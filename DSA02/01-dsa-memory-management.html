Stack, Heap & Arrays

Understand how Java and Python manage memory under the hood — how functions use the call stack, why arrays live on the heap, and what "pass by reference" really means.

01

Introduction to Stack

A stack is a container that stores data following the Last In, First Out (LIFO) principle. Think of a stack of plates in an idli cooker: you can only add or remove plates from the top, and you only ever have access to the topmost one.

Analogy Imagine stacking plates in a dish rack. You always place a new plate on top, and you always take the topmost plate first. The plate at the very bottom was placed first but will be removed last — that is LIFO.
➕
Push — Insert an element on top of the stack.
➖
Pop — Remove the topmost element.
👁️
Access — You can only peek at the topmost element at any given time.
Key Concept The memory region the runtime uses to manage function calls is also called a stack — specifically the call stack — because it follows this exact LIFO principle.
02

The Call Stack

Every time a function is called, a new stack frame is pushed onto the call stack. The frame holds the function's local variables and parameters. When the function returns, its frame is popped and control passes back to the caller.

Example 1 — Three chained functions (Java)

Java
int add(int x, int y)      { return x + y; }
int product(int x, int y)  { return x * y; }
int subtract(int x, int y) { return x - y; }

public static void main() {
    int x = 10, y = 20;
    int temp1 = add(x, y);       // 30
    int temp2 = product(x, y);   // 200
    int temp3 = subtract(x, y);  // -10
    System.out.println(temp1 + temp2 + temp3); // 220
}

As execution proceeds, the call stack builds up and tears down:

Call Stack (grows up ↑)
subtract: x→10, y→20
product: x→10, y→20
add: x→10, y→20
main: x=10, y=20, temp1=30, temp2=200, temp3=-10

Each frame is independent. When add returns, its frame is destroyed; main's frame remains intact with the correct value in temp1.

Example 2 — Nested calls

Java
int add(int x, int y) { return x + y; }

public static void main() {
    int x = 10, y = 20;
    int temp1 = add(x, y);       // 30
    int temp2 = add(temp1, 30);  // 60
    int temp3 = add(temp2, 40);  // 100
    System.out.println(temp3);   // 100
}

The same add function is invoked three times sequentially. Each invocation gets its own stack frame with its own local copies of x and y. Frames are pushed and popped in LIFO order.

03

Reference Variables

A reference variable does not store data directly. Instead it stores a memory address — a pointer to where the actual data lives on the heap.

Analogy Think of a library index card. The card is small and lives in a filing cabinet (the stack). But it contains an address — "Row 4, Shelf 2" — that tells you where to find the actual book (on the heap). The card is the reference variable; the book is the heap object.
Stack
a → #4012(reference / address)
Heap
@ #4012: [ 0 | 0 | 0 ]int array of size 3

In Java, primitive types (int, float, long, char, byte, short, boolean) are stored directly in the stack frame. All other types — arrays, ArrayList, objects — live on the heap and are accessed through a reference variable stored on the stack.

Key Rule Stack holds: primitive values + reference variables (addresses).
Heap holds: the actual data that reference variables point to (arrays, objects).
04

Stack vs Heap

PropertyStackHeap
What is storedPrimitive values, reference variables, call framesObjects, arrays, complex data

Access speedVery fast (LIFO, CPU cache-friendly)Slower (pointer dereference)
AllocationAutomatic (compiler/runtime)Explicit (new) or managed (Python)
Why This Matters When you pass an array to a function, you pass the reference (address), not the data. The function can modify the original array because both pointers point to the same heap location.
05

1-D Arrays on the Heap

When you write int[] ar = new int[3], two things happen: a reference variable ar is created in the current stack frame, and a contiguous block of memory for 3 integers is allocated on the heap. The reference ar holds the starting address of that block.

Java
public static void main() {
    int   x  = 10;         // stored directly on the stack
    int[] ar = new int[3]; // ar → #4K on stack; [0,0,0] on heap @ #4K
    System.out.println(ar);    // prints #4K (memory address)
    System.out.println(ar[2]); // 0 (default int value)
    ar[1] = 7;               // modifies heap @ #4K + 4 bytes
}
Stack (main frame)
x → 10
ar → #4K
Heap
@ #4K: [ 0 | 7 | 0 ]indices: 0 1 2

Accessing ar[2]: read address #4K from the stack, compute #4K + (2 × 4 bytes) = #4K+8, return the value stored there. This is always O(1) regardless of array size.

Array access · O(1) Space for N elements · O(N)
06

Shallow Copy

Assigning one array variable to another (int[] ar2 = ar) copies the reference, not the array data. Both variables now point to the same heap object.

Java
int[] ar  = new int[3];
int[] ar2 = ar;           // ar2 holds the same address as ar
System.out.println(ar);   // #4K
System.out.println(ar2);  // #4K  — same address!
Stack
ar → #4K
ar2 → #4K
Heap
@ #4K: [ 0 | 0 | 0 ]shared by both ar and ar2
Warning — Shallow Copy Pitfall Because both references point to the same data, ar2[0] = 99 also changes what you see through ar[0]. This is a shallow copy — only the reference was duplicated. To get an independent copy (deep copy), use Arrays.copyOf(ar, ar.length) in Java or ar[:] in Python.
07

Reassigning an Array Variable

Assigning a new array to a variable does not modify the old heap block. The reference variable is simply updated to point to a newly allocated heap object. The old one becomes unreachable and is eventually garbage-collected.

Java
int[] ar = new int[3];  // ar → #4K  ([0,0,0])
System.out.println(ar);  // #4K
ar[1] = 9; ar[2] = 5;  // modify in-place at #4K

ar = new int[5];        // ar → #7K  (brand new array)
System.out.println(ar);  // #7K — different address
// ar[1] and ar[2] still carry 9 and 5 at #4K, but ar no longer sees them
Stack
ar → #7K (updated reference)
Heap
@ #4K: [0|9|5] ← unreachable, awaiting GC
@ #7K: [0|0|0|0|0] ← new array of size 5
08

Pass by Reference (Arrays)

When you pass an array to a function, the reference (address) is copied into the function's stack frame. Both the caller and the function now point to the same heap data, so any element-level modification the function makes is visible to the caller.

Java
static void fun(int[] a) {
    System.out.println(a); // prints same address #4K
    a[1] = 5;              // modifies shared heap data at #4K
}

public static void main() {
    int[] ar = new int[3];
    ar[0] = 90; ar[1] = 50;
    fun(ar);
    System.out.println(ar[1]); // 5 — changed by fun()
}
Stack
fun: a → #4K
main: ar → #4K
Heap
@ #4K: [90 | 5 | 0]modified by fun()
Why Not Put Arrays on the Stack? If arrays were stored on the stack, passing an array to a function would require copying every element into a new stack frame — an O(N) operation. By keeping arrays on the heap and passing only the address, the copy is always O(1) regardless of array size.
Reassignment Inside a Function If a function does a = new int[1], it only updates its local copy of the reference. The caller's reference still points to the original array. Only element-level writes (a[i] = x) affect the shared heap object.
09

2-D Arrays in Memory

A 2-D array (int[][] arr = new int[5][4]) is actually an array of references. The outer array lives on the heap and holds 5 addresses; each address points to a separate inner array of size 4 — also on the heap.

Java
int[][] arr = new int[5][4];
// Stack: arr → #4K
// Heap #4K (outer array): [#8K, #5K, #12K, #15K, #9K]
// Each address → its own int[4] on the heap

arr[2][3] = 6;
// Step 1: read #4K → get row addresses
// Step 2: arr[2] → #12K  (address of row 2)
// Step 3: #12K[3] = 6   (set column 3 of row 2)
Stack
arr → #4K
Heap — outer @ #4K
row 0 → #8K
row 1 → #5K
row 2 → #12K
row 3 → #15K
row 4 → #9K
Heap — row arrays
@ #8K: [0|0|0|0]
@ #5K: [0|0|8|0]
@ #12K: [0|0|0|6]
@ #15K: [0|0|0|0]
@ #9K: [0|0|0|0]

Because each row is separately allocated, rows can have different lengths — a jagged array:

Java
int[][] arr = new int[4][];  // outer array only
arr[0] = new int[5];         // row 0: 5 elements
arr[1] = new int[2];         // row 1: 2 elements
arr[2] = new int[3];         // row 2: 3 elements
arr[3] = new int[1];         // row 3: 1 element
Use Case Jagged arrays are useful for storing all subarrays of an array, where each subarray naturally has a different length.

Quiz Walkthroughs

These quizzes test your understanding of stack vs heap behavior. Trace through each call stack before reading the answer.

Quiz 1 — Primitive passed to function

Java
static void change(int a) { a = 50; }

public static void main() {
    int a = 10;
    change(a);
    System.out.println(a); // ?
}
Answer: 10Primitives are passed by value. The function gets its own copy of a in a new stack frame. Setting it to 50 only affects the copy; main's a stays 10.

Quiz 2 — Array element modified

Java
static void change(int[] a) { a[0] = 50; }

public static void main() {
    int[] a = {10};
    change(a);
    System.out.println(a[0]); // ?
}
Answer: 50Both main's a and the function's a point to the same heap array. Setting a[0] = 50 modifies the shared heap data.

Quiz 3 — Array reassigned inside function

Java
static void test(int[] a) {
    a = new int[1];  // local reference redirected to new heap object
    a[0] = 50;
}

public static void main() {
    int[] a = {10};
    test(a);
    System.out.println(a[0]); // ?
}
Answer: 10a = new int[1] inside test only redirects the local reference. The caller's reference still points to the original {10} array. Output is 10.

Quiz 4 — Swap primitives

Java
static void swap(int a, int b) {
    int temp = a; a = b; b = temp;
}

public static void main() {
    int a = 10, b = 20;
    swap(a, b);
    System.out.println(a + " " + b); // ?
}
Answer: 10 20Primitives are passed by value. The swap happens inside the function's stack frame only; main's variables are unaffected. Output is 10 20.

Quiz 5 — Return a new array from a function

Java
static int[] fun(int[] a) {
    a = new int[2];
    a[0] = 50; a[1] = 60;
    return a;  // returns address of the NEW array
}

public static void main() {
    int[] a = {10, 20, 30};
    a = fun(a);               // a now points to the returned array
    System.out.println(a[0]); // ?
}
Answer: 50fun creates a new heap array [50, 60] and returns its address. Back in main, a is reassigned to that address. Now a[0] is 50. The original {10,20,30} becomes unreachable.

Quick Recap

Stack (LIFO)

Holds primitive values and reference variables; each function call creates a new frame that is destroyed on return.

Call Stack

The runtime stack tracking active function calls; each invocation pushes a frame, each return pops one.

Heap

Stores objects and arrays; large, dynamically allocated, and lives beyond the function that created it.

Reference Variable

A stack variable holding a heap address (pointer). Assigning copies the address only, not the data.

Shallow Copy

int[] b = a copies the address — both point to the same array. Changes through either variable affect the shared data.

Pass by Reference

Arrays pass their address. Functions can modify shared heap data; reassignment only affects the local copy of the reference.

2-D Array Layout

Array of row-references on the heap; each row is a separately allocated array, enabling jagged (variable-length) rows.

Why Arrays on Heap

Passing by address is O(1) regardless of array size. If arrays were on the stack, every call would copy O(N) data.
