Encapsulation & Inheritance 🧬

Object-oriented programming organizes code around objects — bundles of data and the behavior that operates on it. This page covers the first two of OOP's four pillars: encapsulation (keep your data safe and hand out a clean, controlled interface) and inheritance (build new types on top of existing ones without copy-pasting). Together they're the backbone of almost every C# class you'll ever read or write. No prior coding background assumed.

01

What is OOP?

A way of organizing programs around objects — things that bundle data with the behavior that acts on it.

Imagine modeling a car in code. Instead of scattering loose variables (speed, fuel) and loose functions (accelerate, refuel) all over the place, object-oriented programming (OOP) packages them together into a single Car object: its data and its behavior live in one place. This makes code easier to reason about, reuse, and protect.

🚗 Analogy

An object is like a vending machine. You interact with it through a few buttons (its public interface) — pick a snack, insert coins. You don't reach into the wiring to dispense a drink. Encapsulation is the locked panel hiding the wiring. Inheritance is the manufacturer building a "drinks machine" and a "snacks machine" on the same base chassis.

02

Classes & Objects

A class is the blueprint. An object is a real thing built from that blueprint.

A class describes what something is and can do: its fields (data), properties (controlled data access), and methods (behavior). An object (or instance) is a concrete value created from that class with new.

C#
// The class = the blueprint
class Dog
{
    public string Name;            // a field (data)
    public void Bark()             // a method (behavior)
    {
        Console.WriteLine($"{Name} says Woof!");
    }
}

// Objects = real things built from the blueprint
Dog rex = new Dog { Name = "Rex" };
Dog fido = new Dog { Name = "Fido" };
rex.Bark();   // Rex says Woof!
fido.Bark();  // Fido says Woof!
🍪 Analogy

A class is a cookie cutter; objects are the cookies. One cutter (class) stamps out many cookies (objects), each with its own decorations (field values) but the same shape (structure and behavior).

✅ Vocabulary you'll see

Member = anything declared inside a class (field, property, method, constructor, event…). Constructor = a special method that runs when you new an object, used to set it up. Instance member = belongs to each object; static member = belongs to the class as a whole.

03

Encapsulation — The Idea

Keep the data private; let the outside world touch it only through a controlled, well-behaved interface.

Encapsulation has two parts: bundling data and behavior together in one class, and information hiding — keeping the internal data private so it can't be tampered with directly. Outsiders interact only through the methods and properties you choose to expose.

Why hide data? The bank account 🏦

C#
class BankAccount
{
    private decimal _balance;     // hidden -- nobody can set it directly

    public void Deposit(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("Amount must be positive.");
        _balance += amount;          // the ONLY way in -- and it's validated
    }

    public decimal GetBalance() => _balance;  // read-only window
}

Because _balance is private, no outside code can write account._balance = -500;. Every change must go through Deposit, which enforces the rule that deposits are positive. The class protects its own invariants.

🔑 The payoff

Encapsulation lets you change the inside without breaking the outside. As long as the public methods keep working the same way, you can rewrite the internals freely. It also makes bugs easier to find: if the balance is wrong, you only have a few methods to inspect — not the entire codebase.

💊 Analogy

Think of a medicine capsule (the word "encapsulation" comes from "capsule"). The active ingredients are sealed inside; you swallow the whole capsule and trust it to release contents safely. You never handle the raw powder directly — and that's exactly the point.

04

Access Modifiers

The keywords that decide who can see and use each member. The toolkit of encapsulation.

C# has six access levels for class members. They range from "everyone" to "only this class." By default, class members are private, and top-level types are internal.

ModifierWho can access itThink of it as
publicAny code, any assembly🌍 Open to everyone
privateOnly the same class🔒 Secret, this class only
protectedThis class + derived classes (any assembly)👨‍👩‍👧 Family (incl. descendants)
internalAny code in the same assembly🏢 Same building/project
protected internalSame assembly OR derived classes anywhere🏢 + 👨‍👩‍👧 (the broader of the two)
private protectedDerived classes in the same assembly only🔒👨‍👩‍👧 (the narrower of the two)

The two combinations explained 🧩

➕

protected internal = OR

The more permissive one: reachable if you're in the same assembly or you're a derived class anywhere. It's a union of protected and internal.

✖️

private protected = AND

The more restrictive one (C# 7.2+): reachable only if you're a derived class and in the same assembly. It's the intersection.

C#
public class Base
{
    public int Anyone;              // everywhere
    private int _onlyHere;          // this class only
    protected int Family;           // this + derived (any assembly)
    internal int SameProject;       // anywhere in this assembly
    protected internal int Broader;  // assembly OR derived anywhere
    private protected int Narrower;  // derived AND same assembly
}
⚠️ Rules that catch people out

You can't make a member more accessible than its containing type (a public method can't return a private type). A derived class can't be more accessible than its base class. And structs can't use protected, protected internal, or private protected — because structs don't support inheritance, those "for-derived-classes" levels are meaningless.

✅ Default to private

Start every field as private and only widen access when there's a real need. The less you expose, the smaller your "surface area" for bugs and the freer you are to refactor internals later.

05

Properties

C#'s idiomatic way to do encapsulation: look like a field, behave like a method.

A property is a member that looks like a field from the outside (p.Name = "Sam";) but is backed by a get accessor (runs when read) and a set accessor (runs when assigned). This lets you add validation, logging, or computation without changing how callers use it.

From field to property 🎛️

C#
class Person
{
    // 1) Auto-property: compiler generates the hidden backing field
    public string Name { get; set; }

    // 2) Property with validation in the setter
    private int _age;
    public int Age
    {
        get => _age;
        set
        {
            if (value < 0) throw new ArgumentException("Age can't be negative.");
            _age = value;        // 'value' is the incoming assigned value
        }
    }

    // 3) Computed (read-only) property -- no stored data of its own
    public bool IsAdult => _age >= 18;
}

Controlling read vs write 🔏

The get and set can have different access levels — a common pattern is a public getter with a private setter, so the value is read-only to the outside but changeable inside the class.

C#
public decimal Balance { get; private set; }   // read outside, write inside

// 'init' (C# 9): can ONLY be set during construction / object initializer
public string Id { get; init; }

// 'required' (C# 11): caller MUST set this when creating the object
public required string Email { get; init; }
🔑 get / set / init

get reads, set writes any time, init writes only while the object is being created (great for immutable objects). Add required to force callers to supply a value at construction. Note: you can't pass a property as a ref or out argument — it isn't a variable, it's accessor methods in disguise.

✅ Prefer properties over public fields

Even a plain { get; set; } auto-property is better than a public field: you can later add validation or make the setter private without breaking any caller's code. Public fields lock you in; properties keep the door open.

06

Inheritance — The Idea

Build a new class on top of an existing one, reusing its members and adding or changing behavior.

Inheritance lets a derived class (child) automatically gain the members of a base class (parent), then add its own. It models an "is-a" relationship: a Dog is an Animal, so Dog inherits everything an Animal has, plus dog-specific extras.

An inheritance hierarchy 🌳

Animal
Dog
Cat
Bird

Dog, Cat, and Bird each inherit Animal's members (like Name and Eat()) and add their own.

C#
class Animal
{
    public string Name { get; set; }
    public void Eat() => Console.WriteLine($"{Name} is eating.");
}

// 'Dog : Animal' means "Dog inherits from Animal"
class Dog : Animal
{
    public void Fetch() => Console.WriteLine($"{Name} fetches the ball.");
}

var rex = new Dog { Name = "Rex" };
rex.Eat();    // inherited from Animal
rex.Fetch();  // Dog's own method
⚠️ Single inheritance for classes

A C# class can inherit from only one base class (no multiple inheritance of classes — it avoids the famous "diamond problem"). You can, however, implement many interfaces. Every class ultimately derives from System.Object, which is why every object has ToString(), Equals(), and GetHashCode().

✅ "is-a" vs "has-a"

Use inheritance only for genuine "is-a" relationships (a SavingsAccount is a BankAccount). If it's really "has-a" (a Car has an Engine), prefer composition — store the other object as a field — instead of inheriting. Overusing inheritance leads to rigid, fragile hierarchies.

07

base & Constructors

When you build a derived object, the base part has to be set up first. base is how you do it.

Creating a Dog also creates the Animal inside it. The base class constructor runs before the derived one. The base(...) keyword lets a derived constructor pass arguments up to the base constructor.

C#
class Animal
{
    public string Name { get; }
    public Animal(string name)        // base constructor
    {
        Name = name;
        Console.WriteLine("Animal created");
    }
}

class Dog : Animal
{
    public string Breed { get; }
    // ': base(name)' forwards 'name' up to the Animal constructor
    public Dog(string name, string breed) : base(name)
    {
        Breed = breed;
        Console.WriteLine("Dog created");
    }
}

var d = new Dog("Rex", "Labrador");
// Output order:  Animal created  then  Dog created
🔑 Order of construction

Base before derived — always. The base must finish setting itself up before the derived class adds its layer. If the base has only a parameterized constructor (no parameterless one), the derived class is required to call : base(...) explicitly, or you'll get a compile error.

You also use base to call a base member from inside an override (see the next section): base.Speak(); runs the parent's version.

08

virtual & override

Let a derived class replace a base method — and have the runtime pick the right version automatically.

By default in C#, methods are not overridable. To allow a derived class to provide its own version, mark the base method virtual, and the derived method override. The override keyword is required — C# never overrides implicitly.

C#
class Animal
{
    public virtual void Speak() => Console.WriteLine("Some generic sound");
}

class Dog : Animal
{
    public override void Speak() => Console.WriteLine("Woof!");
}

class Cat : Animal
{
    public override void Speak() => Console.WriteLine("Meow!");
}

// Polymorphism: same variable type, different behavior at runtime
Animal[] zoo = { new Dog(), new Cat(), new Animal() };
foreach (Animal a in zoo)
    a.Speak();   // Woof!  Meow!  Some generic sound

How the runtime decides 🎯

📞You call a.Speak()a is typed as Animal, but holds a Dog
🔍Method is virtual?Runtime looks at the object's actual type, not the variable's type
🐕Runs the most-derived overrideCalls Dog.Speak() → "Woof!"
🔑 This is polymorphism

"Polymorphism" = "many forms." One line — a.Speak() — produces different behavior depending on the real object. The runtime always calls the most-derived override of a virtual member. This is the engine behind flexible, extensible designs: add a new Animal subclass and existing loops just work.

✅ Reuse the base version

Inside an override you can still call the parent's logic with base.Speak(); — handy when you want to extend behavior ("do what Animal does, then add a bit") rather than replace it entirely.

09

abstract Classes

A base class that can't be instantiated and can demand that children fill in the blanks.

Sometimes a base class is only meaningful as a template — there's no such thing as a generic "Shape" you'd create on its own; you create a Circle or a Square. Mark such a class abstract: it can't be instantiated with new, and it can declare abstract methods (no body) that derived classes are required to implement.

C#
abstract class Shape
{
    // abstract = no implementation; children MUST override it
    public abstract double Area();

    // abstract classes can ALSO have normal, shared code
    public void Describe() => Console.WriteLine($"Area is {Area():F2}");
}

class Circle : Shape
{
    public double Radius { get; set; }
    public override double Area() => Math.PI * Radius * Radius;
}

// var s = new Shape();   // ❌ compile error: can't instantiate abstract class
Shape s = new Circle { Radius = 2 };
s.Describe();   // Area is 12.57
🔑 abstract vs virtual

A virtual method has a default body and may be overridden. An abstract method has no body and must be overridden by any non-abstract derived class. An abstract method is implicitly virtual. An abstract class is a partial blueprint: some bricks laid, some left for children to supply.

📋 Analogy

An abstract class is a job description with some duties filled in and others marked "candidate must define." You can't hire the job description itself — you hire a real person (a concrete subclass) who completes every required duty.

10

sealed & new

Two keywords for the opposite of opening up: sealed stops inheritance, new hides instead of overrides.

sealed — "no further down" 🔒

sealed on a class prevents anyone from inheriting it. On an override method, it stops that method from being overridden any further down the chain.

C#
sealed class Logger { }          // nothing can inherit from Logger

class Mammal : Animal
{
    // override here, but seal it so subclasses can't change it again
    public sealed override void Speak() => Console.WriteLine("(mammal sound)");
}

new — hiding, not overriding ⚠️

new on a member hides a base member with the same name instead of overriding it. The big difference: which method runs depends on the variable's declared type, not the object's real type. This is usually a source of bugs, not a feature.

C#
class Base    { public void Show() => Console.WriteLine("Base"); }
class Derived : Base { public new void Show() => Console.WriteLine("Derived"); }

Derived d = new Derived();
Base b = d;        // same object, base-typed variable

d.Show();   // "Derived"  -- variable is Derived
b.Show();   // "Base"     -- variable is Base (hiding, not overriding!)
⚠️ override vs new — the trap

With override, the object's real type wins — b.Speak() would run the derived version. With new, the variable's type wins, so the same object behaves differently depending on how you're holding it. If you see surprising "wrong method ran" behavior, check whether someone used new where they meant override.

🔑 Quick contrast

virtual/override → polymorphic, real type decides (the common, correct choice). new → hides, declared type decides (rare, deliberate). sealed → blocks further inheritance/overriding.

Worked Example

Encapsulation and inheritance working together — a small bank-account hierarchy.

This ties everything together: a private balance (encapsulation), a public validated interface (properties + methods), a base class with shared logic, and a derived class that overrides one behavior while reusing the rest via base.

C#
abstract class Account
{
    // ENCAPSULATION: balance is hidden; read-only to the outside
    public decimal Balance { get; protected set; }
    public string Owner { get; }

    protected Account(string owner) => Owner = owner;

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Must be positive.");
        Balance += amount;
    }

    // each account type withdraws under its own rules
    public abstract bool Withdraw(decimal amount);
}

class SavingsAccount : Account
{
    public SavingsAccount(string owner) : base(owner) { }

    // no overdraft allowed
    public override bool Withdraw(decimal amount)
    {
        if (amount <= 0 || amount > Balance) return false;
        Balance -= amount;
        return true;
    }
}

Account acc = new SavingsAccount("Mara");
acc.Deposit(100);                // inherited, shared logic
bool ok = acc.Withdraw(40);       // SavingsAccount's own rule -> true
Console.WriteLine(acc.Balance);   // 60
✅ What each pillar did here

Encapsulation: Balance can be set only inside the hierarchy (protected set), and Deposit validates input. Inheritance: SavingsAccount reuses Deposit, Balance, and Owner for free. Abstraction/polymorphism preview: Withdraw is abstract, so a future CheckingAccount can allow overdrafts without touching any existing code.

13

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is encapsulation? Junior

Encapsulation means bundling data and the behavior that operates on it into one class, and hiding the internal data so it can only be accessed through a controlled interface. For example, making a balance field private and only allowing changes through a validated Deposit method. It protects the object from being put into an invalid state.

Q2 What is inheritance? Junior

Inheritance lets one class (the derived/child class) reuse the members of another (the base/parent class) and add or change behavior. It models an "is-a" relationship — a Dog is an Animal — and avoids duplicating shared code. In C# you write it as class Dog : Animal.

Q3 What's the difference between a class and an object? Junior

A class is the blueprint — it defines what fields and methods a type has. An object is an actual instance created from that blueprint using new. One class can produce many objects, each with its own data. Think cookie cutter (class) versus the individual cookies (objects).

Q4 Name the access modifiers and what public and private mean. Junior

The six are public, private, protected, internal, protected internal, and private protected. public means any code anywhere can use the member; private means only code inside the same class can. By default, class members are private.

Q5 What is a property, and why use one instead of a public field? Junior

A property looks like a field from outside but runs code through get and set accessors. Prefer it over a public field because you can add validation, make the setter private, or change the internals later without breaking callers. A public field gives you no such control. Even a simple { get; set; } auto-property keeps your options open.

Q6 What does virtual and override do? Junior

virtual on a base method says "derived classes are allowed to replace this." override on a derived method provides that replacement. After that, calling the method runs the derived version even when you're holding the object in a base-typed variable. In C# the override keyword is required — overriding never happens by accident.

Q7 What is an abstract class? Junior

An abstract class is a base class you can't create directly with new — it exists only to be inherited. It can include normal shared code and abstract methods (methods with no body) that every concrete derived class must implement. Shape with an abstract Area() is the classic example.

Q8 Can a C# class inherit from more than one base class? Junior

No — C# classes support single inheritance: at most one base class. This avoids the ambiguity of the "diamond problem." However, a class can implement many interfaces, which is how C# handles cases where you need behavior from multiple sources.

🟠 Mid-level — going deeper (2–5 years)

Q9 What's the difference between protected internal and private protected? Mid

protected internal is the broader one — accessible if you're in the same assembly OR a derived class anywhere (a union). private protected is the narrower one (C# 7.2+) — accessible only if you're a derived class AND in the same assembly (an intersection). One widens access, the other tightens it.

Q10 What's the difference between override and new (method hiding)? Mid

override is polymorphic: the method that runs depends on the object's real type, so a base-typed variable still calls the derived version. new hides the base member: which method runs depends on the variable's declared type. Hold the same object as Base and it runs the base method; as Derived and it runs the derived one. override is almost always what you want; new is a rare, deliberate choice and a common source of confusion.

Q11 What do init and required add to properties? Mid

init (C# 9) makes a setter usable only during object construction or in an object initializer, then read-only afterward — perfect for immutable objects. required (C# 11) forces the caller to assign that property when creating the object, so you can't forget a mandatory value. They're often combined: public required string Email { get; init; }.

Q12 Why prefer composition over inheritance sometimes? Mid

Inheritance creates a tight, permanent coupling and only fits true "is-a" relationships. Composition (storing another object as a field — "has-a") is more flexible: you can swap the contained object, combine several, and avoid deep fragile hierarchies where a base-class change ripples through everything. Rule of thumb: use inheritance for genuine is-a; use composition for has-a and for sharing behavior without a type relationship.

Q13 What does sealed do, and why might you use it? Mid

sealed on a class prevents inheritance; on an override member it prevents further overriding down the chain. Reasons to use it: to lock down a class whose behavior must not be altered (security or correctness), to signal design intent, and occasionally for a minor performance gain since the runtime can resolve sealed members more directly. Use it sparingly — over-sealing makes code harder to test and extend.

Q14 In what order do constructors run in an inheritance chain? Mid

From the base up: the base class constructor runs first, then the derived class constructor. The base part of the object must be fully initialized before the derived class layers its own setup on top. You forward arguments to the base with : base(...); if the base lacks a parameterless constructor, the derived class must call a base constructor explicitly or it won't compile.

Q15 Can you give a property different access for get and set? Mid

Yes. A very common pattern is public T Value { get; private set; } — readable from anywhere, writable only inside the class. The accessor you restrict must be more restrictive than the property itself. This is a clean way to expose read-only state while still mutating it internally. (Note: you can't pass a property as ref/out because it's accessor methods, not a real variable.)

Q16 Why can't a struct use protected? Mid

Because structs can't be inherited — they're implicitly sealed. protected, protected internal, and private protected all grant access to derived classes, but a struct can never have any, so those modifiers would be meaningless. Struct members can only be public, internal, or private.

🔴 Senior — synthesis & judgment (5–7 years)

Q17 Explain how virtual dispatch actually works at runtime. Senior

Each type with virtual members has a method table (v-table) — a list of pointers to the actual method implementations for that type. When you call a virtual member, the runtime looks at the object's real type, finds its v-table, and jumps to the most-derived implementation, regardless of the variable's compile-time type. Non-virtual calls are resolved at compile time directly (and can be inlined). This indirection is why override is polymorphic while new (a compile-time, declared-type decision) is not.

Q18 Why is calling a virtual method from a constructor dangerous? Senior

During base-class construction, the derived constructor hasn't run yet — so the derived object's fields are still at their defaults. But virtual dispatch will already call the derived override (the object's real type is set early). That override may then read fields that haven't been initialized, producing subtle null/zero bugs. The guidance is to avoid calling virtual members from constructors, or to design the override to tolerate a partially-constructed object.

Q19 When would you choose an abstract class over an interface? Senior

Use an abstract class when derived types share real implementation and state (fields, constructors, common method bodies) and form a genuine is-a hierarchy — and recall a class can have only one base. Use an interface to define a capability/contract that unrelated types can implement, and when you need a type to support multiple such contracts. Modern C# blurs this with default interface methods, but the rule of thumb holds: shared implementation + single hierarchy → abstract class; multiple, cross-cutting contracts → interface.

Q20 What's the fragile base class problem? Senior

It's the risk that a seemingly safe change to a base class silently breaks derived classes that depended on its internal behavior — for example, if the base starts calling one of its own methods differently, an override may now run at an unexpected time. Deep inheritance hierarchies amplify this. Mitigations: keep base classes shallow and well-documented about what's overridable, favor sealed by default and open up deliberately, prefer composition for sharing behavior, and design base classes for inheritance explicitly (the "design for inheritance or prohibit it" principle).

Q21 How does encapsulation support maintainability and security? Senior

Maintainability: hiding internals behind a stable public interface means you can refactor or optimize the implementation without touching callers — the change is localized. It also concentrates invariant-enforcing logic in one place, so bugs have fewer hiding spots. Security/robustness: private state can't be put into an invalid or malicious configuration from outside; every mutation runs through validated entry points. It shrinks the trusted surface area and makes reasoning about correctness tractable.

Q22 A derived class needs to extend, not replace, a base behavior. How do you design that? Senior

Mark the base method virtual, override it in the derived class, and call base.Method() inside the override to run the base logic, then add the extra behavior around it. This is the "extend" pattern. If you want to mandate that subclasses always run shared setup, use the template method pattern: a non-virtual public method in the base that calls protected virtual "hook" methods, so the overall flow is fixed but specific steps are customizable. That keeps control in the base while still allowing extension.

Q23 What does it mean that every type derives from System.Object? Senior

It means there's a single rooted type hierarchy: every class, struct, and other type ultimately inherits from object, so every value has ToString(), Equals(), GetHashCode(), and GetType(). This is what lets a variable of type object hold anything, enables reflection, and gives common behavior you can override (e.g. a meaningful ToString()). For value types, treating them as object triggers boxing, which is the cost side of that universality.

Q24 How do access modifiers interact with assemblies and the InternalsVisibleTo attribute? Senior

internal (and the assembly half of protected internal/private protected) scopes access to the current assembly — the compiled .dll/.exe. The InternalsVisibleTo assembly attribute selectively exposes internal (and private protected for derived types) members to a named friend assembly — commonly used so a unit-test project can reach internals without making them public. It's a controlled crack in the assembly boundary, not a blanket opening, so you keep encapsulation intact for everyone except the explicitly trusted assembly.
