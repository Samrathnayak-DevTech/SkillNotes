Delegates, Events & Func/Action/Predicate 📡

Normally you call a method. But what if you want to store a method in a variable, pass it to another method, or let many methods react to one thing happening? That's what delegates are for — type-safe references to methods. Func, Action, and Predicate are ready-made delegates you'll use daily (they're exactly what LINQ's lambdas plug into), and events build on delegates to power the publisher/subscriber pattern behind every button click. No prior coding background assumed.

01

Overview

All four topics share one idea: methods can be treated as values — stored, passed around, and invoked later.

In most early programming you write a method and call it directly. These features unlock something more powerful: methods as data. A delegate is a variable that holds a method. Func / Action / Predicate are pre-built delegate types so you rarely declare your own. Lambdas are a compact way to write the method inline. And events wrap a delegate so objects can broadcast "something happened" to any number of listeners.

02

Delegates — The Idea

A delegate is a type-safe reference to a method — a "function pointer," but the compiler guarantees the signatures match.

A delegate is a type that holds a reference to a method with a specific signature (parameters + return type). You declare the delegate type, assign any matching method to a variable of that type, then invoke it as if it were the method itself. Because the compiler checks the signature, it's type-safe — unlike raw function pointers in C/C++.

C#
// 1) declare a delegate TYPE: "a method taking a string, returning void"
delegate void Greeter(string name);

// 2) a method with the MATCHING signature
void SayHello(string name) => Console.WriteLine($"Hello, {name}");

// 3) assign the method to a delegate variable (note: no parentheses)
Greeter greet = SayHello;

// 4) invoke it -- this actually calls SayHello
greet("Sam");            // Hello, Sam
greet.Invoke("Mara");    // same thing, explicit form
greet
delegate variable
→
SayHello
the method it points to
→
"Hello, Sam"
runs when invoked
⚠️ Signatures must match exactly

You can only assign a method to a delegate if its parameter types and return type match the delegate. Assigning a method with a different signature is a compile error. This is the "type-safe" guarantee — the reason delegates are safer than C++ function pointers.

📺 Analogy

A delegate is like a TV remote's "favorite" button. The button (delegate variable) is wired to a channel (a method). Press it and you jump to that channel. You can re-wire the button to a different channel later — as long as it's a valid channel (matching signature).

03

Why Delegates?

The point: pass behavior as an argument, so one method can do different things without being rewritten.

The killer use of a delegate is the callback — handing a method to another method so it can call you back. This decouples the what from the how: a sorting routine takes a "compare" callback; a processing loop takes an "on each item" callback. You change behavior by passing a different method, not by editing the routine.

C#
delegate int Transform(int x);

// this method takes BEHAVIOR (a delegate) as a parameter
int[] ApplyToAll(int[] items, Transform op)
{
    var result = new int[items.Length];
    for (int i = 0; i < items.Length; i++)
        result[i] = op(items[i]);    // call back into the passed-in method
    return result;
}

int[] nums = { 1, 2, 3 };
int[] doubled = ApplyToAll(nums, x => x * 2);   // 2, 4, 6
int[] squared = ApplyToAll(nums, x => x * x);   // 1, 4, 9 -- same method, new behavior
📞

Callbacks

"Call me back when you're done / for each item." The basis of async APIs, event handlers, and LINQ.

🔌

Decoupling

A method doesn't hard-code what it does with each item — the caller supplies that, so the method stays reusable.

🧩

Strategy pattern

Swap algorithms at runtime by passing a different delegate — sorting order, validation rule, formatting.

🔑 This is "functional" C#

Passing methods as arguments is the heart of functional programming. LINQ is built entirely on it — every operator takes a delegate describing the rule. Once delegates click, LINQ stops feeling like magic.

04

Multicast Delegates

A single delegate can point to several methods at once. Invoking it calls them all, in order.

Delegates are multicast: use += to add methods and -= to remove them. Invoking the delegate calls every attached method in the order they were added. This is the foundation events are built on.

C#
Action notify = () => Console.WriteLine("Email sent");
notify += () => Console.WriteLine("SMS sent");   // add a second method
notify += () => Console.WriteLine("Log written");

notify();   // calls all three, in order:
            // Email sent / SMS sent / Log written

notify -= () => Console.WriteLine("Log written");  // (would need same ref to actually remove)
notify()
one invoke
→
Email

→
SMS
2nd
→
Log
3rd
⚠️ Return values: only the last one survives

If a multicast delegate has a non-void return type, invoking it runs all the methods but you only get the return value of the last one — the rest are lost. That's why multicast delegates (and events) are almost always void. If you need every result, invoke each method in GetInvocationList() yourself.

⚠️ One exception stops the chain

If one method in the chain throws, the remaining methods are not called. For independent subscribers where one failure shouldn't block the others, iterate GetInvocationList() and wrap each call in its own try/catch.

05

Func, Action, Predicate

Three built-in generic delegates that cover almost every callback you'll ever need — so you rarely declare your own.

Declaring a custom delegate type for every signature gets tedious. .NET ships generic delegates that cover the common cases. Reach for these by default:

DelegateParametersReturnsUse for
Func<…, TResult>0–16 inputsA value (the last type arg)Compute & return something
Action<…>0–16 inputsvoidDo something, return nothing
Predicate<T>Exactly 1 inputboolTest a condition (filtering)
C#
// Func: takes inputs, RETURNS a value. Last type arg = return type.
Func<int, int, int> add = (a, b) => a + b;     // (int, int) -> int
int sum = add(3, 4);                          // 7

// Action: takes inputs, returns NOTHING (void).
Action<string> log = msg => Console.WriteLine(msg);
log("started");                              // started

// Predicate: takes ONE input, returns bool. Same as Func<T, bool>.
Predicate<int> isEven = n => n % 2 == 0;
bool b = isEven(4);                          // true

// used directly by framework methods:
var list = new List<int> { 1, 2, 3, 4 };
var evens = list.FindAll(isEven);            // FindAll takes a Predicate<T>
list.ForEach(log);                            // hmm -- type mismatch? see note
🔑 Predicate<T> is just Func<T, bool>

They're interchangeable in meaning. Predicate<T> predates the Func family and survives mainly in older APIs like List<T>.FindAll and RemoveAll. New LINQ-style APIs use Func<T, bool>. Knowing they're equivalent avoids confusion when one API wants one and another wants the other.

⚠️ Func/Action can't do ref/out/params

Because their type parameters are fixed, Func and Action can't express ref, out, or params parameters. For those (e.g. a TryParse-style bool M(string s, out int n)) you must declare a custom delegate type — one of the few remaining reasons to.

06

Lambdas & Anonymous Methods

The compact, inline way to write the method a delegate points to — no separate named method needed.

You don't need a named method to fill a delegate. A lambda expression writes the method inline: parameters on the left of =>, body on the right. This is what you pass to LINQ, Func, Action, and event handlers all the time.

C#
// all three assign the SAME behavior to a delegate:

// 1) named method
int Square(int x) => x * x;
Func<int, int> a = Square;

// 2) anonymous method (older syntax, still valid)
Func<int, int> b = delegate (int x) { return x * x; };

// 3) lambda (modern, concise)
Func<int, int> c = x => x * x;             // expression body
Func<int, int> d = x => { return x * x; };   // statement body
🔑 Lambdas can capture variables (closures)

A lambda can use variables from the surrounding scope — this is called a closure. The lambda "captures" the variable (not just its value at that moment), which is powerful but can surprise you in loops with deferred execution (a pitfall we saw in the LINQ topic).

C#
int factor = 10;
Func<int, int> scale = x => x * factor;   // captures 'factor'
Console.WriteLine(scale(5));               // 50
factor = 100;
Console.WriteLine(scale(5));               // 500 -- it captured the variable, not the value
✅ Lambda vs named method

Use a lambda for short, inline, one-off logic. Extract a named method when the logic is long, reused, or deserves a descriptive name. Readability wins — if a lambda grows past a couple of lines, give it a name.

07

Events — The Idea

A controlled wrapper around a delegate: objects can broadcast "something happened," and any number of listeners can react.

An event implements the publisher/subscriber pattern. A publisher declares an event; subscribers attach handler methods with +=. When the publisher raises the event, every subscribed handler runs — and the publisher doesn't need to know who's listening. It's a multicast delegate with guardrails.

📢 Publisher
raises event
→
🔔 Subscriber A — send email
🔔 Subscriber B — update UI
🔔 Subscriber C — write log
🔑 Why an event instead of a plain delegate?

An event restricts what outsiders can do. With a public delegate field, anyone could overwrite it (x = ... wiping out other subscribers) or invoke it themselves. An event only exposes += and -= to the outside — subscribers can join or leave, but only the declaring class can raise it or clear it. It's encapsulation for the delegate.

C#
class Button
{
    // declare an event based on a delegate type
    public event Action Clicked;

    public void Press()
    {
        Console.WriteLine("(click)");
        Clicked?.Invoke();   // raise it -- ?. guards against no subscribers
    }
}

var btn = new Button();
btn.Clicked += () => Console.WriteLine("Subscriber 1 reacted");  // subscribe
btn.Clicked += () => Console.WriteLine("Subscriber 2 reacted");
btn.Press();   // (click) / Subscriber 1 reacted / Subscriber 2 reacted
⚠️ Always null-check when raising

If no one has subscribed, the event delegate is null and invoking it throws. The idiom Clicked?.Invoke() (null-conditional) safely does nothing when there are no subscribers. Always raise events this way.

08

The Standard Event Pattern

.NET has a convention for events so they're consistent everywhere: EventHandler, an EventArgs, and an On… method.

While you can base an event on any delegate, .NET libraries follow a standard pattern so handlers look the same everywhere. The convention uses the built-in EventHandler<TEventArgs> delegate, whose signature is (object sender, TEventArgs e) — "who raised it" plus "details about it."

C#
// 1) carry event data in a class derived from EventArgs
class PriceChangedEventArgs : EventArgs
{
    public decimal NewPrice { get; init; }
}

class Stock
{
    private decimal _price;

    // 2) declare the event using EventHandler<TEventArgs>
    public event EventHandler<PriceChangedEventArgs> PriceChanged;

    // 3) a protected virtual "On..." method raises it (overridable by subclasses)
    protected virtual void OnPriceChanged(PriceChangedEventArgs e)
        => PriceChanged?.Invoke(this, e);   // 'this' = the sender

    public decimal Price
    {
        get => _price;
        set
        {
            if (_price == value) return;
            _price = value;
            OnPriceChanged(new PriceChangedEventArgs { NewPrice = value });
        }
    }
}

// subscriber: handler matches (object sender, TEventArgs e)
var stock = new Stock();
stock.PriceChanged += (sender, e) => Console.WriteLine($"New price: {e.NewPrice}");
stock.Price = 99.5m;   // New price: 99.5
📨

sender

The object that raised the event (this). Lets one handler serve many publishers and know which fired.

📦

EventArgs

Carries the event's data. Use EventArgs.Empty when there's no data; derive a class when there is.

🔧

On… method

A protected virtual method that raises the event, so derived classes can extend or intercept it.

✅ Use EventHandler / EventHandler<T>

Prefer the built-in EventHandler (no data) or EventHandler<TEventArgs> (with data) over custom delegate types for events. It makes your events instantly familiar to any C# developer and tooling.

09

Events vs Delegates

An event is a delegate with access restrictions. Here's exactly what changes — and the leak to watch for.

AspectPlain delegate fieldEvent
Subscribe / unsubscribe (+=/-=)AnyoneAnyone
Overwrite with =Anyone (dangerous!)Only the declaring class
Invoke / raiseAnyoneOnly the declaring class
IntentA stored callbackA broadcast notification
🔑 One-line summary

An event is a delegate that the outside world can only subscribe to and unsubscribe from — it can't be cleared, reassigned, or fired by anyone except the class that owns it. It's the encapsulated, broadcast-safe form of a delegate.

⚠️ The classic memory leak: forgetting to unsubscribe

When an object subscribes to a long-lived publisher's event, the publisher holds a reference to the subscriber (through the delegate). If the subscriber is never unsubscribed (-=), the garbage collector can't reclaim it — even after you think you're done with it. Always unsubscribe when a subscriber's life is shorter than the publisher's (e.g. in Dispose).

C#
// subscribe -- but keep the SAME reference so you can unsubscribe
EventHandler handler = (s, e) => Console.WriteLine("reacting");
publisher.SomethingHappened += handler;

// later, when done -- prevents the leak
publisher.SomethingHappened -= handler;
✅ Unsubscribe needs the same reference

To remove a handler with -=, you must pass the same delegate instance you added. A fresh lambda (even with identical code) is a different object and won't remove the original. Store the handler in a variable if you'll need to detach it.

10

Worked Example

A small notification system tying delegates, built-in delegates, lambdas, and events together.

An Order publisher raises a Shipped event. Multiple independent subscribers react — and one of them is supplied as a Func/Action callback, showing how the pieces interlock.

C#
class OrderShippedEventArgs : EventArgs
{
    public string OrderId { get; init; }
}

class Order
{
    public string Id { get; init; }
    public event EventHandler<OrderShippedEventArgs> Shipped;

    public void Ship()
    {
        Console.WriteLine($"Order {Id} shipped");
        Shipped?.Invoke(this, new OrderShippedEventArgs { OrderId = Id });
    }
}

// a method that takes BEHAVIOR (Action) -- a callback
void NotifyVia(Order order, Action<string> channel)
    => order.Shipped += (s, e) => channel($"Order {e.OrderId}");

var order = new Order { Id = "A100" };

// subscribers -- lambdas as event handlers
order.Shipped += (s, e) => Console.WriteLine($"  📧 emailing about {e.OrderId}");
NotifyVia(order, msg => Console.WriteLine($"  📱 SMS: {msg}"));   // Action callback

order.Ship();
// Order A100 shipped
//   📧 emailing about A100
//   📱 SMS: Order A100
✅ Every concept in one place

The event (Shipped) broadcasts to many subscribers; each handler is a lambda; NotifyVia takes an Action (a built-in delegate) as a callback and subscribes it; raising the event runs all handlers via the underlying multicast mechanism. That's the whole topic working together.

12

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is a delegate? Junior

A delegate is a type-safe reference to a method — like a variable that holds a method instead of data. You can store a method in it, pass it to other methods, and invoke it later. "Type-safe" means the compiler checks that the method's signature (its parameters and return type) matches the delegate, so it's safer than a raw function pointer.

Q2 What's the difference between Func, Action, and Predicate? Junior

All three are built-in delegates. Func takes inputs and returns a value (the last type argument is the return type). Action takes inputs and returns nothing (void). Predicate<T> takes one input and returns a bool — used for testing a condition. So: returns a value → Func; returns void → Action; tests true/false → Predicate.

Q3 What is a lambda expression? Junior

A lambda is a short, inline anonymous method written with the => arrow: parameters on the left, body on the right, e.g. x => x * 2. It's a concise way to provide the method a delegate needs without declaring a separate named method. LINQ queries use lambdas constantly.

Q4 What is an event? Junior

An event lets one object (the publisher) notify other objects (subscribers) when something happens. Subscribers attach handler methods with +=; when the publisher raises the event, all subscribed handlers run. A button's "click" is the classic example — the button publishes the click, and your code subscribes to react.

Q5 How do you subscribe to and unsubscribe from an event? Junior

Use += to subscribe (add a handler) and -= to unsubscribe (remove it): button.Clicked += MyHandler; then later button.Clicked -= MyHandler;. The handler is a method (or lambda) whose signature matches the event's delegate.

Q6 What is a multicast delegate? Junior

A delegate that points to more than one method at once. You add methods with +=; invoking the delegate then calls all of them in the order they were added. Events rely on this — that's how one event can notify many subscribers.

Q7 Why is a delegate called "type-safe"? Junior

Because the compiler enforces that any method you assign to it has the exact matching signature — same parameter types and return type. If they don't match, you get a compile error. This prevents the kind of crashes that raw function pointers (as in C++) can cause when signatures don't line up.

🟠 Mid-level — going deeper (2–5 years)

Q8 What's the difference between a delegate and an event? Mid

An event is a delegate with access restrictions. With a plain public delegate field, any code could overwrite it with = (wiping other subscribers) or invoke it directly. An event only exposes += and -= to outsiders — they can subscribe or unsubscribe, but only the declaring class can raise it or reassign it. So an event is the encapsulated, broadcast-safe form of a multicast delegate.

Q9 What happens to the return value of a multicast delegate? Mid

All the methods run, but you only receive the return value of the last one — the others are discarded. That's why multicast delegates and events are almost always declared with a void return. If you genuinely need every method's result, call each one yourself via GetInvocationList() and collect the returns.

Q10 Is Predicate<T> the same as Func<T, bool>? Mid

Semantically yes — both take a T and return a bool. They aren't the same type (you can't assign one to a variable of the other directly), but they're interchangeable in meaning. Predicate<T> is older and appears in APIs like List<T>.FindAll and RemoveAll; modern LINQ-style APIs use Func<T, bool>.

Q11 Why must you null-check (or use ?.) before raising an event? Mid

If no one has subscribed, the event's backing delegate is null, and invoking null throws a NullReferenceException. The idiom MyEvent?.Invoke(this, e) uses the null-conditional operator to safely do nothing when there are no subscribers. It's also thread-safer than an explicit if (MyEvent != null) check because it reads the delegate once.

Q12 Describe the standard .NET event pattern. Mid

Use the built-in EventHandler<TEventArgs> delegate, whose signature is (object sender, TEventArgs e) — sender is the publisher, e carries the data. Define a class deriving from EventArgs to hold that data, declare the event as public event EventHandler<…>, and raise it from a protected virtual OnXxx(…) method so subclasses can extend it. This consistency makes events recognizable across all .NET code.

Q13 When would you still declare a custom delegate type instead of using Func/Action? Mid

When the signature needs features Func/Action can't express: ref, out, or params parameters (e.g. a TryParse-style bool M(string s, out int n)). A named custom delegate can also improve readability by giving an intent-revealing name to a recurring signature. Otherwise, prefer the built-ins.

Q14 What is a closure, and why can it surprise you? Mid

A closure is a lambda that captures a variable from its surrounding scope — it captures the variable, not a snapshot of its value. So if the variable changes later, the lambda sees the new value when it runs. Combined with deferred execution (e.g. building lambdas in a loop and running them afterward), this can cause every lambda to use the variable's final value. The fix is to copy the variable into a fresh local inside the loop.

🔴 Senior — synthesis & judgment (5–7 years)

Q15 How can events cause memory leaks, and how do you prevent them? Senior

When a subscriber registers a handler on a publisher's event, the publisher's delegate holds a reference to the subscriber. If the publisher outlives the subscriber and the subscriber never unsubscribes, the GC can't collect the subscriber — a leak that grows over time (common in UI and long-lived services). Prevent it by unsubscribing (-=) when done, typically in Dispose; using the same handler reference you subscribed with; or using weak-event patterns (WeakEventManager) for cross-lifetime subscriptions. Lambdas make this trickier since you must retain the exact delegate instance to detach it.

Q16 One subscriber throws when an event is raised. What happens to the others, and how do you make subscribers independent? Senior

By default, invoking a multicast delegate calls handlers sequentially, and an exception in one stops the chain — later subscribers never run, and the exception propagates to the publisher. To isolate subscribers, iterate myEvent.GetInvocationList(), cast each entry to the delegate type, and invoke it inside its own try/catch, optionally aggregating failures into an AggregateException. That way one misbehaving handler can't block the rest.

Q17 Why are events usually thread-unsafe to raise naively, and what's the safe idiom? Senior

An explicit if (MyEvent != null) MyEvent(...) has a race: another thread could unsubscribe the last handler between the check and the call, making the delegate null and throwing. The safe idiom is the null-conditional invoke MyEvent?.Invoke(this, e), which the compiler implements by reading the delegate into a local once and invoking that snapshot — so a concurrent unsubscribe can't null it out mid-call. (Subscribers may still receive an event slightly after unsubscribing, which is inherent.)

Q18 How do delegates relate to expression trees and LINQ providers? Senior

A lambda can be compiled into a delegate (executable code) or captured as an expression tree (data describing the code), depending on the target type. LINQ to Objects (IEnumerable) takes Func delegates and runs them in memory; LINQ providers like EF (IQueryable) take Expression<Func<…>> and translate the tree into SQL. So the same lambda syntax becomes either a runnable delegate or analyzable data — the bridge between this topic and the deferred-execution topic.

Q19 Compare the event/delegate approach with interface-based callbacks (e.g. an observer interface). Senior

Both implement observer-style notification. Events/delegates are lightweight, allow ad-hoc lambda subscribers, support multiple handlers per event, and decouple publisher from subscriber types — ideal for "fire and forget" notifications. Interface callbacks (subscriber implements IObserver) are better when a subscriber must implement several related notifications as a cohesive contract, when you want to enforce that implementers handle everything, or when you need richer interaction than a single method. Events scale to many small subscribers; interfaces suit structured, multi-method contracts.

Q20 What does it mean that delegates support covariance and contravariance? Senior

Delegate assignment allows related types: covariance lets a method that returns a more derived type be assigned to a delegate that returns a base type (return types vary "outward"); contravariance lets a method with less derived (base) parameter types be assigned to a delegate with more derived parameters (parameters vary "inward"). For example, a delegate Func<object> can accept a method returning string (covariant return), and a handler taking object sender can subscribe to an event expecting a more derived sender (contravariant parameter). This makes delegate assignment more flexible while staying type-safe.

Q21 Underneath, what type is a delegate, and what's the difference between Delegate and MulticastDelegate? Senior

Every delegate type you declare implicitly derives from System.MulticastDelegate, which derives from System.Delegate. A delegate instance stores a target object (for instance methods) and a method pointer; a multicast delegate additionally maintains an invocation list of these. +=/-= create new combined delegate instances (delegates are immutable) via Delegate.Combine/Remove. You can't explicitly derive from these classes in C#; the compiler does it for you when you declare a delegate type.
