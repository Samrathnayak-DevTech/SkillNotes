Async/await & TPL ⏳

When code waits on something slow — a network call, a file, a database — you don't want the whole program frozen. async/await lets a method pause at a slow point and free the thread to do other work, resuming later when the result is ready. The Task Parallel Library (TPL) adds the building blocks: Task, Task.Run, WhenAll, and friends. The single biggest gotcha — the .Result deadlock — gets its own section. This builds on the async-exception ideas from the previous topic. No prior coding background assumed.

⏳ async / await 📦 Task / Task<T> 🧵 not threads! 🔀 WhenAll / Parallel 💀 .Result deadlock 🧠 Interview Prep
01

Overview

async/await is about not blocking while you wait. TPL is the toolkit of tasks underneath it.

Most slow operations are waiting, not computing — waiting for a server to respond, a disk to read, a database to answer. Blocking a thread during that wait is wasteful. async/await lets the method release its thread while waiting and pick up where it left off when the result arrives, keeping apps responsive and servers scalable. The TPL provides the Task type and combinators that make this work.

🔑 Concurrency vs parallelism

Concurrency = dealing with many things at once by interleaving (one cook juggling several dishes, stirring one while another simmers). Parallelism = doing many things at literally the same time on multiple cores (several cooks). async/await is primarily about concurrency (efficient waiting); the TPL's Parallel/Task.Run add parallelism (using more cores). Different problems, different tools.

02

What async/await Is

A way to write non-blocking code that reads like normal sequential code. And — crucially — it does not create threads.

Mark a method async and you can await inside it. await says: "if this operation isn't done yet, pause here and give the thread back; resume this method when the result is ready." The code reads top-to-bottom like synchronous code, but it doesn't block.

C#
async Task<string> GetPageAsync(string url)
{
    using var client = new HttpClient();
    // await: pause here without blocking the thread; resume when the download finishes
    string html = await client.GetStringAsync(url);
    return html.Length + " chars";   // runs after the await completes
}
synchronous
▓▓▓ work ▓▓▓ BLOCKED waiting on network ▓▓▓ work ▓▓▓
async/await
▓ work ▓ ← thread freed for others → ▓ resume ▓
⚠️ async/await does NOT create a thread

This is the most misunderstood point. await doesn't spin up a background thread — the operation (e.g. a network read) runs without one, and the method simply uses no thread while waiting. The thread is returned to do other work and the method resumes later. async is about not occupying a thread during a wait, not about adding threads. (To run CPU work on another thread, you use Task.Run — section 06.)

🍳 Analogy

You put bread in the toaster (start the await) and, instead of standing there staring at it (blocking), you go chop vegetables (the thread does other work). When the toast pops (the result is ready) you come back and butter it (resume after the await). One cook, no extra staff — just no idle standing around.

03

Task & Task<T>

A Task is a promise of future completion. Task<T> is a promise of a future value.

An async method returns a Task (completes, no value) or Task<T> (completes with a value of type T). The task represents the ongoing operation; await-ing it gives you the result (or rethrows its exception) when it's done.

C#
// returns a value -> Task<T>
async Task<int> CountAsync() { await Task.Delay(100); return 42; }

// returns nothing -> Task (still awaitable)
async Task SaveAsync() { await Task.Delay(100); }

// awaiting unwraps the value (or rethrows the exception)
int n = await CountAsync();    // n == 42
await SaveAsync();              // just wait for completion
📦

Task

An operation that will complete (or fault) in the future, with no result value. Like void, but awaitable.

🎁

Task<T>

An operation that will produce a value of type T. Awaiting it gives you the T.

🧵

Task ≠ Thread

A task is a unit of work, not a thread. Many tasks can run on few threads; an awaiting task uses none.

🔑 Naming & a note on ValueTask

By convention, async methods end in Async (GetPageAsync). For hot paths that often complete synchronously, ValueTask<T> (a struct) can avoid the heap allocation that Task<T> (a class) incurs — but it has usage caveats (don't await twice). Stick with Task/Task<T> unless profiling proves ValueTask is worth it.

04

The State Machine

How "pause and resume" actually works: the compiler rewrites your async method into a state machine.

You write linear code, but the C# compiler transforms an async method into a hidden state machine — a structure that remembers where it paused. At each await, if the operation isn't already done, the method returns to its caller (handing back a Task), and registers a continuation to resume at that exact point when the awaited task completes.

▶️Caller invokes the async methodit runs synchronously until the first await
⏸️Hit an await that isn't finishedthe method returns a Task to the caller and frees the thread
⏳Awaited operation completesa continuation is scheduled to resume the method (MoveNext)
▶️Method resumes after the awaitstate is restored; it runs to the next await or to completion
🔑 What the compiler generates

Behind the scenes the compiler creates a type implementing IAsyncStateMachine with a MoveNext() method and a state field tracking the current await point — plus an AsyncTaskMethodBuilder that produces the returned Task. Local variables become fields so they survive across the pause. It's the same "pause/resume" idea as the yield iterators from the collections topic — async just adds waiting for tasks.

⚠️ An async method with no await

If you mark a method async but never await, the compiler warns you: it runs entirely synchronously and the state-machine overhead buys nothing. Either add an await or drop async and return a completed task (Task.FromResult(...)).

05

Context & ConfigureAwait

After an await, where does the rest of your method run? That's the synchronization context — and ConfigureAwait(false) changes it.

By default, await captures the current synchronization context and resumes the continuation on it. In a UI app (WPF/WinForms) that's the UI thread — so you can safely touch controls after an await. In legacy ASP.NET it's the request context. The capture is what makes "update the UI after the await" just work.

C#
// default: resume on the captured context (e.g. UI thread)
await FetchAsync();              // same as .ConfigureAwait(true)

// library/background code: don't capture -> resume on a thread-pool thread
await FetchAsync().ConfigureAwait(false);   // faster, avoids deadlocks
🔑 When to use ConfigureAwait(false)

Use it in library code and any code that doesn't need to return to the original context. It skips the (sometimes costly) marshaling back and — importantly — helps prevent the deadlock in section 09. Microsoft's CA2007 nudges you to call ConfigureAwait explicitly in libraries. Don't use it right before touching UI controls — there you want the UI context.

⚠️ ASP.NET Core has no synchronization context

Modern ASP.NET Core doesn't use a SynchronizationContext, so ConfigureAwait(false) is a no-op there — it neither helps nor hurts, and the classic .Result deadlock doesn't occur for that reason (though blocking can still cause thread-pool starvation). The context-capture behavior matters most in UI apps and legacy ASP.NET.

06

CPU-bound vs I/O-bound

The decision that determines whether you need Task.Run at all. Get this right and most async confusion disappears.

🌐

I/O-bound

Waiting on something external — network, disk, database. Just await the async API (GetStringAsync, ReadAsync). No thread is used during the wait. Do not wrap it in Task.Run.

🧮

CPU-bound

Heavy computation that actually keeps a core busy — image processing, big calculations. Offload with Task.Run so it runs on a thread-pool thread and doesn't freeze the caller (e.g. the UI).

C#
// I/O-bound: await the async API directly -- no Task.Run!
string data = await httpClient.GetStringAsync(url);

// CPU-bound: offload heavy work to the thread pool so the UI stays responsive
int result = await Task.Run(() => CrunchNumbers(hugeArray));
⚠️ Don't wrap I/O in Task.Run

A common mistake: await Task.Run(() => httpClient.GetStringAsync(url)). This pointlessly burns a thread-pool thread to start an operation that needed no thread at all. For I/O, await the async API directly. Task.Run is for CPU-bound work you want off the current thread.

✅ Use the async API, not the blocking one

Inside async code, prefer the non-blocking versions of everything: await Task.Delay(...) not Thread.Sleep(...); await stream.ReadAsync(...) not stream.Read(...); await Task.WhenAll(...) not Task.WaitAll(...). The blocking versions tie up a thread for nothing.

07

WhenAll / WhenAny / Parallel

The TPL combinators for running multiple operations together instead of one after another.

Task.WhenAll — run many, wait for all 🔗

Start several async operations, then await Task.WhenAll to wait for all of them concurrently — far faster than awaiting each in sequence.

C#
// ❌ sequential: total time = sum of all three
var a = await FetchAsync(url1);
var b = await FetchAsync(url2);
var c = await FetchAsync(url3);

// ✅ concurrent: total time ≈ the slowest one
Task<string>[] tasks = { FetchAsync(url1), FetchAsync(url2), FetchAsync(url3) };
string[] all = await Task.WhenAll(tasks);

Task.WhenAny — react to the first to finish 🏁

C#
// e.g. timeout: whichever finishes first wins
Task done = await Task.WhenAny(FetchAsync(url), Task.Delay(3000));
if (done is Task<string> fetch) Use(await fetch);
else Console.WriteLine("Timed out");

Parallel.For / ForEach — CPU work across cores 🧮

For CPU-bound data parallelism (not I/O), Parallel.For/ForEach spreads iterations across cores.

C#
Parallel.ForEach(images, img => Resize(img));   // uses multiple cores
⚠️ WhenAll vs WaitAll

Use await Task.WhenAll(...) (non-blocking) inside async code, not Task.WaitAll(...) (blocks the thread). Same for WhenAny vs WaitAny. The When… versions return an awaitable task; the Wait… versions block.

🔑 WhenAll and exceptions

If several tasks in a WhenAll fault, awaiting it throws the first exception, but all of them are captured. To inspect every failure, check the returned task's Exception (an AggregateException) or await inside a try/catch and examine task.Exception.InnerExceptions.

08

CancellationToken

The standard way to ask an async operation to stop early — cooperatively and cleanly.

Long-running async work should accept a CancellationToken. The caller can signal cancellation; the operation observes the token and stops gracefully, throwing OperationCanceledException. It's cooperative — the work checks the token; nothing is forcibly killed.

C#
async Task DownloadAsync(string url, CancellationToken token)
{
    for (int i = 0; i < 10; i++)
    {
        token.ThrowIfCancellationRequested();        // bail out if asked
        await Task.Delay(500, token);                 // pass the token to async calls too
    }
}

// caller controls cancellation
using var cts = new CancellationTokenSource();
cts.CancelAfter(2000);                         // auto-cancel after 2s
try { await DownloadAsync(url, cts.Token); }
catch (OperationCanceledException) { Console.WriteLine("Cancelled"); }
✅ Catch OperationCanceledException, not TaskCanceledException

Cancellation surfaces as OperationCanceledException. TaskCanceledException derives from it, so catching the base OperationCanceledException handles both — the recommended approach. Flow the same token through every nested async call so cancellation propagates all the way down.

09

The Deadlock ★

The single most-asked async interview gotcha: blocking on an async call with .Result or .Wait().

In an app with a synchronization context (UI / legacy ASP.NET), calling .Result or .Wait() on a task blocks the context thread. But the awaited method's continuation is trying to resume on that same thread (because it captured the context). The thread is blocked waiting for the task; the task can't finish because it needs the blocked thread. Neither can proceed — deadlock.

UI threadblocked on .Result, waiting for the task to finish
⇄waits for
Continuationcan't run — it needs the UI thread, which is blocked

💀 each waits for the other → frozen forever

C#
// ❌ DEADLOCKS in WPF/WinForms/legacy ASP.NET
void Button_Click(...)            // runs on the UI thread
{
    var data = GetDataAsync().Result;   // blocks UI thread; continuation needs it -> 💀
    label.Text = data;
}

// ✅ FIX 1 (best): async all the way -- never block
async void Button_Click(...) { label.Text = await GetDataAsync(); }

// ✅ FIX 2: in the library, don't capture the context
async Task<string> GetDataAsync()
    => await FetchAsync().ConfigureAwait(false);   // continuation won't need the UI thread
✅ The fixes, in order of preference

1. Async all the way — never block on async; await up the whole call stack. 2. ConfigureAwait(false) in library code so the continuation doesn't need the captured context. 3. If you absolutely must block (e.g. a constructor or legacy boundary), prefer GetAwaiter().GetResult() over .Result — it rethrows the original exception instead of wrapping it in an AggregateException — or offload with Task.Run(async () => await X()).GetAwaiter().GetResult().

⚠️ ASP.NET Core is different

ASP.NET Core has no synchronization context, so this exact deadlock doesn't happen there. But blocking on async (sync-over-async) still ties up thread-pool threads and, under load, causes thread-pool starvation — a different failure with the same root cause. The rule "don't block on async" holds everywhere.

10

More Pitfalls

The other traps that bite async newcomers and veterans alike.

🕳️

async void

Use it only for event handlers (which can't return Task). Elsewhere it's dangerous: the caller can't await it, and exceptions can't be caught — they crash the process.

🔒

sync-over-async

Blocking on async with .Result/.Wait() — deadlocks in UI/legacy ASP.NET, starves the thread pool in ASP.NET Core. Await instead.

👻

fire-and-forget

Calling an async method without awaiting it loses exceptions silently. If you must, observe the result (log faults) rather than ignoring the task.

🧊

blocking calls in async

Thread.Sleep, stream.Read, Task.WaitAll inside async tie up a thread. Use the await-able versions.

C#
// ❌ async void -- exception here crashes the app; no caller can catch it
async void ProcessAsync() { await DoAsync(); }

// ✅ return Task so callers can await AND catch exceptions
async Task ProcessAsync() { await DoAsync(); }

// ❌ fire-and-forget: if DoAsync throws, the exception vanishes
DoAsync();

// ✅ at least observe failures
_ = DoAsync().ContinueWith(t => Log(t.Exception),
        TaskContinuationOptions.OnlyOnFaulted);
⚠️ Why async void is so dangerous

An async void method returns nothing to await, so the caller can't know when it finishes or whether it failed. An unhandled exception in an async void isn't caught by a surrounding try/catch at the call site — it's posted to the synchronization context and typically crashes the process. Reserve async void strictly for event handlers; everywhere else return Task.

🔑 async streams (bonus)

Since C# 8, IAsyncEnumerable<T> lets you stream results asynchronously with await foreach — combining the lazy yield idea with async waiting. Great for paging through a remote data source one item at a time without buffering it all.

11

Worked Example

A method that fetches several resources concurrently, supports cancellation, and follows the rules — all together.

C#
// returns Task<T> (not void), takes a CancellationToken, ends in "Async"
public async Task<int> TotalSizeAsync(
    IEnumerable<string> urls, CancellationToken token = default)
{
    using var client = new HttpClient();

    // start ALL downloads concurrently (I/O-bound -> no Task.Run)
    IEnumerable<Task<string>> downloads =
        urls.Select(u => client.GetStringAsync(u, token));   // LINQ + tasks

    // wait for all of them at once -- time ≈ the slowest, not the sum
    string[] pages = await Task.WhenAll(downloads).ConfigureAwait(false);

    // CPU-bound aggregation is trivial here, so just sum directly
    return pages.Sum(p => p.Length);
}

// caller -- async all the way, with a timeout via cancellation
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(10));
try
{
    int total = await TotalSizeAsync(urls, cts.Token);
    Console.WriteLine($"Total: {total} chars");
}
catch (OperationCanceledException) { Console.WriteLine("Timed out"); }
✅ Every rule applied

Returns Task<int> (never async void); ends in Async; awaits the I/O API directly (no Task.Run for I/O); runs the downloads concurrently with Task.WhenAll instead of one-by-one; uses ConfigureAwait(false) in the reusable method; threads a CancellationToken through for a timeout; and the caller is async all the way with no blocking. That's idiomatic, deadlock-free async.

13

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What do async and await do? Junior

async marks a method as asynchronous so it can use await. await pauses the method at a slow operation without blocking the thread and resumes it when the result is ready. The code reads like normal sequential code but doesn't freeze the program while waiting.

Q2 What is a Task? Junior

A Task represents an operation that will complete in the future. Task means "will finish (no value)"; Task<T> means "will produce a value of type T." You await a task to get its result (or have its exception rethrown) once it's done.

Q3 Does async/await create new threads? Junior

No — this is the key misconception. await doesn't start a thread; while waiting on something like a network call, the method uses no thread at all, and the thread is freed to do other work. async is about not occupying a thread during a wait. To run CPU work on another thread, you use Task.Run — that's separate.

Q4 Why use async at all? Junior

Responsiveness and scalability. In a UI app, async keeps the interface from freezing while waiting on slow work. On a server, it frees threads during waits so the same hardware can handle many more concurrent requests. Both come from not blocking a thread while waiting.

Q5 What return types can an async method have? Junior

Normally Task (no result) or Task<T> (a result of type T). async void is allowed but reserved for event handlers only. There's also ValueTask/ValueTask<T> for performance-sensitive cases. By convention, async method names end in Async.

Q6 What's the difference between Task.WhenAll awaited and awaiting tasks one by one? Junior

Awaiting one by one runs them sequentially — total time is the sum. Starting them all and then await Task.WhenAll(...) runs them concurrently — total time is about the slowest single one. For independent operations, WhenAll is much faster.

🟠 Mid-level — going deeper (2–5 years)

Q7 Why does calling .Result or .Wait() cause a deadlock? Mid

In a context with thread affinity (UI app, legacy ASP.NET), .Result/.Wait() blocks the context thread. Meanwhile the awaited method captured that context and its continuation needs to resume on that same thread. The thread is blocked waiting for the task; the task can't finish because it needs the blocked thread. Each waits on the other forever — deadlock. The fix is to go async all the way and never block.

Q8 What does ConfigureAwait(false) do and when should you use it? Mid

By default await captures the current synchronization context and resumes on it. ConfigureAwait(false) tells it not to — the continuation runs on a thread-pool thread instead. Use it in library code and anywhere you don't need the original context: it's faster and prevents the .Result deadlock. Don't use it right before updating UI controls, where you need to be back on the UI thread.

Q9 Why is async void dangerous, and when is it acceptable? Mid

An async void method returns nothing to await, so callers can't know when it finishes or whether it failed, and an unhandled exception isn't caught by a try/catch at the call site — it's posted to the context and usually crashes the process. It's acceptable only for event handlers, which must return void and can't return Task. Everywhere else, return Task so callers can await and catch exceptions.

Q10 When should you use Task.Run? Mid

For CPU-bound work you want off the current thread — heavy computation that would otherwise freeze the UI. Do not use it for I/O-bound work; awaiting an async I/O API (like GetStringAsync) already uses no thread while waiting, so wrapping it in Task.Run just wastes a thread-pool thread. Rule of thumb: I/O → await directly; CPU → Task.Run.

Q11 How does cancellation work in async code? Mid

It's cooperative via CancellationToken. The caller creates a CancellationTokenSource, passes its Token to the async method, and can cancel it (manually or with CancelAfter). The method observes the token — calling ThrowIfCancellationRequested() and passing it to inner async calls — and stops gracefully by throwing OperationCanceledException. Nothing is forcibly killed; the work checks and cooperates.

Q12 What's the difference between concurrency and parallelism? Mid

Concurrency is dealing with many tasks by interleaving progress (one thread juggling several waits). Parallelism is executing many tasks literally simultaneously on multiple cores. async/await mainly enables concurrency (efficient waiting on I/O); the TPL's Parallel.For and Task.Run enable parallelism (using more cores for CPU work). They solve different problems.

Q13 Why prefer GetAwaiter().GetResult() over .Result when you must block? Mid

Both block, but .Result and .Wait() wrap any exception in an AggregateException, whereas GetAwaiter().GetResult() rethrows the original exception directly — cleaner to catch and read. (Blocking still carries deadlock risk; the real fix is to go async all the way. GetAwaiter().GetResult() is the least-bad option when blocking is unavoidable.)

🔴 Senior — synthesis & judgment (5–7 years)

Q14 How does the compiler implement async/await? Senior

It rewrites the method into a state machine implementing IAsyncStateMachine, with a MoveNext() method and a state field tracking the current await point. Local variables become fields so they survive across pauses. At each await, it gets the awaiter; if the operation is already complete it continues synchronously, otherwise it stores the state, hooks a continuation (via AwaitOnCompleted) to call MoveNext when the awaited task finishes, and returns the Task built by an AsyncTaskMethodBuilder to the caller. It's the same pause/resume mechanism as yield iterators, generalized to awaiting tasks.

Q15 Why doesn't the .Result deadlock happen in ASP.NET Core, and what problem replaces it? Senior

ASP.NET Core has no SynchronizationContext, so continuations resume on any thread-pool thread rather than a specific captured one — there's no single thread to be blocked, so the classic deadlock can't form (and ConfigureAwait(false) is a no-op there). But blocking on async (sync-over-async) still consumes a thread-pool thread for the duration of the wait; under load this causes thread-pool starvation: requests queue waiting for threads, latency spikes, and throughput collapses. Same root cause (don't block on async), different symptom.

Q16 How are exceptions handled across await, and in task-returning methods? Senior

An exception thrown in an async method is captured and placed on the returned Task; awaiting that task rethrows it with the stack trace preserved (the framework uses ExceptionDispatchInfo internally). Exceptions thrown in the synchronous part before the first await still go onto the task, not the caller — so validate arguments and throw before the first await if you want them surfaced immediately. Blocking via .Result/.Wait() wraps exceptions in AggregateException; awaiting unwraps to the first one. With WhenAll, awaiting throws the first exception but all are retained on the task.

Q17 When is ValueTask appropriate, and what are its caveats? Senior

Task/Task<T> are reference types, so returning them allocates on the heap — negligible normally, but costly on hot paths that complete synchronously most of the time (e.g. a cache hit). ValueTask<T> is a struct that avoids the allocation in the synchronous-completion case. Caveats: a ValueTask must be awaited only once, not stored/awaited multiple times or blocked on with .Result; misuse causes subtle bugs. Use it only where profiling shows allocation pressure; default to Task for clarity and safety.

Q18 What is thread-pool starvation and how does sync-over-async cause it? Senior

The thread pool has a limited number of worker threads (it grows slowly). Sync-over-async — blocking a pool thread with .Result/.Wait() while an async operation completes — keeps that thread idle-but-occupied. Under load, many such blocks consume all pool threads; new work (including the continuations needed to unblock them) can't get a thread, so the pool slowly adds threads while latency balloons and throughput tanks, sometimes spiraling. The fix is to never block on async on pool threads — await instead, so threads are returned during waits.

Q19 How do you correctly handle fire-and-forget work? Senior

Truly fire-and-forget is risky: if the unawaited task faults, the exception goes unobserved and is lost (and historically could crash the process). If you must not await, observe the task: attach a continuation that logs faults (ContinueWith(t => Log(t.Exception), OnlyOnFaulted)), or run it through a dedicated background-task mechanism (e.g. a hosted service / channel in ASP.NET Core) that supervises failures and lifetime. Never let an async method's exceptions silently disappear, and avoid async void for this — use async Task and observe it.

Q20 Task.Run for I/O is wrong — but why is it actively harmful on a server? Senior

Wrapping an async I/O call in Task.Run moves the kickoff to a thread-pool thread that then immediately awaits — so you've consumed a pool thread to do nothing but wait, the very thing async I/O is designed to avoid. On a server handling many requests, this doubles thread-pool pressure and reduces the concurrency the async model was meant to provide, hurting scalability. (On a desktop UI it's merely wasteful.) For I/O, await the async API directly; reserve Task.Run for genuinely CPU-bound work.
