// .NET  ·  how your C# actually runs

The C# Runtime, demystified 🚀

You write friendly C#. Somewhere between your keyboard and the CPU, a small army of machinery turns it into something the processor understands — and quietly cleans up the memory mess behind you. This page walks through that machinery piece by piece: the CLR, the IL it runs, how JIT and AOT compile your code, and how the garbage collector manages memory in generations. No prior coding background assumed.

⚙️ CLR 📜 IL / MSIL ⚡ JIT 🏭 AOT 🧹 GC Generations 🧠 Interview Prep
01

The Big Picture

Before any term, get the journey straight: what really happens between "I wrote some C#" and "my program is running."

Most beginners imagine the computer reads C# directly. It does not. Your CPU only understands raw machine code (the 1s and 0s for that exact chip). C# takes a two-step trip to get there.

The journey of your code 🧭

Step 1 — Compile (at build time): The C# compiler (called Roslyn) does not produce machine code. It produces a portable, CPU-neutral instruction set called Intermediate Language (IL), packaged with metadata into a .dll or .exe called an assembly.

Step 2 — Execute (at run time): When you launch the program, the CLR (the runtime) loads the assembly and uses the JIT compiler to translate IL into the exact machine code for your processor, right before each method runs. The garbage collector manages memory the whole time.

🍳 Analogy

Think of C# as a recipe written in plain English. IL is that same recipe translated into a universal cooking notation any kitchen understands. The CLR is the head chef in your specific kitchen who reads the notation and, dish by dish (the JIT), turns it into precise actions for the equipment they actually have. The GC is the kitchen porter quietly clearing dirty plates so you never run out.

🔑 Key idea

This whole stack is why .NET is called a managed platform: a runtime manages the execution of your code instead of letting it talk to the hardware directly. That management is what gives you safety, portability, and automatic memory cleanup.

02

The CLR

Common Language Runtime — the engine that actually runs your program.

The CLR (Common Language Runtime) is the heart of .NET. It is the virtual environment that loads your compiled code and runs it. Languages like C#, F#, and VB.NET all compile to the same IL, so they all run on the same CLR — that is the "Common" in the name.

What the CLR does for you 🛠️

📂

Assembly loading

Finds and loads the .dll/.exe files your program needs, reading their IL and metadata.

⚡

JIT compilation

Calls the Just-In-Time compiler to turn IL into native machine code as methods are first used.

🧹

Memory management

Runs the garbage collector so you almost never allocate or free memory by hand.

🛡️

Type safety

Verifies that code only uses memory and types in valid ways, preventing whole classes of bugs.

💥

Exception handling

Provides a uniform way to throw and catch errors across all .NET languages.

🧵

Threads & security

Coordinates threads and enforces security boundaries while your code executes.

🔑 CLR vs CoreCLR vs .NET

Historically the runtime in the old Windows-only .NET Framework was "the CLR." Modern, cross-platform .NET (.NET Core, .NET 5/6/7/8/9/10) uses an evolved, open-source runtime often called CoreCLR. People still say "the CLR" for both — it means the same role: the execution engine.

CTS & CLS — the shared rulebook 📏

Two specifications let multiple languages cooperate on one runtime:

  • CTS (Common Type System): defines all the types (int, string, classes, structs…) and how they behave, so a string means the same thing in C# and F#.
  • CLS (Common Language Specification): a subset of rules a library should follow so it can be consumed from any .NET language without surprises.
✅ Beginner takeaway

You rarely "use" the CLR directly — it works invisibly. But almost every advanced behavior (why memory is cleaned automatically, why a crash gives a stack trace, why .NET runs on Windows, Linux, and macOS) traces back to the CLR doing its job.

03

IL — Intermediate Language

The portable "in-between" code your C# becomes. Also written MSIL or CIL.

Intermediate Language (IL) — also called MSIL (Microsoft IL) or CIL (Common IL) — is what the C# compiler produces. It is a low-level, CPU-independent instruction set. It is not tied to Intel, ARM, or any specific chip, which is exactly why the same compiled .dll can run on different machines.

See it for yourself 👀

Here is a tiny C# method and the IL it roughly compiles to. Notice IL is stack-based — it pushes values onto an evaluation stack and operates on them.

C#
public int Add(int a, int b)
{
    return a + b;
}
IL (simplified)
// method Add(int32 a, int32 b)
ldarg.1      // push argument a onto the stack
ldarg.2      // push argument b onto the stack
add          // pop both, push (a + b)
ret          // return the value on top of the stack
🧱 Analogy

IL is like flat-pack furniture instructions: standardized, picture-based steps that work in any country. The actual assembly (machine code) depends on the tools you have at home — but the instruction sheet is universal.

IL carries metadata too 🏷️

An assembly is not just IL instructions. It also contains metadata: a self-describing catalog of every type, method, property, and parameter. This is what powers:

  • Reflection — code that inspects types at runtime.
  • IntelliSense — your editor knowing what methods exist.
  • Interoperability — different languages understanding each other's libraries.
⚠️ Worth knowing

Because IL + metadata describe your program so completely, compiled .NET assemblies can be decompiled back into readable C# (with tools like ILSpy or dotPeek). If you ship sensitive logic, you may need obfuscation. IL is portable and inspectable by design.

Managed vs Unmanaged

Why "managed code" is a phrase you will hear constantly — and what it costs.

AspectManaged code (C#)Unmanaged code (C / C++)
Runs underThe CLRDirectly on the OS / CPU
Memory cleanupAutomatic (garbage collector)Manual (malloc/free, new/delete)
Type/memory safetyEnforced by the runtimeProgrammer's responsibility
PortabilityHigh (IL is CPU-neutral)Recompile per platform
Raw speed ceilingVery high, slight overheadHighest, full control

Managed code is code the CLR runs and supervises. The runtime manages its memory, checks its type safety, and handles its exceptions. C# is normally managed. Unmanaged code runs outside that supervision — typically C/C++ talking straight to the operating system, where you must free memory yourself.

🔑 The trade

Managed code trades a little raw speed and control for a lot of safety and productivity. You stop worrying about memory leaks and dangling pointers; the runtime handles them. For 95% of applications, that trade is overwhelmingly worth it.

When C# touches the unmanaged world 🌉

Sometimes you must call native libraries (an OS API, a hardware driver, a C++ DLL). The runtime cannot garbage-collect those resources, so C# gives you tools to clean up manually: IDisposable, the using statement, and finalizers.

C#
// 'using' guarantees Dispose() is called even if an error occurs.
// This is how you safely release unmanaged resources like files.
using (var file = new StreamReader("data.txt"))
{
    string text = file.ReadToEnd();
    Console.WriteLine(text);
}   // file handle released right here, automatically
✅ Rule of thumb

If a type implements IDisposable (files, network connections, database connections), wrap it in a using. The GC handles managed memory, but Dispose() handles the unmanaged resources the GC can't see.

05

JIT Compilation

Just-In-Time: the CLR translates IL into machine code while your program runs.

The JIT (Just-In-Time) compiler is the part of the CLR that converts IL into native machine code — but only when a method is actually about to run for the first time. It compiles "just in time," method by method, not all upfront.

How a method gets JIT-compiled ⚡

▶️Method called for the first timee.g. your code calls CalculateTotal()
🔎CLR checks: already compiled?Is native code for this method cached yet?
⚙️JIT compiles IL → machine codeOptimized for the exact CPU it's running on
💾Cache the native codeStored in memory for the rest of the program's life
🔁Next calls reuse the cacheNo re-compilation — it runs at full native speed
🔑 The big win

Because JIT compiles on the actual machine at runtime, it can use that exact CPU's features (special instruction sets, register layout) and even real usage data to produce highly optimized code. A single portable IL file gets tailor-made machine code on every device.

🗣️ Analogy

JIT is like a live interpreter at a conference who translates each sentence the moment it's first spoken — then remembers the translation, so if the same phrase comes up again it's instant. The very first time costs a beat; everything after is fluent.

The downside: warm-up cost 🐢

The first time each method runs, there's a tiny pause while the JIT works. For a long-running web server this is invisible — methods compile once and then fly. But for short-lived programs (a tiny command-line tool, a cloud function that starts, runs once, and stops), that startup cost matters. This is exactly the problem AOT solves (section 07).

later callsfull native speed

06

Tiered Compilation & PGO

A clever upgrade to JIT: compile fast first, then re-compile the hot paths better.

Modern .NET doesn't just JIT a method once. It uses tiered compilation — two levels of quality — to get the best of both worlds: fast startup and fast steady-state performance.

🥉

Tier 0 — Quick JIT

The first time a method runs, the JIT compiles it fast with minimal optimization. The app starts quickly.

🥇

Tier 1 — Optimized

If a method is called many times (it's "hot"), a background thread re-compiles it with full optimizations and swaps it in.

The runtime counts how often each method runs. Cold methods stay at cheap Tier 0; hot methods get promoted to high-quality Tier 1 — without you doing anything.

🔑 OSR — On-Stack Replacement

What about a method with one giant loop that runs for ages? Waiting for the next call to upgrade it would be too late. OSR lets the runtime swap a long-running loop from Tier 0 to optimized code while it is still executing. Clever.

Dynamic PGO — learning from real usage 📈

Profile-Guided Optimization (PGO) means the runtime watches how your code actually behaves (which branches are taken, which types appear most) during Tier 0, then uses that real profile data to produce smarter Tier 1 code. Dynamic PGO is on by default in modern .NET (.NET 8 and later), and often gives a free throughput boost.

🏃 Analogy

Tier 0 is jogging to warm up; Tier 1 is sprinting once your muscles are ready. PGO is a coach watching your first laps and rewriting your training plan around exactly how you run.

07

AOT Compilation

Ahead-of-Time: do the IL → machine code step before shipping, not at runtime.

AOT (Ahead-Of-Time) compilation flips the timing. Instead of translating IL to machine code while the program runs (JIT), AOT does it earlier — at build/publish time. The program ships as native code that's ready to run instantly. .NET offers two flavors.

Two kinds of AOT in .NET 🏭

⚡

ReadyToRun (R2R)

Precompiles IL to native code but keeps the IL too. Faster startup with no code changes. The JIT can still kick in for hot paths and re-optimize them at Tier 1. Bigger file size.

📦

Native AOT

Compiles everything to a single self-contained native executable with no JIT at all. Tiny footprint, instant startup, runs without .NET installed — at the cost of some dynamic features.

🔑 Native AOT timeline

Native AOT arrived for console apps in .NET 7, gained comprehensive support including ASP.NET Core in .NET 8, and continues to expand in recent releases (.NET 9 and .NET 10). It targets a specific platform (e.g. linux-x64, win-x64) — you build a separate binary per target.

Turning Native AOT on 🔧

It's a single project setting. After this, dotnet publish -r <runtime> produces a native executable.

.csproj
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <!-- this one line enables Native AOT -->
    <PublishAot>true</PublishAot>
  </PropertyGroup>
</Project>
terminal
# build a self-contained native binary for Linux x64
dotnet publish -r linux-x64 -c Release
⚠️ Native AOT trade-offs

Because there's no JIT and unused code is trimmed away, some dynamic features break or are restricted: heavy reflection, Reflection.Emit (generating code at runtime), and loading assemblies on the fly. The build also shows AOT warnings — heed them, because an app that warns at publish time may misbehave. Use Native AOT when startup speed and small size truly matter (microservices, serverless functions, CLI tools, containers).

✅ How to choose

Pick by deployment shape, not raw benchmarks. Long-running service where peak speed matters → keep the default JIT (tiered + PGO converge on the fastest steady-state code). Want faster startup with zero code changes → ReadyToRun. Startup time, memory, or running without a JIT is the dominating constraint, and you don't lean on reflection → Native AOT.

🍞 Analogy

JIT is baking bread fresh as customers order — perfectly suited to whoever walks in, but there's a wait. AOT is pre-baking everything before opening — instant service the moment the door opens, but you committed to the menu in advance and can't adapt on the fly.

09

Garbage Collection

The CLR's automatic memory cleaner — so you (almost) never free memory by hand.

When you create an object with new, it lives on the managed heap (memory the CLR controls). Eventually nothing in your program references that object anymore — it's garbage. The Garbage Collector (GC) finds and reclaims that memory automatically.

What "no longer referenced" means 🔗

The GC starts from roots — things definitely still alive: local variables on the stack, static fields, CPU registers. Any object reachable by following references from a root is live. Anything not reachable is unreachable garbage.

Order and Customer are reachable from a root → kept. OldCart has no path from any root → collected.

The three phases: mark → sweep → compact 🧹

🏷️1. MarkWalk from the roots and flag every reachable (live) object
🗑️2. SweepReclaim the memory of everything not marked
📚3. CompactSlide survivors together to remove gaps, then fix up their references
📚 Analogy

Imagine a bookshelf where you keep grabbing and discarding books. Mark = put a sticky note on every book you're still reading. Sweep = throw out everything without a note. Compact = push the remaining books together so there are no awkward gaps, and update the library index so you can still find them.

⚠️ Don't call GC.Collect()

Beginners often discover GC.Collect() and sprinkle it everywhere to "free memory." Don't. The GC is far smarter than you about when to run; forcing it usually hurts performance by triggering expensive collections at the wrong time. Trust it. Let objects fall out of scope naturally.

🔑 Stop-the-world

To safely move objects, the GC briefly pauses your app's threads ("stop-the-world"). The whole point of the generational design in the next section is to keep these pauses tiny by usually collecting only a small slice of memory.

10

GC Generations

The performance secret: most objects die young, so collect the young area often and cheaply.

If the GC had to scan every object every time, it would be painfully slow. Instead it uses a key observation called the generational hypothesis: most objects die very young (a temporary string, a loop variable), while objects that survive a while tend to keep surviving (a config object, a cache).

So the heap is split into three generations, and the GC mostly collects the youngest one.

🌱

Generation 0

Where brand-new small objects are born. Tiny and collected very frequently — most objects die here, so collections are cheap and fast.

🪴

Generation 1

Objects that survived one Gen 0 collection. Acts as a buffer between short-lived and long-lived objects.

🌳

Generation 2

Long-lived objects (statics, caches, app-wide singletons). Collected rarely because Gen 2 collections are expensive.

Promotion: surviving moves you up ⬆️

Survival is the rule of promotion. An object that lives through a collection is "promoted" to the next generation up. You only ever allocate small objects into Gen 0 yourself — the GC does the promoting.

🌱Born in Gen 0every new SmallObject() starts here
⬆️Survives a Gen 0 GC → promoted to Gen 1
⬆️Survives a Gen 1 GC → promoted to Gen 2and stays in Gen 2 thereafter
🔑 Collections are cumulative

A Gen 1 collection also collects Gen 0. A Gen 2 collection collects everything (Gen 0, 1, 2, plus LOH) — this is the expensive "full GC." That's why the GC tries hard to satisfy most requests with a cheap Gen 0 collection alone.

Gen 0 GCcheapest, frequent
Gen 1 GC+ Gen 0
Gen 2 GCfull · everything · costliest

Special heaps: LOH & POH 📦📌

  • LOH — Large Object Heap: objects ≥ 85,000 bytes (big arrays, large buffers) go here. The LOH is not compacted by default (copying huge blocks is too expensive) and is logically collected as part of Gen 2 — so it's only freed during a full collection.
  • POH — Pinned Object Heap: introduced in .NET 5, a dedicated home for pinned objects (objects fixed in place so the GC can't move them, often for native interop). Isolating them here keeps the rest of the heap freely compactable.
⚠️ LOH fragmentation

Because the LOH isn't compacted, frequently allocating and discarding large objects can leave gaps it can't reuse efficiently — fragmentation. The fix usually isn't forcing compaction; it's pooling and reusing large buffers (e.g. ArrayPool<T>) so you allocate them once.

Seeing generations in code 🔬

C#
var obj = new object();
Console.WriteLine(GC.GetGeneration(obj));   // 0  — brand new

GC.Collect();                                // (demo only — don't do this in real apps)
Console.WriteLine(GC.GetGeneration(obj));   // 1  — it survived, so it was promoted

// A large array lands on the Large Object Heap (reported as Gen 2):
var big = new byte[100_000];           // >= 85,000 bytes -> LOH
Console.WriteLine(GC.GetGeneration(big));   // 2

Workstation vs Server GC ⚙️

One last knob worth knowing: the GC has modes.

  • Workstation GC: default for desktop/client apps; tuned to keep the UI responsive with short pauses.
  • Server GC: uses multiple heaps and threads (one set per CPU core) for maximum throughput on servers with many cores. Common for web apps.
  • Background / concurrent GC: does much of the expensive Gen 2 work on a background thread so your app keeps running with minimal pauses.

12

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is the CLR? Junior

The Common Language Runtime is the engine that runs .NET programs. It loads your compiled code, converts it to machine code (via the JIT), manages memory with the garbage collector, enforces type safety, and handles exceptions. Because C#, F#, and VB.NET all compile to the same intermediate code, they all run on this one common runtime.

Q2 What is IL (MSIL/CIL)? Junior

Intermediate Language is the CPU-independent code the C# compiler produces — not machine code. It's stored, along with metadata describing your types, inside a .dll or .exe assembly. The runtime later turns this IL into machine code for the specific computer. IL is what makes one compiled file able to run on different processors.

Q3 Does the C# compiler produce machine code directly? Junior

No. The C# compiler (Roslyn) produces IL, not machine code. Converting IL to machine code happens later — at runtime by the JIT, or at publish time with AOT. This two-step design is what gives .NET its portability.

Q4 What does "managed code" mean? Junior

Managed code runs under the supervision of the CLR. The runtime manages its memory (garbage collection), checks its type safety, and handles its errors. The opposite is unmanaged code (like C/C++) that runs directly on the OS and where the programmer must free memory manually.

Q5 What is garbage collection in simple terms? Junior

It's the runtime's automatic memory cleaner. When you create objects, they use memory. When nothing in your program refers to an object anymore, the garbage collector detects that it's no longer needed and reclaims its memory — so you don't have to free memory by hand and rarely get memory leaks.

Q6 What is the JIT compiler? Junior

The Just-In-Time compiler converts IL into native machine code while the program runs — specifically the first time each method is called. It then caches the result, so subsequent calls run at full native speed. "Just in time" means it compiles each method right before it's needed, not all at once.

Q7 What is an assembly? Junior

An assembly is the unit of deployment in .NET — typically a .dll or .exe file. It contains your compiled IL code plus metadata (a self-describing catalog of all the types and methods inside). Assemblies are what the CLR loads and runs.

Q8 What are the GC generations and why do they exist? Junior

The heap is split into Gen 0, Gen 1, and Gen 2. The reason is a simple observation: most objects die young. So the GC keeps new objects in Gen 0 and collects it often and cheaply. Objects that survive get promoted upward (Gen 0 → 1 → 2). Long-lived objects end up in Gen 2, which is collected rarely because it's expensive. This makes most collections small and fast.

🟠 Mid-level — going deeper (2–5 years)

Q9 Walk through what happens from C# source to a running program. Mid

(1) You write .cs files. (2) The Roslyn compiler turns them into IL + metadata inside an assembly (.dll/.exe). (3) At launch, the CLR loads the assembly. (4) As each method is first called, the JIT compiles its IL into native machine code for the current CPU and caches it. (5) The garbage collector manages memory throughout. With AOT, step 4 happens earlier — at publish time instead of runtime.

Q10 What's the difference between JIT and AOT? Mid

Both turn IL into machine code; the difference is when. JIT compiles at runtime, method by method — slower startup but it can optimize for the exact CPU and real usage (Tier 1 + PGO), giving excellent steady-state speed. AOT compiles ahead of time at publish — instant startup, smaller footprint, can run without .NET installed, but it's platform-specific and restricts dynamic features like reflection.

Q11 Explain mark, sweep, and compact. Mid

Mark: starting from roots (stack variables, statics, registers), the GC flags every object still reachable. Sweep: it reclaims the memory of everything not marked. Compact: it slides the surviving objects together to eliminate gaps (reducing fragmentation) and updates all references to point to the new locations.

Q12 What is the Large Object Heap (LOH)? Mid

Objects ≥ 85,000 bytes (typically big arrays) are allocated on the LOH instead of the normal small-object heap. The LOH is not compacted by default because copying huge blocks is expensive, and it's collected only during a Gen 2 (full) collection. Frequent large allocations can cause fragmentation, so the usual remedy is to pool and reuse large buffers.

Q13 What does object promotion mean? Mid

When an object survives a garbage collection, the GC promotes it to the next higher generation: Gen 0 survivors become Gen 1, Gen 1 survivors become Gen 2. Objects already in Gen 2 stay in Gen 2. You only allocate small objects into Gen 0 directly; the GC does all the promoting based on survival.

Q14 Should you call GC.Collect() to improve performance? Mid

Almost never. The GC self-tunes and knows far better than you when to collect. Manually calling GC.Collect() usually forces an expensive full collection at a bad time, hurting performance. Better practice is to reduce allocations, let objects go out of scope naturally, and dispose unmanaged resources with using. Manual collection is only for rare, well-measured scenarios.

Q15 What is IDisposable and why is it needed if the GC exists? Mid

The GC handles managed memory, but it doesn't know about unmanaged resources like file handles, database connections, or network sockets. IDisposable gives those types a Dispose() method to release such resources promptly and deterministically. The using statement calls Dispose() automatically — even if an exception is thrown.

Q16 Why can .NET assemblies be decompiled, and what can you do about it? Mid

Because assemblies contain IL plus rich metadata describing every type and method, tools like ILSpy can reconstruct readable C#. That self-describing nature is what powers reflection and IntelliSense, but it also exposes your logic. If you need to protect intellectual property, you can apply obfuscation (and for some cases, Native AOT, which compiles to native code without shipping IL, raises the bar).

🔴 Senior — synthesis & judgment (5–7 years)

Q17 Explain tiered compilation and how it balances startup vs throughput. Senior

The JIT compiles a method twice. Tier 0 is a fast, low-optimization compile so the app starts quickly. The runtime counts method calls; once a method is "hot," a background thread recompiles it at Tier 1 with full optimizations and swaps it in. OSR (On-Stack Replacement) handles long-running loops by upgrading them mid-execution. This gives both fast startup (Tier 0) and fast steady-state (Tier 1) without developer effort.

Q18 What is Dynamic PGO and why does it matter? Senior

Profile-Guided Optimization instruments Tier 0 code to collect real runtime behavior — which branches are taken, which concrete types show up most — then feeds that profile into Tier 1 so the optimized code is tailored to actual usage (e.g. devirtualizing hot virtual calls). Dynamic PGO is on by default in .NET 8 and later and typically improves throughput for free, which is a strong reason to keep the default JIT for long-running services.

Q19 When would you choose Native AOT, and what do you give up? Senior

Choose Native AOT when startup time, memory footprint, or running without a JIT dominates — serverless functions, CLI tools, sidecars, scale-to-zero containers, locked-down environments. You give up runtime re-optimization (no Tier 1/PGO), broad reflection and Reflection.Emit, runtime assembly loading, and cross-platform portability of a single binary (it's platform-specific). You must also resolve AOT trim warnings, since unhandled ones can cause runtime failures. For long-running, throughput-critical servers, the default JIT usually wins.

Q20 Why are Gen 2 (full) collections expensive, and how do you minimize them? Senior

A Gen 2 collection is a full collection — it scans and collects Gen 0, 1, 2, and the LOH, and may compact large amounts of memory, causing longer stop-the-world pauses. To minimize them: reduce mid-life and long-lived allocations, avoid promoting short-lived objects (don't hold references longer than needed), pool large buffers to keep them off a churning LOH, prefer struct/Span<T> for hot paths to cut heap allocations, and consider Server GC + background GC for throughput-sensitive services.

Q21 What problem does the Pinned Object Heap (POH) solve? Senior

Pinned objects are fixed in memory so the GC can't move them — common when passing buffers to native code via interop. Before, pinned objects scattered through the normal heap blocked compaction and caused fragmentation. The POH, introduced in .NET 5, gives pinned objects a dedicated region, so the rest of the heap stays freely compactable and fragmentation from pinning is contained.

Q22 Compare Workstation GC, Server GC, and Background GC. Senior

Workstation GC (client default) uses a single heap and prioritizes low latency for responsive UIs. Server GC creates multiple heaps and GC threads (roughly one set per core) to maximize throughput on multi-core servers — ideal for web apps. Background (concurrent) GC performs most of the expensive Gen 2 work on a background thread so application threads keep running with minimal pauses. You choose modes via runtime configuration based on whether you optimize for latency or throughput.

Q23 A short-lived cloud function has slow cold starts. How does the runtime knowledge help? Senior

Cold-start slowness often comes from JIT warm-up: methods compile on first use, which a short-lived function pays every invocation. Mitigations, in increasing commitment: enable ReadyToRun to precompile to native at publish (faster startup, zero code changes), or move to Native AOT to eliminate the JIT entirely for near-instant startup and a smaller image — provided the function avoids reflection-heavy dependencies. The decision is driven by deployment shape, exactly the JIT-vs-AOT trade-off.

Q24 Explain the difference between the stack and the managed heap. Senior

The stack stores method call frames, local value-type variables, and parameters; it's fast, LIFO, and automatically reclaimed when a method returns — no GC needed. The managed heap stores reference-type objects (created with new); it's where the garbage collector operates with its generations. Value types (int, struct) typically live on the stack or inline; reference types (class) live on the heap, with the variable holding a reference to them. Reducing heap allocations (e.g. via structs and Span<T>) reduces GC pressure.
