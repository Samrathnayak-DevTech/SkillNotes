Dependency Injection & Lifetimes 💉

Instead of a class building the things it depends on with new, dependency injection (DI) has them handed in from outside — which makes code loosely coupled, swappable, and testable. A built-in container creates and supplies those dependencies. The other half of the story is lifetimes: how long each service lives — Transient, Scoped, or Singleton — and the classic bug (the captive dependency) when you mix them wrong. This connects to the IDisposable topic, since the container also disposes your services. No prior coding background assumed.

01

Overview

DI is about who supplies a class's dependencies; lifetimes are about how long those dependencies live.

Dependency injection is a technique where a class receives the objects it needs (its dependencies) from an external source instead of creating them itself. It's the most common form of Inversion of Control (IoC). A container (in .NET, the built-in IServiceProvider) is configured once with how to build each service, then constructs and wires them together on demand — including choosing how long each instance lives (its lifetime).

🔑 The one-line idea

"Don't call us, we'll call you." A class declares what it needs (usually as constructor parameters) and the container provides it. The class no longer knows or cares how its dependencies are built — that decision moves outward, which is exactly what makes the code flexible and testable.

02

The Problem It Solves

Creating your own dependencies with new couples you to a specific implementation and makes testing hard.

When a class news up its own dependencies, it's tightly coupled to those exact classes. You can't swap the implementation (e.g. a real email sender for a fake one in tests) without editing the class. DI inverts this: depend on an abstraction (an interface) and let it be injected.

C#
// ❌ WITHOUT DI: tightly coupled, hard to test
class OrderService
{
    private readonly SmtpEmailSender _email = new();   // locked to SmtpEmailSender
    public void Place() { /* ... */ _email.Send(...); }
    // can't substitute a fake sender in a unit test
}

// ✅ WITH DI: depend on an abstraction, receive it via the constructor
class OrderService
{
    private readonly IEmailSender _email;
    public OrderService(IEmailSender email) => _email = email;   // injected
    public void Place() { /* ... */ _email.Send(...); }
    // tests can pass a FakeEmailSender; prod passes SmtpEmailSender
}
🔗

Loose coupling

Classes depend on interfaces, not concrete types — so implementations can change without touching consumers.

🧪

Testability

Inject a mock/fake in unit tests instead of the real database or network service. This is DI's biggest practical win.

🔄

Flexibility

Swap implementations (e.g. SMTP vs SendGrid) by changing one registration, not every class that uses it.

🔑 IoC & the Dependency Inversion Principle

DI implements the "D" in SOLID — the Dependency Inversion Principle: high-level code should depend on abstractions, not concrete details. "Inversion of control" means the framework/container controls object creation and calls into your code, rather than your code controlling everything. DI is the concrete mechanism that achieves it.

03

Injection & the Container

Register services in the container at startup; it constructs them and injects them through constructors.

.NET has a built-in container. You register services into an IServiceCollection at startup (specifying the lifetime), the host builds an IServiceProvider, and when it creates your classes it resolves and injects each constructor dependency automatically — recursively building the whole graph.

C#
var builder = WebApplication.CreateBuilder(args);

// register: "when something needs IEmailSender, give it SmtpEmailSender"
builder.Services.AddSingleton<IEmailSender, SmtpEmailSender>();
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
builder.Services.AddTransient<OrderService>();   // concrete type is fine too

var app = builder.Build();
// when the framework creates OrderService, it injects IEmailSender + IOrderRepository
// -- you never call 'new OrderService(...)' yourself
🔑 Constructor injection is the default

Declaring dependencies as constructor parameters is the preferred form: it makes a class's requirements explicit and guarantees they're supplied before the object is used. (Method injection and property injection exist but are used far less.) Resolve with GetRequiredService<T>() when you need to ask the container directly — though prefer injection over asking.

🍽️ Analogy

Think of a restaurant kitchen. A chef (your class) doesn't go farm the vegetables and raise the chickens — the supplier (the container) delivers exactly the ingredients (dependencies) the recipe lists. The chef just declares what's needed; sourcing is someone else's job. Change suppliers without changing the recipe.

04

The Three Lifetimes ★

When you register a service you choose its lifetime — how often the container creates a new instance.

🔄

Transient

AddTransient — a new instance every time it's requested. Best for lightweight, stateless services. The safe default when unsure.

📦

Scoped

AddScoped — one instance per scope (per HTTP request in web apps). Shared within a request, fresh for the next. EF Core's DbContext is scoped.

♾️

Singleton

AddSingleton — one instance for the whole app, shared everywhere. For expensive-to-create or globally shared state (cache, config). Must be thread-safe.

Here's how the same three services behave across two requests, when each is resolved twice per request (instance identity shown as a letter):

Transient = all different · Scoped = same within a request, different across requests · Singleton = same everywhere

LifetimeNew instance…Use for

✅ Choosing a lifetime

Ask "how much state does this share, and for how long?" No shared state and cheap → Transient. State that belongs to one request (a DB context, the current user's unit of work) → Scoped. Expensive to build or genuinely global (a cache loaded once) → Singleton. When in doubt, the safest choice is the shortest lifetime that works.

05

Scopes

A "scope" is the boundary that defines a Scoped service's lifetime. In web apps, the framework makes one per request.

A scope is a container-created boundary. Within one scope, a Scoped service resolves to the same instance; a new scope gets a fresh one. In ASP.NET Core, the framework automatically creates a scope per HTTP request, so "scoped" effectively means "per request." Outside a request (console apps, background jobs, app startup), you create scopes explicitly.

C#
// explicit scope -- needed in console apps, background services,
// or when a singleton must use a scoped service (see section 06)
using (IServiceScope scope = app.Services.CreateScope())
{
    var repo = scope.ServiceProvider.GetRequiredService<IOrderRepository>();
    repo.DoWork();
}   // scope disposed here -> scoped services in it are disposed too
🔑 Scoped needs a scope

A scoped service must be resolved from within a scope — either the implicit per-request scope or an explicit one from IServiceScopeFactory.CreateScope(). Resolving a scoped service from the root provider (outside any scope) is a mistake: it behaves like a singleton and never gets disposed properly.

⚠️ Scopes aren't hierarchical

Each scope is independent — there's no parent/child relationship between scopes, and a scoped instance in one scope is unrelated to the "same" service in another. Don't assume nesting semantics; a new scope simply gets its own set of scoped instances.

06

Captive Dependencies ★

The headline pitfall: a longer-lived service holding a shorter-lived one captive, accidentally promoting its lifetime.

A captive dependency happens when a service with a longer lifetime depends on one with a shorter lifetime. The classic case: a Singleton that injects a Scoped service. The singleton is built once and holds that scoped instance forever — so the scoped service is effectively promoted to a singleton, living far longer than intended.

♾️ Singleton (lives forever)
holds a reference to ↓
📦 Scoped service
— now trapped, lives forever too 🪤

The scoped service was meant to last one request — now it never gets released or refreshed

C#
// ❌ CAPTIVE DEPENDENCY: singleton captures a scoped DbContext
builder.Services.AddScoped<AppDbContext>();
builder.Services.AddSingleton<CacheService>();   // CacheService(AppDbContext ctx) -- BAD
// the DbContext is held for the app's life: stale data, threading bugs, leaks

// ✅ FIX: the singleton creates a scope per operation via IServiceScopeFactory
class CacheService
{
    private readonly IServiceScopeFactory _scopeFactory;
    public CacheService(IServiceScopeFactory scopeFactory) => _scopeFactory = scopeFactory;

    public void Refresh()
    {
        using var scope = _scopeFactory.CreateScope();
        var ctx = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        // use ctx within the scope, then it's disposed correctly
    }
}
🔑 The lifetime rule

A service may only depend on services with an equal or longer lifetime. So: Singleton → may inject only Singleton. Scoped → may inject Scoped or Singleton. Transient → may inject anything. Injecting "downhill" into a longer life (scoped into singleton, transient into singleton) is the captive-dependency bug.

✅ Scope validation catches it

The container can validate this for you. In the Development environment, scope validation is on by default and throws InvalidOperationException — e.g. "Cannot consume scoped service 'AppDbContext' from singleton 'CacheService'." — so the mistake surfaces immediately instead of as a subtle production bug. You can also force it with BuildServiceProvider(validateScopes: true).

⚠️ Middleware is a singleton trap

Custom middleware is constructed once for the app's lifetime, so constructor-injecting a scoped service into middleware captures it. Instead, inject scoped services into the middleware's Invoke/InvokeAsync method parameters (resolved per request), or use factory-based middleware.

07

Disposal

The container owns cleanup: it calls Dispose on the services it created, based on their lifetime.

This ties back to the IDisposable topic. The container tracks the disposable services it creates and disposes them automatically — calling Dispose() (or DisposeAsync()) at the right time. You should never dispose a service you resolved from the container yourself.

🔄Transient & Scopeddisposed at the end of the scope they were created in (e.g. end of the HTTP request)
♾️Singletondisposed when the application / root container shuts down
⚠️ Disposable transients from the root container leak

If you resolve a disposable Transient service from the root provider (not within a scope), the container holds onto it for disposal at app shutdown — so it accumulates for the entire application lifetime. That's effectively a memory leak. Resolve disposable transients within a scope so they're released when the scope ends.

✅ Let the container manage lifetime & disposal

Don't roll your own singleton with manual disposal — register it as a singleton and let the container handle it. Don't call Dispose() on injected services. Do dispose explicit scopes you create (with using), which in turn disposes the scoped/transient services inside them. The container only disposes instances it created — if you pass in an instance you built (AddSingleton(myInstance)), you may own its disposal.

08

Registration Techniques

Beyond the basic Add{Lifetime}<TInterface, TImpl> — factories, instances, multiples, and conditional registration.

C#
// interface -> implementation (the usual form)
services.AddScoped<IOrderRepository, SqlOrderRepository>();

// factory: build the instance with custom logic / other services
services.AddSingleton<IClock>(sp => new SystemClock(TimeZoneInfo.Utc));

// pre-built instance (you may own its disposal)
services.AddSingleton<IConfigCache>(myCache);

// TryAdd: register only if that service isn't already registered
services.TryAddScoped<IOrderRepository, SqlOrderRepository>();

// multiple implementations of one interface -> inject IEnumerable<T>
services.AddScoped<INotifier, EmailNotifier>();
services.AddScoped<INotifier, SmsNotifier>();
// a class can take IEnumerable<INotifier> to get ALL of them
MethodWhat it does
Add{Lifetime}<TI, TImpl>Map an interface to an implementation with a lifetime
Factory overload (sp => …)Build the instance yourself, using other services if needed
Instance overloadProvide an already-created object (singleton)
TryAdd…Register only if not already present — avoids accidental duplicates
Multiple registrationsResolve all via IEnumerable<T>; a single resolve gets the last one
🔑 GetService vs GetRequiredService

GetService<T>() returns null if the service isn't registered; GetRequiredService<T>() throws a clear exception instead. Prefer GetRequiredService when the service is mandatory — failing fast with a helpful message beats a mysterious NullReferenceException later. Better still, inject via the constructor rather than resolving manually.

09

Best Practices

The habits that keep a DI-based codebase clean, correct, and testable.

🏗️

Prefer constructor injection

Make dependencies explicit as constructor parameters. Easier to read, test, and reason about than pulling from the provider.

🚫

Avoid the service locator

Injecting IServiceProvider and resolving from it inside your code hides dependencies and is considered an anti-pattern. Inject what you need.

⏱️

Mind lifetimes

Never capture a shorter-lived service in a longer-lived one. Keep scope validation on. Default to the shortest lifetime that works.

🔒

Make singletons thread-safe

A singleton is shared across all requests/threads, so its state must be safe for concurrent access (see the threading topic).

✂️

Keep services small

Many injected dependencies hints at too many responsibilities (an SRP smell). Split the class up.

🧹

Let the container dispose

Don't dispose resolved services yourself; do dispose explicit scopes you create. Register, don't hand-manage, lifetimes.

🔑 Depend on abstractions, not the container

Good DI means your business classes don't know the container exists — they just take interfaces in their constructors. Only the composition root (startup) and a few infrastructure spots (like background services creating scopes) ever touch IServiceProvider directly. If you see GetService scattered through business logic, that's the service-locator anti-pattern creeping in.

✅ Singletons: let the container do it

Microsoft's guidance is explicit: if you need singleton behavior, register the service as a singleton and let the container manage its lifetime and disposal — don't implement the classic Singleton pattern (a static Instance property) yourself. The container gives you thread-safe creation and proper cleanup for free.

10

Worked Example

A small web-style setup showing each lifetime used correctly, with constructor injection throughout.

C#
// --- registrations (composition root) ---
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSingleton<IClock, SystemClock>();          // stateless, shared -> singleton
builder.Services.AddScoped<AppDbContext>();                      // per-request unit of work -> scoped
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();  // uses the DbContext -> scoped
builder.Services.AddTransient<IPriceCalculator, PriceCalculator>(); // stateless helper -> transient

var app = builder.Build();

// --- a service that receives its dependencies via the constructor ---
public class OrderService
{
    private readonly IOrderRepository _repo;     // scoped
    private readonly IPriceCalculator _calc;     // transient
    private readonly IClock _clock;               // singleton -- OK: longer-lived

    // the container supplies all three -- no 'new', no service locator
    public OrderService(IOrderRepository repo, IPriceCalculator calc, IClock clock)
    {
        _repo = repo; _calc = calc; _clock = clock;
    }

    public Order Place(Cart cart)
    {
        var total = _calc.Total(cart);
        var order = new Order(total, _clock.Now());
        _repo.Save(order);     // repo + its DbContext are the SAME scoped instances this request
        return order;
    }
}
✅ Right lifetime for each role

IClock is stateless and shared, so Singleton. AppDbContext and the repository represent this request's unit of work, so Scoped — and because the repository (scoped) depends on the DbContext (scoped) and the clock (singleton), the lifetime rule holds. IPriceCalculator is a throwaway helper, so Transient. Every dependency arrives through the constructor, the graph is built by the container, and nothing references IServiceProvider — textbook DI.

Quick Recap

One sentence per idea. If you can explain each card aloud, you've got it.

💉 DI

A class receives its dependencies from outside instead of creating them — an implementation of Inversion of Control.

🔗 Why

Loose coupling, swappable implementations, and testability (inject mocks instead of real services).

📦 Container

Register services in IServiceCollection; the IServiceProvider builds and injects them.

🏗️ Constructor injection

The preferred form — dependencies are explicit constructor parameters supplied by the container.

🔄 Transient

New instance every request. Lightweight, stateless services; the safe default.

📦 Scoped

One instance per scope (per web request). Per-request state, DB contexts, units of work.

♾️ Singleton

One instance for the whole app. Caches/config/expensive shared state — must be thread-safe.

🪤 Captive dependency

A longer-lived service capturing a shorter-lived one (e.g. scoped in a singleton) — promotes its lifetime, causing bugs.

📏 Lifetime rule

Depend only on equal-or-longer lifetimes: singleton→singleton, scoped→scoped/singleton, transient→anything.

🧹 Disposal

The container disposes services it created — scoped/transient at scope end, singletons at shutdown. Never dispose them yourself.

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is dependency injection? Junior

It's a technique where a class receives the objects it depends on from an external source instead of creating them itself with new. A container is configured with how to build each service and supplies them — usually through the class's constructor. This decouples a class from the concrete implementations it uses.

Q2 Why use dependency injection? Junior

Three big reasons: loose coupling (classes depend on interfaces, not concrete types), testability (you can inject a fake/mock in unit tests instead of a real database or network call), and flexibility (swap an implementation by changing one registration). It makes code easier to change and test.

Q3 What are the three service lifetimes? Junior

Transient — a new instance every time it's requested. Scoped — one instance per scope, which in a web app means per HTTP request. Singleton — one instance shared for the entire application lifetime. You choose the lifetime when registering with AddTransient, AddScoped, or AddSingleton.

Q4 What's the difference between Scoped and Singleton? Junior

A Singleton is created once and the same instance is shared across the whole application, including every request. A Scoped service is created once per scope — in web apps, once per HTTP request — so each request gets its own instance, but everything within that one request shares it. Singleton spans all requests; scoped is per request.

Q5 How do you register a service? Junior

In startup code, call the appropriate method on the service collection, e.g. builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();. This tells the container "whenever something needs an IOrderRepository, create a SqlOrderRepository with a scoped lifetime." The container then injects it wherever it's needed.

Q6 What is constructor injection? Junior

It's declaring a class's dependencies as constructor parameters, so the container supplies them when it creates the object. It's the preferred form of DI because the dependencies are explicit and guaranteed to be present before the object is used. (Method and property injection exist but are used much less often.)

🟠 Mid-level — going deeper (2–5 years)

Q7 What is a captive dependency? Mid

It's when a longer-lived service holds onto a shorter-lived one, accidentally extending its lifetime. The classic case is injecting a Scoped service into a Singleton: the singleton is created once and keeps that scoped instance forever, so it's effectively promoted to a singleton — leading to stale data, threading bugs, and leaks. The rule is that a service should only depend on services with equal or longer lifetimes.

Q8 When would you choose each lifetime? Mid

Transient for lightweight, stateless services (formatters, calculators) where a fresh instance is fine. Scoped for things that represent a single request's work — most notably a database context or unit-of-work — so the same instance is shared across the request but not leaked to others. Singleton for expensive-to-create or genuinely global, shared state like a cache or configuration, remembering it must be thread-safe.

Q9 Who disposes services resolved from the container? Mid

The container does — it tracks the disposable services it creates and calls Dispose (or DisposeAsync) automatically: scoped and transient services at the end of their scope, singletons at application shutdown. You should never dispose a service you got from the container. You do dispose explicit scopes you create (via using), which disposes the services inside them.

Q10 Why is EF Core's DbContext registered as Scoped? Mid

A DbContext represents a unit of work and tracks changes for a set of operations; it's also not thread-safe. A scoped lifetime gives each request its own context — shared by all the components handling that request, then disposed at the end — which matches the request's unit-of-work boundary and avoids cross-request state bleed. Making it a singleton would share one context (and its change tracker) across all requests and threads, which breaks badly.

Q11 How do you correctly use a scoped service from a singleton? Mid

Don't inject it directly — that's a captive dependency. Instead, inject IServiceScopeFactory into the singleton, and for each operation call CreateScope() (inside a using), resolve the scoped service from scope.ServiceProvider, use it, and let the scope dispose it. This creates a fresh, properly-bounded scope each time the singleton needs the scoped service.

Q12 What is scope validation? Mid

It's a container check that detects captive dependencies — specifically, a singleton consuming a scoped service. When enabled (it's on by default in the Development environment), the container throws an InvalidOperationException like "Cannot consume scoped service X from singleton Y," surfacing the misconfiguration at startup/resolution rather than as a subtle runtime bug. You can also force it via BuildServiceProvider(validateScopes: true).

Q13 Why can't you inject a scoped service into middleware via the constructor? Mid

Custom middleware is constructed once, when the request pipeline is built, and lives for the whole application. Constructor-injecting a scoped (or transient) service therefore captures a single instance for the app's lifetime — a captive dependency, which the container will reject. The fix is to inject scoped services into the middleware's Invoke/InvokeAsync method parameters (resolved per request), or to use factory-based middleware that's activated per request.

🔴 Senior — synthesis & judgment (5–7 years)

Q14 How does DI relate to the SOLID principles and IoC? Senior

DI is the practical mechanism for the Dependency Inversion Principle (the "D" in SOLID): high-level modules and low-level modules both depend on abstractions, not on each other's concrete types. Inversion of Control is the broader idea that a framework/container — not your code — drives object creation and calls into you ("don't call us, we'll call you"); DI is one form of IoC (others include the service locator and factories). DI also supports the Single Responsibility and Open/Closed principles by making collaborators explicit and swappable.

Q15 Why is the service locator considered an anti-pattern compared to injection? Senior

A service locator (injecting IServiceProvider and calling GetService inside your class) hides a class's real dependencies — you can't tell what it needs from its constructor, so the API lies about its requirements. That hurts readability, makes unit testing harder (you must stub the whole provider), risks runtime "service not found" failures instead of compile-time clarity, and couples business code to the container. Constructor injection makes dependencies explicit and the container invisible to business logic. The locator is acceptable only at the composition root and a few infrastructure seams (e.g. creating scopes in a background service).

Q16 A singleton genuinely needs a scoped dependency for each operation. Walk through the correct design. Senior

Inject IServiceScopeFactory (a singleton-safe service) into the singleton rather than the scoped service itself. For each logical operation, create a scope with using var scope = _scopeFactory.CreateScope();, resolve the scoped service from scope.ServiceProvider, do the work, and dispose the scope — which disposes the scoped instance correctly. This bounds the scoped service's lifetime to the operation, avoids the captive dependency, and keeps disposal correct. It's exactly how hosted/background services use scoped services like a DbContext. Be mindful that you're now responsible for creating an appropriate scope per unit of work, and that the work inside may need its own thread-safety considerations.

Q17 What disposal pitfalls exist with transient services, and how do you avoid them? Senior

If a disposable transient is resolved from the root container (outside any scope), the container captures it to dispose at application shutdown — so every such resolution accumulates for the app's lifetime, a slow memory leak. Avoid it by resolving disposable transients within a scope (so they're released at scope end), or reconsidering whether a disposable service should really be transient. More broadly: the container only disposes instances it created, so an instance you build and register via AddSingleton(instance) is yours to dispose. Enabling scope validation and being deliberate about where disposable services are resolved prevents most of these leaks.

Q18 If a thread-safe singleton depends on a transient service, does the transient need to be thread-safe? Senior

Potentially yes. Although the transient is "created fresh," when injected into a singleton it's created once (when the singleton is built) and then shared for the singleton's entire life — and the singleton may be used concurrently from many threads. So that captured transient instance can be accessed concurrently and must be thread-safe (or you must avoid sharing mutable state through it). This is really a captive-dependency consequence: the transient's effective lifetime and sharing match the singleton's, so it inherits the singleton's thread-safety requirements. The cleaner fix is usually to resolve such a dependency per-operation via a scope/factory rather than capturing it.

Q19 When would you register multiple implementations of one interface, and how are they resolved? Senior

When you want a set of interchangeable handlers — e.g. several INotifiers (email, SMS, push), validators, or pipeline behaviors — all invoked together. Register each implementation against the same interface; a constructor that takes IEnumerable<INotifier> receives all of them in registration order, while a single INotifier resolution returns the last registered one. This underpins patterns like chain-of-responsibility and the mediator/pipeline. Use TryAddEnumerable to avoid duplicate registrations of the same implementation type, and keyed services (added in .NET 8) when you need to resolve a specific named implementation rather than all of them.

Q20 What are the trade-offs of singletons, and when is one truly justified? Senior

Singletons avoid repeated construction and let state be shared globally, which is ideal for things genuinely expensive to build or inherently global — a cache, a connection multiplexer, configuration loaded once, a logger. The costs: they must be thread-safe; they can couple unrelated requests through shared state; they make testing harder (shared state across tests); they can keep a large object graph alive for the app's lifetime (memory impact); and they invite captive-dependency mistakes. Justify a singleton by genuine shared state or creation cost — not for stateless services (where transient/scoped is simpler and contention-free), and always let the container own its lifetime rather than hand-rolling the pattern.
