Polymorphism & Abstraction 🎭

OOP I gave you encapsulation and inheritance. This page finishes the four pillars with the two that make code flexible: polymorphism (one call, many behaviors — including the crucial override vs new distinction) and abstraction (program against ideas, not concrete details, using abstract classes and interfaces). Together they're why you can add new types to a system without rewriting the old ones. No prior coding background assumed.

🔑 Why these two matter most for design

Encapsulation and inheritance organize code; polymorphism and abstraction make it extensible. They're what let you write a loop over Shape objects today and drop in a brand-new Hexagon next year without changing that loop. This is the open/closed principle: open to extension, closed to modification.

02

The Big Idea

"Polymorphism" = "many forms." One name or one interface, many behaviors depending on context.

Polymorphism lets a single method name — or a single base-typed variable — behave differently depending on the situation. C# offers it in two flavors, split by when the decision is made:

🧮

Compile-time (static)

The compiler decides which method to call based on the arguments. This is method overloading (and operator overloading). Also called "early binding."

🎯

Runtime (dynamic)

The runtime decides based on the object's actual type. This is method overriding via virtual/override. Also called "late binding."

🗣️ Analogy

The word "cut" is polymorphic. Say it to a chef, a barber, or a film director and the same word triggers a completely different action — the listener (the runtime/compiler) picks the meaning from context. One name, many forms.

03

Compile-time Polymorphism

Method overloading: many methods share a name but differ in their parameters. The compiler picks.

Method overloading means defining several methods with the same name but different parameter lists — different number, types, or order of parameters. The compiler chooses which one to call by matching the arguments you pass. No inheritance is involved.

C#
class Printer
{
    public void Print(int n)        => Console.WriteLine($"int: {n}");
    public void Print(string s)     => Console.WriteLine($"string: {s}");
    public void Print(int a, int b) => Console.WriteLine($"two ints: {a},{b}");
}

var p = new Printer();
p.Print(5);        // compiler picks Print(int)
p.Print("hi");     // compiler picks Print(string)
p.Print(1, 2);     // compiler picks Print(int, int)
⚠️ Return type alone doesn't count

You cannot overload two methods that differ only by return type — the compiler can't tell them apart from a call. The parameter list (the "signature") is what distinguishes overloads. Likewise, differing only by parameter names doesn't work; the types/count/order must differ.

Operator overloading 🧩

Operators like + can also be "overloaded" to work on your own types — another form of compile-time polymorphism.

C#
public struct Money
{
    public decimal Amount;
    // define what '+' means for two Money values
    public static Money operator +(Money a, Money b)
        => new Money { Amount = a.Amount + b.Amount };
}
🔑 Why it's "compile-time"

The exact method is locked in when the program is compiled, based on the static types of the arguments. There's no runtime decision — hence "static polymorphism" or "early binding." Fast and predictable.

04

Runtime Polymorphism

Method overriding: the same method, redefined in a derived class — and the runtime calls the right one.

This is the form most people mean by "polymorphism." A base class marks a method virtual; derived classes override it. When you call the method through a base-typed variable, the runtime looks at the object's actual type and runs that type's version. It requires inheritance.

C#
abstract class Payment
{
    public abstract void Pay(decimal amount);
}

class CardPayment : Payment
{
    public override void Pay(decimal a) => Console.WriteLine($"Paid {a} by card");
}
class UpiPayment : Payment
{
    public override void Pay(decimal a) => Console.WriteLine($"Paid {a} by UPI");
}

// One loop, one call -- many behaviors. This is the payoff.
Payment[] payments = { new CardPayment(), new UpiPayment() };
foreach (Payment pay in payments)
    pay.Pay(100);   // "Paid 100 by card"  then  "Paid 100 by UPI"
🔑 Late binding

The variable is typed Payment, but the method that runs depends on whether the object is really a CardPayment or a UpiPayment — decided at runtime ("late binding"). Add a new CashPayment tomorrow and the loop above keeps working untouched. That extensibility is the whole point.

✅ The three enablers

Runtime polymorphism needs: (1) a base type, (2) a virtual or abstract member, and (3) an override in the derived type. Miss any one and you don't get dynamic dispatch.

05

override vs new ★

The single most-tested distinction in OOP interviews. Both let a child redefine a base method — but they resolve completely differently.

Both override and new give a derived class its own version of a base member with the same name. The difference is who decides which version runs:

🎯

override → the object decides

Polymorphic. The runtime picks the version based on the object's real type, no matter what type the variable is. Requires the base member to be virtual/abstract.

🏷️

new → the variable decides

Hiding (not overriding). The compiler picks the version based on the variable's declared type. The base member need not be virtual. new just silences the hiding warning.

The same object, two outcomes 🔬

C#
class Base
{
    public virtual void Speak() => Console.WriteLine("Base");
}

class OverrideChild : Base
{
    public override void Speak() => Console.WriteLine("Override!");
}

class NewChild : Base
{
    public new void Speak() => Console.WriteLine("New!");
}

Now hold each object two ways — once as its real type, once as the base type:

🎯 override
OverrideChild c = new();
c.Speak(); // "Override!"
Base b = c;
b.Speak(); // "Override!"

Same result both ways — the object wins. Consistent & polymorphic.

🏷️ new (hiding)
NewChild c = new();
c.Speak(); // "New!"
Base b = c;
b.Speak(); // "Base"

Different results! The variable type wins. Same object, two behaviors.

⚠️ The real-world bug

If you forget override and the compiler warns "hides inherited member," do not silence it by blindly adding new — that gives you hiding, not polymorphism, and your base-typed code will call the wrong version. Add new only when you genuinely intend hiding. 99% of the time, you meant override.

🎭 Analogy

override is an actor who truly becomes the role — whoever calls "the character" gets the new performance. new is a stand-in wearing a name tag: people who look at the name tag (the declared type) see the original; only those who know it's the stand-in (the derived type) see the new act.

06

Covariant Returns

A modern refinement (C# 9): an override may return a more specific type than the base method.

Normally an override must return the exact same type as the base method. Covariant return types (added in C# 9) relax this: the override can return a more derived type. This makes APIs cleaner — callers of the derived class get the precise type without casting.

C#
class Animal { }
class Dog : Animal { }

class AnimalShelter
{
    public virtual Animal Adopt() => new Animal();
}

class DogShelter : AnimalShelter
{
    // returns Dog (more specific) instead of Animal -- legal since C# 9
    public override Dog Adopt() => new Dog();
}

Dog d = new DogShelter().Adopt();  // no cast needed -- it's already a Dog
🔑 Why it's safe

A Dog is an Animal, so anyone expecting an Animal from the base contract still gets one. The derived class simply makes a stronger promise. This keeps polymorphism intact while removing annoying casts in derived-type code.

07

Abstraction — The Big Idea

Expose what something does; hide how it does it. Program against concepts, not details.

Abstraction means modeling the essential behavior of something while hiding the messy implementation. You define a clean contract — "a Shape has an Area()" — and let each concrete type fill in the details. Code that uses shapes only needs the idea of a shape, not the formula for every one.

🚗 Analogy

Driving a car is abstraction in action. The steering wheel, pedals, and gear stick are the contract. You don't need to know whether the engine is petrol, diesel, or electric — the interface is the same. The car abstracts away the mechanism. Swap the engine (the implementation) and your driving (the calling code) doesn't change.

C# gives you two tools for abstraction, and choosing between them is a core skill:

📋

Abstract classes

A partial base class: some real implementation + some members children must fill in. Best when related types share code and state.

📜

Interfaces

A pure contract: "any type that implements me promises these members." Best for capabilities shared by unrelated types.

🔑 Abstraction vs encapsulation

They sound similar but differ in focus. Encapsulation hides data (the internal state) behind access control. Abstraction hides complexity (the implementation) behind a simple contract. Encapsulation is "don't touch my variables"; abstraction is "you don't need to know how I work."

08

Abstract Classes

A base class you can't instantiate, mixing ready-made code with blanks the children must fill.

An abstract class can't be created with new — it exists only to be inherited. It can hold fields, constructors, and fully-implemented methods (shared code) alongside abstract members (no body) that every concrete subclass must implement.

C#
abstract class Shape
{
    public string Name { get; }
    protected Shape(string name) => Name = name;   // constructor: OK in abstract class

    public abstract double Area();              // blank -- children MUST implement

    // shared, concrete method that uses the abstract one
    public void Describe() => Console.WriteLine($"{Name} area = {Area():F2}");
}

class Circle : Shape
{
    public double R { get; }
    public Circle(double r) : base("Circle") => R = r;
    public override double Area() => Math.PI * R * R;
}
🔑 What only abstract classes can do

They can hold instance fields and state, have constructors, and provide shared implementation — all things a derived class inherits for free. The trade-off: a class can inherit from only one abstract class (single inheritance).

09

Interfaces

A pure contract that any number of unrelated types can sign — C#'s answer to "multiple inheritance."

An interface declares a set of members (methods, properties, events, indexers) that an implementing type promises to provide. It says what a type can do, not how. A class can implement many interfaces, which is how C# lets a type take on capabilities from several sources at once.

C#
interface IFlyable { void Fly(); }
interface ISwimmable { void Swim(); }

// a Duck can do BOTH -- multiple interfaces, no multiple class inheritance needed
class Duck : IFlyable, ISwimmable
{
    public void Fly()  => Console.WriteLine("Duck flies");
    public void Swim() => Console.WriteLine("Duck swims");
}

// program to the contract -- works for ANY flyable thing
IFlyable thing = new Duck();
thing.Fly();
🔑 What interfaces can't do (classically)

An interface has no instance fields and no constructors — it carries no per-object state. It's a promise about behavior, not a place to store data. Members are public by default. (Interfaces can, however, also work for struct types, which can't inherit from classes — another reason they're so useful.)

✅ Program to interfaces

Typing variables and parameters as interfaces (IRepository, ILogger) instead of concrete classes makes components swappable and code easy to test (you can drop in a fake implementation). This is the foundation of dependency injection and loosely-coupled design.

10

Default & Explicit Implementation

Two interface features worth knowing: providing a default body (C# 8), and resolving name clashes explicitly.

Default interface methods (C# 8) 🆕

Since C# 8, an interface member can ship with a default implementation. Existing implementers don't have to write it — they inherit the default. This lets library authors add new members to an interface without breaking every class that already implements it.

C#
interface ILogger
{
    void Log(string message);

    // default body -- implementers get this free unless they override it
    void LogError(string message) => Log($"ERROR: {message}");
}
⚠️ Default-method catch

To call a default-implemented member, you must reference the variable through the interface type, not the concrete class. It's a versioning/compatibility tool — not a replacement for putting shared code in an abstract class. Interfaces still can't hold instance state.

Explicit interface implementation 🔏

If a class implements two interfaces that declare the same member, or you want to hide an interface member from the class's normal public surface, implement it explicitly by prefixing the interface name. Such members are only reachable through the interface.

C#
interface IEnglish { void Greet(); }
interface IFrench  { void Greet(); }

class Greeter : IEnglish, IFrench
{
    // explicit: resolve the clash, one body per interface
    void IEnglish.Greet() => Console.WriteLine("Hello");
    void IFrench.Greet()  => Console.WriteLine("Bonjour");
}

Greeter g = new();
((IEnglish)g).Greet();   // Hello
((IFrench)g).Greet();    // Bonjour
// g.Greet();   // won't compile -- explicit members aren't on the class surface

Abstract Class vs Interface

The classic interview question. Both enable abstraction and polymorphism — but they're built for different jobs.

✅ How to choose

Use an abstract class when related types share real implementation and state, and form one hierarchy (e.g. Shape → Circle, Square). Use an interface when you're defining a capability that unrelated types should support, or when a type needs several such capabilities (e.g. IDisposable, IComparable, IFlyable). Rule of thumb: shared code + single hierarchy → abstract class; multiple, cross-cutting contracts → interface.

🔑 You can combine them

A class can inherit one abstract class and implement many interfaces at the same time: class Duck : Bird, IFlyable, ISwimmable. This is the common real-world shape — a base class for the core identity, interfaces for the assorted capabilities.

12

Worked Example

Abstraction + polymorphism together: a notification system that's open to new channels without edits.

We define an abstraction (INotifier), implement it polymorphically for several channels, and write calling code that works for all of them — and any future ones — without change.

C#
// the abstraction: a contract for "something that can notify"
interface INotifier
{
    void Send(string message);
}

class EmailNotifier : INotifier
{
    public void Send(string m) => Console.WriteLine($"📧 Email: {m}");
}
class SmsNotifier : INotifier
{
    public void Send(string m) => Console.WriteLine($"📱 SMS: {m}");
}

// This method NEVER changes, even as you add new notifier types.
void Alert(IEnumerable<INotifier> channels, string msg)
{
    foreach (INotifier c in channels)
        c.Send(msg);     // polymorphism: each channel sends its own way
}

INotifier[] channels = { new EmailNotifier(), new SmsNotifier() };
Alert(channels, "Server is down!");
// 📧 Email: Server is down!
// 📱 SMS: Server is down!
✅ What each pillar did here

Abstraction: INotifier hides how each channel works behind one Send contract. Polymorphism: the single c.Send(msg) call dispatches to the right implementation at runtime. The win: adding a PushNotifier means writing one new class — the Alert method and all existing callers stay untouched. That's the open/closed principle in action.

14

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is polymorphism? Junior

Polymorphism means "many forms." It lets the same method name or the same base-typed variable produce different behavior depending on context. For example, a single Pay() call can charge a card, a UPI account, or cash, depending on which payment object it's running on. It makes code flexible and reusable.

Q2 What are the two types of polymorphism in C#? Junior

Compile-time (static) polymorphism is method overloading — same name, different parameters; the compiler decides which to call. Runtime (dynamic) polymorphism is method overriding with virtual/override — the runtime decides based on the object's actual type. They're also called early binding and late binding.

Q3 What is method overloading? Junior

Defining multiple methods with the same name but different parameter lists (different number, types, or order of parameters) in the same class. The compiler picks the right one from the arguments you pass. You can't overload on return type alone.

Q4 What is method overriding? Junior

Redefining a base class's virtual (or abstract) method in a derived class using the override keyword, keeping the same signature but changing the behavior. When you call it, the runtime runs the derived version even through a base-typed variable. It requires inheritance.

Q5 What is abstraction? Junior

Abstraction is exposing what something does while hiding how it does it. You define a simple contract (like "a shape has an area") and let each concrete type provide the details. Driving a car is the classic example: you use the wheel and pedals without knowing the engine internals.

Q6 What is an interface? Junior

An interface is a contract that lists members (methods, properties, etc.) a type promises to provide, without saying how. A class that implements the interface must supply those members. A class can implement many interfaces, which is how C# allows behavior from multiple sources without multiple class inheritance.

Q7 What's the difference between an abstract class and an interface, simply? Junior

An abstract class is a partial base class that can include real code, fields, and a constructor, and a type can inherit only one. An interface is a pure contract with (classically) no implementation or state, and a type can implement many. Abstract class = shared code + single hierarchy; interface = capability that many types can sign.

Q8 Why can't you instantiate an abstract class? Junior

Because it's incomplete by design — it may contain abstract methods with no body. Creating one directly would leave those methods undefined. Its purpose is to be a base that concrete derived classes complete, so C# blocks new AbstractType() at compile time.

🟠 Mid-level — going deeper (2–5 years)

Q9 What is the difference between override and new? Mid

override is polymorphic: the method that runs is chosen by the object's real type, so even a base-typed variable calls the derived version. new hides the base member: the method that runs is chosen by the variable's declared type. Hold the same object as the base type and override still runs the derived code, while new runs the base code. override requires a virtual/abstract base member; new works on any member and just suppresses the hiding warning.

Q10 Given a base-typed variable holding a derived object, which method runs for each? Mid

If the derived class used override, the derived method runs (polymorphism). If it used new, the base method runs, because the call was bound to the base member via the variable's declared type. This is the canonical gotcha: same object, different result, depending solely on override vs new.

Q11 Can you overload methods that differ only by return type? Mid

No. Overload resolution is based on the parameter list (the signature), and the compiler determines the call from the arguments — it can't tell two same-parameter methods apart by return type alone. You must vary the number, types, or order of parameters. (Covariant return types are a separate feature about overriding, not overloading.)

Q12 When would you choose an interface over an abstract class? Mid

Choose an interface when you're defining a capability that possibly unrelated types should support (e.g. IComparable, IDisposable), when a type needs several such capabilities, or when you need structs to participate. Choose an abstract class when related types share real implementation and state and belong to one hierarchy. You can also combine: inherit one abstract class and implement many interfaces.

Q13 What are default interface methods, and why were they added? Mid

Since C# 8, an interface member can include a default body. Implementers inherit it unless they provide their own. The main reason is versioning: a library author can add a new member to a widely-used interface without breaking every existing class that implements it. Note you must call a default member through the interface type, and interfaces still can't hold instance state.

Q14 What is explicit interface implementation and when is it useful? Mid

It's implementing an interface member with the interface name as a prefix (void IFoo.Method()), so the member is accessible only through the interface, not the class's public surface. It's useful to resolve clashes when two interfaces declare the same member (giving each its own implementation), and to hide an interface member from the class's everyday API while still satisfying the contract.

Q15 What are covariant return types? Mid

A C# 9 feature letting an override return a more derived type than the base method declared. For example, a base Adopt() returns Animal, while DogShelter.Adopt() overrides it to return Dog. It's safe because a Dog is an Animal, and it spares callers of the derived class from casting.

Q16 How does polymorphism support the open/closed principle? Mid

The open/closed principle says code should be open to extension but closed to modification. With polymorphism, you write logic against a base type or interface (e.g. a loop over INotifier). To add new behavior you create a new implementing class — you don't touch the existing loop or its callers. New types extend the system without modifying tested code, reducing the risk of regressions.

🔴 Senior — synthesis & judgment (5–7 years)

Q17 Explain the runtime mechanism that makes override polymorphic but not new. Senior

Virtual members are dispatched through a method table (v-table) keyed by the object's real type: a virtual call indirects through that table, so the most-derived override is found regardless of the variable's compile-time type. A new member isn't an override — it's a separate, non-virtual member, so the call is bound at compile time to whichever member the variable's declared type exposes. No v-table lookup happens, so the declared type's version is what runs.

Q18 What problems do default interface methods introduce, despite solving versioning? Senior

They blur the interface/abstract-class line and add subtle rules: default members are callable only through the interface type, so the same object behaves differently depending on how it's referenced (echoing the new trap). They reintroduce a form of multiple inheritance of behavior, raising diamond-style ambiguity when two interfaces supply conflicting defaults (the implementer must disambiguate). And because interfaces still can't hold instance state, defaults can't build real stateful properties. Use them for compatibility, not as a general code-sharing mechanism — abstract classes remain better for shared implementation with state.

Q19 Why can overloading be ambiguous, and how does the compiler resolve it? Senior

With multiple overloads and implicit conversions, more than one candidate can match a call. The compiler runs overload resolution: it gathers applicable candidates, then picks the most specific one using "better conversion" rules (an exact type beats one requiring conversion; a more derived parameter type beats a less derived). If no single best candidate exists, you get a compile error (CS0121, ambiguous call). Pitfalls include null arguments, numeric literals that fit several types, and optional parameters — which is why deliberately overlapping overloads should be designed carefully or replaced with clearer distinct names.

Q20 Compare abstraction and encapsulation precisely. Senior

They're complementary but distinct. Encapsulation is a mechanism: hiding internal data and bundling it with behavior, enforced by access modifiers, to protect invariants and localize change. Abstraction is a design perspective: exposing a simplified model (a contract) and suppressing implementation complexity so callers depend on the idea, not the details. Encapsulation is largely about data hiding within a type; abstraction is about complexity hiding across a boundary (often via abstract classes/interfaces). Good design uses both: encapsulate the internals, then expose them through a clean abstraction.

Q21 When is method hiding (new) actually the right choice? Senior

Rarely, but legitimately: when you must introduce a member that collides with a base member you can't make virtual (e.g. a sealed third-party base, or a base member that wasn't designed for overriding), and you explicitly want the derived type to expose different behavior while leaving base-typed callers on the original. It's also used to "re-declare" a member with a more specific signature in some library patterns. The decision must be deliberate and documented, because the declared-type dispatch surprises maintainers — if you can use override instead, do.

Q22 How do interfaces enable testability and loose coupling? Senior

Depending on an interface rather than a concrete class means the consumer is decoupled from any specific implementation — you can substitute another at will. In tests you inject a mock or fake implementing the interface to isolate the unit under test from databases, networks, or clocks. This is the basis of dependency injection: code declares the contracts it needs, and an external composition root supplies concrete implementations. The result is modular, swappable, independently testable components — a direct practical payoff of abstraction + polymorphism.

Q23 Is polymorphism possible without inheritance in C#? Senior

Yes — it depends which kind. Compile-time polymorphism (overloading) needs no inheritance at all. Runtime polymorphism can be achieved through interfaces as well as class inheritance — implementing types need not share a base class, just the contract. There's also parametric polymorphism via generics (List<T>), and ad-hoc dispatch via pattern matching. So "polymorphism" is broader than "class inheritance"; inheritance is just one route to it.

Q24 What's the Liskov Substitution Principle and how does it relate to overriding? Senior

The Liskov Substitution Principle (the "L" in SOLID) says a derived type must be usable anywhere its base type is expected, without breaking correctness. Overriding must therefore honor the base method's contract: don't strengthen preconditions, weaken postconditions, throw unexpected exceptions, or change the meaning callers rely on. A classic violation is a Square : Rectangle whose overridden setters break code that assumes width and height vary independently. Polymorphism only delivers its benefits when overrides are substitutable — otherwise base-typed code that calls the override gets surprised, defeating the whole point.
