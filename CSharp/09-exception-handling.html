Exception Handling 🛡️

When something goes wrong at runtime — a missing file, a bad number, a null reference — C# raises an exception that interrupts normal flow. try / catch / finally lets you respond gracefully instead of crashing. This page covers the mechanics, the exception hierarchy, and the single most-asked interview distinction: throw vs throw ex — which one keeps your stack trace intact and which one quietly destroys your ability to debug. No prior coding background assumed.

01

Overview

An exception is the runtime's way of saying "I can't continue normally." Handling it well keeps your app stable and debuggable.

An exception is an object describing an error that occurred during execution — dividing by zero, opening a missing file, calling a method on null. When thrown, it interrupts the normal flow and travels up the call stack looking for code that can handle it. If nothing does, the program crashes. Exception handling is how you catch and respond to these events deliberately.

🔑 Exceptions are for the exceptional

Exceptions are for unexpected conditions, not normal control flow. Throwing and catching is relatively expensive, so don't use exceptions to, say, end a loop. When a failure is expected (parsing user input, looking up a maybe-missing key), prefer the Try… pattern (int.TryParse, TryGetValue) that returns a bool instead of throwing.

02

try / catch / finally

Three blocks: try the risky code, catch the failure, and finally clean up no matter what.

C#
try
{
    var text = File.ReadAllText("data.txt");   // might fail
    Console.WriteLine(text);
}
catch (FileNotFoundException ex)            // most specific first
{
    Console.WriteLine($"Missing file: {ex.FileName}");
}
catch (IOException ex)                       // broader I/O problems
{
    Console.WriteLine($"I/O error: {ex.Message}");
}
finally
{
    Console.WriteLine("Done (runs whether or not it failed)");
}
▶️try block runsyour risky code executes line by line
💥Exception thrown?flow jumps to the first matching catch — remaining try lines are skipped
🧹finally always runswhether or not an exception happened — ideal for cleanup
🎯

try

Wraps the code that might throw. If a line throws, the rest of the try is skipped.

🪤

catch

Handles a specific exception type. Multiple catches are checked top-to-bottom; the first match runs.

🧹

finally

Runs no matter what — exception or not — so it's where you release resources (files, connections).

⚠️ Order catches most-derived → least-derived

The first catch whose type matches (exactly or as a base class) wins. So a broad catch (Exception) placed before a specific catch (FileNotFoundException) would swallow everything and make the specific one unreachable — that's a compile error. Always put specific types first.

🔑 finally vs using

For objects that implement IDisposable (files, DB connections), a using statement is the cleaner way to guarantee cleanup — it compiles to a try/finally that calls Dispose(). Use an explicit finally for cleanup that isn't tied to an IDisposable.

The Exception Hierarchy

Every exception derives from System.Exception. Catching a base type also catches its descendants.

Exceptions form a class hierarchy rooted at System.Exception. Because catching a type also catches anything derived from it, you can catch broadly (IOException covers many file errors) or narrowly (FileNotFoundException specifically).

Message (human-readable description), StackTrace (the call path to where it was thrown — crucial for debugging), InnerException (the original exception, when one was wrapped in another), and Data (a dictionary for extra context). The StackTrace is exactly what the throw vs throw ex question is all about.

04

Throwing

You raise an exception yourself with throw when your code detects a condition it can't proceed past.

Use throw new SomeException(...) to signal a problem — most commonly to validate input at the start of a method (fail fast) rather than letting bad data cause a confusing error deeper down.

C#
decimal Withdraw(decimal balance, decimal amount)
{
    // guard clauses -- validate and fail fast with the RIGHT exception type
    if (amount <= 0)
        throw new ArgumentOutOfRangeException(nameof(amount), "Must be positive.");
    if (amount > balance)
        throw new InvalidOperationException("Insufficient funds.");

    return balance - amount;
}
✅ Throw the most specific type, with a clear message

Pick the exception type that best describes the problem — ArgumentNullException for a null arg, InvalidOperationException for a bad state — and write a message a developer reading the log will understand. Use nameof(param) so the parameter name stays correct if you rename it.

⚠️ throw is also an expression (C# 7+)

Since C# 7, throw can appear in expression positions, enabling concise patterns like name = input ?? throw new ArgumentNullException(nameof(input));. Handy, but don't let it hide important control flow.

05

throw vs throw ex ★

The headline distinction. Both rethrow a caught exception — but one preserves the stack trace and one destroys it.

When you catch an exception, do some work (like logging), and want to let it continue up the stack, you rethrow it. You have two ways to write it, and they are not equivalent:

✅ throw;
C#
catch (Exception ex)
{
    Log(ex);
    throw;        // rethrow as-is
}
at OriginalMethod() ← real origin kept ✓ at MiddleMethod() at Main()

Preserves the original stack trace — you still see where it really came from.

❌ throw ex;
C#
catch (Exception ex)
{
    Log(ex);
    throw ex;     // rethrow the object
}
at CurrentMethod() ← trace RESET here ✗ at Main() (original origin lost)

Resets the stack trace to this line — the real origin is gone.

🔑 The rule in one line

throw; (no operand) rethrows the current exception and keeps its original stack trace. throw ex; rethrows the exception object and resets the stack trace to the current line — making it look like the error originated where you rethrew it. Microsoft's analyzer flags throw ex; as rule CA2200.

⚠️ Why this ruins production debugging

The stack trace is your map to where a bug actually happened. With throw ex;, every rethrow point looks like the source, so a log entry points at your catch block instead of the failing line deep in the code. In a large codebase, that turns a five-minute fix into an hour of hunting. Almost always use throw;

✅ When (rarely) is throw ex acceptable?

Only if you intentionally want to hide internal stack details — e.g. at a trust boundary where you've deliberately decided the inner trace shouldn't leak. Even then, wrapping in a new exception (throw new …(msg, ex)) is usually the better choice, because it preserves the original as InnerException while presenting a clean outer message.

06

Wrapping & Dispatch

Two more ways to rethrow that keep information: wrap with an inner exception, or capture-and-throw from outside the catch.

Wrapping with InnerException 🎁

Sometimes you want to add context — translate a low-level error into a meaningful one for your domain — without losing the original. Pass the caught exception as the inner exception; it's preserved in the chain via the InnerException property.

C#
try
{
    LoadConfig();
}
catch (IOException ex)
{
    // add meaning, but keep the original as InnerException
    throw new ConfigurationException("Could not load config.", ex);
}
// later, both messages + both stack traces are available:
//   outer: ConfigurationException "Could not load config."
//   inner: IOException "..." with its original stack trace

ExceptionDispatchInfo — rethrow from elsewhere 📮

If you need to capture an exception and rethrow it later or from outside the catch block (common in async/Task code), throw; isn't available. ExceptionDispatchInfo (.NET 4.5+) captures the exception with its stack trace intact and rethrows it preserving everything.

C#
using System.Runtime.ExceptionServices;

ExceptionDispatchInfo captured = null;
try { Risky(); }
catch (Exception ex) { captured = ExceptionDispatchInfo.Capture(ex); }

DoCleanup();                   // work between catching and rethrowing
captured?.Throw();              // rethrows with the ORIGINAL stack trace preserved

07

Exception Filters

The when keyword (C# 6) lets a catch block decide — before catching — whether it actually wants this exception.

An exception filter adds a boolean condition with when. The catch only activates if the condition is true; otherwise the exception keeps propagating to the next handler — without unwinding the stack, so the original trace and local debugging context are preserved.

C#
try
{
    response = await http.GetAsync(url);
}
// only handle 429 (rate-limited) here; let everything else bubble up
catch (HttpRequestException ex) when (ex.StatusCode == HttpStatusCode.TooManyRequests)
{
    await RetryAfterDelay();
}
// distinct handling for a different condition, same exception type
catch (HttpRequestException ex) when (ex.StatusCode == HttpStatusCode.NotFound)
{
    UseDefault();
}
🔑 Filters beat catch-then-check

The old way — catch, test a condition, and throw; if it's not yours — works, but the act of catching already unwound the stack. A when filter is evaluated before the stack unwinds, so if it returns false the exception continues untouched, with its full original context intact. Cleaner and better for debugging.

✅ Logging without catching

A neat trick: a filter that calls a logging method and always returns false logs the exception without handling it — so it keeps propagating. catch (Exception ex) when (Log(ex)) where Log returns false records every exception while letting it bubble up unchanged.

08

Custom Exceptions

When built-in types don't carry enough meaning, define your own — with domain data callers can use.

Built-in exceptions like ArgumentException are great for low-level problems, but they don't carry domain context. If callers need to handle "invalid coupon" differently from "invalid email," parsing the message string is fragile. A custom exception gives them a specific type to catch and typed properties to read.

C#
// derive from Exception; add domain-specific data
public class InsufficientFundsException : Exception
{
    public decimal Shortfall { get; }

    // provide the standard constructors, including one taking an inner exception
    public InsufficientFundsException(string message, decimal shortfall)
        : base(message) => Shortfall = shortfall;

    public InsufficientFundsException(string message, Exception inner)
        : base(message, inner) { }
}

// caller can catch the specific type and read typed data
try { account.Withdraw(500); }
catch (InsufficientFundsException ex)
{
    Console.WriteLine($"Short by {ex.Shortfall:C}");   // no string parsing!
}
🔑 Conventions for custom exceptions

End the name with Exception. Derive directly from Exception (modern guidance — the old ApplicationException base is discouraged). Provide the three standard constructors (parameterless, message, message + inner exception) plus any you need for your data. Make it serializable if it may cross process boundaries.

✅ When to create one

Create a custom exception when callers should be able to handle this failure distinctly from others, or when you have domain data worth attaching. If a built-in type already conveys the meaning accurately (ArgumentNullException for a null arg), just use that — don't proliferate types needlessly.

09

Best Practices

The habits that separate robust code from code that hides its own bugs.

🎯

Catch what you can handle

Only catch exceptions you can meaningfully recover from. Otherwise let them propagate to code that can — or to a global handler.

🚫

Don't swallow

An empty catch { } hides bugs and makes failures invisible. If you catch, at minimum log it.

♻️

Use throw, not throw ex

Preserve the stack trace when rethrowing. Reserve wrapping (throw new …(msg, ex)) for adding context.

🧹

Clean up reliably

Use using for IDisposable resources; finally for the rest. Don't throw from finally.

🔢

Don't use exceptions for flow

Prefer TryParse/TryGetValue for expected failures — throwing is costly and obscures intent.

🎚️

Be specific

Catch the narrowest type that fits. Avoid bare catch (Exception) unless you rethrow or it's a top-level handler.

C#
// ❌ exceptions for control flow -- slow and noisy
try { age = int.Parse(input); }
catch (FormatException) { age = 0; }

// ✅ expected failure -> use the Try pattern (no exception thrown)
if (!int.TryParse(input, out age))
    age = 0;

// ❌ swallowing -- the bug vanishes silently
try { DoWork(); } catch { }

// ✅ handle or log, then rethrow if you can't recover
try { DoWork(); }
catch (Exception ex) { Log(ex); throw; }
⚠️ Never throw from a finally block

An exception thrown in finally replaces any exception currently propagating — masking the real error. Microsoft's rule CA2219 flags this. Keep finally blocks to safe cleanup that won't itself throw.

10

Worked Example

A service method that validates, wraps low-level errors with context, rethrows correctly, and cleans up — all the pieces together.

C#
public Order PlaceOrder(string customerId, int quantity)
{
    // 1) validate -- fail fast with the right exception type
    if (string.IsNullOrEmpty(customerId))
        throw new ArgumentException("Required.", nameof(customerId));
    if (quantity <= 0)
        throw new ArgumentOutOfRangeException(nameof(quantity));

    var db = new DbConnection();
    try
    {
        db.Open();
        return db.Save(customerId, quantity);
    }
    // 2) only handle the case we can add meaning to; filter the rest out
    catch (SqlException ex) when (ex.Number == 1205)   // deadlock
    {
        // 3) wrap with domain context, keep original as InnerException
        throw new OrderException("Order failed due to a deadlock; retry.", ex);
    }
    catch (Exception ex)
    {
        Log(ex);
        throw;   // 4) rethrow preserving the stack trace
    }
    finally
    {
        db.Dispose();   // 5) always clean up the connection
    }
}
✅ Every concept in one place

Validation throws specific types up front. An exception filter (when) catches only the deadlock case. That case is wrapped with domain context while keeping the original as InnerException. Anything else is logged and rethrown with throw; to preserve the trace. And finally guarantees the connection is released no matter what. That's production-grade exception handling.

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is an exception? Junior

An exception is an object that represents an error occurring during program execution — like dividing by zero or opening a missing file. When thrown, it interrupts the normal flow and travels up the call stack looking for code that handles it. If nothing handles it, the program crashes.

Q2 What do try, catch, and finally do? Junior

try wraps code that might fail. catch handles a specific exception type if one is thrown — you can have several, checked top to bottom. finally runs afterward whether or not an exception occurred, which makes it the place to release resources like files or connections.

Q3 Does the finally block always run? Junior

Almost always — whether the try succeeds, throws and is caught, or throws and isn't caught, the finally runs before control leaves. (The rare exceptions are catastrophic process termination like Environment.FailFast or a power loss.) That reliability is why cleanup goes there.

Q4 What's the difference between throw and throw ex? Junior

throw; (with no exception named) rethrows the current exception and keeps its original stack trace — you still see where it really came from. throw ex; rethrows the exception object but resets the stack trace to the line where you wrote it, hiding the true origin. Use throw; — it's almost always what you want.

Q5 What is a stack trace and why does it matter? Junior

A stack trace is the list of method calls leading to where the exception was thrown — like a breadcrumb trail from the error back through the code that called it. It's the single most useful piece of debugging information, because it tells you exactly where and how the failure happened. Preserving it (with throw;) is essential.

Q6 Why should catch blocks be ordered specific-to-general? Junior

The first catch whose type matches the exception (exactly or as a base class) runs. If you put a broad catch (Exception) first, it would match everything, so a more specific catch after it could never run — and C# treats that as a compile error. So list specific types first, broad ones last.

🟠 Mid-level — going deeper (2–5 years)

Q7 Exactly why does throw ex reset the stack trace? Mid

When you write throw ex;, you're throwing an exception object with an operand, and the CLR treats that as throwing "from here" — it overwrites the stack trace starting at the current method, discarding the call chain between the original throw site and this point. throw; with no operand tells the runtime to rethrow the in-flight exception without resetting, so the original trace is preserved. Microsoft's analyzer flags throw ex; as rule CA2200.

Q8 How do you rethrow while adding context without losing the original error? Mid

Wrap it: throw new MyException("higher-level message", ex);. The caught exception becomes the new exception's InnerException, so both messages and both stack traces remain available. This is how you translate a low-level error (e.g. SqlException) into a domain-meaningful one while preserving the root cause for debugging.

Q9 What are exception filters and what advantage do they have? Mid

Exception filters (C# 6) add a when (condition) clause to a catch. The catch activates only if the condition is true; otherwise the exception keeps propagating. The advantage: the filter is evaluated before the stack unwinds, so if it returns false the exception's original stack trace and local context are untouched — unlike catch-then-check-then-rethrow, which already unwound and (if you used throw ex) could damage the trace.

Q10 When should you create a custom exception, and how? Mid

Create one when callers need to handle a failure distinctly from others or you have domain data worth attaching. Derive from Exception, name it ending in Exception, add typed properties for your data, and provide the standard constructors (message, and message + inner exception). Avoid deriving from the discouraged ApplicationException. If a built-in type already conveys the meaning, use that instead.

Q11 Why shouldn't you use exceptions for control flow? Mid

Throwing and catching is relatively expensive (capturing a stack trace, unwinding) and signals "something went wrong" — using it for ordinary, expected outcomes is both slow and misleading. For expected failures like parsing user input or a maybe-missing dictionary key, use the Try… pattern (int.TryParse, TryGetValue) that returns a bool. Reserve exceptions for genuinely exceptional conditions.

Q12 What's wrong with an empty catch block? Mid

An empty catch { } swallows the exception silently — the program keeps running as if nothing happened, but the underlying problem is hidden and the bug becomes invisible. At minimum, log the exception. Better: only catch what you can actually handle, and let everything else propagate to code (or a global handler) that can deal with it or surface it.

Q13 What does catching a base exception type catch? Mid

It catches that type and all types derived from it. So catch (Exception) catches everything, and catch (IOException) catches FileNotFoundException, DirectoryNotFoundException, etc. This is why catch order matters and why you should generally catch the narrowest type that fits — broad catches can unintentionally handle (and hide) errors you didn't anticipate.

🔴 Senior — synthesis & judgment (5–7 years)

Q14 How do you rethrow an exception captured outside a catch block while preserving its stack trace? Senior

Use ExceptionDispatchInfo (.NET 4.5+): in the catch, call ExceptionDispatchInfo.Capture(ex) to snapshot the exception with its stack trace; later — even after other work, or from a different method — call .Throw() on it to rethrow with the original trace intact. throw; only works directly inside the catch, so ExceptionDispatchInfo is the tool for async/TPL scenarios and deferred rethrow. It's what the framework uses to surface task exceptions through await without aggregating.

Q15 Compare throw ex, throw, throw new (wrapping), and ExceptionDispatchInfo for rethrowing. Senior

throw ex; resets the stack trace (anti-pattern). throw; rethrows in-place preserving the trace — the default for "log and rethrow." throw new X(msg, ex) preserves the original as InnerException while adding a higher-level message — for translating errors across abstraction boundaries. ExceptionDispatchInfo.Capture/Throw preserves the trace when you must rethrow outside the catch or after intervening work. Choose by intent: rethrow as-is → throw;; add meaning → wrap; rethrow elsewhere → dispatch info.

Q16 How do exception filters interact with the stack and side effects, and what's a risk? Senior

A when filter runs before the stack unwinds, in the context of the throwing frame — so it preserves the original trace and can inspect live state. Two consequences: (1) it enables "log without catching" via a filter that returns false; (2) filter code runs as part of exception dispatch, so it should be fast and must not throw (an exception in a filter is swallowed and the filter is treated as false, which can mask issues). Avoid heavy work or side effects with surprising ordering in filters.

Q17 Why is throwing from a finally block dangerous? Senior

If an exception is already propagating out of the try/catch and the finally throws a new one, the new exception replaces the original — the real error is lost, along with its stack trace, and callers see the cleanup error instead of the root cause. This makes diagnosis very hard. Microsoft's rule CA2219 flags it. Keep finally to cleanup that can't throw, and if cleanup might fail, guard it (e.g. check state) or handle its exception locally without letting it escape.

Q18 What's the performance cost of exceptions, and how should it shape design? Senior

Throwing is expensive relative to normal returns: the runtime captures a stack trace and unwinds frames searching for a handler. A caught exception on a cold path is negligible; throwing thousands per second on a hot path is not. So design APIs to avoid exceptions for expected outcomes — offer Try… variants, validate cheaply, return result objects or status codes for routine failures — and reserve exceptions for truly exceptional conditions. This is why int.TryParse exists alongside int.Parse.

Q19 How should exceptions be handled in async/task-returning methods? Senior

Exceptions thrown inside the asynchronous part of a task-returning method are captured and stored on the returned Task; they don't surface until the task is awaited (or its .Result/.Wait() is accessed, which wraps them in AggregateException). So validate arguments and throw ArgumentException-type errors synchronously before the first await, so callers see them immediately. When awaiting, the original exception is rethrown with its stack trace preserved (the framework uses ExceptionDispatchInfo internally). Prefer catching OperationCanceledException over its derived TaskCanceledException for cancellation.

Q20 What's your strategy for global/unhandled exception handling in an application? Senior

Handle exceptions locally only where you can recover; let the rest bubble to a top-level boundary. There, a global handler logs the full exception (message, type, stack trace, inner exceptions) with correlation context, returns a clean error response (e.g. a sanitized message to users, never raw stack traces), and decides whether to continue or fail fast. In ASP.NET Core that's exception-handling middleware; for unobserved cases there are AppDomain.UnhandledException and TaskScheduler.UnobservedTaskException. The principle: catch low to recover, catch high to log and translate, never swallow in between.
