Collections, Generics & yield 📚

Three features that almost always show up together. Collections store groups of items (lists, dictionaries, sets). Generics let those collections — and your own code — work with any type safely and without boxing. And yield lets you produce sequences lazily, one item at a time, without building a whole list in memory. Learn them as a trio and a huge amount of everyday C# clicks into place. No prior coding background assumed.

📋 List / Dictionary 🔤 HashSet / Queue / Stack 🧰 Generics <T> 🔒 Constraints 🐢 yield / lazy 🧠 Interview Prep
01

Overview

Why these three belong together: collections hold data, generics make them type-safe and fast, and yield streams them efficiently.

Picture a shopping list, a phone book, and a queue at a counter. Programs need to store groups of things just like that — and C#'s collections give you ready-made structures for each shape of problem. They're built on generics, which let one collection type (like List<T>) hold strings, numbers, or your own objects with full type safety. And yield lets you generate sequences on demand instead of materializing everything up front.

🔑 The thread connecting all three

The interface IEnumerable<T> — "something you can iterate one item at a time." Collections implement it, generics make it type-safe, and yield is the easiest way to produce it. foreach and LINQ work on anything that's IEnumerable<T>, which is why these features feel so unified.

02

Arrays & the Landscape

The array is the simplest collection. Everything else trades a little of its speed for flexibility.

An array is a fixed-size, contiguous block of elements of one type. It's the fastest and most memory-compact way to store a known number of items, but its size can't change after creation.

C#
int[] scores = { 90, 85, 72 };   // fixed size of 3
scores[1] = 88;                    // fast index access (O(1))
Console.WriteLine(scores.Length);    // 3 -- can't grow or shrink
90
[0]
88
[1]
72
[2]
🔑 Why we need more than arrays

Most real data is dynamic — you don't know how many items you'll have, you add and remove as you go, or you need to look things up by a key rather than a position. The System.Collections.Generic namespace provides flexible, type-safe collections built for exactly these patterns. Reach for arrays only when the size is fixed and performance is critical.

⚠️ Avoid the old non-generic collections

You may see legacy types like ArrayList and Hashtable from System.Collections. These store everything as object, which means boxing for value types and no compile-time type safety. Always prefer the generic versions (List<T>, Dictionary<K,V>) — that's exactly the problem generics solved.

03

The Big Collections

Six workhorses. Choosing the right one is mostly about how you'll access the data.

📋

List<T>

A resizable array. Fast index access and add-to-end; your default for an ordered, growable sequence.

🔑

Dictionary<K,V>

Key→value lookup. Near-instant find/add/remove by key. Use when you look things up by an identifier.

🎯

HashSet<T>

Unordered set of unique items. Fast "does it contain X?" and automatic de-duplication.

🚶

Queue<T>

First-in, first-out (FIFO). Enqueue at the back, Dequeue from the front. Like a line at a shop.

🥞

Stack<T>

Last-in, first-out (LIFO). Push on top, Pop off the top. Like a stack of plates.

🔗

LinkedList<T>

Nodes linked in a chain. Cheap insert/remove in the middle, but no fast index access. Niche but useful.

C#
var list = new List<string> { "a", "b" };
list.Add("c");                 // grows automatically

var ages = new Dictionary<string, int>();
ages["Sam"] = 30;             // add/update by key
if (ages.TryGetValue("Sam", out var a)) Console.WriteLine(a);  // safe lookup

var unique = new HashSet<int> { 1, 2, 2, 3 };  // stores {1, 2, 3}

var queue = new Queue<string>();
queue.Enqueue("first");        // FIFO
string next = queue.Dequeue();

var stack = new Stack<string>();
stack.Push("top");             // LIFO
string top = stack.Pop();

Performance at a glance ⚡

"Big-O" describes how an operation scales: O(1) = constant (instant regardless of size), O(n) = grows with the number of items.

✅ How to choose, quickly

Need order and indexing? List<T>. Looking up by a key? Dictionary<K,V>. Only care whether something exists / want no duplicates? HashSet<T>. Processing in arrival order? Queue<T>. Most-recent-first? Stack<T>. When unsure, start with List<T> — it's right far more often than not.

🔑 Concurrent & immutable variants

For multi-threaded code, use the thread-safe collections in System.Collections.Concurrent (ConcurrentDictionary, ConcurrentQueue…). For data that must never change after creation, use System.Collections.Immutable (ImmutableList, ImmutableArray…). Same shapes, different guarantees.

04

Collection Interfaces

Why you can foreach over almost anything: they all implement the same family of interfaces.

Collections share a hierarchy of interfaces. Each adds capabilities to the one above. Programming against these (rather than the concrete type) keeps your code flexible — a method that accepts IEnumerable<T> works with a list, an array, a set, or a lazy yield sequence.

IEnumerable<T> — "you can iterate me"
ICollection<T> — + Count, Add, Remove, Contains
IList<T> — + index access [i], Insert, RemoveAt
🔁

IEnumerable<T>

The most basic: "I can be walked one item at a time." All foreach and LINQ needs. Lazy sequences stop here.

📦

ICollection<T>

Adds Count and modification (Add, Remove, Contains). A finite, countable group.

🔢

IList<T>

Adds positional access — get/set by index, insert and remove at a position. Ordered sequences.

✅ Accept the weakest type you need

If your method only iterates, take IEnumerable<T> — then it accepts every collection and lazy sequences. Only ask for IList<T> if you genuinely need indexing. Accepting the most general interface that does the job makes your API maximally reusable.

🔑 IEnumerable vs IEnumerator

An IEnumerable<T> is "a thing you can get an enumerator from" (it has GetEnumerator()). An IEnumerator<T> is the actual cursor that walks the sequence (MoveNext() + Current). foreach calls GetEnumerator() once, then MoveNext() repeatedly. You'll meet this again with yield.

05

Generics — The Idea

Write code once that works with any type, with full type safety and no boxing. The <T> is a placeholder.

Generics let you parameterize a class or method by a type. The T in List<T> is a placeholder filled in when you use it: List<int>, List<string>, List<Customer> — one definition, infinitely reusable, and the compiler enforces the type at every step.

C#
// Without generics: store as 'object' -> boxing + casts + runtime errors
var oldList = new ArrayList();
oldList.Add(5);                 // boxed
oldList.Add("oops");            // no error -- different type slips in!
int n = (int)oldList[1];      // 💥 runtime InvalidCastException

// With generics: type-safe, no boxing, errors caught at COMPILE time
var list = new List<int>();
list.Add(5);                    // stored as a real int
// list.Add("oops");           // ❌ won't even compile
int m = list[0];               // no cast needed

Why generics matter 💡

🛡️

Type safety

The compiler catches type mistakes before you run. No surprise InvalidCastException at runtime.

⚡

No boxing

Value types are stored directly, avoiding heap allocations and GC pressure — a real performance win.

♻️

Reuse

One List<T> serves every element type. No copy-pasting an IntList, StringList, etc.

📦 Analogy

A generic type is a shipping box with a "contents" label you fill in: a Box<T> becomes a "box of books" or a "box of mugs." Same box design, but once labeled, everyone knows exactly what's inside and you can't accidentally toss a mug into the book box.

🔑 A bit of history

Generics arrived in C# 2.0, and System.Collections.Generic came with them — which is why the older boxing-prone collections still linger in the framework. Modern code is generic almost everywhere.

06

Generic Classes & Methods

You can make your own generics — not just consume the built-in ones.

A generic class 📦

C#
// T is declared on the class; every member can use it
class Box<T>
{
    private T _item;
    public void Put(T item) => _item = item;
    public T Take() => _item;
}

var intBox = new Box<int>();
intBox.Put(42);
int x = intBox.Take();      // strongly typed -- no cast

var nameBox = new Box<string>();
nameBox.Put("hello");

A generic method 🔧

A single method can be generic even inside a non-generic class. The compiler can usually infer the type from the arguments, so you rarely write it explicitly.

C#
void Swap<T>(ref T a, ref T b)
{
    T temp = a;
    a = b;
    b = temp;
}

int p = 1, q = 2;
Swap(ref p, ref q);          // T inferred as int -- no <int> needed

string s1 = "x", s2 = "y";
Swap(ref s1, ref s2);        // same method, now T = string
✅ Multiple type parameters

You can have several: Dictionary<TKey, TValue>, or your own Pair<TFirst, TSecond>. By convention, single-letter names start at T, and descriptive ones are prefixed with T (TKey, TResult).

07

Constraints

By default a generic knows nothing about T. Constraints (where) tell the compiler what T can do.

An unconstrained T could be anything, so the compiler only lets you do what every type supports. A constraint narrows T and unlocks more operations — calling interface methods, creating instances, comparing, and so on.

C#
// T must implement IComparable<T> AND have a parameterless constructor
class SortedBox<T> where T : IComparable<T>, new()
{
    public T Max(T a, T b)
        => a.CompareTo(b) >= 0 ? a : b;   // allowed because of IComparable

    public T CreateDefault() => new T();    // allowed because of new()
}

⚠️ Ordering & exclusivity rules

At most one of class / struct / notnull / unmanaged, and it must come first. Then a base class, then interfaces, and new() always last. Mutually exclusive combos won't compile (e.g. struct already guarantees a parameterless constructor, so struct, new() is rejected).

🔑 Why constrain at all?

Without a constraint, the compiler treats T as object — you can't call .CompareTo(), new T(), or anything type-specific. Constraints are how you promise the compiler "every T here will have this capability," which then lets you use it safely.

08

yield — The Idea

Produce a sequence one item at a time, on demand, without ever building the whole thing in memory.

Normally, to return a sequence you build a List, fill it, and return it — all of it, up front. An iterator using yield does it differently: it hands back items one at a time, only when asked. The method's body doesn't even run until something starts iterating, and it pauses after each item, resuming where it left off. This is lazy / deferred execution.

C#
// the old way: build the whole list, then return it
IEnumerable<int> EvensEager(int max)
{
    var result = new List<int>();
    for (int i = 0; i <= max; i++)
        if (i % 2 == 0) result.Add(i);
    return result;                 // allocates the entire list
}

// the yield way: produce items lazily, no list allocated
IEnumerable<int> EvensLazy(int max)
{
    for (int i = 0; i <= max; i++)
        if (i % 2 == 0) yield return i;   // hand back one, then pause
}
🔑 What the compiler does for you

When it sees yield, the C# compiler rewrites your method into a hidden state machine — a class implementing IEnumerator that remembers where it paused. You write simple linear code; the compiler generates the bookkeeping that makes "pause and resume" work.

🥤 Analogy

Eager = filling a hundred cups of water before anyone's thirsty. Lazy (yield) = a tap that pours a cup only when someone holds one out. If only three people show up, you never poured the other ninety-seven — no waste.

09

yield return & yield break

Two statements: one hands back the next item, the other ends the sequence.

  • yield return x; — gives x to the consumer and pauses, remembering its place. Next request resumes right after it.
  • yield break; — ends the sequence immediately, like an early return for iterators. No more items are produced.
C#
IEnumerable<int> FirstThreeEvens(List<int> numbers)
{
    int count = 0;
    foreach (int n in numbers)
    {
        if (n % 2 == 0)
        {
            yield return n;       // emit, then pause
            count++;
        }
        if (count == 3) yield break;  // stop early -- no more items
    }
}

The execution dance 💃

Watch how control ping-pongs between the caller and the iterator. The iterator runs only far enough to produce the next value, then yields control back:

caller
foreach asks for item #1
iterator
runs to first yield return → gives 0, pauses
caller
processes 0, asks for item #2
iterator
resumes after the yield → gives 2, pauses
caller
processes 2, asks for item #3
iterator
resumes → gives 4, pauses
caller
asks again → iterator ends → foreach stops
🔑 Infinite sequences become possible

Because items are produced on demand, an iterator can describe an endless sequence safely — the consumer decides when to stop. Pair it with LINQ's Take to grab just what you need.

C#
IEnumerable<int> Naturals()       // 1, 2, 3, ... forever
{
    int i = 1;
    while (true) yield return i++;   // never ends on its own
}

foreach (int n in Naturals().Take(5))   // consumer stops it at 5
    Console.Write($"{n} ");          // 1 2 3 4 5
10

Deferred Execution & Pitfalls

The lazy nature of iterators is powerful — and the source of the most common bugs. Know these traps.

⚠️ Pitfall 1 — nothing runs until you iterate

Calling an iterator method does not execute its body. The code runs only when you start a foreach (or LINQ materializer). So side effects, exceptions, or expensive work happen later than the call site suggests — sometimes catching people off guard.

⚠️ Pitfall 2 — multiple enumeration re-runs everything

Each time you iterate the same IEnumerable, the iterator starts over from the beginning and does all the work again. Iterating a deferred query twice doubles the cost (and may give different results if the source changed). If you'll use the results more than once, materialize with .ToList() or .ToArray().

C#
var query = numbers.Where(n => IsExpensive(n));  // nothing runs yet (deferred)

var count = query.Count();   // runs the filter once
var list  = query.ToList();  // runs the filter AGAIN -- double work!

// ✅ fix: materialize once, then reuse
var results = numbers.Where(n => IsExpensive(n)).ToList();
var c = results.Count;     // no recompute
⚠️ Pitfall 3 — disposed resources

If an iterator opens a resource in a using block and yields inside it, the resource may be disposed before the (deferred) enumeration actually happens — giving an ObjectDisposedException. Keep the resource alive across enumeration (e.g. try/finally) or use a built-in lazy reader like File.ReadLines.

✅ When deferral is a gift

Lazy evaluation shines for large or infinite data: stream a multi-gigabyte log file line by line without loading it all, or filter a feed and stop as soon as you've found enough. You only pay for the items you actually consume. Just decide consciously whether to keep it lazy or materialize it.

Putting It Together

A custom generic collection that's iterable via yield — all three features in one small type.

Here's a tiny generic ring buffer (a fixed-size collection that overwrites the oldest item when full). It's generic (<T>), it's a collection (implements IEnumerable<T>), and it exposes its items lazily with yield.

C#
class RingBuffer<T> : IEnumerable<T>
{
    private readonly T[] _items;
    private int _next, _count;

    public RingBuffer(int capacity) => _items = new T[capacity];

    public void Add(T item)
    {
        _items[_next] = item;
        _next = (_next + 1) % _items.Length;     // wrap around
        _count = Math.Min(_count + 1, _items.Length);
    }

    // the iterator: yields items lazily, oldest to newest
    public IEnumerator<T> GetEnumerator()
    {
        for (int i = 0; i < _count; i++)
            yield return _items[(_next - _count + i + _items.Length) % _items.Length];
    }

    IEnumerator IEnumerable.GetEnumerator() => GetEnumerator();
}

var recent = new RingBuffer<string>(3);
recent.Add("a"); recent.Add("b"); recent.Add("c"); recent.Add("d");
foreach (var x in recent) Console.Write($"{x} ");   // b c d  (a was overwritten)
✅ What each feature contributed

Generics let one RingBuffer<T> hold strings, ints, or anything. Implementing IEnumerable<T> means it works with foreach and every LINQ operator for free. yield made GetEnumerator a few trivial lines instead of a hand-written enumerator class with manual state tracking.

13

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What's the difference between an array and a List<T>? Junior

An array has a fixed size set at creation and can't grow or shrink. A List<T> is a resizable collection — it grows automatically as you Add items (internally backed by an array it re-allocates as needed). Use an array for a known, fixed count; use a list when the size changes.

Q2 When would you use a Dictionary instead of a List? Junior

Use a Dictionary when you look things up by a key rather than a position — like finding a person's age by their name. Lookups, adds, and removes by key are near-instant (O(1) on average). A List is for ordered items accessed by index, and finding a value in it requires scanning (O(n)).

Q3 What is the difference between a Queue and a Stack? Junior

A Queue is first-in-first-out (FIFO) — items come out in the order they went in, like a line at a shop (Enqueue/Dequeue). A Stack is last-in-first-out (LIFO) — the most recent item comes out first, like a stack of plates (Push/Pop).

Q4 What are generics and why are they useful? Junior

Generics let you write a class or method with a type placeholder (T) that's filled in when used — so one List<T> works for ints, strings, or any type. They give type safety (mistakes caught at compile time), avoid boxing (better performance for value types), and enable reuse (no separate list type per element type).

Q5 What does the <T> in List<T> mean? Junior

It's a type parameter — a placeholder for whatever type you'll store. When you write List<int>, T becomes int; List<string> makes T a string. The same definition adapts to any type while staying strongly typed.

Q6 What does yield return do, in simple terms? Junior

It returns one item from a sequence and then pauses the method, remembering where it stopped. The next time an item is requested, the method resumes right after the yield return. This lets you produce a sequence one item at a time instead of building a whole list at once.

Q7 What is IEnumerable<T>? Junior

It's the basic interface meaning "something you can iterate over one item at a time." Lists, arrays, dictionaries, and yield sequences all implement it, which is why foreach and LINQ work on all of them. If a method only needs to loop over data, accept IEnumerable<T> so it works with any of them.

🟠 Mid-level — going deeper (2–5 years)

Q8 Why are generic collections preferred over ArrayList and Hashtable? Mid

The non-generic collections store everything as object, which causes boxing of value types (heap allocations + GC pressure) and gives no compile-time type safety — you can accidentally mix types and only discover it via a runtime InvalidCastException. Generic collections (List<T>, Dictionary<K,V>) store the real type directly, so they're faster and catch type errors at compile time. Generics were introduced largely to fix exactly this.

Q9 Explain deferred (lazy) execution with yield. Mid

Calling an iterator method doesn't run its body — it returns an object that runs the code only when you start enumerating (a foreach or a LINQ materializer like ToList). Each requested item runs the method just far enough to hit the next yield return, then pauses. This means work and side effects happen at iteration time, not call time, and unconsumed items are never computed.

Q10 What is the "multiple enumeration" problem? Mid

Because an IEnumerable is lazy, iterating it again re-runs the whole sequence from the start. So calling, say, .Count() and then .ToList() on the same deferred query executes the underlying work twice — wasteful, and possibly inconsistent if the source changed between passes. The fix is to materialize once with .ToList() or .ToArray() and reuse that.

Q11 What are generic constraints and why use them? Mid

Constraints (the where clause) restrict what types T can be, which unlocks operations on T. Without a constraint the compiler treats T as object, so you can't call type-specific members. With where T : IComparable<T> you can call CompareTo; with where T : new() you can write new T(); with a base-class or interface constraint you can use that type's members. They make generic code both capable and type-safe.

Q12 What's the difference between IEnumerable and IEnumerator? Mid

IEnumerable<T> represents a sequence you can iterate — it exposes GetEnumerator(). IEnumerator<T> is the cursor that does the walking, with MoveNext() (advance) and Current (the item there). foreach calls GetEnumerator() once to get an enumerator, then loops MoveNext()/Current. A yield method's machinery implements the enumerator for you.

Q13 When should you use yield instead of building and returning a List? Mid

Use yield when the sequence is large or infinite, when the consumer may not need all items (so you avoid computing them), or when you want to stream data (e.g. reading a huge file line by line) without holding it all in memory. Build and return a concrete List when the data is small, when you'll enumerate it multiple times, or when you need a stable snapshot that won't change or re-run.

Q14 What's the difference between Dictionary and HashSet internally? Mid

Both use hashing for fast O(1)-average lookups. A HashSet<T> stores just keys (unique values), answering "is this present?" and de-duplicating. A Dictionary<K,V> stores key→value pairs, mapping each unique key to an associated value. Think of a HashSet as a Dictionary with only keys and no values.

🔴 Senior — synthesis & judgment (5–7 years)

Q15 How does the compiler implement a yield iterator? Senior

It generates a hidden state-machine class implementing IEnumerator<T> (and IEnumerable<T>). The method's local variables become fields on that class, and a state field tracks which yield the method last paused at. Each MoveNext() call jumps (via a switch on the state) to the resume point, runs to the next yield return (setting Current and saving state) or to the end / yield break (returning false). That's how a linear-looking method "pauses and resumes."

Q16 What are the rules and gotchas around resource disposal in iterators? Senior

Because execution is deferred, a resource opened before the first yield stays open across pauses and is disposed when the iterator completes or when the enumerator is disposed (which foreach does automatically, even on early break). The classic bug is opening a resource in a using whose block exits before deferred enumeration occurs, causing ObjectDisposedException. Use try/finally inside the iterator (or using within it, which compiles to try/finally) so cleanup is tied to the iterator's lifetime, or use framework lazy readers like File.ReadLines.

Q17 How do generics avoid the boxing that ArrayList suffers, at the runtime level? Senior

The CLR specializes generic code per value-type argument: List<int> stores ints in an int[] backing array directly, with no object wrapper — so no heap allocation per element and no GC churn. (Reference-type instantiations share one implementation since references are uniform.) ArrayList, by contrast, has an object[], so every value type is boxed on add and unboxed on read. This specialization is why generic collections are both type-safe and faster for value types.

Q18 Why does Dictionary lookup degrade, and how do you keep it O(1)? Senior

Dictionaries bucket entries by hash code. If many keys produce the same hash (poor GetHashCode) they collide into one bucket, turning lookups into a linear scan within it — O(n) in the worst case. Keep it fast by implementing a good, well-distributed GetHashCode (and a consistent Equals) for custom key types, never mutating a key's hash-relevant fields while it's in the dictionary, and pre-sizing capacity to reduce rehashing. Average O(1) depends on a healthy hash distribution.

Q19 What are covariance and contravariance in generic interfaces? Senior

They allow safe assignment between generic types with related type arguments. Covariance (out T, e.g. IEnumerable<out T>) lets you treat IEnumerable<Dog> as IEnumerable<Animal> — valid because you only read T out. Contravariance (in T, e.g. IComparer<in T>) lets an IComparer<Animal> be used where IComparer<Dog> is expected — valid because T only flows in. They apply to interfaces and delegates (not classes), and only to reference-type arguments.

Q20 You need to process a 10 GB file and aggregate matching lines. How do collections, generics, and yield combine? Senior

Stream it: read lazily with an iterator (or File.ReadLines, which yields lines one at a time) so you never load 10 GB into memory. Chain lazy LINQ operators (Where, Select) — all deferred — to filter and transform line by line, and aggregate into a small generic collection (e.g. a Dictionary<string,int> of counts). Generics keep the aggregation type-safe and boxing-free; yield/deferred execution keeps memory flat; the collection holds only the compact result. Materialize only the final summary, not the stream.

Q21 When would a LinkedList<T> beat a List<T>, and why is it rarely used? Senior

LinkedList<T> gives O(1) insertion/removal at a known node (no shifting of elements), which can beat a List<T>'s O(n) middle insert when you frequently splice in the middle and already hold the node. In practice it's rarely worth it: it has no index access (O(n) to find a position), each node is a separate heap allocation with next/prev pointers (poor cache locality and more memory), and modern CPUs make List's contiguous-memory shifts very fast. Usually List<T> wins on real hardware despite the worse Big-O on paper.

Q22 How do the ordering and exclusivity rules for generic constraints work? Senior

At most one "primary" constraint (class, struct, notnull, unmanaged, or default) may appear, and it must be first. Then an optional base-class constraint, then any number of interface constraints, and finally new() if present, which must be last. Mutually exclusive combos are rejected: struct with new() (value types always have a parameterless constructor, so it's redundant), unmanaged with struct/new() (it already implies struct), and a specific base class with struct/class/unmanaged. Getting the order wrong is a compile error (CS0449).
