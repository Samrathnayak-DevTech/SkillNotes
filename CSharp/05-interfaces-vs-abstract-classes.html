Interfaces vs Abstract Classes ⚖️

Both let you program against an abstraction instead of a concrete type, and both support polymorphism — so beginners (and interviewers) constantly ask which to use. The honest answer is "it depends," but the dependency is learnable. This page lays out exactly what each can and can't do, how modern C# (default methods, static abstract members) blurred the line, and a clear decision guide so you'll never have to guess. No prior coding background assumed.

📜 Interface = contract 📋 Abstract = partial base 🧬 1 base, many interfaces 🆕 Default methods 🧭 Decision guide 🧠 Interview Prep
01

Why It Matters

Two tools for the same goal — defining an abstraction that many types fulfill. Picking wrong leads to rigid or duplicated code.

An interface and an abstract class both let you say "here is a shape that several concrete types will take," and let you write code against that shape. So why have both? Because they make different promises: one is a pure contract, the other a partial implementation you build on. Choosing the right one shapes how flexible — and how maintainable — your design will be.

⚖️ Abstraction tools
📜 Interface
  • Pure contract
  • No state, no constructor
  • Implement many
  • Works for structs
📋 Abstract class
  • Partial base class
  • State + constructors
  • Inherit only one
  • Shared code
🆕 The blur
  • Default methods (C# 8)
  • Static abstract (C# 11)
  • Lines now overlap
🧭 The choice
  • Shared code/state? → abstract
  • Many/unrelated types? → interface
  • Often: use both
🔑 One-line summary

Use an interface to describe what a type can do (a capability). Use an abstract class to provide what related types share (common code + state). When a type needs both, give it one abstract base and several interfaces.

🔌 Analogy

An interface is like a power socket standard: anything with the right plug works, no matter who made it — phones, lamps, laptops (unrelated devices, one contract). An abstract class is like a half-built house from a developer: the foundation and plumbing are already there (shared implementation), and you only finish the rooms — but you can only build on one foundation.

02

Interfaces

A contract: a list of members an implementing type must provide. It says what, never (classically) how.

An interface declares members — methods, properties, events, indexers — with no implementation (in the classic model). Any class or struct that implements it must supply every member. Members are public by default. A type can implement any number of interfaces.

C#
// the contract -- by convention interface names start with 'I'
interface IShape
{
    double Area();       // no body -- just the promise
    string Name { get; }
}

class Circle : IShape
{
    public double R { get; set; }
    public double Area() => Math.PI * R * R;   // must implement
    public string Name => "Circle";             // must implement
}

// program to the contract -- works for any IShape
IShape s = new Circle { R = 2 };
Console.WriteLine(s.Area());
📜

Contract only

Lists members without state. No instance fields, no constructors — it carries no data of its own.

➕

Multiple allowed

One type can implement many interfaces — C#'s way to mix in capabilities from several sources.

🧊

Works for structs

Structs can't inherit a class but can implement interfaces — often the only way to give them a shared contract.

✅ The .NET interfaces you already use

IDisposable (cleanup), IComparable (ordering), IEnumerable (iteration), IEquatable<T> (equality) — these are all interfaces precisely because unrelated types need those capabilities. A FileStream and a database connection share nothing but both are IDisposable.

03

Abstract Classes

A partial base class: some real code and state, plus blanks the children must fill. Can't be instantiated.

An abstract class is a base class you can't create with new. It mixes concrete members (real shared code, fields, constructors) with abstract members (no body, which subclasses must implement). It's "inheritance with required homework."

C#
abstract class Shape
{
    public string Name { get; }            // STATE -- interfaces can't hold this
    protected Shape(string name) => Name = name;  // CONSTRUCTOR -- interfaces can't have one

    public abstract double Area();         // blank -- subclasses must fill

    // SHARED concrete method that reuses the abstract one
    public void Describe() => Console.WriteLine($"{Name}: {Area():F2}");
}

class Circle : Shape
{
    public double R { get; }
    public Circle(double r) : base("Circle") => R = r;  // calls base constructor
    public override double Area() => Math.PI * R * R;
}

// var s = new Shape("x");  // ❌ can't instantiate an abstract class
🗄️

Holds state

Can have instance fields and properties with backing data — real per-object state the subclasses inherit.

🏗️

Has constructors

Can run setup logic and require subclasses to pass data up via : base(...).

♻️

Shares code

Provides ready-made method bodies all subclasses reuse — write once, inherit everywhere.

⚠️ The cost: single inheritance

A class can inherit from only one base class. Spend that single "inheritance slot" on an abstract class and you've used it up — the type can't also inherit a different base. This is the central trade-off against interfaces, of which you can have many.

Core Differences

The side-by-side that answers most interview questions. Read it twice.

🔑 The two differences that matter most

Strip away the details and it comes down to: (1) you can implement many interfaces but inherit only one abstract class, and (2) only an abstract class can hold state (fields) and have a constructor. Almost every "which should I use?" answer flows from these two facts.

05

Only Abstract Classes Can…

The unique powers that pull you toward an abstract class.

  • Hold instance state. Fields and full properties backed by data that every subclass inherits.
  • Have constructors. Run initialization logic and require subclasses to supply values via : base(...).
  • Use any access modifier on members — including protected members meant only for subclasses.
  • Share rich implementation with private helper methods and template-method patterns.
C#
abstract class Repository<T>
{
    private readonly List<T> _items = new();   // STATE: shared backing store
    protected int Count => _items.Count;          // PROTECTED: for subclasses only

    public void Add(T item)                      // SHARED concrete logic
    {
        Validate(item);     // calls the subclass's rule
        _items.Add(item);
    }

    protected abstract void Validate(T item);  // subclass fills this blank
}
✅ Reach for an abstract class when…

…your related types share real implementation and data, and you want to define a common workflow once (the template method pattern above). An interface can't store _items or run that shared Add logic — it has no state and (classically) no bodies to inherit.

06

Only Interfaces Can…

The unique powers that pull you toward an interface.

  • Be implemented in multiples. A type can take on many interfaces at once — capabilities mix freely.
  • Span unrelated types. Types with nothing else in common can share one contract.
  • Be implemented by structs, which can't inherit from a class.
  • Decouple contracts from implementations, the foundation of testing and dependency injection.
C#
interface IFlyable  { void Fly(); }
interface ISwimmable { void Swim(); }

// a Duck mixes in BOTH capabilities -- impossible with single inheritance
class Duck : IFlyable, ISwimmable
{
    public void Fly()  => Console.WriteLine("flap flap");
    public void Swim() => Console.WriteLine("paddle paddle");
}

// even a struct can implement an interface
struct Point : IComparable<Point>
{
    public int X;
    public int CompareTo(Point other) => X.CompareTo(other.X);
}
🔑 The "mix-in" superpower

Interfaces let unrelated capabilities combine on one type: a Duck can be IFlyable and ISwimmable; a FileStream can be IDisposable, IComparable, and more. No single base class could capture all those orthogonal abilities at once — this is why C# has both tools.

07

Default Methods (C# 8)

Interfaces can now carry implementation. Did this make abstract classes obsolete? No — but the line did blur.

Before C# 8, interfaces were pure contracts: signatures only. Since C# 8, an interface member can ship with a default implementation. Existing implementers inherit it for free; they only override it if they want different behavior. Interfaces can also now have static members, private helper members, and access modifiers.

C#
interface ILogger
{
    void Log(string message);                 // still abstract -- must implement

    // DEFAULT body: implementers get this unless they override it
    void LogError(string msg) => Log($"ERROR: {msg}");
}

class ConsoleLogger : ILogger
{
    public void Log(string m) => Console.WriteLine(m);
    // LogError inherited for free -- no need to write it
}
🔑 Why this feature exists

The main motivation is versioning: a library author can add a new member to a widely-used interface without breaking the thousands of classes that already implement it (they pick up the default). It is not meant as a general replacement for abstract classes.

⚠️ Why abstract classes still win for shared code

Default interface methods have real limits: you must call them through the interface type (not the concrete class), and interfaces still can't hold instance state — so a default method can't read or write per-object fields. When your shared logic needs state, an abstract class remains the right tool. The line blurred; it didn't disappear.

08

Static Abstract Members (C# 11)

Interfaces can now require static members — including operators — unlocking "generic math." Abstract classes can't do this.

Since C# 11 (.NET 7), an interface can declare static abstract (and static virtual) members that implementing types must provide. This enables compile-time polymorphism over static APIs and operators — the foundation of .NET's generic math, where one generic algorithm works across int, double, decimal, and your own numeric types.

C#
// 'TSelf' is the implementing type itself (self-referencing constraint)
interface IAddable<TSelf> where TSelf : IAddable<TSelf>
{
    static abstract TSelf Zero { get; }
    static abstract TSelf operator +(TSelf a, TSelf b);
}

// one generic method now works for ANY type implementing IAddable
static T Sum<T>(T[] items) where T : IAddable<T>
{
    T result = T.Zero;            // call a static member on the type parameter!
    foreach (T x in items) result += x;
    return result;
}
🔑 Real-world payoff

.NET 7+ ships INumber<T> and friends in System.Numerics, so a single Sum<T> or Average<T> works across all numeric types with no boxing and no per-type overloads. The .NET team used this to delete hundreds of lines of duplicate code. An abstract class cannot express "every implementer must provide this static operator" — this is interface-only territory.

✅ Note on access

Static abstract members aren't called through an instance or interface variable; you call them on the type parameter (T.Zero) inside generic code constrained by the interface. The actual implementation used is the one on the real type supplied as T.

09

Decision Guide

Walk the questions top to bottom. The first "yes" usually settles it.

❓Do the types share real implementation and state (fields)?Common backing data, a shared workflow, protected helpers?
📋Yes → Abstract classOnly it can hold state, run a constructor, and share concrete code with subclasses.
❓Do unrelated types need this capability, or does one type need several capabilities?Or must a struct participate?
📜Yes → InterfaceMany per type, spans unrelated types, works for structs, decouples for testing.
❓Need static operators / generic-math style contracts?
📜Interface with static abstract members (C# 11)
✅ Rules of thumb

Default to an interface — it's more flexible and keeps the single-inheritance slot free. Upgrade to an abstract class when you find yourself wanting to share real code and state among related types. If you need both shared code and multiple capabilities, do both: one abstract base plus interfaces (next section).

🔑 Microsoft's own guidance, summarized

"Use an interface if you expect unrelated types to implement it, or to specify behavior from multiple sources. Use an abstract class if you want to share code among several closely related types, or need to declare non-public members or fields." The two are complementary, not competitors.

10

Combining Both

The real-world answer is usually "both." One abstract base for the shared identity, interfaces for assorted capabilities.

A class can inherit one abstract class and implement many interfaces at once. This is the common production shape: the base supplies shared state and code; the interfaces declare the extra abilities.

Bird (abstract)
IFlyable
ISwimmable
Duck : Bird, IFlyable, ISwimmable
C#
abstract class Bird
{
    public string Name { get; set; }   // shared state from the base
    public void Eat() => Console.WriteLine($"{Name} eats");  // shared code
}

interface IFlyable  { void Fly(); }
interface ISwimmable { void Swim(); }

// one base class (identity + shared code) + many interfaces (capabilities)
class Duck : Bird, IFlyable, ISwimmable
{
    public void Fly()  => Console.WriteLine($"{Name} flies");
    public void Swim() => Console.WriteLine($"{Name} swims");
}
✅ The mental model

Abstract base = "what this thing fundamentally is and shares." Interfaces = "what this thing can do." A Duck is a Bird (base) that can fly and swim (interfaces). That phrasing usually tells you instantly which member goes where.

11

Pitfalls

The traps that bite people choosing or using these.

🔒

Burning your one base

Putting shared code in an abstract base when an interface would do "spends" the single inheritance slot — now the type can't inherit anything else.

🧩

Fat interfaces

Cramming unrelated members into one interface forces implementers to write methods they don't need. Prefer small, focused interfaces (Interface Segregation).

👻

Default-method surprise

Forgetting that a default interface method is only reachable through the interface type — calling it on the concrete class won't compile.

💥

Breaking changes

Adding an abstract member to a published interface breaks every implementer. Add a default member instead, or a new interface.

C#
interface ILogger { void Log(string m); void Warn(string m) => Log($"WARN: {m}"); }

var logger = new ConsoleLogger();
// logger.Warn("hi");        // ❌ won't compile -- Warn is a default interface member
((ILogger)logger).Warn("hi");   // ✅ must call through the interface type
⚠️ Don't over-abstract

Not everything needs an interface or abstract base. Introducing one for a type that will only ever have a single implementation adds indirection with no payoff. Add the abstraction when you actually have (or clearly foresee) multiple implementations or a need to decouple for testing.

Quick Recap

One sentence per idea. If you can explain each card aloud, you can defend any choice in a code review.

📜 Interface

A pure contract of members a type promises to provide; no state, no constructor; implement many.

📋 Abstract class

A non-instantiable partial base with shared code, state, and constructors; inherit only one.

1️⃣ vs ➕

One abstract base per type, but any number of interfaces — the single biggest practical difference.

🗄️ State

Only abstract classes hold instance fields and run constructors; interfaces carry no per-object data.

🧊 Structs

Structs can't inherit a class but can implement interfaces — sometimes the only option.

🆕 Default methods

Since C# 8 interfaces can carry method bodies (for versioning); callable only via the interface type.

➗ Static abstract

Since C# 11 interfaces can require static members/operators — powers generic math; abstract classes can't.

🧭 Choosing

Shared code + state + one hierarchy → abstract class; capability across many/unrelated types → interface.

🤝 Combine

The common answer: one abstract base for identity + several interfaces for capabilities.

🪶 Don't over-abstract

Skip the abstraction entirely if there's only ever one implementation and no need to decouple.

The cheat sheet 📊

If you need…Use
Shared implementation and state across related types📋 Abstract class
A capability across unrelated types📜 Interface
One type to support several independent capabilities📜 Interfaces (multiple)
A struct to honor a contract📜 Interface
Static operators / generic math contracts📜 Interface (static abstract, C# 11)
To add a member to a shipped contract without breaking implementers📜 Interface default method (C# 8)
Both shared code and multiple capabilities🤝 Abstract base + interfaces
13

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is the difference between an interface and an abstract class? Junior

An interface is a pure contract — a list of members a type must provide, with no state and (classically) no implementation; a type can implement many. An abstract class is a partial base class that can include real code, fields, and a constructor, but a type can inherit only one. Interface = "can-do" capability; abstract class = "is-a" with shared code.

Q2 Can you create an instance of an interface or an abstract class? Junior

No, neither can be instantiated directly with new. Both exist to be implemented or inherited by concrete types. You create an instance of a concrete class and can then refer to it through the interface or abstract base type.

Q3 How many interfaces and abstract classes can a single class use? Junior

A class can inherit from one abstract class (single inheritance) but implement any number of interfaces. It can also do both at once: class Duck : Bird, IFlyable, ISwimmable.

Q4 Can an interface have fields? Junior

No — interfaces can't have instance fields and hold no per-object state. They can declare properties (which are accessor methods, not data), and since C# 8 they can have static fields, but not instance fields. Abstract classes can have instance fields.

Q5 Can a struct use an interface or an abstract class? Junior

A struct can implement an interface but cannot inherit from an abstract class — structs don't support inheritance. So if you need a struct to honor a shared contract, an interface is your only option.

Q6 Give a real example of when you'd use each. Junior

Interface: IDisposable — unrelated types (files, connections) all need a Dispose() capability. Abstract class: a Shape base with a stored Name and a shared Describe() method, where Circle and Square just fill in Area().

Q7 Why does C# have both instead of just one? Junior

Because they solve different problems. Interfaces let a type mix in many capabilities and let unrelated types share a contract (and work with structs). Abstract classes let related types share real code and state through single inheritance. Neither fully replaces the other, so C# provides both.

🟠 Mid-level — going deeper (2–5 years)

Q8 What can an abstract class do that an interface cannot? Mid

Hold instance state (fields), have constructors, use any access modifier (including protected members for subclasses), and provide rich shared implementation including private helpers. Even with C# 8 default methods, interfaces still can't store per-object state, which is the key thing abstract classes uniquely offer.

Q9 What are default interface methods and why were they added? Mid

Since C# 8, an interface member can have a default body that implementers inherit unless they override it. The primary motivation is versioning: a library can add a member to a popular interface without breaking the many existing classes that implement it. They're a compatibility tool, not a general replacement for abstract classes — and you must call them through the interface type.

Q10 Did default interface methods make abstract classes obsolete? Mid

No. Interfaces still can't hold instance state or have constructors, so default methods can't read or write per-object fields the way an abstract class's shared methods can. They also can't define protected instance members in a usable inheritance sense, and default members are only callable through the interface type. For shared code that needs state, abstract classes remain the right tool. The line blurred; it didn't vanish.

Q11 When should you prefer an interface over an abstract class? Mid

When the abstraction is a capability that unrelated types should support, when a type needs several independent capabilities at once, when structs must participate, or when you want maximum flexibility and to keep the single inheritance slot free for something else. Interfaces also decouple consumers from implementations, which is ideal for testing and dependency injection.

Q12 Can an interface inherit from another interface? Can it require properties or events? Mid

Yes to both. An interface can extend one or more other interfaces, and an implementer must satisfy all of them. Interfaces can declare methods, properties, events, and indexers (not instance fields). Members are public by default, and since C# 8 you can also add static, private, and default members.

Q13 How do interfaces support unit testing and dependency injection? Mid

By depending on an interface rather than a concrete class, you can swap implementations freely. In tests you inject a mock or fake that implements the interface, isolating the unit under test from real databases, networks, or clocks. In production, a composition root supplies the real implementation. This decoupling — "program to an interface" — is the backbone of testable, loosely-coupled design.

Q14 You need to add a new method to an interface used by 50 classes. How do you avoid breaking them? Mid

Add it as a default interface method (C# 8) so all 50 classes inherit a working implementation and keep compiling; classes that need custom behavior override it. Alternatively, declare a new interface that extends the old one and have only the relevant types implement it. Adding a plain abstract member would force all 50 classes to implement it immediately — a breaking change.

🔴 Senior — synthesis & judgment (5–7 years)

Q15 What are static abstract interface members and what problem do they solve? Senior

Introduced in C# 11 (.NET 7), they let an interface require static members — including operators — that implementing types must provide, and let generic code call them on the type parameter (T.Zero). They solve the long-standing inability to abstract over static APIs and operators, enabling generic math: one generic algorithm works over int, double, custom numerics, etc., resolved at compile time with no boxing. An abstract class cannot express "every implementer must define this static operator," so this is interface-only.

Q16 Default interface methods reintroduce a form of multiple inheritance. What problems does that create? Senior

If a class implements two interfaces that each supply a conflicting default for the same member, there's a diamond-style ambiguity the implementer must resolve explicitly. Default members are also only callable via the interface type, so the same object behaves differently depending on the static type of the reference — echoing the override-vs-new surprise. And because interfaces lack instance state, defaults can't build truly stateful behavior. Net: use them for compatibility, not as a general code-sharing mechanism; prefer abstract classes when shared stateful implementation is the goal.

Q17 How does the Interface Segregation Principle inform your choice? Senior

The ISP (the "I" in SOLID) says clients shouldn't be forced to depend on members they don't use. It pushes you toward small, focused interfaces (e.g. IReadable, IWritable) rather than one fat interface, so implementers only take on what's relevant and consumers depend on the minimum. Abstract classes don't segregate as cleanly because a subclass inherits the whole base. ISP is a strong argument for preferring multiple narrow interfaces for capabilities, reserving abstract classes for genuine shared implementation.

Q18 "Favor composition over inheritance." How does that affect the interface-vs-abstract-class decision? Senior

It cautions against deep abstract-class hierarchies, which create tight coupling and fragile base classes. The modern lean is: depend on interfaces for contracts and compose behavior by injecting collaborators (objects that implement those interfaces) rather than inheriting it. Abstract classes still earn their place for genuinely shared implementation among closely related types, but you reach for them deliberately, not by default. Interfaces + composition keep designs flexible and testable; inheritance is the heavier commitment.

Q19 When is explicit interface implementation the right call, and how does it interact with this choice? Senior

Use explicit implementation (void IFoo.Method()) when a type implements two interfaces with the same member signature and each needs its own behavior, or when you want to keep an interface member off the class's public surface (accessible only by casting to the interface). It's unique to interfaces — abstract classes don't have this mechanism. It's also handy to implement a contract a type must satisfy without cluttering its everyday API. The trade-off is reduced discoverability, since the member isn't visible on the concrete type.

Q20 A type needs shared stateful logic AND must work polymorphically across an unrelated hierarchy. How do you design it? Senior

Split the concerns. Define an interface for the cross-cutting capability so unrelated types (and structs) can participate polymorphically. Provide the shared stateful logic in an abstract base class that implements that interface, so the related types inheriting it get the implementation for free — while unrelated types still implement the interface their own way. This "interface for the contract, abstract base for the common implementation" pairing is exactly how much of the .NET BCL is structured (e.g. IList<T> alongside helper base classes).

Q21 Are there performance differences between calling through an interface vs an abstract base class? Senior

Both involve indirection. Virtual calls through an abstract base go through the type's v-table. Interface dispatch historically used an interface map / dispatch stub, which can be marginally costlier than a class v-table call, though the runtime optimizes common cases (and devirtualization can eliminate it entirely when the concrete type is known). In the vast majority of code the difference is negligible and shouldn't drive the design decision — correctness and flexibility should. For hot paths, the static-abstract/generic approach can avoid virtual dispatch altogether by resolving at compile time. Measure before optimizing.

Q22 How would you evolve a public API that currently exposes an abstract base class but now needs to support unrelated implementers? Senior

Introduce an interface capturing the public contract and have the existing abstract base implement it, so current subclasses keep working unchanged while new, unrelated types can implement just the interface. Change public API signatures to accept the interface type where feasible. This preserves backward compatibility (existing inheritors are unaffected) while opening the contract to types that can't or shouldn't inherit the base — a common, low-risk way to loosen an over-committed inheritance-based API.
