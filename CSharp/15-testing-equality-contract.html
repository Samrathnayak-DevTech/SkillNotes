Testing & the Equality Contract ✅

Two practical pillars of professional C#. Unit testing proves your code does what you claim — using a framework (xUnit/NUnit/MSTest), the Arrange-Act-Assert pattern, and test doubles to isolate the unit. The equality contract is the set of rules your Equals and GetHashCode must obey so that dictionaries, sets, LINQ, and your own test assertions behave correctly. They meet in the middle: assertions rely on equality, and the contract is something you should test. Builds on the DI, collections, and records topics. No prior coding background assumed.

01

Overview

Testing proves behavior; the equality contract defines correct equality. Each reinforces the other.

Unit testing is writing small, automated checks that a piece of code behaves as expected, so you can change code confidently. The equality contract is the agreement every type makes when it defines equality — break it and hash-based collections silently misbehave. The link: test assertions use equality under the hood, and a well-written test suite is exactly how you verify the contract holds.

🔑 Why they belong together

Assert.Equal(expected, actual) calls Equals — so if your type's equality is wrong, your tests can pass or fail for the wrong reasons. And the equality contract has clear, checkable rules, making it a textbook example of what to unit-test. Knowing both is core to writing dependable C#.

02

Why Test & FIRST

Tests are a safety net that lets you change code without fear — if they're fast, isolated, and trustworthy.

A unit test checks one small unit of behavior in isolation. Together, a test suite documents intent, catches regressions, and enables fearless refactoring. An integration test, by contrast, exercises several components together (and may touch a database or network). Good unit tests follow the FIRST principles:

⚡

Fast

Run in milliseconds so you run them constantly. Slow tests get skipped — and skipped tests don't help.

🧩

Independent

No test depends on another or on shared state/order. Each can run alone, in any sequence.

🔁

Repeatable

Same result every time, anywhere — no reliance on clock, network, or random data.

✔️

Self-validating

The test decides pass/fail via assertions — no manual inspection of output.

⏱️

Timely

Written alongside (or before) the code, while the design is fresh — not bolted on much later.

⚠️ Unit vs integration — keep them apart

If a test touches a real database, file system, or network, it's an integration test, not a unit test — it'll be slower and more brittle. Keep unit tests free of infrastructure (use test doubles, section 04), and keep integration tests in a separate project so your fast feedback loop stays fast.

03

AAA & Frameworks

Every good test has three parts, and .NET has three main frameworks to write them in.

A
Arrange

Set up the objects, inputs, and test doubles the test needs.

A
Act

Invoke the one method under test — a single action per test.

A
Assert

Verify the outcome matches what you expected.

The big three frameworks are xUnit (modern, the de-facto choice for .NET Core), NUnit (mature, rich feature set), and MSTest (built into Visual Studio). They differ mostly in attribute names. In xUnit, [Fact] is a test that's always true; [Theory] + [InlineData] runs the same test with different inputs.

C# · xUnit
public class CalculatorTests
{
    // [Fact] -- a single, parameterless test. Name: Method_Scenario_Expected
    [Fact]
    public void Add_TwoPositives_ReturnsSum()
    {
        // Arrange
        var calc = new Calculator();
        // Act
        int result = calc.Add(2, 3);
        // Assert
        Assert.Equal(5, result);
    }

    // [Theory] -- same test body, many inputs (data-driven)
    [Theory]
    [InlineData(2, 3, 5)]
    [InlineData(-1, 1, 0)]
    [InlineData(0, 0, 0)]
    public void Add_VariousInputs_ReturnsSum(int a, int b, int expected)
    {
        Assert.Equal(expected, new Calculator().Add(a, b));
    }
}
🔑 Naming & one act per test

The convention MethodName_Scenario_ExpectedBehavior makes a failing test self-describing in the test runner. Keep one "Act" (and ideally one concern) per test, so a failure points to exactly one thing. Common assertions: Assert.Equal, True/False, Null/NotNull, Throws<T>, Contains, IsType<T>.

✅ xUnit setup vs NUnit/MSTest

xUnit deliberately dropped [SetUp]/[TearDown] attributes: use the test class's constructor for setup and implement IDisposable for teardown (tying back to the IDisposable topic). xUnit also creates a new instance of the test class per test, which enforces independence by default.

04

Test Doubles

To test a unit in isolation, you replace its real dependencies with stand-ins. "Test double" is the umbrella term.

A class under test usually depends on others (a repository, a clock, an email sender). To test it in isolation, you swap those for test doubles — controllable stand-ins. The terms have precise meanings (from Meszaros and Fowler):

DummyPassed but never used — just fills a parameter.
StubReturns canned answers to calls (provides data).
MockA stub you also assert against — verifies it was called as expected.
FakeA working lightweight implementation (e.g. in-memory DB).
C# · Moq
// STUB: set up a return value so the unit can run (provides data)
var clock = new Mock<IClock>();
clock.Setup(c => c.Now()).Returns(new DateTime(2024, 1, 1));

var repo = new Mock<IOrderRepository>();
var sut = new OrderService(repo.Object, clock.Object);   // inject the doubles (DI!)

// Act
sut.Place(new Cart());

// MOCK: assert an interaction happened -- repo.Save was called once
repo.Verify(r => r.Save(It.IsAny<Order>()), Times.Once);
🔑 Mock vs stub — the key distinction

A stub just feeds the unit data so it can run; you don't check the stub itself. A mock additionally verifies an interaction — you Assert/Verify that a method was (or wasn't) called, with what arguments, how many times. Rule of thumb: stub queries (state you read), mock commands (actions with side effects you want to confirm happened).

✅ This is why DI matters

Test doubles are only injectable because the class takes its dependencies through its constructor (the dependency injection topic). A class that news up its own collaborators can't be isolated for testing. So DI is what makes code testable — and libraries like Moq, NSubstitute, and FakeItEasy generate the doubles for you from an interface.

05

Best Practices & TDD

Habits that keep a test suite valuable, plus the test-first discipline.

🎯

One concern per test

Each test verifies a single behavior, so a failure points to exactly one cause. Avoid multiple unrelated asserts.

🚫

No logic in tests

Avoid if/loops in tests — they can hide bugs in the test itself. Prefer straight-line arrange/act/assert.

📛

Descriptive names

The name should state method, scenario, and expectation, so the runner reads like a spec.

🧱

Test behavior, not internals

Assert observable outcomes, not private implementation details, so tests survive refactoring.

🤖

Run in CI

Automate tests in the build pipeline so every change is validated. Track coverage to find gaps.

🔴

TDD (optional)

Red-green-refactor: write a failing test, make it pass minimally, then clean up — design emerges test-first.

🔴 Redwrite a test for behavior that doesn't exist yet — it fails
🟢 Greenwrite the simplest code that makes the test pass
🔵 Refactorclean up the code with the test as a safety net
✅ Coverage is a guide, not a goal

Code coverage shows which lines tests exercise — useful for spotting untested areas. But 100% coverage doesn't mean correct: you can execute a line without asserting anything meaningful about it. Aim for tests that capture behavior and edge cases, and treat coverage as a hint about gaps, not a target to game.

06

The Equality Contract ★

If you override Equals, you're making a promise. These rules are the promise — and .NET's collections depend on them.

When you define what "equal" means for a type, your Equals must satisfy five rules (it must behave like a mathematical equivalence relation), and GetHashCode must stay consistent with it:

RuleMeaning
Reflexivex.Equals(x) is always true
Symmetricx.Equals(y) equals y.Equals(x)
Transitiveif x.Equals(y) and y.Equals(z), then x.Equals(z)
Consistentrepeated calls give the same result if the objects aren't modified
Nullx.Equals(null) is false (for non-null x)

🔑 The hash-code invariant (the one everyone forgets)

If two objects are equal, they must return the same GetHashCode. The reverse isn't required — unequal objects may share a hash code (a "collision," which Equals then resolves). Break this rule and Dictionary/HashSet may fail to find a key that's actually there, because they look in the wrong bucket. This is why you must always override both methods together — overriding just Equals even produces compiler warning CS0659.

⚠️ Who relies on this?

Dictionary<K,V> and HashSet<T> use the hash code to pick a bucket, then Equals to confirm a match. LINQ's Distinct, Contains, Union, GroupBy, and Intersect rely on the same contract. Get it wrong and these silently return wrong results — no exception, just bad data.

07

== vs Equals

Three ways to compare — and they don't all mean the same thing.

ComparisonDefault behaviorNotes
==Reference equality for classes; value for built-insStatic, resolved at compile time; can be overloaded
EqualsReference equality (Object); value for structsVirtual, resolved at runtime; meant to be overridden
ReferenceEqualsAlways reference identityCan't be overridden — true only if same object
C#
var a = new StringBuilder("hi").ToString();
var b = new StringBuilder("hi").ToString();

a == b;                       // True  -- string overrides == for value equality
a.Equals(b);                  // True  -- value equality
Object.ReferenceEquals(a, b);  // False -- different objects in memory

// for your own class WITHOUT overrides, == is reference equality:
var p1 = new Point(1, 2);
var p2 = new Point(1, 2);
p1 == p2;        // False (unless you overload == and override Equals)
🔑 Keep == and Equals consistent

If you override Equals to give value semantics, you should usually also overload == and != so both behave the same — otherwise a.Equals(b) and a == b can disagree, which confuses everyone. string does this (that's why "hi" == "hi" is true), and records do it automatically (the records topic).

⚠️ == is static, Equals is virtual

A subtle trap: == is resolved at compile time based on the variable's declared type, while Equals is virtual and dispatched at runtime on the actual object. So through a base-type reference, == may not call your overload while Equals still calls your override. Keeping them consistent avoids surprises.

08

Implementing It

The standard recipe: override Equals and GetHashCode, implement IEquatable<T>, and overload ==.

C#
public sealed class Money : IEquatable<Money>
{
    public decimal Amount { get; }
    public string Currency { get; }   // immutable -> safe for hashing

    public Money(decimal amount, string currency)
        => (Amount, Currency) = (amount, currency);

    // strongly-typed Equals (IEquatable<T>) -- no boxing, used by collections
    public bool Equals(Money? other) =>
        other is not null && Amount == other.Amount && Currency == other.Currency;

    // object.Equals delegates to the typed one
    public override bool Equals(object? obj) => Equals(obj as Money);

    // hash code from the SAME fields used in Equals; HashCode.Combine is the modern way
    public override int GetHashCode() => HashCode.Combine(Amount, Currency);

    // overload == / != to match Equals
    public static bool operator ==(Money? a, Money? b) => Equals(a, b);
    public static bool operator !=(Money? a, Money? b) => !Equals(a, b);
}
🔑 The checklist

(1) Implement IEquatable<T>.Equals(T) — typed, no boxing for value types. (2) Override object.Equals(object) to delegate to it. (3) Override GetHashCode from the same fields, ideally via HashCode.Combine(...). (4) Overload ==/!=. The analyzers CA1066/CA1067 nudge you to keep IEquatable and the overrides in sync.

✅ Or just use a record

For pure data, a record (from the modern-C# topic) generates this entire recipe — value Equals, a contract-correct GetHashCode, and ==/!= — from a single line. Hand-write the recipe only when you need custom equality semantics (e.g. equality by an ID field, ignoring other properties) that a record's "compare every member" default doesn't give you.

09

Pitfalls

The ways equality goes subtly, silently wrong.

C# · the mutable-key bug
class BadKey { public string Id { get; set; }
    public override int GetHashCode() => Id.GetHashCode(); /* ...Equals on Id... */ }

var set = new HashSet<BadKey>();
var k = new BadKey { Id = "A" };
set.Add(k);
k.Id = "B";            // 💥 hash code changed AFTER insertion
set.Contains(k);      // False! -- it's in the wrong bucket now, effectively lost
🔓

Mutable hash fields

If a field used in GetHashCode changes after the object is in a set/dictionary, the key is "lost." Hash on immutable fields only.

↔️

Inheritance breaks symmetry

If a base and derived type can compare equal one way but not the other, symmetry breaks. Use GetType() checks or seal equality.

🐌

Struct default equality

ValueType.Equals/GetHashCode work via reflection — correct but slow. Override them on any struct used as a key or in hot paths.

⚖️

== / Equals disagree

Overriding one but not the other lets a == b and a.Equals(b) differ — a confusing inconsistency.

⚠️ The inheritance + symmetry trap

Suppose Base compares by one field and Derived adds another. base.Equals(derived) might say "true" (matching field) while derived.Equals(base) says "false" (missing field) — breaking symmetry. Two common fixes: compare exact runtime types with GetType() != obj.GetType() (so base and derived are never equal), or only implement equality on sealed classes.

10

Testing the Contract

The two topics meet: the equality contract is precisely the kind of well-defined behavior unit tests exist to verify.

Because the contract is a set of concrete rules, it's an ideal target for tests. A small suite locks in that your type's equality behaves correctly — and keeps behaving correctly as the code evolves.

C# · xUnit
public class MoneyEqualityTests
{
    [Fact]
    public void Equals_SameValues_AreEqual()         // value equality
    {
        var a = new Money(10, "USD");
        var b = new Money(10, "USD");
        Assert.Equal(a, b);          // uses Equals
        Assert.True(a == b);          // == matches Equals
    }

    [Fact]
    public void Equal_Objects_ShareHashCode()       // THE invariant
    {
        var a = new Money(10, "USD");
        var b = new Money(10, "USD");
        Assert.Equal(a.GetHashCode(), b.GetHashCode());
    }

    [Fact]
    public void Equals_IsSymmetric()                  // symmetry
    {
        var a = new Money(10, "USD");
        var b = new Money(10, "USD");
        Assert.Equal(a.Equals(b), b.Equals(a));
    }

    [Fact]
    public void Equals_Null_IsFalse() =>            // null rule
        Assert.False(new Money(10, "USD").Equals(null));

    [Fact]
    public void WorksAsDictionaryKey()                // the real-world payoff
    {
        var dict = new Dictionary<Money, string> { [new(10, "USD")] = "ten" };
        Assert.True(dict.ContainsKey(new Money(10, "USD")));   // found via hash + Equals
    }
}
✅ The full-circle payoff

That last test is the point of the whole page: a correct equality contract means a value works as a dictionary key — and a unit test proves it. The Assert.Equal(a, b) calls rely on Equals; the dictionary relies on the hash invariant; and the tests document and guard the contract. Testing and equality, working together.

12

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is a unit test, and how does it differ from an integration test? Junior

A unit test checks one small piece of behavior in isolation, with no external dependencies, so it runs fast and pinpoints failures. An integration test exercises several components working together — and may touch a real database, file system, or network — so it's slower and broader. You keep unit tests fast and numerous, and use a smaller set of integration tests for the seams between components.

Q2 What is the AAA pattern? Junior

Arrange-Act-Assert: the three parts of a well-structured test. Arrange sets up the objects, inputs, and any test doubles. Act calls the one method under test. Assert verifies the result matches what you expected. It keeps tests readable and makes the single thing being tested obvious.

Q3 What's the difference between [Fact] and [Theory] in xUnit? Junior

[Fact] marks a test with no parameters that should always hold. [Theory] marks a data-driven test that runs the same body with multiple inputs, supplied by [InlineData(...)] (or other data sources). Use a Theory instead of copy-pasting near-identical Facts that differ only by input values.

Q4 What does it mean for two objects to be "equal"? Junior

It depends on the type. By default, reference types use reference equality — two variables are equal only if they point to the same object. Value equality means two objects are equal if their contents match, even if they're different objects in memory. string and records use value equality; a plain class uses reference equality unless you override Equals.

Q5 Why must you override GetHashCode when you override Equals? Junior

Because hash-based collections (Dictionary, HashSet) use the hash code to decide where to store and look for an object. The rule is: equal objects must have equal hash codes. If you change what "equal" means via Equals but leave the default GetHashCode, two equal objects can get different hash codes, so the collection looks in the wrong place and fails to find them. The compiler even warns you (CS0659).

Q6 What is a mock? Junior

A mock is a stand-in for a real dependency that you use in tests to both supply controlled behavior and verify how it was used — for example, confirming that a method was called once with certain arguments. Libraries like Moq create mocks from an interface, so you can test a class in isolation without its real collaborators (like a database).

🟠 Mid-level — going deeper (2–5 years)

Q7 What's the difference between a stub and a mock? Mid

A stub provides canned data so the code under test can run — you don't check the stub itself. A mock goes further: you assert against it, verifying that an expected interaction happened (a method was called, with given arguments, a certain number of times). Rule of thumb: stub the queries (state you read), mock the commands (side-effecting actions you want to confirm occurred). Both are "test doubles," along with dummies and fakes.

Q8 State the equality contract. Mid

Five rules for Equals: reflexive (x.Equals(x) true), symmetric (x.Equals(y) == y.Equals(x)), transitive (if x=y and y=z then x=z), consistent (same result on repeated calls if unchanged), and x.Equals(null) is false. Plus the GetHashCode invariant: equal objects must return equal hash codes (the converse isn't required — collisions are allowed).

Q9 Why should hash codes be based on immutable fields? Mid

Because hash-based collections compute an object's hash code when it's inserted and use it to choose a bucket. If a field that feeds GetHashCode changes after insertion, the object's hash code changes, but it's still sitting in the old bucket — so Contains/lookup computes the new hash, looks in the new bucket, and doesn't find it. The key is effectively lost. Hashing only on immutable data avoids this.

Q10 What is IEquatable<T> and why implement it? Mid

IEquatable<T> provides a strongly-typed Equals(T) method. For value types it avoids boxing (the untyped Equals(object) would box the struct), and it lets collections call the typed comparison directly, which is faster. When you implement it you should also override object.Equals and GetHashCode to stay consistent (analyzers CA1066/CA1067 enforce this), typically by having object.Equals delegate to the typed version.

Q11 What's the difference between == and Equals? Mid

== is a static operator resolved at compile time from the declared types; by default for reference types it's reference equality (unless overloaded). Equals is a virtual method dispatched at runtime on the actual object, meant to be overridden for value semantics. string overloads == so it matches its value Equals. Best practice: if you override Equals, also overload ==/!= so they agree.

Q12 Why does DI make code more testable? Mid

Because dependencies arrive through the constructor as interfaces, a test can inject test doubles (stubs/mocks) instead of the real database, clock, or service — isolating the class so you test only its logic. A class that creates its own collaborators with new can't be isolated, so you'd be forced into slow, brittle integration tests. So the dependency-injection pattern is what unlocks fast, focused unit testing.

Q13 What happens if two equal objects return different hash codes? Mid

You violate the core invariant, and hash-based collections break silently. A Dictionary or HashSet stores the object in the bucket for its hash code; if an equal object hashes differently, a lookup checks a different bucket and reports "not found" even though an equal key is present. LINQ's Distinct, GroupBy, Union, etc. misbehave too. No exception is thrown — you just get wrong results, which is why this bug is so dangerous.

🔴 Senior — synthesis & judgment (5–7 years)

Q14 How does inheritance threaten the symmetry of Equals, and how do you handle it? Senior

If a base type compares by some fields and a derived type adds more, you can get asymmetry: base.Equals(derived) may return true (the shared fields match) while derived.Equals(base) returns false (the base lacks the extra field), violating symmetry — which corrupts collections relying on the contract. Two robust fixes: (1) compare exact runtime types with GetType() != obj.GetType() so a base and derived instance are never considered equal; or (2) only define value equality on sealed classes, removing the inheritance dimension entirely. Microsoft recommends implementing IEquatable on sealed types for exactly this reason.

Q15 Why are value types' default Equals and GetHashCode problematic, and when does it matter? Senior

A struct inherits ValueType.Equals/GetHashCode, which compare fields via reflection — correct but significantly slower than a hand-written comparison, and historically the default GetHashCode could derive from only the first field (poor distribution → collisions). For a struct used as a Dictionary/HashSet key or in a tight loop, that reflection cost and weak hashing hurt. The fix is to override both with a typed implementation (and implement IEquatable<T> to avoid boxing). For a struct that's never compared or hashed in hot paths, the default is acceptable.

Q16 What kinds of equality might you choose for a domain type, and what are the trade-offs? Senior

Two common models: value equality (equal if all relevant fields match — good for value objects like Money or Coordinate) and identity equality (equal if a stable identifier matches — good for entities, e.g. two Customer objects are "the same customer" if their Id matches even if other fields differ). The trade-off: value equality is intuitive for immutable data but wrong for entities whose attributes change over time; identity equality models persistence/ORM semantics but means objects with different data can be equal. Choose based on whether the type represents a value or an entity with identity, and document the choice — records default to whole-object value equality, which suits values but not entities.

Q17 How do you decide what to test, and what to avoid testing? Senior

Test behavior and contracts: business logic, branching, edge cases, error paths, and invariants (like the equality contract) — the things that, if broken, cause real bugs. Avoid testing implementation details (private methods, exact internal calls) since those make tests brittle and block refactoring; instead assert observable outcomes. Don't test the framework or third-party libraries themselves. Favor a few high-value tests over chasing 100% coverage — coverage shows untested lines but not whether your assertions are meaningful. Mock external commands with side effects; prefer real objects for simple queries to avoid over-mocking that just restates the implementation.

Q18 What is over-mocking, and why is it a problem? Senior

Over-mocking is replacing too many collaborators with mocks and asserting on their every interaction, so the test ends up encoding the implementation rather than the behavior. Such tests break whenever you refactor internals even if behavior is unchanged (false failures), give a false sense of safety (you tested that methods were called, not that the result is correct), and become hard to read. Prefer testing through the public surface with real objects where cheap, mock only true external boundaries (I/O, network, time), and assert on outcomes and essential interactions rather than on every call.

Q19 How do records change the equality story, and when would you still hand-roll equality? Senior

Records (C# 9) auto-generate a contract-correct equality: value-based Equals/GetHashCode over all members, plus ==/!= — eliminating the boilerplate and the classic "forgot to override GetHashCode" bug. That's ideal for immutable value/DTO types. You'd still hand-roll when you need custom semantics the record default can't express: equality by a subset of fields (e.g. identity by Id only), case-insensitive or normalized comparison, ignoring computed/transient members, or a domain-specific notion of "equal." You can also override a record's equality members selectively. And records are shallow — for members that are mutable reference types, you may need custom logic.

Q20 Tie it together: how do testing and the equality contract reinforce each other? Senior

They're mutually dependent. Test assertions like Assert.Equal call Equals under the hood, so a broken equality implementation makes tests pass or fail for the wrong reasons — your equality must be correct for your tests to be trustworthy. Conversely, the equality contract is a set of precise, checkable rules (reflexivity, symmetry, transitivity, the hash invariant, dictionary-key behavior), making it a perfect subject for unit tests that document and guard those rules as code evolves. In practice you write a small equality test suite per value type, and you rely on correct equality everywhere you assert. Good engineering treats both as part of the same discipline: define correct behavior, then prove it automatically.
