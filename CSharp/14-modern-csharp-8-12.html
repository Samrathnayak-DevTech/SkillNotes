// C#  ·  five years of language evolution

Modern C# 8–12 ✨

Between C# 8 (2019) and C# 12 (2023), the language gained a wave of features that reshaped how everyday C# is written: records for immutable data, far richer pattern matching, nullable reference types for null safety, and a steady drive to cut ceremony — top-level statements, file-scoped namespaces, primary constructors, collection expressions. This page maps what landed in each version, then dives into the features interviewers ask about most. It draws on the type-system, pattern, and DI topics from earlier. No prior coding background assumed.

01

Overview

Five releases, each tied to a .NET version, pushing a few consistent themes: safety, immutability, expressiveness, and less boilerplate.

C# ships a new version with each annual .NET release. Knowing which feature came in which version is a common interview check, but the bigger picture is the themes these releases share. Here's the map:

C# 8 — .NET Core 3.0 (2019)
Nullable reference types, switch expressions, async streams, ranges & indices, default interface members, using declarations
C# 9 — .NET 5 (2020)
Records, init-only setters, top-level statements, target-typed new, relational & logical patterns, covariant returns
C# 10 — .NET 6 (2021)
Record structs, global usings, file-scoped namespaces, extended property patterns, lambda improvements
C# 11 — .NET 7 (2022)
Raw string literals, required members, generic math (static abstract members), list patterns, UTF-8 literals, file-local types
C# 12 — .NET 8 (2023)
Primary constructors (all types), collection expressions, alias any type, default lambda parameters, ref readonly params

🔑 C# version ↔ .NET version

By default, your target framework picks the C# version: .NET 8 → C# 12, .NET 7 → C# 11, .NET 6 → C# 10, .NET 5 → C# 9, .NET Core 3.x → C# 8. You can override <LangVersion>, but some features need newer runtime/library support (e.g. async streams and ranges needed .NET Core 3.0 types). When unsure "what version is feature X," anchor on this table.

// switch expression -- returns a value, no break, arm per case
string label = day switch
{
    0 or 6 => "weekend",   // (or-pattern arrived in C# 9; see note)
    _      => "weekday"
};

// ranges & indices: ^1 = last, 1..4 = items 1,2,3
int[] a = { 10, 20, 30, 40, 50 };
int last  = a[^1];      // 50
int[] mid = a[1..4];    // 20, 30, 40

// using declaration: disposed at end of scope, no extra braces
using var reader = new StreamReader(path);

// async streams: await over an asynchronous sequence
await foreach (var item in GetPagesAsync()) Use(item);

x ??= ComputeDefault();   // assign only if x is null
🔑 The headliners

Nullable reference types (the flagship safety feature — section 09), switch expressions and richer pattern matching (section 08), and async streams (which the async topic referenced as IAsyncEnumerable) are the C# 8 features most worth knowing cold.

Records are the marquee feature (section 07). Top-level statements let a program be just its statements — no class Program / static void Main wrapper. Relational and logical patterns (and/or/not) made switch expressions far more powerful (section 08).

⚠️ Primary constructors: class vs record differ

In a record, primary-constructor parameters become public init properties automatically. In a regular class/struct (C# 12), they do not become properties — they're captured as private state for use in the body. That's deliberate: records model data (expose it); classes model behavior (hide it). A common interview gotcha.

07

Records & Immutability

The C# 9 headline: a concise way to declare immutable types with value-based equality built in.

A record is a reference type (or value type with record struct) whose compiler generates value-based equality, a readable ToString, and support for non-destructive copying with with. It's purpose-built for immutable data — DTOs, domain values, events.

C#
public record Person(string Name, int Age);   // positional record

var a = new Person("Sam", 30);
var b = new Person("Sam", 30);

// VALUE equality: equal because their contents match (not references)
Console.WriteLine(a == b);        // True  (a class would print False)

// non-destructive copy: new record, one property changed
var older = a with { Age = 31 };   // Person { Name = Sam, Age = 31 }, 'a' unchanged

Console.WriteLine(a);   // Person { Name = Sam, Age = 30 } -- nice ToString, free

vs

🔑 What the compiler generates for a record

Value-based Equals/GetHashCode (compare all members), ==/!=, a ToString that prints the members, a Deconstruct for positional records, and a protected copy constructor used by with. You write one line; you get all of that. Records are still reference types by default — record struct (C# 10) makes a value-type version.

✅ When to use a record vs a class

Use a record for immutable data where two instances with the same contents should be considered equal — DTOs, value objects, messages, configuration snapshots. Use a class for entities with identity and mutable behavior (a service, a stateful object). Records pair naturally with init and C# 11 required members for safe, immutable construction.

08

Pattern Matching

Across C# 8–11, pattern matching grew from a curiosity into one of the language's most expressive tools.

Pattern matching lets you test a value's shape and extract data in one step. Each version added new pattern kinds; together they make switch expressions read like a concise decision table.

PatternExampleSince
Type / declarationobj is string sC# 7
Property{ Age: 30 }C# 8
Positional / tuple(0, 0)C# 8
Relational> 0, <= 10C# 9
Logicaland, or, notC# 9
Extended property{ Addr.City: "X" }C# 10
List[1, .., 9]C# 11
C#
// a switch expression combining several pattern kinds
decimal Fee(Order o) => o switch
{
    // property + relational + logical patterns
    { Total: <= 0 }                  => throw new ArgumentException(),
    { Express: true, Total: > 100 }  => 0,          // free express over 100
    { Total: > 100 }                 => 5,
    { Customer.IsVip: true }          => 2,          // extended property pattern
    _                              => 10
};

// list pattern: match by sequence shape, bind the middle
string Describe(int[] xs) => xs switch
{
    []           => "empty",
    [var only]    => $"one: {only}",
    [var first, .. var rest] => $"first {first}, then {rest.Length} more"
};
🔑 Switch expressions vs switch statements

A switch expression (C# 8) returns a value, uses => arms, needs no break, and the compiler can warn if it isn't exhaustive. Use it when every branch produces a result. Keep the older switch statement for branches that perform side effects rather than yield a single value.

09

Nullable Reference Types

C# 8's flagship safety feature: make the compiler track which references can be null, turning runtime crashes into compile-time warnings.

Historically, any reference could be null, so NullReferenceException was the most common crash. Nullable reference types (NRT) flip the default: with the feature enabled, a plain string is assumed non-null, and you must write string? to allow null. The compiler then warns wherever you might dereference a null.

C# 8
// enable per-file (or project-wide in the .csproj: <Nullable>enable</Nullable>)
#nullable enable

string  name  = null;   // ⚠️ warning: can't assign null to non-nullable
string? maybe = null;   // ✅ fine: explicitly nullable

Console.WriteLine(maybe.Length);   // ⚠️ warning: 'maybe' may be null here

if (maybe is not null)
    Console.WriteLine(maybe.Length);   // ✅ ok: null-checked, compiler knows it's safe

int len = maybe!.Length;   // ! = null-forgiving: "trust me, not null" (use sparingly)
?

Nullable annotation

string? says "this may be null." Plain string says "this won't be" — and the compiler holds you to it.

!

Null-forgiving

x! tells the compiler to suppress its null warning. An escape hatch — overusing it defeats the purpose.

🔎

Flow analysis

After an if (x is not null) check, the compiler tracks that x is safe in that branch.

⚠️ It's compile-time only — warnings, not guarantees

NRT is a static analysis feature: it produces warnings, not runtime enforcement. A non-nullable reference can still be null at runtime (e.g. data from a non-annotated library, deserialization, or reflection). NRT dramatically reduces null bugs by flagging risky code, but it doesn't physically prevent null — so validate data crossing trust boundaries.

✅ Adopt it project-wide

Turn it on with <Nullable>enable</Nullable> in the project file (it's the default in new .NET templates). Treat the warnings as a guide to where null was being ignored, annotate intent with ?, and reserve ! for the rare cases you genuinely know better than the compiler.

Less Ceremony

A theme running through all five versions: say the same thing with less boilerplate.

A striking amount of modern-C# work went into removing ceremony — the ritual code that adds noise without meaning. Stacked up, these features make a typical file dramatically shorter.

then → now
// ── THEN (classic C#) ──────────────────────────────
using System;
namespace MyApp
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("Hello");
        }
    }
}

// ── NOW (top-level statements + implicit usings) ───
Console.WriteLine("Hello");   // that's the whole program

🔑 Less ceremony ≠ less clarity

The goal isn't terseness for its own sake — it's removing noise so the meaning stands out. A primary constructor puts a class's dependencies right next to its name; a collection expression shows the data without the machinery. Used well, these features make intent more visible, not less.

✅ It connects to everything else

These tie back to earlier topics: primary constructors streamline the dependency injection pattern; records build on value vs reference semantics; required/init pair with safe construction; and collection expressions complement collections & LINQ. Modern C# is less a pile of tricks than a coherent push toward safe, expressive, low-noise code.

Interview Questions

Click any question to reveal a clear answer. Tagged by level — Junior (0–2 yrs), Mid (2–5 yrs), Senior (5–7 yrs). Written for someone with no prior coding background.

🌱 Junior — fundamentals (0–2 years)

Q1 What is a record, and how does it differ from a class? Junior

A record (C# 9) is a type designed for immutable data. The key difference is value-based equality: two records with the same property values are considered equal (== returns true), whereas two class instances are equal only if they're the same object. Records also give you a readable ToString and the with expression for copies — all generated by the compiler.

Q2 What are nullable reference types? Junior

A C# 8 feature where the compiler tracks whether a reference is allowed to be null. With it enabled, a plain string means "won't be null" and string? means "might be null." The compiler then warns you wherever you might use a null value, helping prevent NullReferenceException at compile time instead of crashing at runtime.

Q3 What is a switch expression? Junior

A C# 8 form of switch that returns a value. Instead of case/break, each arm uses pattern => result, and the whole expression evaluates to the matching arm's result. It's more concise than a switch statement and the compiler can warn if you haven't covered all cases.

Q4 What are top-level statements? Junior

A C# 9 feature that lets a program's entry point be just the statements themselves, without wrapping them in a class Program with a static void Main. So a whole program can be one line like Console.WriteLine("Hello");. The compiler generates the boilerplate behind the scenes. Great for small programs and reducing noise.

Q5 What does the init keyword do? Junior

An init-only setter (C# 9) lets a property be set during object creation (in the constructor or an object initializer) but not changed afterward. It's how you make properties immutable while still allowing the nice new Thing { Prop = value } initializer syntax. It pairs with records and (in C# 11) required members.

Q6 What are collection expressions? Junior

A C# 12 syntax for creating collections with square brackets: int[] a = [1, 2, 3]; works for arrays, lists, spans, and more. The spread element .. inlines another collection: [.. a, 4, 5]. It unifies what used to require different initializer syntaxes per collection type.

🟠 Mid-level — going deeper (2–5 years)

Q7 Which C# version introduced records, and what does the compiler generate? Mid

Records arrived in C# 9 (record structs in C# 10). For a record the compiler generates value-based Equals and GetHashCode (comparing all members), ==/!= operators, a ToString that prints the members, a Deconstruct method for positional records, and a copy constructor used by the with expression. You declare one line and get all that for free.

Q8 How do primary constructors differ between records and classes? Mid

In a record (C# 9), primary-constructor parameters automatically become public init-only properties — records model data, so the values are exposed. In a regular class or struct (C# 12), the parameters do not become properties; they're captured as private state for use in the body (initializing fields, methods, base calls). The difference reflects intent: records expose data, classes encapsulate it. It's a frequent gotcha.

Q9 What does the with expression do? Mid

It creates a copy of a record with some properties changed, leaving the original untouched — "non-destructive mutation." For example var b = a with { Age = 31 }; produces a new record identical to a except Age. It relies on the compiler-generated copy constructor and is the idiomatic way to "modify" an immutable record.

Q10 What's the difference between string? and the null-forgiving operator !? Mid

string? is an annotation declaring that a reference is allowed to be null, so the compiler will warn if you dereference it without checking. The null-forgiving operator ! (as in x!) is a suppression: it tells the compiler "I know this isn't null here, stop warning." Use ? to express intent honestly; use ! sparingly, only when you genuinely know more than the compiler, since it silences a real safety check.

Q11 When would you use a raw string literal? Mid

Raw string literals (C# 11), delimited by at least three double-quotes ("""), let you write multi-line strings containing quotes, backslashes, and braces with no escaping. They're ideal for embedded JSON, regular expressions, SQL, file paths, or any text where escape sequences would be noisy and error-prone. Leading whitespace up to the closing quotes is stripped, so you can indent the literal naturally.

Q12 What do required members add over init? Mid

init (C# 9) makes a property settable only at creation; required (C# 11) goes further and makes the compiler enforce that the property is set during initialization — omitting it is a compile error. Together (public required string Name { get; init; }) they give you immutable properties that callers can't forget to provide, replacing constructor boilerplate or runtime null checks.

Q13 How has pattern matching evolved across C# 8–11? Mid

C# 8 added property, tuple, and positional patterns plus switch expressions. C# 9 added relational patterns (> 0) and logical patterns (and/or/not). C# 10 added extended property patterns (dotted access like { Addr.City: "X" }). C# 11 added list patterns ([1, .., 9]) for matching sequences. Together they turn switch into a powerful, readable decision tool that tests shape and extracts data at once.

🔴 Senior — synthesis & judgment (5–7 years)

Q14 Is the nullable reference types feature a runtime guarantee? Senior

No — it's compile-time static analysis that emits warnings; it doesn't change runtime behavior or physically prevent null. A reference annotated non-nullable can still hold null at runtime — e.g. values from a library compiled without the annotations, deserialization, reflection, default-initialized arrays, or simply ignoring the warnings. The annotations are also erased to attributes, not enforced by the CLR. So NRT massively reduces null bugs by surfacing risky code, but you still validate external/untrusted data and shouldn't assume a non-null annotation is a hard guarantee.

Q15 Records are immutable — but how "deep" is that immutability? Senior

Records give shallow immutability by default: with positional/init properties the record's own references can't be reassigned, but if a property points to a mutable object (say a List<T> or a mutable class), the contents of that object can still change. Value equality is likewise member-wise and, for reference-typed members, depends on those members' own equality. For true deep immutability, use immutable collections (e.g. ImmutableArray) and immutable member types, and be aware that with copies are shallow too.

Q16 Why don't class primary-constructor parameters become properties like record ones do? Senior

It's a deliberate design choice reflecting the types' purposes. Records model immutable data, so exposing the constructor parameters as public init properties is exactly what you want. Regular classes and structs model behavior and often have internal state they shouldn't expose, so C# 12 captures primary-constructor parameters as private state for use in the body (fields, base calls, method logic) without auto-exposing them. If you want a class to expose one, you declare the property explicitly and initialize it from the parameter. This keeps encapsulation the default for classes while keeping records terse.

Q17 What problem do static abstract interface members (generic math) solve? Senior

Before C# 11, you couldn't write one generic algorithm over "any number type," because operators like + and constants like Zero are static and interfaces couldn't require static members. Static abstract (and virtual) interface members let interfaces declare static requirements — operators, Parse, identity values — so interfaces like INumber<T> can constrain a generic type to "supports these static operations." Now you can write T Sum<T>(IEnumerable<T> xs) where T : INumber<T> once and have it work for int, double, decimal, etc. This underpins the generic-math library.

Q18 How do you decide between a record, a record struct, and a class for a new type? Senior

Ask about identity, mutability, and size. Use a record (class) for immutable reference data where value equality is desired and instances may be largish or shared — DTOs, domain events, value objects. Use a record struct for small, immutable value-semantic data where you also want to avoid heap allocation (watch the usual struct caveats: copy cost if large, defensive copies). Use a plain class for entities with identity and mutable behavior (services, stateful aggregates) where reference equality is the right model. The earlier value-vs-reference trade-offs (boxing, copying, equality semantics) all still apply — records just generate the equality/ToString boilerplate for you.

Q19 Collection expressions look like array initializers — what's actually different and why does it matter? Senior

The old { … } initializer worked only in limited spots (mainly arrays and collection-initializer types) and had no spread. Collection expressions ([ … ], C# 12) are target-typed and unified: the same syntax builds arrays, List<T>, Span<T>/ReadOnlySpan<T>, and any type with a [CollectionBuilder], and the compiler picks an optimized construction per target (e.g. stack-allocating a span). The spread element ..e inlines other sequences. Caveat: there's no natural type, so you can't use var x = [1,2,3]; — you must give a target type. It matters because it both reduces verbosity and lets the compiler emit more efficient code than a hand-written initializer.

Q20 Taken together, what's the through-line of C# 8–12, and how does language version relate to runtime? Senior

The through-line is safety, immutability, expressiveness, and reduced ceremony: NRT and required members make illegal states harder to represent; records and init make immutability ergonomic; pattern matching and switch expressions make decisions declarative; top-level statements, file-scoped namespaces, primary constructors, and collection expressions strip boilerplate. On versioning: the C# compiler version is generally tied to the target framework (.NET 8 → C# 12, etc.), and crucially some features require runtime/library support — default interface members needed CLR changes in .NET Core 3.0; async streams and ranges needed new .NET Core 3.0 types; generic math needs the .NET 7 numeric interfaces. So "can I use feature X" depends on both the LangVersion and the target runtime, not the compiler alone.
