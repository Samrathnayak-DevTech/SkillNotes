UML Diagrams

The Unified Modelling Language is the industry-standard visual vocabulary for communicating software design. This session covers the two diagrams most critical for LLD: Use Case Diagrams (what your system does) and Class Diagrams (how it is structured).

01

Why UML?

In software teams, communication happens between clients, product managers, engineers, and other stakeholders. Each group has a different level of technical knowledge. Describing a complex system with plain text is ambiguous and error-prone.

Core Purpose

UML (Unified Modelling Language) is a standardised notation for representing software system components and their relationships visually. Everyone on the team — regardless of their preferred programming language — reads the same diagram the same way. It removes ambiguity when explaining complex designs.

Without UMLWith UML
Text descriptions are ambiguousVisual notation is unambiguous
Different engineers imagine different structuresEveryone reads the diagram identically
Hard to communicate complex relationshipsRelationships are shown with standard arrows and symbols
Hard for non-engineers to followUse Case diagrams are readable by product managers and clients

03

What Is a Use Case Diagram?

Definition

A Use Case Diagram is a behavioural UML diagram that represents the features, behaviours, and functionalities that a system supports, and the actors (users or other systems) who interact with them. It answers the question: "What can users do with this system?"

Use Case Diagrams are deliberately high-level and readable by non-engineers. Product managers use them to sign off on scope. Architects use them to derive system boundaries. Engineers use them to plan classes and APIs.

Analogy

Think of a Use Case Diagram as a restaurant menu. The menu doesn't show you how each dish is cooked (that's the class diagram) — it just tells you what dishes are available and who can order them. Each dish is a use case; each type of customer (dine-in, takeaway) is an actor.

Importance of System Boundary

The system boundary is crucial in design discussions: it explicitly defines what is in scope and what is out. Anything outside the boundary (e.g. a payment gateway, a bank's system) is modelled as an external actor, not a use case.

06

<<include>> Relationship

The <<include>> relationship indicates that one use case always incorporates the behaviour of another as part of its execution. It is a mandatory dependency — the base use case cannot complete without the included use case.

When to Use

Use <<include>> when a use case always needs another to run as a prerequisite. The included use case is typically a common sub-flow extracted to avoid duplication (DRY principle applied to use cases).

Checkout
«include» ─ ─ ─ →
Payment
Find Address

To complete Checkout, the system must always execute Payment and Find Address — these are mandatory inclusions.

In UML notation: the dashed arrow labelled <<include>> points from the base use case to the included use case. Think of it as: "Checkout includes (calls/requires) Payment."

07

<<extend>> Relationship

The <<extend>> relationship indicates that one use case optionally adds behaviour to another. Unlike <<include>>, the extension only happens under certain conditions — it is optional.

Key Difference from Include

<<include>> = mandatory sub-flow (parent always calls child). <<extend>> = optional variation (child conditionally adds to parent). The arrow direction is also different: an <<extend>> arrow points from the extending use case to the base use case — the extension "points at" what it extends.

What Is a Class Diagram?

Definition

A Class Diagram is a structural UML diagram that describes the classes in a system, their attributes and methods, and the relationships between them. It is the blueprint of your codebase — it shows how you will structure the solution, not just what it does.

Class diagrams are used to: design and communicate architecture, generate code scaffolding, document existing systems, identify refactoring opportunities, and verify that a design satisfies the requirements in the Use Case Diagram.

Dashed border. The stereotype <<interface>> appears above the class name. Methods are abstract by default.

ABSTRACT CLASS — italic name
<<abstract>>Animal

Class name in italics. Abstract methods also in italics. Can have concrete methods and fields.

Uses <<enumeration>> stereotype. Lists constant values in the body. No methods section required for simple enums.

An interface extending another interface is shown with a dashed arrow (implementation/realization arrow) from the child interface to the parent interface — the same notation used when a class implements an interface.

HAS-A Relationship — Aggregation

When an object contains a reference to another object, but both can exist independently, the relationship is called Aggregation. It represents a "has-a" relationship where the contained object has an independent lifetime.

Aggregation

Aggregation: a relationship between two entities A and B where even if the object of A is destroyed (goes out of scope), B can still exist independently. B's lifecycle is not controlled by A.

Show
- tickets: List<Ticket>
◇──
Ticket
- seatNumber: int - price: double

A Show has many Tickets. If the Show is cancelled (object destroyed), the Tickets can still exist — they might be transferred to another show or refunded.

UML notation: a hollow (open) diamond (◇) on the side of the "owner" class (Show), with a line to the "part" class (Ticket). The hollow diamond indicates the part can exist without the whole.

15

HAS-A Relationship — Composition

Composition is a stronger form of aggregation. When an object owns another object such that the contained object cannot exist independently of the container, the relationship is Composition.

Composition

Composition: a relationship between A and B where if the object of A is destroyed, B cannot exist independently. B's lifetime is entirely controlled by A. B is an integral part of A — it has no meaningful existence outside A.

Customer
- address: Address
◆──
Address
- street: String - city: String - pincode: String

A Customer has an Address. If the Customer is deleted from the system, their Address record has no meaning and should also be deleted. Address cannot exist without a Customer.

Another example: a Class and an Assignment. An assignment only exists for a specific class — it has no independent existence once the class ends.

UML notation: a filled (solid) diamond (◆) on the side of the owner. Compare:

AspectAggregation ◇Composition ◆
DiamondHollow (open)Filled (solid)
Lifetime dependencyPart lives independentlyPart dies with the whole
OwnershipWeak (shared ownership possible)Strong (exclusive ownership)
ExampleShow ◇── TicketCustomer ◆── Address
Memory Trick

Aggregation: "A library has books. If the library burns down, the books (that survived) can still be read." Composition: "A human has a heart. If the human dies, the heart has no function — its existence is tied to the human." The filled diamond = the inseparable bond.

Quick Recap

UML Purpose

A standardised notation for communicating software designs visually. Removes ambiguity between engineers, product managers, and clients.

Diagram Categories

Structural (Class, Component, Object) describe static architecture. Behavioural (Use Case, Sequence, Activity) describe dynamic behaviour.

Use Case Diagram

Shows what the system can do (use cases = ovals) and who interacts with it (actors = stick figures). The rectangle is the system boundary.

<<include>>

Mandatory sub-flow. Base use case always executes the included use case. Arrow: base → included. Example: Checkout includes Payment.

<<extend>>

Optional variation. Extension conditionally adds behaviour to base. Arrow: extending → base. Example: Login via Google extends Login.

Generalisation

IS-A in use case land. A TA is a Student (inherits all Student use cases + adds instructor-like ones). Solid arrow: child → parent.

Class Diagram

Blueprint of the codebase. Three compartments: class name, attributes (visibility name: Type), methods (visibility name(params): return).

Visibility Symbols

+ public, - private, # protected, ~ package-private. Underlined members are static. Abstract class name and methods are in italics.

Interface vs Abstract

Interface: dashed border + <<interface>> stereotype. Abstract class: italic name + <<abstract>> stereotype. Both use hollow-triangle arrows for IS-A.

Aggregation ◇

HAS-A with independent lifetime. Hollow diamond on the owner. If the owner is destroyed, the part can still exist (e.g. Show ◇── Ticket).

Composition ◆

HAS-A with dependent lifetime. Filled diamond on the owner. If the owner is destroyed, the part cannot exist (e.g. Customer ◆── Address).

Choosing Association Type

Ask: "Can this part exist without its container?" Yes → Aggregation. No → Composition. When in doubt about ownership, lean toward Aggregation.
