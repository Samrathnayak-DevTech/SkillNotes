Factory Method & Abstract Factory

Two creational patterns for delegating object creation. Factory Method lets subclasses decide which object to instantiate. Abstract Factory groups related factory methods into a cohesive "family" — a factory of factories.

01

The Problem: instanceof Branching

Consider a UserService that saves data to a database. The database could be MySQL, PostgreSQL, or MongoDB. A naïve implementation checks the runtime type to construct the correct query object:

Java — instanceof branching (Bad)
class UserService {
    Database db;   // MySQLDB | PostgreDB | MongoDB

    void saveUser() {
        Query q;
        if (db instanceof MySQLDB) {
            q = new MySQLQuery();
        } else if (db instanceof PostgreDB) {
            q = new PostgreQuery();
        }
        // … each new DB type forces editing this method
    }
}
Violations

This violates both SRP (UserService is now responsible for Query creation logic) and OCP (adding a MongoDB branch requires editing this method). The UserService is coupled to every concrete Query and Database implementation.

02

Factory Method Pattern

Definition

The Factory Method pattern defines an interface for creating an object, but lets subclasses decide which class to instantiate. The factory method defers instantiation to subclasses. The creator knows when to create; the subclass knows what to create.

The core idea: instead of the client deciding which concrete class to instantiate, add a method to the object that produces the appropriate related object. This method is the "factory method". Calling db.createQuery() makes each database subclass responsible for returning its own compatible query object.

Analogy

A laptop manufacturer (the creator) decides when a laptop needs a charger, but the specific charger (USB-C, MagSafe, barrel) is determined by the laptop model (the subclass). The manufacturer doesn't hard-code a brand of charger — it delegates to the specific model.

Java — Factory Method applied
abstract class Database {
    abstract Query createQuery();   // ← THE factory method
    void connectToDB() { /* generic logic */ }
    void refreshDB()    { /* generic logic */ }
}

class MySQLDB extends Database {
    Query createQuery() { return new MySQLQuery(); }
}
class PostgreDB extends Database {
    Query createQuery() { return new PostgreQuery(); }
}
class MongoDB extends Database {
    Query createQuery() { return new MongoDBQuery(); }
}

// UserService — completely decoupled from concrete Query types
class UserService {
    Database db;

    void saveUser() {
        Query q = db.createQuery();  // polymorphism — no instanceof needed
        // use q …
    }
}

Adding a new database (e.g., CassandraDB) now requires only: creating a new CassandraDB class and a CassandraQuery class. The UserService and all existing classes are untouched. Pure OCP.

04

Adding Transactions

The pattern scales cleanly. When we need Transaction objects in addition to queries, we add another factory method to Database:

Java — Multiple factory methods
abstract class Database {
    abstract Query       createQuery();
    abstract Transaction createTransaction();
    // non-factory methods remain generic
    void connectToDB() { … }
    void refreshDB()   { … }
}

class MySQLDB extends Database {
    Query       createQuery()       { return new MySQLQuery(); }
    Transaction createTransaction() { return new MySQLTransaction(); }
}

// Usage remains clean
Database db = … ;  // injected — could be any subtype
Query       q   = db.createQuery();
Transaction txn = db.createTransaction();

Each database subclass (MySQLDB, PostgreDB, MongoDB) creates its own compatible Transaction implementation. The factory methods on the Database class act as the glue — they guarantee that a MySQLDB always produces MySQL-compatible objects.

05

Cross-Platform UI Example

The same pattern applies to cross-platform UI frameworks like Flutter or React Native. A Platform class has a createButton() factory method; each platform subclass returns its native button.

Java — Platform UI Factory Method
abstract class Platform {
    abstract Button createButton();
}
class IOSPlatform     extends Platform { Button createButton() { return new IOSButton(); }}
class AndroidPlatform extends Platform { Button createButton() { return new AndroidButton(); }}
class WindowsPlatform extends Platform { Button createButton() { return new WindowsButton(); }}

// Client
Platform p = getPlatform(); // injected at runtime
Button   b = p.createButton();
b.render();
06

The Problem: Too Many Factory Methods

As the system grows, the Platform class accumulates factory methods: createButton(), createDropDown(), createMenu(), createNavbar(). The class has a lot of factory methods — this is a signal that the responsibilities can be separated further.

UIComponent hierarchy (one factory method per component type per platform)
<<interface>>Button

<<interface>>DropDown
render()
<<interface>>Menu
render()

Each interface has 3 implementations (Windows, iOS, Android) = 9 concrete classes for 3 component types. The Platform class now needs to carry a factory method for every one of these component types.

Signal for Abstract Factory

When you notice a class accumulating many factory methods for a coherent family of related products, it's time to extract those factory methods into a dedicated Factory interface. This is the Abstract Factory pattern.

07

Abstract Factory — Factory of Factories

Definition

The Abstract Factory pattern provides an interface for creating families of related objects without specifying their concrete classes. Each concrete factory creates one complete family. The client works with both the factory and its products through abstract interfaces only.

The Abstract Factory is often described as a "factory of factories." Whereas Factory Method puts a factory method directly on the creator class, Abstract Factory extracts all the factory methods into a separate, dedicated factory interface, and gives each "family variant" its own factory implementation.

Analogy — IKEA vs Crate & Barrel

Think of furniture families. An Abstract Factory is like a furniture brand (IKEA, Crate & Barrel). Each brand produces a complete, consistent family — sofa, coffee table, bookshelf — all in the same style. You pick one brand (factory) and get a matched set. Mixing factories (IKEA sofa, Crate & Barrel table) is possible but you have to do it deliberately.

Abstract Factory: Database Example

The Database class had two factory methods: createQuery() and createTransaction(). With Abstract Factory, we split these responsibilities: Database keeps only non-factory methods, and we introduce a DatabaseFactory interface that owns all the factory methods.

Java — Abstract Factory: Database
// The Abstract Factory interface
interface DatabaseFactory {
    Query       createQuery();
    Transaction createTransaction();
    Updater     createUpdater();
}

// One concrete factory per database family
class MySQLDBFactory implements DatabaseFactory {
    public Query       createQuery()       { return new MySQLQuery(); }
    public Transaction createTransaction() { return new MySQLTransaction(); }
    public Updater     createUpdater()     { return new MySQLUpdater(); }
}

// The Database superclass now just produces a factory — one method
abstract class Database {
    abstract DatabaseFactory createDBFactory(); // ← single factory method
    void connectToDB() { … }
    void refreshDB()   { … }
}
class MySQLDB extends Database {
    DatabaseFactory createDBFactory() { return new MySQLDBFactory(); }
}

// Client usage
Database        db  = … ;
DatabaseFactory dbf = db.createDBFactory();
Query           q   = dbf.createQuery();
Transaction     txn = dbf.createTransaction();
09

Abstract Factory: Cross-Platform UI

The Platform class had too many factory methods. With Abstract Factory, the factory methods move to a UIFactory interface. Each platform subclass produces its own UIFactory implementation:

Java — Abstract Factory: UI
// Abstract Factory for UI components
interface UIFactory {
    Button   createButton();
    DropDown createDropDown();
    Menu     createMenu();
    Navbar   createNavbar();
}

// Concrete factories — one per platform
class WindowsUIFactory implements UIFactory {
    public Button   createButton()   { return new WindowsButton(); }
    public DropDown createDropDown() { return new WindowsDropDown(); }
    public Menu     createMenu()     { return new WindowsMenu(); }
    public Navbar   createNavbar()   { return new WindowsNavbar(); }
}
// Similarly: IOSUIFactory, AndroidUIFactory …

// Platform now holds a single factory method — clean!
abstract class Platform {
    abstract UIFactory createUIFactory();
    void refreshUI() { … }
}
class WindowsPlatform extends Platform {
    UIFactory createUIFactory() { return new WindowsUIFactory(); }
}

// Client
Platform  p   = getPlatform();
UIFactory uif = p.createUIFactory();
Button    b   = uif.createButton();   // correct platform button

Factory Method vs Abstract Factory

AspectFactory MethodAbstract Factory
CreatesOne product at a timeA family of related products
Where definedOverridden method on the creator classSeparate Factory interface, implemented per family
Adding a new product typeAdd a new factory method on the creatorAdd a method to the factory interface (affects all concrete factories)
Adding a new family variantAdd a new creator subclassAdd a new concrete factory class implementing the interface
When to useOne or a few related objects need subclass-specific creationMultiple families of related objects must always be used together consistently
ExamplesDatabase.createQuery(), Platform.createButton()DatabaseFactory, UIFactory, JDBC DataSource providers
Real-World Usage

Java's JDBC is a classic Abstract Factory: Connection, Statement, and ResultSet are abstract products. MySQL, PostgreSQL, and Oracle provide their concrete factories (drivers). Spring's ApplicationContext is another example — it's an abstract factory for Spring beans. GUI toolkits like SWT and AWT use Abstract Factory to produce native OS widgets.

Quick Recap

Factory Method

An abstract/interface method on a creator class. Each subclass overrides it to return the appropriate concrete product. Delegates instantiation to subclasses.

Why not instanceof?

instanceof branching violates OCP (must edit for each new type) and SRP (creator takes on instantiation responsibility).

createQuery() pattern

Database → abstract createQuery(). MySQLDB → returns MySQLQuery. UserService calls db.createQuery() — no instanceof, no coupling.

Abstract Factory

Groups related factory methods into a dedicated Factory interface. Each "family variant" provides its own concrete factory. A factory of factories.

When: Abstract Factory

When the creator accumulates many factory methods for a coherent family. Extract them into a UIFactory / DatabaseFactory interface.

Consistency guarantee

Abstract Factory ensures that all products from one factory (MySQLDBFactory) are compatible with each other — you never accidentally mix MySQL queries with Postgres transactions.

Cross-platform UI

Flutter / React Native use the Abstract Factory approach: Platform → UIFactory → Button, DropDown, Menu. Each platform family is self-consistent.

Real-world: JDBC

JDBC is a canonical Abstract Factory implementation — Connection, Statement, ResultSet are abstract; MySQL/Postgres drivers are the concrete factories.
