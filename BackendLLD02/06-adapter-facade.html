Adapter & Facade Patterns

Two structural design patterns that tame interface mismatches and subsystem complexity. The Adapter translates one interface into another. The Facade provides a single simplified entry point to a complex system of collaborating objects.

What Are Structural Design Patterns?

Design patterns are broadly divided into three categories: Creational (how objects are made), Structural (how objects are composed and related), and Behavioural (how objects communicate). The Adapter and Facade patterns both belong to the Structural category.

🏗️
Creational

Deal with object creation mechanisms. Examples: Singleton, Factory Method, Abstract Factory, Builder.

🔗
Structural

Deal with object composition — how classes and objects are assembled into larger structures. Examples: Adapter, Facade, Decorator, Flyweight, Proxy.

💬
Behavioural

Deal with communication and responsibility between objects. Examples: Strategy, Observer, Command, Iterator, Template Method.

Key Insight

Structural patterns are about how classes and objects fit together — they describe composition relationships that make a system easier to work with, extend, or integrate with external code you don't control.

02

The Problem: Incompatible Interfaces

When building real applications, you often need to integrate with third-party libraries, legacy code, or external APIs. The problem is that these external components don't implement the interface your application expects — their method signatures, naming conventions, or parameter orders differ from what your code calls.

Real-World Analogy

Think of a UK plug and a US socket. Both carry electricity, but their shapes are incompatible. You need a physical adapter to bridge the gap — you don't replace the plug or redesign the socket. The adapter sits between them and makes the connection work. This is exactly what the Adapter design pattern does in software.

Consider PhonePe: it needs to connect to multiple banks (YesBank, ICICI, HDFC) to process payments. Each bank exposes its own proprietary SDK with completely different method names. PhonePe can't modify the bank SDKs — they're external, third-party code.

The Problem

❌→
YesBank SDK

PhonePe

ICICI SDK

Each bank has a different method name and parameter order — PhonePe can't call them uniformly.

03

Adapter Pattern Defined

Definition

The Adapter Pattern converts the interface of a class into another interface that clients expect. It lets classes work together that could not otherwise because of incompatible interfaces. The adapter wraps the incompatible class and exposes a target interface that your application already understands.

The Adapter pattern involves four participants:

ParticipantRoleIn PhonePe Example
TargetThe interface your application usesBankApi interface
ClientThe class that uses the Target interfacePhonePe
AdapteeThe incompatible third-party classYesBank, ICICI
AdapterWraps the Adaptee, implements TargetYesBankHelper, ICICIHelper
SOLID Alignment

The Adapter pattern enforces the Open/Closed Principle: you add support for a new bank by creating a new adapter class without touching PhonePe at all. It also supports the Dependency Inversion Principle: PhonePe depends on the BankApi abstraction, not on concrete bank implementations.

04

PhonePe + Bank API Example

PhonePe defines a BankApi interface with the methods it needs. Each bank has its own SDK. Adapter classes implement BankApi and internally delegate to the bank's SDK with the correct signature translation.

<<interface>>BankApi
+ register(): void + sendMoney(from, to, x): void
YesBankHelper (Adapter)
- yb: YB + sendMoney(from,to,x) → yb.transfer(from,to,x)
ICICIHelper (Adapter)
- icici: ICICI + sendMoney(from,to,x) → icici.sendMoney(x,from,to)
YB (Adaptee)
+ transfer(from,to,x)
ICICI (Adaptee)
+ sendMoney(x,from,to)

Notice how ICICI's sendMoney takes (x, from, to) — the amount first — while our BankApi uses (from, to, x). The adapter silently reorders the arguments.

05

Full Implementation in Java

Java — BankApi interface (Target)
// The interface PhonePe works with — the Target
interface BankApi {
    void register();
    void sendMoney(String from, String to, double amount);
}
Java — YesBankHelper (Adapter)
// YesBank's own SDK — we cannot modify this
class YB {
    void transfer(String from, String to, double x) { /* YB logic */ }
}

// Adapter: implements BankApi, wraps YB
class YesBankHelper implements BankApi {
    private YB yb = new YB();

    public void register() { /* delegate to YB's registration */ }

    public void sendMoney(String from, String to, double x) {
        // YB uses the same parameter order — straightforward delegation
        yb.transfer(from, to, x);
    }
}
Java — ICICIHelper (Adapter with reordering)
// ICICI's SDK — note the different parameter order!
class ICICI {
    void sendMoney(double x, String from, String to) { /* ICICI logic */ }
}

// Adapter: reorders arguments to match BankApi contract
class ICICIHelper implements BankApi {
    private ICICI icici = new ICICI();

    public void register() { /* delegate */ }

    public void sendMoney(String from, String to, double x) {
        // ICICI's method takes amount first — adapter fixes the ordering
        icici.sendMoney(x, from, to);
    }
}
Java — PhonePe client (works with BankApi only)
class PhonePe {
    private BankApi api;

    // Inject the desired adapter at construction time
    PhonePe(BankApi api) { this.api = api; }

    void sendMoney(String from, String to, double amount) {
        api.sendMoney(from, to, amount);
        // PhonePe has NO idea whether it's talking to YesBank or ICICI
    }
}

// Usage — just swap the adapter, PhonePe code never changes
PhonePe pp1 = new PhonePe(new YesBankHelper());
PhonePe pp2 = new PhonePe(new ICICIHelper());
pp1.sendMoney("A", "B", 500);
What We Achieved

PhonePe only knows about BankApi. Adding a new bank (HDFC, Axis) means writing one new adapter class — zero changes to PhonePe. This is a clean OCP win.

Scaler + Payment Gateways

The same pattern applies to Scaler's payment integration. Scaler needs to accept payments via multiple gateways (Razorpay, Stripe). Rather than coupling Scaler directly to each gateway's proprietary SDK, Scaler defines a Payment interface and creates adapter classes for each gateway.

Scaler (Client)
- payment: Payment + processPayment()
depends on ↓
<<interface>>Payment
+ pay(amount): void
RazorpayAdapter
- rzp: Razorpay + pay(amount)
StripeAdapter
- stripe: Stripe + pay(amount)
Pattern Summary

Use the Adapter pattern whenever you need to integrate a class whose interface is incompatible with what your system expects — particularly when dealing with third-party SDKs, legacy APIs, or any external code you cannot modify.

07

The Problem: Complex Subsystems

As systems grow, a single action can require orchestrating many collaborating services. Consider what happens when a customer places an order on Amazon. A naïve implementation might look like this:

Java — Amazon without Facade (too many dependencies)
class Amazon {
    LogisticsService ls;
    NotificationService ns;
    TrackingService ts;
    DeliveryService ds;
    SellerService ss;
    InventoryService invs;

    void onOrderPlaced() {
        ls.updateLogistics();
        ns.sendNotification();
        ts.updateTracking();
        ds.updateDelivery();
        ss.notifySeller();
        invs.updateInventory();
    }
}
The Problem

The Amazon class is directly coupled to six different service classes. Every time a new service is added (or removed) for order placement, you must modify Amazon. The class has too many responsibilities — it's managing the coordination logic itself. This violates SRP and makes the class fragile.

08

Facade Pattern Defined

Definition

The Facade Pattern provides a simplified interface to a complex subsystem. It doesn't hide the subsystem — the subsystem classes remain available directly — but the facade offers a convenient, high-level interface for the most common use cases. Clients that need the simple behaviour use the facade; clients that need fine-grained control can still bypass it.

The word "facade" comes from architecture — the front face of a building. You see one clean, unified surface from the street, even though the building contains many complex systems (plumbing, wiring, load-bearing structure) behind it. In software, the facade is that clean front face.

Analogy

A television remote control is a facade. The TV has a complex circuit board, tuner, display driver, and audio processor. The remote exposes only the buttons you need: on/off, volume, channel. You don't need to understand the subsystem to operate it effectively.

Without FacadeWith Facade
Client is coupled to many service classesClient depends only on the Facade
Adding/removing a service requires changing the clientOnly the Facade needs to change
The orchestration logic lives in the clientOrchestration logic is centralised in the Facade
Hard to test the client in isolationEasy to mock or stub the Facade
09

Amazon Order Placed — Facade Example

We extract all the order-placement coordination into a dedicated AmazonOnOrderPlacedHelper class. Amazon now holds a reference to this helper and calls a single method.

Amazon
- obj: AmazonOnOrderPlacedHelper + onOrderPlaced() obj.onOrderPlacedHelper()
→
AmazonOnOrderPlacedHelper (Facade)
- ls: LogisticsService - ns: NotificationService - ts: TrackingService - ds: DeliveryService - ss: SellerService + onOrderPlacedHelper() ls.updateLogistics() ns.sendNotification() ts.updateTracking() ...
Java — Facade Class
// Facade: encapsulates all subsystem coordination
class AmazonOnOrderPlacedHelper {
    private LogisticsService ls = new LogisticsService();
    private NotificationService ns = new NotificationService();
    private TrackingService ts = new TrackingService();
    private DeliveryService ds = new DeliveryService();
    private SellerService ss = new SellerService();

    void onOrderPlacedHelper() {
        ls.updateLogistics();
        ns.sendNotification();
        ts.updateTracking();
        ds.updateDelivery();
        ss.notifySeller();
    }
}

// Amazon is now clean — single dependency, single method call
class Amazon {
    private AmazonOnOrderPlacedHelper obj = new AmazonOnOrderPlacedHelper();

    void onOrderPlaced() {
        obj.onOrderPlacedHelper();
    }
}
Benefit

Now if Amazon adds an InventoryService to the order flow, only AmazonOnOrderPlacedHelper changes. The Amazon class is completely untouched. The facade centralises all coupling to the subsystem.

10

Adapter vs Facade — Key Differences

Students often confuse these two patterns because they both "wrap" other objects. The distinction lies in intent.

AspectAdapterFacade
IntentConvert one interface into anotherSimplify access to a complex system
Problem solvedInterface incompatibilitySubsystem complexity
Number of classes wrappedUsually one (the Adaptee)Usually many (the subsystem)
Changes interface?Yes — maps one interface to anotherNo — provides a new, simpler interface on top
Subsystem accessClient loses direct access to AdapteeSubsystem classes remain accessible directly
CategoryStructuralStructural
Memory Trick

Adapter = plug converter. Makes incompatible things work together. Facade = reception desk at a hospital. You tell the receptionist what you need; they coordinate the complex internal system (radiology, pharmacy, labs) so you don't have to know where everything is.

Quick Recap

Structural Patterns

Deal with how objects and classes are composed. Adapter and Facade are both structural patterns — they shape how components fit together.

Adapter Intent

Convert the interface of an incompatible class into the interface your client expects — without modifying either the client or the incompatible class.

Adapter Participants

Target (your interface), Client (uses Target), Adaptee (incompatible third party), Adapter (wraps Adaptee, implements Target).

PhonePe / YesBank

YesBankHelper and ICICIHelper implement BankApi and delegate internally to the bank SDKs. PhonePe only knows BankApi.

Facade Intent

Provide a single, simplified entry point to a complex subsystem. Reduces coupling between client and subsystem classes.

Amazon Order Example

AmazonOnOrderPlacedHelper is the Facade. Amazon delegates to it. All service coordination is centralised inside the Facade.

Adapter vs Facade

Adapter bridges two incompatible interfaces (one-to-one). Facade simplifies a complex group of classes (one-to-many). Both are structural, but different problems.

SOLID Benefits

Adapter supports OCP (add a new bank → new adapter, no existing code changes). Facade supports SRP (removes coordination logic from the client).
