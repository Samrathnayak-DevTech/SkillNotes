Behavioural Patterns Overview

Behavioural design patterns are concerned with algorithms and the assignment of responsibilities between objects. They describe not just the structure of objects, but also the patterns of communication between them.

Key Insight

Behavioural patterns deal with methods, actions, and behaviours — specifically, how to cleanly distribute those actions across a system when there are many possible ways to do something, or many objects that need to react to the same event.

This session covers two important behavioural patterns:

PatternCore ProblemSolution
StrategyMultiple if-else branches for different "ways to do something"Each way becomes its own class; client picks the right one
ObserverAn event source hard-codes all the things that should happen when the event firesSubscribers register themselves; source iterates and calls them
02

Google Maps — The Problem

Google Maps shows a path from source A to destination B. It supports multiple transport modes: Car, Bike, Bus, Train, Walk. Each mode calculates a completely different path — a car uses highways, a cyclist avoids them, a pedestrian uses footpaths.

Mobile UI

The Google Maps app shows a source field, a destination field, and then mode buttons: Bike | Car | Bus | Train | Walk. The route displayed changes entirely depending on which button you press.

A naïve implementation puts all the logic inside a single findPath method with a chain of if-else branches:

Java — Naïve GoogleMaps (bad design)
class GoogleMaps {
    void findPath(String src, String dest, String mode) {
        if (mode.equals("Car")) {
            // ... car routing logic ...
        } else if (mode.equals("Bike")) {
            // ... bike routing logic ...
        } else if (mode.equals("Walk")) {
            // ... walking routing logic ...
        } // and so on for Bus, Train ...
    }
}
03

SRP & OCP Violations

① SRP Violation

GoogleMaps is now responsible for all routing algorithms — car, bike, walk, bus, train. It has multiple reasons to change: fix a bug in car routing → edit GoogleMaps. Improve bike routing → edit GoogleMaps. Each mode is a separate responsibility crammed into one class.

② OCP Violation

Adding a new transport mode (e.g. "Ferry") requires editing the existing GoogleMaps.findPath method — adding another else if branch. Classes should be open for extension, closed for modification. Every new mode breaks this principle.

The root cause: when there are multiple ways of doing something, packing them all into one method leads to SRP and OCP violations through if-else proliferation. This is the exact scenario the Strategy pattern is designed to solve.

04

Strategy Pattern Defined

Definition

The Strategy Pattern defines a family of algorithms, encapsulates each one in a separate class, and makes them interchangeable. Rather than implementing different behaviours as branches inside a single method, each behaviour gets its own class. The client holds a reference to a strategy interface and delegates work to whichever concrete strategy is currently set.

The rule is: for every way to do a particular thing, create a class.

Analogy

Think of a Swiss Army knife vs a toolbox. Packing every tool into one handle (the naïve approach) makes each tool clunky. A toolbox keeps each tool separate — you pick the right one for the job. The Strategy pattern is the toolbox approach for algorithms.

GoogleMaps no longer contains any routing logic. It holds a PathCalculator reference and delegates to it:

Java — GoogleMaps with Strategy
class GoogleMaps {
    private PathCalculator pc;

    void findPath(String src, String dest, Mode mode) {
        PathCalculator pc = PathCalculatorFactory.getPCForMode(mode);
        pc.findPath(src, dest);
    }
}
06

PathCalculatorFactory

A factory class maps the Mode enum to the correct PathCalculator implementation. The mode parameter is typed as an enum (not a raw String) to prevent typos and enable compile-time checking.

Java — Mode Enum + PathCalculatorFactory
enum Mode { CAR, BIKE, WALK, BUS, TRAIN }

class PathCalculatorFactory {
    public static PathCalculator getPCForMode(Mode mode) {
        if (mode == Mode.CAR)   return new CarPathCalculator();
        if (mode == Mode.BIKE)  return new BikePathCalculator();
        if (mode == Mode.WALK)  return new WalkPathCalculator();
        throw new IllegalArgumentException("Unknown mode: " + mode);
    }
}
Combining Patterns

Notice that PathCalculatorFactory is both a Factory (it creates objects) and a companion to the Strategy pattern (it selects the right strategy). Real systems commonly combine two or more design patterns this way.

07

Full Strategy Implementation

Java — Complete Strategy Pattern for Google Maps
// Strategy interface
interface PathCalculator {
    void findPath(String src, String dest);
}

// Concrete strategies — each encapsulates one algorithm
class CarPathCalculator implements PathCalculator {
    public void findPath(String src, String dest) {
        System.out.println("Car route: " + src + " → " + dest);
        // highway-aware routing logic
    }
}

class BikePathCalculator implements PathCalculator {
    public void findPath(String src, String dest) {
        System.out.println("Bike route: " + src + " → " + dest);
        // cycle-lane-aware routing logic
    }
}

class WalkPathCalculator implements PathCalculator {
    public void findPath(String src, String dest) {
        System.out.println("Walking route: " + src + " → " + dest);
        // footpath-based routing logic
    }
}

enum Mode { CAR, BIKE, WALK }

class PathCalculatorFactory {
    public static PathCalculator getPCForMode(Mode mode) {
        switch(mode) {
            case CAR:  return new CarPathCalculator();
            case BIKE: return new BikePathCalculator();
            case WALK: return new WalkPathCalculator();
            default:   throw new IllegalArgumentException();
        }
    }
}

class GoogleMaps {
    void findPath(String src, String dest, Mode mode) {
        PathCalculator pc = PathCalculatorFactory.getPCForMode(mode);
        pc.findPath(src, dest);
    }
}

// Usage
GoogleMaps gm = new GoogleMaps();
gm.findPath("Bangalore", "Chennai", Mode.CAR);
gm.findPath("Bangalore", "Chennai", Mode.BIKE);
08

Amazon Order — Observer Problem

When an order is placed on Amazon, many services need to react: logistics must update delivery routing, inventory must decrement stock, notifications must be sent to the customer, the seller must be informed. Without a pattern, Amazon either calls all these services directly (tight coupling) or uses a Facade (better, but still hard-coded).

Problems

① Every time a new action must happen on order placement (e.g. start a loyalty points calculation), source code must change and be recompiled. ② You cannot add or remove a handler at runtime without editing the core class. ③ The event source (Amazon) is tightly coupled to every handler — it must know all of them by name.

09

Observer Pattern Defined

Definition

The Observer Pattern (also called Publish-Subscribe or Pub/Sub) defines a one-to-many dependency between objects. When a publisher object changes state or fires an event, all registered subscribers are notified and updated automatically. The key idea is reversing the dependency: instead of the publisher knowing about each subscriber, the subscribers register themselves with the publisher.

Analogy

A YouTube channel (publisher) doesn't call each subscriber individually. Subscribers click "Subscribe" to opt in. When a video is uploaded (event), YouTube notifies all current subscribers. New subscribers can join and old ones can unsubscribe — the channel doesn't care who is currently watching.

Full Observer Implementation

Java — Subscriber interface + concrete subscribers
// Subscriber contract
interface OnOrderPlacedSubscriber {
    void onOrderPlaced();
}

class LogisticsService implements OnOrderPlacedSubscriber {
    public void onOrderPlaced() {
        System.out.println("Logistics: updating shipping route...");
    }
}

class InventoryService implements OnOrderPlacedSubscriber {
    public void onOrderPlaced() {
        System.out.println("Inventory: decrementing stock count...");
    }
}

class NotificationService implements OnOrderPlacedSubscriber {
    public void onOrderPlaced() {
        System.out.println("Notifications: sending order confirmation email...");
    }
}
Java — Amazon (Publisher)
class Amazon {
    private List<OnOrderPlacedSubscriber> subs = new ArrayList<>();

    // Subscribers register themselves at runtime
    void registerOnOrderPlacedSub(OnOrderPlacedSubscriber sub) {
        subs.add(sub);
    }

    // Subscribers can also remove themselves at runtime
    void unregisterOnOrderPlacedSub(OnOrderPlacedSubscriber sub) {
        subs.remove(sub);
    }

    // Called when a customer places an order
    void onOrderPlacedEvent() {
        for (OnOrderPlacedSubscriber sub : subs) {
            sub.onOrderPlaced(); // notify each subscriber
        }
    }
}

// Usage — register subscribers at startup (or any time at runtime)
Amazon amazon = new Amazon();
amazon.registerOnOrderPlacedSub(new LogisticsService());
amazon.registerOnOrderPlacedSub(new InventoryService());
amazon.registerOnOrderPlacedSub(new NotificationService());

// When an order is placed, all registered subscribers are notified
amazon.onOrderPlacedEvent();

// Remove a subscriber at runtime — no code change needed in Amazon
amazon.unregisterOnOrderPlacedSub(inventoryService);
Runtime Flexibility

Adding or removing a subscriber no longer requires touching Amazon's source code. A new microservice just registers itself. Maintenance windows, A/B tests, and feature toggles can all be modelled by registering or unregistering subscribers at runtime.

Quick Recap

Behavioural Patterns

Deal with how objects communicate and assign responsibilities. Strategy handles "multiple ways to do something". Observer handles "multiple things that should react to an event".

Strategy Pattern

Encapsulate each variant algorithm in its own class. All variants implement a common interface. The client picks and delegates to the right one, eliminating if-else chains.

SRP + OCP Wins

Strategy restores SRP (each class has one algorithm) and OCP (add a new mode → add a new class, no edits to existing code).

PathCalculatorFactory

A factory maps a Mode enum to the correct PathCalculator strategy. Combines the Factory Method and Strategy patterns in one tidy class.

Observer Pattern

Reverses the dependency: subscribers register themselves with the publisher. The publisher iterates its subscriber list and calls onEvent() on each.

Register / Unregister

Publisher exposes register() and unregister() methods. Subscribers can be added or removed at runtime — no source changes required.

OnOrderPlacedSubscriber

Common interface for all Amazon event handlers. LogisticsService, InventoryService, NotificationService all implement it. Amazon only knows the interface.

Real-World Observer

Java's EventListener API, JavaScript's addEventListener, React's state subscriptions, and message brokers like Kafka all implement the Observer pattern at different scales.
