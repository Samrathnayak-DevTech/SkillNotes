01

What are Design Patterns?

A design pattern is a standard, reusable solution to a commonly occurring problem in software design. Think of them as templates — not finished code, but proven blueprints you can adapt to your situation.

Gang of Four (GoF)

In 1994, Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides published Design Patterns: Elements of Reusable Object-Oriented Software, cataloguing 23 patterns. These became the canonical reference for OOP design. The book is entirely focused on object-oriented programming.

Design principles (like SOLID) are the broader philosophical rules that guide how you design. Design patterns are concrete, named solutions that apply those principles to recurring structural problems. Good code aims to be:

⇡
Extensible

Adding new features requires new code, not changes to existing stable code.

⚙
Maintainable

Bugs can be isolated and fixed without cascading side-effects across the codebase.

◎
Readable

Another engineer can understand the intent of a class or method at a glance.

⬡
Modular

The system is composed of loosely coupled, independently testable units.

↺
Reusable

Components can be composed in new contexts without modification.

03

S — Single Responsibility Principle

Every code unit — method, class, or interface — should have a single, well-defined responsibility. Put differently: there should be only one reason to change a code unit.

Robert C. Martin's Definition

"A class should have one, and only one, reason to change." If two distinct actors (teams, requirements, stakeholders) can cause a class to change for different reasons, the class has too many responsibilities.

The Bird Problem

Consider a single Bird class designed to represent all birds — Pigeon, Sparrow, Owl, and more. Its fly() method looks like this:

Java
void fly() {
    if (type.equals("Pigeon")) {
        // pigeon-specific flight logic …
    } else if (type.equals("Sparrow")) {
        // sparrow flight logic …
    } else if (type.equals("Owl")) {
        // owl flight logic …
    }
    // … every new bird adds another else-if
}

This violates SRP in four compounding ways:

Problems with the God-Class approach

1. Readability: A single method that handles every bird type becomes enormous and hard to follow. 2. Testing: You cannot test Pigeon flight logic in isolation without running the entire if-else chain. 3. Code duplication: Shared utility logic tends to get copy-pasted between branches. 4. Low reusability: You cannot reuse the Pigeon logic elsewhere without dragging the whole method along.

How to Spot SRP Violations

Two reliable signals:

Too many if-else conditions. If the branches contain independent, unrelated logic (each branch is a completely different behaviour), that logic probably belongs in separate classes. If the conditions are part of a unified algorithm or business rule, they may be fine.

Monster methods. A method that does more than its name suggests. saveToDB(User user) that also builds a DB connection from scratch violates SRP — connection management and query execution are separate concerns.

Java — SRP Violation
void saveToDB(User user) {
    Database db = new Database();
    db.setURL("jdbc:mysql://…");   // connection concern
    db.setPort(3306);
    String q = "INSERT INTO users …"; // query concern
    db.execute(q);
}
Java — SRP Fixed
void saveToDB(User user) {
    String q   = createQuery(user);          // delegated
    Database db = getDBConnection();          // delegated
    db.execute(q);
}

A practical project-level signal: a Utils package that contains one giant Utils.java file. Split it: StringUtils.java, DateUtils.java, DBUtils.java. Each file has one responsibility.

04

O — Open/Closed Principle

A module should be open for extension but closed for modification. Adding a new feature should require writing new code, not editing existing, tested code.

The Core Idea

Adding a new feature should require minimal (ideally zero) changes to the existing codebase. You extend the system by adding new classes or methods, not by touching stable ones.

Why Modification is Dangerous

Every time you touch existing, production-tested code you risk introducing regressions — bugs in previously working functionality. This is the testing & regression problem. The more if-else branches you add to an existing method, the greater the blast radius of a change.

Applying OCP to the Bird Example

The single Bird class with a fly() that branches on type violates OCP: adding an Eagle means editing fly(). The fix is to make Bird abstract and give each bird its own fly() override.

<<abstract>>Bird
name: String age: int color: String weight: double
fly(): void eat(): void walk(): void makeSound(): void
↑ extends
Pigeon
fly() { … }
Eagle
fly() { … }

Adding a new bird (Eagle, Flamingo, …) only requires creating a new subclass — no changes to the Bird base class or any existing bird class.

Rule of Thumb

The generic behaviour belongs in the parent. Behaviour that is specific to a particular variant belongs in that variant's own class. The parent class is the stable contract; subclasses are the extensions.

05

L — Liskov Substitution Principle

An object of any child class must be freely substitutable wherever a parent type is expected, without requiring any special-case code or causing unexpected behaviour.

Barbara Liskov (1987)

If S is a subtype of T, then objects of type T may be replaced with objects of type S without altering any of the desirable properties of the program.

The Penguin Problem

We add a Penguin to our Bird hierarchy. Penguins are birds, but they cannot fly. If Penguin extends the abstract Bird that has a fly() method, Penguin must override it — but with what? Two bad options: throw an exception, or leave the body empty.

Java — LSP Violation
Bird b = new Pigeon();
b.fly();  // ✓ works fine

b = new Penguin();
b.fly();  // ✗ throws RuntimeException — unexpected outcome!

The client code has a reference of type Bird. It reasonably expects fly() to work. Substituting a Penguin breaks that contract — a clear LSP violation.

Ideal Solution

If an entity doesn't support a behaviour, it should not even have that method. Do not force a fly() override onto a class that cannot fly. Instead, restructure the hierarchy: separate FlyingBird and NonFlyingBird subclasses of the abstract Bird. Only FlyingBird carries the fly() method.

<<abstract>>Bird
eat() walk() makeSound()
FlyingBird
fly()
↑ Pigeon, Sparrow, Owl
NonFlyingBird
(no fly method)
↑ Penguin

Now Bird b = new Penguin() is perfectly safe — the NonFlyingBird type never exposes a fly() method, so there is no false contract.

Class Explosion Warning

If you combine behaviours purely through inheritance (FlyingDancingBird, FlyingNonDancingBird, NonFlyingDancingBird, …), you get a class explosion: 2 binary traits → 4 classes, 10 binary traits → 1024 classes. The next principle — ISP — solves this using interfaces.

06

I — Interface Segregation Principle

Interfaces should be as small (lean) as possible. No class should be forced to implement methods it does not use. This is SRP applied to interfaces.

The Problem with Fat Interfaces

If you combine unrelated capabilities into one interface — say a FlyDance interface with both fly() and dance() — then every bird that can fly but not dance is still forced to implement dance(), either with a no-op or an exception. Both are bad.

The Solution: Many Small Interfaces

Split capabilities into separate, single-method interfaces. A class opts in by implementing only the interfaces that match its capabilities.

<<abstract>>Bird
eat() walk() makeSound()
<<interface>>Flyable
fly()
<<interface>>Danceable
dance()
<<interface>>Swimmable
swim()
Java
// Pigeon: extends Bird, can fly AND dance
class Pigeon extends Bird implements Flyable, Danceable {
    public void fly()   { /* flap wings */ }
    public void dance() { /* head-bob */ }
}

// Penguin: extends Bird, can swim — NOT flyable, NOT danceable
class Penguin extends Bird implements Swimmable {
    public void swim() { /* torpedo mode */ }
}

// Collect all flying birds — clean, type-safe list
List<Flyable> flyingBirds = new ArrayList<>();
flyingBirds.add(new Pigeon());
flyingBirds.add(new Sparrow());
// new Penguin() would be a compile error here — perfect!

This also eliminates the class-explosion problem. With 10 behaviour types, no new subclasses are needed — each concrete class simply picks the interfaces that apply to it.

07

D — Dependency Inversion Principle

No two concrete classes should depend directly on each other. High-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces or abstract classes).

The Principle

1. High-level modules should not import low-level modules. Both should depend on abstractions. 2. Abstractions should not depend on details. Details (concrete implementations) should depend on abstractions.

The Code Duplication Problem

Both Pigeon and Sparrow implement Flyable and contain identical flight logic. The naive fix — copying the same fly() body into both — creates code duplication. A bug in the flight logic must now be fixed in two (or more) places.

Java — DIP Violation (direct concrete dependency)
class Pigeon extends Bird implements Flyable {
    public void fly() {
        // 10 lines of pigeon-sparrow flight logic here
    }
}
class Sparrow extends Bird implements Flyable {
    public void fly() {
        // exact same 10 lines — duplication!
    }
}

The Solution: Depend on an Abstraction

Extract the flight logic into a FlyBehaviour interface and a concrete implementation class. Both Pigeon and Sparrow hold a reference to FlyBehaviour (the abstraction) and delegate to it. This is also the foundation of the Strategy design pattern.

Java — DIP Applied
// Abstraction
interface FlyBehaviour {
    void makeFly();
}

// Concrete implementation — encapsulates the shared logic once
class PigeonSparrowFlyBehaviour implements FlyBehaviour {
    public void makeFly() {
        // flight logic lives here exactly once
    }
}

// High-level classes depend on the interface, not the concrete class
class Pigeon extends Bird implements Flyable {
    FlyBehaviour fb = new PigeonSparrowFlyBehaviour();
    public void fly() { fb.makeFly(); }
}
class Sparrow extends Bird implements Flyable {
    FlyBehaviour fb = new PigeonSparrowFlyBehaviour();
    public void fly() { fb.makeFly(); }
}
Analogy

Think of a power socket. Your laptop (high-level) doesn't depend directly on the power plant (low-level). Both depend on the standardised socket interface (abstraction). You can change the power provider without touching the laptop.

Dependency Injection (Bonus)

A complementary technique (not one of the five SOLID principles) is Dependency Injection: instead of a class creating its own FlyBehaviour with new, the behaviour is injected from the outside (via constructor or setter). This makes the class fully decoupled and trivially testable — you can inject a mock behaviour in tests.

✦

Quick Recap

S — Single Responsibility

Every class/method has one reason to change. Watch for monster methods and Utils god-classes.

O — Open/Closed

Extend with new classes; don't modify stable existing ones. Abstract base class + subclasses = OCP in practice.

L — Liskov Substitution

A subclass must be a perfect behavioural substitute for its parent. Never force a subclass to throw exceptions for inherited methods it can't support.

I — Interface Segregation

Keep interfaces lean — ideally one method each. Classes implement only what they actually support. Eliminates class explosion and forced no-op methods.

D — Dependency Inversion

Concrete classes depend on abstractions (interfaces), not on each other directly. Enables swappable implementations and easy testing.

Design Patterns

23 GoF patterns grouped as Creational, Structural, and Behavioural. All are built on SOLID foundations.

Class Explosion

Combining N binary behaviours purely through inheritance needs 2ᴺ classes. Interface segregation solves this by letting classes opt-in selectively.

Dependency Injection

Passing dependencies from the outside rather than constructing them inside. Not a SOLID principle, but a natural companion to DIP.
