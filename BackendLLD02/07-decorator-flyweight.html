Decorator & Flyweight Patterns

Two structural patterns. The Decorator adds behaviour to an object at runtime through recursive wrapping — no subclass explosion. The Flyweight slashes memory by sharing common immutable state across thousands of similar objects.

01

The Ice Cream Problem

Imagine an application that takes orders for custom ice cream cones. An ice cream can be built with different combinations of cone types, scoops, and syrups. The requirements are:

① Build the ice cream from a set of selectable ingredients. ② Compute its total cost. ③ Build its description string.

EXAMPLE ORDER
Cherry (topping)
Blueberry Scoop
Choco Scoop
Choco Syrup
Orange Cone (base)

Each layer is a separate ingredient. The final ice cream is the composition of all layers.

The two interfaces every ingredient must implement are: getCost() — returns the cumulative price — and getDesc() — returns a description string listing all components.

02

Naïve Approach and Its Flaws

The first instinct is to create an IceCream class that holds a List<Cone> and a List<Topping>, with separate hierarchies for each. This leads to an explosion of concrete classes for every combination, and the IceCream class needs to know about both Cone and Topping types.

Problems With This Approach

① Class explosion: with 3 cone types and 4 toppings you get dozens of combinations. ② The getDesc() output depends on the order of addition, which a flat list can't capture cleanly. ③ You can't add ingredients "at runtime" — the list is a static structure, not a dynamic composition chain.

03

Decorator Pattern Defined

Definition

The Decorator Pattern attaches additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality. Each decorator wraps an existing object of the same interface and adds its own behaviour before or after delegating to the wrapped object.

The core mechanic is that each ingredient class implements the IceCream interface and holds a reference to another IceCream object. Calling getCost() on the outermost wrapper recursively adds costs down through the chain until it hits the base (or null).

THE WRAPPING CHAIN
Cherry
Vanilla Scoop
Choco Cone
Choco Syrup
Orange
Cone

Each outer ring wraps an inner ring. getCost() sums inward to the core; getDesc() builds the string inward to the core.

04

Step I — Define the IceCream Interface

Every ingredient — whether a base cone or an add-on — must implement the same IceCream interface. This is what makes the recursive wrapping possible.

Java — IceCream interface
// Step I: Define the interface all ingredients share
interface IceCream {
    double getCost();
    String getDesc();
}
Why One Interface?

Because a ChocoSyrup must be able to wrap an OrangeCone (or any other IceCream). If they had different types, a syrup couldn't hold a cone reference polymorphically. Unifying them under one interface is the key insight of the Decorator pattern.

05

Step II — Two Types of Ingredients

There are two categories of ingredient:

Base Addon (mandatory)

The starting object — has no inner IceCream reference (or holds null). Must always be present. Examples: OrangeCone, ChocolateCone.

  • OrangeCone
  • ChocolateCone
Normal Addon (optional)

Wraps an existing IceCream. Adds its own cost and name to whatever is already there. Examples: VanillaScoop, SBScoop, Cherry, ChocoSyrup.

  • VanillaScoop
  • SBScoop (Strawberry)
  • BBScoop (Blueberry)
  • Cherry
  • ChocoSyrup
Design Difference

Base Addon classes have a constructor that optionally accepts an IceCream reference (can be null). Normal Addon (decorator) classes require an IceCream argument in their constructor — there is no default/no-arg constructor. This enforces that every decorator must wrap something.

06

Step III — How Addons Compute Cost and Description

Every base addon (e.g. OrangeCone) handles the base case: if its inner ic reference is null, return just its own cost/name. Otherwise, add to what the inner object already has.

Every normal addon (decorator, e.g. ChocoSyrup) always has an ic object — it simply adds its own cost/description on top:

Base Addon — OrangeCone
getCost() {
  if (ic == null) return 5;
  else return ic.getCost() + 5;
}

getDesc() {
  if (ic == null) return "OC";
  return ic.getDesc() + "OC";
}
Normal Addon — ChocoSyrup
getCost() {
  return ic.getCost() + 10;
}

getDesc() {
  return ic.getDesc() + "ChocoSyrup";
}

The trace for ic = new ChocoSyrup(new OrangeCone()):

ChocoSyrup.getCost()
→ OrangeCone.getCost() + 10
→ (ic == null, so return 5) + 10
→ 15
07

Full Code + Nested Wrapping

Java — OrangeCone (Base Addon)
class OrangeCone implements IceCream {
    private IceCream ic;

    // Base constructor — no inner object
    OrangeCone() { this.ic = null; }

    // Also accepts a wrapping context (when used inside a chain)
    OrangeCone(IceCream ic) { this.ic = ic; }

    public double getCost() {
        if (ic == null) return 5;
        return ic.getCost() + 5;
    }

    public String getDesc() {
        if (ic == null) return "OC";
        return ic.getDesc() + " OC";
    }
}
Java — ChocoSyrup (Normal Addon / Decorator)
class ChocoSyrup implements IceCream {
    private IceCream ic;

    // No default constructor — must always wrap something
    ChocoSyrup(IceCream ic) { this.ic = ic; }

    public double getCost() {
        return ic.getCost() + 10;
    }

    public String getDesc() {
        return ic.getDesc() + " ChocoSyrup";
    }
}
Java — Building a full ice cream (nested constructors)
// Build: SBScoop(BBScoop(ChocolateCone(ChocoSyrup(OrangeCone()))))
IceCream ic = new SBScoop(
    new BBScoop(
        new ChocolateCone(
            new ChocoSyrup(
                new OrangeCone()
            )
        )
    )
);

// SBScoop.getCost() → BBScoop → ChocolateCone → ChocoSyrup → OrangeCone(5)
// 5 (OC) + 10 (CS) + 40 (CC) + 30 (BB) + 100 (SB) = 185
System.out.println(ic.getCost());   // 185.0
System.out.println(ic.getDesc());   // OC ChocoSyrup ChocolateCone BBScoop SBScoop

Decorator — When to Use

Rule of Thumb

Use the Decorator pattern when you have an entity to which you may want to add behaviours dynamically at runtime, especially when the set of possible combinations would cause a subclass explosion if handled through inheritance.

Other real-world examples of Decorator in use:

DomainBase EntityDecorators
UI componentsButtonBlueMargin, OrangeMargin, PurpleShade
Coffee orderingSimpleCoffeeMilk, Sugar, Caramel, WhipCream
Java I/O streamsFileInputStreamBufferedInputStream, GZIPInputStream, DataInputStream
Pizza orderingPlainPizzaTomatoTopping, CheeseTopping, MushroomTopping
Java I/O — Classic Example

Java's I/O library is the canonical Decorator implementation. BufferedReader(new FileReader(file)) wraps a FileReader (base) with buffering behaviour (decorator). You can keep stacking: new DataInputStream(new BufferedInputStream(new FileInputStream(file))).

The PUBG Memory Problem

Flyweight is a structural pattern that optimises memory when a large number of similar objects exist. Consider PUBG: a multiplayer battle game where each player carries up to 300 bullets. With 100 players in a match, you have up to 30,000 flying bullets active simultaneously — some scenarios can push this to 105 bullets across a session.

A straightforward Bullet class with all properties stored per object:

1.1 KB
10⁵ bullets × 1.1 KB
≈ 110 MB RAM

110 MB just for bullet state — for a mobile game this is unacceptable.

Problem

Despite having 100,000 bullets, the game only has 10 distinct bullet types (5.56mm, 7mm, 7.62mm, etc.). The image, weight, radius, length, colour, maxDamage, and maxRange are identical for all bullets of the same type. Storing them per-bullet is massive duplication.

10

Intrinsic vs Extrinsic Properties

Key Concepts

Intrinsic properties: values that are the same for all objects of a particular type — they don't change over the object's lifetime. These can be safely shared. Extrinsic properties: values that are unique to each object instance and change over time. These must be stored per instance.

Intrinsic (shared)
  • weight
  • radius
  • length
  • colour
  • maxDamage
  • maxRange
  • image (1 KB)

Total: ≈ 1 KB per bullet type
× 10 types = 10 KB

Extrinsic (per instance)
  • curCoordinate (24 B)
  • targetCoordinate (24 B)
  • speed (8 B)
  • direction (8 B)
  • bulletRef (8 B pointer)

Total: 72 B per flying bullet
× 105 bullets = 7.2 MB

11

Flyweight Solution

Definition

The Flyweight Pattern reduces memory usage by sharing as much data as possible with similar objects. When a class has both intrinsic and extrinsic state, split it into two classes: one that holds the shared intrinsic state (the Flyweight), and one that holds the per-instance extrinsic state (the context object).

We split Bullet into two classes: Bullet (intrinsic — one object per type, cached in a factory) and FlyingBullet (extrinsic — one object per active bullet in flight, holds a reference to its Bullet type).

Java — Bullet (intrinsic / shared Flyweight)
// Intrinsic state — shared; one instance per bullet TYPE
class Bullet {
    double weight, radius, length, maxDamage, maxRange;
    int colour;
    Image image;   // 1 KB — shared across all bullets of this type

    Bullet(double weight, double radius, double length,
           int colour, double maxDamage, double maxRange, Image img) {
        this.weight = weight; this.radius = radius; this.length = length;
        this.colour = colour; this.maxDamage = maxDamage;
        this.maxRange = maxRange; this.image = img;
    }
}
Java — FlyingBullet (extrinsic / per-instance context)
// Extrinsic state — unique per bullet in flight
class FlyingBullet {
    Coordinate curCoordinate;    // 24 B
    Coordinate targetCoordinate; // 24 B
    double speed;                 //  8 B
    double direction;             //  8 B
    Bullet bullet;               //  8 B reference — points to shared Bullet object
    // Total: 72 B per flying bullet
}
Java — BulletFactory (Flyweight cache)
class BulletFactory {
    private static Map<String, Bullet> cache = new HashMap<>();

    public static Bullet getBullet(String type) {
        if (!cache.containsKey(type)) {
            cache.put(type, createBullet(type));
        }
        return cache.get(type); // returns the SAME shared object
    }
}

Memory Calculation — Before vs After

Before Flyweight — 110 MB
10⁵ Bullet objects × 1.1 KB each
110 MB
After Flyweight — ≈ 8 MB
10⁵ FlyingBullet × 72 B = 7.2 MB
7.2 MB
10 Bullet types × 1 KB = 0.01 MB
0.01 MB

Total: ≈ 8 MB — a 93% reduction from 110 MB, achieved purely by sharing the intrinsic state.

When to Use Flyweight

Use Flyweight when a class has a large number of instances and those instances share a significant portion of their state (intrinsic properties). If you notice a lot of memory being wasted because of duplicated immutable data across objects, consider splitting the class and applying Flyweight.

Quick Recap

Decorator Intent

Add behaviours to a base entity at runtime through recursive wrapping. All decorators and the base share one common interface.

IceCream Interface

getCost() and getDesc() — every ingredient (cone, scoop, syrup, cherry) implements the same IceCream interface. This enables wrapping.

Base vs Normal Addon

Base addons can have a null inner reference (they are the start). Normal addons (decorators) always require an inner IceCream — no default constructor.

Recursive Delegation

getCost() = ic.getCost() + own cost. getDesc() = ic.getDesc() + own name. Each call chains inward until it hits the base.

Flyweight Intent

Share intrinsic (immutable, type-level) state across thousands of instances. Each instance only stores its extrinsic (unique, mutable) state plus a reference to the shared object.

Intrinsic Properties

Same for all instances of a type. Never change. Examples: bullet weight, radius, image, maxDamage. These belong in the shared Flyweight object.

Extrinsic Properties

Unique per instance, change over time. Examples: bullet's current position, speed, direction. These stay in the context (FlyingBullet) object.

PUBG Calculation

110 MB → 8 MB. 10⁵ FlyingBullet objects at 72 B each = 7.2 MB, plus 10 shared Bullet objects at ~1 KB each = 0.01 MB.

Flyweight Factory

A factory/cache stores one Bullet object per type. All FlyingBullets of the same type share the same Bullet reference.
