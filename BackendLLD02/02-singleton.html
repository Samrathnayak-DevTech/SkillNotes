Singleton Design Pattern

A creational pattern that guarantees a class has exactly one instance across the entire application, and provides a global access point to that instance.

01

What is the Singleton Pattern?

The Singleton pattern restricts the instantiation of a class to one object and ensures that the entire codebase shares that single instance. Any request for the object returns the same pre-existing instance rather than creating a new one.

Definition

Singleton ensures that a class has only one instance across the whole application, and provides a global, controlled access point to it.

Without Singleton, consider this scenario: three different service classes each hold a Database field. If each creates its own instance with new Database(), you have three separate database connection objects floating in memory — wasteful and potentially inconsistent.

02

When to Use Singleton

Two conditions typically justify Singleton:

Use Singleton when …

1. Object creation is expensive and you need to share the same instance. 2. The resource is inherently shared and having multiple instances would cause inconsistency or waste.

Classic examples in backend systems:

Database connection. A DatabaseConnection object stores a URL, username, password, and a pool of TCP connections to the database. Creating it is expensive (authentication, connection establishment). All services should reuse one instance.

Logger. Multiple services (UserService, ProductService, SearchService, CartService) all need to write to the same log server. A single Logger instance manages the connection to the log server and serialises writes from all services.

Config / properties manager. Application configuration loaded from a file or environment variables should be read once and shared.

03

Building Singleton Step-by-Step

We build up to the correct Singleton implementation in stages, identifying the flaw at each step.

Step 1 — Naive class (not Singleton)

A plain class with a public constructor. Anyone can call new DatabaseConnection() an unlimited number of times.

Java — Step 1: Not Singleton
class DatabaseConnection {
    String username;
    String password;
    String url;
    List<TCPConnection> connections;
    // public constructor → anyone can create unlimited instances
}

DatabaseConnection dbc1 = new DatabaseConnection(); // ✓ creates one
DatabaseConnection dbc2 = new DatabaseConnection(); // ✓ creates another — bad!

Step 2 — Private constructor

Make the constructor private. Now no external code can call new DatabaseConnection().

Java — Step 2: Private Constructor
class DatabaseConnection {
    String username, password, url;
    List<TCPConnection> connections;

    private DatabaseConnection() { } // ← locked
}
DatabaseConnection dbc1 = new DatabaseConnection(); // ✗ compile error

But now there is no way to get an instance at all. We need a way in.

Step 3 — Static getInstance(), but still broken

Add a non-static getInstance(). Still broken: you need an existing object to call it on.

Step 4 — Static getInstance() (correct signature, wrong body)

Making the method static means you can call it on the class itself: DatabaseConnection.getInstance(). But if the body just returns new DatabaseConnection() every time, it still creates multiple instances.

Java — Step 4: Static Method (still not singleton)
public static DatabaseConnection getInstance() {
    return new DatabaseConnection(); // ← still creates a new one each time
}

Step 5 — Lazy Initialization (single-threaded)

Store the single instance in a private static field (static so the method can access it). Create it only on the first call, then return the stored reference on every subsequent call.

Java — Step 5: Correct Single-Threaded Singleton
class DatabaseConnection {
    private static DatabaseConnection dbc = null; // ← static field

    String username, password, url;
    List<TCPConnection> connections;

    private DatabaseConnection() { }

    public static DatabaseConnection getInstance() {
        if (dbc == null) {          // first call: create
            dbc = new DatabaseConnection();
        }
        return dbc;               // all calls: return stored instance
    }
}
// Usage
DatabaseConnection dbc = DatabaseConnection.getInstance();
Why static field?

A static method can only access static members of its class. The stored instance dbc must therefore be declared static so getInstance() can read and write it without needing an object reference.

04

The Multithreading Problem

Step 5's implementation is not thread-safe. Consider two threads T1 and T2 calling getInstance() simultaneously for the very first time:

Race condition timeline
T1 checks: dbc == null → true
T2 checks: dbc == null → true (T1 hasn't written yet)
T1 creates new DatabaseConnection() — instance 1
T2 creates new DatabaseConnection() — instance 2 ← BAD

Both threads see dbc == null before either writes the new instance. Result: two instances created, breaking the Singleton guarantee. All three thread-safe approaches below solve this problem.

05

Approach 1 — Eager Initialization

Create the instance at class-load time — before any thread can request it. Because the JVM handles class loading in a thread-safe manner, no race condition is possible.

Java — Eager Initialization
class DatabaseConnection {
    // Instance created at class-load time — thread-safe by JVM guarantee
    private static final DatabaseConnection dbc = new DatabaseConnection();

    String username, password, url;
    List<TCPConnection> connections;

    private DatabaseConnection() { }

    public static DatabaseConnection getInstance() {
        return dbc; // T1 and T2 both safely get the same pre-built instance
    }
}
Cons of Eager Initialization

1. Startup time increase: The instance is created even if it is never used, lengthening application boot time. 2. Cannot pass constructor arguments: Since the instance is created during static initialisation (before any caller can supply values), runtime configuration cannot be passed to the constructor.

06

Approach 2 — Synchronized Lazy Initialization

Keep lazy initialization but mark getInstance() as synchronized. This forces threads to acquire a lock before entering the method — only one thread executes the check-and-create at a time.

Java — Synchronized getInstance()
public static synchronized DatabaseConnection getInstance() {
    if (dbc == null) {
        dbc = new DatabaseConnection();
    }
    return dbc;
}
Con: Low Performance

Synchronization is needed only once — during the very first call when the instance does not yet exist. After that, every subsequent getInstance() call unnecessarily acquires and releases the lock. In high-throughput systems this becomes a significant bottleneck.

07

Approach 3 — Double-Checked Locking ✓ Best

The production-grade solution: check once before locking (to avoid unnecessary synchronization on every call), lock, then check again inside the lock (to handle the race condition during the first initialisation).

Java — Double-Checked Locking
class DatabaseConnection {
    private static volatile DatabaseConnection dbc = null;

    String username, password, url;
    List<TCPConnection> connections;

    private DatabaseConnection() { }

    public static DatabaseConnection getInstance() {
        if (dbc == null) {                  // Step 1: check without lock
            lock();                           // Step 2: acquire lock
            if (dbc == null) {              // Step 3: check again inside lock
                dbc = new DatabaseConnection();
            }
            unlock();
        }
        return dbc;
    }
}

Why the Second Check?

Imagine T1 and T2 both pass the first null check simultaneously. T1 acquires the lock and creates the instance. T1 releases the lock. T2 acquires the lock. If there were no second check, T2 would create a second instance. The inner check prevents this.

Why volatile?

Without volatile, due to CPU instruction reordering, a thread might see a partially constructed object referenced by dbc before the constructor has fully completed. The volatile keyword prevents instruction reordering and ensures all threads see the most recently written value from main memory.

Java Idiom: Initialization-on-Demand Holder

Java has a JVM-guaranteed thread-safe lazy singleton using a private static inner class. The inner class is not loaded until getInstance() is called, and the JVM handles thread safety during class loading automatically. This is cleaner than explicit locking in Java.

Java — Initialization-on-Demand Holder (Bonus Pattern)
class DatabaseConnection {
    private DatabaseConnection() { }

    private static class Holder {
        static final DatabaseConnection INSTANCE = new DatabaseConnection();
    }

    public static DatabaseConnection getInstance() {
        return Holder.INSTANCE; // Holder class loaded lazily on first call
    }
}
08

Pros & Cons

Pros

Resource efficiency: Expensive objects (database connections, loggers) are created only once and shared, saving memory and CPU. Ensures a single consistent state for shared resources.

Cons

Hard to test: Because a Singleton holds global state, unit tests can interfere with each other — one test's side-effects on the singleton affect subsequent tests. You typically need to reset or mock the singleton between tests, which is awkward. This is a known violation of the DIP (you cannot inject a different implementation easily) unless you use an interface.

✦

Quick Recap

Singleton

Guarantees one instance of a class across the whole application. Provides a global access point via a static factory method.

Private constructor

The first step: make the constructor private so external code cannot call new ClassName() directly.

Static field + getInstance()

Store the one instance in a private static field. Return it from a public static getInstance() method.

Thread-safety problem

In a multithreaded environment, two threads can both see null and both create an instance simultaneously — breaking the guarantee.

Eager initialization

Create the instance at class-load time. Thread-safe by JVM contract. Con: increases startup time; can't pass constructor args.

Synchronized method

Wrap the entire method in a lock. Thread-safe but slow — every call pays the lock overhead even after the instance exists.

Double-checked locking

Check → lock → check again. Only synchronizes during first creation. Use volatile to prevent reordering. Best production approach.

Use cases

Database connections, loggers, configuration managers, thread pools — any expensive or shared-state resource.
