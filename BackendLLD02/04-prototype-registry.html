Prototype & Registry Patterns

Two complementary creational patterns: Prototype lets objects clone themselves cleanly without exposing internals; Registry stores named prototypes so new objects are created by copying a pre-configured template.

01

The Problem: Copying Objects

Sometimes you have an existing object and need to create a new one in memory with the exact same attribute values — a deep copy. The naïve approach is to write copy code in the client:

Java — Naïve copy at the call site
// st1 is an existing Student object
Student st2 = new Student();
st2.name = st1.name;
st2.age  = st1.age;
// … repeat for every field
Cons

1. Too much boilerplate code. Every field must be copied manually; adding a new field to the class means updating all copy sites. 2. Private fields are inaccessible. The client cannot copy fields marked private, so a complete copy is impossible from outside the class.

02

Copy Constructor

A better step: add a copy constructor — a constructor that takes an existing instance of the same class and copies all fields into the new instance.

Java — Copy Constructor
class Student {
    String name;
    int age;

    // Copy constructor
    Student(Student st) {
        this.name = st.name;
        this.age  = st.age;
        // private fields are accessible here — inside the class
    }
}
// Usage
Student copy = new Student(st1);  // clean, no manual field access

This is good but still has a problem when inheritance is involved.

03

Why the Client Shouldn't Copy

Suppose Student has a subclass IntelligentStudent. If the client holds a Student reference that may actually point to an IntelligentStudent object, it must check the runtime type to make the right copy:

Java — Client-side type checking violates OCP
Student st;  // could be Student or IntelligentStudent at runtime

Student copy;
if (st instanceof Student) {
    copy = new Student(st);
} else if (st instanceof IntelligentStudent) {
    copy = new IntelligentStudent((IntelligentStudent) st);
}
// Adding a new subclass (e.g., ScholarStudent) forces editing this block — OCP violation
Problems

Putting the copy logic on the client is error-prone and violates OCP. Every new subclass forces the client code to be updated. The internal details of the class are also leaked to the client.

Ideal Solution

The object itself should know how to copy itself. Expose a simple copy() (or clone()) method to the client, and let all the complex logic be handled inside the class hierarchy.

04

The copy() / clone() Method

Move the copy logic into the class. Each class provides a copy() method that creates and returns a complete copy of itself — including private fields.

Java — copy() inside the class
class Student {
    private String name;
    private int    age;

    Student copy() {
        Student copyObj = new Student();
        copyObj.name = this.name;  // private field — accessible inside own class
        copyObj.age  = this.age;
        return copyObj;
    }
}

// Client code — clean and simple
Student copyObj = st.copy();  // no type checks, no instanceof

Pros: no OCP violation — adding a new subclass only requires adding a copy() to the new class; the client never changes. Private fields are never exposed externally.

05

Runtime Polymorphism & the Critical Note

The magic is in runtime polymorphism. When the client calls st.copy() on a reference of type Student that actually holds an IntelligentStudent, Java dispatches to IntelligentStudent.copy() — producing the correct subtype copy automatically.

Java — IntelligentStudent overrides copy()
class IntelligentStudent extends Student {
    private int iq;

    IntelligentStudent copy() {         // override with covariant return type
        IntelligentStudent is = new IntelligentStudent();
        is.name = this.name;
        is.age  = this.age;
        is.iq   = this.iq;
        return is;
    }
}
Critical Note

All child classes must override the copy() method. If a subclass forgets to override it, calling copy() on that subclass will invoke the parent's copy(), which only copies the parent's fields — silently producing an incomplete, partially-copied object. This is a common and subtle bug.

06

The Prototype Design Pattern

Definition

The Prototype pattern uses an existing object (the prototype) as a blueprint for creating new objects by copying it. The prototype knows how to clone itself, hiding the cloning complexity from the client.

The name "prototype" comes from the industrial concept: you create one master template (the prototype), then stamp out copies of it. In software terms, it is a blueprint-and-copy mechanism.

Use Prototype when:

You need to create many objects that share most of their attributes, with only a few differences. Constructing the object from scratch each time would be expensive or complex. You want to decouple the client from the concrete classes it instantiates.

Consider a printing company that needs to create 100,000 notebooks of a specific type: A4 size, ruled, 100 pages. The Notebook class has many attributes, some shared across all copies and some unique per notebook.

Way 1 (Naïve): Create each of the 100,000 notebooks from scratch with new Notebook() and set all 5 common fields every single time — 500,000 setter calls of repetitive code.

Way 2 (Prototype): Create one prototype object with all the common fields set. Then create each notebook as prototype.copy() and only set the unique fields (frontPage, lastPage) on the individual copy. Common fields come along for free.

Java — Prototype approach
// Build the prototype once
Notebook a4RuledPrototype = new Notebook();
a4RuledPrototype.setWidth(100);
a4RuledPrototype.setHeight(50);
a4RuledPrototype.setNumPages(100);
a4RuledPrototype.setType("Ruled");
a4RuledPrototype.setMrp(60.0);

// Mass-produce copies — only set the unique fields
for (int i = 0; i < 100_000; i++) {
    Notebook nb = a4RuledPrototype.copy();
    nb.setFrontPage(generateFrontPage(i));
    nb.setLastPage(generateLastPage(i));
    printNotebook(nb);
}
08

The Registry Pattern

Prototype tells objects how to copy themselves. Registry is the companion pattern that answers: where do you store the prototypes so they can be found efficiently?

Definition

A Registry is a centralised store (typically a HashMap<String, PrototypeObject>) that maps string keys (the prototype's name) to pre-configured prototype objects. When you need a new object, you look up the appropriate prototype by key, call copy(), and customise the result.

The Registry supports two operations:

register(key, prototype) — saves a named prototype into the map.

get(key) — retrieves the prototype for copying.

Analogy

Think of a stamp collection office. Each named stamp design (prototype) is stored in a drawer labelled with its name. When you need 500 "First Class" envelopes, you pull the "First Class" stamp from the drawer and press it 500 times, personalising each envelope's address separately. The drawer is the Registry; the stamp is the prototype.

09

Full Implementation — Registry with Student

Consider a learning platform with multiple course batches. Each batch has a fixed university name, batch name, and average PSP — but each individual student has their own name and age. Instead of setting the common fields for every new student, we store one prototype per batch in a Registry.

Java — Registry Implementation
// ── Registry class ───────────────────────────────────────────
class StudentRegistry {
    private HashMap<String, Student> map = new HashMap<>();

    void register(String key, Student st) {
        map.put(key, st);
    }

    Student get(String key) {
        return map.get(key);   // caller should call .copy() on result
    }
}

// ── Setup: create one prototype per batch ────────────────────
Student mar24Prototype = new Student();
mar24Prototype.setBatch("Mar24-MWFLLD");
mar24Prototype.setAvgBatchPsp(65.0);
mar24Prototype.setUnivName("Scaler");

StudentRegistry registry = new StudentRegistry();
registry.register("Mar24MWFLLD", mar24Prototype);

// ── Usage: create individual student from batch prototype ─────
Student shubham = registry.get("Mar24MWFLLD").copy();
shubham.setName("Shubham");
shubham.setAge(24);
// batch, avgBatchPsp, univName are already set — came from the prototype copy

Another example using a Search API — building multiple search requests that share protocol, domain, and type, differing only in the query string:

Java — SearchAPI with Registry
// Way 1: repetitive setup for each request (bad)
SearchAPI api1 = new SearchAPI();
api1.setProtocol("https"); api1.setDomain("google.com");
api1.setType("search");    api1.setQuery("India");

SearchAPI api2 = new SearchAPI(); // exact same three setup lines duplicated
api2.setProtocol("https"); api2.setDomain("google.com");
api2.setType("search");    api2.setQuery("World Cup");

// Way 2: Registry + Prototype (good)
SearchAPI googleSearchPrototype = new SearchAPI();
googleSearchPrototype.setProtocol("https");
googleSearchPrototype.setDomain("google.com");
googleSearchPrototype.setType("search");
// Store this prototype in a Registry …

SearchAPI api = googleSearchPrototype.copy();
api.setQuery("India");   // only the unique field

Implementation Steps

Follow these three steps to apply the Prototype + Registry pattern:

Step 1
Identify the class for which you want to use Prototype, and implement a copy() (or clone()) method inside it. Remember: all child classes must also implement the method.
Step 2
Create a Registry (a HashMap<String, T> wrapper) to store prototype objects by a meaningful string key. Populate it with one pre-configured prototype per variant.
Step 3
When a new object is needed, fetch the appropriate prototype from the Registry, call .copy() on it, then set only the attributes that differ for this specific instance.

Quick Recap

Prototype Pattern

Objects clone themselves via a copy() method. Hides copy complexity, supports polymorphism, keeps private fields internal.

copy() / clone()

Implemented inside the class. Creates a new object with all the same field values. All child classes must override it.

Copy Constructor

A constructor that takes an instance of the same class and copies its fields. Works well for a single class, but falls apart with inheritance.

OCP & Prototype

With copy() on the class itself, adding a new subclass never requires changes to client code — the client always just calls .copy().

When to Use Prototype

When you need many objects with only a few different attributes and many shared ones. Avoids expensive re-construction from scratch each time.

Registry Pattern

A HashMap<String, Prototype> that stores named prototype objects. Clients look up a prototype by key, copy it, then set unique fields.

Registry Operations

register(key, prototype) saves a template. get(key).copy() retrieves and clones it. Clean separation between template management and object creation.

Critical Warning

Every subclass must override copy(). Forgetting results in a partially copied object — a silent, hard-to-debug error.
