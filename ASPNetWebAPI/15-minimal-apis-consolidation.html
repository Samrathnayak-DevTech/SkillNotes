Minimal APIs & Consolidation 🏁

Twelve days ago an endpoint meant a controller class, attributes, and boilerplate. Minimal APIs strip that to the essence: a route, a lambda, done — while keeping everything you've learned (validation, DTOs, results, DI, EF Core, paging, docs, error handling, auth, versioning). Today we rebuild one resource the minimal way, wire together the entire series into a single versioned, documented, secured CRUD API, and ship it to GitHub. This is graduation. 🎓

01

Minimal APIs — What & Why 🪶

A Minimal API defines HTTP endpoints directly on the WebApplication with a route and a handler function — no controller class, no attributes required. Introduced in .NET 6 and matured every release since, it's the leaner counterpart to the controller model (Day 3 onward). Both run on the same ASP.NET Core foundation; they're two front-ends to the same routing, DI, and middleware.

C# — the entire app, in a few lines
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/hello", () => "Hello, world!");

app.Run();
🍽️ Analogy — à la carte vs. prix fixe

Controllers are a prix-fixe menu: a full place setting arrives whether or not you need every fork. Minimal APIs are à la carte — you take exactly the pieces this endpoint needs and nothing else. Same kitchen, same ingredients; just less on the table you didn't order.

🔑 It's not a different framework

Minimal APIs aren't a "lite" ASP.NET Core — they share the same request pipeline, dependency injection (Day 7), model binding, validation, authentication (Day 12), and OpenAPI (Day 10). What changes is how you declare endpoints, not what the platform can do. Everything from Days 4–14 still applies.

02

Core Syntax & Binding 🧱

Each verb has a Map* method taking a route template and a handler. Parameters are bound automatically by source: route-template names come from the path, complex types from the JSON body, and registered services from DI — all inferred, no attributes needed in the common case.

C# — the four verbs & automatic binding
// route param "id" binds from the path; ISomething comes from DI
app.MapGet("/products/{id:int}", (int id, IProductService svc) =>
    svc.GetAsync(id));

// query string: ?page=2 binds to the "page" parameter
app.MapGet("/products", (int page, IProductService svc) =>
    svc.ListAsync(page));

// complex type binds from the JSON request body
app.MapPost("/products", (CreateProductDto dto, IProductService svc) =>
    svc.CreateAsync(dto));

app.MapPut("/products/{id:int}", (int id, UpdateProductDto dto, IProductService svc) =>
    svc.UpdateAsync(id, dto));

app.MapDelete("/products/{id:int}", (int id, IProductService svc) =>
    svc.DeleteAsync(id));

Name matches a route token ({id})The URL path
Simple type, no route matchThe query string
Complex type (a class/record)The JSON request body
A registered service typeDependency injection
HttpContext, ClaimsPrincipal, CancellationTokenSupplied by the framework
✅ Group many params with [AsParameters]

When a handler needs a lot of query parameters (like the paging/filter/sort inputs from Day 9), bundle them into a struct or record and mark it [AsParameters] — the framework binds each property from its matching source, keeping the signature clean. It's the Minimal API counterpart to the bound query DTO you used with [FromQuery].

⚠️ You can still be explicit

Inference covers most cases, but when it's ambiguous you annotate: [FromQuery], [FromRoute], [FromBody], [FromHeader], [FromServices]. For example, a string you want from the body (not the query) needs [FromBody]. Reach for these only when the default guess would be wrong.

03

Results & TypedResults 📤

Day 6's return-type lessons land squarely here. A handler can return a plain object (serialized as 200 JSON), or an IResult for control over status codes. Prefer TypedResults — the concrete, strongly-typed factory — over the older Results, because it enables testing and feeds OpenAPI metadata automatically.

C# — TypedResults with a union return type
// Results<...> declares every outcome — self-documenting & compile-checked
app.MapGet("/products/{id:int}", async (int id, IProductService svc)
    : Task<Results<Ok<ProductDto>, NotFound>> =>
{
    var product = await svc.GetAsync(id);
    return product is null
        ? TypedResults.NotFound()          // 404
        : TypedResults.Ok(product);        // 200 + body
});
🔑 Why the Results<T1,T2> union matters

Declaring Results<Ok<ProductDto>, NotFound> tells the framework — and every reader — exactly which responses this endpoint can produce. OpenAPI (Day 10) infers the 200 and 404 automatically, with no [ProducesResponseType] to hand-write, and the compiler stops you returning anything outside the declared set. Self-documenting and checked.

⚠️ TypedResults skips content negotiation

As noted on Day 14: HttpResults (what TypedResults produces) write the response directly and bypass the output formatters, so they don't honor the Accept header. That's fine for a JSON API — but if you genuinely need XML/JSON negotiation on a Minimal API, you'd return a plain object rather than a TypedResults value.

04

Endpoint Groups 📦

Repeating /api/products on every endpoint — and re-applying auth, tags, and versioning each time — gets old fast. MapGroup creates a route group: a shared prefix plus shared metadata that every endpoint in the group inherits.

C# — a group with shared prefix & metadata
var products = app.MapGroup("/api/products")
    .WithTags("Products")             // OpenAPI grouping (Day 10)
    .RequireAuthorization();          // every endpoint needs auth (Day 12)

// routes below are RELATIVE to the group prefix
products.MapGet("/", GetAll);
products.MapGet("/{id:int}", GetById);
products.MapPost("/", Create).RequireAuthorization("AdminOnly"); // tighten one
products.MapPut("/{id:int}", Update);
products.MapDelete("/{id:int}", Delete);
✅ Groups nest and layer

Anything you can put on an endpoint you can put on a group: .WithTags, .RequireAuthorization, .RequireCors (Day 14), .CacheOutput (Day 14), .ProducesProblem (Day 10), and endpoint filters. Groups can even nest — a versioned outer group containing a resource inner group — so cross-cutting concerns are declared once at the right level.

🔑 Endpoint filters — the Minimal API "action filter"

Minimal APIs support AddEndpointFilter, letting you run logic before/after a handler — validation, logging, timing — across a whole group. It's the leaner cousin of MVC action filters, and a clean place to centralize a concern (like DTO validation) for every endpoint in a group at once.

05

Splitting Into Files 🗂️

The one fair criticism of Minimal APIs — "everything piles into Program.cs" — is solved by convention, not framework magic. Move each resource's endpoints into an extension method on IEndpointRouteBuilder, with handlers as named static methods. Program.cs stays a table of contents.

C# — ProductEndpoints.cs
public static class ProductEndpoints
{
    public static IEndpointRouteBuilder MapProductEndpoints(
        this IEndpointRouteBuilder app)
    {
        var group = app.MapGroup("/products").WithTags("Products");

        group.MapGet("/{id:int}", GetById);
        group.MapPost("/", Create);
        return app;
    }

    // named handlers = unit-testable like controller actions
    public static async Task<Results<Ok<ProductDto>, NotFound>> GetById(
        int id, IProductService svc)
    {
        var dto = await svc.GetAsync(id);
        return dto is null ? TypedResults.NotFound() : TypedResults.Ok(dto);
    }

    public static async Task<Created<ProductDto>> Create(
        CreateProductDto dto, IProductService svc)
    {
        var created = await svc.CreateAsync(dto);
        return TypedResults.Created($"/products/{created.Id}", created);
    }
}
C# — Program.cs stays a clean index
var app = builder.Build();

app.MapProductEndpoints();   // one line per resource
app.MapOrderEndpoints();

app.Run();
✅ Named handlers are testable

Because GetById and Create are ordinary static methods, you unit-test them exactly like controller actions — call the method with a fake service, assert on the returned TypedResults type and value. The "Minimal APIs can't be tested" myth comes from inlining logic into lambdas; named handlers dissolve it.

06

Rebuild the Resource as a Minimal API 🔧

Today's first "Do": take the Products resource you built with controllers (Day 8's CRUD) and rebuild it minimally — versioned, secured, documented, all in one group. This is the whole series in one artifact.

C# — Program.cs (the composition root)
using Asp.Versioning;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<AppDbContext>(o =>
    o.UseSqlServer(builder.Configuration.GetConnectionString("Default")));
builder.Services.AddScoped<IProductService, ProductService>();  // Day 7
builder.Services.AddApiVersioning();                                // Day 14
builder.Services.AddAuthentication().AddJwtBearer();               // Day 12
builder.Services.AddAuthorization();                               // Day 13
builder.Services.AddProblemDetails();                              // Day 11
builder.Services.AddExceptionHandler<GlobalExceptionHandler>();  // Day 11
builder.Services.AddOpenApi();                                     // Day 10

var app = builder.Build();

app.UseExceptionHandler();          // Day 11 — first
app.MapOpenApi();                   // Day 10
app.UseAuthentication();
app.UseAuthorization();            // Days 12–13

app.MapProductEndpoints();
app.Run();
C# — the versioned, secured, documented CRUD group
public static IEndpointRouteBuilder MapProductEndpoints(
    this IEndpointRouteBuilder app)
{
    var versionSet = app.NewApiVersionSet()          // Day 14
        .HasApiVersion(new ApiVersion(1, 0))
        .HasApiVersion(new ApiVersion(2, 0))
        .ReportApiVersions()
        .Build();

    var group = app
        .MapGroup("/api/v{version:apiVersion}/products")
        .WithApiVersionSet(versionSet)
        .WithTags("Products")                      // Day 10 — docs
        .RequireAuthorization()                   // Days 12–13
        .ProducesProblem(StatusCodes.Status500InternalServerError);

    group.MapGet("/", ListAsync);                  // paging/filter/sort — Day 9
    group.MapGet("/{id:int}", GetAsync);
    group.MapPost("/", CreateAsync)
         .MapToApiVersion(2, 0);              // v2-only shape
    group.MapPut("/{id:int}", UpdateAsync);
    group.MapDelete("/{id:int}", DeleteAsync)
         .RequireAuthorization("AdminOnly");      // tighter policy — Day 13

    return app;
}
🔑 Read that group declaration again

In roughly ten lines, that one group is versioned (v1 & v2), secured (auth required, admin-only delete), documented (tagged, problem responses declared), and error-safe (global handler upstream). Every concern from the series is present — declared once, at the level where it belongs. That density is the reward for the last eleven days.

🔑 The mental model to keep

A request enters, passes cross-cutting middleware (errors, auth), routes to a versioned endpoint, binds & validates its input, delegates to a service that talks to EF Core, and returns a typed result shaped as a DTO — with the whole contract documented in OpenAPI. Every arrow in that sentence is a day you've done. The API is not twelve topics; it's one system with twelve facets.

🏛️ Analogy — you've built the whole house

Foundations (routing, DI), plumbing and wiring (cross-cutting concerns), locks on the doors (auth), a floor plan others can read (OpenAPI), and rooms that do their jobs (CRUD endpoints). Early days felt like isolated bricks. Today you can stand in the finished building.

08

Ship It — Push to GitHub 🐙

Today's second "Do": get the project into version control and up on GitHub. Two things first — a proper .gitignore so build artifacts and secrets don't get committed, and a README so the repo explains itself.

bash — generate a .NET-aware .gitignore
# The .NET SDK ships a curated ignore file — use it, don't hand-write one
dotnet new gitignore

# It excludes bin/, obj/, .vs/, *.user, and other build/IDE noise
bash — initialize, commit, and push
git init
git add .
git commit -m "Initial commit: versioned, secured, documented Products API"
git branch -M main

# create the empty repo on github.com first, then wire the remote:
git remote add origin https://github.com/<you>/products-api.git
git push -u origin main
✅ Or let the GitHub CLI do it in one step

With the GitHub CLI installed and authenticated (gh auth login), gh repo create products-api --public --source=. --push creates the remote repo and pushes your current commit in a single command — no clicking around the website. Handy once you've done the manual flow enough to understand it.

⚠️ Never commit secrets

Your JWT signing key and database connection string (Days 8, 12) must not land in the repo. Keep them out of appsettings.json: use user-secrets in development (dotnet user-secrets set "Jwt:Key" "…", stored outside the project tree) and environment variables or a secrets vault in production. If a secret ever is committed, rotate it — Git history remembers, so deleting the line later isn't enough. This is the single most common way beginners leak credentials.

🔑 What belongs in the README

A good API README states: what the service does, prerequisites (.NET version, SQL Server), how to configure secrets, how to run migrations and start it (dotnet run), where the docs live (the OpenAPI/Scalar URL from Day 10), and how to authenticate (get a token, send the Bearer header from Day 12). Someone should go from clone to a working call without asking you a question.

A Request, End to End 🔄

To close the loop, trace one authenticated POST /api/v2/products through the finished Minimal API — every layer of the series, in order.

Step 1 · 🧵 middleware
Exception handler, then auth

The request passes the global exception handler (Day 11), then UseAuthentication validates the JWT and builds the user (Day 12).

Step 2 · 🗺️ route + version
Matched to the v2 handler

Routing resolves v2 from the URL and selects CreateAsync; the group's RequireAuthorization confirms access (Days 13–14).

Step 3 · 🎁 bind + validate
Body → DTO, checked

The JSON body binds to CreateProductDto and its data annotations validate (Days 4–5); invalid input short-circuits to a 400.

Step 4 · 🧠 service + EF Core
Business logic → database

The handler calls IProductService (Day 7), which persists via AppDbContext and SaveChangesAsync (Day 8).

Step 5 · 📤 typed result
201 Created + DTO

TypedResults.Created returns a 201 with the response DTO and Location header — a shape already described in the OpenAPI doc (Days 6, 10).

11

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1What is a Minimal API?🌱 Junior

A way to define HTTP endpoints directly on the WebApplication with a route and a handler function, without a controller class or attributes. Introduced in .NET 6, it reduces boilerplate for smaller services and microservices. It runs on the same ASP.NET Core platform as controllers — same routing, DI, middleware, and auth.

Q2How do you define a GET endpoint in a Minimal API?🌱 Junior

Call app.MapGet("/route", handler), where the handler is a lambda or method. For example, app.MapGet("/products/{id:int}", (int id) => ...). There are matching methods for the other verbs: MapPost, MapPut, MapDelete.

Q3How does a handler get a route parameter, query value, or the request body?🌱 Junior

By parameter binding convention: a parameter whose name matches a route token ({id}) comes from the path, a simple type with no route match comes from the query string, and a complex type (a class or record) comes from the JSON body. You can be explicit with [FromRoute], [FromQuery], [FromBody] when needed.

Q4What is an endpoint group?🌱 Junior

A set of endpoints sharing a route prefix and metadata, created with app.MapGroup("/prefix"). Endpoints mapped on the group have routes relative to the prefix and inherit anything applied to the group — tags, authorization, versioning — so you configure cross-cutting settings once instead of per endpoint.

Q5How do you use a service (dependency injection) in a Minimal API handler?🌱 Junior

Just add it as a handler parameter — (int id, IProductService svc) => .... The framework resolves any registered service type from DI automatically (Day 7). You rarely need [FromServices]; it's inferred. This is the same container you registered services in via builder.Services.

Q6What's the main difference between Minimal APIs and controllers?🌱 Junior

How you declare endpoints. Controllers use classes, attributes, and convention-based discovery; Minimal APIs use explicit Map* calls with handler functions and less ceremony. They share the same underlying platform, so both support DI, validation, auth, and OpenAPI — the choice is about structure and preference, not capability.

Q7What is a .gitignore, and why does a .NET project need one?🌱 Junior

A file listing paths Git should not track. A .NET project produces build output (bin/, obj/) and IDE files that shouldn't be committed — they're regenerated and would bloat the repo. Run dotnet new gitignore to get the official .NET ignore file. It also helps keep secret files out of version control.

Q8What are the basic git commands to push a project to GitHub?🌱 Junior

git init to create the repo, git add . to stage files, git commit -m "message" to save a snapshot, git branch -M main to name the branch, git remote add origin <url> to link the GitHub repo, and git push -u origin main to upload. You create the empty repo on GitHub first (or use the gh CLI to do both at once).

Q9How do you return different status codes from a Minimal API handler?🟠 Mid

Return an IResult via the Results or TypedResults factories: TypedResults.Ok(dto), TypedResults.NotFound(), TypedResults.Created(uri, dto), TypedResults.BadRequest(), etc. To declare multiple possible outcomes, use a union return type like Results<Ok<ProductDto>, NotFound>, which is compile-checked and self-documenting.

Q10TypedResults vs. Results — what's the difference and which is preferred?🟠 Mid

Results returns an opaque IResult; TypedResults returns concrete strongly-typed values like Ok<T>. TypedResults is preferred because it makes handlers unit-testable (you can assert on the concrete return type) and automatically contributes response metadata to OpenAPI. Combined with a Results<T1, T2> union, it documents every outcome without manual attributes.

Q11How do you apply authorization or tags to a whole group of endpoints?🟠 Mid

Chain the metadata onto the group: app.MapGroup("/api/products").WithTags("Products").RequireAuthorization(). Every endpoint mapped on that group inherits it. You can still tighten an individual endpoint — e.g. .RequireAuthorization("AdminOnly") on the delete — and groups can nest to layer concerns at different levels.

Q12How do you version Minimal API endpoints?🟠 Mid

With the Asp.Versioning.Http package: build a version set with app.NewApiVersionSet().HasApiVersion(new ApiVersion(1,0)).HasApiVersion(new ApiVersion(2,0)).Build(), attach it to a group or endpoint via .WithApiVersionSet(set), and bind specific endpoints to a version with .MapToApiVersion(2, 0). A URL-segment route like /api/v{version:apiVersion}/products then routes v1 and v2 correctly.

Q13How do you keep Minimal APIs from cluttering Program.cs?🟠 Mid

Move each resource's endpoints into an extension method on IEndpointRouteBuilder in its own file (e.g. ProductEndpoints.MapProductEndpoints), with the handlers as named static methods. Program.cs then just calls app.MapProductEndpoints() per resource, staying a readable index. This also makes the handlers unit-testable.

Q14How do you secure a Minimal API endpoint?🟠 Mid

Call .RequireAuthorization() on the endpoint or its group to require an authenticated user, or .RequireAuthorization("PolicyName") / .RequireAuthorization(p => p.RequireRole("Admin")) for role/policy rules (Days 12–13). Ensure UseAuthentication and UseAuthorization are in the pipeline. Use .AllowAnonymous() to exempt specific endpoints like login.

Q15How do Minimal API endpoints show up in OpenAPI/Swagger?🟠 Mid

The native Microsoft.AspNetCore.OpenApi package (Day 10) discovers them automatically. You enrich the docs with fluent metadata — .WithTags, .WithSummary, .WithDescription, .Produces<T>(), .ProducesProblem() — and returning TypedResults or a Results<...> union lets responses be inferred. (With Swashbuckle you'd also call AddEndpointsApiExplorer.)

Q16What should a good API project README contain?🟠 Mid

What the service does; prerequisites (.NET version, database); how to configure secrets (user-secrets/env vars); how to run migrations and start it; where the API docs live (the OpenAPI/Scalar URL); and how to authenticate. The goal is that a new developer can go from git clone to a successful authenticated request without asking the author anything.

Q17When would you choose Minimal APIs over controllers for a real project?🔴 Senior

Minimal APIs suit smaller services, microservices, and high-throughput or AOT-targeted APIs where you want less ceremony and lower overhead. Controllers still shine for large apps with many endpoints that benefit from convention-based organization, model binding features, filters, and teams already fluent in MVC. It's not capability — both do DI, auth, validation, and docs — it's a fit-to-context and team-familiarity decision. Many codebases mix both.

Q18How do you unit test Minimal API endpoints?🔴 Senior

Write handlers as named static methods rather than inline lambdas, then call them directly in a test with fake dependencies and assert on the returned TypedResults type and value — exactly like testing a controller action. For full-stack coverage, add integration tests with WebApplicationFactory<Program> (Day 8) that send real HTTP requests and assert status codes and bodies. The "Minimal APIs aren't testable" claim only applies to logic buried in lambdas.

Q19What are endpoint filters, and how do they compare to MVC action filters?🔴 Senior

Endpoint filters (AddEndpointFilter) run logic before and after a Minimal API handler — inspecting or short-circuiting the request, transforming the result — and can be applied to a single endpoint or a whole group. They're the Minimal API analog of MVC action filters, useful for cross-cutting concerns like input validation, logging, or timing. They compose in order and can pass control down the chain, similar to middleware but scoped to endpoints with access to the handler's arguments.

Q20How do you keep secrets like connection strings and signing keys out of the repo?🔴 Senior

Don't put them in committed config. In development use the Secret Manager (dotnet user-secrets), which stores values outside the project directory; in production use environment variables or a managed secrets store (Azure Key Vault, AWS Secrets Manager). The .gitignore should exclude any local secrets files. Critically, if a secret is ever committed, rotate it — Git history preserves it even after you delete the line, so removal alone doesn't undo the exposure.

Q21Walk through how a request flows through the consolidated API.🔴 Senior

It enters the middleware pipeline: the global exception handler wraps everything (Day 11), then authentication validates the JWT and builds the user (Day 12). Routing matches the versioned endpoint and authorization enforces the group's policy (Days 13–14). The body binds to a DTO and validates (Days 4–5); on failure it short-circuits to a 400. The handler delegates to a service (Day 7) that uses EF Core to hit the database (Day 8), then returns a TypedResults shaped as a DTO (Day 6) — a response already described in OpenAPI (Day 10). Each stage is one day of the series.

Q22What's the performance and AOT story for Minimal APIs?🔴 Senior

Minimal APIs have lower per-request overhead than controllers (less of the MVC pipeline) and are the recommended model for Native AOT, where the request-delegate generator produces trimming- and AOT-friendly code with reduced startup time and memory. For most apps the difference is modest, but for high-throughput microservices or serverless/cold-start-sensitive workloads it's a real advantage. Note that some reflection-heavy features are constrained under AOT, which Minimal APIs' source-generation approach is designed to accommodate.

Q23How would you structure a production-ready versioned, secured, documented CRUD API?🔴 Senior

Register services in Program.cs (DbContext, services, versioning, auth, ProblemDetails, OpenAPI) and order the middleware correctly (exception handler first, then auth, then caching — Days 11, 14). Put each resource's endpoints in an extension method with named, testable handlers, grouped via MapGroup with a versioned prefix, shared authorization, and OpenAPI tags. Use DTOs and a service layer to keep endpoints thin and entities internal (Days 5, 7). Keep secrets out of source control, document responses via typed results, add integration tests, and publish with a clear README. The result is thin endpoints over a well-layered core — every concern declared once, at the right level.
