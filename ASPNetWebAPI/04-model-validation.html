Model Validation in ASP.NET Core 🛡️

Every request that reaches your API is a stranger holding a form. Model validation is the polite-but-firm doorkeeper that checks the form before your code runs — rejecting nonsense with a clean, predictable error and letting good data through untouched. Today we wire up rules with data annotations, read the verdict from ModelState, let [ApiController] return automatic 400s, and build our own rules when the built-in ones fall short.

🏷️ Data annotations 📋 ModelState 🚫 Automatic 400s 🧩 Custom attributes 🔗 IValidatableObject 📨 Proper error responses
01

What Model Validation Is 🤔

Model validation is the step where ASP.NET Core checks that the data a client sent obeys your rules before your controller action runs. A "model" here is just the C# class the incoming data is copied into — a CreateBookRequest, a LoginForm, and so on. Validation asks: is the title actually filled in? Is the price a sensible number? Is that really an email address?

It sits directly after model binding (the step that copies raw HTTP values — JSON, form fields, query strings — into your model's properties). The two are easy to confuse, so hold them apart:

🔑 Key distinction

Model binding catches conversion errors — e.g. someone typed "x" where an int was expected. Model validation catches business-rule errors — e.g. a rating of 0 was sent where the range is 1–5. Both finish before your action executes, and both dump their findings into the same place: ModelState.

Why bother — isn't the front-end enough? 🧐

Client-side (browser) validation is a convenience: it saves a round-trip and gives instant feedback. It is not security. Anyone can bypass the browser with a tool like Postman or curl and send whatever they like. Server-side validation is the real gate. The rule of thumb: validate on the client for friendliness, validate on the server for truth.

🎟️ Analogy — the concert door

Client-side validation is the sign at the parking lot saying "tickets required." Helpful, but a determined person can walk right past it. Server-side validation is the staff member at the actual door scanning every ticket. Only one of them actually keeps the venue safe — and it's not the sign.

02

The Validation Pipeline 🔄

A single request travels through a fixed sequence of stages. Knowing the order tells you exactly where each kind of error comes from — and why your action either runs with clean data or never runs at all.

Step 1 · 📥 incoming
HTTP request arrives

Raw JSON, form data, route values, and query strings — all still just text.

Step 2 · 🔗 model binding
Bind values into the model

The framework copies raw values into your C# object. Type mismatches (letters in a number field) become binding errors here.

Step 3 · 🛡️ model validation
Run the validation rules

Every data annotation, custom attribute, and IValidatableObject.Validate runs. Failures are added to ModelState.

Step 4 · ⚖️ the verdict
Is ModelState valid?

With [ApiController], an invalid state short-circuits into an automatic 400. Otherwise, your action inspects ModelState.IsValid itself.

Step 5 · ✅ your action runs
Action executes with trusted data

Only reached when the data is valid (or you chose to handle invalid data manually).

Think of ModelState as a scorecard that fills up as the request moves through steps 2 and 3. By the time your action is considered, the scorecard is either empty (valid) or holds a list of complaints keyed by field name:

too short
Price ✗
out of range

IsValid = false

✅ Tip

Both binding and validation always run to completion before your action is invoked. You never have to call validation manually in the normal flow — it has already happened. Your only job is to react to the result.

03

Data Annotations — Rules as Attributes 🏷️

The most common way to declare rules is with data annotations: small attributes you place in square brackets above a property. They live in the System.ComponentModel.DataAnnotations namespace and read almost like plain English.

C#
using System.ComponentModel.DataAnnotations;

public class CreateBookRequest
{
    [Required]
    [StringLength(100, MinimumLength = 2)]
    public string Title { get; set; } = null!;

    [Required]
    [StringLength(13, MinimumLength = 13,
        ErrorMessage = "ISBN must be exactly 13 characters.")]
    public string Isbn { get; set; } = null!;

    [Range(0, 999.99)]
    public decimal Price { get; set; }

    [EmailAddress]
    public string? ContactEmail { get; set; }
}

Read top to bottom, this says: a title is mandatory and 2–100 characters; an ISBN is mandatory and exactly 13 characters (with a friendlier message); a price must fall between 0 and 999.99; and if a contact email is present, it must look like an email. No if statements, no repetition — the rules travel with the data.

The built-in attributes you'll actually use 📚

AttributeChecks that…Example
[Required]the value isn't null (and, for strings, not empty)[Required]
[StringLength]a string's length is within a max (and optional min)[StringLength(100, MinimumLength = 2)]
[MaxLength] / [MinLength]length ceilings/floors for strings or collections[MaxLength(250)]
[Range]a number (or date) falls within min–max[Range(1, 5)]
[RegularExpression]a string matches a pattern[RegularExpression(@"^\d{3}-\d{4}$")]
[EmailAddress]the value has an email shape[EmailAddress]
[Phone]the value has a phone-number shape[Phone]
[Url]the value has a URL shape[Url]
[CreditCard]the value has a credit-card format[CreditCard]
[Compare]two properties match (e.g. password + confirm)[Compare("Password")]
[Remote]a value is valid by calling a server action (client-side)[Remote("VerifyEmail", "Users")]
[ValidateNever]skips a property/parameter during validation[ValidateNever]
🔑 StringLength vs. MaxLength

They look similar but differ in intent. [MaxLength] also feeds the database schema (via EF Core) to size the column. [StringLength] is purely a validation rule and adds a convenient MinimumLength. On an API DTO, either validates input; on an entity, prefer [MaxLength] so the column and the rule stay in sync.

Custom error messages & placeholders 💬

Every attribute accepts an ErrorMessage. Internally the message is run through String.Format, so you can use numbered placeholders — {0} is the field name, and later numbers are the attribute's own arguments (order varies per attribute).

C#
// {0} = field name, {1} = max, {2} = min (for StringLength)
[StringLength(8, MinimumLength = 6,
    ErrorMessage = "{0} length must be between {2} and {1}.")]
public string Name { get; set; } = null!;

// Applied to "Name" this produces:
// "Name length must be between 6 and 8."
04

ModelState — Reading the Verdict 📋

ModelState is a property on ControllerBase of type ModelStateDictionary. It's a dictionary keyed by field name; each entry holds any errors found for that field (from binding or validation). Two members do most of the work:

⚖️

ModelState.IsValid

Returns true only if every field is error-free. This is the single boolean your action checks.

➕

AddModelError(key, msg)

Lets you inject your own error — for rules that need the database or services the attributes can't reach.

The classic MVC pattern (controllers & views) 🖥️

In a page-rendering app, the action inspects IsValid and, if the data is bad, redisplays the same page so the user can fix it. The errors are already attached to the model, so the view can show them.

C#
[HttpPost]
public async Task<IActionResult> Create(CreateBookRequest request)
{
    if (!ModelState.IsValid)
    {
        // Send the user back to the form with error messages.
        return View(request);
    }

    await _repository.AddAsync(request);
    return RedirectToAction(nameof(Index));
}

Adding an error the attributes can't express ➕

Some rules need data only the server has — "is this email already taken?" No attribute can answer that, so you check it in the action and push the result into ModelState with the property name as the key. Using nameof keeps the key correct even if you rename the property later.

C#
if (_repository.EmailExists(request.ContactEmail))
{
    ModelState.AddModelError(
        nameof(request.ContactEmail),
        "This email is already registered.");
}

if (!ModelState.IsValid)
{
    return ValidationProblem(ModelState);
}
⚠️ Watch out

The key you pass to AddModelError must match how the client sees the field, or the error won't attach to the right input. For nested models the key is dotted, e.g. "Contact.Email". Get the key wrong and the error still shows — just as a general, unattached message.

05

Automatic HTTP 400 Responses 🚫

For Web APIs, checking ModelState.IsValid by hand in every action gets tedious. The [ApiController] attribute removes that chore. Applied to a controller, it registers a filter that runs after model binding but before your action; if ModelState is invalid, it short-circuits the request and returns an automatic 400 Bad Request — your action never runs.

C#
[ApiController]
[Route("api/[controller]")]
public class BooksController : ControllerBase
{
    [HttpPost]
    public IActionResult Create(CreateBookRequest request)
    {
        // No ModelState check needed. If validation failed,
        // [ApiController] already returned a 400 for us.
        _repository.Add(request);
        return CreatedAtAction(nameof(Get), new { id = request.Id }, request);
    }
}

What the client receives 📦

The body isn't a raw string. The framework builds a ValidationProblemDetails object — a standardized error shape defined by RFC 9457 (formerly RFC 7807) — and serializes it with the content type application/problem+json. Errors are grouped under an errors dictionary keyed by field name:

JSON — 400 response body
{
  "type": "https://tools.ietf.org/html/rfc7231#section-6.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "traceId": "00-e98…f00-87ee…46-00",
  "errors": {
    "Isbn":  [ "ISBN must be exactly 13 characters." ],
    "Title": [ "The Title field is required." ]
  }
}
🔑 Where it happens

The filter runs during OnActionExecuting — after binding, before your action. That timing is why you can trust the data inside your action body: anything invalid was already turned away at the door.

The complexity of it all, at a glance:

06

Returning Proper Validation Errors 📨

"Proper" errors are consistent, machine-readable, and keyed to the field that failed — so a front-end can highlight the exact input. You have three levels of control, from fully automatic to fully custom.

🤖

Automatic

Let [ApiController] emit the 400 + ValidationProblemDetails. Zero code. Best default.

✍️

Manual, consistent

Call ValidationProblem(ModelState) so your custom errors match the automatic shape exactly.

🛠️

Global override

Replace InvalidModelStateResponseFactory to reshape or re-status every validation response app-wide.

⚠️ Don't reach for BadRequest by reflex

Returning BadRequest(ModelState) works, but its shape can drift from the automatic response. Prefer ValidationProblem(ModelState) — it returns the same ValidationProblemDetails the framework produces automatically, so manual and automatic errors look identical to the client.

400 vs. 422 — which status? ⚖️

Both signal "your input was rejected." The distinction some teams draw: 400 Bad Request for malformed requests (couldn't even be parsed) and 422 Unprocessable Entity for well-formed requests that break business rules. ASP.NET Core defaults to 400 for both; if your team prefers 422 for rule violations, you override the response factory. Consistency across your API matters more than which number you pick.

Customizing the automatic response globally 🛠️

Every automatic 400 flows through InvalidModelStateResponseFactory. Swap it out to add logging, reshape the body, or change the status code to 422 for the whole app:

C# — Program.cs
builder.Services.AddControllers()
    .ConfigureApiBehaviorOptions(options =>
    {
        options.InvalidModelStateResponseFactory = context =>
        {
            var problem = new ValidationProblemDetails(context.ModelState)
            {
                Status = StatusCodes.Status422UnprocessableEntity,
                Title  = "Validation failed"
            };
            return new UnprocessableEntityObjectResult(problem);
        };
    });

Turning the automatic behavior off entirely 🔌

If you'd rather handle everything by hand (perhaps to log first, or return a bespoke shape per action), disable the filter with SuppressModelStateInvalidFilter. Then it's back to you to check ModelState.IsValid.

C# — Program.cs
builder.Services.Configure<ApiBehaviorOptions>(options =>
{
    options.SuppressModelStateInvalidFilter = true;
});
✅ Tip

Reach for the global factory (not per-action code) when you want one consistent error contract across the whole API. Front-end teams love a single predictable error shape far more than clever per-endpoint variations.

07

Custom Validation Attributes 🧩

When no built-in attribute fits — "this date can't be in the future," "this code must start with a department prefix" — you write your own. Create a class that inherits ValidationAttribute and override IsValid. Return ValidationResult.Success when the value passes, or a ValidationResult carrying an error message when it fails.

C#
using System.ComponentModel.DataAnnotations;

public class NotInFutureAttribute : ValidationAttribute
{
    protected override ValidationResult? IsValid(
        object? value, ValidationContext context)
    {
        // A missing value isn't THIS rule's job — let [Required] handle it.
        if (value is DateTime date && date > DateTime.Now)
        {
            return new ValidationResult(
                $"{context.DisplayName} cannot be in the future.");
        }

        return ValidationResult.Success;
    }
}

Now it drops onto any date property like a built-in attribute:

C#
[NotInFuture]
public DateTime PublishedOn { get; set; }

Reaching the whole object via ValidationContext 🔎

The ValidationContext parameter is powerful: context.ObjectInstance gives you the entire model, so one property's rule can consider another property. It also exposes context.DisplayName (for friendly messages) and even service resolution for dependency-injected checks.

C#
protected override ValidationResult? IsValid(
    object? value, ValidationContext context)
{
    var book = (CreateBookRequest)context.ObjectInstance;
    var year = ((DateTime)value!).Year;

    if (book.Genre == Genre.Classic && year > 1960)
    {
        return new ValidationResult(
            "Classic titles must be published no later than 1960.");
    }
    return ValidationResult.Success;
}
⚠️ Attribute tied to one type

Casting context.ObjectInstance to a specific model (CreateBookRequest above) locks the attribute to that type. If you reuse it on a different model, the cast throws. For genuinely reusable cross-field logic, prefer IValidatableObject (next section) or keep the custom attribute focused on a single property's value.

08

Cross-Field Validation: IValidatableObject 🔗

Attributes validate one property at a time. When a rule spans several properties — "end date must be after start date," "if paying by card, a card number is required" — implement IValidatableObject on the model itself. Its single Validate method sees the whole object and can yield return as many errors as it finds, each tied to the field(s) it concerns.

C#
using System.ComponentModel.DataAnnotations;

public class DateRange : IValidatableObject
{
    public DateTime Start { get; set; }
    public DateTime End   { get; set; }

    public IEnumerable<ValidationResult> Validate(
        ValidationContext validationContext)
    {
        if (End < Start)
        {
            // Second arg names the member(s) the error attaches to.
            yield return new ValidationResult(
                "End date must be after the start date.",
                new[] { nameof(End) });
        }
    }
}

Attribute vs. IValidatableObject — which one? 🤷

Use…WhenReusable across models?
ValidationAttributethe rule is about one property and you'll reuse it in many places (e.g. [NotInFuture])✅ Yes — that's the point
IValidatableObjectthe rule compares several properties of this model❌ No — it lives on one class
Check in the action / servicethe rule needs a database, external API, or DI service➖ It's just code
🔑 Ordering note

IValidatableObject.Validate runs after the property-level attributes. In practice a badly-formed field is usually caught by [Required]/[Range] first, and your cross-field method runs once the individual pieces are at least present — so guard against nulls just in case.

09

Configuration & Gotchas ⚙️

Non-nullable reference types quietly imply [Required] 🕳️

This trips up newcomers constantly. With nullable reference types enabled (<Nullable>enable</Nullable>, the modern default), the validation system treats a non-nullable string Name as if it were marked [Required(AllowEmptyStrings = true)]. So a missing Name fails validation even without an explicit attribute. If a value is genuinely optional, make the type nullable:

C#
public class Person
{
    public string  Name { get; set; } = null!;  // implicitly required
    public string? Nickname { get; set; }        // truly optional
}

Re-running validation manually 🔁

Validation is automatic, but occasionally you compute a value inside an action and want to validate again. Clear the old state, then call TryValidateModel:

C#
book.PublishedOn = DateTime.Now.Date;   // recomputed value

ModelState.ClearValidationState(nameof(book));
if (!TryValidateModel(book, nameof(book)))
{
    return ValidationProblem(ModelState);
}

Safety limits & short-circuiting 🛑

max errors200 (default) optionMaxModelValidationErrors max depth32 (default) optionMaxValidationDepth

Validation stops once it hits the maximum error count (200 by default) so a hostile payload can't generate endless errors. MaxValidationDepth (default 32) protects against stack overflow on deeply nested or recursive models. And validation is automatically short-circuited — skipped entirely — for object graphs with nothing to validate, such as collections of primitives (byte[], string[], Dictionary<string,string>).

.NET 10 packaging change 📦

As of .NET 10, the unified validation APIs moved into the Microsoft.Extensions.Validation NuGet package so they can be used outside ASP.NET Core HTTP scenarios too. The behavior is unchanged, but you now add an explicit package reference and register the services:

XML — .csproj
<PackageReference Include="Microsoft.Extensions.Validation" Version="10.0.0" />
C# — Program.cs
builder.Services.AddValidation();
🧯 Analogy — the sprinkler cap

The error and depth limits are like the maximum flow on a building's sprinkler system: you never expect to hit them in normal use, but they exist so one runaway situation can't flood the whole floor. They protect the server, not the honest user.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1What is model validation in ASP.NET Core?🌱 Junior

It's the step where the framework checks that incoming request data obeys your rules — before your controller code runs. It answers questions like "is this field filled in?" and "is this number in range?" so your action only deals with data you can trust.

Q2What is a data annotation? Give a few examples.🌱 Junior

A data annotation is a small attribute you place above a model property to declare a rule. Common ones are [Required] (must be present), [StringLength] (length limits), [Range] (numeric bounds), and [EmailAddress] (must look like an email). They live in the System.ComponentModel.DataAnnotations namespace.

Q3What does the [Required] attribute do?🌱 Junior

It marks a property as mandatory: validation fails if the value is null (and, for strings, empty). It's the most common rule and often the first line of defense before more specific checks.

Q4What is ModelState, and what does ModelState.IsValid tell you?🌱 Junior

ModelState is a dictionary that collects any errors found while binding and validating the request, keyed by field name. ModelState.IsValid is a single boolean that's true only when every field is error-free — it's the flag your action checks to decide whether to proceed.

Q5What's the difference between model binding and model validation?🌱 Junior

Binding copies raw request values into your C# object and catches conversion errors (letters where a number was expected). Validation runs next and catches business-rule errors (a value out of an allowed range). Both happen before your action and both record their findings in ModelState.

Q6How do you set a custom error message on a validation attribute?🌱 Junior

Pass an ErrorMessage argument, e.g. [Required(ErrorMessage = "Please enter a title.")]. You can even include placeholders — {0} is the field name — because the message is formatted with String.Format.

Q7Which HTTP status code signals a validation failure, and why?🌱 Junior

400 Bad Request by default — it tells the client the request was rejected because the data didn't meet requirements. Some teams use 422 Unprocessable Entity for well-formed-but-rule-breaking data, but 400 is the framework default.

Q8Why validate on the server if the browser already validates?🌱 Junior

Browser validation is only a convenience — it can be bypassed with tools like Postman or by disabling JavaScript. The server is the only place you fully control, so it's the real security boundary. Client validation improves the experience; server validation protects the data.

Q9What does the [ApiController] attribute do for validation?🟠 Mid

Among other conveniences, it wires up automatic model validation: if ModelState is invalid, the framework short-circuits the request and returns a 400 with a ValidationProblemDetails body before your action runs. That means you don't have to write if (!ModelState.IsValid) in every Web API action.

Q10How do you add a validation error that no built-in attribute can express?🟠 Mid

Do the check in your action or service and call ModelState.AddModelError(key, message), using the property name (ideally via nameof) as the key. This is how you enforce rules that need the database or external services — like "this email is already taken."

Q11What's the difference between [StringLength] and [MaxLength]?🟠 Mid

Both cap a string's length for validation. The key difference: [MaxLength] also informs the database schema (via EF Core) to size the column, while [StringLength] is validation-only and additionally supports a MinimumLength. On entities prefer [MaxLength] to keep the column and rule aligned; on API DTOs either works.

Q12What is ValidationProblemDetails and what does it contain?🟠 Mid

It's the standardized error body (based on RFC 9457 / formerly 7807) the framework returns for validation failures, sent as application/problem+json. It includes type, title, status, a traceId, and an errors dictionary that maps each field name to its list of error messages — perfect for a front-end to highlight specific inputs.

Q13How do you write a custom validation attribute?🟠 Mid

Create a class inheriting ValidationAttribute and override IsValid(object? value, ValidationContext context). Return ValidationResult.Success when the value passes, or new ValidationResult("message") when it fails. Then apply your attribute to a property like any built-in one.

Q14When would you use IValidatableObject instead of a custom attribute?🟠 Mid

Use IValidatableObject for rules that compare several properties of the same model — like "end date must be after start date." Its Validate method sees the whole object and can yield multiple errors. A custom attribute is better for a single-property rule you want to reuse across many models.

Q15How do you make manual error responses match the automatic ones?🟠 Mid

Return ValidationProblem(ModelState) rather than BadRequest(ModelState). ValidationProblem produces the same ValidationProblemDetails shape the framework emits automatically, so clients see one consistent error contract whether the error was automatic or hand-added.

Q16How do you disable the automatic 400 response?🟠 Mid

Set SuppressModelStateInvalidFilter = true on ApiBehaviorOptions in Program.cs. You'd do this when you want full manual control — for example to log before responding or to return a bespoke shape. Once disabled, it's back to checking ModelState.IsValid yourself.

Q17How do non-nullable reference types affect [Required] behavior?🔴 Senior

With nullable reference types enabled, the validation system treats a non-nullable property (e.g. string Name) as if it carried [Required(AllowEmptyStrings = true)], so a missing value fails even without an explicit attribute. To make a value optional you must use a nullable type (string?). You can turn this implicit behavior off with SuppressImplicitRequiredAttributeForNonNullableReferenceTypes.

Q18How do you customize the automatic 400 response globally?🔴 Senior

Replace ApiBehaviorOptions.InvalidModelStateResponseFactory via ConfigureApiBehaviorOptions. The delegate receives the action context and returns any IActionResult, so you can log, reshape the body, or change the status code (e.g. to 422). Doing it here gives the whole API one consistent error contract instead of per-action variations.

Q19400 vs. 422 for validation — which should you return?🔴 Senior

A common convention: 400 for requests that couldn't be parsed at all (malformed syntax), and 422 Unprocessable Entity for well-formed requests that break business rules. ASP.NET Core defaults to 400 for both. Either is defensible; what matters most is picking one and applying it consistently across the API so clients aren't guessing.

Q20How does validation short-circuit, and what is MaxValidationDepth?🔴 Senior

Validation is automatically skipped for object graphs with nothing to validate — like collections of primitives (byte[], string[], Dictionary<string,string>) — which saves work. MaxValidationDepth (default 32) caps how deep the validator recurses through a nested object graph, preventing a stack overflow on deep or self-referencing models. There's also a max error count (200) that stops runaway error accumulation.

Q21When would you choose FluentValidation over data annotations?🔴 Senior

FluentValidation is a popular library that expresses rules in code (a fluent builder) rather than attributes. It shines when rules are complex, conditional, or need dependency-injected services and unit testing in isolation — keeping validation logic out of the model. Data annotations stay ideal for simple, declarative rules with minimal ceremony and built-in client-side integration. Many teams mix both.

Q22What is overposting (mass assignment) and how do you guard against it?🔴 Senior

Overposting is when a client sends extra fields hoping to set properties they shouldn't — like IsAdmin = true. The defense is to bind to a purpose-built DTO or view model that exposes only the properties clients may set, rather than binding directly to your database entity. Validation and DTOs work together here: the DTO limits what can be assigned, and validation enforces the values that are allowed.

Q23What changed about validation packaging in .NET 10?🔴 Senior

The unified validation APIs moved into the Microsoft.Extensions.Validation NuGet package so they can be used outside ASP.NET Core HTTP scenarios. The functionality is the same, but you now add an explicit package reference and call builder.Services.AddValidation() to register the services.

✅ Study-order tip

Learn it in the order the request travels: first binding vs. validation (Q5) so you know where errors come from, then data annotations + ModelState (Q2–Q4) to declare and read rules, then automatic 400s + ProblemDetails (Q9, Q12) for the response, and finally the custom rules and config (Q13–Q23). If you can explain the pipeline diagram from section 02 out loud, you can answer most of these on the spot.
