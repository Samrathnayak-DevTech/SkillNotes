Day 01 · ASP.NET Core Web API

🌐 Web API & REST: The Language of the Web

Before you write a line of ASP.NET Core, you need to know what you're building and the rules it plays by. Today: what a Web API actually is, the REST principles that shape it, how the web names things (resources & URLs), and the vocabulary every request speaks — HTTP verbs and status codes. Then you'll scaffold a real project and watch it answer its first request.

🌐 What's a Web API 🧭 REST 📦 Resources 🔤 HTTP Verbs 🚦 Status Codes

01🌐 What Is a Web API?

An API — Application Programming Interface — is a contract that lets one piece of software talk to another. A Web API is an API you reach over the internet using HTTP, the same protocol your browser uses. Instead of returning a web page for a human to look at, it returns data (usually JSON) for another program to use.

🍽️ Analogy

A Web API is a restaurant kitchen with a waiter. You (a phone app, a website, another server) don't march into the kitchen and cook — you give the waiter an order from a menu, the waiter takes it to the kitchen, and brings back exactly what you asked for. The menu is the API's list of what you can request; the waiter is HTTP carrying your order back and forth; the kitchen is the server doing the real work. You never see the kitchen; you just trust the contract on the menu.

The same Web API can feed many different clients at once — a mobile app, a website, a smart watch, another company's server — because they all speak the same neutral language: HTTP requests in, data responses out.

🔑 Key Idea

A Web API separates the front end (what users see — apps, websites) from the back end (data and business logic). That separation is why the same API can power an iPhone app and a website simultaneously, and why teams can rebuild the app without touching the server. You're building the back end that everything else talks to.

03🧭 REST Principles

REST (Representational State Transfer) isn't a technology or a library — it's an architectural style, a set of conventions for designing web APIs so they're predictable, scalable, and easy to consume. An API that follows them is called RESTful. You don't install REST; you design for it.

📦Resource-based

Everything is a "resource" (a customer, an order) identified by a URL. You act on resources, not on function calls.

🔤Uniform interface

The same small set of HTTP verbs (GET, POST, …) works across every resource, so once you know one endpoint you can guess the others.

🎫Stateless

Each request carries everything the server needs. The server remembers nothing between requests — no "session" of what you did last.

🖥️Client–server

The client (UI) and server (data) are separate and independent — either can change without breaking the other, as long as the contract holds.

🗂️Cacheable

Responses can say whether they're safe to cache, so clients and proxies can reuse them instead of re-asking — faster and lighter.

🎭Representations

A resource (the order) is separate from how it's sent (usually JSON). The same order could be JSON, XML, etc. — JSON is the norm today.

📮 Analogy

Stateless is the confusing one, so: think of each request as a complete letter, not a phone call. On a phone call, both sides remember the conversation so far ("as I was saying…"). A letter must contain everything needed to be understood on its own — who you are, what you want — because the reader has no memory of previous letters. REST requests are letters: self-contained. That's what lets any server in a farm handle any request, which is how APIs scale.

🔑 Why REST Won

Because it's predictable. Once you learn that GET /products lists products and GET /products/5 gets one, you can guess how any RESTful API's customers, orders, or invoices work without reading much documentation. That consistency across the entire industry is REST's superpower — and it's why interviewers care that you understand the conventions, not just the syntax.

✅ "RESTful" Is a Spectrum

Few real APIs follow every REST principle perfectly, and that's fine. In practice "RESTful" means "uses resources, HTTP verbs, and status codes sensibly." Don't get dogmatic — aim for clear, consistent, conventional. The goal is an API another developer can predict, not a purity badge.

04📦 Resources & URLs

In REST, a resource is any "thing" your API exposes — a product, a customer, an order. Each resource is named by a URL (the address). The golden rule: URLs are nouns, not verbs. The verb comes from the HTTP method (next section), so the URL just names the thing.

URLRefers toKind
/productsThe whole collection of productsCollection
/products/42The single product with id 42Item
/customers/7/ordersAll orders belonging to customer 7Sub-collection
/customers/7/orders/3Order 3 of customer 7Nested item
⚠️ URLs Are Nouns, Not Verbs

A classic beginner mistake is putting actions in the URL: /getProducts, /createProduct, /deleteProduct/42. Don't. The URL names the resource (/products, /products/42); the HTTP verb says what to do with it (GET to read, POST to create, DELETE to remove). DELETE /products/42 — not GET /deleteProduct/42. Nouns in the path, verbs in the method.

Query strings: filtering, not identity 🔎

The path identifies which resource; the query string (after the ?) refines how you want the collection — filtering, sorting, paging. It doesn't name a different thing, it tunes the request:

URLs · path vs query string
GET /products                      # all products
GET /products?category=keyboards   # filter the collection
GET /products?sort=price&page=2     # sort + paginate
GET /products/42                   # one specific product (path, not query)
✅ Naming Conventions

Use plural nouns for collections (/products, not /product), lowercase with hyphens for multi-word paths (/order-items), and keep nesting shallow (one or two levels — /customers/7/orders is fine, five levels deep is a smell). Consistency matters more than any single rule: pick a convention and hold to it across the whole API.

05🔤 HTTP Verbs

The HTTP verb (or method) states the action to perform on the resource named by the URL. There's a small, standard set, and each has an agreed meaning. Master these five and you can read almost any API:

VerbMeansExampleBody?
GETRead — fetch a resource, change nothingGET /products/42No
POSTCreate — add a new resource to a collectionPOST /productsYes (the new item)
PUTReplace — update a resource wholesalePUT /products/42Yes (full item)
PATCHModify — update part of a resourcePATCH /products/42Yes (changed fields)
DELETERemove — delete a resourceDELETE /products/42No

Two properties worth knowing: safe & idempotent 🧮

🕊️Safe

The verb doesn't change anything on the server. GET is safe — reading a product a hundred times changes nothing. POST/PUT/DELETE are not safe.

🔁Idempotent

Doing it many times has the same effect as doing it once. GET, PUT, and DELETE are idempotent; POST is not (two POSTs create two things).

🛗 Analogy

Idempotent is the elevator-button rule. Pressing "call the elevator" once or five times summons the same one elevator — that's idempotent (DELETE, PUT). But putting a coin in a vending machine five times dispenses five snacks — that's not idempotent (POST). It matters because if a network glitch makes a client retry, an idempotent request is safe to repeat; a POST retried blindly might create a duplicate order.

🔑 Verb + Noun = Intent

The whole grammar of a REST request is verb + resource. GET /orders = "read the orders." POST /orders = "create an order." DELETE /orders/3 = "remove order 3." The URL never needs a verb because the HTTP method already is the verb. This is the single most important habit to build on Day 1.

06🚦 Status Codes

Every HTTP response comes with a three-digit status code that tells the client, at a glance, how the request went. They're grouped by the first digit — learn the families first, then the common members.

✅ Success

5xx

The five families · the first digit tells you the category before you read the number
🚦 Analogy

Status codes are the facial expression the server gives back with every answer. 2xx is a smile ("here you go"). 4xx is a raised eyebrow ("you asked wrong — that doesn't exist, or you're not allowed"). 5xx is a wince ("I messed up — something broke on my end"). You can read the room from the first digit before you even hear the words.

The ones you'll use constantly

CodeNameMeans
200OKSuccess — here's your data (typical GET response).
201CreatedSuccess — a new resource was created (typical POST response).
204No ContentSuccess — done, nothing to return (typical DELETE response).
400Bad RequestThe request was malformed or failed validation — the client's fault.
401UnauthorizedYou're not authenticated — the API doesn't know who you are.
403ForbiddenIt knows who you are, but you're not allowed to do this.
404Not FoundThe resource doesn't exist.
409ConflictClashes with current state (e.g. duplicate, or a concurrency conflict).
500Internal Server ErrorSomething threw on the server — the classic "it broke."
⚠️ 401 vs 403 — a Favorite Interview Trap

They sound alike but mean different things. 401 Unauthorized = "I don't know who you are" — you haven't logged in / your token is missing or invalid. 403 Forbidden = "I know exactly who you are, and you're not allowed to do this." The names are historically confusing (401 is really about authentication, 403 about authorization), so the memory hook is: 401 = who are you?, 403 = you can't.

🔑 Return the Right Code — It's an API's Manners

Returning 200 OK with an error message buried in the body is a broken API — clients rely on the code to branch. A missing item is 404, not a 200 with "not found" text. Bad input is 400. A created order is 201, not 200. Using accurate status codes is how your API communicates clearly — and it's exactly what these study notes will keep coming back to.

07📨 Anatomy of a Request & Response

Put it all together and watch one round trip. A client wants product 42; here's the full conversation, piece by piece:

📤
1. The client sends a requestA verb + URL (GET /products/42), plus headers (e.g. Accept: application/json — "send me JSON"). GET has no body.
🌐
2. It travels over HTTP to the serverAcross the internet to your Web API, which is listening on a host and port.
⚙️
3. The API handles itIt matches the verb + URL to the right code, does the work (often reading a database), and builds a response.
📥
4. The server sends a responseA status code (200 OK), headers (Content-Type: application/json), and a body — the product as JSON.
✅
5. The client reads the resultIt checks the status code first (did it work?), then parses the JSON body into an object to use.
HTTP · a request and its response
# ── REQUEST ──
GET /products/42 HTTP/1.1
Host: api.myshop.com
Accept: application/json

# ── RESPONSE ──
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 42,
  "name": "Mechanical Keyboard",
  "price": 4999.00
}
🔑 Four Parts Each Way

A request is: verb + URL + headers + (optional) body. A response is: status code + headers + (optional) body. That's the entire shape of every HTTP exchange on the web — an app, your browser, an API call all use exactly this structure. Once you see it, the whole thing stops being mysterious.

08🏗️ Scaffolding the Project

Enough theory — let's make one. The .NET SDK ships a Web API template that gives you a working, runnable API in one command. You'll need the .NET SDK installed (check with dotnet --version).

Terminal · create & run
# Create a new Web API project called MyShop.Api:
dotnet new webapi -n MyShop.Api

# Move into it and run:
cd MyShop.Api
dotnet run

When it starts, the console prints the URL it's listening on, e.g. http://localhost:5xxx. The template gives you a tiny demo endpoint — a /weatherforecast route that returns five days of made-up weather as JSON. Hit it in a browser or with curl:

Terminal · call the demo endpoint
curl http://localhost:5xxx/weatherforecast

# → a JSON array of forecasts:
# [{"date":"2026-07-08","temperatureC":22,"summary":"Mild"}, ...]

That JSON coming back is your proof: you have a running Web API answering HTTP requests. Look at what the template created:

FileWhat it is
Program.csThe entry point — sets up and starts the app, and (in the minimal template) defines the weather endpoint.
MyShop.Api.csprojThe project file — target framework and package references.
appsettings.jsonConfiguration — settings like logging levels (you'll use this a lot later).
Properties/launchSettings.jsonLocal run settings — which URLs and ports, which browser to open.
MyShop.Api.httpA file of sample requests you can run right from your editor to test endpoints.
✅ Two Template Flavors

The default dotnet new webapi uses minimal APIs — endpoints defined as compact lambdas in Program.cs. Add --use-controllers for the classic controller style (separate controller classes). Both are valid and current; minimal APIs are the lean default, controllers suit larger apps. Either way, the weather demo runs the same. Don't worry about which yet — just get it running.

09📖 Swagger & OpenAPI

Calling endpoints with curl works, but there's a friendlier way to explore an API: a web page that lists every endpoint and lets you try them in the browser. That's what Swagger UI does — and it's built on OpenAPI, a standard, machine-readable description of your API (which endpoints exist, what they take, what they return).

📋 Analogy

OpenAPI is the printed menu of your API — a precise, structured list of everything on offer. Swagger UI is a nicely designed, interactive version of that menu you can read and order from directly. The kitchen (your API) generates the menu automatically from the code, so it's never out of date.

⚠️ Important Change: Swagger Isn't in the Template Anymore

Older tutorials show Swagger UI appearing automatically. Since .NET 9, the Web API template no longer includes Swagger (Swashbuckle). Instead it ships Microsoft's built-in Microsoft.AspNetCore.OpenApi, which generates the OpenAPI description (served as JSON at /openapi/v1.json) but gives you no UI page out of the box. If a tutorial says "just go to /swagger" and it 404s, this is why. You add a UI yourself.

Step 1 — See the raw OpenAPI document 📄

The template already generates it. Run the app and open, in your browser:

Browser · the OpenAPI JSON
http://localhost:5xxx/openapi/v1.json

# → raw JSON describing every endpoint. Great for tools, ugly for humans.

Step 2 — Add a UI to read it 🖼️

Pick a viewer and point it at that JSON. Two common choices — Scalar (the modern default many teams now use) or the classic Swagger UI:

C# · Program.cs — add an interactive UI
// The template already has this — it generates the OpenAPI document:
builder.Services.AddOpenApi();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();               // serves /openapi/v1.json (already there)

    // Option A — Scalar (dotnet add package Scalar.AspNetCore):
    app.MapScalarApiReference();     // browse at /scalar

    // Option B — classic Swagger UI
    // (dotnet add package Swashbuckle.AspNetCore.SwaggerUi):
    // app.UseSwaggerUI(o => o.SwaggerEndpoint("/openapi/v1.json", "v1")); // at /swagger
}

Now run again and open /scalar (or /swagger). You'll see the /weatherforecast endpoint listed with a "try it out" button — click it, send a real request, and watch the JSON and 200 status come back, right in the browser. That's you "reading the API via Swagger."

⚠️ Keep the Docs to Development

Notice the UI is registered inside if (app.Environment.IsDevelopment()). That's deliberate: an OpenAPI document is a complete map of your API's surface, so you generally don't expose it publicly in production. Explore freely on your machine; lock it down when you ship.

🔑 OpenAPI Is the Contract, the UI Is Just a Reader

The valuable artifact is the OpenAPI document — that JSON is a precise, standard description other tools can consume to generate client code, tests, mock servers, and docs. Swagger UI and Scalar are just human readers of it. Separating "the contract" (OpenAPI, built into .NET) from "the viewer" (a UI you choose) is exactly why the template changed in .NET 9.

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is a Web API?🌱 JUNIOR

It's an application interface you access over the internet using HTTP, which returns data (usually JSON) rather than web pages. It lets different programs — a mobile app, a website, another server — talk to your back end using a shared, neutral language. The Web API is the single "front door" to your data and business logic, so many kinds of clients can use the same service without knowing anything about its internals.

Q2What does REST stand for, and is it a technology?🌱 JUNIOR

REST stands for Representational State Transfer. It's not a technology, library, or protocol you install — it's an architectural style: a set of conventions for designing web APIs so they're predictable and consistent. An API that follows those conventions (resources named by URLs, standard HTTP verbs, meaningful status codes, stateless requests) is called "RESTful." You design for REST; you don't import it.

Q3What is a "resource" in REST, and how should you name its URL?🌱 JUNIOR

A resource is any "thing" your API exposes — a product, a customer, an order — identified by a URL. URLs should be nouns, not verbs: /products for the collection, /products/42 for one item. The action comes from the HTTP verb, not the URL, so you never write /getProducts or /deleteProduct. Use plural nouns for collections and keep nesting shallow, like /customers/7/orders.

Q4Name the main HTTP verbs and what each does.🌱 JUNIOR

GET reads a resource (changes nothing). POST creates a new resource in a collection. PUT replaces a resource wholesale. PATCH updates part of a resource. DELETE removes a resource. The verb states the action, and the URL names the thing it acts on — so DELETE /products/42 means "remove product 42."

Q5What do the status code families 2xx, 4xx, and 5xx mean?🌱 JUNIOR

The first digit gives the category. 2xx = success (the request worked, e.g. 200 OK, 201 Created). 4xx = client error — your fault, the request was wrong (e.g. 400 bad request, 404 not found). 5xx = server error — the server's fault, something broke on its end (e.g. 500 internal server error). There's also 3xx for redirects. Reading just the first digit tells you who's responsible before you look up the exact number.

Q6What status code would you return for a successful GET, a successful create, and a resource that doesn't exist?🌱 JUNIOR

A successful GET returns 200 OK with the data in the body. A successful create (POST) returns 201 Created (ideally with the new resource, and often a location header pointing to it). A request for something that doesn't exist returns 404 Not Found. Using the right code matters because clients branch on it — returning 200 with a "not found" message in the body is a broken API.

Q7How do you create and run a new ASP.NET Core Web API?🌱 JUNIOR

With the .NET SDK installed, run dotnet new webapi -n MyShop.Api to scaffold the project from the template, then cd MyShop.Api and dotnet run. It prints a localhost URL it's listening on. The template includes a demo /weatherforecast endpoint you can call in a browser or with curl to confirm it's working — you'll get back a JSON array of forecasts.

Q8What is Swagger / OpenAPI used for?🌱 JUNIOR

OpenAPI is a standard, machine-readable description of an API — what endpoints exist, what they accept, and what they return. Swagger UI (and alternatives like Scalar) is an interactive web page, generated from that description, that lets you browse the endpoints and try them out in the browser without writing any client code. It's great for exploring, testing, and documenting an API, and the OpenAPI document can also feed tools that generate client code or tests.

🟠 Mid

Q9What does it mean that REST is "stateless," and why does it matter?🟠 MID

Stateless means each request carries everything the server needs to handle it — the server keeps no memory of previous requests from that client (no server-side "session" of what you were doing). Every request is like a self-contained letter, not a phone call that assumes shared context. Why it matters: it makes the API scalable and resilient. Because no request depends on server memory, any server in a farm (behind a load balancer) can handle any request, and servers can be added, removed, or restarted freely. It also simplifies reasoning — a request's behavior depends only on the request itself. State that must persist (like who you are) travels with each request, typically as a token in a header.

Q10Explain "safe" and "idempotent" HTTP methods. Classify GET, POST, PUT, DELETE.🟠 MID

Safe means the method doesn't modify server state — it's read-only. Idempotent means making the same request multiple times has the same effect as making it once. Classification: GET is both safe and idempotent (reading repeatedly changes nothing). PUT and DELETE are idempotent but not safe (replacing a resource with the same data, or deleting an already-deleted resource, ends in the same state — but they do change things). POST is neither safe nor idempotent (two POSTs typically create two resources). This matters for retries: if a network failure makes a client retry, idempotent requests are safe to repeat, whereas a blindly retried POST can create duplicates — which is why create endpoints sometimes use idempotency keys.

Q11What's the difference between 401 and 403?🟠 MID

401 Unauthorized is about authentication: the server doesn't know who you are — you haven't logged in, or your credentials/token are missing or invalid. The fix is to authenticate. 403 Forbidden is about authorization: the server knows exactly who you are, but you don't have permission to do this — and authenticating again won't help. Memory hook: 401 = "who are you?", 403 = "you can't." The names are historically confusing (401 says "unauthorized" but really means unauthenticated), which is exactly why interviewers ask it.

Q12When would you use PUT versus PATCH?🟠 MID

PUT replaces the entire resource — you send the full representation, and the server overwrites it wholesale (fields you omit are typically cleared/reset). PATCH applies a partial update — you send only the fields that change, and the rest stay as they were. Use PUT when the client has and intends to send the complete object; use PATCH when you want to update just a field or two (like changing only a product's price) without resending everything. PUT is idempotent by design; PATCH usually is too but can be defined not to be. In practice many APIs lean on PATCH for updates because sending the whole object for every small change is wasteful and risks clobbering fields.

Q13Why did .NET 9 remove Swagger/Swashbuckle from the Web API template, and what replaced it?🟠 MID

Swashbuckle was a third-party (community-maintained) package that Microsoft had shipped as a first-class default since .NET 5 — a dependency they didn't own or control, and it had fallen behind on maintenance and new runtime releases. Starting in .NET 9 the team removed it from the template and shipped their own Microsoft.AspNetCore.OpenApi package, which natively generates the OpenAPI document (served at /openapi/v1.json) as a supported platform feature. The key consequence: the template now generates the OpenAPI document but includes no UI — you add your own viewer (Scalar, Swagger UI, or ReDoc) and point it at the generated JSON. Swashbuckle isn't deprecated and can still be added manually; it was just removed as the default.

Q14What's the difference between the path and the query string, and when do you use each?🟠 MID

The path identifies which resource — it's part of the resource's identity: /products/42 is a specific product, /customers/7/orders is a specific customer's orders. The query string (after ?) refines how you want a collection returned — filtering, sorting, paging, searching: /products?category=keyboards&sort=price&page=2. Rule of thumb: if it changes what thing you're addressing, it goes in the path; if it changes how you want the results shaped or subset, it goes in the query string. Individual resource identity in the path; collection modifiers in the query.

Q15What are the parts of an HTTP request and an HTTP response?🟠 MID

A request has: a method/verb (GET, POST…), a URL/path, headers (metadata like Accept, Content-Type, Authorization), and an optional body (the data sent, e.g. on POST/PUT). A response has: a status code (200, 404…), headers (like Content-Type), and an optional body (the returned data, e.g. JSON). Headers carry metadata about the message; the body carries the actual payload. Every HTTP exchange on the web fits this shape, which is why understanding it once unlocks reading any API traffic.

Q16Why is returning the correct status code important? Give an example of getting it wrong.🟠 MID

Because clients — and infrastructure like caches, proxies, and monitoring — branch on the status code, not on the body text. The code is the machine-readable verdict. Getting it wrong breaks consumers: a common anti-pattern is returning 200 OK for everything and putting {"error": "not found"} in the body. Now every client must parse the body to discover failure, error monitoring shows zero errors while things are actually broken, and caches may store the "error" as a valid response. The correct behavior is a missing resource → 404, invalid input → 400, no permission → 403, a created resource → 201, a server exception → 500. Accurate codes are how an API communicates unambiguously across the whole ecosystem.

Q17How would you design the URLs and verbs for a shop's orders and their line items?🔴 SENIOR

Model resources as noun collections and items, use verbs for actions, and nest only where there's a genuine ownership relationship. Orders: GET /orders (list, with query params for filtering/paging), POST /orders (create → 201), GET /orders/{id} (read → 200 or 404), PUT/PATCH /orders/{id} (update), DELETE /orders/{id} (remove → 204). Line items belong to an order, so nest them: GET /orders/{id}/items, POST /orders/{id}/items, DELETE /orders/{id}/items/{itemId}. Keep nesting to one level even when relationships go deeper — for an item's product you'd link to /products/{productId} rather than nesting three levels. For actions that don't map cleanly to CRUD (e.g. "cancel an order"), the pragmatic REST approach is either a state change via PATCH (PATCH /orders/{id} with {"status":"cancelled"}) or a sub-resource (POST /orders/{id}/cancellation) — avoid POST /orders/{id}/cancel verbs if you're being strict, though many production APIs accept them for clarity. Consistency and predictability across the API matter more than purity.

Q18REST vs GraphQL vs gRPC — when would you choose each?🔴 SENIOR

REST (HTTP + JSON, resource-oriented) is the default for public and general-purpose APIs: universally understood, cacheable via HTTP, tooling everywhere, easy to debug. Its weakness is over-/under-fetching — fixed endpoints may return too much or require multiple round trips. GraphQL lets clients request exactly the fields they need in one query, great for rich front ends with varied data needs and for avoiding many round trips; costs are added server complexity, harder HTTP caching, and query-cost/security concerns. gRPC (HTTP/2 + Protobuf, contract-first, binary) excels for high-performance internal service-to-service communication and streaming: fast, strongly typed, small payloads; but it's less browser-friendly and less human-readable, so it's less ideal for public APIs. Rule of thumb: public/general API → REST; flexible client-driven data graphs → GraphQL; low-latency internal microservice calls → gRPC. They also coexist — a system might expose REST publicly and use gRPC internally.

Q19What is HATEOAS and the Richardson Maturity Model? Do you need full REST maturity?🔴 SENIOR

The Richardson Maturity Model grades how "RESTful" an API is in levels: Level 0 (one URL, one verb — RPC over HTTP), Level 1 (multiple resources, still one verb), Level 2 (resources + proper HTTP verbs + status codes — where most good APIs live), and Level 3 (HATEOAS). HATEOAS (Hypermedia As The Engine Of Application State) means responses include links to related actions/resources, so a client can navigate the API by following links rather than hard-coding URLs — like how a web page links you onward. In theory it makes clients resilient to URL changes and self-discovering. In practice, full Level 3/HATEOAS is rare: it adds significant complexity, most clients hard-code URLs anyway, and the payoff seldom justifies the cost. Most production APIs deliberately target Level 2 — correct resources, verbs, and status codes — which delivers the predictability REST is prized for. Knowing the model matters more than achieving Level 3; it's a vocabulary for discussing trade-offs, not a checklist to complete.

Q20How should an API evolve without breaking existing clients (versioning)?🔴 SENIOR

The core principle: once clients depend on your contract, breaking changes need a new version while the old keeps working. First, distinguish non-breaking changes (adding an optional field, a new endpoint) — which don't need a version bump if clients are tolerant readers — from breaking ones (removing/renaming fields, changing types, changing semantics), which do. Versioning strategies: URL path (/v1/products, /v2/products) — most visible and simplest for clients, common in practice; query string (/products?api-version=2); header (a custom or Accept header with a version) — cleaner URLs, "purer" REST, but less discoverable. ASP.NET Core has the Asp.Versioning libraries supporting all of these. Beyond the mechanism: publish a deprecation policy, support old versions for a defined window, communicate changes, and prefer additive evolution so you version as rarely as possible. Design DTOs deliberately (don't expose entities directly) so internal changes don't force external breaks. The senior insight is that versioning is as much about process and communication — deprecation timelines, client migration — as about the URL scheme.

Q21Is exposing an OpenAPI document in production a good idea? What are the considerations?🔴 SENIOR

It's a deliberate trade-off, which is why the .NET template gates the docs behind a Development-only check by default. An OpenAPI document is a complete map of your API's surface — every endpoint, parameter, and shape — so publicly exposing it hands potential attackers a reconnaissance blueprint and can reveal internal or not-yet-public endpoints. Reasons you might still expose it: it's a genuine public API where discoverability is the point (partners, third-party developers), or you want interactive docs for consumers. Mitigations when you do: serve it only for intended audiences (behind auth, on an internal network, or via a developer portal), expose only the intended public endpoints (not internal admin routes), strip sensitive detail, and consider a read-only doc renderer (like ReDoc) rather than a "try it out" UI in production. For internal or private APIs, the safe default is: generate the document (it's useful for your own tooling and CI) but don't serve the UI publicly. The judgment is matching exposure to audience and threat model, not a blanket yes/no.

Q22A client complains your API is "slow" because it makes many calls to render one screen. How do you think about it?🔴 SENIOR

This is the classic REST over-fetching / chatty-API problem: a screen needing a customer, their orders, and each order's items might make N+1 round trips, and network latency dominates. I'd approach it in layers. First, measure — confirm it's round-trip count/latency, not slow individual queries (which is a different fix, e.g. database indexing or the N+1 query problem on the server). Options to reduce chattiness: design purpose-built endpoints or resource expansion (?include=orders,items or a dedicated /dashboard aggregate) so one call returns what a screen needs; support field selection and paging so responses aren't bloated; add caching (HTTP caching headers, or a cache layer) for data that tolerates staleness; and enable HTTP/2 so multiple requests multiplex efficiently. If the client's data needs are highly variable, this is exactly the case where GraphQL or a Backend-for-Frontend (an aggregation layer tailored to that client) earns its complexity. The tension to name explicitly: pure REST favors granular, reusable resources, but real UIs want aggregated, screen-shaped responses — good API design balances the two rather than dogmatically insisting on one resource per call. The right answer depends on measured cost, how many clients share the API, and how variable their needs are.

Q23How do you handle errors consistently across a Web API so clients can rely on them?🔴 SENIOR

Two pillars: correct status codes and a consistent error body. Use the status code as the primary, machine-readable signal — 400 for validation, 401/403 for auth, 404 for missing, 409 for conflicts, 422 for semantic validation if you distinguish it, 500 for unexpected server faults — never 200-with-an-error-in-the-body. For the body, standardize on a single error shape so every client parses errors the same way; in ASP.NET Core the built-in choice is ProblemDetails (RFC 9457), a standard JSON structure with type, title, status, detail, and room for validation specifics, and the framework already emits ValidationProblemDetails for model-validation failures. Centralize it: a global exception handler (middleware or IExceptionHandler) converts unhandled exceptions into a clean ProblemDetails 500 — logging full detail internally but never leaking stack traces, SQL, or internal messages to clients. Include a correlation/trace id in every error so a client can quote it to support and you can find the matching logs. Consider stable, documented error codes so clients branch on codes rather than fragile message strings. The goal: whether a request fails validation, hits a business rule, or trips an unexpected error, the client always gets the right status and the same predictable structure — and the API never divulges internals.

Q24Why does the client–server separation and a stable contract matter architecturally?🔴 SENIOR

Because it's what lets the two halves of a system evolve independently, which is the whole economic argument for building an API in the first place. When the client (apps, web front ends, third-party integrations) and server (data, business logic) communicate only through a well-defined HTTP contract, each side can change freely as long as the contract holds: you can rebuild the mobile app, add a new web front end, or swap the server's database and internal architecture without coordinating a lockstep release across everything. It enables independent teams, independent deployment cadences, and reuse of one back end by many clients — the same API serving iOS, Android, web, and partner servers. It also localizes change: a UI redesign doesn't touch the server; a server refactor doesn't touch clients. The flip side and the senior responsibility is that the contract itself becomes a durable commitment — once clients you don't control depend on it, you can't casually change it, which is why DTO design, versioning strategy, and backward compatibility (the earlier questions) matter so much. The contract is both the enabler of independence and the constraint you must steward carefully; treating the API as a product with a stable, well-documented interface is what makes the architecture pay off over years.
