Day 03 · ASP.NET Core Web API

🔌 Model Binding: From Raw Request to Real Objects

Yesterday routing decided which method answers a request. Today: how the pieces of that request — the URL path, the query string, the body, the headers, a form — become the typed arguments your method actually receives. That translation is model binding, and the [From…] attributes are how you tell it exactly where to look.

🛣️ [FromRoute] 🔎 [FromQuery] 📦 [FromBody] 🏷️ [FromHeader] 📋 [FromForm]

01🔌 What Is Model Binding?

An HTTP request is, underneath, just text and bytes — a URL string, some header lines, maybe a JSON payload. Your C# action method, though, wants clean typed arguments: an int id, a string category, a ProductDto product. Model binding is the framework's automatic process of reading the raw request and filling in those parameters for you.

🛂 Analogy

Model binding is the customs officer who unpacks and sorts an arriving shipment. The crate (the HTTP request) shows up stuffed with mixed items — some taped to the outside (the URL), some on the shipping label (headers), some inside the box (the body). The officer knows which slot each item belongs in and lays them out neatly on the counter — labelled, typed, ready for you to use. You never rummage through the raw crate; you just receive the sorted parameters.

Without binding, every action would start with tedious plumbing — reading Request.RouteValues["id"], parsing strings to ints, deserializing JSON by hand. Binding does all of that, so your method signature simply declares what it needs and the values arrive:

C# · binding fills these parameters for you
[HttpGet("{id:int}")]                       // GET /api/products/42
public IActionResult GetById(int id)          // id arrives as 42 — already an int
{
    // No parsing, no Request.Query rummaging — just use it.
    return Ok(id);
}
🔑 Binding Runs Before Your Action

Model binding happens after routing picks your action (Day 2) but before your code runs. By the time the first line of your method executes, all its parameters are already populated. If binding fails or validation (Day 14) rejects the input, [ApiController] can short-circuit with a 400 before your action is even called — so your code only ever sees data that at least parsed correctly.

  • Posted form fields
  • File uploads
  • multipart/form-data

Five places data can live in a request; five attributes to name each one. The art of Day 3 is matching the kind of data to the right source: identity in the path, filters in the query, payloads in the body, metadata in headers, uploads in forms. Get that mapping right and your API reads exactly as a RESTful one should (Day 1).

Each [From…] attribute tells binding exactly where to pull a parameter from. Here's the whole family at a glance:

🔑 Match the Data to the Source

The choice isn't arbitrary — it follows Day 1's conventions. Identity → route. Optional filters/sorting/paging → query. The payload you're creating or updating → body. Cross-cutting metadata (who's calling, what language, a trace id) → header. File uploads / browser form posts → form. When each piece of data lives where it belongs, your endpoints are predictable and your OpenAPI docs (Day 1) are accurate.

04🛣️ [FromRoute] — the URL Path

[FromRoute] binds a parameter from a route token in the URL path — the {id} pieces you defined in the route template on Day 2. It's how the resource's identity reaches your method.

C# · binding from the path
[HttpGet("{id:int}")]                    // GET /api/products/42
public IActionResult GetById([FromRoute] int id)
    => Ok($"route id = {id}");          // id = 42

// The parameter name must match the route token — {id} ↔ id.
// If names differ, use the Name property:
[HttpGet("orders/{orderId:int}")]
public IActionResult GetOrder([FromRoute(Name = "orderId")] int id)
    => Ok(id);
🔑 The Token Name Is the Link

Binding matches the route token to the parameter by name: {id} fills id. When they can't match (the token is orderId but you want the parameter called id), pass [FromRoute(Name = "orderId")] to bridge them. This Name option exists on [FromQuery], [FromHeader], and [FromForm] too — any time the request's name and your C# name differ.

✅ Route Constraints Still Apply

The {id:int} constraint from Day 2 runs during routing, before binding — so by the time [FromRoute] int id is bound, the segment is already guaranteed to be a valid integer. Constraints filter the URL shape; binding then converts that vetted string into your typed parameter. They work as a team.

05🔎 [FromQuery] — the Query String

[FromQuery] binds from the query string — the ?key=value&key2=value2 part after the path. This is the home of optional, result-shaping inputs: filters, sorting, and paging. Query parameters are naturally optional and easy to give defaults.

C# · filtering, sorting, paging from the query
// GET /api/products?category=audio&sort=price&page=2
[HttpGet]
public IActionResult Search(
    [FromQuery] string? category,
    [FromQuery] string? sort,
    [FromQuery] int page = 1,        // default when omitted
    [FromQuery] int pageSize = 20)
    => Ok(new { category, sort, page, pageSize });

Binding many query values into one object 🧳

When a search has lots of parameters, bundle them into a single class and bind the whole thing from the query — cleaner signatures and reusable filters:

C# · a query parameter object
public class ProductQuery
{
    public string? Category { get; set; }
    public string? Sort { get; set; }
    public int Page { get; set; } = 1;
    public int PageSize { get; set; } = 20;
}

// GET /api/products?category=audio&page=2  → fills the object's properties
[HttpGet]
public IActionResult Search([FromQuery] ProductQuery query)
    => Ok(query);
🔑 Query = "How", Not "Which"

This is Day 1 and Day 2's path-vs-query distinction, now in binding. The route says which resource (identity — /products/42); the query says how you want a collection returned (filter/sort/page — ?category=audio). Don't push filters into the path (/products/active/page/2 conflates identity with state); keep them in the query where they belong.

⚠️ Don't Put Secrets in the Query String

Query strings show up in server access logs, browser history, and the Referer header sent to other sites. Never pass auth tokens, passwords, or personal data as query parameters — they leak. Sensitive values belong in a header ([FromHeader]) or the body ([FromBody]) instead. Query strings are for harmless, shareable filter values.

06📦 [FromBody] — the Request Body

[FromBody] binds a parameter from the request body — the JSON payload sent with POST, PUT, and PATCH. The framework hands the body to an input formatter (the JSON one by default) that deserializes it into your object. This is how the data you're creating or updating arrives.

C# · binding a JSON payload
public record CreateProductRequest(string Name, string Category, decimal Price);

// POST /api/products   Body: { "name": "Headphones", "category": "audio", "price": 2999 }
[HttpPost]
public IActionResult Create([FromBody] CreateProductRequest request)
{
    // request.Name = "Headphones", request.Price = 2999m — deserialized for you
    return CreatedAtAction(nameof(GetById), new { id = 99 }, request);  // 201
}
⚠️ Only One [FromBody] per Action

The request body is a stream that can be read only once. Once an input formatter consumes it to bind one parameter, it's gone — so you can have at most one [FromBody] parameter per action. If you need several values from the body, wrap them in a single object (a DTO) and bind that. Trying to bind two body parameters is a bug.

⚠️ [FromBody] Ignores Property-Level Source Attributes

When you bind a complex type from the body, the JSON input formatter reads only the body — it doesn't understand [FromQuery] or [FromRoute] placed on that type's properties. So a [FromQuery] on a property of a [FromBody] object is silently ignored; that property just gets whatever the JSON body provides. Mix sources at the parameter level (one param from body, another from route), not by decorating properties inside a body object.

✅ Content-Type Drives the Formatter

The framework picks the input formatter from the request's Content-Type header. application/json → the JSON formatter (the built-in default, using System.Text.Json). You can add XML or other formatters, but JSON is the standard for modern APIs — which is why [FromBody] "just works" for the JSON your clients send.

07🏷️ [FromHeader] & 📋 [FromForm]

🏷️ [FromHeader] — request headers

[FromHeader] binds a parameter from an HTTP header — the metadata lines a client sends alongside the request. Headers carry cross-cutting concerns: a correlation/trace id, the caller's preferred language, a custom API key, content negotiation hints.

C# · reading headers
[HttpGet]
public IActionResult Get(
    [FromHeader(Name = "X-Correlation-Id")] string? correlationId,
    [FromHeader(Name = "Accept-Language")] string? language)
{
    // Header names often contain hyphens, which aren't valid C# identifiers —
    // so the Name property maps "X-Correlation-Id" → correlationId.
    return Ok(new { correlationId, language });
}
🔑 Headers Almost Always Need Name =

Real header names — X-Correlation-Id, Accept-Language, X-Api-Key — contain hyphens, which can't be C# parameter names. So you nearly always pair [FromHeader] with Name = "..." to map the wire name to a valid identifier. This is the same Name mechanism from [FromRoute], just used more often here.

📋 [FromForm] — posted form data & file uploads

[FromForm] binds from form fields submitted as multipart/form-data — the format used by HTML form posts and, importantly, file uploads. When you need to accept an uploaded file, this is the source, paired with IFormFile.

C# · a file upload via form data
// POST /api/products/{id}/image  (Content-Type: multipart/form-data)
[HttpPost("{id:int}/image")]
public IActionResult UploadImage(
    [FromRoute] int id,
    [FromForm] string altText,
    IFormFile file)                     // the uploaded file itself
{
    // file.FileName, file.Length, file.OpenReadStream() ...
    return Ok(new { id, altText, size = file.Length });
}
⚠️ Headers & Form Aren't Inferred — Be Explicit

Unlike route/query/body, the framework does not automatically infer [FromHeader] or [FromForm] (they're treated as special sources). You must write the attribute explicitly, or the value won't bind. Also, [FromForm] and [FromBody] are mutually exclusive on one action — both read the request body, so a request is either JSON ([FromBody]) or form data ([FromForm]), never both.

08🧠 Inference & Combining Sources

With [ApiController] on your controller (Day 2), the framework infers the source for most parameters, so you often don't write the attributes at all. Knowing the inference rules lets you decide when to stay implicit and when to be explicit.

ParameterInferred sourceInferred?
Name matches a route token[FromRoute]✅ Yes
Simple type (int, string…), not in route[FromQuery]✅ Yes
Complex type (class/record)[FromBody]✅ Yes
Header value[FromHeader]❌ No — be explicit
Form field / file[FromForm]❌ No — be explicit
C# · inference means these attributes are optional
[HttpGet("{id:int}")]
public IActionResult GetById(int id) => ...;
// id matches the {id} token → inferred [FromRoute], no attribute needed

[HttpGet]
public IActionResult Search(string? category) => ...;
// simple type, not in route → inferred [FromQuery]

[HttpPost]
public IActionResult Create(CreateProductRequest request) => ...;
// complex type → inferred [FromBody]
✅ Explicit vs Implicit — a Judgment Call

Inference removes noise from routine CRUD actions, and many teams lean on it. But writing the attributes explicitly has real benefits: it makes intent obvious to the next reader, produces clearer OpenAPI specs (which keep generated client SDKs correct), and avoids subtle surprises when a signature changes. A reasonable rule: rely on inference for simple, obvious cases; be explicit whenever the source isn't glaringly clear, when a name needs mapping, or when accurate API docs matter.

Combining sources in one action 🎛️

A single action commonly pulls from several places at once — and that's exactly right, as long as each piece comes from where it belongs:

C# · the classic enterprise pattern
// PUT /api/products/42   Body: {...}   Header: X-Correlation-Id: abc123
[HttpPut("{id:int}")]
public IActionResult Update(
    [FromRoute]  int id,                              // identity ← path
    [FromBody]   UpdateProductRequest request,          // payload  ← body
    [FromHeader(Name = "X-Correlation-Id")] string? correlationId)  // trace ← header
    => NoContent();                                    // 204
🔑 The Clean Combination

The idiomatic pattern: route identifies the resource, body carries the payload, header carries metadata like a correlation id. Each source does its one job, so the action is unambiguous and its OpenAPI description is accurate. Combining sources is normal and encouraged — the discipline is matching each value to its correct source, not avoiding the mix.

09🧱 All Five Sources in One Controller

Here's a products controller with one endpoint for each binding source — a complete map of everything today, continuing the shop domain with an in-memory list (real data arrives with EF Core in Days 7–9):

C# · ProductsController.cs — every source
using Microsoft.AspNetCore.Mvc;

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    // 🛣️ FROM ROUTE — identity from the path
    // GET /api/products/42
    [HttpGet("{id:int}")]
    public IActionResult GetById([FromRoute] int id)
        => Ok(new { source = "route", id });

    // 🔎 FROM QUERY — filter/sort/page from the query string
    // GET /api/products?category=audio&page=2
    [HttpGet]
    public IActionResult Search(
        [FromQuery] string? category,
        [FromQuery] int page = 1)
        => Ok(new { source = "query", category, page });

    // 📦 FROM BODY — JSON payload for create
    // POST /api/products   { "name": "Headphones", "category": "audio", "price": 2999 }
    [HttpPost]
    public IActionResult Create([FromBody] CreateProductRequest request)
        => CreatedAtAction(nameof(GetById), new { id = 99 },
                             new { source = "body", request.Name });

    // 🏷️ FROM HEADER — metadata from a request header
    // GET /api/products/whoami   (Header: X-Api-Key: ...)
    [HttpGet("whoami")]
    public IActionResult WhoAmI(
        [FromHeader(Name = "X-Api-Key")] string? apiKey)
        => Ok(new { source = "header", hasKey = apiKey is not null });

    // 📋 FROM FORM — file upload via multipart/form-data
    // POST /api/products/42/image
    [HttpPost("{id:int}/image")]
    public IActionResult UploadImage(
        [FromRoute] int id,
        [FromForm] string altText,
        IFormFile file)
        => Ok(new { source = "form", id, altText, size = file.Length });
}

public record CreateProductRequest(string Name, string Category, decimal Price);

How a request reaches the right parameters, step by step:

🎯
1. Routing selects the actionDay 2: verb + URL are matched to one method, e.g. POST /api/products → Create.
🔌
2. Binding fills each parameterFor every parameter, its source (explicit or inferred) is read: route values, query string, body via the JSON formatter, headers, form fields.
✅
3. Validation checks the resultDay 14: [ApiController] validates the bound model; if it fails, it returns 400 automatically before your code runs.
⚡
4. Your action runsEvery parameter is populated and typed — you just use them and return a result.

10🛠️ Hands-On: Bind from Every Source

Today's Do: build endpoints that bind from each source. Use the MyShop.Api / ProductsController from Day 2. The path:

  1. 🛣️ Route: confirm GetById([FromRoute] int id) on [HttpGet("{id:int}")] — call /api/products/42 and see id = 42.
  2. 🔎 Query: add Search([FromQuery] string? category, [FromQuery] int page = 1) on [HttpGet]; call /api/products?category=audio&page=2 and confirm both bind (and that omitting page defaults to 1).
  3. 🧳 Query object: refactor those into a ProductQuery class bound with [FromQuery] ProductQuery query; confirm the same URL fills its properties.
  4. 📦 Body: add Create([FromBody] CreateProductRequest request) on [HttpPost]; POST some JSON and confirm the object deserializes. Then try sending malformed JSON and watch [ApiController] return a 400.
  5. 🏷️ Header: add an action reading [FromHeader(Name = "X-Api-Key")] string? apiKey; send the header (Swagger/Scalar or curl -H) and confirm it binds — then confirm it's null when omitted (headers aren't required unless you make them so).
  6. 📋 Form: add an upload action with [FromForm] string altText and an IFormFile file; post multipart form data and confirm the file's name and length arrive.
  7. 🎛️ Combine: write one PUT /api/products/{id} that takes [FromRoute] int id, [FromBody] UpdateProductRequest request, and [FromHeader(Name="X-Correlation-Id")] string? correlationId — all three at once.
  8. 🧠 Test inference: remove the [FromRoute]/[FromQuery]/[FromBody] attributes from a few actions and confirm they still work (inferred) — then note which ones (header, form) break without the attribute.
  9. 📖 Open Swagger/Scalar and confirm each parameter shows in the right place (path, query, body, header) — proof your sources are annotated correctly.
✅ Self-Check

From your endpoints: (1) Why can you have only one [FromBody] parameter? (2) Which two sources are not inferred, so require an explicit attribute? (3) Why does [FromHeader] almost always need Name = "..."? (4) In your combined PUT, which piece came from where — and does that match Day 1's "identity vs payload vs metadata" split? (5) What happened when you sent invalid JSON to the [FromBody] endpoint, and before which step did that 400 fire?

11🧾 Quick Recap

🔌 Model bindingThe framework's automatic mapping of raw request data (path, query, body, headers, form) into your action's typed parameters — running after routing, before your code.
🛣️ [FromRoute]Binds from the URL path (route tokens). Resource identity: {id} → int id. Matches by name; use Name= to remap.
🔎 [FromQuery]Binds from the query string. Filter/sort/page, naturally optional with defaults. Can bind a whole query object. Never put secrets here.
📦 [FromBody]Binds the JSON body via input formatters. Create/update payloads. Max one per action (the body reads once); wrap multiple values in a DTO.
🏷️ [FromHeader]Binds an HTTP header. Correlation id, locale, API key. Almost always needs Name="X-…" because header names have hyphens.
📋 [FromForm]Binds multipart form fields & IFormFile uploads. Mutually exclusive with [FromBody] (both read the body).
🧠 InferenceWith [ApiController]: route-token→route, simple→query, complex→body are inferred; header and form are not — annotate them explicitly.
🎛️ CombiningThe clean pattern: route = identity, body = payload, header = metadata. Mix sources at the parameter level; match each value to where it belongs.

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is model binding in ASP.NET Core?🌱 JUNIOR

Model binding is the framework's automatic process of taking raw data from an incoming HTTP request — the URL path, query string, body, headers, or form fields — and mapping it onto your action method's parameters as strongly-typed .NET objects. Instead of manually reading Request.Query or parsing JSON, you just declare the parameters your method needs (an int id, a ProductDto product), and binding populates them for you before your code runs. It removes tedious, error-prone plumbing from every action.

Q2Name the main binding source attributes and where each reads from.🌱 JUNIOR

[FromRoute] reads from the URL path (route tokens like {id}). [FromQuery] reads from the query string (after the ?). [FromBody] reads from the request body (usually JSON). [FromHeader] reads from an HTTP request header. [FromForm] reads from posted form fields (multipart/form-data, including file uploads). There's also [FromServices], which isn't request data — it pulls a dependency from the DI container.

Q3Which source would you use for a product's id in /api/products/42?🌱 JUNIOR

[FromRoute], because the id is part of the URL path — it identifies which product. You'd have a route template like [HttpGet("{id:int}")] and a parameter [FromRoute] int id, and binding captures 42 from the path into id. Route is the right source for resource identity, following the REST convention that the path names the resource.

Q4Which source is used for a JSON payload sent with a POST?🌱 JUNIOR

[FromBody]. When a client sends structured data (like a new product) as JSON in the request body, [FromBody] tells the framework to deserialize that body into your object using an input formatter (the JSON formatter by default). It's the standard source for create and update operations — POST, PUT, and PATCH requests that carry a payload.

Q5Which source fits filtering and paging, like ?category=audio&page=2?🌱 JUNIOR

[FromQuery], because those values live in the query string and describe how you want a collection returned rather than identifying a specific resource. Query parameters are naturally optional and easy to give defaults (like int page = 1), which suits filtering, sorting, and paging. You'd write [FromQuery] string? category, [FromQuery] int page = 1.

Q6Why can you only have one [FromBody] parameter per action?🌱 JUNIOR

Because the request body is a stream that can be read only once. When an input formatter reads the body to bind one [FromBody] parameter, the stream is consumed and can't be read again for a second one. So you're limited to one [FromBody] parameter per action. If you need several values from the body, put them in a single object (a DTO) and bind that one object.

Q7Why does [FromHeader] usually include a Name property?🌱 JUNIOR

Because real HTTP header names often contain hyphens — like X-Correlation-Id or Accept-Language — and hyphens aren't valid in C# parameter names. The Name property maps the actual header name to a legal C# identifier: [FromHeader(Name = "Accept-Language")] string language. The Name option exists on the other attributes too, but headers need it most because of those hyphenated names.

Q8What is [FromForm] used for?🌱 JUNIOR

[FromForm] binds data submitted as form fields in multipart/form-data — the format used by HTML form posts and, most commonly in APIs, file uploads. You pair it with an IFormFile parameter to receive the uploaded file, plus [FromForm] for any accompanying text fields. It's the source to reach for whenever a client is uploading a file rather than sending JSON.

🟠 Mid

Q9How does binding source inference work under [ApiController], and which sources aren't inferred?🟠 MID

With [ApiController], the framework infers a source for most parameters so you can omit the attributes: a parameter whose name matches a route token is bound from the route; a simple type not in the route is bound from the query string; and a complex type (class/record) is bound from the body. The two sources that are not inferred are [FromHeader] and [FromForm] — they're treated as special and must be annotated explicitly, or the value simply won't bind. So header and form parameters always need their attribute; route, query, and body can rely on inference (though being explicit is often still clearer).

Q10When would you bind a complex object from the query string versus the body?🟠 MID

Bind a complex object from the query string ([FromQuery] ProductQuery query) when it's a bundle of optional, result-shaping filter/sort/page values on a GET — GETs have no body, and grouping many query parameters into one class keeps the signature clean and the filter reusable. Bind from the body ([FromBody] CreateProductRequest request) when it's the payload you're creating or updating on a POST/PUT/PATCH — structured data that belongs in the request body as JSON. Rule of thumb: read-only filters for a collection → query object; the resource data being submitted → body. One caveat: very large or deeply nested objects bind awkwardly from the query and produce poor OpenAPI specs, so keep query objects flat and modest.

Q11What happens to [FromQuery] or [FromRoute] attributes placed on the properties of a [FromBody] object?🟠 MID

They're ignored. When a parameter is bound with [FromBody], the request body is handed to an input formatter (the JSON deserializer), which reads only the body and doesn't understand model-binding source attributes. So a [FromQuery] decorating a property inside a body object has no effect — that property just gets whatever value the JSON body supplies (or its default if absent). The takeaway: you mix sources at the parameter level (one parameter from body, another from route or header), not by decorating individual properties within a body-bound object.

Q12Can [FromForm] and [FromBody] be used together on the same action? Why or why not?🟠 MID

No — they're mutually exclusive because both read the request body, just in different formats: [FromBody] expects the body to be JSON (parsed by an input formatter), while [FromForm] expects it to be multipart/form-data (form fields and files). A single request has one body with one content type, and it can be read only once, so an action is either JSON-based ([FromBody]) or form-based ([FromForm]), never both. If you need file uploads alongside structured data, put the structured fields as individual [FromForm] parameters in the same form, rather than mixing a JSON body with a form.

Q13Why shouldn't sensitive data go in the query string, and where should it go?🟠 MID

Because query strings are highly visible and get recorded in places you don't control: server and proxy access logs, browser history, bookmarks, and the Referer header that browsers send to third-party sites. Putting auth tokens, passwords, API keys, or personal data there exposes them in all those places — a real security and privacy leak. Sensitive values should travel in a header (e.g. an Authorization header, bound with [FromHeader]) or in the request body ([FromBody]), which aren't logged the same way. The query string is for harmless, shareable filter values only.

Q14Describe a clean way to combine multiple binding sources in one action.🟠 MID

The idiomatic enterprise pattern is: route identifies the resource, body carries the payload, header carries metadata. For example, an update endpoint: [FromRoute] int id for which product, [FromBody] UpdateProductRequest request for the new data, and [FromHeader(Name="X-Correlation-Id")] string? correlationId for tracing. Each source does exactly one job, so the action is unambiguous, matches REST conventions (identity in the path, payload in the body), and generates accurate OpenAPI output. Combining sources is normal and encouraged — the discipline is matching each value to its correct source, and mixing them at the parameter level (not by decorating a body object's properties).

Q15Should you rely on inference or annotate sources explicitly? Argue both sides.🟠 MID

For inference: it removes boilerplate from routine CRUD actions — a simple GetById(int id) or Create(ProductDto dto) reads cleanly without attribute noise, and the rules (route-token→route, simple→query, complex→body) are predictable once learned. For explicit annotation: it makes intent obvious to the next reader (no need to recall inference rules), produces more accurate OpenAPI specs — which matters because SDK/client generators rely on correct in: path/query/body metadata, and wrong inference silently emits broken clients — and it's more robust to refactoring (renaming a parameter or adding a route token won't silently change where a value comes from). A balanced position: lean on inference for simple, unambiguous actions; be explicit whenever the source isn't obvious, when a Name mapping is needed, for header/form (which aren't inferred anyway), and in public APIs where OpenAPI accuracy is important. Consistency within a team matters more than the exact choice.

Q16When exactly does model binding run in the request pipeline, and what happens if it fails?🟠 MID

Binding runs after routing has selected the action (Day 2) and before the action executes. The sequence: the request comes through the middleware pipeline, routing matches it to an action, then for each parameter the binding system reads the appropriate source (route values, query, body via input formatter, headers, form), converts the raw value to the parameter's type, and populates it; then model validation runs on the bound values. If binding can't convert a value (e.g. non-numeric text for an int) or validation fails, and the controller has [ApiController], the framework automatically returns a 400 Bad Request with a ProblemDetails body before your action method is called. That's why, by the time your code runs, its parameters are already populated with values that at least parsed to the right types — you don't write the parsing or the 400 yourself.

🔴 Senior

Q17Walk through what the model binding system does internally for one request.🔴 SENIOR

After routing selects an action, the framework inspects each parameter and determines its binding source — from an explicit [From…] attribute, or by inference (route-token match → route, simple type → query, complex type → body; header/form require explicit attributes). For non-body sources, value providers supply the raw data: route value providers, query-string providers, header providers, form providers — a value provider abstracts "give me the value for this key from my part of the request." A model binder then converts that raw string (or strings) into the target type: a SimpleTypeModelBinder for primitives, a complex-object binder that recursively binds each property of a class from the value providers, and so on. For [FromBody], it's different — an input formatter (chosen by the request's Content-Type, JSON by default via System.Text.Json) deserializes the whole body stream into the object. As values are bound, any conversion failures and validation results are recorded in ModelState. After binding, validation runs; with [ApiController], an invalid ModelState short-circuits to a 400. Finally the populated parameters are passed to the action. The extensibility points — custom value providers (e.g. reading from cookies) and custom model binders — let you plug in sources or conversions the defaults don't cover.

Q18Model binding is a security surface. What risks does it introduce and how do you mitigate them?🔴 SENIOR

The core risk is over-posting / mass assignment: if you bind directly to your domain entity, a client can set fields you never intended — sending {"isAdmin": true} or {"price": 0} and having binding happily populate them. Mitigation: bind to purpose-built DTOs / request models that expose only the fields a client may set, then map to the entity — never bind an EF entity straight from [FromBody]. Related risks: sensitive data in query strings (leaks via logs/history — use headers or body); large/hostile payloads (set request size limits, and be cautious with unbounded collections that could exhaust memory); file uploads via [FromForm] (validate size, content type, and file name; don't trust the client-supplied name; store outside the web root); and trusting bound values without validation (binding only guarantees the value parsed to the type, not that it's valid or authorized — you still validate ranges/rules per Day 14 and check authorization, e.g. that the caller may act on the route's id). The mindset: binding maps input into your types, but every bound value is still untrusted user input until validated and authorized. DTOs plus explicit validation plus authorization are the layered defense.

Q19How would you implement a custom model binder or value provider, and when is that justified?🔴 SENIOR

A custom value provider supplies raw data from a source the framework doesn't read by default — cookies, session, a custom store. You implement IValueProvider (a way to fetch a value by key) and IValueProviderFactory, and register the factory in MVC options; bound parameters can then draw from that source. A custom model binder controls how raw values are converted into a specific type — implement IModelBinder's BindModelAsync to read from the binding context and set a bound result, and apply it via [ModelBinder(typeof(...))] or an IModelBinderProvider. When is it justified? When you have a recurring, non-trivial conversion the defaults can't express cleanly — e.g. parsing a comma-separated range string into a strongly-typed DateRange, binding a domain value object from a compact string form, or reading a value from an unusual source consistently across many actions. When it's not justified: for one-off parsing (just take a string and parse it in the action), for business validation (that's validation's job, not binding's), or anything that needs I/O or services on the hot path (binders should be cheap and side-effect-free). And note union types are only supported via [FromBody] since the other sources bind strings without JSON parsing. The bar: reusable, purely-syntactic conversion the built-ins genuinely can't handle — otherwise keep binding simple and handle it in the action or a service.

Q20Why does binding to DTOs rather than domain entities matter beyond security?🔴 SENIOR

Security (avoiding over-posting) is the headline, but DTOs pay off in several other ways. Decoupling the API contract from the domain model: your external request/response shapes can stay stable while the internal entity evolves (rename a column, split a table) — without a DTO layer, every entity change is a breaking API change. Shaping input precisely: a CreateProductRequest exposes only what a client may send (name, price) and omits server-controlled fields (id, createdAt, audit columns), so the contract documents intent. Validation placement: validation attributes live on the DTO at the boundary, keeping the entity clean. Serialization control: you decide exactly what's exposed, avoiding accidentally leaking internal fields or navigation properties (and the circular-reference/over-fetch problems of serializing entities directly). Versioning: different API versions can use different DTOs while sharing one entity. Accurate OpenAPI: the generated schema reflects the real contract, not internal storage details. The cost is mapping code (mitigated by mapping libraries or simple manual maps), which is a worthwhile trade in any non-trivial API. The principle from Day 2 recurs: the web edge (controllers, DTOs, binding) should be a thin, deliberate boundary around a stable domain core, not a direct window into it.

Q21A GET endpoint needs 15 filter parameters. How do you design the binding?🔴 SENIOR

Fifteen individual [FromQuery] parameters make an unwieldy signature, so bind a query parameter object: a class whose properties are the filters, bound with [FromQuery] ProductSearchQuery query. That keeps the signature clean, makes the filter set reusable and testable, and lets you give sensible defaults. But at 15 fields I'd apply judgment: keep the object flat (deeply nested objects bind poorly from the query and produce confusing OpenAPI/SDK output — one of the documented anti-patterns), annotate it clearly so the OpenAPI spec lists each query parameter correctly, add validation for ranges and enums, and enforce paging with sane limits so a client can't request everything at once. If the "filter" is genuinely large and structured, that's a signal it might be better modeled as a POST to a /search sub-resource with the criteria in the body (a pragmatic, if not-purely-REST, choice many APIs make for complex searches) — trading GET's cacheability for a cleaner payload. Also consider whether some of those 15 are really distinct endpoints or could be simplified. So: query object for cleanliness, flat and well-annotated for tooling, validated and paged for safety, and reconsider a body-based search endpoint if the criteria are truly complex. The decision balances usability, OpenAPI/SDK quality, caching, and how variable the client's needs are.

Q22How does model binding relate to validation, and where should each responsibility sit?🔴 SENIOR

They're distinct, sequential steps with different jobs. Binding is syntactic: it maps request data to typed parameters and records conversion failures (a non-numeric string for an int) in ModelState. Validation is semantic: it checks that the bound values satisfy rules — required fields present, numbers in range, strings within length, cross-field constraints — via data annotations ([Required], [Range]) or a library like FluentValidation. Order: bind first, then validate; both feed ModelState, and with [ApiController] an invalid ModelState auto-returns a 400 before the action runs. Where responsibilities sit: binding is the framework's job (you just choose sources); simple field-level validation belongs on the DTO via attributes at the boundary; complex or business-rule validation belongs in explicit validators or the domain/service layer (Day 14), because binding and attribute validation can't express things like "this email isn't already taken" or cross-entity rules. The key senior insight is that binding success does not mean the input is valid or safe — it only means it parsed to the right type. Every bound value is still untrusted until validated (for correctness) and authorized (for permission). So the layering is: bind (framework) → validate shape/rules (DTO attributes + validators) → enforce business rules and authorization (services) — never treat "it bound" as "it's good."

Q23Does model binding work the same way in minimal APIs as in controllers?🔴 SENIOR

The concepts are the same — the framework maps request parts to handler parameters, and the same [FromRoute]/[FromQuery]/[FromBody]/[FromHeader]/[FromForm]/[FromServices] attributes apply — but minimal APIs use a somewhat different, more explicit binding mechanism than the MVC model binder. In minimal APIs, parameters are bound by a set of rules: route-token-name matches bind from the route, other simple types from the query, complex types from the body (JSON), and services from DI — broadly similar inference to [ApiController], but resolved at endpoint-construction time and with some differences (e.g. minimal APIs support special parameter types like HttpContext, HttpRequest, CancellationToken, and [AsParameters] to bundle parameters into a struct/record; and custom binding is done via static TryParse/BindAsync methods on the type rather than IModelBinder). Minimal APIs also don't run the full MVC filter/model-validation pipeline the same way — validation isn't automatic out of the box the way [ApiController] gives you auto-400s (though it can be added). So: same mental model and attributes, different implementation and some different conventions/extensibility. A senior should know that "model binding" as a concept spans both, but the MVC model binder (value providers, IModelBinder, automatic ModelState validation) is specific to controllers, while minimal APIs have their own lighter-weight parameter-binding system.

Q24Why does choosing the right binding source matter beyond "it works"?🔴 SENIOR

Because the source you choose is part of your API's public contract, and getting it wrong has downstream costs even when the endpoint "works" in a quick test. OpenAPI/SDK correctness: the binding source determines whether a parameter appears in the spec as in: path, in: query, in: header, or requestBody. If sources are wrong or misinferred, the spec misrepresents the API, SDK generators emit broken clients, and integration tests can pass locally while failing in real client environments. RESTful predictability: putting identity in the path, filters in the query, payloads in the body, and metadata in headers makes the API behave the way every other well-designed API does, so consumers can predict it (Day 1's core value). Security: source choice affects exposure — secrets in the query leak via logs; sensitive data belongs in headers/body. Caching & semantics: query parameters participate in URL-based caching and are shareable/bookmarkable; body content isn't — so a filter in the query behaves differently from the same filter in a body. Maintainability: conflating identity with filter state (e.g. /orders/active/page/2 instead of /orders?status=active&page=2) makes URLs rigid and hard to evolve. So the source isn't a mere implementation detail — it shapes the contract, the tooling, the security posture, and the evolvability of the API. Choosing deliberately is part of designing the API, not just wiring up a parameter.
