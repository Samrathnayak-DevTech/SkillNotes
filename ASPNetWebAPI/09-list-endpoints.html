List Endpoints 📃

"Get all products" sounds harmless — until the table has two million rows and your API tries to serialize every one into a single response. Real list endpoints hand the client control: show me page 2, only the products under $20, sorted by price. Today we build a listing that's async, filterable, sortable, and paged — pushing all that work down to SQL Server via IQueryable so the database does the heavy lifting, not your server's memory.

⚡ Async actions 📄 Pagination 🔍 Filtering ↕️ Sorting 🧮 IQueryable 📦 Result shaping
01

Why List Endpoints Are Hard 🪨

A "read one" endpoint returns a single row — easy. A "read all" endpoint is deceptively dangerous, because all can mean millions of rows. Return them unbounded and you overload the database, saturate the network, and can crash the client's browser. A production list endpoint never returns everything; it returns a controlled slice, shaped by what the client asked for.

Four levers give the client that control, and they combine into one query:

🔍

Filtering

Narrow which rows come back — "only active products under $20."

↕️

Sorting

Decide the order — "newest first," "cheapest first."

📄

Pagination

Return one page at a time — "rows 21–40," not all of them.

📦

Shaping

Trim the columns to a DTO and wrap them with page metadata.

🔑 The golden rule

Do all four at the database, not in memory. If you load every row and then filter/sort/page in C#, you've already paid the cost you were trying to avoid. The whole game today is building one IQueryable that translates into a single efficient SQL query.

📚 Analogy — a library request desk

You don't ask the librarian to wheel out every book in the building. You say: "cookbooks (filter), alphabetical by author (sort), just the first shelf-full (page)." They fetch exactly that. A good list endpoint is that request desk — the client describes the slice, the database delivers only it.

The big picture 🗺️

List Endpoints

02

Async Actions & the Magic of IQueryable ⚡

Two ideas make efficient list endpoints possible. First, async: database calls are I/O, so awaiting them (with ToListAsync, etc.) frees the thread to serve other requests instead of blocking — essential for scalability. Second, and more subtly: IQueryable<T> uses deferred execution.

🔑 Deferred execution, explained

When you write db.Products.Where(...).OrderBy(...).Skip(...).Take(...), nothing runs yet. You're building up a description of a query. EF Core only translates it to SQL and hits the database when you materialize it — calling ToListAsync(), FirstOrDefaultAsync(), CountAsync(), and so on. That's why you can add filtering, sorting, and paging step by step, and still get one optimized SQL query at the end.

C# — building a query without running it
// Each line refines the query — no database call happens here.
IQueryable<Product> query = db.Products.AsNoTracking();

query = query.Where(p => p.Price < 20);          // filter
query = query.OrderBy(p => p.Name);                // sort
query = query.Skip(20).Take(10);                    // page

// NOW it runs — one SQL query with WHERE + ORDER BY + OFFSET/FETCH.
var results = await query.ToListAsync();
⚠️ Materialize too early and you lose everything

The classic bug: calling ToListAsync() (or ToList()) before filtering/paging. Once you do, the entire table is pulled into memory and every later Where/Skip runs in C# on the full set — the exact inefficiency you were avoiding. Keep it IQueryable until the very last step. As a signal: if you see .ToList() followed by more .Where(), that's a red flag.

✅ AsNoTracking for lists

List endpoints are read-only, so start the query with .AsNoTracking() (Day 8). EF Core skips change-tracking setup for the returned rows — less memory, faster reads. You're returning DTOs, not editing entities, so there's nothing to track.

03

Filtering — Narrowing the Rows 🔍

Filtering means adding Where clauses based on optional query-string parameters. The trick: only apply a filter when the client actually supplied it. You conditionally compose the IQueryable, and unsupplied filters simply aren't added.

C# — conditional filtering
IQueryable<Product> query = db.Products.AsNoTracking();

// Only add each Where if the parameter was provided.
if (!string.IsNullOrWhiteSpace(request.Search))
    query = query.Where(p => p.Name.Contains(request.Search));

if (request.MinPrice is not null)
    query = query.Where(p => p.Price >= request.MinPrice);

if (request.MaxPrice is not null)
    query = query.Where(p => p.Price <= request.MaxPrice);

if (request.InStockOnly == true)
    query = query.Where(p => p.Stock > 0);

A request like GET /api/products?search=note&maxPrice=20&inStockOnly=true composes three Where clauses; EF Core folds them into one SQL WHERE. The parameters ride in on a bound DTO:

C# — the query parameters as a DTO
public class ProductQuery
{
    public string? Search { get; set; }
    public decimal? MinPrice { get; set; }
    public decimal? MaxPrice { get; set; }
    public bool? InStockOnly { get; set; }

    public string? SortBy { get; set; }        // used in §04
    public bool Desc { get; set; }
    public int Page { get; set; } = 1;         // used in §05
    public int PageSize { get; set; } = 20;
}
⚠️ Contains vs. real search

Contains translates to SQL LIKE '%term%'. It works, but a leading wildcard can't use a normal index, so it scans — fine for modest tables, slow for huge ones. For serious text search, reach for full-text search or a search engine. Also validate/whitelist filter inputs; blindly trusting client-supplied filter fields can leak data or hurt performance.

✅ Nullable params = "optional"

Making filter parameters nullable (decimal?, bool?) is what lets you distinguish "not supplied" from "supplied as zero/false." That distinction is the whole basis of conditional filtering.

04

Sorting — Choosing the Order ↕️

Sorting adds OrderBy / OrderByDescending (and ThenBy for tie-breakers). The danger here is security and stability: you must never pass a raw client string into a dynamic sort blindly, and your order must be fully unique.

C# — safe, whitelisted sorting
// Whitelist the allowed sort fields — never trust a raw column name.
query = (request.SortBy?.ToLower()) switch
{
    "price" => request.Desc
        ? query.OrderByDescending(p => p.Price)
        : query.OrderBy(p => p.Price),
    "name"  => request.Desc
        ? query.OrderByDescending(p => p.Name)
        : query.OrderBy(p => p.Name),
    _       => query.OrderBy(p => p.Name)      // safe default
};

// Tie-breaker on the unique key guarantees a STABLE order.
query = ((IOrderedQueryable<Product>)query).ThenBy(p => p.Id);
⚠️ Ordering must be fully unique — this bites everyone

Microsoft is explicit: whatever pagination method you use, always make the ordering fully unique. If you sort only by Price and several products share a price, the database is free to return those tied rows in a different order on each query — so paging can silently skip or duplicate rows across pages. Fix it by adding a unique tie-breaker like .ThenBy(p => p.Id).

🔑 Why the whitelist matters

Accepting an arbitrary property name to sort by (via string-based dynamic LINQ) opens the door to errors and abuse — clients could sort by expensive un-indexed columns or probe your schema. A switch over known fields keeps sorting safe, predictable, and index-friendly. Add a database index on the columns you actually sort by; it's the single biggest performance win.

05

Pagination — Offset (Skip/Take) 📄

Offset pagination is the familiar "page number + page size" model. You Skip past earlier pages and Take the current one. In SQL these become OFFSET and FETCH/LIMIT.

C# — offset pagination
// Clamp the page size so a client can't ask for a million rows.
int pageSize = Math.Clamp(request.PageSize, 1, 100);
int page = Math.Max(request.Page, 1);

// Total count runs BEFORE Skip/Take, on the filtered query.
int total = await query.CountAsync();

var items = await query
    .Skip((page - 1) * pageSize)   // jump past earlier pages
    .Take(pageSize)                // take just this page
    .Select(p => new ProductDto(p.Id, p.Name, p.Price))
    .ToListAsync();

👍

Pros

Dead simple, and supports jumping to any page number — great for classic "page 1 2 3 … 50" UIs and admin dashboards.

👎

Cons

The database must still scan and discard every skipped row — page 1,000 processes ~10,000 rows to return 10. And concurrent inserts/deletes can shift rows, causing skips or duplicates between pages.

🔑 Count on the filtered query, before paging

Call CountAsync() on the query after filtering but before Skip/Take, so the total reflects the whole filtered set (that's how the client knows how many pages exist). It's a second SQL query — the cost of showing "page 3 of 27." Keyset pagination (next) skips this cost entirely.

06

Keyset Pagination — For Large Data 🚀

Microsoft's docs recommend keyset pagination (also called seek or cursor pagination) as the alternative to offset for large datasets. Instead of skipping N rows, you use a WHERE clause to start right after the last row the client already saw. No skipping means no scanning discarded rows.

C# — keyset pagination (single key)
// The client sends the last Id it saw; we fetch the next page after it.
var nextPage = await db.Products
    .AsNoTracking()
    .OrderBy(p => p.Id)
    .Where(p => p.Id > request.AfterId)   // seek, don't skip
    .Take(pageSize)
    .ToListAsync();

When you order by a non-unique column, you need a composite comparison to keep the seek deterministic — the exact pattern from the EF Core docs:

C# — keyset with two keys (date, then id)
var nextPage = await db.Products
    .AsNoTracking()
    .OrderBy(p => p.CreatedOn).ThenBy(p => p.Id)
    .Where(p => p.CreatedOn > lastDate
             || (p.CreatedOn == lastDate && p.Id > lastId))
    .Take(pageSize)
    .ToListAsync();

✅ Which to pick

Use offset for most APIs and admin screens where users jump to specific page numbers on small-to-medium data. Use keyset for large datasets and infinite-scroll feeds where performance and consistency matter more than random page access. Either way, index the columns you order by — for a composite keyset, use a composite index in the same direction as your sort.

🔑 SQL row-value note

Most databases support a tidy form — WHERE (CreatedOn, Id) > (@lastDate, @lastId) — but EF Core doesn't yet express that in LINQ, so you write the expanded OR condition shown above. It produces the same correct result.

07

Shaping the Result Set 📦

Two more shaping steps make the response client-friendly. First, project to a DTO inside the query (Day 5) so SQL selects only needed columns. Second, wrap the page in an envelope that carries metadata — the client needs to know there are more pages.

C# — a paged-result envelope
public record PagedResult<T>(
    IReadOnlyList<T> Items,
    int Page,
    int PageSize,
    int TotalCount)
{
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
    public bool HasNext => Page < TotalPages;
    public bool HasPrevious => Page > 1;
}
JSON — what the client receives
{
  "items": [
    { "id": 21, "name": "Notebook", "price": 4.99 },
    { "id": 22, "name": "Pen",      "price": 1.20 }
  ],
  "page": 2,
  "pageSize": 20,
  "totalCount": 137,
  "totalPages": 7,
  "hasNext": true,
  "hasPrevious": true
}
🔑 Where should the metadata live?

Two common conventions: an envelope in the body (shown above — easy to consume, self-describing), or the raw array in the body with pagination info in a response header (e.g. a custom X-Pagination header or the standard Link header). Envelopes are the most common in typical JSON APIs. Pick one and apply it consistently across every list endpoint.

✅ Always cap the page size

Never let the client dictate an unbounded pageSize. Clamp it to a sane maximum (e.g. 100) as shown in §05 — otherwise ?pageSize=1000000 reintroduces the exact problem pagination exists to solve. Provide sensible defaults (page 1, size 20) so a bare GET /api/products still behaves.

08

The Full Endpoint 🧩

Here's today's "Do," assembled: a paged, filterable, sortable list endpoint. The controller binds the query DTO and delegates; the service composes one IQueryable and materializes it once.

C# — the controller action
[HttpGet]
public async Task<ActionResult<PagedResult<ProductDto>>> GetAll(
    [FromQuery] ProductQuery request) =>
    await service.ListAsync(request);
C# — the service: compose, count, page, shape
public async Task<PagedResult<ProductDto>> ListAsync(ProductQuery q)
{
    int pageSize = Math.Clamp(q.PageSize, 1, 100);
    int page = Math.Max(q.Page, 1);

    // 1) BASE — read-only
    IQueryable<Product> query = db.Products.AsNoTracking();

    // 2) FILTER — only what was supplied
    if (!string.IsNullOrWhiteSpace(q.Search))
        query = query.Where(p => p.Name.Contains(q.Search));
    if (q.MinPrice is not null) query = query.Where(p => p.Price >= q.MinPrice);
    if (q.MaxPrice is not null) query = query.Where(p => p.Price <= q.MaxPrice);

    // 3) SORT — whitelist + unique tie-breaker
    query = q.SortBy?.ToLower() switch
    {
        "price" => q.Desc ? query.OrderByDescending(p => p.Price)
                          : query.OrderBy(p => p.Price),
        _       => q.Desc ? query.OrderByDescending(p => p.Name)
                          : query.OrderBy(p => p.Name)
    };
    query = ((IOrderedQueryable<Product>)query).ThenBy(p => p.Id);

    // 4) COUNT — filtered total, before paging
    int total = await query.CountAsync();

    // 5) PAGE + SHAPE — skip/take, project to DTO, run once
    var items = await query
        .Skip((page - 1) * pageSize)
        .Take(pageSize)
        .Select(p => new ProductDto(p.Id, p.Name, p.Price))
        .ToListAsync();

    return new PagedResult<ProductDto>(items, page, pageSize, total);
}
✅ Try it

GET /api/products?search=note&maxPrice=20&sortBy=price&desc=true&page=2&pageSize=10 → returns page 2 of in-name-"note", ≤ $20 products, most expensive first, ten per page, wrapped with total-count metadata. One request, one SQL query, bounded response.

09

Query Build Order — Filter → Sort → Page → Shape 🔄

The order you compose the query in isn't arbitrary — each step depends on the last. Follow this sequence every time.

Step 1 · 🧱 base query
Start IQueryable, AsNoTracking

Begin from the DbSet; nothing executes yet. Read-only, so no change tracking.

Step 2 · 🔍 filter
Add Where clauses

Conditionally, per supplied parameter. Filtering first means everything after works on the smaller set.

Step 3 · ↕️ sort
OrderBy + unique ThenBy

Whitelist the field; add a unique tie-breaker so paging is stable. Order must exist before Skip/Take.

Step 4 · 🔢 count
CountAsync on the filtered query

Offset only. Run before Skip/Take so the total reflects all filtered rows (for page metadata).

Step 5 · 📄📦 page + shape
Skip/Take → Select DTO → ToListAsync

Page the ordered set, project to a DTO, then materialize. This is the single database round-trip for the items.

⚠️ Order matters

Sorting must come before paging — Skip/Take without a stable OrderBy gives arbitrary, shifting results. And counting must come before Skip/Take (it counts the filtered set, not one page). Get the sequence wrong and you get wrong data, not just slow data.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1Why not just return every row from a list endpoint?🌱 Junior

Because a table can have millions of rows. Returning them all overloads the database, saturates the network, and can crash the client. A real list endpoint returns a controlled slice — filtered, sorted, and paged — so the response stays small and fast no matter how big the table grows.

Q2What is pagination?🌱 Junior

Returning data in pages instead of all at once. The client asks for a page number and page size; the server returns just that subset. In EF Core the common approach uses Skip() and Take(), which become SQL OFFSET and FETCH/LIMIT, so only the requested rows come back.

Q3How do you filter a list based on query parameters?🌱 Junior

Add Where clauses to the query, but only when the parameter is supplied. You bind the query-string values to a DTO with nullable properties, then check each one — e.g. if (maxPrice is not null) query = query.Where(p => p.Price <= maxPrice). Unsupplied filters simply aren't added.

Q4How do you sort results by a column?🌱 Junior

Use OrderBy (ascending) or OrderByDescending (descending), with ThenBy for tie-breakers. In an API you map the client's requested field to a known column — usually with a switch over allowed values — rather than trusting a raw column name.

Q5Why should list actions be async?🌱 Junior

Database calls are I/O operations. Awaiting them (ToListAsync, CountAsync) frees the request thread to serve other requests instead of blocking while waiting on the database. That lets the server handle many more concurrent requests — it's important for scalability.

Q6What do Skip and Take do?🌱 Junior

Skip(n) bypasses the first n rows; Take(n) returns the next n. Together they select one page: for page p of size s, Skip((p-1)*s).Take(s). EF Core translates them to SQL OFFSET and FETCH, so the paging happens in the database.

Q7Why cap the page size?🌱 Junior

So a client can't request an unbounded number of rows (?pageSize=1000000), which would reintroduce the exact overload pagination is meant to prevent. You clamp the incoming value to a safe maximum (e.g. 100) and provide sensible defaults so a bare request still works.

Q8What extra info should a paged response include?🌱 Junior

Beyond the items, metadata that lets the client navigate: the current page, page size, total count (and thus total pages), and often flags like hasNext/hasPrevious. This is usually wrapped in an envelope object in the response body, or placed in a response header.

Q9What is IQueryable's deferred execution and why does it matter here?🟠 Mid

An IQueryable doesn't run when you build it — adding Where, OrderBy, Skip, Take just describes the query. It executes only when you materialize it (ToListAsync, CountAsync, etc.). That's what lets you compose filtering, sorting, and paging step by step and still get one efficient SQL query that does all the work in the database.

Q10What's the danger of calling ToList too early?🟠 Mid

It materializes the query immediately, pulling the whole (unfiltered, unpaged) result into memory. Any Where/OrderBy/Skip after that runs in C# on the full set — client-side evaluation — defeating the purpose. Keep the query as IQueryable until the final materialization so filtering and paging happen in SQL.

Q11Why must the ordering be fully unique when paginating?🟠 Mid

If the sort isn't unique — say you order only by price and several rows share a price — the database can return those tied rows in a different order on each query. Across pages, that means rows can be skipped or shown twice. Adding a unique tie-breaker like ThenBy(p => p.Id) guarantees a stable, deterministic order.

Q12Offset vs. keyset pagination — what's the difference?🟠 Mid

Offset uses Skip/Take to jump past N rows — simple and supports any page number, but the database still scans all skipped rows (slow at deep pages) and can skip/duplicate rows if data changes. Keyset uses a WHERE clause to start right after the last row seen — fast and stable even on huge datasets, but only supports next/previous navigation, not jumping to arbitrary pages.

Q13Where do you compute the total count, and why there?🟠 Mid

On the filtered query, before Skip/Take. You want the total of all rows matching the filters (not just the current page) so the client can compute how many pages exist. It's a separate CountAsync query — the cost of showing "page 3 of 27." Keyset pagination avoids this count entirely.

Q14How do you sort safely by a client-provided field?🟠 Mid

Whitelist the allowed fields — map the incoming string to a known column with a switch, with a safe default. Never feed a raw client string into dynamic LINQ, which could let callers sort by expensive un-indexed columns or probe the schema. And index the columns you actually allow sorting on.

Q15How does Contains translate to SQL, and what's the catch?🟠 Mid

Name.Contains(term) becomes LIKE '%term%'. It works, but the leading wildcard prevents the database from using a normal index, so it scans the table — fine for small data, slow for large. For serious text search, use full-text search or a dedicated search engine rather than Contains.

Q16In what order should you compose filter, sort, and page?🟠 Mid

Filter → sort → count → page (then shape). Filter first so everything works on a smaller set; sort before paging because Skip/Take needs a stable order; count the filtered set before paging for metadata; then Skip/Take, project to a DTO, and materialize. Because it's all one IQueryable, the sequence you write becomes one SQL query.

Q17Why does offset pagination degrade at deep pages?🔴 Senior

Because OFFSET still requires the database to locate and discard every skipped row before returning the page. Page 1,000 with size 10 means processing ~10,000 rows to hand back 10. The cost grows linearly with the offset. Keyset pagination avoids it by seeking directly to the starting point via an indexed WHERE, so performance stays flat regardless of how deep you go.

Q18How do you implement keyset pagination with a non-unique sort column?🔴 Senior

Add a unique tie-breaker to both the ordering and the seek condition. Ordering by date then id, the WHERE becomes: date > lastDate OR (date == lastDate && id > lastId). This picks up exactly where the previous page ended. Most databases support the tidy row-value form (date, id) > (@lastDate, @lastId), but EF Core doesn't express that in LINQ yet, so you write the expanded OR condition. Back it with a composite index matching the sort order.

Q19What indexing strategy supports filtering and sorting?🔴 Senior

Index the columns you filter and order by — it's the single biggest performance lever. For multi-column ordering (or composite keyset), use a composite index in the same direction as the sort (EF Core supports specifying sort order in composite indexes since EF Core 7). Watch the trade-off: indexes speed reads but slow writes and use space, so index the columns you actually query, not every column.

Q20How can a count query become a bottleneck, and what can you do?🔴 Senior

On large tables, COUNT over a heavily-filtered set can be as expensive as the page query itself, doubling the work per request. Options: switch to keyset pagination (no count needed), return an approximate/cached count, omit the exact total and just expose hasNext (fetch pageSize+1 and check for overflow), or cache counts for common filter combinations. Exact "page X of Y" is a feature with a real cost.

Q21How do you keep pagination consistent when data changes between requests?🔴 Senior

Offset pagination is inherently vulnerable — an insert or delete in earlier rows shifts the whole result set, so page 3 might repeat or skip a row seen on page 2. Keyset pagination fixes this because it anchors on the last-seen key value rather than a positional offset, so concurrent changes to other rows don't shift your window. For infinite-scroll feeds especially, keyset is the consistent choice.

Q22How should filtering and sorting be exposed and validated on the API surface?🔴 Senior

Bind a strongly-typed query DTO with [FromQuery], validate it (ranges on page/size, allowed sort values), and whitelist filterable/sortable fields explicitly. Avoid generic "filter by any field/operator" surfaces unless you deliberately build a safe query language — they invite injection-like abuse, expensive scans, and schema leakage. Document the supported params and enforce sane defaults and caps server-side.

Q23When would you avoid returning a total count at all?🔴 Senior

When the extra count query is too costly or the UI doesn't need exact page numbers — e.g. infinite scroll or "load more" feeds. There you use keyset pagination and expose only a "next cursor"/hasNext, often by fetching one extra row (pageSize + 1) and checking whether it came back. This gives smooth forward navigation with no count overhead, trading away the "page X of Y" display that users of a feed don't need anyway.
