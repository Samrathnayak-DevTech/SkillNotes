// day 13 — who's allowed to do what

Authorization 🛂

Yesterday we proved who a user is. Today we decide what they're allowed to do. ASP.NET Core gives you a spectrum: the blunt-but-simple [Authorize(Roles=…)], the flexible claims-based checks, and the fully general policy-based model with custom requirements and handlers that can encode any rule your app can compute. We'll restrict an endpoint to a role and to a custom policy — and see that all three styles are really one engine wearing different hats.

🚧 [Authorize] 👥 Roles 🏷️ Claims 📜 Policies 🧩 Requirements ⚙️ Handlers
01

What Authorization Is 🛂

Authorization (AuthZ) runs after authentication (Day 12). Authentication established who the user is — building their ClaimsPrincipal and setting HttpContext.User. Authorization asks the next question: given who they are, are they permitted to do this? A logged-in user is not automatically an allowed user.

🏢 Analogy — the office building

Your badge (authentication) proves you're an employee and gets you in the front door. But the same badge only opens certain doors: the server room needs IT clearance, the exec floor needs a manager badge, the vault needs both a role and a specific permission. Authorization is the set of rules on each door — not "are you an employee?" but "does this employee get through this door?"

🔑 401 vs. 403, one more time

These two live at the authentication/authorization boundary. 401 Unauthorized = not authenticated (no valid identity). 403 Forbidden = authenticated, but authorization said no. If your authorization rule fails, the user gets a 403 — proof they're logged in but lack the required role, claim, or policy.

Claims & the ClaimsPrincipal 🏷️

Everything today builds on claims. A claim is a name–value statement about the user — role: Admin, department: Engineering, age: 34 — asserted by a trusted issuer. After authentication, the user's claims live on HttpContext.User, a ClaimsPrincipal. Every authorization decision is ultimately "does this principal's claims satisfy the rule?"

🔑 A role is just a claim

Here's the unifying idea: a role is simply a claim of type role. When your JWT (Day 12) carries ClaimTypes.Role = "Admin", "role-based" authorization is really claims-based authorization with a well-known claim type. That's why all three styles converge — they're all inspecting claims on the principal.

✅ Reading claims in code

You can inspect the principal directly: User.IsInRole("Admin"), User.HasClaim("department", "Engineering"), or User.FindFirst("tenure")?.Value. The authorization system does this checking for you declaratively, so you rarely write it by hand — but knowing it's just claim lookups demystifies the whole thing.

03

Role-Based Authorization 👥

The simplest style, from Day 12: gate an endpoint on one or more named roles. Great for coarse buckets like Admin, Manager, User.

C# — roles on controllers & actions
[Authorize(Roles = "Admin")]                 // must be in Admin
[HttpDelete("{id}")]
public IActionResult Delete(int id) => NoContent();

[Authorize(Roles = "Admin,Manager")]         // Admin OR Manager (comma = OR)
[HttpPost]
public IActionResult Create() => Ok();

Stacking attributes means AND, while a comma-separated list means OR:

C# — AND vs OR with roles
[Authorize(Roles = "Admin")]     // must be Admin…
[Authorize(Roles = "Finance")]   // …AND also Finance (both required)
public class PayrollController : ControllerBase { }
⚠️ Roles don't scale as rules get real

Roles answer "which bucket are you in?" — but real rules quickly outgrow buckets: "must be over 18," "must own this record," "only during business hours," "must belong to the same tenant." You can keep inventing roles (AdminNightShiftEU…), but that's a combinatorial mess. When matching a name isn't enough, reach for policies.

✅ Roles need to be in the token

For [Authorize(Roles=…)] to work with JWT (Day 12), the token must carry role claims — you added them at login with GetRolesAsync → ClaimTypes.Role. No role claim in the token, no match, and the user gets a 403 even if they "are" an admin in your database.

04

Claims-Based Authorization via Policies 📜

The next step up: instead of matching a role name, require a specific claim. You express this as a named policy registered once in AddAuthorization, then apply it with [Authorize(Policy=…)].

C# — defining claim policies
builder.Services.AddAuthorization(options =>
{
    // must simply HAVE the "EmployeeId" claim (any value)
    options.AddPolicy("IsEmployee", policy =>
        policy.RequireClaim("EmployeeId"));

    // must have "department" claim equal to one of these values
    options.AddPolicy("EngineeringOnly", policy =>
        policy.RequireClaim("department", "Engineering", "Platform"));

    // requirements chain with AND — needs BOTH
    options.AddPolicy("SeniorEngineer", policy => policy
        .RequireClaim("department", "Engineering")
        .RequireClaim("level", "Senior"));
});
C# — applying a policy
[Authorize(Policy = "EngineeringOnly")]
[HttpGet("builds")]
public IActionResult GetBuilds() => Ok();

// Minimal API equivalent
app.MapGet("/builds", () => Results.Ok())
   .RequireAuthorization("EngineeringOnly");

RequireClaim("dept", "Eng")has that claim with one of the listed values
RequireUserName("bob")has that exact username
RequireAssertion(ctx => …)satisfies an inline lambda (escape hatch for OR/simple logic)
⚠️ Chained requirements are AND, not OR

Chaining .RequireClaim(a).RequireClaim(b) means the user needs both — every requirement in a policy must pass. If you need OR ("VIP or employee or CEO"), chaining won't do it. For simple OR, use RequireAssertion with a lambda; for richer OR logic, use a custom requirement with multiple handlers (§06), since any one handler succeeding satisfies the requirement.

Here's the insight that ties the three styles together: they're all policy-based authorization. Roles and claims are just convenient shorthand that the framework compiles into policies at runtime.

🔑 Why this matters

Once you see that [Authorize(Roles="Admin")] and RequireClaim(...) are just pre-built requirements in a policy, custom policies stop feeling exotic. They're the same mechanism with a requirement and handler you write yourself. The framework ships requirements/handlers for the common cases; you extend the system for everything else. A policy is a named bundle of requirements, and all requirements must pass (AND) for the policy to succeed.

✅ DefaultPolicy vs. FallbackPolicy

Two options on AuthorizationOptions people mix up. DefaultPolicy applies when [Authorize] is used with no policy named (default: require an authenticated user). FallbackPolicy applies to endpoints with no authorization metadata at all — setting it to RequireAuthenticatedUser() is a handy way to make your whole API secure-by-default, requiring an explicit [AllowAnonymous] to opt out.

06

Custom Policies — Requirements & Handlers 🧩

When matching a value isn't enough — you need to compute something — you write a custom policy from two small pieces. A requirement carries the rule's data; a handler holds the logic.

C# — 1) the requirement (just data)
using Microsoft.AspNetCore.Authorization;

// A marker interface — the requirement just holds the rule's data.
public class MinimumTenureRequirement(int months) : IAuthorizationRequirement
{
    public int Months { get; } = months;
}
C# — 2) the handler (the logic, from DI)
public class MinimumTenureHandler : AuthorizationHandler<MinimumTenureRequirement>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        MinimumTenureRequirement requirement)
    {
        // Always confirm identity first — handlers run even for anonymous users.
        var claim = context.User.FindFirst("tenure_months");
        if (claim is null)
            return Task.CompletedTask;   // neither pass nor fail; just don't succeed

        if (int.TryParse(claim.Value, out var months) && months >= requirement.Months)
            context.Succeed(requirement);   // ✅ requirement met

        return Task.CompletedTask;
    }
}
C# — 3) register the handler + define the policy
// Register the handler in DI (singleton is fine when it holds no per-request state)
builder.Services.AddSingleton<IAuthorizationHandler, MinimumTenureHandler>();

builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("VestedEmployee", policy =>
        policy.AddRequirements(new MinimumTenureRequirement(12)));
});
🔑 Succeed / Fail / do nothing — the three outcomes

In a handler: call context.Succeed(requirement) when it's met. Call context.Fail() to hard-fail — this vetoes the requirement even if another handler would succeed (use for "banned user" style overrides). Or return without calling either to simply not satisfy it — leaving room for another handler on the same requirement to succeed. That last distinction is how OR logic works across multiple handlers.

⚠️ Handlers run even for unauthenticated users

All registered handlers for the pending requirements are invoked — the pipeline doesn't short-circuit on an anonymous user before your handler runs. So always null-check the claim (as above) rather than assuming a user/claim exists; otherwise you risk a NullReferenceException. Pair a custom policy with RequireAuthenticatedUser() if identity is a precondition.

✅ Handlers get DI — that's the superpower

Because handlers are resolved from the container, they can inject anything: a TimeProvider for "business hours only" (and a fake clock in tests), your DbContext to look up ownership, an HttpClient to an entitlement service. Any rule your app can compute, the authorization layer can enforce — this is what roles and simple claims can't do. If a handler injects a scoped service, register it scoped rather than singleton.

07

Applying & Combining Policies 🔗

Today's "Do": restrict one endpoint to a role and another to a custom policy — here's the payoff, side by side.

C# — role-restricted AND policy-restricted endpoints
[ApiController]
[Route("api/reports")]
[Authorize]                              // baseline: must be logged in
public class ReportsController : ControllerBase
{
    // 🔒 restricted to a ROLE
    [HttpGet("summary")]
    [Authorize(Roles = "Manager")]
    public IActionResult Summary() => Ok();

    // 🔒 restricted to a custom POLICY
    [HttpGet("vested")]
    [Authorize(Policy = "VestedEmployee")]
    public IActionResult VestedReport() => Ok();
}

You can stack a role and a policy on the same endpoint — they combine with AND (both must pass):

C# — combining role + policy (AND)
[HttpDelete("{id}")]
[Authorize(Roles = "Manager")]           // must be a Manager…
[Authorize(Policy = "VestedEmployee")]     // …AND satisfy the policy
public IActionResult Delete(int id) => NoContent();
✅ Verify it end-to-end

With three seeded users you can prove each path: a plain user hits /summary → 403 (not a Manager); a Manager → 200. A user with 6 months' tenure hits /vested → 403; one with 18 months → 200. No token at all → 401. That sequence exercises role, policy, and the authentication boundary together.

🔑 Name policies once, reuse everywhere

The big win of named policies over inline checks: define the rule once in AddAuthorization, apply it by name across many endpoints. When the rule changes (tenure goes from 12 to 6 months), you edit one line, not every controller. This is the same "centralize the cross-cutting concern" theme as error handling (Day 11).

08

Resource-Based Authorization 🎯

Some rules can't be decided from claims alone — they depend on the specific thing being accessed. "Can the user edit this document?" needs the document in hand to check ownership. Attributes can't do this (the resource isn't loaded yet), so you invoke authorization imperatively in the action via IAuthorizationService.

C# — checking ownership at runtime
public class DocumentsController(IAuthorizationService auth, IDocumentService docs)
    : ControllerBase
{
    [HttpPut("{id}")]
    [Authorize]
    public async Task<IActionResult> Update(int id, DocumentDto dto)
    {
        var doc = await docs.GetAsync(id);
        if (doc is null) return NotFound();

        // evaluate a policy against THIS document (the resource)
        var result = await auth.AuthorizeAsync(User, doc, "SameOwner");
        if (!result.Succeeded) return Forbid();   // 403

        await docs.UpdateAsync(id, dto);
        return NoContent();
    }
}

The handler receives the resource via context.Resource:

C# — a resource-based handler
public class SameOwnerHandler : AuthorizationHandler<SameOwnerRequirement, Document>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        SameOwnerRequirement requirement,
        Document resource)                      // the doc passed to AuthorizeAsync
    {
        var userId = context.User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
        if (userId is not null && resource.OwnerId == userId)
            context.Succeed(requirement);
        return Task.CompletedTask;
    }
}
🔑 When to reach for resource-based

Use it whenever the decision depends on the relationship between the user and a specific entity — ownership, membership, document state, tenant match. The AuthorizationHandler<TRequirement, TResource> two-type overload gives your handler the strongly-typed resource. It's the tool for "you can edit your own posts but not others'," which no attribute can express.

09

The Authorization Flow 🔄

Follow a request through the authorization middleware to see how the pieces connect — from token to allow/deny.

Step 1 · 🪪 identity established
UseAuthentication built the user

The JWT was validated (Day 12); HttpContext.User now holds the ClaimsPrincipal with all its claims.

Step 2 · 🚧 endpoint has metadata
Middleware reads [Authorize]

UseAuthorization finds the endpoint's requirements — a role, a named policy, or the default policy.

Step 3 · ⚙️ handlers evaluate
Each requirement checked

For every requirement, its handler(s) inspect the user's claims and call Succeed, Fail, or neither.

Step 4 · 🧮 combine results
ALL requirements must pass

The policy succeeds only if every requirement succeeded and none hard-failed. AND across requirements; OR across a requirement's handlers.

Step 5 · ✅/⛔ decision
Endpoint runs, or 403

Pass → the action executes. Fail → 403 Forbidden (or 401 if the user wasn't authenticated at all).

🔑 AND across requirements, OR across handlers

Two rules govern the combination, and they're the most-tested detail: a policy needs all its requirements satisfied (AND), but a single requirement is satisfied if any one of its handlers calls Succeed (OR). Design accordingly — split "must have X and Y" into two requirements; express "X or Y or Z" as one requirement with three handlers.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1What's the difference between authentication and authorization?🌱 Junior

Authentication proves who you are (verifying identity, building the user). Authorization decides what that identity is allowed to do (checking permissions). Authorization always runs after authentication — being logged in doesn't automatically mean you're allowed to perform a given action.

Q2How do you restrict an endpoint to a role?🌱 Junior

Add [Authorize(Roles = "Admin")] to the controller or action (or .RequireAuthorization(p => p.RequireRole("Admin")) on a Minimal API). A comma-separated list like "Admin,Manager" means the user needs either role. The user's token must carry the matching role claim for it to pass.

Q3What is a claim?🌱 Junior

A name–value statement about the user asserted by a trusted issuer — like role: Admin, department: Engineering, or email: a@b.com. After authentication, the user's claims live on HttpContext.User (a ClaimsPrincipal), and authorization decisions are based on them.

Q4What is a policy in ASP.NET Core authorization?🌱 Junior

A named, reusable authorization rule. You define it once with AddAuthorization(o => o.AddPolicy("Name", policy => ...)) — describing what the user must satisfy — then apply it with [Authorize(Policy = "Name")]. Policies let you name a rule once and reuse it across many endpoints.

Q5How do you require a specific claim?🌱 Junior

In a policy, use policy.RequireClaim("EmployeeId") to require the claim exist (any value), or policy.RequireClaim("department", "Engineering") to require a specific value. Then apply the policy to your endpoint with [Authorize(Policy = "...")].

Q6What status code does a failed authorization return?🌱 Junior

403 Forbidden — the user is authenticated but lacks permission. That's distinct from 401 Unauthorized, which means the user isn't authenticated at all (no or invalid token). Getting a 403 confirms your identity is valid but the rule denied you.

Q7How do you allow anonymous access to one endpoint on a protected controller?🌱 Junior

Put [Authorize] on the controller and [AllowAnonymous] on the specific action you want public. [AllowAnonymous] overrides the controller-level requirement for that action — commonly used for login or health-check endpoints.

Q8What are the three main styles of authorization?🌱 Junior

Role-based (match a role name), claims-based (require a specific claim, expressed as a policy), and policy-based (a named bundle of requirements with custom logic). There's also resource-based authorization for per-entity decisions like ownership. Under the hood they're all the same policy engine.

Q9How do role-based and claims-based authorization relate?🟠 Mid

A role is just a claim of type role. So role-based authorization is a special case of claims-based authorization with a well-known claim type. Both, in turn, are implemented as policies internally — [Authorize(Roles=…)] compiles to a RolesAuthorizationRequirement and RequireClaim to a ClaimsAuthorizationRequirement. It's one engine with convenient shorthands.

Q10When would you use a policy instead of a role?🟠 Mid

When the rule needs anything beyond matching a name: comparisons (minimum age or tenure), OR-logic across different facts, data from a database or external service, environmental conditions like time of day, or per-resource ownership. Also use a named policy whenever the same check applies to multiple endpoints, so it's defined once and reused.

Q11What are a requirement and a handler?🟠 Mid

A requirement implements IAuthorizationRequirement (a marker interface) and just carries the rule's data — e.g. a minimum tenure of 12 months. A handler extends AuthorizationHandler<TRequirement> and contains the logic: it inspects the user in HandleRequirementAsync and calls context.Succeed(requirement) if the rule is met. The requirement holds data; the handler evaluates it.

Q12In a handler, what's the difference between Succeed, Fail, and doing nothing?🟠 Mid

context.Succeed(requirement) marks it satisfied. context.Fail() hard-fails it — a veto that stands even if another handler succeeds (useful for "banned user" overrides). Returning without calling either simply doesn't satisfy the requirement, leaving another handler free to succeed. That "do nothing" option is how OR logic works across multiple handlers for one requirement.

Q13How do multiple requirements in a policy combine? And multiple handlers?🟠 Mid

Requirements combine with AND — every requirement in the policy must be satisfied for the policy to pass. A single requirement can have multiple handlers, and it's satisfied if any one handler succeeds (OR). So chaining RequireClaim(a).RequireClaim(b) needs both; to express OR, use one requirement with several handlers, or a RequireAssertion lambda.

Q14How do you register a custom authorization handler?🟠 Mid

Add it to DI: builder.Services.AddSingleton<IAuthorizationHandler, MyHandler>(). Singleton is fine when the handler is stateless. If the handler injects a scoped dependency (like a DbContext), register it as scoped instead, to avoid a captive-dependency problem. Then reference its requirement in a policy via policy.AddRequirements(new MyRequirement(...)).

Q15What's the difference between DefaultPolicy and FallbackPolicy?🟠 Mid

DefaultPolicy is used when [Authorize] is applied with no named policy — by default it requires an authenticated user. FallbackPolicy applies to endpoints that have no authorization metadata at all. Setting the fallback to RequireAuthenticatedUser() makes the whole app secure-by-default, so you must add [AllowAnonymous] to expose anything publicly.

Q16How do you express OR logic across two different claims?🟠 Mid

Chaining RequireClaim won't work — that's AND. For simple cases use RequireAssertion(ctx => ctx.User.HasClaim(...) || ctx.User.IsInRole(...)). For richer scenarios, define one requirement and register multiple handlers for it; since any single handler succeeding satisfies the requirement, that naturally models OR while keeping each handler small and testable.

Q17What is resource-based authorization and when is it necessary?🔴 Senior

When the decision depends on a specific entity — "can this user edit this document?" — attributes can't help because the resource isn't loaded yet. You inject IAuthorizationService and call AuthorizeAsync(User, resource, "PolicyName") inside the action after loading the entity, then Forbid() if it fails. The handler uses the two-type AuthorizationHandler<TRequirement, TResource> overload to receive the resource. It's the tool for ownership, tenancy, and per-record rules.

Q18Why must authorization handlers null-check the user's claims?🔴 Senior

Because all registered handlers for the pending requirements run even when the user is unauthenticated — the pipeline doesn't necessarily short-circuit before your handler executes. If you assume a claim or identity exists, you can hit a NullReferenceException. Guard by checking the claim exists (and returning without succeeding if not), and pair the policy with RequireAuthenticatedUser() when identity is a precondition.

Q19How would you enforce a rule that depends on runtime data like time or a database?🔴 Senior

Write a custom requirement and handler, and let the handler inject its dependencies from DI — a TimeProvider for "business hours only" (swappable with a fake clock in tests), a DbContext or entitlement-service client for data-driven rules. Because handlers are DI-resolved, any rule the app can compute becomes enforceable declaratively. Register the handler with the appropriate lifetime for its dependencies.

Q20How do you make an API secure-by-default?🔴 Senior

Set a FallbackPolicy that requires an authenticated user (e.g. options.FallbackPolicy = new AuthorizationPolicyBuilder().RequireAuthenticatedUser().Build()). Then every endpoint without explicit authorization metadata still requires auth, and you must consciously add [AllowAnonymous] to expose public endpoints — flipping the default from "open unless protected" to "protected unless opened," which prevents accidentally-unsecured endpoints.

Q21When would you implement a custom IAuthorizationPolicyProvider?🔴 Senior

When you can't enumerate policies up front — e.g. parameterized policies like MinimumAge18, MinimumAge21, or policies built at runtime from a database. Rather than registering each with AddPolicy, a custom provider parses the policy name (or looks it up) and produces the policy dynamically. It should defer to DefaultAuthorizationPolicyProvider as a fallback for names it doesn't handle, since ASP.NET Core uses only one provider instance.

Q22Where should authorization live — controller, service, or both?🔴 Senior

Coarse, endpoint-level checks (roles, policies) belong at the boundary via attributes, so unauthorized requests never reach your logic. But fine-grained, data-dependent rules (ownership, business invariants) often belong closer to the domain, enforced via IAuthorizationService in the action or a domain check in the service layer. Defense in depth: gate at the edge, and re-verify per-resource rules where the data is loaded. Avoid scattering ad-hoc if (User.IsInRole(...)) checks — prefer named policies so rules are centralized and testable.

Q23Roles vs. claims vs. policies — how do you choose in a real design?🔴 Senior

Use roles for a small set of coarse, stable job functions. Use claims/policies for finer permissions that change independently of job title (feature flags, entitlements, department). Prefer permission-style claims and named policies over proliferating roles, because roles combinatorially explode (AdminEU, AdminNightShift…). Reserve custom requirement/handler policies for computed and data-driven rules, and resource-based checks for per-entity ownership. The guiding principle: keep the rule expressed once, in the layer that has the data to decide it, and apply it by name.
