DTOs & Mapping in ASP.NET Core 📦

Your database entities know everything — internal costs, admin flags, private notes, tangled relationships. Handing them straight to clients is like mailing someone your entire filing cabinet when they only asked for one form. A DTO (Data Transfer Object) is that one clean form: a purpose-built shape for exactly what goes in and out over the wire. Today we separate entities from API models, shut down the over-posting vulnerability, and weigh how to move data between the two — by hand, with AutoMapper, or with a source generator.

📦 Request & response DTOs 🛡️ Avoid over-posting ✍️ Manual mapping 🤖 AutoMapper ⚡ Source generators 🧱 Layer separation
01

What a DTO Is 🤔

A Data Transfer Object is a simple class (or record) whose only job is to define how data travels between your API and its clients. It carries data and nothing else — no methods, no business logic, just properties. Microsoft's own guidance calls it "an object that defines how the data will be sent over the network," and notes it's also known as an input model or view model.

The core idea for today: your public API contract should be separate from your internal database model. A DTO is the boundary between those two worlds.

🔑 The one-line definition

An entity models what your data is (a row in a table). A DTO models what your API says (a message on the wire). They change for different reasons, so they should be different classes.

🍽️ Analogy — the restaurant menu

The kitchen (your database) has raw ingredients, supplier invoices, and staff notes. The customer gets a menu — a curated, presentable subset. No diner needs to see the wholesale cost of the salmon or the chef's private prep notes. The menu is your DTO; the kitchen is your entity. Change the kitchen all you like; as long as the menu stays the same, customers never notice.

Why bother separating them? 🎯

Microsoft lists concrete reasons to shape data with a DTO instead of exposing the entity directly. Each one is a real problem DTOs solve:

🛡️

Prevent over-posting

Stop clients from setting fields they shouldn't — the security headline of today (section 03).

🙈

Hide properties

Keep internal or sensitive fields (costs, flags, secrets) out of the response entirely.

🪶

Shrink the payload

Send only the fields the client needs — smaller, faster responses.

🫓

Flatten object graphs

Turn nested relationships (Book → Author → Name) into a single convenient field.

♻️

Remove circular references

Avoid serialization loops from two entities that point at each other.

🧱

Decouple the layers

Change the database without breaking clients — and version the API cleanly.

02

Entities vs. DTOs — Two Worlds 🌗

Let's make it concrete. Here's a domain entity — a full-fat class that maps to a database table via Entity Framework Core. It contains everything the business needs to store, including things no external client should ever touch.

C# — the entity (stays private)
// Maps to a database table. Knows EVERYTHING.
public class Book
{
    public int Id { get; set; }
    public string Title { get; set; } = null!;
    public decimal Price { get; set; }
    public decimal Cost { get; set; }            // internal purchase cost 🙈
    public bool IsPublished { get; set; }         // server decides this ⚠️
    public string InternalNotes { get; set; } = null!; // never leaves the building 🙈
    public Author Author { get; set; } = null!;    // navigation property 🔗
}

Now the two DTOs. Notice how each is a deliberate subset shaped for a single direction of travel:

C# — the DTOs (the public contract)
// What a client is ALLOWED to send us.
public class CreateBookRequest
{
    [Required, StringLength(100)]
    public string Title { get; set; } = null!;

    [Range(0, 999.99)]
    public decimal Price { get; set; }
    // No Id, no Cost, no IsPublished, no InternalNotes. Can't be set from outside.
}

// What we choose to send BACK. A record = concise + immutable.
public record BookResponse(
    int Id,
    string Title,
    decimal Price,
    string AuthorName);   // flattened from Author.Name 🫓
🔑 Why records for DTOs?

C# record types (C# 9+) give you concise syntax, value-based equality, and immutability by default — a natural fit for a data-only object that shouldn't be mutated after it's received or built. Immutability is a widely recommended DTO trait.

⚠️ The lazy-loading trap

Serializing an entity directly can be worse than leaking fields. Many ORMs lazy-load relationships on access — so a JSON serializer walking Book.Author.Books.Author… can trigger a cascade of surprise database queries or an infinite loop. A flat DTO simply has no navigation properties to follow, sidestepping the whole problem.

03

Over-Posting — The Vulnerability DTOs Kill 🛡️

Over-posting (also called mass assignment) happens when a client sends more properties than you intended, and the model binder happily copies every one of them onto your entity. If you bind straight to the entity, the client can set fields you never meant to expose — like an IsPublished flag, a discounted Price, or even someone else's Id.

The vulnerable version ❌

C# — DON'T do this
[HttpPost]
public async Task<IActionResult> Create(Book book)   // ❌ binds to the ENTITY
{
    _context.Books.Add(book);          // whatever the client sent is now saved
    await _context.SaveChangesAsync();
    return CreatedAtAction(nameof(Get), new { id = book.Id }, book);
}

A well-behaved client sends title and price. An attacker sends this, and the binder sets the extra fields too:

JSON — malicious request body
{
  "title": "My Book",
  "price": 9.99,
  "isPublished": true,        // 😈 skipped the review process
  "cost": 0,                  // 😈 tampered with internal data
  "internalNotes": "pwned"     // 😈 wrote to a private field
}

How the attack flows 🔄

Step 1 · 📤 crafted request
Client sends extra JSON fields

Beyond the intended title and price, the payload includes isPublished and internalNotes.

Step 2 · 🔗 model binding
Binder matches every field by name

Because the action takes a Book entity, the binder sets every matching property — including the ones you never intended.

Step 3 · 💥 the breach
Entity now holds attacker values

IsPublished = true and tampered internal fields are on the object — before any of your code runs.

Step 4 · 🗄️ persisted
SaveChanges writes it to the database

The unauthorized change is now permanent. The review process was bypassed entirely.

The fix — bind to a DTO ✅

Bind to a request DTO that only exposes the fields a client may set. Fields like IsPublished aren't on the DTO, so there's simply nothing for a malicious value to bind to. The server sets the protected fields itself.

C# — safe by construction
[HttpPost]
public async Task<IActionResult> Create(CreateBookRequest request)  // ✅ binds to the DTO
{
    var book = new Book
    {
        Title = request.Title,
        Price = request.Price,
        IsPublished = false       // the SERVER decides — client can't touch it
    };

    _context.Books.Add(book);
    await _context.SaveChangesAsync();
    return CreatedAtAction(nameof(Get), new { id = book.Id }, book.ToResponse());
}

Visually, the DTO acts as a filter: only whitelisted fields survive the crossing into your entity.

✅ Tip

A dedicated request DTO is the cleanest defense, but not the only one. You can also use [Bind] to whitelist properties or [BindNever] to exclude one. In practice, most teams prefer a DTO because it makes the contract explicit and self-documenting rather than relying on scattered attributes.

04

Request DTOs — Shaping Input 📥

A request DTO (input model) defines exactly what a client is allowed to send for a given operation. Because it's separate from the entity, it becomes the natural home for your input validation rules and your over-posting protection at once.

Different operations often deserve different request DTOs. Creating and updating a resource usually have different rules — a create might require every field, while an update might allow partial changes.

C#
// Create: Title is mandatory.
public class CreateBookRequest
{
    [Required, StringLength(100, MinimumLength = 2)]
    public string Title { get; set; } = null!;

    [Range(0, 999.99)]
    public decimal Price { get; set; }
}

// Update: same shape, but the Id comes from the route, not the body.
public class UpdateBookRequest
{
    [Required, StringLength(100, MinimumLength = 2)]
    public string Title { get; set; } = null!;

    [Range(0, 999.99)]
    public decimal Price { get; set; }
}
🔑 Where does the Id come from?

On an update, take the resource's Id from the URL route (PUT /api/books/42), never from the request body. That keeps identity server-controlled and removes one more thing a client could tamper with.

📝 Analogy — the application form

A request DTO is a printed form with a fixed set of blanks. Applicants can only write in the boxes provided — there's no line for "approve my own application." Whatever isn't on the form simply can't be submitted.

05

Response DTOs — Shaping Output 📤

A response DTO (output model) defines exactly what you send back. It hides internal fields, flattens relationships into plain values, and trims the payload to what the client actually needs.

C#
public record BookResponse(
    int Id,
    string Title,
    decimal Price,
    string AuthorName);   // Cost, IsPublished, InternalNotes are simply absent

Build response DTOs efficiently with projection ⚡

When reading from the database, don't load whole entities and then trim them — project straight into the DTO inside your LINQ query. Entity Framework Core translates the Select into SQL that fetches only the columns you asked for, so the database never even sends the fields you're hiding.

C# — EF Core projection
var books = await _context.Books
    .Select(b => new BookResponse(
        b.Id,
        b.Title,
        b.Price,
        b.Author.Name))          // flattened; EF joins Authors for us
    .ToListAsync();

return Ok(books);
✅ Tip — projection beats "load then map"

Projecting into a DTO inside the query is a genuine performance win: the generated SQL selects only the needed columns instead of SELECT *, and there's no wasted round-trip loading fields you'll immediately discard. This is the single most common place a DTO makes an app measurably faster.

⚠️ One shape per direction

Resist the urge to reuse a single class for both request and response. They evolve differently: a response often includes a server-generated Id and computed fields, while a request must forbid them. Sharing one class quietly reopens the over-posting door and couples two contracts that should move independently.

06

Manual Mapping — The Explicit Way ✍️

"Mapping" just means copying values from one object's shape into another's — entity → response DTO, or request DTO → entity. The simplest approach is to write that copy yourself. Microsoft's official tutorials do exactly this: they convert to DTOs manually in code.

Keep the mapping tidy with extension methods so the logic lives in one place and reads fluently:

C#
public static class BookMappings
{
    // entity  ->  response DTO
    public static BookResponse ToResponse(this Book book) =>
        new(book.Id, book.Title, book.Price, book.Author.Name);

    // request DTO  ->  entity
    public static Book ToEntity(this CreateBookRequest request) =>
        new() { Title = request.Title, Price = request.Price };
}

Now controllers read cleanly, and every field crossing the boundary is visible and intentional:

C#
var book = request.ToEntity();
_context.Books.Add(book);
await _context.SaveChangesAsync();

return CreatedAtAction(nameof(Get),
    new { id = book.Id }, book.ToResponse());
👁️

Fully explicit

Every mapped field is right there in code — easy to read, review, and debug.

🛡️

Compile-time safe

Rename a property and the compiler flags the mapping immediately.

🪶

Zero dependencies

No library, no licensing questions, no configuration to learn.

😮‍💨

Verbose at scale

Dozens of large DTOs mean a lot of repetitive copy code to maintain.

🔑 When manual mapping is the right call

For small-to-medium APIs, or anywhere the mapping is simple, hand-written mapping is often the best choice — not a compromise. It's transparent, dependency-free, and impossible to misconfigure. Reach for a tool only when the repetition genuinely starts to hurt.

07

AutoMapper — Convention-Based Mapping 🤖

AutoMapper is the long-standing .NET library that maps objects automatically by matching property names, so you write configuration instead of copy code. You define profiles that declare which types map to which, inject an IMapper, and call Map.

C# — a mapping profile
using AutoMapper;

public class BookProfile : Profile
{
    public BookProfile()
    {
        // AuthorName doesn't match Author.Name by name, so we help it.
        CreateMap<Book, BookResponse>()
            .ForCtorParam("AuthorName",
                opt => opt.MapFrom(src => src.Author.Name));

        CreateMap<CreateBookRequest, Book>();
    }
}
C# — register + use
// Program.cs — register profiles (and a license key, see below)
builder.Services.AddAutoMapper(cfg =>
{
    cfg.LicenseKey = builder.Configuration["AutoMapper:LicenseKey"];
}, typeof(BookProfile).Assembly);

// In the controller — inject IMapper, then map
public BooksController(IMapper mapper) => _mapper = mapper;

var response = _mapper.Map<BookResponse>(book);
⚠️ Important — AutoMapper is no longer free for everyone (2025)

In April 2025 the maintainer, Jimmy Bogard, announced that AutoMapper (and MediatR) would move to a commercial model; the commercial editions launched in mid-2025 under his company, Lucky Penny Software. It now uses a dual license: a free community tier plus a paid tier.

The community tier is free for individuals and for organizations under US $5 million in annual revenue; larger organizations need a paid license (priced by team size, not per seat). Newer versions log a prompt to supply a LicenseKey. The library also adopted the copyleft Reciprocal Public License (RPL 1.5) and left the .NET Foundation in late 2025. Older MIT-licensed versions remain usable and are archived.

This licensing shift is precisely why "manual vs. AutoMapper" is a live question again in 2025–2026. For many hobby and small-company projects nothing changes — the free tier covers them. For larger enterprises, teams now weigh the license cost against alternatives like manual mapping or a source generator.

The trade-off, plainly ⚖️

✨

Less boilerplate

Matching-name properties map with zero code, which shines on many large DTOs.

🕵️

Runtime reflection

Classic AutoMapper resolves mappings at runtime, so mistakes surface as runtime errors, not compile errors.

🧭

Learning curve

Profiles, custom resolvers, and config validation are their own skill to maintain.

💳

Licensing to track

Revenue thresholds and keys are now part of the decision for bigger teams.

✅ Tip — validate your AutoMapper config

If you use AutoMapper, call configuration.AssertConfigurationIsValid() at startup (or in a test). It turns silent "this property never got mapped" bugs into a loud failure the moment the app boots, recovering some of the compile-time safety you give up with a reflection-based mapper.

08

Source Generators — Mapperly & Mapster ⚡

A middle path has grown popular: mappers that generate the copy code at compile time instead of using runtime reflection. You get the low-boilerplate feel of AutoMapper and the transparency and speed of hand-written code — plus compile-time errors for unmapped properties.

Mapperly (Riok.Mapperly) 🏭

Mapperly is a .NET source generator (Apache-2.0 licensed, free, no runtime dependencies). You declare a partial class with a [Mapper] attribute and partial method signatures; Mapperly writes the method bodies during the build. The generated code is plain, readable C# you can inspect.

C# — you write this…
using Riok.Mapperly.Abstractions;

[Mapper]
public partial class BookMapper
{
    public partial BookResponse ToResponse(Book book);
    public partial Book ToEntity(CreateBookRequest request);
}
C# — …Mapperly generates the body at build time
// auto-generated — readable, debuggable, zero reflection
public partial BookResponse ToResponse(Book book)
{
    var target = new BookResponse(book.Id, book.Title, book.Price, book.Author.Name);
    return target;
}
C# — usage
var mapper = new BookMapper();
BookResponse dto = mapper.ToResponse(book);

Need a property that doesn't line up by name? Annotate the method with [MapProperty(...)]; to exclude a field, mark it [MapperIgnore]. If Mapperly can't figure out a conversion, you just provide a normal method body and it uses yours.

🔑 Mapster, the other popular option

Mapster is a comparable alternative: it offers convention-based mapping like AutoMapper and optional compile-time code generation, and benchmarks often show it running faster than reflection-based AutoMapper. Both Mapperly and Mapster are common landing spots for teams re-evaluating their mapper after the AutoMapper licensing change.

✅ Why "compile-time" matters

Because the mapping code exists before the app runs, the compiler checks it. Forget to map a property and you get a build-time warning, not a production surprise. There's also no reflection cost at runtime — the generated code is as fast as what you'd write by hand.

Best practices to carry away 📌

🚫

Never expose entities

No entity should appear in a request or response signature. The DTO is the contract.

↔️

Separate in & out

Distinct request and response DTOs — don't share one class across directions.

🧊

Prefer immutable

Use records; a DTO shouldn't mutate after it's built or received.

🧠

No logic in DTOs

Data only. Business rules belong in your domain and service layers.

🗄️

Project when reading

Map inside the EF query so the database returns only needed columns.

🧱

Mind the layers

DTOs live in the API layer; entities in the domain layer. Keep the domain unaware of the API.

⚠️ DTOs aren't free

They add classes and mapping code, and for a tiny internal API the ceremony can outweigh the benefit — Microsoft even notes DTOs take "some initial effort." The honest rule: use DTOs whenever an API is public, security-sensitive, or expected to evolve. For a throwaway prototype, exposing the model briefly may be fine — just don't ship it that way.

🧳 Analogy — carry-on vs. the whole closet

Manual mapping is packing a carry-on by hand: total control, a little tedious. AutoMapper is a valet who packs by convention: fast, but you trust them to guess right (and now there may be a tip involved). A source generator is a smart packing machine that shows you exactly what it packed before the trip. All three get your clothes to the destination; pick the one whose trade-offs fit your journey.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1What is a DTO?🌱 Junior

A Data Transfer Object is a simple class or record whose only job is to carry data between your API and its clients. It has properties but no business logic, and it defines the exact shape of what goes over the network for a given request or response.

Q2Why not just return your database entities directly?🌱 Junior

Because entities often contain things clients shouldn't see (internal costs, flags, private notes) and relationships that cause serialization problems. Exposing them also ties your public API to your database schema, so any storage change risks breaking clients. A DTO lets you send a safe, stable, curated subset instead.

Q3What's the difference between an entity and a DTO?🌱 Junior

An entity models how data is stored — it maps to a database table and holds everything the business needs. A DTO models how data is communicated — it's the API contract. They live in different layers and change for different reasons, which is exactly why they should be separate classes.

Q4What's a request DTO versus a response DTO?🌱 Junior

A request DTO (input model) defines what a client is allowed to send in — and is where you put validation rules. A response DTO (output model) defines what you send back — hiding internal fields and flattening relationships. Keeping them separate lets input and output evolve independently.

Q5What is "mapping" in this context?🌱 Junior

Mapping is copying values from one object shape to another — for example, taking a Book entity and producing a BookResponse DTO, or turning a CreateBookRequest DTO into a new entity. You can do it by hand, with a library like AutoMapper, or with a source generator.

Q6Name two benefits of using DTOs.🌱 Junior

Any two of: they prevent over-posting; they hide sensitive fields; they reduce payload size; they flatten nested objects; they decouple the API from the database (easier versioning); and they give clients predictable, documented shapes. Security and decoupling are the two most commonly cited.

Q7What does it mean to "flatten" an object graph with a DTO?🌱 Junior

An entity might have a nested relationship like Book.Author.Name. Flattening means the DTO exposes that as a single top-level field, e.g. AuthorName. This gives the client a simpler shape and avoids serializing an entire related object (and its relationships).

Q8Should a DTO contain business logic?🌱 Junior

No. A DTO is data only — properties, no behavior. Business rules belong in your domain entities and service layer. Keeping DTOs logic-free is what makes them safe to serialize and share across layers or even across languages.

Q9What is over-posting and how do DTOs prevent it?🟠 Mid

Over-posting (mass assignment) is when a client sends extra properties and the model binder copies them onto your entity — letting them set fields like IsPublished or someone else's Id. Binding to a request DTO that only exposes the allowed fields fixes it: the protected fields aren't on the DTO, so there's nothing for malicious values to bind to. The server sets those fields itself.

Q10How do you map manually between an entity and a DTO?🟠 Mid

You write the copy in code, typically as extension methods — e.g. a ToResponse() that returns new BookResponse(book.Id, book.Title, …) and a ToEntity() going the other way. It's explicit and compile-time safe: rename a property and the compiler flags the mapping.

Q11What is AutoMapper and what problem does it solve?🟠 Mid

AutoMapper is a .NET library that maps objects automatically by matching property names, so you configure mappings once (in a Profile) instead of writing repetitive copy code. You inject IMapper and call Map<TDto>(entity). It reduces boilerplate on projects with many DTOs, at the cost of resolving mappings via runtime reflection.

Q12How do you register AutoMapper in ASP.NET Core?🟠 Mid

In Program.cs, call builder.Services.AddAutoMapper(...) and point it at the assembly (or assemblies) containing your Profile classes. Recent versions also expect a LicenseKey to be set in that configuration. Then inject IMapper wherever you need to map.

Q13Why are C# records popular for DTOs?🟠 Mid

Records give concise declaration, value-based equality, and immutability by default — all desirable for a data-only object that shouldn't change after it's built or received. A one-line positional record replaces a page of boilerplate properties, which suits DTOs perfectly.

Q14How do you efficiently produce response DTOs from a database query?🟠 Mid

Project directly into the DTO inside the LINQ query with Select, rather than loading whole entities and mapping afterward. EF Core translates the projection into SQL that fetches only the needed columns — smaller reads, no wasted data, no extra round-trips. (AutoMapper offers ProjectTo<T>() for the same idea.)

Q15Should you reuse one DTO for both requests and responses?🟠 Mid

Generally no. Requests and responses have different needs: a response usually includes a server-generated Id and computed fields, while a request must forbid them. Sharing one class couples the two contracts and can quietly reopen the over-posting hole. Separate DTOs let each side evolve safely.

Q16How do DTOs help with API versioning?🟠 Mid

Because the DTO is the contract, you can introduce a v2 DTO with new or reshaped fields without touching your entities or breaking v1 clients. The internal model stays stable while the public surface evolves per version — something impossible if clients depend directly on your entities.

Q17Manual mapping vs. AutoMapper — what are the trade-offs?🔴 Senior

Manual mapping is explicit, compile-time safe, dependency-free, and easy to debug, but verbose across many large DTOs. AutoMapper cuts boilerplate via convention but resolves at runtime, so mapping mistakes become runtime errors (mitigated by AssertConfigurationIsValid), and it adds a library with a learning curve — and, since 2025, licensing to track. The right pick depends on scale, team familiarity, and how much you value compile-time guarantees.

Q18What changed with AutoMapper's licensing in 2025, and how should it affect your choice?🔴 Senior

In April 2025 the maintainer announced AutoMapper (and MediatR) going commercial, with commercial editions launching mid-2025 under Lucky Penny Software. It's now dual-licensed: a free community tier for individuals and organizations under about $5M USD annual revenue, and a paid tier (priced by team size) above that. It also moved to the copyleft RPL 1.5 and left the .NET Foundation. Practically: small teams are unaffected, but larger orgs should factor license cost into the decision — and it's a reasonable prompt to evaluate free alternatives like Mapperly, Mapster, or manual mapping. Existing MIT versions remain usable.

Q19What are source-generator mappers like Mapperly, and why prefer them?🔴 Senior

They generate the mapping code at compile time from a declared [Mapper] partial class, rather than using runtime reflection. Benefits: no runtime overhead (as fast as hand-written code), compile-time errors for unmapped properties, and readable generated code you can inspect and step through. Mapperly is free (Apache-2.0) with no dependencies. You give up some of the advanced dynamic configuration reflection-based mappers offer, but gain safety and performance.

Q20Where should DTOs and mapping live in a layered / Clean Architecture app?🔴 Senior

DTOs belong in the presentation/API (or application) layer — they're part of the public contract. Entities belong in the domain layer and should stay unaware of the API. Mapping code sits at that boundary. The key rule is dependency direction: the domain must not reference the API layer, so entities never depend on DTOs. In CQRS, commands and queries often carry DTOs to enforce this cleanly.

Q21What are the downsides or costs of introducing DTOs?🔴 Senior

More classes and mapping code to write and maintain, some duplication between DTO and entity shapes, and initial setup effort. For a tiny internal or throwaway API the ceremony can outweigh the benefit. The judgment call: adopt DTOs whenever the API is public, security-sensitive, or expected to evolve; skip them only for short-lived internal code you won't ship.

Q22How do you handle mapping when entities have navigation properties?🔴 Senior

Flatten them into plain DTO fields (Author.Name → AuthorName) and, when reading, project inside the EF query so only the needed related columns are fetched. This avoids two classic traps: serializing an entity can trigger lazy-loading cascades (surprise queries) or circular-reference loops when related objects point back at each other. A flat DTO has no navigation properties for the serializer to follow.

Q23Besides DTOs, what other techniques limit which properties a client can set?🔴 Senior

You can whitelist bound properties with [Bind(...)], exclude specific ones with [BindNever], or bind explicitly from a source like [FromBody] onto a constrained model. These work, but they're scattered and easy to get wrong. A dedicated request DTO is usually preferred because it makes the allowed contract explicit and self-documenting in one place, rather than relying on attributes spread across the code.
