Error Handling for APIs 🧯

Things go wrong: a record isn't found, a rule is violated, a database times out, a bug throws. What separates a professional API from a fragile one is what the client sees when that happens — a consistent, structured error, never a raw stack trace or an inconsistent shape that changes per endpoint. Today we centralize all of it: a single global handler that catches unhandled exceptions and returns standardized ProblemDetails, built on ASP.NET Core's IExceptionHandler.

🎯 Centralized 🧾 ProblemDetails 🪝 IExceptionHandler 🔗 UseExceptionHandler 🔒 No leaks 📝 Logged
01

Why Centralize Error Handling? 🎯

Without a central strategy, error handling scatters into a try/catch in every action. That's repetitive, easy to get wrong, and guarantees inconsistency — one endpoint returns { "error": "not found" }, another a plain string, a third leaks a full stack trace. Clients can't reliably parse any of it, and sensitive internals leak to the outside world.

Centralized handling flips this: your business code simply throws when something is wrong, and one component at the edge of the pipeline catches every unhandled exception, logs it, and shapes a single consistent response. Your controllers and services stay clean and focused on the happy path.

🚑 Analogy — the emergency room

You don't put a doctor in every hallway of a hospital. You have one ER where anyone in trouble is routed, triaged, and treated consistently. A global exception handler is that ER: no matter where in the app something collapses, it ends up in one place that responds calmly and uniformly — instead of every room improvising its own first aid.

🔑 Three jobs of a good error strategy

1. Consistency — every error looks the same shape (we use ProblemDetails). 2. Safety — clients never see stack traces or internal details in production. 3. Observability — every failure is logged with enough context to debug, and correlated by a trace id.

02

ProblemDetails — the Error Contract 🧾

We met ProblemDetails back on Day 6: it's the standard, machine-readable error format defined by RFC 9457 (formerly 7807), served as application/problem+json. Centralized handling and ProblemDetails are a pair — the handler catches the exception; ProblemDetails is the shape it returns.

JSON — the standard error shape
{
  "type":     "https://tools.ietf.org/html/rfc9110#section-15.5.5",
  "title":    "Not Found",
  "status":   404,
  "detail":   "No product with id 42 exists.",
  "instance": "GET /api/products/42",
  "traceId":  "00-8fa7...c3-01"
}

Registering it is one line — and it does more than you might expect. AddProblemDetails() registers the IProblemDetailsService, which the framework uses to turn error status codes (and unhandled exceptions, once a handler is wired) into this shape automatically.

C# — Program.cs
builder.Services.AddProblemDetails();
🔑 Two flavors, from earlier days

ProblemDetails is the base shape (Day 6). ValidationProblemDetails extends it with an errors dictionary keyed by field, and the [ApiController] attribute produces it automatically on model-validation failures (Day 4). Today's global handler covers everything else — the exceptions your code throws or that escape unhandled.

✅ Add a trace id for correlation

Include a correlation id (from Activity.Current?.Id or HttpContext.TraceIdentifier) in every error. When a client reports "I got a 500 at 3pm," that id lets you find the exact log entry instantly. AddProblemDetails() can be configured to attach it to every response automatically.

03

The Evolution of Error Handling 🪜

ASP.NET Core has offered several ways to handle errors globally. Knowing the progression explains why IExceptionHandler is the modern recommendation.

ApproachHowVerdict
Per-action try/catchWrap logic in each method❌ Repetitive, inconsistent; use only for local recovery
Custom middlewareA try/catch around next(context)🟠 Works everywhere, but you hand-write the plumbing
UseExceptionHandler(lambda)Inline handler in Program.cs🟠 Built-in, but logic crammed into startup
IExceptionHandlerA class per concern, DI-registered✅ Clean, testable, modular — the modern default

Here's the "old but everywhere" middleware version, so you recognize it — a single try/catch wrapping the rest of the pipeline:

C# — the classic middleware pattern (for context)
public class ExceptionMiddleware(RequestDelegate next, ILogger<ExceptionMiddleware> logger)
{
    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await next(context);          // run the rest of the pipeline
        }
        catch (Exception ex)
        {
            logger.LogError(ex, "Unhandled exception");
            context.Response.StatusCode = StatusCodes.Status500InternalServerError;
            await context.Response.WriteAsJsonAsync(new ProblemDetails
            {
                Status = StatusCodes.Status500InternalServerError,
                Title = "Server error"
            });
        }
    }
}
🔑 Why IExceptionHandler wins

It plugs directly into ASP.NET Core's built-in exception-handling middleware, so you inherit all its edge-case handling (client disconnects, response-already-started, clearing the context) for free — and get improvements automatically as the framework evolves. It works across the whole pipeline including Minimal APIs, relies on DI, and lets you split concerns into small, testable classes instead of one growing try/catch.

04

IExceptionHandler — the Modern Way 🪝

IExceptionHandler (introduced in .NET 8) is an interface with a single method, TryHandleAsync, giving you a central callback for handling exceptions. You implement it in a class, and ASP.NET Core's exception middleware invokes it when something is thrown.

C# — a global exception handler
using Microsoft.AspNetCore.Diagnostics;

internal sealed class GlobalExceptionHandler(
    IProblemDetailsService problemDetailsService,
    ILogger<GlobalExceptionHandler> logger) : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext,
        Exception exception,
        CancellationToken cancellationToken)
    {
        logger.LogError(exception, "Unhandled exception: {Message}", exception.Message);

        httpContext.Response.StatusCode = StatusCodes.Status500InternalServerError;

        // Let the ProblemDetails service write the standardized body.
        return await problemDetailsService.TryWriteAsync(new ProblemDetailsContext
        {
            HttpContext = httpContext,
            Exception = exception,
            ProblemDetails = new ProblemDetails
            {
                Status = StatusCodes.Status500InternalServerError,
                Title = "An unexpected error occurred."
            }
        });
    }
}
🔑 The return value is the whole contract

TryHandleAsync returns a bool: return true to say "I handled this exception — stop here," or false to pass it to the next handler (or, if none remain, the default handling). That single boolean is what makes handler chaining work (section 06).

1️⃣

One method

TryHandleAsync(context, exception, ct) — that's the entire interface.

💉

DI-friendly

Inject loggers, the IProblemDetailsService, or anything else through the constructor.

♻️

Singleton

Handlers are registered as singletons, so keep them stateless (resolve scoped services per-request if needed).

✅ Prefer TryWriteAsync over manual JSON

Using problemDetailsService.TryWriteAsync(...) (rather than hand-serializing with WriteAsJsonAsync) lets your registered ProblemDetails customizations — like the trace id and content type — apply consistently. It also respects the content negotiation the service is configured for.

05

Wiring It Up — the Two Steps 🔗

Like all DI in ASP.NET Core (Day 7), it's register + use. Two calls, plus AddProblemDetails().

C# — Program.cs (the complete wiring)
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddProblemDetails();                       // the error shape service
builder.Services.AddExceptionHandler<GlobalExceptionHandler>(); // register the handler

var app = builder.Build();

app.UseExceptionHandler();   // add the middleware — FIRST in the pipeline

app.MapControllers();
app.Run();
⚠️ The gotcha that catches everyone

Calling AddExceptionHandler<T>() alone isn't enough — you must also add the middleware with UseExceptionHandler(). And bare UseExceptionHandler() only works if you've also registered AddProblemDetails(); otherwise it throws "the ExceptionHandler property must be set." If you haven't added ProblemDetails, you'd need UseExceptionHandler(_ => {}) with an empty lambda. Register AddProblemDetails() and the plain call just works.

⚠️ Put UseExceptionHandler first

Place UseExceptionHandler() at the very start of the pipeline so it wraps everything downstream — middleware, routing, and your endpoints. An exception thrown in middleware registered before it won't be caught. First position = catches the most.

🔑 .NET 10 change — no more duplicate logs

In .NET 8 and 9, the exception middleware always logged the exception at Error level even when your handler already logged it — producing duplicate log entries. .NET 10 changes the default: when your TryHandleAsync returns true, the middleware no longer emits its own diagnostic log. You can fine-tune this with the middleware's SuppressDiagnosticsCallback if you want different behavior.

06

Chaining Multiple Handlers 🔀

You can register several handlers, each responsible for one kind of exception. They run in the order registered, and the first one whose TryHandleAsync returns true wins — so order specific handlers before your catch-all fallback.

C# — a specific handler that only claims its exception
internal sealed class NotFoundExceptionHandler(
    IProblemDetailsService problemDetailsService) : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception, CancellationToken ct)
    {
        // Not my exception? Decline — let the next handler try.
        if (exception is not NotFoundException notFound)
            return false;

        httpContext.Response.StatusCode = StatusCodes.Status404NotFound;
        return await problemDetailsService.TryWriteAsync(new ProblemDetailsContext
        {
            HttpContext = httpContext,
            Exception = notFound,
            ProblemDetails = new ProblemDetails
            {
                Status = StatusCodes.Status404NotFound,
                Title = "Resource not found",
                Detail = notFound.Message
            }
        });
    }
}
C# — register specific first, fallback last
builder.Services.AddExceptionHandler<NotFoundExceptionHandler>();   // tried 1st
builder.Services.AddExceptionHandler<ValidationExceptionHandler>(); // tried 2nd
builder.Services.AddExceptionHandler<GlobalExceptionHandler>();     // catch-all, last
NotFound?
→ true → 404
false ↓
Validation?
→ true → 400
false ↓
Global
→ true → 500

Each handler either claims the exception (true, stop) or declines (false, pass on). The fallback returns true for anything, so it must be registered last.

✅ One job per handler

Chaining keeps each handler tiny and single-purpose — one for "not found," one for validation, one catch-all — which is far more testable than a giant switch on exception type. That said, a switch expression inside one handler is a perfectly good simpler alternative for small apps.

07

Custom Exceptions → Status Codes 🔀

The cleanest pattern: define domain exceptions that mean something, throw them from your service layer (Day 7), and let handlers translate each into the right status code. Your business logic reads naturally and never touches HTTP.

C# — meaningful domain exceptions
public class NotFoundException(string message) : Exception(message);
public class DuplicateNameException(string name)
    : Exception($"A product named '{name}' already exists.");
C# — the service just throws; no HTTP concerns
public async Task<ProductDto> GetByIdAsync(int id)
{
    var product = await _repo.GetByIdAsync(id)
        ?? throw new NotFoundException($"No product with id {id} exists.");
    return product.ToDto();
}

A single handler can map many exception types to codes with a switch — a compact alternative to one handler per type:

C# — mapping exceptions to status codes in one handler
(int status, string title) = exception switch
{
    NotFoundException      => (StatusCodes.Status404NotFound,   "Not found"),
    DuplicateNameException => (StatusCodes.Status409Conflict,   "Conflict"),
    ValidationException    => (StatusCodes.Status400BadRequest, "Validation failed"),
    _                       => (StatusCodes.Status500InternalServerError, "Server error")
};

httpContext.Response.StatusCode = status;
return await problemDetailsService.TryWriteAsync(new()
{
    HttpContext = httpContext,
    Exception = exception,
    ProblemDetails = new() { Status = status, Title = title, Detail = exception.Message }
});
🔑 Ties the whole week together

This is the payoff of the layered design: the service (Day 7) expresses what went wrong by throwing a domain exception; the handler decides the HTTP status (Day 6); and ProblemDetails gives it a consistent body. Controllers no longer need try/catch or manual NotFound() calls for these cases — throwing is enough.

⚠️ Don't use exceptions for ordinary control flow

Exceptions are for exceptional conditions. Throwing on a genuinely-missing resource is reasonable, but for high-frequency expected outcomes, returning a result (or a 404 directly from the controller, Day 6) can be cheaper than throwing, since exceptions carry overhead. Use judgment: domain exceptions for real rule violations, plain returns for routine "not found" on hot paths.

08

Safety & Logging 🔒

Two responsibilities the handler must never neglect: don't leak internals to clients, and do record enough to debug.

⚠️ Never serve sensitive error detail to clients

Microsoft is explicit: serving raw exception detail is a security risk. A stack trace or database error message can reveal your schema, file paths, library versions, and logic to an attacker. In production, return a generic Detail ("An unexpected error occurred"); keep the specifics in your logs. Only in development should you surface exception details for debugging.

C# — dev vs. prod detail
public GlobalExceptionHandler(
    IProblemDetailsService pds, IHostEnvironment env, ILogger<GlobalExceptionHandler> log)
    { _pds = pds; _env = env; _log = log; }

// ... inside TryHandleAsync:
var detail = _env.IsDevelopment()
    ? exception.ToString()          // full detail — DEV ONLY
    : "An unexpected error occurred."; // safe generic message in prod
📝

Log the real thing

Always log the full exception server-side with logger.LogError(ex, ...) — that's where the detail belongs, not in the response.

🔖

Correlate

Attach a trace id to both the log and the response so a reported error maps to one log entry.

🎚️

Right level

Log unexpected 500s at Error; expected 404/400 domain cases can be Information/Warning to avoid noise.

✅ Customize ProblemDetails globally

AddProblemDetails(options => options.CustomizeProblemDetails = ctx => { ... }) lets you enrich every error once — e.g. always add the trace id and the request path — so individual handlers stay lean and every response is uniform.

09

Request-to-Error Flow 🔄

Here's the full journey of a request that throws, from endpoint to structured response — today's "Do" in motion.

Step 1 · 📥 request enters
Passes UseExceptionHandler

The exception middleware sits first, wrapping everything downstream in its safety net, then hands off to routing.

Step 2 · 🧠 service throws
A domain exception is raised

The service can't find the product and throws NotFoundException — no HTTP code, no try/catch, just intent.

Step 3 · 🪝 middleware catches
Handlers run in order

The exception unwinds to the middleware, which calls each IExceptionHandler until one returns true.

Step 4 · 🧾 shape + log
ProblemDetails written, error logged

The matching handler sets 404, writes a ProblemDetails body via TryWriteAsync, and logs with a trace id.

Step 5 · 📤 client responds
Consistent structured error

The client receives application/problem+json with a clear status, title, and safe detail — same shape every time.

🔑 The clean-code payoff

Notice what's absent: no try/catch in the controller, no manual error JSON, no leaked stack trace. The business code throws; the edge of the app handles. That separation is the entire point of centralized error handling.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1Why centralize error handling instead of using try/catch everywhere?🌱 Junior

A try/catch in every action is repetitive and leads to inconsistent error responses — each endpoint might return a different shape. Centralizing means one component catches all unhandled exceptions and returns a single consistent format, keeps controllers focused on the happy path, and ensures internals aren't leaked. Business code just throws; the edge handles it.

Q2What is ProblemDetails?🌱 Junior

A standardized, machine-readable error format (RFC 9457, formerly 7807) served as application/problem+json. It has fields like type, title, status, and detail. Using it means every client gets the same predictable error shape to parse. You enable it with AddProblemDetails().

Q3What is IExceptionHandler?🌱 Junior

An interface (introduced in .NET 8) for handling exceptions centrally. It has one method, TryHandleAsync(HttpContext, Exception, CancellationToken), where you write your handling logic. You implement it in a class, register it with AddExceptionHandler<T>(), and ASP.NET Core's exception middleware calls it when something is thrown.

Q4What two steps wire up an exception handler?🌱 Junior

First, register the handler in DI: builder.Services.AddExceptionHandler<GlobalExceptionHandler>(). Second, add the middleware to the pipeline: app.UseExceptionHandler(). You also register AddProblemDetails() so the standardized error body is produced and the bare UseExceptionHandler() call works.

Q5What does TryHandleAsync return, and why?🌱 Junior

A bool. Return true to say "I handled this exception; stop here," or false to let the next handler (or the default behavior) deal with it. That boolean is what lets you chain multiple handlers, each responsible for a different exception type.

Q6Why shouldn't you return a stack trace to the client?🌱 Junior

It's a security risk. A stack trace or raw exception message can reveal your database schema, file paths, library versions, and internal logic to an attacker. In production you return a generic message and keep the real detail in your server logs; full detail is only appropriate in a development environment.

Q7Where should UseExceptionHandler go in the pipeline?🌱 Junior

At the very start, before other middleware. It works by wrapping everything downstream, so placing it first means it catches exceptions from all subsequent middleware, routing, and endpoints. Anything registered before it wouldn't be caught by it.

Q8What's the benefit of a consistent error format?🌱 Junior

Clients can write one piece of code to parse and handle errors instead of special-casing every endpoint. Consistency also makes generated SDKs, logging, and monitoring reliable. ProblemDetails gives you that one standard shape across success-adjacent failures and unexpected crashes alike.

Q9How do you chain multiple exception handlers?🟠 Mid

Register several with AddExceptionHandler<T>(). They run in registration order, and the first whose TryHandleAsync returns true handles the exception. Each specific handler checks the exception type and returns false if it's not theirs. Register specific handlers first and a catch-all fallback last, since the fallback always returns true.

Q10Why does bare UseExceptionHandler() sometimes throw at startup?🟠 Mid

Because the middleware needs to know how to produce a response. With no arguments it requires either an ExceptionHandlingPath/handler configured, or a registered ProblemDetails service. If you've called AddProblemDetails(), the plain UseExceptionHandler() works; otherwise you must pass an empty lambda UseExceptionHandler(_ => {}) or configure options. Registering ProblemDetails is the clean fix.

Q11How would you map custom exceptions to specific status codes?🟠 Mid

Define domain exceptions (e.g. NotFoundException, DuplicateNameException), throw them from the service layer, and in the handler map each type to a code — either one handler per type, or a single handler with a switch on the exception type setting the status and title. The service stays HTTP-free; the handler owns the translation.

Q12IExceptionHandler vs. exception-handling middleware — which and why?🟠 Mid

IExceptionHandler is the modern recommendation. Both work across the pipeline, but the handler plugs into the framework's built-in middleware, so you inherit its edge-case handling (client disconnects, response already started, context clearing) and future improvements for free, keep concerns in small testable DI-registered classes, and avoid hand-writing the try/catch plumbing. Custom middleware is still valid but reinvents what the framework gives you.

Q13What lifetime is an IExceptionHandler, and what does that imply?🟠 Mid

Singleton — one instance for the app's lifetime. So keep handlers stateless and don't inject scoped services (like a DbContext) directly into the constructor, or you'll create a captive dependency (Day 7). If you need a scoped service, resolve it per-request from the HttpContext.RequestServices inside TryHandleAsync.

Q14Why prefer IProblemDetailsService.TryWriteAsync over WriteAsJsonAsync?🟠 Mid

TryWriteAsync routes the response through the registered ProblemDetails service, so your global customizations (trace id, content type, extension members) apply uniformly, and it respects the configured negotiation. Hand-serializing with WriteAsJsonAsync bypasses all that, risking inconsistent bodies across handlers.

Q15How do you handle validation errors versus thrown exceptions?🟠 Mid

They're separate paths. Model-validation failures are handled automatically by [ApiController], which returns ValidationProblemDetails (a 400 with a per-field errors dictionary) before your action runs (Day 4). The global exception handler covers everything else — exceptions your code throws or that escape unhandled. Both produce a ProblemDetails-family response, so the client sees a consistent shape either way.

Q16How do you correlate an error report with your logs?🟠 Mid

Include a trace/correlation id in both the logged entry and the ProblemDetails response — typically from Activity.Current?.Id or HttpContext.TraceIdentifier. When a user reports "I got a 500 and it said traceId X," you search logs for X and find the exact request. You can attach it globally via AddProblemDetails' customize callback.

Q17What changed about exception logging in .NET 10?🔴 Senior

In .NET 8/9 the exception-handling middleware always emitted an Error-level diagnostic log for every exception, even when your IExceptionHandler already logged it — causing duplicate log entries. In .NET 10 the default changed: when TryHandleAsync returns true, the middleware no longer emits its own diagnostic by default. You can control this precisely with the middleware's SuppressDiagnosticsCallback.

Q18Should you use exceptions for expected control flow like "not found"?🔴 Senior

Cautiously. Exceptions carry real overhead (stack capture), so for high-frequency, fully-expected outcomes on hot paths, returning a result or a direct 404 from the controller can be cheaper and clearer. Domain exceptions are appropriate for genuine rule violations and keeping deep call stacks clean. It's a judgment call — many teams use a result pattern for expected cases and reserve exceptions for the truly exceptional.

Q19How do you globally enrich every ProblemDetails response?🔴 Senior

Configure AddProblemDetails(options => options.CustomizeProblemDetails = ctx => { ... }). In the callback you can add extension members to every response — a trace id, the request path/method, a support code, an environment tag — so all errors are uniform without repeating that code in each handler. It runs for both exception responses and generated status-code responses.

Q20How would you test exception handlers?🔴 Senior

Unit test the handler in isolation: construct it with a fake logger and ProblemDetails service, build a DefaultHttpContext, call TryHandleAsync with a sample exception, and assert the return value and the status code set on the response. For end-to-end confidence, add integration tests with WebApplicationFactory (Day 8) that hit an endpoint which throws and assert the response is the expected status and ProblemDetails shape.

Q21How does centralized handling interact with the service/layered architecture?🔴 Senior

It completes the separation of concerns: the service layer (Day 7) throws domain exceptions expressing what went wrong with no HTTP knowledge; the handler at the edge translates those into HTTP status codes (Day 6) and a ProblemDetails body. Controllers shrink to the happy path — no try/catch, no manual error results. The layers stay decoupled and each has one responsibility for errors.

Q22How do you keep a scoped dependency out of a singleton handler safely?🔴 Senior

Don't inject the scoped service into the singleton handler's constructor — that's a captive dependency and scope validation will flag it. Instead resolve it per request from httpContext.RequestServices.GetRequiredService<T>() inside TryHandleAsync, which gives you the correctly-scoped instance for that request. Better still, keep handlers free of scoped dependencies where possible; they should mostly log and shape responses.

Q23What's the difference between the developer exception page and a global handler?🔴 Senior

The Developer Exception Page (UseDeveloperExceptionPage, on by default in development) shows rich diagnostic detail — stack trace, headers, query — to help you debug, and must never run in production. The global exception handler (UseExceptionHandler + IExceptionHandler) is the production strategy that returns safe, structured ProblemDetails to clients. Typical setup: dev page in development, exception handler in production — gated by the environment.
