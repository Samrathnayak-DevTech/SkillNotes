// day 12 — prove who you are, on every request

Authentication 🔐

APIs are stateless — there's no session on the server remembering who you are, so every single request must prove its identity. The dominant way to do that in ASP.NET Core is a JWT: the user logs in once, receives a signed token, and sends it on every request thereafter. Today we validate those tokens with JWT bearer authentication, issue one from a login endpoint, protect routes with [Authorize], and meet ASP.NET Core Identity for managing users.

🎟️ JWT ✍️ Issue tokens ✅ Validate tokens 🚧 [Authorize] 🆔 Identity 🔑 Claims
01

Authentication vs. Authorization 🪪

Two words that sound alike and get muddled constantly. Authentication (AuthN) answers "who are you?" — proving identity. Authorization (AuthZ) answers "what are you allowed to do?" — granting or denying access based on that identity. You authenticate first, then authorize. Today is mostly about authentication, with a taste of authorization via [Authorize].

🎫 Analogy — the concert

Authentication is showing your ID at the door to prove you're really you. Authorization is the wristband that says whether you can enter the general floor or the VIP area. The bouncer checks who you are once; the wristband (your claims) decides where you can go all night.

🔑 Why APIs need this on every request

A traditional web app can keep a server-side session. A REST API is stateless — it deliberately keeps no memory between requests. So each request must independently carry proof of identity. A JWT is that portable proof: self-contained, signed, and verifiable without a database lookup.

02

What Is a JWT? 🎟️

A JWT (JSON Web Token, pronounced "jot") is a compact, signed string carrying information about the user. It has three parts, separated by dots: header.payload.signature.

Header
alg + type
.
Payload
claims
.
Signature
verifies integrity

The payload holds claims — statements about the user, like their id, name, and roles, plus standard fields such as iss (issuer), aud (audience), and exp (expiry). The signature is computed from the header, payload, and a secret key, so the server can verify the token wasn't tampered with.

⚠️ Signed ≠ encrypted

A standard JWT is signed, not encrypted. The payload is only Base64-encoded — anyone can decode and read it (paste one into jwt.io and see). The signature proves it wasn't altered, but it doesn't hide the contents. So never put secrets (passwords, sensitive data) in a JWT payload, and always transmit tokens over HTTPS.

🔑 Why JWTs scale

The server never stores the token — it only needs the secret key to verify the signature. That makes JWT stateless: any server in a farm can validate a request without a shared session store or database lookup. Log in on server A, make a request to server B — both trust the same signed token.

✅ The client sends it in a header

On every protected request, the client puts the token in the Authorization header with the Bearer scheme: Authorization: Bearer eyJhbGci.... "Bearer" means "whoever bears (holds) this token gets access" — which is exactly why keeping it secret matters.

03

Configuring JWT Bearer Validation ✅

The Microsoft.AspNetCore.Authentication.JwtBearer package lets your API validate incoming tokens. You register the scheme and tell it exactly how to check each token via TokenValidationParameters.

C# — Program.cs (validating tokens)
// dotnet add package Microsoft.AspNetCore.Authentication.JwtBearer
var key = new SymmetricSecurityKey(
    Encoding.UTF8.GetBytes(builder.Configuration["Jwt:Key"]!));

builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer           = true,
            ValidIssuer              = builder.Configuration["Jwt:Issuer"],
            ValidateAudience         = true,
            ValidAudience            = builder.Configuration["Jwt:Audience"],
            ValidateLifetime         = true,   // reject expired tokens
            ValidateIssuerSigningKey = true,   // verify the signature
            IssuerSigningKey         = key
        };
    });

builder.Services.AddAuthorization();
ParameterChecks…Why it matters
ValidateIssuerSigningKeyThe signatureProves the token was signed by you and not forged — always true
ValidateIssuerThe iss claimConfirms the token came from the expected issuer
ValidateAudienceThe aud claimConfirms the token was meant for this API
ValidateLifetimeexp / nbfRejects expired or not-yet-valid tokens
⚠️ The #1 cause of surprise 401s

If a valid-looking token is rejected, the culprit is almost always a mismatch: the signing key differs between issuer and validator, or the token's iss/aud don't match your ValidIssuer/ValidAudience. There's also a default 5-minute clock skew tolerance for expiry — if your servers' clocks drift more than that, lifetime validation fails. Enable Microsoft.AspNetCore.Authentication logging at Information to see the exact reason.

✅ In real systems, prefer an identity provider

Signing your own tokens with a symmetric key is fine for learning and closed systems. For production, Microsoft recommends using an OpenID Connect / OAuth identity provider (Entra ID, Auth0, Keycloak, Duende IdentityServer). Then you set options.Authority to the provider's URL and it fetches the signing keys automatically — no shared secret to manage.

04

Middleware Order — the Silent Trap 🔀

Registering the services isn't enough; you must add two middleware components to the pipeline, in the right order. This is one of the most common "why is my auth not working?" mistakes.

C# — Program.cs (pipeline)
var app = builder.Build();

app.UseAuthentication();   // 1) WHO are you? (reads & validates the token)
app.UseAuthorization();    // 2) are you ALLOWED? (checks [Authorize])

app.MapControllers();
app.Run();
Request
Bearer token
→
UseAuthentication
build ClaimsPrincipal
→
UseAuthorization
enforce [Authorize]
→
Endpoint
runs if allowed
⚠️ Authentication must come before Authorization

UseAuthentication() reads the token and builds the user's ClaimsPrincipal; UseAuthorization() then checks whether that user satisfies each endpoint's rules. Reverse them and authorization runs against an empty user, so every protected endpoint returns 401 even with a valid token. Both must also sit after UseRouting (which the minimal hosting model wires up for you) and before your endpoints.

🔑 What authentication actually produces

When a valid token arrives, the JWT bearer handler validates it and builds a ClaimsPrincipal from its claims, setting it as HttpContext.User. From then on, your code can read User.Identity.Name, check User.IsInRole("Admin"), or pull a claim with User.FindFirst(...) — all populated straight from the token.

05

Protecting Endpoints 🚧

With validation wired up, protecting a route is one attribute. [Authorize] requires a valid authenticated user; without a good token, the request gets 401 Unauthorized.

C# — controller protection
[ApiController]
[Route("api/[controller]")]
[Authorize]                        // whole controller requires auth
public class ProductsController : ControllerBase
{
    [HttpGet]
    [AllowAnonymous]              // ...but this one is public
    public IActionResult List() => Ok();

    [HttpDelete("{id}")]
    [Authorize(Roles = "Admin")]   // only users with the Admin role
    public IActionResult Delete(int id) => NoContent();
}

Minimal APIs use the fluent equivalent, RequireAuthorization():

C# — Minimal API protection
app.MapGet("/api/profile", (ClaimsPrincipal user) =>
    Results.Ok(new { name = user.Identity!.Name }))
   .RequireAuthorization();                       // any authenticated user

app.MapDelete("/api/products/{id}", (int id) => Results.NoContent())
   .RequireAuthorization(p => p.RequireRole("Admin"));   // role-gated
🔑 401 vs. 403 — know the difference

401 Unauthorized means "I don't know who you are" — no token or an invalid one (an authentication failure). 403 Forbidden means "I know who you are, but you're not allowed" — a valid token that lacks the required role or policy (an authorization failure). Getting a 403 is actually a sign your token is valid.

✅ Roles vs. policies

[Authorize(Roles = "Admin")] is the simple case. For anything richer — "must be over 18," "must own this resource," "must have a specific claim" — define a policy with AddAuthorization(o => o.AddPolicy(...)) and apply it with [Authorize(Policy = "...")]. Policies scale far better than sprinkling role names everywhere.

06

Issuing a Token — the Login Endpoint ✍️

Now the other half: issuing a token when someone logs in. The login endpoint verifies credentials, builds a set of claims, signs a token, and returns it. Note this endpoint is [AllowAnonymous] — you can't require a token to get a token.

C# — building & signing a JWT (JsonWebTokenHandler)
[HttpPost("login")]
[AllowAnonymous]
public IActionResult Login(LoginRequest request)
{
    // 1) verify credentials (here simplified; use Identity in §07)
    if (!CredentialsAreValid(request))
        return Unauthorized();

    // 2) build the claims that describe the user
    var claims = new[]
    {
        new Claim(JwtRegisteredClaimNames.Sub, request.Username),
        new Claim(ClaimTypes.Role, "Admin")
    };

    // 3) describe & sign the token
    var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(_config["Jwt:Key"]!));
    var descriptor = new SecurityTokenDescriptor
    {
        Subject            = new ClaimsIdentity(claims),
        Issuer             = _config["Jwt:Issuer"],
        Audience           = _config["Jwt:Audience"],
        Expires            = DateTime.UtcNow.AddHours(1),
        SigningCredentials = new SigningCredentials(key, SecurityAlgorithms.HmacSha256)
    };

    // 4) JsonWebTokenHandler is the modern token creator
    var token = new JsonWebTokenHandler().CreateToken(descriptor);
    return Ok(new { accessToken = token });
}
🔑 JsonWebTokenHandler is the modern API

JsonWebTokenHandler is the current, recommended way to create and read tokens — it replaces the older JwtSecurityTokenHandler and is faster. The same signing key used here to create the token must match the IssuerSigningKey used in §03 to validate it — that shared secret is the whole basis of trust.

✅ Test it without writing a login

For local development you don't even need a login endpoint to try protected routes. The dotnet user-jwts CLI tool mints valid development tokens for your API — handy for testing [Authorize] in Swagger UI, Scalar, or a .http file (Day 8) before the real auth flow exists.

07

ASP.NET Core Identity 🆔

So far we hand-waved "verify credentials." You could hash passwords and manage users yourself — but that's exactly the wheel you shouldn't reinvent. ASP.NET Core Identity is the built-in, battle-tested membership system: user registration, secure password hashing, lockout, roles, claims, and more.

C# — registering Identity with EF Core
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseSqlServer(connectionString));

builder.Services
    .AddIdentityCore<IdentityUser>()
    .AddRoles<IdentityRole>()
    .AddEntityFrameworkStores<AppDbContext>();

Its two workhorse services get injected into your endpoints: UserManager creates/finds users and checks passwords; SignInManager handles sign-in. A real login uses them, then issues your JWT:

C# — register & login with Identity, then issue a JWT
// REGISTER — UserManager hashes the password & stores the user
[HttpPost("register"), AllowAnonymous]
public async Task<IActionResult> Register(RegisterRequest req)
{
    var user = new IdentityUser { UserName = req.Email, Email = req.Email };
    var result = await _userManager.CreateAsync(user, req.Password);
    return result.Succeeded ? Ok() : BadRequest(result.Errors);
}

// LOGIN — verify the password, then mint a JWT (from §06)
[HttpPost("login"), AllowAnonymous]
public async Task<IActionResult> Login(LoginRequest req)
{
    var user = await _userManager.FindByEmailAsync(req.Email);
    if (user is null ||
        !await _userManager.CheckPasswordAsync(user, req.Password))
        return Unauthorized();

    var roles = await _userManager.GetRolesAsync(user);
    return Ok(new { accessToken = CreateJwt(user, roles) });
}
⚠️ Big gotcha — Identity's built-in tokens are NOT JWTs

.NET 8+ added a built-in Identity API (MapIdentityApi) with token-based endpoints. But the tokens it issues are a proprietary format, not standard JWTs — the built-in API is meant for simple first-party scenarios. If you specifically need JWTs (e.g. for a mobile app or to share identity across services), use Identity for user management (as above) and issue the JWT yourself, or use a full identity provider. Don't assume "Identity token" means "JWT."

🔑 Cookies vs. tokens — Microsoft's actual guidance

For a browser-based SPA on your own domain, Microsoft recommends cookie authentication (the browser manages it safely, out of reach of JavaScript). Reach for JWT bearer tokens when the client can't use cookies well — native mobile/desktop apps, or when a third party issues the identity. JWT isn't automatically "more modern"; it's the right tool for stateless, cross-service, non-browser scenarios.

08

Login-to-Access Flow 🧩

Here's the full round trip — today's "Do": log in to receive a JWT, then use it to reach a protected endpoint.

Step 1 · 🔑 client logs in
POST /login with credentials

An anonymous endpoint. The server checks the username/password (via UserManager.CheckPasswordAsync).

Step 2 · ✍️ server issues a token
Build claims, sign, return JWT

The server assembles the user's claims and roles, signs a token with the secret key, and returns { accessToken }.

Step 3 · 💾 client stores it
Keeps the token

The client holds the token (in memory, or secure storage) and attaches it to future requests.

Step 4 · 📨 request with Bearer
Authorization: Bearer <token>

Calling a protected route, the client sends the token in the header. UseAuthentication validates it and builds the user.

Step 5 · ✅ access granted
[Authorize] passes → 200

Authorization confirms the user meets the rule; the endpoint runs. No token → 401; wrong role → 403.

✅ Verify it end-to-end

Call a protected route with no token → expect 401. POST /login with good credentials → get a token back. Re-call the route with Authorization: Bearer <token> → expect 200. Try an Admin-only route as a non-admin → expect 403. That sequence proves issuing, validating, and authorizing all work.

09

Security Practices That Matter 🔒

Auth is where mistakes are most costly. A short list of non-negotiables:

🔐

HTTPS always

Bearer tokens are like cash — anyone holding one is trusted. Only ever transmit them over HTTPS so they can't be sniffed.

🗝️

Guard the signing key

Keep it out of source control — use user-secrets in dev, a secrets vault in prod. Anyone with the key can forge tokens.

⏱️

Short-lived tokens

Give access tokens a short expiry (minutes to an hour). Use refresh tokens to get new ones without re-login.

🙈

No secrets in the payload

The payload is readable by anyone. Put only non-sensitive claims (id, roles) in it — never passwords or private data.

🔁

Don't roll your own crypto

Use Identity for password hashing and a standard provider for tokens. Home-grown auth is where breaches happen.

📉

Least privilege

Give tokens only the claims/roles they need. Broad "god" tokens turn one leak into a total compromise.

⚠️ Revocation is JWT's weak spot

Because JWTs are stateless and self-validating, you can't easily "log someone out" server-side before the token expires — there's no session to kill. That's why short lifetimes plus refresh tokens matter: the short expiry is your revocation window. For hard revocation you need extra machinery (a denylist, or reference tokens checked against a store), which trades away some of JWT's stateless benefit.

🔑 The whole series, secured

Authentication layers cleanly onto everything you've built: protected controllers still return the right status codes (Day 6), throw domain exceptions caught by your global handler (Day 11), and document their 401/403 responses in OpenAPI (Day 10). Security isn't a separate app — it's a cross-cutting concern threaded through the API you already have.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1What's the difference between authentication and authorization?🌱 Junior

Authentication (AuthN) is proving who you are — verifying identity, e.g. by validating a token or password. Authorization (AuthZ) is deciding what you're allowed to do — checking whether that identity has permission for an action, e.g. via roles or policies. You authenticate first, then authorize.

Q2What is a JWT and what are its three parts?🌱 Junior

A JSON Web Token — a compact, signed string proving a user's identity. It has three dot-separated parts: the header (algorithm and type), the payload (claims about the user, like id and roles), and the signature (computed from header, payload, and a secret key to prove the token wasn't tampered with).

Q3Where does the client put the token on a request?🌱 Junior

In the Authorization HTTP header using the Bearer scheme: Authorization: Bearer <token>. The API's JWT bearer handler reads that header, validates the token, and builds the user from its claims. "Bearer" means whoever holds the token is granted access.

Q4How do you protect an endpoint?🌱 Junior

Add the [Authorize] attribute to a controller or action (or .RequireAuthorization() on a Minimal API endpoint). It requires a valid authenticated user; requests without a good token get a 401. Use [AllowAnonymous] to exempt specific endpoints like login.

Q5Why must APIs authenticate every request?🌱 Junior

REST APIs are stateless — they keep no server-side session remembering who you are between requests. So each request must independently carry proof of identity. A JWT provides that: the client logs in once, gets a token, and sends it on every subsequent request.

Q6What's the correct middleware order for auth?🌱 Junior

app.UseAuthentication() before app.UseAuthorization(). Authentication reads and validates the token and builds the user; authorization then checks whether that user is allowed. If you reverse them, authorization runs against an empty user and everything returns 401.

Q7What is ASP.NET Core Identity?🌱 Junior

A built-in membership system that handles user registration, secure password hashing, sign-in, roles, claims, and lockout. Its main services are UserManager (create/find users, check passwords) and SignInManager (sign users in/out). It saves you from writing — and getting wrong — your own auth plumbing.

Q8What's the difference between a 401 and a 403?🌱 Junior

401 Unauthorized means the request isn't authenticated — no token or an invalid one. 403 Forbidden means the request is authenticated but the user lacks permission for that action (e.g. wrong role). A 403 actually tells you the token is valid.

Q9What does TokenValidationParameters check?🟠 Mid

It controls how each incoming token is validated: ValidateIssuerSigningKey verifies the signature (proving the token is genuine — always on), ValidateIssuer/ValidateAudience confirm the iss/aud claims match this API, and ValidateLifetime rejects expired or not-yet-valid tokens. Mismatches here are the usual cause of unexpected 401s.

Q10Is a JWT encrypted? What does that mean for its contents?🟠 Mid

No — a standard JWT is signed, not encrypted. The payload is only Base64-encoded, so anyone can decode and read it. The signature guarantees it wasn't altered, not that it's secret. So never put sensitive data in the payload, and always send tokens over HTTPS to prevent interception.

Q11How do you issue a JWT on login?🟠 Mid

In an anonymous login endpoint: verify credentials (e.g. UserManager.CheckPasswordAsync), build a list of claims (subject, roles), then create a SecurityTokenDescriptor with the issuer, audience, expiry, and signing credentials, and call JsonWebTokenHandler().CreateToken(descriptor). Return the token to the client. The signing key must match the one the API validates with.

Q12Why is JWT considered stateless, and why does that help scaling?🟠 Mid

The server doesn't store the token or a session; it only needs the secret key to verify the signature and read the claims. So any server in a farm can validate any request independently — no shared session store, no sticky sessions, no database lookup per request. That makes horizontal scaling and load balancing simple.

Q13What's the difference between role-based and policy-based authorization?🟠 Mid

Role-based is the simple case: [Authorize(Roles = "Admin")] checks the user has a named role. Policy-based is richer — you define a named policy with requirements (a specific claim, a minimum age, resource ownership) via AddAuthorization, then apply [Authorize(Policy = "...")]. Policies centralize complex rules and scale better than scattering role strings across the codebase.

Q14Are the tokens from ASP.NET Core Identity's built-in API JWTs?🟠 Mid

No — a common misconception. The built-in Identity API (MapIdentityApi, added in .NET 8) issues tokens in a proprietary format, not standard JWTs, and it's aimed at simple first-party scenarios. If you specifically need JWTs, use Identity for user management and issue the JWT yourself, or use a full OpenID Connect/OAuth provider.

Q15When should you use cookies instead of JWT bearer tokens?🟠 Mid

For browser-based apps (SPAs) served on your own domain, Microsoft recommends cookies — the browser manages them automatically and keeps them out of JavaScript's reach, which is safer. Use JWT bearer tokens when the client can't handle cookies well (native mobile/desktop apps) or when a third party issues the identity across services. JWT isn't inherently "better" — it fits stateless, cross-service, non-browser cases.

Q16A valid-looking token gets a 401. How do you debug it?🟠 Mid

Check the usual mismatches: the signing key differs between issuer and validator (most common), the token's iss/aud don't match ValidIssuer/ValidAudience, or the token is expired (mind the default 5-minute clock-skew tolerance and any clock drift). Decode the token at jwt.io to inspect its claims, and enable Microsoft.AspNetCore.Authentication logging at Information level to see the exact validation failure.

Q17What are refresh tokens and why are they needed?🔴 Senior

Access tokens should be short-lived to limit damage if leaked, but forcing a re-login every few minutes is bad UX. A refresh token is a longer-lived credential the client exchanges for a new access token when the old one expires, without re-entering credentials. Refresh tokens are typically stored server-side (or as reference tokens) so they can be revoked — recovering some of the control that stateless access tokens give up.

Q18How do you revoke a JWT before it expires?🔴 Senior

You can't easily — that's JWT's core trade-off. Because validation is stateless and signature-based, there's no session to kill. Practical mitigations: keep access tokens short-lived so the expiry window is the revocation window; maintain a denylist of revoked token ids (jti) checked on each request; rotate/revoke refresh tokens; or switch to reference tokens validated against a store. Each of these reintroduces some state, trading away pure statelessness for control.

Q19Symmetric vs. asymmetric signing — when would you use each?🔴 Senior

Symmetric (e.g. HMAC-SHA256) uses one shared secret to both sign and validate — simple, fine when the same party does both. Asymmetric (e.g. RSA/ECDSA) signs with a private key and validates with a public key — essential when the issuer and validators are separate parties, since validators only need the public key and can't forge tokens. Identity providers use asymmetric keys and publish the public keys via a JWKS endpoint that the API fetches automatically when you set Authority.

Q20Why prefer an identity provider over signing your own tokens?🔴 Senior

Microsoft explicitly discourages hand-rolling tokens outside testing/closed systems. A provider (Entra ID, Auth0, Keycloak, Duende IdentityServer) gives you standards-compliant OpenID Connect/OAuth flows, key rotation, MFA, consent, federation, and battle-tested security you won't reproduce correctly yourself. You just set options.Authority and the API fetches signing keys automatically. Self-signed tokens are fine to learn on, risky in production.

Q21How does the JWT bearer handler turn a token into HttpContext.User?🔴 Senior

During UseAuthentication, the handler reads the Bearer token, validates it against TokenValidationParameters (signature, issuer, audience, lifetime), and if valid, constructs a ClaimsPrincipal from the token's claims and assigns it to HttpContext.User. Downstream, authorization and your code read that principal — User.IsInRole(...), User.FindFirst(...) — all sourced from the token, with no database hit.

Q22Why is JsonWebTokenHandler preferred over JwtSecurityTokenHandler?🔴 Senior

JsonWebTokenHandler is the modern successor in the Microsoft.IdentityModel libraries — it's faster, more memory-efficient, and works directly with the newer JsonWebToken type. JwtSecurityTokenHandler is the older API (and does some claim-type remapping by default that surprises people). New code should use JsonWebTokenHandler for both creating and validating tokens.

Q23What are the main security pitfalls when implementing JWT auth?🔴 Senior

Transmitting tokens over plain HTTP; storing the signing key in source control; long-lived access tokens with no revocation strategy; putting sensitive data in the readable payload; over-broad claims that make one leak catastrophic; accepting the alg: none algorithm or not validating the signature; and skipping issuer/audience validation. The defenses: HTTPS everywhere, secrets in a vault, short lifetimes plus refresh tokens, minimal claims, and always validating signature, issuer, audience, and lifetime.
