CRUD End-to-End 🔁

Today everything from the past week clicks into place. We take one resource — a Product — and build the full set of operations that let a client create it, read it, update it, and delete it, with the data living in a real SQL Server database via Entity Framework Core. That's five endpoints: list all, get one, create, update, delete. Then we verify each one actually works, by hand and with automated tests. This is the shape of nearly every real API you'll ever build.

➕ Create 📖 Read (list + one) ✏️ Update 🗑️ Delete 🧩 EF Core 🗃️ SQL Server 🧪 Tested
01

What "CRUD End-to-End" Means 🔁

CRUD stands for Create, Read, Update, Delete — the four things you can do to almost any piece of data. "End-to-end" means we build the whole path for one resource: from an HTTP request, through the layers we learned this week, into a SQL Server table, and all the way back.

In a REST API those four operations become five endpoints, because "Read" splits into "read the whole collection" and "read one item":

#OperationHTTPRouteSuccess code
1Read allGET/api/products200 OK
2Read oneGET/api/products/{id}200 OK / 404
3CreatePOST/api/products201 Created
4UpdatePUT/api/products/{id}204 No Content / 404
5DeleteDELETE/api/products/{id}204 No Content / 404
🔑 This is the week, assembled

Every layer we built shows up here: validation (Day 4) on the incoming DTO, DTOs (Day 5) separating the API from the entity, correct status codes (Day 6) on each response, and the service + data layers (Day 7) doing the work. CRUD is where all of it meets a real database.

🗂️ Analogy — a filing cabinet

Think of a drawer of index cards. Create = write a new card and file it. Read = pull one card, or flip through them all. Update = erase and rewrite a card. Delete = shred a card. EF Core is the clerk who does the physical filing; SQL Server is the cabinet itself.

02

Wiring EF Core to SQL Server 🔌

Entity Framework Core is an ORM — an Object-Relational Mapper. It lets you work with C# objects and LINQ instead of hand-writing SQL; EF Core translates between your classes and database tables. Three pieces get us connected.

Step 1 — Install the packages 📦

bash
# The SQL Server provider (pulls in EF Core itself)
dotnet add package Microsoft.EntityFrameworkCore.SqlServer

# Design-time support for the migrations commands
dotnet add package Microsoft.EntityFrameworkCore.Design

Step 2 — The entity and the DbContext 🧱

The entity is a plain C# class that maps to a table. The DbContext is your gateway to the database: it exposes a DbSet<T> per table and takes its options through the constructor (so DI can configure it).

C# — Product.cs (the entity)
public class Product
{
    public int Id { get; set; }          // primary key — SQL Server auto-assigns it
    public string Name { get; set; } = null!;
    public decimal Price { get; set; }
}
C# — AppDbContext.cs
using Microsoft.EntityFrameworkCore;

public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options) { }

    // One DbSet per table. This becomes the "Products" table.
    public DbSet<Product> Products => Set<Product>();
}

Step 3 — Register it + a connection string 🔗

Put the connection string in appsettings.json, then register the context in Program.cs. Recall from Day 7 that AddDbContext registers the context as scoped (one per request).

JSON — appsettings.json
{
  "ConnectionStrings": {
    "Default": "Server=localhost;Database=ShopDb;Trusted_Connection=True;TrustServerCertificate=True;"
  }
}
C# — Program.cs
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseSqlServer(builder.Configuration.GetConnectionString("Default")));

// .NET 9+ ships native OpenAPI (no Swashbuckle) — more in section 07
builder.Services.AddOpenApi();

var app = builder.Build();
if (app.Environment.IsDevelopment()) app.MapOpenApi();
app.MapControllers();
app.Run();
🔑 Connection string, decoded

Server=localhost is your SQL Server instance; Database=ShopDb is created if it doesn't exist; Trusted_Connection=True uses Windows authentication (no username/password); TrustServerCertificate=True avoids a local dev TLS certificate error. For local development, LocalDB (Server=(localdb)\\MSSQLLocalDB) is a lightweight, on-demand SQL Server that needs no setup.

03

Migrations — Building the Schema 🏗️

Your C# entity describes the shape of the data, but the database doesn't have a table yet. Migrations bridge that gap: EF Core generates versioned files that create and evolve the schema to match your model, without you writing SQL by hand.

bash — the two commands you'll run constantly
# 1) Generate a migration by comparing your model to the last snapshot
dotnet ef migrations add InitialCreate

# 2) Apply pending migrations to the database (creates it if needed)
dotnet ef database update

The first command creates a Migrations/ folder with an InitialCreate class (an Up method that builds the table and a Down that reverses it) plus a model snapshot. The second runs that Up against SQL Server.

🔑 Two things EF Core creates for bookkeeping

A __EFMigrationsHistory table records which migrations have been applied, so they never run twice. A ModelSnapshot.cs captures your model's current state — the next migrations add diffs against it to figure out what changed. Because of the snapshot, you undo a migration with dotnet ef migrations remove, not by deleting the file.

✅ The change workflow

Add a property to the entity → dotnet ef migrations add AddDescription → dotnet ef database update. Migrations update the schema while preserving existing data, which is exactly what you need once the app is live. Repeat this loop every time the model changes.

⚠️ EnsureCreated vs. Migrate — don't mix them

EnsureCreated() quickly builds the database from the model but writes no migration history and can't evolve the schema later — fine for throwaway demos and tests, wrong for real apps. Migrate() applies migrations properly. Never use both on the same database. And avoid auto-running migrations on startup in production — prefer generating a SQL script (dotnet ef migrations script) and applying it in your deployment pipeline.

04

How EF Core Tracks Changes 🧠

Understanding this one idea makes all five operations obvious. When you load or add an entity, the DbContext tracks it and remembers its state. You don't write SQL — you change objects, then call SaveChangesAsync(), and EF Core looks at each tracked entity's state to decide what SQL to run.

Added
→ INSERT
Unchanged
→ (nothing)
Modified
→ UPDATE
Deleted
→ DELETE
⟶ SaveChangesAsync() ⟶
SQL runs
one transaction
You do this…State becomesSaveChangesAsync issues
Add(entity)AddedINSERT

Read from DB, leave it aloneUnchangednothing
🔑 SaveChangesAsync is your Unit of Work

All the changes you make between reads are batched and committed together in a single transaction when you call SaveChangesAsync() (recall Day 7 — the DbContext is the Unit of Work). If it throws, nothing is persisted. Always use the async version in a web app for scalability.

✅ AsNoTracking for read-only queries

When you're only reading data to return it (never editing), add .AsNoTracking() to the query. EF Core then skips setting up change-tracking for those entities — less memory, faster reads. Use it for your "read all" and "read one" list queries; skip it when you intend to modify and save.

05

Create & Read 📖

Here's the controller so far, wired to a service (Day 7) that returns DTOs (Day 5). We'll show the service methods alongside each operation so you see the full path. Start with the two Read operations and Create.

C# — ProductsController (Read all, Read one, Create)
[ApiController]
[Route("api/[controller]")]
public class ProductsController(IProductService service) : ControllerBase
{
    // 1) READ ALL -> 200 with a list
    [HttpGet]
    public async Task<ActionResult<List<ProductDto>>> GetAll() =>
        await service.GetAllAsync();

    // 2) READ ONE -> 200 if found, 404 if not
    [HttpGet("{id:int}")]
    public async Task<ActionResult<ProductDto>> GetById(int id)
    {
        var product = await service.GetByIdAsync(id);
        return product is null ? NotFound() : product;
    }

    // 3) CREATE -> 201 with a Location header
    [HttpPost]
    public async Task<ActionResult<ProductDto>> Create(CreateProductRequest request)
    {
        var dto = await service.CreateAsync(request);
        return CreatedAtAction(nameof(GetById), new { id = dto.Id }, dto);
    }
}

And the matching data-access work inside the service. Notice the EF Core calls: ToListAsync, FindAsync, Add, and SaveChangesAsync.

C# — ProductService (the EF Core calls)
public class ProductService(AppDbContext db) : IProductService
{
    // READ ALL — project straight into the DTO, no tracking needed
    public Task<List<ProductDto>> GetAllAsync() =>
        db.Products
          .AsNoTracking()
          .Select(p => new ProductDto(p.Id, p.Name, p.Price))
          .ToListAsync();

    // READ ONE — FindAsync looks up by primary key
    public async Task<ProductDto?> GetByIdAsync(int id)
    {
        var p = await db.Products.FindAsync(id);
        return p is null ? null : new ProductDto(p.Id, p.Name, p.Price);
    }

    // CREATE — Add marks it Added; SaveChanges issues the INSERT
    public async Task<ProductDto> CreateAsync(CreateProductRequest request)
    {
        var product = new Product { Name = request.Name, Price = request.Price };
        db.Products.Add(product);
        await db.SaveChangesAsync();   // product.Id is now populated by SQL Server
        return new ProductDto(product.Id, product.Name, product.Price);
    }
}
🔑 The Id fills itself in

You never set product.Id yourself — it's the primary key, and SQL Server assigns it on INSERT. After SaveChangesAsync(), EF Core copies the generated value back into your entity, which is how CreatedAtAction can build the Location URL to the brand-new resource. (And because we bind a CreateProductRequest DTO, not the entity, there's no over-posting — Day 5.)

06

Update & Delete ✏️🗑️

These two both return 204 No Content on success and 404 if the target doesn't exist. The pattern is the same: load the entity first (so EF Core tracks it), then modify or remove, then save.

C# — ProductsController (Update, Delete)
    // 4) UPDATE -> 204 on success, 404 if missing
    [HttpPut("{id:int}")]
    public async Task<IActionResult> Update(int id, UpdateProductRequest request)
    {
        var ok = await service.UpdateAsync(id, request);
        return ok ? NoContent() : NotFound();
    }

    // 5) DELETE -> 204 on success, 404 if missing
    [HttpDelete("{id:int}")]
    public async Task<IActionResult> Delete(int id)
    {
        var ok = await service.DeleteAsync(id);
        return ok ? NoContent() : NotFound();
    }
C# — ProductService (Update, Delete)
    // UPDATE — load, mutate tracked entity, save (EF diffs the changes)
    public async Task<bool> UpdateAsync(int id, UpdateProductRequest request)
    {
        var product = await db.Products.FindAsync(id);
        if (product is null) return false;

        product.Name = request.Name;      // entity is tracked -> state becomes Modified
        product.Price = request.Price;
        await db.SaveChangesAsync();       // issues UPDATE for only the changed columns
        return true;
    }

    // DELETE — load, Remove marks it Deleted, save issues DELETE
    public async Task<bool> DeleteAsync(int id)
    {
        var product = await db.Products.FindAsync(id);
        if (product is null) return false;

        db.Products.Remove(product);
        await db.SaveChangesAsync();
        return true;
    }
🔑 Why "load first" for Update?

Loading the entity means EF Core is tracking it, so when you change a property its state flips to Modified and SaveChangesAsync generates an UPDATE touching only the changed columns. It also gives you a natural 404 when the id doesn't exist. (There's a faster "set-based" ExecuteUpdateAsync for bulk edits without loading, but load-then-save is the clearest default and the one to know first.)

⚠️ PUT means "replace," so validate the whole thing

PUT replaces the resource, so the UpdateProductRequest should carry all editable fields and be validated just like create (Day 4). The id comes from the route, never the body — that keeps identity server-controlled and avoids over-posting. For partial edits you'd use PATCH instead, which is a separate topic.

07

Verifying by Hand — OpenAPI & .http Files 🔎

Before automated tests, you'll want to click through each endpoint yourself. Here's the modern .NET story, which changed recently and trips people up.

⚠️ Swagger is no longer in the box (.NET 9+)

For years the Web API template bundled Swashbuckle (Swagger UI). Starting in .NET 9, ASP.NET Core dropped that dependency and ships its own native Microsoft.AspNetCore.OpenApi package instead. In .NET 10 it generates OpenAPI 3.1 documents natively. The catch: the native package produces the OpenAPI document but no UI. Swashbuckle still exists as a community package if you want it.

With AddOpenApi() and MapOpenApi() (from section 02), the raw document is served at /openapi/v1.json. To get an interactive UI back, add a renderer — Scalar is the popular modern choice:

C# — add Scalar UI (Program.cs)
// dotnet add package Scalar.AspNetCore
if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();                 // serves /openapi/v1.json
    app.MapScalarApiReference();       // interactive UI at /scalar/v1
}

Or skip the UI entirely — use a .http file 📝

Visual Studio and VS Code (with the REST Client extension) can send requests straight from a checked-in .http file. It's the fastest way to exercise all five operations in order, and it lives in your repo as runnable documentation.

HTTP — products.http (run each request in order)
@base = https://localhost:5001

### 3) CREATE -> expect 201 Created + Location header
POST {{base}}/api/products
Content-Type: application/json

{ "name": "Notebook", "price": 4.99 }

### 1) READ ALL -> expect 200 + JSON array
GET {{base}}/api/products

### 2) READ ONE -> expect 200 (use the id from the create response)
GET {{base}}/api/products/1

### 4) UPDATE -> expect 204 No Content
PUT {{base}}/api/products/1
Content-Type: application/json

{ "name": "Notebook XL", "price": 6.49 }

### 5) DELETE -> expect 204; a second GET /1 then returns 404
DELETE {{base}}/api/products/1
✅ The manual smoke test

Run them top to bottom and watch the status codes: 201 → 200 → 200 → 204 → 204, then a final GET /1 should give 404. If every code matches the table in section 01, all five operations are wired correctly. This 60-second loop catches most mistakes before you write a single automated test.

08

Automated Tests — Proving All Five 🧪

Manual checks are great once; automated tests prove it stays working forever. The gold standard for a CRUD API is an integration test: spin up the whole app in memory with WebApplicationFactory and send real HTTP requests through an HttpClient.

C# — integration test with WebApplicationFactory (xUnit)
public class ProductsApiTests(WebApplicationFactory<Program> factory)
    : IClassFixture<WebApplicationFactory<Program>>
{
    private readonly HttpClient _client = factory.CreateClient();

    [Fact]
    public async Task Create_then_Get_returns_the_product()
    {
        // CREATE -> 201
        var create = await _client.PostAsJsonAsync("/api/products",
            new { name = "Pen", price = 1.20 });
        Assert.Equal(HttpStatusCode.Created, create.StatusCode);

        var created = await create.Content.ReadFromJsonAsync<ProductDto>();

        // READ ONE -> 200 and the same data
        var get = await _client.GetAsync($"/api/products/{created!.Id}");
        Assert.Equal(HttpStatusCode.OK, get.StatusCode);
    }

    [Fact]
    public async Task Get_missing_product_returns_404()
    {
        var res = await _client.GetAsync("/api/products/999999");
        Assert.Equal(HttpStatusCode.NotFound, res.StatusCode);
    }
}
🔑 What database do tests use?

You don't want tests hitting your real SQL Server. Two common choices: the EF Core InMemory provider (fast, but doesn't enforce relational rules like foreign keys or unique constraints — good for pure logic, misleading for query semantics), or SQLite in-memory / a containerized SQL Server (slower, but behaves like a real relational database). For CRUD you're testing real query behavior, so SQLite-in-memory or a test container gives the most trustworthy results.

✅ Cover the five, plus the sad paths

A solid suite asserts each operation's happy path and its failure: create returns 201; read-all returns the list; read-one returns 200 (and 404 for a missing id); update returns 204 (and 404); delete returns 204, after which the item is really gone (404 on re-read). That last "delete then confirm gone" check is the one people forget — and it's exactly what proves the operation reached the database.

End-to-End Trace: One POST 🧩

Let's follow a single POST /api/products all the way down to SQL Server and back, so every layer's job is visible in one picture.

Step 1 · 📥 bind & validate
Request → CreateProductRequest DTO

Model binding fills the DTO; [ApiController] auto-returns 400 if validation fails (Day 4). No entity is exposed (Day 5).

Step 2 · 🎛️ controller
Delegate to the service

The thin action calls service.CreateAsync(request) and awaits a ProductDto back (Day 7).

Step 3 · 🧠 service + EF Core
Build entity, Add, SaveChanges

Creates a Product, calls Add (state → Added), then SaveChangesAsync — EF Core emits an INSERT.

Step 4 · 🗃️ SQL Server
Row inserted, Id generated

The database writes the row and assigns the identity Id, which flows back into the tracked entity.

Step 5 · 📤 response
201 Created + Location

The service returns a ProductDto; the controller wraps it in CreatedAtAction with a Location header (Day 6).

🔑 The whole week in one request

Validation → DTO → Controller → Service → EF Core → SQL Server → status code. Each day of this week owns one hop. When you can narrate this trace out loud — and repeat it for GET, PUT, and DELETE — you understand end-to-end CRUD.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1What does CRUD stand for, and why is it five endpoints?🌱 Junior

Create, Read, Update, Delete — the four basic operations on data. In a REST API it usually becomes five endpoints because Read splits into "read the whole collection" (GET /products) and "read one item" (GET /products/{id}), alongside POST, PUT, and DELETE.

Q2What is Entity Framework Core?🌱 Junior

It's an ORM (Object-Relational Mapper) for .NET. It lets you work with C# classes and LINQ queries instead of writing SQL by hand, and it translates between your objects and database tables — reading rows into objects and turning object changes back into SQL.

Q3What is a DbContext and a DbSet?🌱 Junior

The DbContext is your gateway to the database — it manages the connection, tracks changes, and saves them. A DbSet<T> is a property on the context representing a table for one entity type (e.g. DbSet<Product> maps to the Products table), and you query and modify it like a collection.

Q4What are migrations, and which two commands do you use most?🌱 Junior

Migrations are versioned files that keep the database schema in sync with your C# model. The two commands: dotnet ef migrations add <Name> to generate a migration from model changes, and dotnet ef database update to apply pending migrations to the database (creating it if needed).

Q5Which method actually saves changes to the database?🌱 Junior

SaveChangesAsync() (or its sync version). You add, edit, or remove tracked entities in memory, and nothing hits the database until you call it — at which point EF Core generates and runs the appropriate INSERT/UPDATE/DELETE statements in a single transaction.

Q6How do you create a new record with EF Core?🌱 Junior

Make a new entity object, call db.Products.Add(product) to mark it as Added, then await db.SaveChangesAsync(), which issues the INSERT. You don't set the Id — SQL Server generates it and EF Core copies it back into your object.

Q7What status code should each CRUD operation return on success?🌱 Junior

Read (list or one) → 200 OK. Create → 201 Created with a Location header. Update and Delete → 204 No Content. And when a read/update/delete targets an id that doesn't exist → 404 Not Found.

Q8How do you read one record versus all records?🌱 Junior

All records: await db.Products.ToListAsync(). One record by primary key: await db.Products.FindAsync(id) (returns null if not found). For read-only queries you can add .AsNoTracking() to skip change-tracking and go faster.

Q9How does EF Core know whether to INSERT, UPDATE, or DELETE?🟠 Mid

Through change tracking. Each tracked entity has a state: Added → INSERT, Modified → UPDATE, Deleted → DELETE, Unchanged → nothing. Add sets Added, editing a property of a loaded entity sets Modified, and Remove sets Deleted. SaveChangesAsync inspects those states and generates the matching SQL.

Q10Why load an entity before updating it?🟠 Mid

Loading it makes EF Core track it, so changing a property flips its state to Modified and SaveChangesAsync emits an UPDATE for just the changed columns. It also gives you a clean 404 path when the id doesn't exist. (There's a set-based ExecuteUpdateAsync for bulk updates without loading, but load-then-save is the clearest default.)

Q11What's the difference between EnsureCreated and Migrate?🟠 Mid

EnsureCreated() builds the database from the current model but records no migration history and can't evolve the schema afterward — fine for tests/demos. Migrate() applies migrations, tracks them in __EFMigrationsHistory, and supports schema evolution — the right choice for real apps. Never mix the two on one database.

Q12What is AsNoTracking and when should you use it?🟠 Mid

It tells EF Core not to set up change-tracking for the entities a query returns, which uses less memory and runs faster. Use it for read-only queries where you'll only display or return the data (your list and read-one endpoints). Don't use it when you intend to modify the loaded entities and save them.

Q13Why shouldn't the client set the Id on create?🟠 Mid

The Id is a database-generated primary key — SQL Server assigns it on insert. Accepting it from the client invites conflicts and is a form of over-posting. That's why you bind a create DTO that omits Id (Day 5), let the database generate it, and read it back off the entity after SaveChangesAsync to build the Location header.

Q14What changed with Swagger in .NET 9 and 10?🟠 Mid

ASP.NET Core dropped the Swashbuckle (Swagger UI) dependency from the Web API template in .NET 9 and shipped native OpenAPI generation via Microsoft.AspNetCore.OpenApi (AddOpenApi()/MapOpenApi()), producing OpenAPI 3.1 documents in .NET 10 at /openapi/v1.json. The native package makes the document but no UI, so teams add a renderer like Scalar — or just use .http files. Swashbuckle still exists as a community package.

Q15How would you test all five CRUD operations?🟠 Mid

Write integration tests with WebApplicationFactory<Program>, which hosts the app in memory and gives you an HttpClient. Send real requests and assert status codes and bodies: create → 201, read-all → 200, read-one → 200/404, update → 204/404, delete → 204 then a re-read returns 404. Point the app at a test database rather than production.

Q16What database should integration tests use — InMemory or something else?🟠 Mid

The EF Core InMemory provider is fast but doesn't enforce relational rules (foreign keys, unique constraints) and doesn't translate SQL, so it's good for pure logic and misleading for query behavior. For trustworthy CRUD tests, prefer SQLite in-memory or a containerized SQL Server, which behave like a real relational database.

Q17Why avoid running migrations automatically on app startup in production?🔴 Senior

Auto-migrating on startup can cause downtime and race conditions when multiple app instances start at once, and it couples schema changes to deploys with no review step. The safer pattern is to generate a SQL script (dotnet ef migrations script), review it, and apply it in a controlled deployment step — ideally with backward-compatible, phased migrations for zero-downtime releases.

Q18How do you handle concurrent updates to the same row?🔴 Senior

Use optimistic concurrency with a concurrency token — on SQL Server, a rowversion column mapped via .IsRowVersion(), which the database maintains automatically. EF Core includes the original version in the UPDATE's WHERE; if the row changed since you read it, no rows match and it throws DbUpdateConcurrencyException. Catch it, return 409 Conflict with the current values, and let the client retry.

Q19PUT vs. PATCH — what's the difference for update?🔴 Senior

PUT is a full replacement: the body carries all editable fields and you validate it like a create. PATCH is a partial update: the body describes only the changes (often JSON Patch or a merge-patch document). PUT is simpler and idempotent; PATCH avoids sending the whole resource but needs careful handling of which fields are actually present versus intentionally null.

Q20When would you use ExecuteUpdate/ExecuteDelete instead of load-then-save?🔴 Senior

For bulk operations. ExecuteUpdateAsync / ExecuteDeleteAsync issue a single set-based SQL statement without loading entities into memory or tracking them — ideal for "delete all expired" or "raise all prices 10%." The trade-offs: they bypass change tracking, don't run SaveChanges interceptors or concurrency checks, and won't fire domain events, so use them for genuine bulk work, not single-entity edits where load-then-save is clearer and safer.

Q21How do you keep tests isolated so they don't pollute each other's data?🔴 Senior

Give each test a clean database state. Common approaches: a fresh SQLite in-memory connection per test, wrapping each test in a transaction that's rolled back, or a customized WebApplicationFactory that swaps the real DbContext registration for a per-test database and re-seeds it. The goal is that tests can run in any order, in parallel, and repeatedly without shared state leaking between them.

Q22What N+1 and performance pitfalls show up in CRUD reads?🔴 Senior

The classic is the N+1 query: loading a list, then lazily loading a related property per item, firing one query per row. Fix it by eager-loading with Include, or better, projecting directly into a DTO with Select so EF Core fetches only needed columns in one query (Day 5). Also add AsNoTracking for reads, and paginate list endpoints (Skip/Take) so "read all" doesn't return an unbounded result set.

Q23How should a CRUD API represent "delete" — hard or soft?🔴 Senior

A hard delete removes the row (Remove + DELETE) — simple, but the data is gone and referential integrity can block it. A soft delete sets an IsDeleted/DeletedAt flag and filters it out (often via an EF Core global query filter), preserving history and audit trails at the cost of extra query complexity and care around unique constraints. Choose based on whether the business needs recoverability and audit; both still return 204 to the client.
