Returning Results in ASP.NET Core 📤

An HTTP response is more than a blob of JSON — it's a status code that tells the client what happened, optional headers, and an optional body. Return the wrong code and a caller can't tell success from failure, a browser won't cache, and a client SDK guesses. Today we learn the tools ASP.NET Core gives you to shape responses cleanly — IActionResult, ActionResult<T>, Results/TypedResults, and ProblemDetails — and when to reach for 200, 201, 204, and 404.

🟢 200 OK 🆕 201 Created 🫗 204 No Content 🔎 404 Not Found 🎛️ IActionResult ⚡ TypedResults 🧾 ProblemDetails
01

Why Return Types Matter 🎯

When your action finishes, ASP.NET Core has to turn whatever you returned into a real HTTP response. That response has three parts, and a good API is deliberate about all three:

Status code
what happened
Headers
metadata, Location…
Body
the data (maybe)
=
HTTP response
what the client sees

The status code is the headline. A client reads it first to decide what to do — retry, show data, show an error, follow a link. Returning 200 OK for everything (even failures) is one of the most common API mistakes: it forces every caller to dig into the body to guess what actually happened.

📦 Analogy — the delivery slip

A status code is the sticker on a package: "Delivered," "Address not found," "Signature required." A courier who slapped "Delivered" on every parcel — even the ones they couldn't deliver — would be useless. The sticker (status code) lets the recipient act without opening the box (the body). Return the honest sticker every time.

Two worlds, one goal 🌗

ASP.NET Core has two styles of building APIs, and each has its own family of return helpers:

🎛️

Controllers (MVC)

Actions return IActionResult, ActionResult<T>, or a plain type. Helpers live on ControllerBase: Ok(), NotFound(), etc.

⚡

Minimal APIs

Route handlers return IResult via the Results / TypedResults classes: Results.Ok(), TypedResults.NotFound(), etc.

The big picture 🗺️

Here's the full family of ways to return a result. Each branch is a section below.

🔑 The four this lesson targets

⚠️ 200-for-everything is an anti-pattern

Returning 200 with a body like { "success": false } defeats the purpose of HTTP. Caching, retries, client SDKs, and monitoring all key off the status code. Let the status line carry the truth; don't smuggle failure inside a 200 body.

03

Returning a Plain Type (T) 📦

The simplest thing an action can do is return the data directly — a string, an int, or a custom object. ASP.NET Core serializes it to JSON and sends 200 OK automatically.

C#
[HttpGet]
public async Task<List<Product>> Get() =>
    await _db.Products.OrderBy(p => p.Name).ToListAsync();

On success this returns 200 with a JSON array; on an unhandled exception the framework produces a 500. It's clean and reads well — but notice what it can't do: there's no way to also return a 404 or a 400 from this signature, because the return type is locked to List<Product>.

✅ When a plain type is enough

Use it when the action has a single, always-successful path and no validation — like a "list everything" endpoint with no parameters. Microsoft's guidance: "without known conditions to safeguard against, returning a specific type could suffice." The moment you need a second outcome (not found, invalid input), reach for IActionResult or ActionResult<T>.

🔑 Bonus: self-documenting

Because the return type is right there in the signature (List<Product>), OpenAPI/Swagger can describe the 200 response automatically — no extra attributes needed. That's an advantage plain types share with ActionResult<T>.

04

IActionResult — Full Flexibility 🎛️

IActionResult is an interface representing "any action result." Returning it lets a single action produce different responses down different paths — a 200 here, a 404 there. It's the right choice when an action has multiple possible outcomes.

C#
[HttpGet("{id}")]
public IActionResult GetById(int id)
{
    var product = _db.Products.Find(id);
    if (product is null)
        return NotFound();   // 404

    return Ok(product);      // 200 + JSON body
}

Those helpers — Ok(), NotFound(), BadRequest(), NoContent(), CreatedAtAction() — are convenience methods on ControllerBase. Each creates a result object representing a status code. For example, return BadRequest(); is just shorthand for return new BadRequestResult();.

⚠️ The catch — the type is hidden

Because the signature only says IActionResult, tools like Swagger can't tell what a successful response contains. You have to spell it out with attributes, or your API docs will be blank on the body:

C# — documenting an IActionResult action
[HttpGet("{id}")]
[ProducesResponseType<Product>(StatusCodes.Status200OK)]
[ProducesResponseType(StatusCodes.Status404NotFound)]
public IActionResult GetById(int id) { /* ... */ }
🔑 IActionResult in one line

Maximum flexibility, but you carry the documentation burden yourself. That trade-off is exactly what ActionResult<T> was created to fix.

05

ActionResult<T> — The Sweet Spot 🎯

Introduced in ASP.NET Core 2.1, ActionResult<T> combines the two previous approaches: you can return either a status-code result (NotFound(), BadRequest()) or the object itself, and the compiler handles the rest via implicit conversions.

C#
[HttpGet("{id}")]
public async Task<ActionResult<Product>> GetById(int id)
{
    var product = await _db.Products.FindAsync(id);
    if (product is null)
        return NotFound();   // 404 — a status result

    return product;         // 200 — no Ok() wrapper needed; implicit cast
}

Two things make this the modern default for controller APIs:

🪄

Implicit casts

Both T and ActionResult convert to ActionResult<T>, so you can return product; for success and return NotFound(); for failure in the same method.

📖

Self-documenting

The T tells OpenAPI the success body's shape, so you can drop the Type from [ProducesResponseType].

✅ Rule of thumb

For controller-based Web APIs, reach for ActionResult<T> by default. Use plain IActionResult only when there's genuinely no single success type to name (e.g. an action that returns a file or a redirect). And wrap I/O in async Task<...> whenever you touch a database or network.

06

Results & TypedResults — Minimal APIs ⚡

Minimal APIs don't use IActionResult. Instead, route handlers return IResult, and two static helper classes create those results: Results and TypedResults. They live in the HttpResults family and work in both Minimal APIs and controllers.

C# — Results (returns IResult)
app.MapGet("/products/{id}", async (int id, AppDb db) =>
{
    var product = await db.Products.FindAsync(id);
    return product is null
        ? Results.NotFound()      // 404
        : Results.Ok(product);   // 200
});

The one difference that matters 🔍

Results and TypedResults offer the same helpers, but they differ in return type. Results.Ok(x) returns the interface IResult — the compiler knows nothing about the shape. TypedResults.Ok(x) returns a concrete type, Ok<T>, that the compiler and tooling can inspect.

C#
IResult a = Results.Ok(product);        // type: IResult (opaque)
Ok<Product> b = TypedResults.Ok(product); // type: Ok<Product> (concrete)

The runtime behavior is identical — both send 200 with the JSON body. The typing difference unlocks two real benefits, which is why Microsoft recommends TypedResults as the default for new Minimal API code:

📖

Automatic OpenAPI

TypedResults.Ok<Product>() tells the framework the 200 returns a Product — no .Produces<T>() call needed. With plain Results, the contract is invisible.

🧪

Easier testing

Unit tests can assert Assert.IsType<Ok<Product>>(result) and read .Value directly, instead of converting an opaque IResult first.

🔑 HttpResults vs. MVC results, under the hood

Unlike IActionResult, HttpResults are executed by IResult.ExecuteAsync and don't use the configured formatters. That means no content negotiation — the Content-Type is decided by the result type itself. This is why they're lighter, and why controllers (which want content negotiation) idiomatically stick with ActionResult<T>.

⚠️ TypedResults ≠ controller default

Although HttpResults technically compile inside controllers, the idiomatic split is: controllers → ActionResult<T>, Minimal APIs → TypedResults. Don't rip working IActionResult controllers out to chase TypedResults; apply it to new Minimal API work instead.

07

Union Return Types — Results<T1, …, TN> 🔗

There's a wrinkle: because every TypedResults helper returns a different concrete type, a handler with more than one outcome (say 200 and 404) has two different return types. The compiler can't reconcile that on its own. The fix is the union type Results<T1, …, TN>, which lists every result the handler may produce.

C# — Minimal API with a union return type
app.MapGet("/products/{id}",
    async Task<Results<Ok<Product>, NotFound>> (int id, AppDb db) =>
{
    var product = await db.Products.FindAsync(id);
    return product is null
        ? TypedResults.NotFound()
        : TypedResults.Ok(product);
});

The same pattern works on controller actions too:

C# — union type on a controller action
[HttpGet("{id}")]
public Results<Ok<Product>, NotFound> GetById(int id)
{
    var product = _db.Products.Find(id);
    return product is null
        ? TypedResults.NotFound()
        : TypedResults.Ok(product);
}
✅ Why the union type is powerful

It declares the endpoint's full contract right in the signature — this endpoint returns a Product (200) or nothing (404), and the OpenAPI docs pick that up automatically. Better still, the union types use implicit casts and give you compile-time checking: try to return a result you didn't declare, and the code won't build. Microsoft prefers Results<T1, …, TN> over bare IResult for exactly this reason.

🔑 Rule of thumb

Returning Results (plain)? The compiler infers IResult, so multiple outcomes "just work" — at the cost of type info. Returning TypedResults with more than one outcome? You must declare a Results<…> union — and you get richer docs and safety in return.

08

ProblemDetails — Standardized Errors 🧾

When something goes wrong, don't invent your own error shape per endpoint. ProblemDetails is a standard machine-readable error format (defined by RFC 9457, formerly RFC 7807) that ASP.NET Core produces out of the box. A consistent error contract makes every client's life easier.

JSON — a ProblemDetails body
{
  "type":   "https://tools.ietf.org/html/rfc9110#section-15.5.10",
  "title":  "Insufficient stock",
  "status": 409,
  "detail": "Only 3 units of 'Widget' remain.",
  "instance": "/api/orders/1042"
}

Register it once in Program.cs, and error responses (status 400 and up) get turned into ProblemDetails automatically:

C# — Program.cs
builder.Services.AddProblemDetails();

To return one explicitly with your own message and status code, use the Problem helper — available on both ControllerBase and the Results/TypedResults classes:

C#
// Controller: ControllerBase.Problem(...)
return Problem(
    title: "Insufficient stock",
    detail: $"Only {available} units of '{product.Name}' remain.",
    statusCode: StatusCodes.Status409Conflict);

// Minimal API: TypedResults.Problem(...)
return TypedResults.Problem(
    detail: "There is an error",
    statusCode: StatusCodes.Status403Forbidden);
🔑 Problem vs. ValidationProblem

ValidationProblem is a specialized cousin that returns a ValidationProblemDetails — same shape plus an errors dictionary keyed by field — and it always uses 400. Use it for input-validation failures (recall Day 4's automatic 400s). Use Problem when you need to choose the status code yourself (403, 409, 500, …).

✅ Tip

Pair AddProblemDetails() with a global exception handler so even unexpected 500s come back as tidy ProblemDetails instead of raw stack traces. One consistent error shape across success codes and failures is a hallmark of a professional API.

09

Putting It Together: A CRUD Controller 🧩

Here's the whole lesson applied — one controller returning 200, 201, 204, and 404 in the right places. This is the "Do" for today.

C# — ProductsController
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    // GET all -> 200 with a list
    [HttpGet]
    public async Task<ActionResult<List<Product>>> GetAll() =>
        await _db.Products.ToListAsync();

    // GET one -> 200 if found, 404 if not
    [HttpGet("{id}")]
    public async Task<ActionResult<Product>> GetById(int id)
    {
        var product = await _db.Products.FindAsync(id);
        return product is null ? NotFound() : product;   // 404 : 200
    }

    // POST -> 201 with a Location header pointing to the new resource
    [HttpPost]
    public async Task<ActionResult<Product>> Create(CreateProductRequest request)
    {
        var product = new Product { Name = request.Name, Price = request.Price };
        _db.Products.Add(product);
        await _db.SaveChangesAsync();

        return CreatedAtAction(nameof(GetById),
            new { id = product.Id }, product);          // 201 + Location
    }

    // PUT -> 204 on success, 404 if the target is missing
    [HttpPut("{id}")]
    public async Task<IActionResult> Update(int id, UpdateProductRequest request)
    {
        var product = await _db.Products.FindAsync(id);
        if (product is null) return NotFound();       // 404

        product.Name = request.Name;
        product.Price = request.Price;
        await _db.SaveChangesAsync();
        return NoContent();                            // 204
    }

    // DELETE -> 204 on success, 404 if the target is missing
    [HttpDelete("{id}")]
    public async Task<IActionResult> Delete(int id)
    {
        var product = await _db.Products.FindAsync(id);
        if (product is null) return NotFound();       // 404

        _db.Products.Remove(product);
        await _db.SaveChangesAsync();
        return NoContent();                            // 204
    }
}

Choosing the status code — a decision flow 🔄

When you write an action, walk this path to pick the response:

Step 1 · ❓ did it succeed?
Success or failure?

If the request couldn't be honored, jump to the failure branch (404 / 400 / 409 / 500). Otherwise continue.

Step 2 · 🆕 did you create something?
Was a new resource made?

Yes → 201 Created with a Location header via CreatedAtAction. No → continue.

Step 3 · 📦 is there a body?
Do you have data to return?

Yes → 200 OK with the body. No → 204 No Content (typical for PUT/DELETE).

Step 4 · 🔎 the failure branch
Pick the honest error code

Missing resource → 404. Bad input → 400. State conflict → 409. Our bug → 500 (usually automatic, as ProblemDetails).

🔑 Why CreatedAtAction, not just Ok

CreatedAtAction(nameof(GetById), new { id }, product) does three things at once: sends 201, puts the new resource in the body, and adds a Location header with the URL to fetch it later. That Location header is what makes a POST truly RESTful — the client learns where its new thing lives without guessing.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1Why return a status code instead of just data?🌱 Junior

The status code tells the client what happened without opening the body — success, missing resource, bad input, server error. Caching, retries, client SDKs, and monitoring all rely on it. Returning 200 for everything forces callers to guess by parsing the body, which defeats the point of HTTP.

Q2What does 200 OK mean and how do you return it?🌱 Junior

200 OK means the request succeeded and there's a body with the result. In a controller you return it with Ok(value), or by returning the object directly from an ActionResult<T> or plain-type action. In a Minimal API, Results.Ok(value) or TypedResults.Ok(value).

Q3What does 201 Created mean, and what extra thing does it include?🌱 Junior

201 means a new resource was created (typically after a POST). Besides the status, it includes a Location response header holding the URL of the new resource, so the client knows where to fetch it. CreatedAtAction sets the status, body, and Location header together.

Q4What does 204 No Content mean, and when do you use it?🌱 Junior

204 means the request succeeded but there's nothing to send back in the body. It's the natural response to a successful PUT or DELETE where you don't need to return the updated or deleted resource. You return it with NoContent().

Q5What does 404 Not Found mean and how do you return it?🌱 Junior

404 means the resource the client asked for doesn't exist — for example, a lookup by id that finds nothing. Return it with NotFound() (optionally passing a message or ProblemDetails). It signals the client to stop looking rather than retry.

Q6What is IActionResult?🌱 Junior

It's an interface representing the result of a controller action. Returning IActionResult lets one action produce different responses on different paths — a 200 here, a 404 there — because any result type (like OkObjectResult or NotFoundResult) implements it.

Q7What's the simplest thing an action can return?🌱 Junior

The data itself — a plain type like a string, int, or a custom object (or a list of them). ASP.NET Core serializes it to JSON and returns 200 on success. It's fine when there's a single always-successful path and no need for other status codes.

Q8What do Ok(), NotFound(), and NoContent() do?🌱 Junior

They're convenience methods on ControllerBase that build result objects for common status codes: Ok() → 200 (with an optional body), NotFound() → 404, NoContent() → 204. For instance, return NotFound(); is shorthand for return new NotFoundResult();.

Q9IActionResult vs. ActionResult<T> — what's the difference?🟠 Mid

IActionResult only expresses "some result" — the success body's type is invisible in the signature. ActionResult<T> adds a type parameter, so you can return either a status result (NotFound()) or the object T directly (thanks to implicit casts), and the T documents the success shape. ActionResult<T> is essentially IActionResult plus type information.

Q10Why is ActionResult<T> often preferred over IActionResult?🟠 Mid

Two reasons: you can return the object directly without wrapping it in Ok(), and the return type is self-documenting — OpenAPI/Swagger infers the success body from T, so you can omit the Type from [ProducesResponseType]. You keep full status-code flexibility while getting cleaner code and better docs.

Q11What are Results and TypedResults in Minimal APIs?🟠 Mid

They're two static classes that create IResult objects for Minimal API responses, with matching helpers like Ok(), NotFound(), and Problem(). Results helpers return the interface IResult; TypedResults helpers return concrete types like Ok<T>. Both produce the same HTTP response at runtime.

Q12What's the difference between Results.Ok() and TypedResults.Ok()?🟠 Mid

The return type. Results.Ok(x) returns IResult — opaque to the compiler and tooling. TypedResults.Ok(x) returns Ok<T> — a concrete type. The runtime result (200 + JSON) is identical, but the typed version gives automatic OpenAPI metadata and easier unit testing, which is why it's the recommended default for new code.

Q13What is ProblemDetails and when should you return it?🟠 Mid

It's a standardized, machine-readable error body (RFC 9457) with fields like type, title, status, and detail. Return it for error responses so every client gets a consistent shape. Register AddProblemDetails() for automatic conversion, and use Problem(...) to return one explicitly with a chosen status code.

Q14Which method returns a 201 with a Location header, and why does Location matter?🟠 Mid

CreatedAtAction (or CreatedAtRoute / Created). It returns 201, the created resource in the body, and a Location header pointing to the new resource's URL. The Location matters because it tells the client exactly where its new resource lives — the RESTful convention for answering a successful POST.

Q15What status code for a successful DELETE with no body? For a POST that creates?🟠 Mid

A successful DELETE (or PUT) with nothing to return → 204 No Content. A POST that creates a new resource → 201 Created with a Location header. If the DELETE's target didn't exist, 404 is appropriate. Matching the verb to the right code is the heart of today's "Do."

Q16What is CreatedAtAction and what arguments does it take?🟠 Mid

It builds a 201 response and computes the Location header from another action. Typical call: CreatedAtAction(nameof(GetById), new { id = product.Id }, product) — the action name to link to, the route values to build its URL, and the body to return. Using nameof avoids hard-coding the action name.

Q17When would you return a plain type vs. IActionResult vs. ActionResult<T>?🔴 Senior

Plain type: a single, always-successful path with no validation (e.g. "list all"). IActionResult: multiple outcomes where there's no single success type to name — a redirect, a file, or genuinely heterogeneous responses. ActionResult<T>: the common case — multiple status codes but a clear success type; you get flexibility and self-documentation. In practice ActionResult<T> is the default for controller actions.

Q18What is the Results<T1, …, TN> union type and why use it?🔴 Senior

Because each TypedResults helper returns a different concrete type, a handler with multiple outcomes needs a return type that covers them all. Results<Ok<Product>, NotFound> is a union declaring exactly the results the handler may return. It uses implicit casts, retains OpenAPI metadata automatically, and gives compile-time checking — return an undeclared type and it won't build. Microsoft prefers it over bare IResult for multi-outcome endpoints.

Q19Why does TypedResults improve OpenAPI docs and testing over Results?🔴 Senior

Because the concrete type carries the response contract. TypedResults.Ok<Product>() lets the OpenAPI generator infer a 200 returning a Product with no extra .Produces<T>() call; Results.Ok() returns opaque IResult, so the schema is invisible unless you add metadata by hand. For tests, you can assert IsType<Ok<Product>> and read .Value directly instead of converting an IResult.

Q20How do HttpResults differ from MVC results under the hood?🔴 Senior

HttpResults (from Results/TypedResults) are executed by IResult.ExecuteAsync and don't use the configured output formatters — so there's no content negotiation, and the result type itself decides the Content-Type. MVC's IActionResult pipeline does run formatters and supports content negotiation. That's why HttpResults are lighter and shareable between Minimal APIs and controllers, while controllers idiomatically keep ActionResult<T> when they want negotiation.

Q21400 vs. 404 vs. 409 vs. 422 — how do you pick?🔴 Senior

400 Bad Request: the request was malformed or couldn't be parsed/validated. 404 Not Found: the target resource doesn't exist. 409 Conflict: the request clashes with current state (duplicate, concurrency conflict). 422 Unprocessable Entity: the request was well-formed but semantically invalid (some teams use this instead of 400 for business-rule failures). ASP.NET Core defaults validation failures to 400; consistency across your API matters more than the exact choice between 400 and 422.

Q22How do you standardize error responses across an API?🔴 Senior

Adopt ProblemDetails everywhere. Call AddProblemDetails() so error status codes are converted to the RFC 9457 shape automatically, return Problem()/ValidationProblem() from your own error paths, and register a global exception handler so unexpected 500s also come back as ProblemDetails rather than stack traces. The result is one predictable error contract clients can code against.

Q23Should you migrate existing IActionResult controllers to TypedResults?🔴 Senior

Not as a priority. TypedResults is the recommended default for new Minimal API work, but rewriting stable IActionResult controllers carries risk for little day-to-day gain unless you have a concrete trigger — rewriting the module, generating a client SDK, or fixing weak tests. Apply the modern approach to new code and let legacy controllers evolve naturally; for existing controllers, ActionResult<T> already gives type-safe responses.

✅ Study-order tip

Start with the status codes (Q2–Q5, Q15) — they're the vocabulary everything else uses and the most-asked. Then learn the controller return types in order of power: plain type → IActionResult → ActionResult<T> (Q6–Q10). Layer on the Minimal API side (Results/TypedResults/union types, Q11–Q12, Q18–Q20) and finish with ProblemDetails (Q13, Q22). If you can build the CRUD controller in section 09 from memory and justify each status code, you own this topic.
