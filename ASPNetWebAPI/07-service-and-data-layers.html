Service & Data Layers 🧱

A controller's job is to take a request and hand back a response — not to run business rules, talk to the database, and format results all at once. When everything piles into the controller, it becomes untestable and unchangeable. Today we push that work outward: dependency injection supplies collaborators, a service layer holds the business logic, and (optionally) a repository wraps the data access. The "Do": lift logic out of a fat controller into an injected service.

💉 Dependency injection ⏳ Service lifetimes 🧠 Service layer 🗄️ Repository pattern 🧩 EF Core 🔌 Loose coupling
01

Why Layers? 🎂

A "layer" is just a group of classes with one kind of responsibility, talking to the next layer through a small, well-defined interface. In a typical Web API the layers stack like this:

Controller
HTTP in/out
→
Service
business logic
→
Repository / EF Core
data access
→
Database
storage

Each layer only knows about the one directly below it, and only through an interface. That single rule buys you a lot: you can test the service without a database, swap the data layer without touching controllers, and find any given kind of code in one predictable place.

🍽️ Analogy — a restaurant

The waiter (controller) takes your order and brings your food — but never cooks. The chef (service) knows the recipes and makes decisions. The pantry clerk (repository) just fetches and stores ingredients. If the waiter also cooked and managed inventory, the kitchen would be chaos. Clear roles keep everyone fast and replaceable.

⚠️ The "fat controller" smell

A controller action that validates input, runs business rules, queries the database, and shapes the response all in one method is doing four jobs. It can't be unit-tested without spinning up a database, and the same logic gets copy-pasted into the next controller that needs it. Today's refactor fixes exactly this.

02

Dependency Injection in a Nutshell 💉

Dependency injection (DI) means a class doesn't create the things it needs — they're handed to it from outside. Instead of a controller calling new ProductService(), the framework's built-in container constructs the service and passes it in. This flips control of object creation to the container (a form of Inversion of Control).

🔑 Two steps, always

1. Register the service with the container in Program.cs (say what implements what, and for how long). 2. Request it by listing it as a constructor parameter wherever you need it. The container wires the two together automatically.

C# — Program.cs (registration)
// Map the interface to its implementation, with a lifetime.
builder.Services.AddScoped<IProductService, ProductService>();
builder.Services.AddScoped<IProductRepository, ProductRepository>();

// EF Core's DbContext is registered scoped by this helper.
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseSqlServer(connectionString));
✅ Why depend on an interface, not the concrete class?

When a controller asks for IProductService instead of ProductService, it no longer cares how the service works — only what it does. That lets you swap the implementation (a real one, a fake for tests, a cached one) without changing the controller. Microsoft's own guidance: request dependencies as constructor parameters, and avoid directly instantiating dependent classes because it couples your code to one implementation.

🔌 Analogy — the wall socket

Your lamp doesn't come with a power station wired inside it. It has a plug, and the socket delivers electricity. DI is the socket: your class exposes what it needs (the plug — a constructor parameter), and the container supplies it. Swap the power source (coal, solar) and the lamp neither knows nor cares.

03

Injecting Services Into Controllers 🎛️

The standard way to get a service into a controller is constructor injection. You declare the dependencies as constructor parameters, store them in readonly fields, and use them in your actions. The container fills them in when it creates the controller.

C#
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private readonly IProductService _service;

    // The container injects IProductService here.
    public ProductsController(IProductService service) => _service = service;

    [HttpGet("{id}")]
    public async Task<ActionResult<ProductDto>> GetById(int id)
    {
        var product = await _service.GetByIdAsync(id);
        return product is null ? NotFound() : product;
    }
}

Notice the controller is now tiny: it receives the request, calls one service method, and maps the result to an HTTP response. It contains no business logic and no database code. That's the goal.

Two other injection styles 🧰

🎯

[FromServices]

Inject a service into a single action method's parameters instead of the constructor — handy when only one action needs it.

🧵

Minimal API params

In Minimal APIs, services are just parameters on the route handler lambda; the container matches them by type automatically.

C# — the two alternatives
// [FromServices] on one action only
[HttpGet]
public async Task<IActionResult> GetAll([FromServices] IProductService service) =>
    Ok(await service.GetAllAsync());

// Minimal API: the service is just a handler parameter
app.MapGet("/products/{id}", async (int id, IProductService service) =>
    await service.GetByIdAsync(id) is { } p ? Results.Ok(p) : Results.NotFound());
🔑 Prefer the constructor

Constructor injection is the default for a reason: it makes a class's dependencies obvious and explicit, and yields classes that are easy to test (you just pass fakes to the constructor). Reach for [FromServices] only when a dependency is needed by a single action.

04

Service Lifetimes — The Gotcha ⏳

When you register a service, you choose how long each instance lives. Pick wrong and you get subtle, painful bugs. There are three lifetimes:

LifetimeNew instance…Register withGood for
🟢 TransientEvery single time it's requestedAddTransientLightweight, stateless helpers (mappers, formatters)
🔵 ScopedOnce per HTTP requestAddScopedServices and repositories; anything using DbContext
🟣 SingletonOnce for the whole appAddSingletonStateless shared or expensive-to-create services
🔑 DbContext is scoped by default

AddDbContext<T> registers your EF Core context with a scoped lifetime — one per request. That's deliberate: a DbContext is a short-lived unit of work that tracks changes for a single request and isn't thread-safe. Any service or repository that uses it should therefore also be scoped.

⚠️ The captive dependency trap

Never inject a shorter-lived service into a longer-lived one — for example, a scoped DbContext into a singleton. The singleton holds onto that one DbContext forever, sharing it across all requests and threads, which corrupts change-tracking and throws errors. This is a "captive dependency." In development, ASP.NET Core's scope validation throws an exception to catch exactly this.

Singleton
app lifetime
✅ may use →
Scoped
per request
✅ may use →
Transient
per resolve

Safe direction: longer-lived may depend on equal-or-longer-lived. Injecting right-to-left (a scoped into a singleton) is the captive-dependency bug.

✅ Simple rule of thumb

Default your business services and repositories to scoped — it matches the request-per-instance model of DbContext and is almost always correct for web APIs. Use transient for tiny stateless helpers, and singleton only for genuinely global, thread-safe, stateless (or carefully-managed) services.

05

Building a Service Layer 🧠

The service layer is where business logic lives — the rules and decisions that make your app your app. It sits between controllers (HTTP) and the data layer (persistence). Each service is defined by an interface so callers depend on the contract, not the implementation.

C# — the contract
public interface IProductService
{
    Task<List<ProductDto>> GetAllAsync();
    Task<ProductDto?> GetByIdAsync(int id);
    Task<ProductDto> CreateAsync(CreateProductRequest request);
}
C# — the implementation
public class ProductService : IProductService
{
    private readonly IProductRepository _repo;

    public ProductService(IProductRepository repo) => _repo = repo;

    public async Task<ProductDto?> GetByIdAsync(int id)
    {
        var product = await _repo.GetByIdAsync(id);
        return product is null ? null : product.ToDto();
    }

    public async Task<ProductDto> CreateAsync(CreateProductRequest request)
    {
        // Business rule lives HERE, not in the controller.
        if (await _repo.NameExistsAsync(request.Name))
            throw new DuplicateNameException(request.Name);

        var product = new Product { Name = request.Name, Price = request.Price };
        await _repo.AddAsync(product);
        await _repo.SaveChangesAsync();
        return product.ToDto();
    }
}
🔑 What belongs in a service?

Business rules ("a product name must be unique"), orchestration across multiple repositories, mapping between entities and DTOs, and calls to other services or external APIs. What does not belong: HTTP concerns (status codes, routing) — those stay in the controller; and raw SQL/query mechanics — those stay in the data layer.

✅ Watch the constructor

Microsoft's guidance: if a class ends up with a long list of injected dependencies, that's a sign it has too many responsibilities (a Single Responsibility Principle violation). Split it into smaller, focused services rather than growing one giant one.

06

The Refactor — Fat Controller → Service 🔧

This is today's "Do." Here's a controller doing everything itself — validation, business rule, database access, mapping — all crammed into one action.

C# — BEFORE: the fat controller ❌
[HttpPost]
public async Task<IActionResult> Create(CreateProductRequest request)
{
    // data access in the controller...
    var exists = await _db.Products
        .AnyAsync(p => p.Name == request.Name);

    // ...business rule in the controller...
    if (exists)
        return Conflict("A product with that name already exists.");

    // ...entity creation in the controller...
    var product = new Product { Name = request.Name, Price = request.Price };
    _db.Products.Add(product);
    await _db.SaveChangesAsync();

    // ...and mapping in the controller. Four jobs, one method.
    var dto = new ProductDto(product.Id, product.Name, product.Price);
    return CreatedAtAction(nameof(GetById), new { id = product.Id }, dto);
}

Now the same behavior, with the logic moved into ProductService. The controller shrinks to pure request/response translation.

C# — AFTER: thin controller ✅
[HttpPost]
public async Task<ActionResult<ProductDto>> Create(CreateProductRequest request)
{
    try
    {
        var dto = await _service.CreateAsync(request);   // all logic delegated
        return CreatedAtAction(nameof(GetById), new { id = dto.Id }, dto);
    }
    catch (DuplicateNameException ex)
    {
        return Conflict(ex.Message);   // controller only maps to HTTP
    }
}

What the refactor buys you 🎁

🧪

Testable

Unit-test ProductService with a fake repository — no web server, no database, no HTTP.

♻️

Reusable

A background job or another endpoint can call CreateAsync without duplicating the rule.

🔍

Readable

The controller reads like a table of contents; the service reads like the actual story.

🔧

Changeable

Business rules change in one place. The controller and database code stay untouched.

✅ Handling service errors cleanly

The try/catch above works, but many teams prefer a global exception handler (recall Day 6's ProblemDetails) or a result object so the service reports "duplicate" without throwing. Either way, the principle holds: the service decides what happened; the controller decides the HTTP status for it.

07

The Repository Pattern Over EF Core 🗄️

A repository is a class that wraps data access behind a collection-like interface, so the rest of your app asks for entities without knowing SQL or EF Core exists. The Unit of Work is the companion idea: a single object that batches several changes and commits them together in one transaction.

C# — a repository interface + EF Core implementation
public interface IProductRepository
{
    Task<Product?> GetByIdAsync(int id);
    Task<List<Product>> GetAllAsync();
    Task<bool> NameExistsAsync(string name);
    Task AddAsync(Product product);
    Task SaveChangesAsync();
}

public class ProductRepository : IProductRepository
{
    private readonly AppDbContext _db;
    public ProductRepository(AppDbContext db) => _db = db;

    public Task<Product?> GetByIdAsync(int id) =>
        _db.Products.FirstOrDefaultAsync(p => p.Id == id);

    public Task<List<Product>> GetAllAsync() =>
        _db.Products.ToListAsync();

    public Task<bool> NameExistsAsync(string name) =>
        _db.Products.AnyAsync(p => p.Name == name);

    public async Task AddAsync(Product product) =>
        await _db.Products.AddAsync(product);

    public Task SaveChangesAsync() => _db.SaveChangesAsync();
}
🔑 The classic argument FOR a repository

It decouples your business code from EF Core, so the service never sees a DbContext or an IQueryable. In theory that means you could swap the persistence technology, and it gives you one obvious place to centralize complex or shared queries. It also lets you hand your service a fake repository in unit tests.

📚 Analogy — the librarian

A repository is a librarian. You ask "do you have books by this author?" and get a stack back — you never touch the shelving system, the catalog software, or the storage room. If the library switches from paper cards to a database, your request to the librarian doesn't change.

08

Do You Even Need a Repository? ⚖️

Here's where honest engineers disagree — so let's lay out both sides fairly. The crux: EF Core's DbContext already implements the Unit of Work pattern, and each DbSet<T> is already a repository. A hand-written repository over EF Core can therefore be "an abstraction over an abstraction."

Skip the repository (use DbContext directly)Add a repository
EF Core is Repository + Unit of Work alreadyDecouples business code from EF Core entirely
Fewer layers of indirection to read throughOne place for shared/complex queries
Full access to EF features (Include, projections, etc.)Enforces a boundary so Includes don't sprawl everywhere
Test with the EF InMemory or SQLite providerHand services a fake/mock repository in tests
Generic repositories often just re-wrap DbSetUseful with DDD, CQRS, or multi-source data access
⚠️ The "swap the database" myth

The most-cited reason for a repository — "so we can change databases later" — rarely pays off. Switching SQL Server → PostgreSQL is already handled by EF Core providers; switching to a document database like Cosmos DB changes your data model and query patterns so fundamentally that you can't just swap a repository implementation. Don't add the abstraction for a swap that almost never happens.

🔑 The pragmatic middle ground (2026 consensus)

There's no universal winner. Small/CRUD apps: inject DbContext straight into your service — simpler and honest. Domain-heavy or large apps: a thin repository (or the Specification pattern for reusable queries) creates a clean seam worth the cost. The common advice: start simple, add the abstraction only when complexity genuinely demands it — don't over-engineer up front, but don't fear refactoring later.

✅ The one non-negotiable

Whichever you choose, keep the service layer. Even without a repository, injecting DbContext into a ProductService (rather than into the controller) still gives you the testability and separation that today's refactor is about. The service layer is the win; the repository is optional.

C# — the "no repository" option: DbContext in the service
public class ProductService : IProductService
{
    private readonly AppDbContext _db;   // inject the context directly
    public ProductService(AppDbContext db) => _db = db;

    public async Task<ProductDto?> GetByIdAsync(int id)
    {
        var product = await _db.Products.FindAsync(id);
        return product?.ToDto();
    }
}

Full-Stack Walkthrough 🧩

Let's trace a single POST /api/products request all the way down and back up, so you can see each layer do exactly its one job.

Step 1 · 📥 controller
Receive & delegate

The action binds the request DTO, then calls _service.CreateAsync(request). No logic here.

Step 2 · 🧠 service
Apply business rules

Checks the name isn't taken, builds the Product entity, tells the repository to add and save, maps to a DTO.

Step 3 · 🗄️ repository
Talk to EF Core

Runs the AnyAsync check, calls AddAsync, then SaveChangesAsync — the unit of work commits.

Step 4 · 💾 database
Persist the row

EF Core generates the INSERT; the new Id flows back up into the entity.

Step 5 · 📤 back to the controller
Return 201 Created

The DTO bubbles up; the controller wraps it in CreatedAtAction with a Location header.

The wiring, all together 🔩

C# — Program.cs (the full registration)
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();

// Data layer: DbContext (scoped) + repository (scoped)
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseSqlServer(builder.Configuration.GetConnectionString("Default")));
builder.Services.AddScoped<IProductRepository, ProductRepository>();

// Service layer: business logic (scoped)
builder.Services.AddScoped<IProductService, ProductService>();

var app = builder.Build();
app.MapControllers();
app.Run();
🔑 Read the dependency chain

Controller → IProductService → IProductRepository → AppDbContext → Database. Every arrow is an interface (except the context), every hop is one responsibility, and the container assembles the whole chain for each request. Change any link's implementation and the others don't notice.

11

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1What is dependency injection?🌱 Junior

It's a pattern where a class receives the objects it needs from the outside instead of creating them itself. In ASP.NET Core a built-in container constructs those dependencies and passes them into your class (usually through its constructor), so your class just declares what it needs and the framework supplies it.

Q2What are the two steps to use a service with DI?🌱 Junior

First, register it in Program.cs — map the interface to its implementation and choose a lifetime, e.g. AddScoped<IProductService, ProductService>(). Second, request it by listing the interface as a constructor parameter where you need it. The container connects the two automatically.

Q3What is a service layer and why have one?🌱 Junior

It's a set of classes that hold your business logic, sitting between the controllers and the data access. Having one keeps controllers thin, makes the logic reusable and unit-testable, and gives every rule one clear home instead of being scattered across controllers.

Q4Why inject an interface instead of a concrete class?🌱 Junior

Depending on an interface means your class cares only about what a dependency does, not how. That lets you swap the implementation — a real one in production, a fake in tests, a cached one later — without changing the class that uses it. It's the core of loose coupling.

Q5What is constructor injection?🌱 Junior

Declaring your dependencies as parameters on the class's constructor. The DI container fills them in when it creates the object, and you typically store them in readonly fields. It's the default and recommended injection style because it makes dependencies explicit and the class easy to test.

Q6What are the three service lifetimes?🌱 Junior

Transient — a new instance every time it's requested (AddTransient). Scoped — one instance per HTTP request (AddScoped). Singleton — one instance for the whole application (AddSingleton). You pick one when registering the service.

Q7What does a controller do once you have a service layer?🌱 Junior

Very little, on purpose. It receives the HTTP request, calls the relevant service method, and turns the result into an HTTP response (status code plus body). It holds no business rules and no database code — it's a thin translation layer between HTTP and your services.

Q8What is a repository?🌱 Junior

A class that wraps data access behind a simple, collection-like interface — methods like GetByIdAsync or AddAsync — so the rest of the app can work with entities without knowing about SQL or the ORM underneath. It's the bridge between your business code and the database.

Q9Which lifetime should a service that uses DbContext have, and why?🟠 Mid

Scoped. AddDbContext registers the context as scoped (one per request) because a DbContext is a short-lived, non-thread-safe unit of work that tracks changes for a single request. Any service or repository that depends on it should also be scoped so their lifetimes line up.

Q10What is a captive dependency?🟠 Mid

When a longer-lived service captures a shorter-lived one — classically, injecting a scoped DbContext into a singleton. The singleton holds that one instance forever, so it's shared across all requests and threads, breaking change-tracking and causing errors. ASP.NET Core's scope validation throws in development to catch it.

Q11How would you refactor a fat controller?🟠 Mid

Extract the business logic, data access, and mapping into a service class behind an interface (e.g. IProductService), register it in DI, and inject it into the controller. The action then just calls one service method and maps the result to an HTTP response. The logic becomes testable and reusable, and the controller becomes a thin translator.

Q12What belongs in the service layer vs. the controller vs. the data layer?🟠 Mid

Controller: HTTP concerns — model binding, status codes, routing. Service: business rules, orchestration across repositories/services, and entity-to-DTO mapping. Data layer: query mechanics and persistence (EF Core / SQL). Keeping HTTP out of the service and business rules out of the controller is the whole point.

Q13What is the Unit of Work pattern, and where is it in EF Core?🟠 Mid

Unit of Work tracks a set of changes and commits them together in a single transaction. In EF Core the DbContext is the Unit of Work: you make several changes across entities, then one call to SaveChangesAsync() persists them all atomically. That's why layering your own Unit of Work on top is often redundant.

Q14Why is DbContext often described as already being a repository?🟠 Mid

Because each DbSet<T> behaves like a repository for that entity type — you query, add, and remove through it — and the DbContext coordinates saving as a unit of work. So EF Core already gives you the Repository + Unit of Work combination, which is why a hand-written repository over it can be "an abstraction over an abstraction."

Q15When would you use [FromServices] instead of constructor injection?🟠 Mid

When only a single action needs a particular dependency, so putting it in the constructor (and a field) would be noise for every other action. [FromServices] injects it directly into that one action's parameters. For anything used across multiple actions, constructor injection is cleaner and more discoverable.

Q16How do you unit-test a service that depends on a repository?🟠 Mid

Because the service depends on IProductRepository, you pass in a fake or mock implementation in the test — one that returns canned data — and assert the service's behavior without a real database or web server. That testability is a major reason for both the interface and the service layer.

Q17Is the repository pattern worth it over EF Core?🔴 Senior

It depends — and there's no universal answer. EF Core already implements Repository + Unit of Work, so for small or CRUD-heavy apps a repository is often needless indirection; injecting DbContext into services is cleaner. For domain-heavy or large systems, or with DDD/CQRS, a thin repository (or the Specification pattern) creates a useful boundary that stops query logic sprawling. The pragmatic stance: start simple, add the abstraction only when complexity earns it.

Q18Why is "so we can swap databases" a weak argument for a repository?🔴 Senior

Because the swap it protects against rarely happens the way people imagine. Moving between relational databases (SQL Server ↔ PostgreSQL) is already handled by EF Core providers with no repository needed. Moving to a document store like Cosmos DB changes the data model and query patterns so deeply that you can't just swap a repository implementation anyway. The abstraction cost is paid up front for a benefit that seldom materializes.

Q19What's the downside of a generic repository over EF Core?🔴 Senior

A generic Repository<T> tends to just re-wrap DbSet<T> with the same methods, adding a layer without adding value, and it can hide or block useful EF Core features (custom includes, projections, split queries). Many argue that if you use a repository at all, it should be a specific, intention-revealing one per aggregate — not a generic pass-through.

Q20What is the Specification pattern and how does it relate to repositories?🔴 Senior

A specification is a reusable, composable object that encapsulates a query's filtering/including logic, which you then apply to EF Core (directly or through a repository). It gives you query reuse and testability without stuffing dozens of methods into a repository — you add a new specification as requirements grow. It's a popular alternative to fat repositories for keeping query logic clean and separated.

Q21How should a singleton safely use a scoped service like DbContext?🔴 Senior

Not by constructor injection — that captures it. Instead, inject IServiceScopeFactory, create a scope explicitly when needed (using var scope = factory.CreateScope();), resolve the scoped service from that scope, and dispose the scope when done. This gives the singleton a fresh, correctly-scoped DbContext per operation instead of freezing one for the app's lifetime.

Q22How do you keep a service class from becoming a "god object"?🔴 Senior

Watch the constructor: a long list of injected dependencies is a signal the class has too many responsibilities and is violating the Single Responsibility Principle. Split it into smaller, focused services (or use patterns like MediatR/CQRS handlers so each use case is its own class). Small, well-factored services are easier to test and reason about.

Q23If you drop the repository, how do you still test data-dependent logic?🔴 Senior

Test against a real (but lightweight) EF Core provider rather than mocking. The InMemory provider works for pure logic tests but doesn't enforce relational rules like foreign keys; for realistic query behavior, use SQLite in-memory or a containerized database in integration tests. This tests how your LINQ actually translates and how relationships load — something mocking a repository can't verify.

✅ Study-order tip

Nail DI basics first (Q1–Q5) — register/request, interfaces, constructor injection — because everything else builds on it. Then master lifetimes and the captive-dependency trap (Q6, Q9, Q10, Q21), the single most common source of DI bugs and interview follow-ups. Get comfortable articulating the refactor (Q11–Q12), and finish with the repository debate (Q17–Q20) where senior interviewers want to hear a balanced, pragmatic answer — not dogma. If you can walk the request through all four layers in section 09 out loud, you're ready.
