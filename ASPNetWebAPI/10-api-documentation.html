API Documentation 📘

Your API might be flawless, but if a client can't discover what endpoints exist, what to send, and what comes back, it's useless to them. Good docs turn your API into a self-service product. In .NET the machine-readable contract is an OpenAPI document, rendered by tools like Swagger UI or Scalar. Today we generate that document, enrich it with XML comments, and precisely describe every response — the difference between docs that merely exist and docs people actually enjoy using.

📄 OpenAPI 🧭 Swagger UI 📦 Swashbuckle 💬 XML comments 🎯 ProducesResponseType ✨ Scalar
01

Why Document an API? 📖

An API is a contract between your server and everyone who calls it — front-end teams, mobile apps, partners, and increasingly AI agents. Documentation is that contract made visible: which endpoints exist, what parameters and bodies they accept, what shapes and status codes they return, and what each field means. Without it, callers resort to guesswork, trial and error, and messaging you on Slack.

The modern goal is machine-readable documentation. A structured spec doesn't just render nice web pages — it powers interactive test consoles, auto-generates client SDKs, seeds Postman collections, and gives LLM tools the context to call your API correctly.

🍴 Analogy — the menu, again

An API without docs is a restaurant with no menu: you'd have to walk into the kitchen and ask what's cookable. The OpenAPI document is the menu — every dish (endpoint), its ingredients (parameters), and what arrives on the plate (responses). Swagger UI and Scalar are the beautifully printed, laminated version the customer actually holds.

🔑 Two things, often confused

The document (the OpenAPI spec, usually JSON) is the data. The UI (Swagger UI, Scalar, Redoc) is a viewer that renders that data into an interactive page. Today's job is to produce a rich document, then point a UI at it.

02

OpenAPI vs. Swagger — the Vocabulary 🏷️

People use these words interchangeably, but they're distinct — and interviewers love the distinction.

TermWhat it actually is
OpenAPIThe open specification (standard) for describing REST APIs in a language-neutral format. The document itself.
SwaggerA set of tools built around OpenAPI (Swagger UI, Swagger Editor, Codegen). "Swagger" was the original name; the spec was renamed OpenAPI in 2015.
Swagger UIThe specific interactive web page that renders an OpenAPI document and lets you try requests.
SwashbuckleA .NET library that generates OpenAPI docs (and bundles Swagger UI) for ASP.NET Core.
🔑 One-liner to remember

OpenAPI is the specification; Swagger is the tooling around it. An OpenAPI document is a JSON (or YAML) file describing your whole API. In .NET it's typically served at /openapi/v1.json (native) or /swagger/v1/swagger.json (Swashbuckle).

JSON — a tiny slice of an OpenAPI document
{
  "openapi": "3.1.1",
  "info": { "title": "Shop API", "version": "v1" },
  "paths": {
    "/api/products/{id}": {
      "get": {
        "summary": "Gets a product by its id.",
        "responses": {
          "200": { "description": "The product was found." },
          "404": { "description": "No product with that id exists." }
        }
      }
    }
  }
}
✅ Version note

.NET 10's native generator emits OpenAPI 3.1 documents (and .NET 11 will default to 3.2); Swashbuckle historically emits 3.0. The version mostly matters for advanced schema features like nullable handling and JSON Schema alignment — the concepts you learn today carry across all of them.

03

Native OpenAPI (.NET 9+) ⚙️

Here's the big change to internalize: for years the Web API template bundled Swashbuckle. Starting in .NET 9, ASP.NET Core dropped that dependency and shipped its own first-party package, Microsoft.AspNetCore.OpenApi. It's maintained by the ASP.NET Core team, is AoT-friendly, and generates the OpenAPI document natively — no third-party dependency.

C# — Program.cs (the whole native setup)
// dotnet add package Microsoft.AspNetCore.OpenApi
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddOpenApi();          // register the generator

var app = builder.Build();

if (app.Environment.IsDevelopment())
    app.MapOpenApi();               // serves /openapi/v1.json

app.MapControllers();
app.Run();
⚠️ The catch: the document, but no UI

The native package generates only the OpenAPI document — it does not include an interactive UI. Navigate to /openapi/v1.json and you'll see raw JSON, not a pretty page. To get a clickable interface, you add a renderer separately (section 05). This surprises people migrating from .NET 8, where the template gave you Swagger UI out of the box.

🏛️

First-party

Maintained by the ASP.NET Core team and versioned with the framework, so it tracks new features.

🪶

AoT-friendly

Uses source generation and System.Text.Json schema support, so it works with trimming and Native AOT.

🧩

Transformers

Customize the output via document, operation, and schema transformers instead of Swashbuckle filters.

🔑 Build-time docs, too

Add the Microsoft.Extensions.ApiDescription.Server package to generate the OpenAPI JSON at build time — handy for committing the spec to source control, spec-based integration tests, or serving it statically.

04

Swashbuckle — Still Here, Now Opt-In 📦

Swashbuckle was the default for years and remains a perfectly good, actively-published community package — it's just no longer bundled. You'd choose it when you have existing Swashbuckle configuration, need a specific filter/extension it provides, or want its bundled Swagger UI in one package. On .NET 10 you use Swashbuckle v10.x.

C# — the classic Swashbuckle setup
// dotnet add package Swashbuckle.AspNetCore
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();  // needed for minimal APIs
builder.Services.AddSwaggerGen(options =>
{
    options.SwaggerDoc("v1", new OpenApiInfo
    {
        Title = "Shop API",
        Version = "v1",
        Description = "Manages products in the shop."
    });
});

var app = builder.Build();
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();      // serves /swagger/v1/swagger.json
    app.UseSwaggerUI();    // serves the Swagger UI at /swagger
}

⚠️ Don't mix the two blindly

Adding both Microsoft.AspNetCore.OpenApi and Swashbuckle in one project can cause namespace/version conflicts (the Microsoft.OpenApi types moved between versions). Pick one generator per project. A common migration path is: keep Swashbuckle's Swagger UI, but generate the document with the native package.

🔑 The 2026 default recommendation

For a new project, prefer native OpenAPI for generation (it's first-party and future-proof) plus a modern UI like Scalar. Reach for Swashbuckle when you have an existing investment in its filters/extensions, or specifically want its all-in-one Swagger UI package. Both are valid; the choice is about your situation, not one being "wrong."

05

Adding an Interactive UI 🧭

With native OpenAPI you get a JSON document but no viewer. Point one of these UIs at it. All three read the same OpenAPI document; they differ only in looks and features.

C# — Scalar (the popular modern choice)
// dotnet add package Scalar.AspNetCore
if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();                 // /openapi/v1.json
    app.MapScalarApiReference();       // interactive UI at /scalar/v1
}
C# — or Swagger UI against the native document
// dotnet add package Swashbuckle.AspNetCore.SwaggerUI
if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();
    app.UseSwaggerUI(o =>
        o.SwaggerEndpoint("/openapi/v1.json", "Shop API v1"));
}
🧭

Swagger UI

The classic, universally recognized "try it out" console. Great familiarity; served at /swagger.

✨

Scalar

Modern, fast, readable by default, with a polished request builder. The popular .NET 9/10 pick; at /scalar/v1.

📗

Redoc

Clean, read-only, three-panel reference — nice for polished public docs you show customers.

⚠️ Gate the UI to non-production

Notice the if (IsDevelopment()) around the UI mapping. Exposing an interactive API explorer publicly can reveal internal endpoints and invite probing. Many teams keep the UI to development/staging, and if the doc must be public, serve it read-only and protected. Generating the document is cheap; exposing a live "try it" console to the world is a choice, not a default.

06

XML Comments — Prose From Your Code 💬

A generated document knows your shapes (types, status codes) but not your intent. XML documentation comments — the /// triple-slash comments C# developers already write — fill that gap with human-readable summaries, parameter descriptions, and examples that flow straight into the OpenAPI document.

🔑 This changed in .NET 10 — know it cold

Historically only Swashbuckle read XML comments; the native package ignored them. As of .NET 10, Microsoft.AspNetCore.OpenApi reads XML doc comments natively via a build-time source generator. It intercepts your AddOpenApi() call and injects transformers that pull <summary>, <remarks>, <param>, <returns>, and <example> into the document — with no runtime reflection cost.

Step 1 — turn on the XML doc file 🔛

For either generator, you must enable XML documentation output in the project file. This one property is the switch everyone forgets:

XML — .csproj
<PropertyGroup>
  <TargetFramework>net10.0</TargetFramework>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
</PropertyGroup>

With native OpenAPI (.NET 10) that's all you need — the source generator finds the comments in your assembly (and referenced projects that also set the property) automatically. With Swashbuckle you additionally wire the file in:

C# — Swashbuckle only: include the XML file
builder.Services.AddSwaggerGen(options =>
{
    var xmlFile = $"{Assembly.GetExecutingAssembly().GetName().Name}.xml";
    options.IncludeXmlComments(Path.Combine(AppContext.BaseDirectory, xmlFile));
});

Step 2 — write the comments ✍️

C# — XML doc comments on an action
/// <summary>Gets a single product by its id.</summary>
/// <remarks>Returns 404 when no product with the id exists.</remarks>
/// <param name="id" example="42">The product's unique identifier.</param>
/// <returns>The matching product.</returns>
[HttpGet("{id:int}")]
public async Task<ActionResult<ProductDto>> GetById(int id) { /* ... */ }
⚠️ Minimal API gotcha — comments on lambdas are lost

The C# compiler doesn't capture XML doc comments placed on lambda expressions. So for a Minimal API endpoint, XML comments only work if you define the handler as a named method, put the comments on that method, and reference it from MapGet(...). Comments written directly above an inline () => ... lambda simply won't appear.

✅ Document the model, not just the endpoint

Put <summary> comments on your DTO properties too — they become field descriptions in the schema, so a client sees what each JSON field means. You can also add <example> values. Well-described schemas are what separate "technically documented" from "actually pleasant."

07

Documenting Responses Precisely 🎯

The most valuable — and most skipped — part of API docs is describing every response an endpoint can return, with its status code and body type. Recall from Day 6 that a bare IActionResult hides the success type; you spell out the full picture with the [ProducesResponseType] attribute.

C# — declaring all responses
[HttpGet("{id:int}")]
[ProducesResponseType<ProductDto>(StatusCodes.Status200OK)]
[ProducesResponseType(StatusCodes.Status404NotFound)]
public async Task<IActionResult> GetById(int id) { /* ... */ }

The generic form [ProducesResponseType<T>(code)] declares both the status code and the body type in one line (the older form is [ProducesResponseType(typeof(ProductDto), StatusCodes.Status200OK)]). List each outcome the endpoint can produce so the document is complete.

🔑 TypedResults documents itself

From Day 6: in Minimal APIs, returning TypedResults (e.g. via a Results<Ok<ProductDto>, NotFound> union) makes the framework infer the response metadata automatically — no .Produces<T>() needed. And ActionResult<T> in controllers lets OpenAPI infer the 200 body from T. Choosing self-documenting return types means less to annotate by hand.

✅ Standardize the error shape

Document your failures as ProblemDetails (Day 6). For validation failures the framework already produces ValidationProblemDetails (Day 4). Declaring [ProducesResponseType<ProblemDetails>(StatusCodes.Status404NotFound)] tells clients not just that a 404 can happen but exactly what the error body looks like — so they can code against it.

⚠️ Undocumented responses = broken trust

If your endpoint can return a 409 or 422 but the doc only lists 200, a client's generated SDK won't handle those cases and will break in production. The rule: the documented responses must match what the code actually returns — treat drift between them as a bug.

08

A Fully Annotated Endpoint 🧩

Today's "Do," assembled: an endpoint whose documentation is complete, precise, and readable — XML prose plus every response declared.

C# — the annotated action
/// <summary>Creates a new product.</summary>
/// <remarks>
/// The server assigns the id. On success the response includes a
/// Location header pointing to the new resource.
/// </remarks>
/// <param name="request">The product's name and price.</param>
/// <response code="201">The product was created.</response>
/// <response code="400">The request failed validation.</response>
/// <response code="409">A product with that name already exists.</response>
[HttpPost]
[Consumes(MediaTypeNames.Application.Json)]
[ProducesResponseType<ProductDto>(StatusCodes.Status201Created)]
[ProducesResponseType<ValidationProblemDetails>(StatusCodes.Status400BadRequest)]
[ProducesResponseType<ProblemDetails>(StatusCodes.Status409Conflict)]
public async Task<ActionResult<ProductDto>> Create(CreateProductRequest request)
{
    var dto = await _service.CreateAsync(request);
    return CreatedAtAction(nameof(GetById), new { id = dto.Id }, dto);
}

And the DTO, documented so its fields carry meaning and examples into the schema:

C# — a documented request DTO
/// <summary>The data needed to create a product.</summary>
public class CreateProductRequest
{
    /// <summary>Display name, 2–100 characters.</summary>
    /// <example>Wireless Mouse</example>
    [Required, StringLength(100, MinimumLength = 2)]
    public string Name { get; set; } = null!;

    /// <summary>Price in USD; must be non-negative.</summary>
    /// <example>19.99</example>
    [Range(0, 999.99)]
    public decimal Price { get; set; }
}
✅ Bonus — data annotations enrich the schema

Notice the validation attributes from Day 4 ([Required], [StringLength], [Range]) also feed the OpenAPI schema — they show up as required, maxLength, minimum/maximum constraints. So good validation and good docs reinforce each other for free. This one endpoint now tells a client exactly what to send, what constraints apply, and every response it might get back.

09

The Documentation Pipeline 🔄

Here's how your code becomes a browsable, interactive page — the journey from /// comment to rendered UI.

Step 1 · ✍️ you annotate
Code, attributes, XML comments

Return types, [ProducesResponseType], data annotations, and /// summaries describe intent and shape.

Step 2 · 🏭 build time
Source generator gathers XML

With GenerateDocumentationFile on, the generator caches your XML comments and injects transformers into AddOpenApi.

Step 3 · ⚙️ generation
Build the OpenAPI document

ASP.NET Core inspects endpoints, schemas, and metadata to produce the OpenAPI JSON, applying any transformers.

Step 4 · 🌐 serve
Expose the document

MapOpenApi() serves it at /openapi/v1.json — the machine-readable contract, ready to consume.

Step 5 · 🧭 render
UI reads & displays it

Scalar / Swagger UI / Redoc fetch that JSON and render the interactive, human-friendly reference.

🔑 One source of truth

The beauty of this pipeline: the docs are generated from the code itself, so they can't drift as far as hand-written docs do. Your job is to make the code expressive — accurate return types, complete [ProducesResponseType], and clear XML comments — and the document (and every tool that reads it) stays in sync automatically.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1What is the difference between OpenAPI and Swagger?🌱 Junior

OpenAPI is the specification — the open standard for describing a REST API in a machine-readable format. Swagger is the set of tools built around that spec (Swagger UI, Editor, Codegen). "Swagger" was the original name before the spec was renamed OpenAPI. In short: OpenAPI is the document; Swagger is the tooling.

Q2What is an OpenAPI document and where is it served?🌱 Junior

It's a JSON (or YAML) file describing every endpoint, its parameters, request/response shapes, and status codes. In .NET with the native package it's served at /openapi/v1.json; with Swashbuckle it's at /swagger/v1/swagger.json. Tools and UIs read this document to render docs, generate clients, and test the API.

Q3What is Swashbuckle?🌱 Junior

A .NET library that generates an OpenAPI document from your ASP.NET Core app and bundles Swagger UI to display it. For years it was included in the Web API template; since .NET 9 it's an opt-in community package rather than the default. You set it up with AddSwaggerGen, UseSwagger, and UseSwaggerUI.

Q4How do you add API docs in a new .NET 9 or 10 project?🌱 Junior

Add the Microsoft.AspNetCore.OpenApi package, call builder.Services.AddOpenApi(), and map it with app.MapOpenApi(). That serves the OpenAPI JSON. Since no UI is included, add one — e.g. Scalar via MapScalarApiReference() — to get an interactive page.

Q5What are XML documentation comments used for here?🌱 Junior

The /// comments on your methods, classes, and properties provide human-readable prose — summaries, parameter descriptions, examples — that gets pulled into the OpenAPI document. They turn a doc that only knows shapes and status codes into one that also explains intent and meaning.

Q6What single project setting is needed to use XML comments?🌱 Junior

<GenerateDocumentationFile>true</GenerateDocumentationFile> in the .csproj. It tells the compiler to emit the XML documentation file that both the native generator and Swashbuckle read. It's the step people most often forget, and without it the comments silently don't appear.

Q7How do you tell the docs what type a 200 response returns?🌱 Junior

Use the [ProducesResponseType<T>(StatusCodes.Status200OK)] attribute on the action, where T is the body type. Or return ActionResult<T> and the framework infers the 200 body from T automatically. For non-body codes like 404 you use [ProducesResponseType(StatusCodes.Status404NotFound)].

Q8What is Swagger UI?🌱 Junior

An interactive web page that reads an OpenAPI document and renders it as browsable docs with a "try it out" console, so you can send real requests to each endpoint from the browser. It's just a viewer for the document — alternatives that do the same job include Scalar and Redoc.

Q9What changed about API docs in .NET 9?🟠 Mid

ASP.NET Core dropped the Swashbuckle dependency from the Web API template and introduced its own first-party generator, Microsoft.AspNetCore.OpenApi (AddOpenApi/MapOpenApi). It generates the OpenAPI document but includes no UI, so you now add a renderer like Scalar or Swagger UI separately. Swashbuckle still exists as an opt-in community package.

Q10Does the native OpenAPI package read XML comments?🟠 Mid

Not originally — in .NET 9 it ignored them. As of .NET 10 it does: a build-time source generator discovers your XML comments (when GenerateDocumentationFile is on), intercepts AddOpenApi(), and injects transformers that add the summaries, descriptions, and examples to the document — with no runtime reflection cost.

Q11Why don't XML comments work on Minimal API lambdas?🟠 Mid

The C# compiler doesn't emit XML doc comments for lambda expressions, so comments above an inline () => ... handler are never captured. To document a Minimal API endpoint, define the handler as a named method, put the XML comments on that method, and reference it from MapGet/MapPost. Then the comments flow into the document.

Q12How do you document all the responses an endpoint can return?🟠 Mid

Add one [ProducesResponseType] per outcome — e.g. [ProducesResponseType<ProductDto>(201)], [ProducesResponseType<ValidationProblemDetails>(400)], [ProducesResponseType<ProblemDetails>(409)]. In Minimal APIs the equivalents are .Produces<T>(code) and .ProducesProblem(code). The documented set must match what the code actually returns.

Q13How do return types like ActionResult<T> and TypedResults help documentation?🟠 Mid

They're self-documenting. ActionResult<T> lets OpenAPI infer the success body from T without a Type on the attribute. In Minimal APIs, returning concrete TypedResults (e.g. via a Results<Ok<T>, NotFound> union) makes the framework infer the response metadata automatically. Choosing these types means less manual annotation.

Q14How do data annotations affect the generated schema?🟠 Mid

Validation attributes double as schema documentation: [Required] marks a field required, [StringLength] becomes maxLength/minLength, [Range] becomes minimum/maximum. So the same annotations that enforce validation (Day 4) also tell clients the constraints on each field — good validation and good docs reinforce each other.

Q15Can you use both Swashbuckle and the native package together?🟠 Mid

You should pick one generator per project — mixing them can cause version/namespace conflicts (the Microsoft.OpenApi types moved between versions). A supported hybrid is to generate the document with the native package and use Swashbuckle's Swagger UI package purely as the viewer, pointing it at /openapi/v1.json.

Q16Should Swagger UI be exposed in production?🟠 Mid

Usually not openly. An interactive explorer reveals your internal endpoints and invites probing, so many teams gate the UI behind IsDevelopment() or authentication, or restrict it to staging. If a public spec is genuinely needed, serve a read-only rendering (e.g. Redoc) and consider protecting the document endpoint. Generating the document is fine; exposing a live "try it" console publicly is a deliberate choice.

Q17How do you customize the generated document in the native package?🔴 Senior

Through transformers registered on AddOpenApi: document transformers (modify the whole doc — e.g. add contact info, security schemes), operation transformers (per-endpoint tweaks), and schema transformers (adjust type schemas). This is the native equivalent of Swashbuckle's filters (IOperationFilter, ISchemaFilter, IDocumentFilter), and it's how you add things the framework can't infer, like global auth requirements.

Q18Why generate the OpenAPI document at build time?🔴 Senior

Adding Microsoft.Extensions.ApiDescription.Server emits the JSON during the build rather than at runtime. That lets you commit the spec to source control (so contract changes show up in code review as diffs), run spec-based integration/contract tests in CI, generate client SDKs in the pipeline, and serve the doc statically — all without booting the app to hit a live endpoint.

Q19How would you document a versioned API?🔴 Senior

Produce a separate OpenAPI document per version (the native package supports multiple documents from one app, and API versioning integrates with the API Explorer). Each version's endpoints map to its own doc name (v1, v2), served at /openapi/v1.json, /openapi/v2.json, and the UI exposes a version selector. This keeps each version's contract stable and independently documented for its consumers.

Q20How do you keep docs from drifting out of sync with the code?🔴 Senior

Generate the docs from the code (attributes, return types, XML comments) rather than maintaining them by hand, so most drift is impossible. Then enforce the rest in CI: generate the spec at build time and diff it against the committed version to catch unintended contract changes, and add contract tests that assert each endpoint returns the documented status codes. Treat a mismatch between documented and actual responses as a build-failing bug.

Q21How do you document authentication in the OpenAPI document?🔴 Senior

Define a security scheme (e.g. JWT bearer, OAuth2) and apply it to operations. In the native package you add it via a document transformer that sets the security scheme and requirement; in Swashbuckle you use AddSecurityDefinition + AddSecurityRequirement. This makes the UI show an "Authorize" button and lets generated clients know which endpoints need credentials and how to supply them.

Q22Why is a good OpenAPI document valuable beyond human docs?🔴 Senior

Because it's machine-readable, it drives automation: generating strongly-typed client SDKs (NSwag, Kiota), scaffolding server stubs, seeding Postman/Bruno collections, contract and fuzz testing, API gateways, and increasingly giving LLM/agent tools the structured context to call your API correctly. The richer and more accurate the document — precise response types, descriptions, examples — the better every one of those downstream tools works.

Q23When would you choose native OpenAPI vs. Swashbuckle vs. NSwag today?🔴 Senior

Native OpenAPI: the default for new projects — first-party, AoT-friendly, OpenAPI 3.1, tracks the framework. Swashbuckle: when you have significant existing configuration (custom filters, security setup) or specifically want its bundled Swagger UI in one package. NSwag: when your workflow centers on generating typed C#/TypeScript clients from the spec, where it remains the strongest tool. A common combo is native generation + Scalar for docs, and NSwag or Kiota for client generation.

✅ Study-order tip

Lock down the vocabulary first (Q1–Q3) — the OpenAPI-vs-Swagger distinction is the single most common opener. Then the .NET 9/10 shift (Q9–Q10) since it's current and interviewers probe whether you're up to date. Practice annotating — XML comments and [ProducesResponseType] (Q5–Q7, Q11–Q13) — because that's the day's "Do." Finish with the senior automation angle (Q17–Q23): transformers, build-time generation, versioning, security, and why the document powers SDKs and tools.
