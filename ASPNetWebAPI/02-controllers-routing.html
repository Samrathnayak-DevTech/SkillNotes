Day 02 · ASP.NET Core Web API

🎛️ Controllers & Routing: Directing the Traffic

Yesterday you learned the rules of the web — verbs, URLs, status codes. Today you learn how ASP.NET Core connects an incoming request to your code. A controller groups related endpoints; action methods are the endpoints; and routing is the address system that decides which method answers which URL. You'll build a real controller with GET routes for the shop's products.

🎛️ Controllers ⚡ Action methods 🏷️ Attribute routing 🧩 Route templates 🚧 Constraints

01🎛️ What Is a Controller?

A controller is a class that groups together the endpoints for one resource. All the operations on products — list them, fetch one, filter them — live together in a ProductsController. It's the organizing unit of a Web API: one controller per resource, each holding the actions for that resource.

🏢 Analogy

Think of your API as an office building, and routing as the front-desk receptionist. A visitor (request) arrives asking for "Products, please." The receptionist checks the directory and sends them to the right department (the controller) and the right person in it (the action method) who handles that exact request. The controller is the department; each action is a specialist at a desk; routing is the receptionist reading the directory to route each visitor.

In ASP.NET Core, a Web API controller is a class that inherits from ControllerBase and (by convention) ends its name in Controller:

C# · the smallest controller
using Microsoft.AspNetCore.Mvc;

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    [HttpGet]                        // GET /api/products
    public IActionResult GetAll() => Ok("all products");
}
🔑 ControllerBase, Not Controller

For a Web API you inherit from ControllerBase, which gives you everything an API needs — model binding, routing, and helper methods like Ok(), NotFound(), BadRequest(), CreatedAtAction() that return proper HTTP responses. Its cousin Controller adds view support (for returning HTML pages via Razor), which an API doesn't need. APIs return data, so ControllerBase is the right base class.

✅ Controllers Are Discovered Automatically

You don't register each controller by hand. In Program.cs you call builder.Services.AddControllers() and app.MapControllers() once, and ASP.NET Core scans your assembly at startup, finds every controller class, reads its route attributes, and wires up all the endpoints. Add a new controller and it just works — no central list to maintain.

The chain to hold in mind: a request's URL + verb are matched by routing to a specific action method inside a controller; the URL's variable parts (route parameters) are handed to the method as arguments; and constraints filter which URLs even qualify. Everything today is about making that match precise and predictable.

03⚡ Action Methods

An action method is a public method on a controller that handles one endpoint. It's tagged with an HTTP verb attribute ([HttpGet], [HttpPost], …) so routing knows which verb it answers, and it returns a result that becomes the HTTP response.

C# · two actions, two verbs
[HttpGet]                     // GET /api/products
public IActionResult GetAll()
{
    return Ok(_products);       // 200 OK + the list as JSON
}

[HttpGet("{id}")]              // GET /api/products/42
public IActionResult GetById(int id)
{
    var product = _products.FirstOrDefault(p => p.Id == id);
    return product is null
        ? NotFound()          // 404 if it doesn't exist
        : Ok(product);         // 200 + the product
}

What an action returns 🎁

Those helper methods (Ok(), NotFound()) come from ControllerBase and produce the right status code plus body. You have three common return-type styles:

Return typeUse forExample
IActionResultWhen you return different results (Ok, NotFound…) from one actionreturn NotFound();
ActionResult<T>Same, but documents the success type T for tooling/SwaggerActionResult<Product>
T (the data itself)Simple cases — the framework wraps it in 200 OKreturn product;
🔑 The Helper Methods Map to Status Codes

This is where Day 1's status codes become code. Ok() → 200, CreatedAtAction() → 201, NoContent() → 204, BadRequest() → 400, NotFound() → 404, Conflict() → 409. You don't hand-write status numbers; you call the named helper and it produces the correct code and shape. Choosing the right helper is choosing the right status code.

✅ One Action per Verb + Route

Keep each action focused: one HTTP verb, one clear route, one job. GetAll answers GET /api/products; GetById answers GET /api/products/{id}. Resist cramming multiple behaviors into one method with if branches on the verb — separate actions are clearer, easier to route, and show up cleanly in your OpenAPI docs (Day 1).

04✨ The [ApiController] Attribute

You'll see [ApiController] on every Web API controller. It's a small attribute that switches on a bundle of API-friendly behaviors so you write less boilerplate. It's opt-in, and for APIs you always want it.

✅Automatic 400s

If model validation fails (a required field missing, a bad type), it returns 400 Bad Request automatically — you don't write the check.

🔌Binding inference

It infers where each parameter comes from — route, query, or body — so you rarely need explicit [FromBody]/[FromQuery] attributes.

📋ProblemDetails errors

Error responses follow the standard ProblemDetails (RFC 9457) JSON shape, so clients parse errors consistently.

📎Attribute routing required

It requires attribute routing (next section) — which is exactly what you want for a clean, explicit API.

Binding source inference 🔌

Thanks to [ApiController], the framework guesses where each argument comes from, so typical actions need no [From…] attributes:

Parameter kindInferred sourceMeaning
Matches a route token ({id})[FromRoute]Comes from the URL path
Simple type, not in route[FromQuery]Comes from the query string
Complex type (a class/record)[FromBody]Comes from the request body (JSON)
✅ Be Explicit When It Helps

Inference removes noise from ordinary CRUD actions, but add the explicit attribute when a parameter's name doesn't match its route token, when you need a [FromHeader] value, or simply when you want the binding intent obvious to the next reader. Clarity beats cleverness — the inference is a convenience, not a rule you must hide behind.

🔑 [ApiController] = Less Boilerplate, More Convention

Without it, you'd manually check ModelState.IsValid and return 400s in every action, and annotate every parameter's source. With it, those become automatic. It encodes the sensible defaults for HTTP APIs so your controllers stay focused on logic, not plumbing — which is why it's on essentially every ASP.NET Core Web API controller.

05🏷️ Attribute Routing

Attribute routing means you declare each endpoint's URL right on the code, using attributes like [Route(...)] and [HttpGet(...)]. The route lives next to the method it belongs to — read the code, know the URL. It's the standard, preferred approach for Web APIs.

C# · controller route + action routes combine
[ApiController]
[Route("api/[controller]")]      // prefix for every action → "api/products"
public class ProductsController : ControllerBase
{
    [HttpGet]                    // GET  api/products
    public IActionResult GetAll() => ...;

    [HttpGet("{id}")]             // GET  api/products/42
    public IActionResult GetById(int id) => ...;

    [HttpGet("featured")]         // GET  api/products/featured
    public IActionResult GetFeatured() => ...;
}

The [Route] on the controller sets a prefix; the template on each action is appended to it. So the controller-level api/[controller] plus an action's {id} yields api/products/{id}. They combine.

🔑 The [controller] Token

[controller] in a route template is a placeholder that's replaced with the controller's name minus the "Controller" suffix — ProductsController becomes products. So [Route("api/[controller]")] produces api/products automatically. Rename the class and the route follows. (There's also [action], but hard-coding action names in URLs is discouraged for REST APIs — your URLs should be resource nouns, per Day 1.)

⚠️ Attribute Routing vs Conventional Routing

There's an older style, conventional routing, where you define one global pattern like {controller}/{action}/{id?} in Program.cs. It's still supported and common in MVC apps that render pages, but for Web APIs the standard is attribute routing — it's explicit, per-endpoint, RESTful, and required by [ApiController]. Know conventional routing exists; reach for attribute routing.

✅ Relative vs Absolute

An action route without a leading slash is relative — it's appended to the controller's prefix ([HttpGet("featured")] → api/products/featured). A leading / or ~/ makes it absolute, ignoring the controller prefix entirely. Relative is the norm; reach for absolute only when an endpoint genuinely belongs outside its controller's path.

06🧩 Route Templates & Parameters

A route template is the pattern string inside the attribute. It mixes fixed literal segments with route parameters in curly braces. When a URL matches, each {parameter} is captured and handed to your action method's argument of the same name.

C# · templates that mix literals & parameters
[HttpGet("{id}")]                          // api/products/42       → id = 42
public IActionResult GetById(int id) => ...;

[HttpGet("{category}/{id}")]               // api/products/audio/7  → category, id
public IActionResult GetInCategory(string category, int id) => ...;

[HttpGet("{id}/reviews")]                  // api/products/42/reviews
public IActionResult GetReviews(int id) => ...;

Optional parameters & defaults

Add ? to make a route parameter optional, or =value to give it a default:

C# · optional & default
[HttpGet("{id?}")]        // id is optional — matches api/products AND api/products/42
public IActionResult Get(int? id) => ...;

[HttpGet("page/{page=1}")]   // page defaults to 1 if omitted
public IActionResult GetPage(int page) => ...;
🔑 Route Parameter vs Query String

A route parameter ({id}) is part of the path and usually identifies which resource — api/products/42. A query string value (?sort=price&page=2) refines how you want a collection and is bound from [FromQuery] arguments. This is Day 1's path-vs-query distinction, now in code: identity → route parameter; filtering/sorting/paging → query string.

C# · query string binding
[HttpGet]   // GET api/products?category=audio&sort=price&page=2
public IActionResult Search(
    [FromQuery] string? category,
    [FromQuery] string? sort,
    [FromQuery] int page = 1) => ...;

07🚧 Route Constraints

A route constraint restricts what a route parameter will match, written as {parameter:constraint}. {id:int} only matches if the segment is an integer. Constraints do two valuable jobs: they disambiguate similar routes, and they reject invalid URLs early — before your action even runs.

C# · constraints in action
[HttpGet("{id:int}")]                // only matches api/products/42
public IActionResult GetById(int id) => ...;

[HttpGet("{slug}")]                 // matches api/products/wireless-mouse
public IActionResult GetBySlug(string slug) => ...;
⚠️ The Ambiguity Problem Constraints Solve

Imagine two GET actions both mapped to {id} — one taking an int, one a string. A request to api/products/42 matches both templates, and ASP.NET Core throws an AmbiguousMatchException — it can't decide. Add {id:int} to the numeric one and {slug} to the other, and the routing engine can tell them apart: 42 only fits the int route, wireless-mouse only fits the slug route. Constraints resolve the tie.

Common built-in constraints

ConstraintMatchesExample
:intAn integer{id:int}
:guidA GUID{id:guid}
:alphaAlphabetic characters{name:alpha}
:bool, :datetimeA bool / date{active:bool}
:min(n) / :max(n)Number ≥ / ≤ n{id:int:min(1)}
:range(a,b)Number within a range{page:int:range(1,1000)}
:minlength(n) / :maxlength(n)String length limits{code:minlength(3)}
:regex(...)A custom pattern{sku:regex(^PRD\\d+$)}

You can chain constraints with more colons: {id:int:min(1)} means "an integer, and at least 1." A request like api/products/0 or api/products/abc simply won't match — the client gets a 404 without your code running.

🔑 Constraints Run Before Your Action

A constraint is checked during routing, before model binding or your action executes. If the URL fails the constraint, the request never reaches your method — it's rejected at the door. That's good for both performance (no wasted work on invalid input) and clarity (your action can assume id is a valid int). Constraints are a routing-level filter, not a substitute for real validation (Day 14) of request bodies — but for URL segments they're the first line of defense.

⚠️ Constraints Validate Format, Not Existence

{id:int} guarantees id is a number — it does not guarantee product 42 exists. That's still your action's job: look it up and return NotFound() if it's missing. Constraints filter shape; your code checks reality.

08🧭 How a Request Finds Its Action

Put the pieces in motion. A request arrives; here's how ASP.NET Core routes it to exactly one action method:

📥
1. Request arrivese.g. GET /api/products/42 — a verb and a URL path.
🔎
2. Match candidate routesRouting compares the path against every action's template, keeping those whose literals and parameters fit.
🚧
3. Apply constraints & verbDrop candidates that fail a constraint ({id:int}) or whose HTTP verb doesn't match (GET vs POST).
🎯
4. Pick the most specificIf several still match, the most specific template wins (a literal or constrained segment beats a bare parameter). A true tie throws AmbiguousMatchException.
🔌
5. Bind & invokeRoute values (id = 42) and query/body are bound to the method's parameters, then your action runs and returns a result.
🔑 Specificity Wins

When templates compete, routing prefers the more specific one. featured (a literal) beats {id} (a parameter), so api/products/featured hits GetFeatured(), not GetById(). A constrained parameter ({id:int}) beats a bare one ({slug}). This precedence is why you rarely need to think about ordering — the engine picks the tightest fit automatically. Only genuine, un-distinguishable ties error out.

09🧱 A Full Products Controller

Here's a complete, runnable controller with several GET routes for the shop's products — using an in-memory list so it works without a database (that comes in Days 7–9). Read it as a summary of everything today:

C# · ProductsController.cs
using Microsoft.AspNetCore.Mvc;

[ApiController]
[Route("api/[controller]")]                         // → api/products
public class ProductsController : ControllerBase
{
    // Stand-in data until we add EF Core (Days 7–9)
    private static readonly List<Product> _products =
    [
        new(1, "Mechanical Keyboard", "audio", 4999m),
        new(2, "Wireless Mouse", "audio", 1499m),
        new(3, "Desk Lamp", "home", 899m),
    ];

    // GET api/products            → the whole collection
    [HttpGet]
    public ActionResult<IEnumerable<Product>> GetAll(
        [FromQuery] string? category)               // optional ?category=audio
    {
        var result = category is null
            ? _products
            : _products.Where(p => p.Category == category).ToList();
        return Ok(result);                            // 200
    }

    // GET api/products/42         → one product by id (int only)
    [HttpGet("{id:int}")]
    public ActionResult<Product> GetById(int id)
    {
        var product = _products.FirstOrDefault(p => p.Id == id);
        return product is null ? NotFound() : Ok(product);   // 404 or 200
    }

    // GET api/products/featured   → literal beats {id}, so this wins for "featured"
    [HttpGet("featured")]
    public ActionResult<IEnumerable<Product>> GetFeatured() =>
        Ok(_products.Take(2));

    // GET api/products/42/reviews → nested sub-resource
    [HttpGet("{id:int}/reviews")]
    public ActionResult<IEnumerable<string>> GetReviews(int id) =>
        Ok(new[] { $"Great product #{id}!", "Would buy again." });
}

public record Product(int Id, string Name, string Category, decimal Price);

And the one-time wiring in Program.cs so controllers are found and served:

C# · Program.cs — enable controllers
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllers();          // register controller services
builder.Services.AddOpenApi();             // OpenAPI doc (Day 1)

var app = builder.Build();
if (app.Environment.IsDevelopment()) app.MapOpenApi();
app.MapControllers();                        // wire up every controller's routes
app.Run();
✅ This Maps Straight to Day 1

Every route here is a noun (products, not getProducts), the verb is GET via [HttpGet], filtering rides in the query string (?category=), identity rides in the path ({id:int}), and results carry proper status codes (200 via Ok, 404 via NotFound). Day 1 gave you the rules; today wires them to real methods.

10🛠️ Hands-On: A Controller with GET Routes

Today's Do: create a controller with GET routes for a resource. Use the MyShop.Api project from Day 1 (or a fresh dotnet new webapi --use-controllers). The path:

  1. 🎛️ Create Controllers/ProductsController.cs: a class inheriting ControllerBase, decorated with [ApiController] and [Route("api/[controller]")].
  2. 🗃️ Add a Product record and a static in-memory list of a few products so you have data without a database yet.
  3. 📋 Add GetAll — [HttpGet] returning Ok(_products) — and confirm GET /api/products returns the JSON array.
  4. 🔎 Add GetById — [HttpGet("{id:int}")] — returning the product or NotFound(). Test both a real id and a missing one; confirm 200 vs 404.
  5. 🚧 Prove the constraint: request /api/products/abc and confirm you get a 404 (the :int constraint rejected it before your code ran).
  6. ⭐ Add a literal route [HttpGet("featured")] and confirm /api/products/featured hits it — not GetById — because a literal beats a parameter.
  7. 🔗 Add a nested route [HttpGet("{id:int}/reviews")] returning some placeholder reviews, and hit /api/products/1/reviews.
  8. 🎚️ Add query filtering: a [FromQuery] string? category parameter on GetAll, and test /api/products?category=audio.
  9. 📖 Open your Swagger/Scalar UI (Day 1) and confirm all your GET routes appear and can be tried in the browser.
✅ Self-Check

From your running controller: (1) Why does /api/products/featured reach GetFeatured and not GetById? (2) What status code comes back for /api/products/999 (a valid-but-missing id) versus /api/products/abc (a non-int) — and why are they different code paths? (3) Where did category come from — route or query — and how did the framework know? (4) What does [controller] in your route expand to, and what happens to the URL if you rename the class?

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is a controller in ASP.NET Core Web API?🌱 JUNIOR

A controller is a class that groups together the endpoints (action methods) for one resource — for example, a ProductsController holds all the operations on products: list them, get one, and so on. For a Web API it inherits from ControllerBase and by convention its name ends in "Controller." ASP.NET Core discovers controllers automatically at startup (once you call AddControllers/MapControllers), reads their route attributes, and wires up the endpoints. It's the main organizing unit of an API — roughly one controller per resource.

Q2What is an action method?🌱 JUNIOR

An action method is a public method on a controller that handles one endpoint. It's tagged with an HTTP verb attribute like [HttpGet] or [HttpPost] so routing knows which verb it answers, and it returns a result (often IActionResult or ActionResult<T>) that becomes the HTTP response. For example, a GetById(int id) method decorated with [HttpGet("{id}")] handles requests like GET /api/products/42 and returns that product (or a 404 if it's missing).

Q3What's the difference between ControllerBase and Controller?🌱 JUNIOR

ControllerBase is the base class for Web API controllers — it provides everything an API needs: routing, model binding, and helper methods like Ok(), NotFound(), and CreatedAtAction() that return proper HTTP responses. Controller inherits from ControllerBase and adds support for views (rendering HTML pages with Razor), which is for MVC web apps, not APIs. Since a Web API returns data (JSON) rather than pages, you inherit from ControllerBase and skip the extra view machinery.

Q4What does the [ApiController] attribute do?🌱 JUNIOR

It opts a controller into a set of API-friendly behaviors so you write less boilerplate: automatic 400 Bad Request responses when model validation fails, binding-source inference (so you don't have to annotate every parameter with [FromBody]/[FromQuery]), standardized ProblemDetails error responses, and a requirement to use attribute routing. It's on essentially every ASP.NET Core Web API controller because it encodes the sensible defaults for HTTP APIs.

Q5What is attribute routing?🌱 JUNIOR

Attribute routing means you define each endpoint's URL directly on the code using attributes — [Route("api/products")] on the controller and [HttpGet("{id}")] on the action. The route lives right next to the method it maps to, so reading the code tells you the URL. The controller-level route acts as a prefix that combines with each action's template. It's the standard, preferred approach for Web APIs because it's explicit and per-endpoint, unlike the older conventional routing that defines one global pattern centrally.

Q6What is a route parameter, and how does it reach my method?🌱 JUNIOR

A route parameter is a variable part of the URL template written in curly braces, like {id} in api/products/{id}. When a request URL matches, the value in that segment (say 42) is captured and automatically passed to your action method's argument of the same name (int id). This is model binding: the framework maps the URL segment to your parameter by matching names. So GET /api/products/42 calls GetById(int id) with id = 42.

Q7What does {id:int} mean in a route?🌱 JUNIOR

It's a route parameter id with an int constraint — the route only matches if that URL segment is an integer. So api/products/42 matches, but api/products/abc does not (the client gets a 404, and your action never runs). Constraints like this restrict what a parameter accepts; other examples are :guid, :alpha, and :min(1). They help ensure your action only receives well-formed values.

Q8How do you return a 404 from an action?🌱 JUNIOR

Return the NotFound() helper method from ControllerBase: return NotFound();. It produces a 404 Not Found response. Similarly, Ok(data) returns 200 with a body, CreatedAtAction(...) returns 201, BadRequest() returns 400, and NoContent() returns 204. You don't write raw status numbers — you call the named helper for the outcome, and it sets the correct status code and response shape. A typical pattern is return product is null ? NotFound() : Ok(product);.

🟠 Mid

Q9How does binding source inference work under [ApiController]?🟠 MID

With [ApiController], the framework infers where each action parameter comes from, so you rarely need explicit [From…] attributes. The rules: a parameter whose name matches a route template token ({id}) is bound from the route ([FromRoute]); a complex type (a class or record) is bound from the request body ([FromBody]); and a simple type that isn't in the route is bound from the query string ([FromQuery]). This removes most annotations from typical CRUD actions. You'd still add an explicit attribute when a parameter name doesn't match its route token, when you need a header ([FromHeader]), or when you want the binding intent to be obvious to readers.

Q10When would you use a route parameter versus a query string parameter?🟠 MID

Use a route parameter when the value identifies which resource — it's part of the resource's identity and belongs in the path: api/products/42, api/customers/7/orders/3. Use a query string parameter when the value refines how you want a collection returned — filtering, sorting, paging, searching — which are modifiers, not identity: api/products?category=audio&sort=price&page=2. Rule of thumb: identity and hierarchy go in the path (route parameters, often constrained like {id:int}); optional result-shaping goes in the query string (bound via [FromQuery], frequently with defaults). This mirrors REST's path-vs-query distinction.

Q11What is an AmbiguousMatchException and how do route constraints prevent it?🟠 MID

It's thrown at request time when a URL matches two or more actions equally and routing can't decide which to invoke. Classic example: two GET actions both mapped to {id}, one taking int and one taking string — a request to api/products/42 fits both templates identically, so the engine can't choose. Route constraints break the tie by making the templates distinguishable: mark the numeric one {id:int} and the other {slug}. Now 42 only matches the int route and wireless-mouse only matches the slug route — no ambiguity. More broadly, routing prefers the most specific template, and constraints (plus literal segments) are how you make one candidate more specific than another.

Q12If you have GET /products/{id} and GET /products/featured, why doesn't "featured" get treated as an id?🟠 MID

Because routing ranks candidates by specificity, and a literal segment is more specific than a parameter segment. The template featured is a fixed literal, while {id} is a variable placeholder, so for the URL api/products/featured the literal route wins and GetFeatured() is invoked. If you'd also constrained the id route as {id:int}, that reinforces it — featured isn't an int, so it couldn't match GetById at all. Either way, you don't have to worry about ordering the actions; the routing engine automatically prefers the tightest, most specific match.

Q13What's the difference between attribute routing and conventional routing? Which do you use for APIs?🟠 MID

Attribute routing declares routes with attributes directly on controllers and actions ([Route], [HttpGet("{id}")]) — explicit, per-endpoint, and local to the code it maps. Conventional routing defines one or more global URL patterns centrally in Program.cs (like {controller}/{action}/{id?}) that apply across all controllers. For Web APIs, attribute routing is the standard: it gives precise control over RESTful URLs, keeps each route beside its action, supports constraints and versioning cleanly, and is required by [ApiController]. Conventional routing is more common in MVC apps that render views with predictable controller/action URLs. They can coexist, but for an API you reach for attribute routing.

Q14What does the [controller] token in a route template do, and why is it useful?🟠 MID

[controller] is a token replaced at startup with the controller's name minus the "Controller" suffix — so in ProductsController, [Route("api/[controller]")] becomes api/products. It's useful because it keeps the route in sync with the class name automatically: rename the class and the URL follows, and it enforces a consistent naming convention across controllers without repeating the literal path. There's also an [action] token, but hard-coding action names into API URLs is discouraged — REST URLs should be resource nouns, with the HTTP verb (not the URL) expressing the action.

Q15When should you return IActionResult versus ActionResult<T> versus the raw type?🟠 MID

IActionResult is the flexible choice when an action can return different result kinds (Ok, NotFound, BadRequest) — it doesn't tie the signature to a data type. ActionResult<T> gives you that same flexibility and documents the success payload type T, which improves OpenAPI/Swagger output and lets you return either an ActionResult (like NotFound()) or a T directly — it's often the best default for a typed API. Returning the raw type T is simplest for actions that always succeed with data (the framework wraps it in 200), but you lose the ability to return other status codes cleanly. In practice ActionResult<T> is a common sweet spot: type information for tooling plus the freedom to return status-code results.

Q16Route constraints run before your action. Why does that matter, and what are their limits?🟠 MID

Constraints are evaluated during routing, before model binding and before your action executes. That matters for two reasons: performance — a URL that fails a constraint (e.g. abc against {id:int}) is rejected immediately with a 404, so the server doesn't waste work binding and running the action; and clarity/safety — your action can assume the parameter already has the right shape, so there's less defensive code. The limits: constraints only validate the format of URL segments, not business reality — {id:int} guarantees id is a number but not that product 42 exists (your action still checks and returns NotFound()). They also aren't a substitute for validating request bodies, which is model validation's job. Think of constraints as a fast, routing-level filter on URL shape, not full input validation.

🔴 Senior

Q17Walk through how ASP.NET Core selects an action for an incoming request.🔴 SENIOR

At startup, the framework scans controllers and builds an endpoint table from the route attributes. Per request: the routing middleware takes the URL path and finds all endpoint templates whose literal segments and parameter positions match. It then removes candidates that fail their route constraints (e.g. {id:int} against a non-numeric segment) and those whose HTTP method constraint doesn't match the request verb (GET vs POST — the verb attributes implement action constraints). From the survivors it ranks by route precedence: more specific templates win — a literal segment beats a parameter, a constrained parameter beats a bare one, and catch-all parameters are least specific. The highest-priority match is selected. If two survivors have identical precedence and both still match, it throws an AmbiguousMatchException. Once an action is chosen, model binding populates its parameters from route values, query string, and body (per [ApiController] inference), any filters run, and the action executes. The key senior insight is that selection is a filter-then-rank pipeline, and specificity — not declaration order — decides the winner.

Q18How would you design the routes for a nested resource like a customer's orders and an order's items?🔴 SENIOR

Model the ownership hierarchy in the path, but keep nesting shallow and constrain identity segments. A dedicated OrdersController could use a controller route like [Route("api/customers/{customerId:int}/orders")] so every action is scoped to a customer: GET (list that customer's orders), GET {orderId:int} (one order), POST (create under the customer). For an order's line items, either an OrderItemsController with [Route("api/orders/{orderId:int}/items")] or item actions on the orders controller — but I'd stop nesting at one or two levels. Deeper relationships (an item's product) are better expressed as a link to the top-level resource (/api/products/{id}) than as /customers/x/orders/y/items/z/product, which is unwieldy and couples URLs to a rigid hierarchy. Constrain every id (:int/:guid) both to disambiguate and to reject malformed URLs early. And validate that the nested resource actually belongs to its parent (order Y really is customer X's) rather than trusting the path — a common security gap. The judgment: reflect genuine ownership in the URL, cap depth for usability, and don't let routing structure substitute for authorization checks.

Q19Controllers vs minimal APIs — how do you choose?🔴 SENIOR

Both are first-class and current. Minimal APIs define endpoints as lambdas mapped directly in Program.cs (or grouped with MapGroup) — less ceremony, lower overhead, and great for small services, microservices, and focused endpoints; the trade-off is that as an app grows, cramming logic into Program.cs gets unwieldy without disciplined organization, and some MVC features (certain filters, model-binding conveniences, conventions) historically came more naturally to controllers, though minimal APIs have closed much of that gap. Controllers give structured grouping of related endpoints, a mature filter pipeline, attribute-based conventions, and a familiar organization that scales well for large APIs with many resources and cross-cutting concerns. My rule of thumb: reach for minimal APIs for small or performance-sensitive services and simple endpoints; prefer controllers for larger applications where grouping, conventions, and the filter pipeline pay off — and it's fine to mix them in one app. The decision is about team familiarity, app size, and how much you lean on MVC's conventions, not about one being obsolete.

Q20How would you keep controllers thin as an application grows?🔴 SENIOR

The controller's job is HTTP concerns only: bind and validate input, call into the domain, and translate the result into an HTTP response with the right status code. Everything else — business logic, data access, orchestration — belongs behind an injected service (Day 3's dependency injection). So a ProductsController action should be a few lines: take the request, call _productService.GetAsync(id), and return Ok/NotFound. Techniques to hold that line: inject services rather than doing work in the controller; use DTOs/request-response models rather than exposing domain entities directly, mapping at the boundary; push validation into model attributes or FluentValidation; centralize error handling (a global exception handler / ProblemDetails) rather than try/catch in every action; and consider a mediator pattern (like MediatR) for larger apps so actions dispatch a command/query and stay trivial. The payoff is testability (the logic lives in services you can unit-test without HTTP — Day 13), reusability (the same service serves multiple endpoints or a background worker), and clarity (the controller reads as a thin HTTP adapter). "Fat controllers" that mix HTTP, business rules, and data access are the anti-pattern this avoids.

Q21How do you handle API versioning with controllers and routing?🔴 SENIOR

Once external clients depend on your routes, breaking changes need a new version while the old keeps working, and routing is one place versioning lives. Common approaches: URL path versioning — embed the version in the route, e.g. [Route("api/v{version:apiVersion}/[controller]")] producing /api/v1/products and /api/v2/products; most visible and easy for clients. Query string (?api-version=2) or header-based versioning keep URLs clean but are less discoverable. The Asp.Versioning (formerly Microsoft.AspNetCore.Mvc.Versioning) libraries support all of these: you decorate controllers with [ApiVersion("1.0")], can have two controller classes for v1 and v2, map versions to actions, and integrate with OpenAPI to produce per-version docs. Beyond the mechanism, the senior concerns are process: deprecation policy and timelines, communicating changes, supporting old versions for a defined window, and — importantly — designing DTOs so internal model changes don't force external version bumps (don't expose entities directly). Version as rarely as possible by favoring additive, non-breaking changes; when you must break, the routing scheme is the least interesting part compared to the client-migration strategy.

Q22A teammate writes GET api/products/getAll and POST api/products/createProduct. What's wrong and how do you fix it?🔴 SENIOR

The problem is verbs in the URL — those routes duplicate the HTTP method's meaning in the path, which violates REST conventions (Day 1: URLs are nouns, the HTTP verb is the action). It produces redundant, non-idiomatic URLs (POST .../createProduct says "create" twice) that are harder to predict and don't leverage HTTP semantics — caching, idempotency, and tooling all key off the method. The fix is resource-noun routes with the verb expressed by the HTTP method: GET /api/products (list), GET /api/products/{id} (one), POST /api/products (create), PUT/PATCH /api/products/{id} (update), DELETE /api/products/{id} (remove) — all on a ProductsController with [Route("api/[controller]")] and verb attributes carrying the (nounless) templates. I'd also note this is exactly what the [action] route token encourages if misused, which is why it's discouraged for APIs. The teammate's instinct likely came from RPC-style or older conventional-routing habits; the coaching point is that in REST the method is the verb, so the URL should name only the resource. Beyond correctness, consistent noun-based routes make the whole API predictable, which is REST's core value.

Q23When and why would you write a custom route constraint?🔴 SENIOR

When the built-in constraints (int, guid, alpha, range, regex, etc.) can't express a domain-specific URL-segment rule you want enforced at the routing layer — for example, a product SKU with a specific checksum/format, a tenant code drawn from a known set, or a slug pattern too complex or reused too often to inline as a regex everywhere. You implement IRouteConstraint (a Match method returning whether the value qualifies) and register it in routing options with a name, then use it as {sku:sku}. The benefits mirror built-in constraints: it disambiguates routes and rejects malformed URLs before the action runs, and centralizing the rule keeps it DRY and readable versus copy-pasted regexes. The caveats: keep custom constraints cheap and side-effect-free — they run on the hot path during matching for potentially many candidate routes, so they must not hit a database or do heavy work (existence checks belong in the action, not a constraint). And don't overuse them; if the rule is really business validation rather than URL shape, it belongs in validation (Day 14), not routing. So: custom constraints are for reusable, cheap, format-level URL rules the built-ins can't express — not for business logic or data lookups.

Q24Why does separating HTTP concerns (controllers/routing) from business logic matter architecturally?🔴 SENIOR

Because it's what keeps an API maintainable, testable, and adaptable as it grows, and it's the natural consequence of treating the controller as a thin HTTP adapter. When controllers only handle HTTP (routing, binding, validation, status-code translation) and delegate real work to injected domain services, several things follow. Testability: the business logic lives in plain services you can unit-test without spinning up the HTTP stack (Day 13), while controllers stay trivial enough to need little testing. Reusability: the same service can back a controller endpoint, a minimal API, a background worker (Day 12), or a gRPC method — the logic isn't trapped in the web layer. Adaptability: you can change the transport (add a new API version, swap controllers for minimal APIs, expose the same logic over a message queue) without rewriting business rules, and change business rules without touching routing. Clarity and single responsibility: routing/controllers answer "how does this reach the web?", services answer "what does the business do?" — mixing them produces fat controllers that are hard to read, test, and reuse. This layering is why the series moves from controllers (the HTTP edge) into dependency injection and services next: the edge should be thin, and the value should live behind it. The architectural principle is that the delivery mechanism (HTTP, controllers, routes) is a detail that should be swappable around a stable core of business logic.
