Cross-Cutting API Concerns 🧵

Some concerns don't belong to any one endpoint — they thread through the whole API. How do you evolve it without breaking existing clients? How do you let a browser app on another domain call it? How do you serve the format each client wants, and avoid recomputing responses that never changed? Today we cover the four classics — versioning, CORS, content negotiation, and caching — and stand up v1 and v2 of an endpoint with a working CORS policy.

🔢 Versioning 🌐 CORS 🤝 Content negotiation ⏱️ Caching headers 📦 Output caching 🧵 Middleware order
01

What "Cross-Cutting" Means 🧵

A cross-cutting concern is a capability that applies broadly across an application rather than to a single feature. Logging, error handling (Day 11), and authentication (Day 12) are cross-cutting — and so are today's four. They share a trait: you configure them once, centrally (usually in Program.cs and the middleware pipeline), and they shape the behavior of many or all endpoints.

🏗️ Analogy — a building's utilities

Individual rooms are your endpoints. But plumbing, wiring, and climate control run through every room — you don't install a separate furnace per office. Cross-cutting concerns are the building's shared systems: designed once at the infrastructure level, benefiting the whole structure. Today's four are the API's utilities.

🔑 The through-line

Each of today's concerns answers a "how does the API behave as a whole?" question: versioning — how it evolves over time; CORS — who may call it from a browser; content negotiation — what format it returns; caching — how it avoids redundant work. All four are configured centrally and felt everywhere.

02

Why Version an API 🔢

Your API is a contract (Day 10). Once clients depend on it, you can't freely change it — removing a field, renaming a property, or changing a type is a breaking change that shatters existing consumers you may not control. Versioning is the escape hatch: run multiple versions side by side, so legacy clients keep using v1 while new clients adopt v2.

🔑 What counts as breaking?

Adding a new optional field is usually safe — well-written clients ignore unknowns. Removing or renaming fields, changing a type, or altering behavior are breaking. When you must make a breaking change, that's when you cut a new version rather than mutating the old one.

There are several ways for a client to say which version it wants; you pick one (or combine them):

StrategyExampleTrade-off
URL segment/api/v2/productsExplicit, easy to see/test; most common for public APIs
Query string/api/products?api-version=2.0Keeps one path; less visible
HeaderX-Version: 2.0Clean URLs; requires setting a header
Media typeAccept: …;v=2.0Ties version to content negotiation
✅ The package moved — know the name

Versioning isn't built in. The current library is Asp.Versioning (e.g. Asp.Versioning.Mvc for controllers, Asp.Versioning.Http for Minimal APIs) — it was formerly Microsoft.AspNetCore.Mvc.Versioning. On .NET 10 you use the 10.0 packages, and Asp.Versioning.Mvc.ApiExplorer integrates versioning with OpenAPI (Day 10).

03

Adding v1 & v2 of an Endpoint 🚀

Today's first "Do." Install the packages, register versioning, and expose two versions of a products endpoint. First, the setup:

C# — Program.cs (register versioning)
// dotnet add package Asp.Versioning.Mvc
// dotnet add package Asp.Versioning.Mvc.ApiExplorer
using Asp.Versioning;

builder.Services.AddControllers();
builder.Services.AddApiVersioning(options =>
{
    options.DefaultApiVersion = new ApiVersion(1, 0);
    options.AssumeDefaultVersionWhenUnspecified = true;   // legacy clients get v1
    options.ReportApiVersions = true;                    // advertise versions in headers
    options.ApiVersionReader = new UrlSegmentApiVersionReader();
})
.AddMvc()                                              // controllers support
.AddApiExplorer(options =>
{
    options.GroupNameFormat = "'v'VVV";                  // e.g. "v1", "v2"
    options.SubstituteApiVersionInUrl = true;
});

Now expose both versions. Two clean patterns: one controller declaring both versions with [MapToApiVersion] per action, or separate controllers per version. Here's the single-controller approach:

C# — one controller, two versions
[ApiController]
[Route("api/v{version:apiVersion}/products")]
[ApiVersion("1.0")]
[ApiVersion("2.0")]
public class ProductsController : ControllerBase
{
    // GET /api/v1/products — simple shape
    [HttpGet, MapToApiVersion("1.0")]
    public IActionResult GetV1() =>
        Ok(new { Id = 1, Name = "Notebook", Price = 4.99 });

    // GET /api/v2/products — richer shape (a breaking change)
    [HttpGet, MapToApiVersion("2.0")]
    public IActionResult GetV2() =>
        Ok(new
        {
            Id = 1,
            Name = "Notebook",
            Pricing = new { Amount = 4.99, Currency = "USD" }   // changed shape
        });
}

GET /api/v2/products

✅ Deprecating a version gracefully

When it's time to retire v1, mark it [ApiVersion("1.0", Deprecated = true)]. Because you set ReportApiVersions = true, responses then include an api-deprecated-versions: 1.0 header — a signal to clients to migrate, without breaking them yet. Later, once traffic drains, you remove it.

🔑 Ties into OpenAPI

With AddApiExplorer registered, each version can get its own OpenAPI document (Day 10) — on .NET 10 the Asp.Versioning.OpenApi package plus MapOpenApi().WithDocumentPerVersion() produces /openapi/v1.json and /openapi/v2.json automatically, so your docs and versions stay in sync.

04

CORS & the Same-Origin Policy 🌐

Browsers enforce the same-origin policy: a page served from https://app.example.com can't, by default, make JavaScript requests to a different origin like https://api.other.com. An "origin" is the scheme + host + port. This stops a malicious page from quietly reading another site's data using your logged-in session.

CORS (Cross-Origin Resource Sharing) is the W3C standard by which a server can relax that restriction — telling the browser "requests from these specific origins are OK." Your API opts in per policy.

⚠️ CORS relaxes security — it isn't a security feature

Microsoft is explicit about this: CORS loosens the same-origin policy; it does not make your API safer. It's enforced by the browser, not the server — the server just returns headers, and the browser decides whether to expose the response to the page. So CORS is not a substitute for authentication or authorization (Days 12–13); a non-browser client (curl, a server) ignores CORS entirely.

🚪 Analogy — the guest list

Same-origin policy is a club that only admits its own members. CORS is the bouncer holding a guest list: origins on the list get in, others are turned away. But the guest list is checked by the browser (the bouncer works for the visitor, not the club) — which is why CORS protects users, not your server, and why anything that isn't a browser walks right past it.

How a cross-origin call works 🔎

Step 1 · ✈️ preflight (sometimes)
Browser sends OPTIONS

For "complex" requests (custom headers, PUT/DELETE), the browser first asks permission with an OPTIONS request naming the method and headers it wants to use.

Step 2 · 📋 server answers
Access-Control-Allow-* headers

If the origin is allowed, the API returns headers like Access-Control-Allow-Origin listing what's permitted.

Step 3 · 📨 real request
Browser proceeds — or blocks

If the headers allow it, the browser sends the actual request and exposes the response. If not, it blocks the response and raises a CORS error.

05

Configuring CORS ⚙️

Today's second "Do." Two steps as always: register a policy with AddCors, then enable it with UseCors.

C# — a named CORS policy
const string FrontendPolicy = "FrontendPolicy";

builder.Services.AddCors(options =>
{
    options.AddPolicy(FrontendPolicy, policy =>
        policy.WithOrigins("https://app.example.com", "http://localhost:3000")
              .WithMethods("GET", "POST", "PUT", "DELETE")
              .WithHeaders("Authorization", "Content-Type")
              .WithExposedHeaders("X-Pagination-TotalCount")  // let JS read it (Day 9)
              .SetPreflightMaxAge(TimeSpan.FromMinutes(10)));  // cache preflight
});
C# — enable it in the pipeline (order matters!)
var app = builder.Build();

app.UseRouting();
app.UseCors(FrontendPolicy);   // AFTER UseRouting, BEFORE auth
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();
Builder methodEffect
WithOrigins("…")Allow specific origins (no trailing slash!)
AllowAnyOrigin()Allow all origins (can't combine with credentials)
WithMethods(…) / AllowAnyMethod()Permit specific / all HTTP verbs
WithHeaders(…) / AllowAnyHeader()Permit specific / all request headers
AllowCredentials()Allow cookies/auth headers (needs explicit origins)
WithExposedHeaders(…)Let JS read custom response headers
SetPreflightMaxAge(…)Cache the preflight response
⚠️ Two traps that bite everyone

1. You cannot combine AllowAnyOrigin() with AllowCredentials() — the CORS spec forbids wildcards on credentialed requests, and it'll throw. Use explicit WithOrigins when you need credentials. 2. Origins in WithOrigins must have no trailing slash — "https://app.example.com/" silently fails to match. Small detail, hours of debugging.

✅ Per-endpoint CORS

Beyond a global UseCors("policy"), you can apply policies per endpoint: [EnableCors("policy")] on a controller/action, or .RequireCors("policy") on a Minimal API endpoint. Handy when different endpoints serve different client origins.

Content Negotiation 🤝

Content negotiation is how the client and server agree on the response format. The client sends an Accept header stating what it can handle; the server picks a matching formatter to serialize the response. In ASP.NET Core the default is JSON, but the mechanism is general.

Accept:
application/json
client asks
→
JSON formatter
server picks
→
Content-Type:
application/json
server answers

Recall from Day 6 that returning an object (via Ok(dto) or ActionResult<T>) runs it through the output formatters, which honor content negotiation. By default only JSON is configured; to also serve XML, add the XML formatter:

C# — enabling XML alongside JSON
builder.Services.AddControllers()
    .AddXmlSerializerFormatters();   // now honors Accept: application/xml
🔑 The Day 6 connection — and a caveat

Content negotiation is exactly why the TypedResults/Results distinction from Day 6 mattered: the HttpResults types (like TypedResults.Ok) write the response directly and don't run through the formatters, so they skip content negotiation. Traditional controller results (Ok(obj), ActionResult<T>) do negotiate. If a client's Accept header seems ignored on a Minimal API, that's usually why.

✅ Force or restrict formats

You can constrain what an endpoint produces or consumes with [Produces("application/json")] and [Consumes(...)] (Day 10 — these also document the API). Setting options.ReturnHttpNotAcceptable = true makes the server return 406 Not Acceptable when it can't satisfy the Accept header, instead of silently falling back to JSON.

07

Caching Headers ⏱️

The fastest response is the one you never compute. HTTP caching lets clients and intermediaries reuse a response instead of re-fetching it, and it's driven by response headers that follow standard HTTP semantics.

HeaderPurpose
Cache-ControlThe main control: max-age=60 (cache 60s), no-cache, private/public
ExpiresOlder absolute expiry time (superseded by Cache-Control)
ETagA version fingerprint; client sends it back via If-None-Match
Last-ModifiedTimestamp; client revalidates via If-Modified-Since
VaryWhich request headers produce different cached responses (e.g. Accept-Encoding)
🔑 The 304 round-trip saver

With an ETag, the client re-requests with If-None-Match: "abc". If nothing changed, the server replies 304 Not Modified with an empty body — the client reuses its cached copy. You still make a request, but skip re-sending (and often re-computing) the payload. This conditional caching is ideal for data that changes occasionally.

In controllers, the [ResponseCache] attribute sets these headers declaratively:

C# — setting cache headers with [ResponseCache]
[HttpGet]
[ResponseCache(Duration = 60,                    // Cache-Control: max-age=60
               Location = ResponseCacheLocation.Any, // public
               VaryByQueryKeys = ["page"])]          // separate cache per page
public IActionResult GetProducts(int page) => Ok(/* … */);
⚠️ Headers ask; they don't guarantee

[ResponseCache] only sets headers — it tells clients and proxies how they may cache. It doesn't store anything on your server, and browsers often override caching on refresh (sending Cache-Control: no-cache). For caching that actually lives on the server, you need caching middleware (next).

08

Output Caching — Server-Side ⚙️

Two middleware options actually store responses on the server. Response Caching (older) respects HTTP cache headers like a proxy. Output Caching (added in .NET 7) is the newer, more capable option — you control what's cached regardless of client headers, which is exactly what an API wants.

C# — output caching setup
builder.Services.AddOutputCache(options =>
{
    // a reusable named policy
    options.AddPolicy("Products60", b => b.Expire(TimeSpan.FromSeconds(60)));
});

var app = builder.Build();
app.UseOutputCache();   // after UseCors, UseAuth* (see §09)
C# — apply per endpoint
// Controllers: attribute
[HttpGet, OutputCache(PolicyName = "Products60")]
public IActionResult GetProducts() => Ok(/* … */);

// Minimal API: fluent — vary the cache by a query param
app.MapGet("/products", GetProducts)
   .CacheOutput(b => b.Expire(TimeSpan.FromSeconds(60)).SetVaryByQuery("page"));
Response CachingOutput Caching
SinceEarly ASP.NET Core.NET 7+
Driven byClient HTTP headersServer-side policies
Cache locationIn-memory (per server)Pluggable (memory, Redis, …)
ControlLimited; client can prevent itFull server control
Best forSimple proxy-style cachingAPIs & UI apps — the modern pick
⚠️ Never cache personalized or authorized data carelessly

If a response depends on the user (auth'd data from Days 12–13), caching it globally can leak one user's data to another. That's exactly why UseOutputCache must come after UseAuthentication/UseAuthorization (§09), and why you vary the cache key by the relevant dimensions — or simply don't cache per-user responses.

09

Pipeline Order — the Part People Break 🧵

These concerns are middleware, and middleware order is behavioral, not cosmetic. Here's a correct ordering that satisfies all the documented constraints.

Step 1 · 🧭 UseRouting
Match the endpoint

Routing runs first so later middleware knows which endpoint (and its metadata, like CORS or cache policies) was selected.

Step 2 · 🌐 UseCors
After routing, before caching & auth

CORS must run before output/response caching — otherwise the cache can store a response without CORS headers and later serve it to a browser, causing intermittent CORS errors.

Step 3 · 🔐 UseAuthentication → UseAuthorization
Establish & check identity

Auth (Days 12–13) runs before caching so the cache never serves protected content to the wrong user.

Step 4 · ⏱️ UseOutputCache
After CORS + auth

Now caching sees fully-formed, correctly-authorized, CORS-headed responses — safe to store and replay.

Step 5 · 🎯 MapControllers
Execute the endpoint

Finally the versioned, negotiated, cacheable response is produced by your action.

🔑 The rules to memorize

UseCors goes after UseRouting and before caching and UseAuthorization. UseOutputCache/UseResponseCaching go after UseCors and after auth. Get this wrong and the bugs are nasty and intermittent — a cached response missing CORS headers, or worse, one user seeing another's cached data. Order is correctness, not style.

Interview Questions 🎤

Click any question to reveal a plain-English answer. Tags mark the experience level a question is usually aimed at.

🌱 Junior 0–2 yrs 🟠 Mid 2–5 yrs 🔴 Senior 5–7 yrs
Q1Why do APIs need versioning?🌱 Junior

Because an API is a contract that clients depend on. Once you can't control every client, breaking changes — removing or renaming fields, changing types or behavior — would break them. Versioning lets you run multiple versions side by side, so existing clients keep using v1 while new ones adopt v2, and you can evolve the API without a flag day.

Q2What are the common ways to specify an API version?🌱 Junior

URL segment (/api/v2/products), query string (?api-version=2.0), a custom header (X-Version: 2.0), or the media type in the Accept header. URL versioning is the most common and most explicit; header versioning keeps URLs clean but requires clients to set a header. You choose a strategy via the ApiVersionReader.

Q3What is CORS?🌱 Junior

Cross-Origin Resource Sharing — a standard that lets a server tell browsers which other origins (scheme+host+port) are allowed to make JavaScript requests to it. Browsers block cross-origin requests by default (the same-origin policy); CORS is how a server opts in to relaxing that for specific origins.

Q4How do you enable CORS in ASP.NET Core?🌱 Junior

Two steps: register a policy with builder.Services.AddCors(options => options.AddPolicy("Name", p => p.WithOrigins("https://app.example.com").AllowAnyHeader().AllowAnyMethod())), then enable it in the pipeline with app.UseCors("Name"). You can also apply policies per endpoint with [EnableCors] or RequireCors.

Q5What is content negotiation?🌱 Junior

The process where the client and server agree on the response format. The client sends an Accept header (e.g. application/json); the server picks a matching formatter and sets the Content-Type accordingly. ASP.NET Core returns JSON by default; you can add other formatters like XML.

Q6What does the Cache-Control header do?🌱 Junior

It's the main HTTP header for controlling caching. Values like max-age=60 say a response can be reused for 60 seconds, no-cache forces revalidation, and public/private say whether shared caches (proxies) may store it. It tells clients and intermediaries how they may cache the response.

Q7What's a breaking change vs. a non-breaking change?🌱 Junior

Adding a new optional field is generally non-breaking — well-written clients ignore fields they don't recognize. Removing or renaming a field, changing a field's type, or changing behavior are breaking, because existing clients rely on the old shape. Breaking changes are what justify a new API version.

Q8What is an ETag?🌱 Junior

A response header carrying a version fingerprint of the content. The client stores it and sends it back on the next request via If-None-Match. If the content hasn't changed, the server responds 304 Not Modified with no body, and the client reuses its cached copy — saving bandwidth and often computation.

Q9Which package provides API versioning, and how do you register it?🟠 Mid

The Asp.Versioning family — Asp.Versioning.Mvc for controllers, Asp.Versioning.Http for Minimal APIs, plus Asp.Versioning.Mvc.ApiExplorer for OpenAPI integration (it was formerly Microsoft.AspNetCore.Mvc.Versioning). You register with AddApiVersioning(options => …).AddMvc().AddApiExplorer(…), setting things like DefaultApiVersion, AssumeDefaultVersionWhenUnspecified, and the ApiVersionReader.

Q10How do you expose two versions of the same endpoint?🟠 Mid

Either put both versions on one controller — annotate it with [ApiVersion("1.0")] and [ApiVersion("2.0")] and mark each action with [MapToApiVersion("…")] — or use separate controllers per version. With a URL-segment route like api/v{version:apiVersion}/products, requests to /api/v1/products and /api/v2/products reach the right action.

Q11Why is CORS enforced by the browser and not the server?🟠 Mid

Because the same-origin policy it relaxes is a browser security model protecting the user. The server just returns Access-Control-Allow-* headers stating what's permitted; the browser reads them and decides whether to expose the response to the page. This is why CORS isn't a server security feature — a non-browser client like curl or another server ignores it entirely, so CORS never replaces authentication/authorization.

Q12Why can't you combine AllowAnyOrigin with AllowCredentials?🟠 Mid

The CORS spec prohibits wildcards on credentialed requests for security — allowing any origin to send cookies/auth headers would let any site make authenticated requests on the user's behalf. ASP.NET Core throws if you configure both. When you need credentials, you must list explicit origins with WithOrigins and call AllowCredentials().

Q13What's a preflight request?🟠 Mid

For "complex" cross-origin requests (custom headers, or methods like PUT/DELETE), the browser first sends an OPTIONS request asking permission — naming the intended method and headers via Access-Control-Request-*. The server responds with the allowed methods/headers; only if they permit the real request does the browser send it. You can cache preflight responses with SetPreflightMaxAge to cut this overhead.

Q14Response caching vs. output caching — what's the difference?🟠 Mid

Response caching (older) works like a proxy — it caches based on the client's HTTP cache headers, and the client can prevent caching. Output caching (added in .NET 7) is server-controlled: you decide via policies what gets cached, independent of client headers, with pluggable stores (memory, Redis). For APIs, output caching is the modern, more capable choice because you keep control.

Q15Does [ResponseCache] cache anything on the server?🟠 Mid

No — it only sets response headers (Cache-Control, Vary, etc.) that tell clients and proxies how they may cache. Nothing is stored on your server, and clients can ignore or override it (browsers often do on refresh). To actually store responses server-side you add caching middleware — output caching or response caching.

Q16How does content negotiation relate to TypedResults from Minimal APIs?🟠 Mid

Traditional controller results (Ok(obj), ActionResult<T>) run through the output formatters, which honor the Accept header — so they negotiate content. The HttpResults types from Minimal APIs (TypedResults.Ok, etc.) write the response directly and bypass the formatters, so they don't do content negotiation. If an Accept header seems ignored on a Minimal API endpoint, that's usually the reason.

Q17Why does middleware order matter for CORS and caching?🔴 Senior

UseCors must run after UseRouting (so the selected endpoint's policy is known) and before caching. If caching ran before CORS, the cache could store a response without the CORS headers and later replay it to a browser, producing intermittent CORS failures for some users. Caching must also run after authentication/authorization, or it could serve one user's protected, cached response to another. Order here is correctness, and the resulting bugs are intermittent and hard to trace.

Q18How do you deprecate an API version gracefully?🔴 Senior

Mark it with [ApiVersion("1.0", Deprecated = true)] and set ReportApiVersions = true, so responses carry an api-deprecated-versions header telling clients to migrate — without removing the version yet. Communicate a sunset timeline (often via a Sunset header or docs), monitor traffic to the deprecated version, and only remove it once usage drains. This gives consumers time to adapt instead of a hard break.

Q19What are the risks of caching authenticated responses, and how do you handle them?🔴 Senior

The big risk is cross-user data leakage — caching a user-specific response and serving it to someone else. Mitigations: place caching after authentication/authorization in the pipeline; mark per-user responses Cache-Control: private (or don't cache them at all); vary the cache key by the relevant dimension (user, tenant, role) if you must cache; and be conservative — cache public, non-personalized GETs, not authorized data. Getting this wrong is a serious security bug, not just a performance one.

Q20How does API versioning integrate with OpenAPI documentation?🔴 Senior

The Asp.Versioning.Mvc.ApiExplorer package feeds version info into ASP.NET Core's API Explorer, letting you generate a separate OpenAPI document per version. On .NET 10, Asp.Versioning.OpenApi bridges into the built-in Microsoft.AspNetCore.OpenApi pipeline, and MapOpenApi().WithDocumentPerVersion() produces /openapi/v1.json, /openapi/v2.json, etc. — each containing only that version's operations, so docs and versions stay in sync automatically.

Q21When would you choose URL versioning vs. header versioning?🔴 Senior

URL versioning (/api/v2/…) is explicit, visible, easy to test in a browser, cache-friendly (distinct URLs), and dominant for public APIs — at the cost of "polluting" the URL and implying the resource identity changes. Header/media-type versioning keeps a single canonical URL (arguably more RESTful) and suits internal or content-negotiated APIs, but is invisible, harder to test, and easy for clients to forget. Many teams combine readers with a sensible default. The choice is about your consumers and tooling, not correctness.

Q22How would you invalidate or vary an output cache correctly?🔴 Senior

Vary the cache key by the inputs that change the response — query params (SetVaryByQuery), headers (SetVaryByHeader), or a custom value (VaryByValue) — so different inputs get different cached entries. For invalidation on writes, use cache tags: tag entries when caching and evict them via IOutputCacheStore.EvictByTagAsync when the underlying data changes. Combined with a short expiry as a safety net, tag-based eviction keeps cached list/detail endpoints fresh after mutations.

Q23Is enabling CORS enough to secure a cross-origin API? Explain.🔴 Senior

No — CORS relaxes security, it doesn't provide it. It only governs whether a browser exposes a response to a page; it does nothing against non-browser clients, and it's not authentication or authorization. A cross-origin API still needs real auth (Day 12), authorization (Day 13), HTTPS, input validation, and rate limiting. CORS just decides which browser origins may attempt requests — the server must still independently verify every request is permitted. Treating a permissive CORS policy as "security" is a classic misconception.

✅ Study-order tip

Start with why each concern exists (Q1, Q3, Q5, Q7) — the motivations are the most common openers. Nail versioning mechanics (Q2, Q9–Q10) and CORS setup + the browser-enforced insight (Q4, Q11–Q13), which together are the day's "Do." Understand caching layers — headers vs. middleware, response vs. output (Q6, Q8, Q14–Q15). Then the senior material that separates candidates: pipeline ordering, caching authenticated data safely, graceful deprecation, and the "CORS isn't security" point (Q17–Q23). If you can stand up v1/v2 of an endpoint and a working CORS policy with correct middleware order from memory, you own this — and the whole API series.
