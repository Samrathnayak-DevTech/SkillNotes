Log Basics

The logarithm is the inverse of the exponential function. Informally: logb(a) asks "to what power must b be raised to get a?"

logb(a) = x   ⟺   bx = a

Key identity:  loga(ax) = x
Derived:   2x = N  ⟹  x = log2(N)

Worked Examples

Memory Tip log2(N) tells you how many times you can halve N before reaching 1. This is exactly the number of iterations in any "divide by 2" loop.
02

Log Iteration Problems

The link between logarithms and iteration count becomes concrete when analyzing loops that halve (or double) their variable each step.

Question: How many times must N be divided by 2 to reach 1?

NStepsAnswer
100100 → 50 → 25 → 12 → 6 → 3 → 16
324324 → 162 → 81 → 40 → 20 → 10 → 5 → 2 → 18
99 → 4 → 2 → 13
2727 → 13 → 6 → 3 → 14

After x iterations of halving: N / 2x = 1  ⟹  N = 2x  ⟹  x = log2(N).

Quiz 1 — While loop halving

How many iterations?
pseudocode
i = N;
while(i > 1) {
    i = i / 2;
}

The sequence is N → N/2 → N/4 → … → 1. After x steps, i = N/2x = 1, so x = log2(N). Iterations = O(log N).

Quiz 2 — For loop doubling

How many iterations?
pseudocode
for(i = 1; i < N; i = i * 2) {
    // ...
}

The sequence is i = 1, 2, 4, 8, 16, … After x steps, i = 2x = N so x = log2(N). Iterations = O(log N).

Edge Case — Infinite Loop for(i=0; i≤N; i=i*2) — if i starts at 0, then 0×2 = 0 forever. This is an infinite loop. Always check the initial value when multiplying.
03

Loop Iteration Quizzes

Mastering iteration counting is essential before Big-O analysis. Work through these systematically: identify the range of each loop variable, then multiply (nested) or add (sequential).

Quiz 4 — Constant outer loop
pseudocode
for(i = 1; i <= 10; i++) {
    for(j = 1; j <= N; j++) {
        // ...
    }
}

Outer loop runs exactly 10 times (constant). Inner runs N times each. Total = 10 × N = O(N). A constant multiplier gets dropped.

Quiz 5 — Both loops linear
pseudocode
for(i = 1; i <= N; i++) {
    for(j = 1; j <= N; j++) {
        // ...
    }
}

Each of the N outer iterations runs the inner N times. Total = N × N = O(N²).

Quiz 6 — Outer linear, inner doubling
pseudocode
for(i = 1; i <= N; i++) {
    for(j = 1; j <= N; j = j * 2) {
        // ...
    }
}

Outer runs N times. Inner doubles j until N → log2(N) iterations. Total = N × log(N) = O(N log N).

Quiz 7 — Triangular inner loop (j ≤ i)
pseudocode
for(i = 1; i <= N; i++) {
    for(j = 1; j <= i; j++) {
        // ...
    }
}

When i=1: 1 iteration. When i=2: 2. … When i=N: N. Total = 1+2+3+…+N = N(N+1)/2 = O(N²).

Quiz 9 — Inner loop runs 2i times
pseudocode
for(i = 1; i <= N; i++) {
    for(j = 1; j <= 2^i; j++) {
        // ...
    }
}

Inner runs 2¹ + 2² + 2³ + … + 2N. This is a GP with a=2, r=2, N terms. Sum = 2(2N–1) = O(2N).

04

Asymptotic Analysis & Big-O

Asymptotic analysis means studying how an algorithm's performance (number of iterations) behaves as the input size N grows toward infinity. We care about the rate of growth, not the exact count.

Big-O Rules 1. Count the total iterations as a function of N.
2. Drop all lower-order terms.
3. Drop all constant coefficients.
Only the dominant (fastest-growing) term remains.

Worked Examples

Iteration CountBig-O

4N² + 3N + 1O(N²)

4N·log N + 3N·√N + 10⁶O(N√N)
100 × log NO(log N)
N / 10O(N)

Complexity Hierarchy

From best (fewest operations) to worst (most operations) as N grows large:

O(1)< O(log N)< O(√N)< O(N)< O(N log N)< O(N√N)< O(N²)< O(N³)< O(2N)< O(N!)< O(NN)
05

Why Drop Lower-Order Terms & Coefficients?

Why ignore lower-order terms?

As N grows, the dominant term overwhelms everything else. Consider iterations = N² + 10·N:

NN² + 10·N (total)10·N as % of total
10100 + 100 = 20050%
10010,000 + 1,000 = 11,000≈9%
1,00010⁶ + 10⁴ ≈ 10⁶≈1%
10,00010⁸ + 10⁵ ≈ 10⁸≈0.1%

As N increases, the contribution of lower-order terms in the total iteration count decreases. For large inputs — which is exactly the regime we care about — they become negligible.

Why ignore coefficients?

A coefficient does not change the class (rate of growth) of an algorithm. Consider two algorithms:

Algo 1Algo 2Winner
10 × log NNAlgo 1
100 × log NNAlgo 1
100 × log NN / 10Algo 1
10 × NN² / 100Algo 1 (for large N)
Key Insight The rate of growth — not the coefficient — determines which algorithm wins for large inputs. O(log N) will eventually beat O(N) regardless of their coefficients. This is why coefficients are dropped in Big-O.
06

Issues with Big-O

Issue 1 — "Sufficiently large" N required

Big-O describes behaviour for large inputs. Algo 1 (10³·N) and Algo 2 (N²) both express O(N) vs O(N²), but Algo 1 is not always better. For small N (say N < 1000), Algo 2 might actually be faster because 10³·N > N² when N < 10³.

Analogy Think of an India vs. Pakistan cricket match with 50 million viewers vs. the "most watched YouTube video" with 15 billion views. For the YouTube video scale (huge N), O(log N) streaming algorithms are essential. For small audiences, even O(N²) works fine. Always consider your actual data scale.

Issue 2 — Same Big-O doesn't mean same speed

Two algorithms can share the same Big-O class yet differ significantly in actual performance. Compare:

two O(N) algorithms
// Algorithm A — N iterations
for(i = 1; i <= N; i++) {
    if(i % 2 != 0) { c = c + 1; }
}

// Algorithm B — N/2 iterations
for(int i = 1; i <= N; i = i + 2) {
    c = c + 1;
}

Both are O(N), but Algorithm B runs in exactly N/2 iterations while Algorithm A runs in N. When two algorithms share the same Big-O, compare their actual iteration counts directly.

Rule If two algorithms have the same Big-O class, fall back to counting iterations directly. Big-O is a coarse filter — not the final arbiter.
07

Online Judges & Time Limit Exceeded (TLE)

When you submit code to an online judge (LeetCode, Codeforces, etc.), it runs on a server with a fixed time limit — typically 1–2 seconds. The server processes roughly 109 instructions per second (1 GHz processor).

Each iteration of a typical loop body contains about 6–10 simple instructions (comparisons, additions, assignments, increments). So the practical rule of thumb:

Practical Rule In the worst case, you can execute roughly 107 to 108 iterations within a 1-second time limit. Design your algorithm so that its total iteration count stays within this bound for the given constraints.

Approx. 1 (10 instructions / iteration)

109 instructions ÷ 10 instructions/iter = 108 iterations/second

Approx. 2 (100 instructions / iteration)

109 instructions ÷ 100 instructions/iter = 107 iterations/second

Constraint-Based Complexity Selection

Constraint on NAcceptable ComplexitiesNotes
N ≤ 20O(2N), O(N!)220 ≈ 106 — fine
N ≤ 100O(N³) and better100³ = 10⁶ — fine

N ≤ 104O(N²) — borderline, O(N log N) preferred10⁸ — tight

Remember Always read the constraints first! They tell you exactly how efficient your solution needs to be before you write a single line of code.

Quick Recap

Logarithm

logb(a) = power b must be raised to get a. The inverse of exponentiation.

Halving / Doubling

Any loop that halves or doubles its variable runs in O(log N) iterations.

Big-O Rules

Count iterations, drop lower-order terms, drop coefficients — keep only the dominant term.

Complexity Order

O(1) < O(log N) < O(√N) < O(N) < O(N log N) < O(N²) < O(N³) < O(2N)

Coefficients Don't Matter

Rate of growth, not constant multipliers, determines which algorithm is asymptotically better.

Same Big-O → Compare Directly

If two algorithms share the same complexity class, compare their actual iteration formulas.

TLE Budget

Assume 107–108 iterations per second. Choose algorithm from constraint on N.

Read Constraints First

The constraint on N is the single most important input to algorithm selection.
