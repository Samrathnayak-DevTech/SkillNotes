Importance of Constraints

Before writing a single line of code, always read the constraints on N. The constraint directly tells you the maximum acceptable Big-O complexity, given that a server runs roughly 107–108 iterations per second.

If 1 ≤ N ≤ 105, which complexities pass?

ComplexityIterationsResult
O(N³)(10⁵)³ = 10¹⁵✗ TLE
O(N² log N)(10¹⁰) · log 10⁵✗ TLE
O(N²)(10⁵)² = 10¹⁰✗ TLE
O(N log N)10⁵ · log 10⁵ ≈ 1.7×10⁶✓ Passes
O(N)10⁵✓ Passes

If 1 ≤ N ≤ 106, which complexities pass?

ComplexityIterationsResult
O(N³)10¹⁸✗ TLE
O(N²)10¹²✗ TLE
O(N log N)≈ 2×10⁷⚠ Maybe
O(N)10⁶✓ Passes

Small N — more aggressive complexities allowed

🔢

1 ≤ N ≤ 100

O(N³) passes — 10⁶ iterations. Even O(2N) might be feasible for very small N.

✦

1 ≤ N ≤ 20

O(2N) passes — 220 ≈ 10⁶. Exponential algorithms are acceptable.

∞

1 ≤ N ≤ 10¹²

Need O(√N), O(log N), or O(1). Linear is too slow (10¹² iterations).

Golden Rule Constraint → Complexity → Algorithm. Always determine the required Big-O before brainstorming the algorithm. This prevents wasted effort on approaches that are guaranteed to TLE.
02

How to Approach a Problem

A systematic 9-step framework prevents TLE and bugs before you write a line of code.

📖
Step 1: Read the question and constraints carefully.
💡
Step 2: Formulate an idea or logic.
✅
Step 3: Verify the correctness of the logic by tracing through a few examples by hand.
📝
Step 4: Mentally develop pseudocode or a rough idea of loops.
⏱
Step 5: Determine the time complexity based on the pseudocode.
🔍
Step 6: Assess whether the time complexity is feasible given the constraint. In the worst case, you can have at most 107–108 iterations.
🔁
Step 7: If the complexity is too high, re-evaluate the idea. Otherwise, proceed.
💻
Step 8: Code the solution.
Pro Tip Step 3 ("verify with examples") is the most skipped step and the biggest source of bugs in interviews. Always test your logic manually before touching the keyboard.

Space Complexity

Space complexity is the maximum memory (worst case) used at any point during the algorithm's execution. Like time complexity, we express it using Big-O notation. Note: memory used by the input itself is typically called input space and is usually not counted in the algorithm's space complexity — we count only the auxiliary space (extra memory the algorithm allocates).

Memory Sizes int → 4 bytes   |   long → 8 bytes   |   int[N] → 4·N bytes   |   long[N][N] → 8·N² bytes

Example 1 — O(1) space

java
void fun(int N) {
    int  x = N;        // 4 bytes
    int  y = 2 * x;    // 4 bytes
    long z = x + y;   // 8 bytes
}
// Total = 16 bytes → constant → S.C = O(1)

Example 2 — O(N) space

java
void func(int N) {
    int[] a   = new int[10];   // 40 bytes (constant)
    int   x   = 0;              // 4 bytes
    int   y   = 0;              // 4 bytes
    long  z   = 0;              // 8 bytes
    int[] arr = new int[N];    // 4·N bytes  ← dominates
}
// Total = 4N + 56 → S.C = O(N)

Example 3 — O(N²) space

java
void fun(int N) {
    int    x   = N;                 // 4 bytes
    int    y   = x * x;             // 4 bytes
    long   z   = y + y;             // 8 bytes
    int[]  arr = new int[N];       // 4·N bytes
    long[][] b = new long[N][N];  // 8·N² bytes  ← dominates
}
// Total ≈ 8N² + 4N + 16 → S.C = O(N²)

Example 4 — Input space is NOT counted

java
int maxArr(int[] arr, int N) {
    // arr[] is input space — given, not created by us
    int ans = arr[0];   // 4 bytes extra
    for (int i = 0; i < N; i++) {
        ans = Math.max(ans, arr[i]);
    }
    return ans;
}
// T.C = O(N)   S.C = O(1)  ← only 1 extra variable

Introduction to Arrays

An array is a collection of elements of the same data type, stored in contiguous memory. Because memory is contiguous, accessing any element by index is instantaneous — O(1).

Memory Layout If an int[] array starts at address 1000, then element at index i is at address 1000 + i × 4. This formula is why random access is O(1) — no searching needed.

Declaring an Array

java / c++ / python
// Java
int[] arr = new int[10];   // size 10, zero-initialized

// C++
int arr[10];               // size 10 (values uninitialized)

// Python (list)
arr = [0] * 10            # size 10, all zeros

Code

java · O(N) time · O(1) space
void reverse(int[] arr, int N) {
    int l = 0, r = N - 1;
    while (l < r) {
        // swap arr[l] and arr[r]
        int temp = arr[l];
        arr[l]   = arr[r];
        arr[r]   = temp;
        l++;
        r--;
    }
    // Iterations = N/2  →  T.C = O(N)   S.C = O(1)
}
Time O(N) Space O(1) Swaps N/2

void reverse(int[] arr, int l, int r) {
    while (l < r) {
        int temp = arr[l];
        arr[l]   = arr[r];

        l++; r--;

Rotate an Array by K — O(N×K) vs O(N)

Right-rotating an array by K means moving the last K elements to the front. For example, rotating [9, 6, 2, 8, 3, 5, 4, 1] by K=5 gives [8, 3, 5, 4, 1, 9, 6, 2].

Edge Case If K ≥ N, rotating by K is the same as rotating by K % N (since rotating N times returns to the original). Always normalise: K = K % N.

Approach 1 — Brute Force: O(N×K)

Repeat K times: save the last element, shift all others right by one, place the saved element at index 0.

java · O(N×K)
void rotateBrute(int[] arr, int N, int K) {
    K = K % N;   // normalise
    for (int i = 0; i < K; i++) {
        int temp = arr[N - 1];        // save last element
        for (int j = N - 2; j >= 0; j--) {
            arr[j + 1] = arr[j];       // shift right
        }
        arr[0] = temp;                // insert at front
    }
    // T.C = O(N×K)   S.C = O(1)
}
Problem With N ≤ 10⁵ and K ≤ 10⁵, the brute force reaches N×K = 10¹⁰ iterations — a guaranteed TLE.

Approach 2 — Optimal: Three Reverses O(N)

A beautiful observation: right-rotating by K is equivalent to three reversal operations.

0 Normalise: K = K % N
1 Reverse the entire array: reverse(arr, 0, N-1)
2 Reverse the first K elements: reverse(arr, 0, K-1)
3 Reverse the remaining N-K elements: reverse(arr, K, N-1)

java · O(N) optimal
void rotate(int[] arr, int N, int K) {
    K = K % N;                // handle K >= N
    reverse(arr, 0, N - 1);   // step 1: reverse all
    reverse(arr, 0, K - 1);   // step 2: reverse first K
    reverse(arr, K, N - 1);   // step 3: reverse rest
    // T.C = O(N)   S.C = O(1)
}
Brute Time O(N×K) Optimal Time O(N) Space O(1)

Dynamic Arrays

Drawback of Static Arrays

Static arrays have a fixed size that must be declared before use. If you don't know the size upfront, or if the array needs to grow, static arrays are inflexible and wasteful.

🔒

Fixed Size

You must declare the size in advance. Allocating too much wastes memory; too little causes overflow.

💥

No Auto-Resize

Adding more elements than the declared size causes out-of-bounds errors at runtime.

Dynamic Arrays — Automatic Resizing

Dynamic arrays (called ArrayList in Java, vector in C++, and list in Python) automatically resize when they run out of capacity. Internally, when full, a new array of double the size is allocated and all elements are copied over.

How Resizing Works Imagine a hotel that doubles its rooms every time it fills up. Individual check-ins (appends) are O(1) most of the time. Occasionally the hotel rebuilds — that's the copy step, O(N). On average, each append costs O(1) amortized.

Quick Recap

Read Constraints First

Constraint on N tells you the required Big-O before you think about algorithms.

9-Step Problem Approach

Read → Ideate → Verify → Pseudocode → Complexity → Feasibility check → Rethink/Code.

Space Complexity

Worst-case memory at any point. Input space is usually excluded; only auxiliary space counts.

Array Access

Random access is O(1) due to contiguous memory. Address of arr[i] = base + i × element_size.

Reverse — Two Pointers

l=0, r=N-1; swap and converge. O(N) time, O(1) space, N/2 swaps.

Rotate — Three Reverses

Normalise K, then: reverse all → reverse [0,K-1] → reverse [K,N-1]. O(N) time, O(1) space.

K % N Normalisation

Rotating N times = no change. K = K % N handles all cases including K ≥ N.

Dynamic Arrays

ArrayList / vector / list auto-resize. Append is O(1) amortized; access is O(1).
