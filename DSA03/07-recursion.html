01Power Function — Naive Approach

Given a base a and exponent n (where n ≥ 0), compute aⁿ using recursion.

Idea 1 — Reduce Exponent by 1 Each Step

Observe that aⁿ = a × a × … × a (n times). The recursive relationship is:

aⁿ = aᴺ⁻¹ × a power(a, N) = power(a, N−1) × a power(a, 0) = 1 ← base case (a⁰ = 1 for all a)
C++ — Idea 1 (Naive)
// Assumption: power(a, n) returns aⁿ
int power(int a, int n) {
    if (n == 0) return 1;          // Base case: a⁰ = 1
    return power(a, n - 1) * a;    // Recursive step
}

Dry Run — power(3, 4)

power(3, 0)returns 1
→ 1 × 3 = 3
power(3, 1)power(3,0) × 3 → 3
→ 3 × 3 = 9
power(3, 2)power(3,1) × 3 → 9
→ 9 × 3 = 27
power(3, 3)power(3,2) × 3 → 27
→ 27 × 3 = 81
power(3, 4)power(3,3) × 3 → 81

Complexity

TimeO(N) SpaceO(N)

N + 1 recursive calls, each doing O(1) work. Stack depth is N + 1.

02Fast Power — O(log N)

Idea 1 is correct but slow for large exponents. We can exploit the observation that multiplying a number by itself is equivalent to squaring — and squaring halves the exponent each time.

Key Insight — Halving the Exponent

If N is even: aᴺ = aᴺ/² × aᴺ/² If N is odd: aᴺ = aᴺ/² × aᴺ/² × a

Examples:

2¹⁶ = 2⁸ × 2⁸
3⁶⁴ = 3³² × 3³² ← N even: square the half-result
2¹⁷ = 2⁸ × 2⁸ × 2
3¹⁵ = 3⁷ × 3⁷ × 3 ← N odd: square the half-result, then × a

Idea 2 — Two Recursive Calls (Redundant)

C++ — Idea 2
int power(int a, int N) {
    if (N == 0) return 1;
    if (N % 2 == 0) {
        return power(a, N/2) * power(a, N/2);  // two calls — wasteful!
    } else {
        return power(a, N/2) * power(a, N/2) * a;
    }
}
Problem with Idea 2 Two calls are made with identical arguments — the same sub-problem is computed twice at every level, producing a binary tree of calls. Total calls ≈ 2^(log₂N) = N, giving O(N) time — no better than Idea 1.

Optimized Approach — Store the Half-Result

The fix is simple: compute power(a, N/2) once, store it in a variable p, then reuse it. This keeps the recursion linear (one call per level).

C++ — Optimized O(log N)
// Assumption: power(a, N) returns aᴺ in O(log N) time
int power(int a, int N) {
    if (N == 0) return 1;

    int p = power(a, N / 2);   // Compute ONCE — key optimisation

    if (N % 2 == 0) {
        return p * p;           // Even exponent
    } else {
        return p * p * a;       // Odd exponent: one extra factor of a
    }
}

Why O(log N)?

Each recursive call halves the exponent: N → N/2 → N/4 → … → 1 → 0. The number of levels in this chain equals log₂(N). Since only one call is made per level, total function calls = log₂(N). The total call count from the geometric series is:

1 + 2 + 4 + … (log₂N terms) = 2^(log₂N) − 1 = N − 1 ≈ O(N) ← Idea 2 (bad) Optimized: 1 call per level × log₂N levels = O(log N) ← Idea 3 (good)

Dry Run — power(2, 17) with optimized version

power(2, 0)returns 1
→ p=1, odd: 1×1×2 = 2
power(2, 1)p=1, odd → 2
→ p=2, even: 2×2 = 4
power(2, 2)p=2, even → 4
→ p=4, odd: 4×4×2 = 32
power(2, 4) → n=4? wait — n=17→8

For N=17: calls are power(2,17) → power(2,8) → power(2,4) → power(2,2) → power(2,1) → power(2,0). That is only 5 levels instead of 17 — the logarithmic depth in action.

Complexity

TimeO(log N) SpaceO(log N)

Comparison

Naive — Idea 1

Decrements exponent by 1 each call. N+1 calls total. Slow for large N.

T: O(N)   S: O(N)

Optimized — Idea 3

Halves exponent each call, stores result in p. log₂(N) calls total.

T: O(log N)   S: O(log N)

Real-World Use This algorithm is called Binary Exponentiation (or fast exponentiation). It is used in cryptography (RSA, modular exponentiation), competitive programming, and standard library implementations of Math.pow.

03Print Array Using Recursion

Given an array of size N, print all elements using recursion — no loops. The idea is to carry an index as a parameter: print the current element, then recurse with idx + 1.

Example Array

5
0
2
1
7
2
3
3
4
4

arr[N=5] = [5, 2, 7, 3, 4]

Java/C++ — Print Array
// Assumption: printArr(arr, idx) prints arr[idx], arr[idx+1], ..., arr[N-1]
void printArr(int[] arr, int idx) {
    if (idx == arr.length) return;  // Base case: past the last element
    print(arr[idx]);              // ① Print current element (going down)
    printArr(arr, idx + 1);       // ② Recurse on the rest
}

// Call: printArr(arr, 0)  →  Output: 5, 2, 7, 3, 4

Complexity

TimeO(N) SpaceO(N)
Pattern — Index Threading Passing an idx parameter is the standard way to traverse an array recursively. The base case is always idx == arr.length. This pattern appears in all recursive array problems: print, find max, find sum, search, etc.

04Find Maximum & Sum of Array

The index-threading pattern extends naturally to compute aggregates over an array. Two classic examples:

Find Maximum Element

The maximum of an array from index idx onward is the larger of arr[idx] and the maximum of the rest.

Java — Find Max
// Assumption: findMax(arr, idx) returns the maximum of arr[idx..N-1]
int findMax(int[] arr, int idx) {
    if (idx == arr.length - 1) return arr[idx]; // Base: single element
    return Math.max(arr[idx], findMax(arr, idx + 1));
}

Find Sum of All Elements

The sum from index idx onward equals arr[idx] plus the sum of the rest.

Java — Find Sum
// Assumption: sum(arr, idx) returns arr[idx] + arr[idx+1] + … + arr[N-1]
int sum(int[] arr, int idx) {
    if (idx == arr.length) return 0;   // Base: empty range sums to 0
    return arr[idx] + sum(arr, idx + 1);
}

Complexity (both)

TimeO(N) SpaceO(N)

Strategy — Count First, Fill on the Way Back

The key challenge: we need to allocate the result array, but we don't know how many matches exist until we've scanned everything. The recursive approach solves this elegantly:

Going Down (forward pass) — Count how many times B appears by incrementing count whenever arr[idx] == B.
At the base case — Allocate a new int[count] result array of the exact required size.
Coming Back Up (backward pass) — Fill the result array by placing each matched index at its correct position using res[count−1] = idx.
Java — All Indices
// Assumption: getAllIndices returns all indices where arr[idx..N-1] equals B
int[] getAllIndices(int[] arr, int idx, int count, int B) {

    // Base case: past end of array — allocate result of exact size
    if (idx == arr.length) {
        int[] res = new int[count];
        return res;
    }

    // Count matches on the way down
    if (arr[idx] == B) count++;

    // Recurse forward
    int[] res = getAllIndices(arr, idx + 1, count, B);

    // Fill on the way back up
    if (arr[idx] == B) {
        res[count - 1] = idx;
        count--;   // decrement local count to fill next position
    }

    return res;
}

// Call: getAllIndices(arr, 0, 0, B)
Why This Works When we return from the base case, count holds the total number of matches. As we unwind, each matched index fills position count−1 from right to left. Because the unwinding goes right-to-left (last index first), and we decrement count after each fill, indices end up in the correct ascending order in the result array.

Complexity

TimeO(N) SpaceO(N)

06Check Palindrome

Given a string, determine recursively whether it is a palindrome — meaning it reads the same forwards and backwards.

Examples

✓
"radar" → true
r=r, a=a, center d — palindrome.
✗
"area" → false
a≠a but r≠e — fails inner check.

Two-Pointer Recursive Strategy

Use two pointers l (left) starting at 0 and r (right) starting at N−1. At each step:

Base case: if l ≥ r, the entire string has been checked — return true.
Mismatch check: if str[l] ≠ str[r], return false immediately.
Recurse inward: return isPalindrome(str, l+1, r−1).

Visual — "radar"

Pointers move inward; matching characters are highlighted green.

r
a
d
a
r
l=0 l=1 mid r=3 r=4
Java — isPalindrome
// Assumption: isPalindrome(str, l, r) checks if str[l..r] is a palindrome
boolean isPalindrome(String str, int l, int r) {
    if (l >= r) return true;            // Base: pointers crossed — done

    if (str.charAt(l) != str.charAt(r)) {
        return false;                   // Mismatch found
    }

    return isPalindrome(str, l + 1, r - 1); // Check inner substring
}

// Call: isPalindrome(str, 0, str.length() - 1)

Dry Run — "abcba"

isPal("abcba", l=2, r=2)l≥r → true
b == b ✓ → recurse inward
isPal("abcba", l=1, r=3)b==b → true
a == a ✓ → recurse inward
isPal("abcba", l=0, r=4)a==a → true

Complexity

TimeO(N) SpaceO(N)

At most N/2 recursive calls (pointers meet in the middle), each doing O(1) work.

07Print Before & After — Tracing Both Phases

This is a tracing exercise that reinforces the "before/after recursive call" observation from Session 1. What happens when you print both before and after the recursive call?

C++ — solve(N)
void solve(int N) {
    if (N == 0) return;
    print(N);     // ① Before recursive call — executes going DOWN
    solve(N - 1);
    print(N);     // ③ After recursive call — executes coming BACK UP
}

Output for solve(3)

↓ downsolve(3) — print ①3
↓ downsolve(2) — print ①2
↓ downsolve(1) — print ①1
⊥ basesolve(0) — return—
↑ upsolve(1) — print ③1
↑ upsolve(2) — print ③2
↑ upsolve(3) — print ③3

Output: 3, 2, 1, 1, 2, 3

Analogy — Echo in a Canyon Shouting numbers into a canyon: you hear them once as the sound travels out (going down), and again as the echo returns (coming back up) — in reverse order.

Why This Matters

This pattern — doing work both before and after the recursive call — is the foundation of:

  • In-order tree traversal (left subtree → visit node → right subtree)
  • Merge Sort (split going down, merge coming back up)
  • Backtracking (choose before recurse, undo after recurse)

08Stack Overflow Pitfall

What happens when the base case is never reached? Consider calling solve(N) from Section 07 with a negative value such as N = −3:

C++ — Dangerous Call
void solve(int N) {
    if (N == 0) return;  // Base case: only triggered when N == 0
    print(N);
    solve(N - 1);        // N = -3 → -4 → -5 → … never reaches 0!
}

solve(-3);  // → Error: Stack Overflow!
What Happens With N = −3, the call goes solve(−3) → solve(−4) → solve(−5) → … The base case N == 0 is never triggered because N moves away from 0, not toward it. The call stack keeps growing until the program runs out of memory — this is a Stack Overflow error.

How to Prevent Stack Overflow

🎯
Correct Base Case
Make sure the base condition is actually reachable from any valid input. For decreasing inputs, use N <= 0 instead of N == 0.
📉
Converging Input
Every recursive call must move the input closer to the base case. Verify this manually for all edge-case inputs.
🔍
Input Validation
Guard at the call site: reject invalid inputs before recursing. For power(a, N), assert N ≥ 0 before calling.

Fixed Version

C++ — Fixed
void solve(int N) {
    if (N <= 0) return;  // Handles N=0 AND negative N safely
    print(N);
    solve(N - 1);
}

★Quick Recap

Naive Power — O(N)
power(a,N) = power(a,N−1) × a. N+1 calls, each reducing exponent by 1.
Fast Power — O(log N)
Halve the exponent each call; store result in p to avoid duplicate work. Even: p×p; Odd: p×p×a.
Print Array
Thread an idx parameter; base case is idx == arr.length. Print before or after the call to control order.
Find Max / Sum
Return Max(arr[idx], findMax(arr, idx+1)) or arr[idx] + sum(arr, idx+1). Both O(N) time and space.
All Indices
Count matches going down; allocate array at base case; fill positions coming back up. O(N) time and space.
Check Palindrome
Two-pointer recursion: compare str[l] vs str[r], recurse with l+1 and r−1. Base: l≥r → true. O(N).
Before & After
Printing both before and after the call gives symmetric output (N,…,1,1,…,N). Foundation of backtracking.
Stack Overflow
Occurs when the base case is never reached. Always verify input moves toward the base case; use <= guards.
