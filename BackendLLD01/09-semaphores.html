01

What is a Semaphore?

A semaphore is a generalization of a MUTEX. Where a MUTEX has a count of exactly 1 (either locked or unlocked), a semaphore has an internal counter that can be initialized to any non-negative integer. It allows up to N threads to access a resource simultaneously.

Definition

A Semaphore is a MUTEX with a count greater than 1. It maintains an internal counter. acquire() decrements the counter (blocks if it would go below 0). release() increments the counter (waking a blocked thread if any are waiting).

FeatureMUTEXSemaphore
Internal countBinary: 0 or 1Integer ≥ 0 (initialized to N)
Concurrent accessExactly 1 threadUp to N threads

02

acquire() and release()

Pseudocode — Semaphore Internals
// acquire() — sometimes called wait() or P()
void acquire() {
    if (count > 0) {
        count--;          // resource available — take it
    } else {
        block();          // count == 0 — block the calling thread
    }
}

// release() — sometimes called signal() or V()
void release() {
    count++;              // return resource
    if (waitingThread) {
        wakeUp(waitingThread);  // notify a blocked thread
    }
}
Java
import java.util.concurrent.Semaphore;

// Semaphore initialized to 3 — allows up to 3 concurrent threads
Semaphore sem = new Semaphore(3);

sem.acquire();   // count: 3 → 2. Blocks if count == 0
    /* ... protected work ... */
sem.release();   // count: 2 → 3. Wakes a blocked thread if any

The Producer–Consumer Problem

The Producer–Consumer problem (also called the Bounded Buffer problem) is a classic concurrency scenario: multiple producer threads add items to a shared store, and multiple consumer threads remove items from it. The store has a fixed capacity.

∞ Producers → add items to store

Store (maxSize = 6 slots)
∞ Consumers → remove items from store

# of producers can produce = # of empty slots in the store (don't overfill)

# of consumers can consume = # of filled slots in the store (don't consume from empty store)

Two constraints must always hold: producers must not add to a full store, and consumers must not remove from an empty store. Both checks and modifications must be atomic — otherwise a race condition can breach these constraints.

04

Why MUTEX Alone Is Not Enough

A MUTEX (single synchronized block) prevents multiple threads from entering the critical section simultaneously — but it doesn't handle the coordination problem: a producer blocked because the store is full, or a consumer blocked because the store is empty.

With a plain MUTEX, a producer that finds the store full must either busy-wait (polling in a loop — bad) or use wait()/notifyAll() (complex, error-prone). Semaphores encode both the limit and the blocking/waking logic in a single clean abstraction.

The Problem with Only MUTEX

A producer checking if (store.size() < maxSize) and then calling store.add() has a TOCTOU (time-of-check to time-of-use) window. Another producer may add between the check and the add — overfilling the store — unless both operations are protected together. A semaphore atomically tracks remaining capacity.

05

Solving with Two Semaphores

The elegant solution uses two semaphores — one counting empty slots (for producers) and one counting filled slots (for consumers):

prods semaphore (empty slots)
6 → 5 → 4 → 3 → 2 → 1 → 0
conss semaphore (filled slots)
0 → 1 → 2 → 3 → ...

Initially: 6 empty slots (prods=6, conss=0). Each item produced decrements prods and increments conss. Each item consumed decrements conss and increments prods.

Producer Thread
prods.acquire()
← blocks if store is full (prods==0)
store.add(item)
conss.release()
← notifies consumers: 1 item available
Consumer Thread
conss.acquire()
← blocks if store is empty (conss==0)
store.remove()
prods.release()
← notifies producers: 1 slot free
06

Full Java Implementation

Java
import java.util.concurrent.*;
import java.util.*;

class ProducerConsumer {
    static final int      MAX_SIZE = 6;
    static List<String>  store    = new LinkedList<>();

    // prods = available empty slots (starts at MAX_SIZE)
    static Semaphore prods = new Semaphore(MAX_SIZE);
    // conss = available filled slots (starts at 0)
    static Semaphore conss = new Semaphore(0);

    static class Producer implements Runnable {
        private final String item;
        Producer(String item) { this.item = item; }

        @Override
        public void run() {
            try {
                prods.acquire();        // wait for an empty slot (prods--)
                synchronized (store) {
                    store.add(item);     // critical section
                }
                conss.release();        // signal: one more item available (conss++)
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        }
    }

    static class Consumer implements Runnable {
        @Override
        public void run() {
            try {
                conss.acquire();        // wait for a filled slot (conss--)
                synchronized (store) {
                    store.remove(0);   // critical section
                }
                prods.release();        // signal: one slot is free again (prods++)
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        }
    }
}
Why Both a Semaphore AND synchronized?

The semaphores handle coordination (blocking producers when full, blocking consumers when empty). The synchronized block on store prevents two producers (or two consumers) from modifying the list at the exact same time — which would cause a ConcurrentModificationException or data corruption even with the semaphore in place.

Quick Recap

Semaphore

MUTEX with count > 1. acquire() decrements (blocks at 0). release() increments (wakes a blocked thread).

Producer–Consumer

Producers add to a bounded store; consumers remove. Neither can violate store capacity or consume from an empty store.

Two-Semaphore Pattern

prods = empty slots (starts at maxSize). conss = filled slots (starts at 0). Producer acquires prods, releases conss. Consumer does the opposite.

acquire()

Wait for a permit. If count > 0: decrement and proceed. If count == 0: block the calling thread.

release()

Return a permit. Increment count. If threads are waiting, wake one up.

Semaphore + synchronized

Semaphore coordinates capacity; synchronized protects the actual list modification from concurrent corruption.
