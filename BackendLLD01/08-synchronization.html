Introduction to Synchronization

Why parallel threads produce wrong results when they share data, what a race condition is, and how Java's synchronized keyword enforces mutual exclusion.

Race ConditionCritical SectionPreemptionMutual ExclusionMUTEXsynchronized
01

The Race Condition

Consider a shared count variable initialized to 0. Two threads run concurrently: T1 adds numbers 1–100 to count; T2 subtracts numbers 1–100 from count. If both finish perfectly, the result should be 0. But in practice, it is often not.

The reason: a single count += i statement is not atomic at the machine level. It compiles to three CPU instructions — read, modify, write — and the CPU can context-switch between any two of them.

count = 0, T1 adds, T2 subtracts — interleaved execution:
①
T1: x ← count
x = 0, count = 0
②
T1: x ← x + 1
x = 1, count = 0
③
T2: y ← count
y = 0, count = 0
④
T2: y ← y - 1
y = -1, count = 0
⑤
T2: count ← y
count = -1
⑥
T1: count ← x
count = 1 ← WRONG! T2's write was overwritten
Synchronization Problem

When more than one thread is working on shared data at the same time, the result can be unpredictably wrong. This non-deterministic corruption is called a race condition — the output depends on the exact interleaving (race) of thread operations, which the OS scheduler controls non-deterministically.

02

Critical Section

The critical section is the segment of code where shared data is being read or written. When more than one thread enters the critical section simultaneously, a race condition can occur.

Definition

A critical section is any code that accesses shared mutable state. The goal of synchronization is to ensure that at most one thread executes the critical section at any given time.

Java
int count = 0;

// CRITICAL SECTION — both T1 and T2 try to execute this simultaneously:
count += i;   // read count, add i, write back
              // NOT atomic! Three separate CPU ops.
03

Preemption Inside the Critical Section

Preemption is when the CPU forcibly pauses a thread mid-execution to give another thread CPU time. This is normal and necessary for fairness — but if preemption happens while a thread is inside the critical section, the shared data is left in a partially-updated state that the next thread will read incorrectly.

Key Point

If a thread is preempted inside the critical section — having read the shared variable but not yet written back its update — and another thread enters the critical section during that gap, the second thread reads stale data. Both threads end up writing independent updates, and one of them is silently lost.

04

Four Properties of a Good Synchronization Solution

Property 1
Mutual Exclusion

Only one thread can be inside the critical section at a time. This is the most fundamental requirement.

Property 2
Progress

If no thread is in the critical section and some threads want to enter, one of them must be allowed to proceed. The system must not deadlock.

Property 3
Bounded Waiting

Once a thread requests to enter the critical section, it must eventually be granted access. No thread should wait indefinitely (starvation).

Property 4
No Busy Waiting

A thread waiting to enter the critical section should be blocked (sleeping), not spinning in a tight loop consuming CPU cycles.

Analogy — Trial Room

A fitting room in a store is the critical section — only one person at a time. Mutual exclusion is the lock on the door. Progress means the store attendant must eventually let a waiting customer in. Bounded waiting means nobody waits forever. No busy waiting means customers sit down rather than pacing back and forth in front of the door.

05

MUTEX — The Lock

A MUTEX (short for Mutual Exclusion) is a locking mechanism. A thread must acquire the lock before entering the critical section and release it afterward. If another thread tries to acquire a lock that is already held, it blocks until the lock is released.

Java
import java.util.concurrent.locks.ReentrantLock;

ReentrantLock lock = new ReentrantLock();
int count = 0;

// T1 — Adder
lock.acquire();   // or lock.lock() — blocks if already held
    x = count;
    x = x + 1;
    count = x;
lock.release();   // or lock.unlock() — wakes a waiting thread

// T2 — Subtractor (runs concurrently)
// T2 calls lock.acquire(); if T1 holds the lock, T2 blocks here
// Once T1 calls lock.release(), T2 gets the lock and proceeds
06

The synchronized Block

Java has a built-in locking mechanism: every object has an implicit monitor lock (intrinsic lock). The synchronized block lets you grab that lock for any object.

Java
Object count = new Object();  // used as the lock object
int value = 0;

// T1 — Adder
synchronized (count) {    // acquires count's monitor lock
    x = value;
    x = x + 1;
    value = x;
}                          // automatically releases lock on exit (even on exception)

// T2 — Subtractor
// synchronized(count) { ... } — tries to acquire same lock, blocks if T1 holds it

// You can nest synchronized blocks for different locks:
// synchronized(count)   { synchronized(student) { ... } }
// But be careful of deadlock when acquiring multiple locks!
Tip

synchronized blocks are preferred over synchronizing the entire method because they narrow the critical section to exactly the code that needs protection, allowing other non-critical code to run concurrently.

07

Synchronized Methods

You can also declare an entire method as synchronized. The lock used is the instance's own monitor (this). Only one thread can be inside any synchronized method of a particular object at a time.

Java
class Counter {
    private int value = 0;

    public synchronized void addV(int x)  { value += x; }  // sync on 'this'
    public synchronized void subV(int x)  { value -= x; }
    public int               getV()        { return value; }  // not synchronized
}

Counter c1 = new Counter();
Counter c2 = new Counter();

// ✗  T1: c1.addV() and T2: c1.subV() — BLOCKED (same object c1's lock)
// ✓  T1: c1.addV() and T2: c2.subV() — OK (different objects, different locks)
// ✓  T1: c1.addV() and T2: c1.getV() — OK (getV is not synchronized)
Key Rule

Only one synchronized method can execute on a given object at a time. Two threads can call synchronized methods on different objects simultaneously — each object has its own lock. Unsynchronized methods are always accessible regardless.

✦

Quick Recap

Race Condition

Multiple threads accessing shared data concurrently produce wrong results. The output depends on unpredictable thread scheduling.

Critical Section

Code that accesses shared mutable state. Must be protected so only one thread executes it at a time.

Preemption

OS pauses a thread mid-execution. If this happens inside the critical section, the next thread sees inconsistent data.

Mutual Exclusion

Only one thread in the critical section at a time. The core requirement of any sync solution.

MUTEX

A lock mechanism: acquire before entering critical section, release after. Blocking — waiting threads sleep, not spin.

synchronized block

Acquires the monitor lock of a specified object. Automatically released on exit. Narrower and preferred over method-level sync.

synchronized method

Locks the entire method on this. Only one sync method per object at a time. Non-sync methods remain accessible.

4 Properties

Mutual exclusion, progress (no deadlock), bounded waiting (no starvation), no busy waiting.
