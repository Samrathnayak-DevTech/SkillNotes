Interfaces, Abstract Classes & Static

The trio of Java constructs that separate what something does from how it does it — enabling loose coupling, clean inheritance hierarchies, and class-level shared state.

Interfaces Program to Interface Tight vs Loose Coupling Diamond Problem Abstract Classes Default Methods Static Keyword
01

What is an Interface?

Recall that a class is a blueprint of an entity — it captures both attributes (fields) and behaviours (methods). An interface goes one step further in abstraction: it is a blueprint of behaviours only. It says "anything that calls itself an Animal must be able to eat, walk, and breathe" — but says nothing about how.

Key Concept

An interface is a pure contract. It declares method signatures without providing bodies. Any class that implements the interface is legally bound to provide concrete implementations of every declared method.

In Java, interface methods are implicitly public abstract, and interface fields are implicitly public static final (constants). You cannot instantiate an interface directly.

Java 
interface Animal {
    void eat();       // implicitly public abstract
    void walk();
    void breathe();
}

// ✗ Cannot instantiate an interface
Animal a = new Animal();  // Compile error!
Analogy

Think of an interface as a job description posting. It lists everything the candidate must be able to do. A concrete class is the actual employee who shows up and performs those duties in their own style.

02

Implementing an Interface

A class uses the implements keyword. It must override every method in the interface; otherwise the compiler throws an error (unless the class is also declared abstract).

Java 
class Dog implements Animal {

    @Override
    public void eat() {
        System.out.println("Dog is eating");
    }

    @Override
    public void walk() {
        System.out.println("Dog is walking");
    }

    @Override
    public void breathe() {
        System.out.println("Dog is breathing");
    }
}

// ✓ You can create objects of implementing classes
Dog d = new Dog();

// ✓ You can also hold them through the interface reference
Animal a = new Dog();
a = new Cat();   // perfectly legal — reassign to another Animal
Tip

Always add the @Override annotation. It makes your intent explicit and causes a compile-time error if you accidentally misspell the method name — catching bugs early.

03

Program to Interface

This is one of the most important design principles you will ever learn. Instead of writing code that depends on a specific concrete class, depend on an interface. Your code becomes flexible, testable, and easy to extend.

Design Principle

Program to an interface, not an implementation. This is the "D" in SOLID (Dependency Inversion Principle) expressed informally. Your high-level modules should depend on abstractions, not on concrete classes.

Java 
interface Runner {
    void run();
}

// Dog, Cat, Tiger all implement Runner
// Now you can write a single method that works for ALL of them:

void startRace(List<Runner> runners) {
    for (Runner r : runners) {
        r.run();   // works for every Runner implementation
    }
}

// One list holds all animal types — no separate List<Dog>, List<Cat>...
List<Runner> runners = new ArrayList<>();
runners.add(new Dog());
runners.add(new Cat());
runners.add(new Tiger());

Without this principle you would need separate lists and methods for every subtype — a maintenance nightmare. With it, adding a new Horse class requires zero changes to startRace.

Real-World Example: The Stack Interface

A stack is a last-in, first-out (LIFO) data structure. The core behaviour is always push and pop — but there can be multiple implementations: array-based, linked-list-based, and so on. This is an ideal scenario for an interface.

User code depends only on «Stack»
«Stack»
+ push(x) : void
+ pop() : int
+ isEmpty() : boolean
▲ implements

int[] A;
int top = -1;
+ push, pop, isEmpty
LLBasedStack

Java 
interface Stack {
    void    push(int x);
    int     pop();
    boolean isEmpty();
}

class ArrayBasedStack implements Stack {
    private int[] A = new int[100];
    private int   top = -1;

    public void push(int x) { A[++top] = x; }
    public int  pop()       { return A[top--]; }
    public boolean isEmpty() { return top == -1; }
}

class LLBasedStack implements Stack {
    // linked-list-based internals...
    public void push(int x) { /* add to head */ }
    public int  pop()       { /* remove from head */ }
    public boolean isEmpty() { /* check head == null */ }
}

// ✗  Stack s = new Stack();     — can't instantiate interface
// ✓  Swap implementations freely:
Stack s = new ArrayBasedStack();  // or LLBasedStack()
s.push(10);
s.push(20);
s.pop();    // returns 20

The calling code uses only Stack. Changing from ArrayBasedStack to LLBasedStack requires editing exactly one line — the new call. That is the power of programming to an interface.

Java Collections & the List Interface

Java's own standard library is built on this exact principle. List<T> is an interface. ArrayList, LinkedList, and Vector are all concrete implementations.

Java 
// List is the interface — ArrayList is the implementation
List<Integer> l = new ArrayList<>();

// You can swap to LinkedList later with zero changes elsewhere:
List<Integer> l2 = new LinkedList<>();

LinkedList

Tip

Always declare variables as the interface type (List<T>) rather than the concrete type (ArrayList<T>). This makes it trivial to swap implementations without touching the rest of your code.

06

PhonePe Case Study: Tight vs Loose Coupling

This real-world example shows why programming to an interface is critical in production code. Imagine PhonePe needs to process bank transactions.

The Tight-Coupling Problem

PhonePe hard-codes a dependency on YesBank. The PhonePe class holds a field YB yb = new YesBank() and calls yb.transferMoney() and yb.checkBalance() directly. This is tight coupling:

❌ Tight Coupling
class PhonePe {
  YesBank yb = new YesBank();
  sendMoney(src,dest,x) {
    yb.transferMoney(src,dest,x);
  }
}

If YesBank fails or you want to add ICICI, you must rewrite PhonePe.

✓ Loose Coupling
class PhonePe {
  BankAPI api = new ICICI();
  sendMoney(src,dest,x) {
    api.transferMoney(src,dest,x);
  }
}

PhonePe only knows about BankAPI. Swap any bank with one line change.

Java 
// The contract PhonePe depends on
interface BankAPI {
    void checkBalance(String account);
    void transferMoney(String src, String dest, double amount);
}

// Multiple banks implement the same contract
class YesBank  implements BankAPI { /* ... */ }
class ICICI    implements BankAPI { /* ... */ }
class SBI      implements BankAPI { /* ... */ }
class HDFC     implements BankAPI { /* ... */ }

class PhonePe {
    BankAPI api = new ICICI();  // swap to any bank here

    void sendMoney(String src, String dest, double x) {
        api.transferMoney(src, dest, x);
    }

    void getAllBalance(String account) {
        api.checkBalance(account);
    }
}
Key Insight

Loose coupling = PhonePe depends on the BankAPI interface, not on any specific bank. This is exactly "Program to Interface" applied to a real payment system. Adding HDFC requires creating a new HDFC class — not touching PhonePe at all.

07

The Diamond Problem & Multiple Interface Implementation

Java does not support multiple inheritance for classes. The reason is the Diamond Problem: if class X extends both A and B, and both A and B define fun() with different bodies, which version does X inherit? The compiler cannot decide.

Z — fun() { "Hello" }
↙ ↘
A — fun() { "Hi" }
B — fun() { "Hey" }
↖ ↗
X extends A, B → ERROR 💥

Which fun() does X.fun() call? Ambiguous → compile error.

Java 
// ✗ Multiple class inheritance — not allowed in Java
class X extends A, B { }   // Compile error

// ✓ Multiple interface implementation — perfectly fine!
class X implements A, B {
    // X must provide its OWN body for all methods from A and B
    // No ambiguity — X defines exactly one implementation
}

Interfaces sidestep the Diamond Problem entirely because their methods have no body (prior to Java 8). When X implements A and B, X writes its own implementation — there is nothing to be ambiguous about.

Important

Interface fields are implicitly public static final. This means all constants declared in an interface are shared across all implementing classes and cannot be reassigned. They are effectively named constants.

08

Abstract Classes

An abstract class sits between a regular class and an interface. It can have fields (attributes), constructors, and concrete method implementations — but it can also declare abstract methods that subclasses must implement. Like an interface, you cannot instantiate an abstract class directly.

Key Rule

If a class contains even one abstract method, the class itself must be declared abstract. A concrete subclass must implement all inherited abstract methods — or declare itself abstract too.

Java 
abstract class Animal {
    // Concrete fields — shared by all subclasses
    String name;
    int age;

    // Abstract methods — subclasses MUST implement these
    abstract void eat();
    abstract void walk();

    // Concrete method — provides a default implementation
    void breathe() {
        System.out.println(name + " is breathing"); // default impl
    }
}

// Dog provides bodies for eat() and walk()
class Dog extends Animal {
    void eat()  { System.out.println("Dog is eating"); }
    void walk() { System.out.println("Dog is walking"); }
}

// abstract Cat still leaves walk() abstract — also must be declared abstract
abstract class Cat extends Animal {
    void eat() { System.out.println("Cat is eating"); }
    abstract void walk();   // still abstract — Tiger must implement it
}

10

Default Methods in Interfaces (Java 8+)

Before Java 8, adding a new method to a widely-used interface would break every class that implemented it — hundreds or even thousands of classes. Java 8 solved this with default methods: concrete method implementations inside an interface that all implementing classes inherit automatically.

Why It Matters

Java 8 introduced default methods specifically to allow the Collections Framework to be extended (e.g., adding forEach, stream()) without breaking the thousands of existing classes that implemented List, Collection, etc.

Java 
interface X {
    void abstractMethod();   // must be implemented by all

    default void fun() {
        System.out.println("Default implementation in interface");
        // Classes A, B, C, D, E ... all inherit this for free
        // They can still @Override it if they need custom behaviour
    }
}

// A, B, C, D, E ... implement X
// They all get fun() without any code changes!
class A implements X {
    public void abstractMethod() { /* ... */ }
    // fun() inherited automatically — no need to override
}

class B implements X {
    public void abstractMethod() { /* ... */ }

    @Override
    public void fun() {
        System.out.println("B's custom implementation");
    }
}

The Static Keyword

The keyword static means "belongs to the class, not to any particular object." Static members are initialized at class-loading time (essentially compile/load time), whereas regular instance members are created when an object is instantiated at runtime.

Class Level (static)
Shared by ALL objects of the class.
Initialized once when the class loads.
Accessed via ClassName.member.

Examples: Math.PI, Integer.MAX_VALUE
Instance Level (non-static)
Each object gets its own copy.
Initialized when new is called at runtime.
Accessed via objectRef.member.

Examples: student.name, student.age

Static Attributes

A static field is stored once in the class area of memory, not in each object's heap space. All instances read and write the same memory location. The classic use case is a counter that tracks how many objects have been created.

Java 
class Student {
    String name;           // instance field — each object has its own
    static int count = 0;   // class field — shared by ALL Student objects

    Student(String n) {
        this.name = n;
        count++;   // increments the single shared counter
    }
}

Student s1 = new Student("Amit");
Student s2 = new Student("Priya");
System.out.println(Student.count);  // prints 2

Static Methods

A static method belongs to the class and can be called without creating an object. public static void main(String[] args) is the most common example — the JVM calls it directly on the class, before any object exists.

Java 
class Student {
    String name;
    static int count = 0;

    void solveAssignment() {
        System.out.println(name + " is solving");
    }

    static void staticMethod() {
        System.out.println("Total students: " + count);
    }
}

Student s = new Student();
s.solveAssignment();          // instance method — called on object
Student.staticMethod();         // static method  — called on class

Static Rules & Common Gotchas

Can a static method access non-static members?

No. A static method has no implicit this reference — it does not belong to any object. So it cannot access instance fields or call instance methods directly. The fix is to pass an object reference as a parameter.

Java 
class Example {
    String name = "Alice";   // instance field
    static int count = 0;    // static field

    static void staticMethod() {
        System.out.println(count);  // ✓ OK — static accessing static
        // System.out.println(name); ✗ Compile error — can't access instance field
    }
}

// Correct invocation:
Example.staticMethod();   // via class name
Common Gotcha

Objects are created at runtime (when new is called on the heap). Static attributes are initialized at class-load time, which happens before any object is created. This is why main() must be static — no object exists yet when the JVM starts.

Static Classes (Nested)

Java supports static nested classes — classes declared inside another class with the static modifier. A static nested class does not hold a reference to an instance of the outer class. This is commonly used in the Builder design pattern, where SomeClass.Builder is a static inner class that constructs SomeClass instances step by step.

Java 
class Student {
    private final String name;
    private final int    age;

    private Student(Builder b) { this.name = b.name; this.age = b.age; }

    static class Builder {      // static inner class — Builder pattern
        String name;
        int    age;

        Builder name(String n) { this.name = n; return this; }
        Builder age(int    a) { this.age  = a; return this; }
        Student build()        { return new Student(this); }
    }
}

Student s = new Student.Builder()
                .name("Deepak")
                .age(30)
                .build();
✦

Quick Recap

Interface

A pure behavioural contract — no fields (only constants), no constructors, all methods abstract by default. Use implements.

Program to Interface

Declare variables and parameters as interface types so any implementation can be swapped with one line change.

Loose Coupling

When your class depends on an interface rather than a concrete class, swapping implementations requires no changes to the calling code.

Diamond Problem

Java bans multiple class inheritance to avoid ambiguity. Multiple interface implementation is allowed because X supplies its own method bodies.

Abstract Class

A partial class — may have fields, constructors, concrete methods, and abstract methods. Cannot be instantiated. Use extends.

Default Methods (Java 8+)

Concrete method implementations inside an interface. Let library authors add new methods without breaking existing implementors.

Static Attribute

A class-level field shared by all instances. Initialized at class-load time, not per-object. Accessed via ClassName.field.

Static Method

Belongs to the class, has no this. Cannot access instance fields. The entry point main() is static because no object exists yet.

Static Class (Nested)

A class nested inside another with static. Holds no outer-instance reference. Cornerstone of the Builder design pattern.
