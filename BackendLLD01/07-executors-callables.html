Executors, Callables & Futures

Why creating raw threads is wasteful, how thread pools fix that, and how to get return values from concurrent tasks using Callable and Future.

ExecutorServiceThread PoolCallableFutureMerge Sort
01

The Thread-Reuse Problem

Creating and destroying threads is expensive. A thread needs its own stack (typically 512 KB–1 MB), OS registration, and scheduler bookkeeping. If you spin up a new thread for every task and discard it afterward, you waste CPU time and memory — especially when tasks are short-lived.

Analogy — Tata Motors Assembly Line

Imagine Tata Motors builds a brand-new factory line for each individual Nexon car — then demolishes it after that one car is made. Absurd. Instead, they build a set of manufacturing lines once and reuse them for every car that comes through. The line is the thread; the car is the task.

Principle

Ideally, we should try to reuse threads as much as possible, assigning new tasks to idle threads rather than creating new threads for every task. This is the core idea behind the thread pool.

ExecutorService & Thread Pool

ExecutorService is Java's framework for managing a pool of reusable threads. Instead of creating threads manually, you submit tasks to the executor — it handles all thread lifecycle management for you.

When a thread finishes its task, it picks up the next one from the waiting queue rather than being destroyed.

Java
import java.util.concurrent.*;

// Create a fixed thread pool with 4 threads
ExecutorService executor = Executors.newFixedThreadPool(4);

// Submit 100 tasks — only 4 threads do the work, reusing as tasks complete
for (int i = 1; i <= 100; i++) {
    final int num = i;
    executor.submit(() -> System.out.println("Printing: " + num));
}

executor.shutdown();  // no new tasks; waits for current tasks to finish

03

Using ExecutorService

Java
ExecutorService executor = Executors.newFixedThreadPool(4);

// submit() with a Runnable (no return value)
executor.submit(() -> System.out.println("Hello from pool"));

// execute() — equivalent, but doesn't return a Future
executor.execute(() -> System.out.println("Another task"));

// Always shut down when done to release thread resources
executor.shutdown();           // graceful — waits for running tasks
executor.shutdownNow();        // forceful — interrupts running tasks
executor.awaitTermination(10, TimeUnit.SECONDS);  // wait up to 10s
04

Callable vs Runnable

Runnable's run() returns void — once you submit it, you have no way to get a result back. Callable<T> solves this: its call() method returns a value of type T and can also throw checked exceptions.

FeatureRunnableCallable<T>
Methodrun()call()
Return typevoidGeneric T (any type)
Throws checked exceptions?NoYes — throws Exception
Submit to executor viasubmit() or execute()submit() only (returns Future<T>)
Java
class SumTask implements Callable<Integer> {
    private final int[] arr;

    SumTask(int[] arr) { this.arr = arr; }

    @Override
    public Integer call() {
        int sum = 0;
        for (int x : arr) sum += x;
        return sum;          // ✓ returns a value
    }
}
05

Future — Non-Blocking Results

When you submit a Callable to an executor, it returns a Future<T> immediately — without blocking. The Future is a promise: a placeholder for the result that will eventually be available. You can continue doing other work and retrieve the result later by calling future.get().

Analogy

You send your friend (M) to pick up a package for person (B). Your friend says "I'll get it" and leaves — that's the Future. You don't stand at the door waiting (blocking). You go do other things. When you need the package, you call your friend and ask "got it yet?" That's future.get().

Java
ExecutorService executor = Executors.newFixedThreadPool(2);

Callable<Integer> task1 = new SumTask(new int[]{1,2,3});
Callable<Integer> task2 = new SumTask(new int[]{4,5,6});

// submit() returns immediately (non-blocking) — tasks run in parallel
Future<Integer> f1 = executor.submit(task1);
Future<Integer> f2 = executor.submit(task2);

// Do other work here while tasks run concurrently...

// get() BLOCKS until result is available
int result1 = f1.get();   // 6
int result2 = f2.get();   // 15
System.out.println(result1 + result2);  // 21

executor.shutdown();
Key Methods on Future

get() — blocks until result is ready (throws InterruptedException, ExecutionException).
get(timeout, unit) — blocks up to a timeout; throws TimeoutException if result not ready.
isDone() — returns true if task completed (non-blocking check).
cancel(true) — attempts to interrupt the running task.

06

Case Study: Multithreaded Merge Sort

Merge sort is a perfect candidate for parallelism — the two halves are completely independent and can be sorted simultaneously. With Callable and Future, we can assign each half to a separate thread pool and merge the results.

4 | 1 | 8 | 7 | 2 | 6 | 3 | 5 Pool-1
4 | 1 | 8 | 7 T₁ (Pool-2)
2 | 6 | 3 | 5 T₂ (Pool-2)
1 | 4 | 7 | 8 + 2 | 3 | 5 | 6
↓ Merge → 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8
Java
class MergeSorter implements Callable<List<Integer>> {
    private final List<Integer> list;
    private final ExecutorService executor;

    MergeSorter(List<Integer> list, ExecutorService executor) {
        this.list = list; this.executor = executor;
    }

    @Override
    public List<Integer> call() throws Exception {
        if (list.size() <= 1) return list;

        int mid = list.size() / 2;
        List<Integer> leftHalf  = list.subList(0, mid);
        List<Integer> rightHalf = list.subList(mid, list.size());

        // Submit both halves to the thread pool — they run in parallel!
        Future<List<Integer>> leftFuture  = executor.submit(new MergeSorter(leftHalf,  executor));
        Future<List<Integer>> rightFuture = executor.submit(new MergeSorter(rightHalf, executor));

        // Collect results (get() blocks until each half is sorted)
        List<Integer> leftSorted  = leftFuture.get();
        List<Integer> rightSorted = rightFuture.get();

        return merge(leftSorted, rightSorted);
    }

    private List<Integer> merge(List<Integer> l, List<Integer> r) {
        List<Integer> result = new ArrayList<>();
        int i = 0, j = 0;
        while (i < l.size() && j < r.size())
            result.add(l.get(i) <= r.get(j) ? l.get(i++) : r.get(j++));
        while (i < l.size()) result.add(l.get(i++));
        while (j < r.size()) result.add(r.get(j++));
        return result;
    }
}

Quick Recap

Thread Reuse

Creating/destroying threads is expensive. Reuse threads via a pool for efficient resource utilization.

ExecutorService

Manages a thread pool. Submit tasks via submit() or execute(). Always call shutdown() when done.

Thread Pool

A fixed set of reusable worker threads. Idle threads pick up queued tasks. Prevents thread explosion.

Callable<T>

Like Runnable but returns a value via call(). Can throw checked exceptions. Use when you need the task's result.

Future<T>

Returned by executor.submit(callable). Non-blocking submission. get() blocks until result ready.

Multithreaded Merge Sort

Split array in two halves, submit each to executor, get both futures, then merge. Left and right halves sort in parallel.
