Generics

Why using Object for reusable classes loses type safety, how generics (introduced in Java 5) add compile-time type checking, and what happens to all those type parameters at runtime.

01

The Pair Problem

Suppose you need a reusable Pair class — a container for two related values. You might use it for coordinates (x, y), for a student's (name, marks), or for any two values that belong together. The naive first attempt uses fixed types:

Java — Naive Approach
class Pair {
    Double first;
    Double second;
}

Pair coord = new Pair();
coord.first  = 8.17;
coord.second = 7.14;

// But what about Pair for student marks?
// You'd need a whole new class — Marks { String name; Double marks; }
// This creates code duplication for every new type combination.

The problem: a hard-coded Pair only works for one type combination. Every new combination requires a new class.

02

Using Object — Generality Without Safety

Since every Java class extends Object, you can use Object fields to accept anything. This gives you generality, but you lose all compile-time type checking — you can accidentally assign a String to a field that should hold a Double, and the compiler won't warn you.

Java — Object Approach (Unsafe)
class Pair {
    Object first;
    Object second;
}

Pair coord = new Pair();
coord.first  = 8.79;
coord.second = 6.14;
coord.first  = "Scaler";  // ✗ No compile error! But semantically wrong.

// To get the value back you need an explicit cast — which can fail at runtime:
Double x = (Double) coord.first;  // ClassCastException if first holds a String!
The Problem with Object

Object provides generality but NO compile-time safety. Bugs only surface at runtime as ClassCastException — the worst possible time to find them.

03

Generic Classes — The Right Solution

Generics (introduced in Java 5) allow you to define a class with type parameters — placeholders for data types that are filled in at the use site. You get both generality (works with any type) and compile-time safety (wrong types are caught by the compiler).

❌ Fixed Types

Hard-coded type. Must duplicate for every new type combination.

⚠ Object

Any type accepted. No compile-time safety. Casting errors at runtime.

✓ Generics

Works with any type AND catches type mistakes at compile time.

Java — Generic Pair<T, U>
// T and U are type parameters — placeholders, like variables for types
class Pair<T, U> {
    T first;
    U second;

    T getFirst()          { return first; }
    void setFirst(T val)  { this.first = val; }
    U getSecond()         { return second; }
    void setSecond(U val) { this.second = val; }
}
Convention

Common type parameter names: T = Type, U / V = second/third type, E = Element (collections), K = Key, V = Value. You'll see these everywhere in the Java standard library: List<E>, HashMap<K,V>.

04

Using a Generic Class

Java
// Coordinates — both Double
Pair<Double, Double> p1 = new Pair<>();
p1.setFirst(3.14);
p1.setSecond(2.71);
p1.setSecond("Scaler");  // ✗ Compile error — second must be Double

// Student marks — String name, Double marks
Pair<String, Double> p2 = new Pair<>();
p2.setFirst("Samable");
p2.setSecond(82.5);

// No cast needed — getFirst() returns String directly
String s = p2.getFirst();   // ✓ no (String) cast required

// Generic method — accepts any Animal or subclass
void fun(Animal animal) { /* ... */ }

Animal a = new Animal(); fun(a);  // ✓
Dog    d = new Dog();    fun(d);  // ✓ Dog IS-A Animal
05

Wildcards — ? extends T

Generics are invariant — List<Dog> is NOT a subtype of List<Animal>, even though Dog IS-A Animal. This means you cannot pass a List<Dog> to a method that expects List<Animal>.

Java
void fun(List<Animal> animals) { /* ... */ }

List<Dog> dogs = new ArrayList<>();
fun(dogs);  // ✗ Compile error — List<Dog> is NOT a List<Animal>

// Without wildcards, you'd need separate overloads for every animal type — code duplication!
void fun(List<Dog>   d) { /* ... */ }  // duplicated
void fun(List<Cat>   c) { /* ... */ }  // duplicated
void fun(List<Tiger> t) { /* ... */ }  // duplicated

The solution is a bounded wildcard: ? extends Animal means "a List of Animal or any of its subclasses."

Java — Bounded Wildcard
// ? extends Animal = "any List whose elements are Animal or a subclass of Animal"
void fun(List<? extends Animal> animals) {
    for (Animal a : animals) { /* treat each as Animal */ }
}

fun(new ArrayList<Dog>());    // ✓
fun(new ArrayList<Cat>());    // ✓
fun(new ArrayList<Car>());    // ✗ Car is not an Animal
06

Type Erasure

Generics are a compile-time feature only. At runtime, all type parameters are erased — replaced with Object (or the upper bound if bounded). This is called type erasure, and it was the backward-compatibility mechanism that allowed generics to be added in Java 5 without breaking existing pre-Java-5 code.

Java
// Before Java 5 — raw type (no type parameter)
List list = new ArrayList();

// After Java 5 — generic types
List<Integer> nos     = new ArrayList<>();
List<String>  strings = new ArrayList<>();
List<Car>     cars    = new ArrayList<>();

// At runtime, ALL of the above are just:  List (holding Objects)
// The JVM erases Integer, String, Car — it only sees List.
// Type checks and casts are injected by the compiler automatically.
Implication of Type Erasure

You cannot do instanceof List<Integer> at runtime — the JVM doesn't know the type parameter. You also cannot create a new T[10] generic array for this reason. Generic types only exist at compile time.

Quick Recap

Generics

Placeholders for data types. Introduced Java 5. Give you reusability (works with any type) + compile-time type safety.

Type Parameter

A placeholder like T, U, K, V in a class or method declaration. Filled in at use site: Pair<String, Double>.

Object vs Generics

Object: general but unsafe. Generics: general AND safe. Wrong types caught at compile time instead of runtime.

Invariance

List<Dog> is NOT a subtype of List<Animal>, even though Dog IS-A Animal. Generics are invariant by default.

Wildcard ? extends T

Accepts List of T or any subclass. List<? extends Animal> accepts List<Dog>, List<Cat>, etc.

Type Erasure

At runtime all type parameters become Object. Generics only exist at compile time. Cannot use instanceof with generic types.
