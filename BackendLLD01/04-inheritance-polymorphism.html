Inheritance & the IS-A Relationship

Inheritance lets a child class automatically acquire all attributes and behaviours of its parent class. This eliminates code duplication for entities that share common characteristics. The test for when to use inheritance is the IS-A check: if you can say "a Mentor IS-A User" or "a Dog IS-AN Animal" and it makes semantic sense, inheritance is appropriate.

A child class inherits ALL attributes and behaviours from its parent — as if all those fields and methods had been written inside the child class itself. But the child can also add its own unique attributes and methods on top.

02

The extends Keyword

Java
class User {
    String name, email, password;

    void login()  { /* ... */ }
    void signUp() { /* ... */ }
}

class Mentor extends User {
    String company;       // Mentor-specific fields
    double avgRating;
    // name, email, password, login(), signUp() all inherited from User
}

Mentor m = new Mentor();
m.name      = "Deepak";  // ✓ inherited from User
m.login();               // ✓ inherited from User
m.company   = "Scaler"; // ✓ Mentor's own field
Java Rule

Java supports only single inheritance for classes — a class can extend only one parent. This avoids the Diamond Problem (covered in Session 05). Multiple interface implementation is allowed as a workaround.

03

Constructor Chaining

When you create an object of a child class, Java automatically calls the parent's default constructor first, before the child's own constructor body executes. This cascade up the hierarchy is called constructor chaining.

A
B
C
D

D d = new D() → A's constructor runs first, then B, then C, then D.

Java
class A { A() { System.out.println("A constructor"); } }
class B extends A { B() { System.out.println("B constructor"); } }
class C extends B { C() { System.out.println("C constructor"); } }
class D extends C { D() { System.out.println("D constructor"); } }

D d = new D();
// Output:
// A constructor
// B constructor
// C constructor
// D constructor
Tip

Java inserts an implicit super() call as the first statement of every constructor. If you want to call a specific parameterized parent constructor, use super(arg1, arg2) explicitly as the very first line of your constructor.

04

Reference Type vs Object Type

In Java, the reference type (the declared type of the variable) and the object type (the actual class of the instantiated object) can differ. A parent reference can hold any child object.

Java
class A { String name; int age; }
class B extends A { String college; }
class C extends A { String company; }

// ✗  C c = new A() — parent object can't fill a child reference
A a = new C();         // ✓ reference=A, object=C

a.name    = "Amit";   // ✓ name is in A — accessible via A reference
a.age     = 25;       // ✓ age is in A
// a.company = "Scaler"; ✗ Compile error — 'company' is not in A
//   Even though the runtime object IS a C, the compiler only knows about A
Compile vs Runtime

Compile time: the compiler checks field/method access against the reference type. Runtime: the JVM determines which overridden method to execute based on the actual object type. This is what makes polymorphism work.

05

What is Polymorphism?

Polymorphism (from Greek: "many forms") means that the same method call can behave differently depending on the object it is invoked on. In Java, polymorphism comes in two flavours:

TypeMechanismResolved At
Compile-Time PolymorphismMethod Overloading — same name, different parameter listsCompile time
Runtime PolymorphismMethod Overriding — child redefines a parent method with the same signatureRuntime (JVM)
06

Method Overloading (Compile-Time)

Overloading means defining multiple methods with the same name but different parameter lists in the same class. The compiler picks the correct version based on the argument types and count at the call site.

Java
class Calculator {
    int sum(int a, int b)               { return a+b; }
    int sum(int a, int b, int c)          { return a+b+c; }
    int sum(int a, int b, int c, int d) { return a+b+c+d; }
}

Calculator calc = new Calculator();
calc.sum(1, 2);          // calls first overload
calc.sum(1, 2, 3);       // calls second overload
calc.sum(1, 2, 3, 4);    // calls third overload
Overloading vs Returning Different Types

Two methods that differ only by return type (e.g., void fun(String s) and int fun(String s)) have identical signatures and are not valid overloads — the compiler can't distinguish them at a call site like a.fun("hello"). The method signature is defined as: name + parameter types only (not return type).

07

Method Signature

The method signature in Java is the combination of the method's name and the data types of its parameters (in order). The return type is NOT part of the signature. Two methods are considered to have different signatures — and can therefore coexist as overloads — if they have different parameter types, different parameter counts, or different parameter order.

MethodsDifferent Signature?Valid Overloads?
fun(String s) vs fun(int x)Yes — different param types✓
fun(String s, int x) vs fun(int x, String s)Yes — different param order✓
int fun() vs void fun(int x)Yes — different params✓
void fun(String s) vs int fun(String s)No — only return type differs✗ Compile error
08

Method Overriding (Runtime)

Overriding means a child class provides its own implementation of a method that already exists in the parent, with the exact same signature (same name, same parameter types). The child's version replaces the parent's version for any object of the child class.

Overriding Rules

The overriding method must have: the same name, the same parameter list, and the same (or covariant) return type. It cannot reduce the access level (e.g., you cannot override a public method with private). The @Override annotation is strongly recommended — it catches mistakes at compile time.

Java
class A {
    void fun() { System.out.println("Hello from A"); }
}

class B extends A {
    @Override
    void fun() { System.out.println("Hey from B"); }
    // Note: changing return type to int would NOT be a valid override
    // (int fun() with no params would be a compile error — same signature, different return)
}
09

Dynamic Dispatch (Runtime Polymorphism)

When you call an overridden method through a parent reference, the JVM decides at runtime which version to execute based on the actual object stored — not the declared type of the reference. This is called dynamic method dispatch.

A a
= new
A()
→ a.fun() → "Hello"
A a
= new
B()
→ a.fun() → "Hey"

Same reference type (A), same method call (a.fun()), different output — because the actual stored object differs.

Java
// A has fun() → "Hello"
// B extends A, overrides fun() → "Hey"
// C extends B, overrides fun() → "Hi"

A a = new A();  a.fun();  // "Hello"
   a = new B();  a.fun();  // "Hey"  — B's override runs

B b = new C();  b.fun();  // "Hi"   — C's override runs (actual object is C)
   b = new B();  b.fun();  // "Hello" — wait, B doesn't override → walks up to A

// The rule: walk up the inheritance chain from the actual object type
// until you find an implementation of the method.
Key Rule

Which method executes depends on the object stored, not the reference type. The JVM looks at the actual class of the object, checks if that class overrides the method, and if not, walks up to the parent — continuing until it finds an implementation.

Quick Recap

Inheritance

Child inherits all parent attributes and behaviours. Use extends. Models IS-A relationships. Single parent only in Java.

Constructor Chaining

Creating a child object triggers the parent's constructor first (via implicit super()). Propagates up the entire chain.

Reference vs Object

Parent reference can hold child object. Compile-time access governed by reference type; runtime method dispatch governed by object type.

Method Overloading

Same name, different parameter signature. Resolved at compile time. Return type alone does NOT constitute a different signature.

Method Signature

Name + parameter types (order matters). Return type is NOT part of the signature in Java.

Method Overriding

Same name + same signature in child class. Replaces parent behaviour. Use @Override. Resolved at runtime.

Dynamic Dispatch

JVM looks at the actual object at runtime to decide which overridden method to run. Walks up the chain if no override is found.
