Exception Handling

What exceptions are, the full Java exception hierarchy, the difference between checked and unchecked exceptions, and how to handle or propagate them gracefully.

01

What is an Exception?

An exception is something unusual or unexpected that happens during program execution — a condition the normal code flow didn't anticipate. If left unhandled, an exception crashes the program.

Key Rule

If an exception is not handled anywhere in the call stack, the JVM terminates the thread (and in a single-threaded program, the entire application) and prints a stack trace. Handle exceptions — don't let them crash your users.

Common examples: reading a file that doesn't exist (FileNotFoundException), accessing array index 10 in an array of size 10 (ArrayIndexOutOfBoundsException), dividing by zero (ArithmeticException), or a failed database query (SQLException).

02

The Throwable Hierarchy

Everything throwable in Java extends Throwable, which has two direct children: Exception (recoverable) and Error (not recoverable — don't try to catch Errors).

04

try-catch Block

Wrap code that might throw an exception in a try block. If an exception is thrown, execution jumps immediately to the matching catch block. Lines after the throwing line in the try block are skipped.

Java
void readFile(String path) {
    try {
        File f = new File(path);
        f.read();     // ← if this throws, execution jumps to catch
        // L5, L6 are SKIPPED if f.read() throws
    } catch (FileNotFoundException e) {
        System.out.println("File not found: " + e.getMessage());
    } catch (IOException e) {
        System.out.println("IO error: " + e.getMessage());
    } catch (Exception e) {
        System.out.println("Unexpected: " + e.getMessage());
    }
    // L8, L9 — execute normally after exception is handled
}
05

Multiple catch & Ordering Rules

You can have multiple catch blocks. The JVM checks them top-to-bottom and executes the first matching one. This means you must order catches from most specific to most general. Putting a parent exception type before a child is a compile error (unreachable code).

Java
// ✗ WRONG — Exception catches everything, making FileNotFoundException unreachable
try { /* ... */ }
catch (Exception e)           { /* ... */ }   // too broad — catches all
catch (FileNotFoundException e) { /* ... */ }   // Compile error: unreachable!

// ✓ CORRECT — specific first, general last
try { /* ... */ }
catch (FileNotFoundException e) { /* most specific */ }
catch (IOException e)           { /* parent of FNFE */ }
catch (Exception e)             { /* catch-all fallback */ }
06

The finally Block

Code inside a finally block always executes — whether an exception was thrown or not, whether it was caught or not. Use it for cleanup that must always happen, like closing database connections or releasing file handles.

Java
Connection conn = null;
try {
    conn = DB.connect();
    conn.executeQuery("SELECT ...");
} catch (SQLException e) {
    System.out.println("DB error: " + e.getMessage());
} catch (Exception e) {
    e.printStackTrace();
} finally {
    // ALWAYS runs — even if exception is uncaught!
    if (conn != null) conn.close();   // prevent resource leak
}
07

throws — Propagating Up the Call Stack

Instead of handling a checked exception yourself, you can declare that your method throws it — passing responsibility to the caller. Use throws in the method signature. The exception propagates up the call stack until something handles it (or the JVM crashes the program).

Java
// Method declares it might throw IOException — caller must handle or re-throw
void readFile(String path) throws IOException {
    File f = new File(path);
    f.read();   // if this throws, propagates to caller
}

// Caller has two choices:
// Option A — handle it here
try { readFile("data.txt"); }
catch (IOException e) { /* handle */ }

// Option B — propagate further up
void processFile(String path) throws IOException {
    readFile(path);   // caller's problem now
}

08

Custom Exceptions

When the built-in exceptions don't semantically describe your error, create your own. For a checked exception, extend Exception. For an unchecked one, extend RuntimeException.

Java
// Custom checked exception
class InsufficientFundsException extends Exception {
    private final double shortfall;

    InsufficientFundsException(double shortfall) {
        super("Insufficient funds. Short by: " + shortfall);
        this.shortfall = shortfall;
    }

    public double getShortfall() { return shortfall; }
}

// Usage
void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance)
        throw new InsufficientFundsException(amount - balance);
    balance -= amount;
}
09

try-with-resources (Java 7+)

Any class that implements AutoCloseable (which includes all I/O streams, database connections, etc.) can be declared in a try statement. Java will automatically call close() when the block exits — whether normally or via exception. This eliminates the need for a finally block just to close resources.

Java
// OLD: manual finally
BufferedReader br = null;
try { br = new BufferedReader(new FileReader("file.txt")); /* ... */ }
catch (IOException e) { /* handle */ }
finally { if (br != null) br.close(); }

// NEW: try-with-resources — close() called automatically
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    String line;
    while ((line = br.readLine()) != null) System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
}
// br.close() is automatically called — no finally needed
✦

Quick Recap

Exception

An unusual event during execution. Must be handled or the program crashes. Extends Throwable → Exception.

Error

Severe JVM problems (OutOfMemoryError, StackOverflowError). No recovery possible — do not catch.

Checked Exception

Extends Exception (not RuntimeException). Compiler forces you to handle or declare with throws. Examples: IOException, SQLException.

Unchecked Exception

Extends RuntimeException. No compiler enforcement. Caused by bugs — NullPointerException, ArrayIndexOutOfBoundsException.

try-catch

Wrap risky code in try. Matching catch block handles the exception. Skips remaining try lines after throw.

catch ordering

Most specific (child) types first, general (parent) types last. Reversed order = compile error (unreachable code).

finally

Always executes regardless of exception. Use for cleanup: close DB connection, release file handle.

throws

Declare in method signature to propagate a checked exception to the caller. Caller must then handle or re-throw.

try-with-resources

Declare AutoCloseable resources in try(). close() called automatically. No finally block needed.
