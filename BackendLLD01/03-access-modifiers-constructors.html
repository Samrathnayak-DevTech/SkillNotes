Access Modifiers & Constructors

Who can see what, how objects get initialised, and the crucial difference between copying a reference and copying an object.

01

The Four Access Modifiers

Access modifiers determine the visibility of a class member (field, method, constructor) from other parts of the program. Java has four, listed here from most permissive to most restrictive:

ModifierAccessible From
publicAnywhere — any class, any package.
protectedSame package, plus any subclass (child class) even if it's in a different package.
(default / package-private)Only within the same package. No keyword required; just omit the modifier.
privateOnly within the declaring class itself. Not even subclasses can access it.
Best Practice — Encapsulation

Declare all fields private. Expose them through public getter and setter methods. This way you control every read and write — you can add validation, logging, or lazy initialization inside the setter without changing any code that uses the class.

Java
class Student {
    private   String name;      // only Student can access
    protected int    age;       // Student + subclasses + same package
              String batch;    // default — same package only
    public    double psp;       // anyone

    public String getName()               { return name; }
    public void   setName(String name)  { this.name = name; }
}
02

Default Constructor

A constructor is a special method that is called when you create an object with new. It has the same name as the class and no return type (not even void).

If you don't write any constructor, the Java compiler silently inserts a default (no-argument) constructor. It creates the object and initialises all fields to their default values: 0 for numeric types, false for booleans, and null for object references.

Java
class Student {
    String name;
    int    age;
    String batch;

    // Java inserts this implicitly if you write no constructor:
    Student() {
        name  = null;
        age   = 0;
        batch = null;
    }
}

Student st = new Student();   // calls the default constructor
st.name = "Mahavendra";
st.age  = 25;
st.batch = "LLD";
Important

As soon as you define any constructor yourself, the compiler stops inserting the default one. If you want to keep no-arg construction available alongside a parameterized constructor, you must write both explicitly.

03

Parameterized (Custom) Constructor

A parameterized constructor accepts arguments so you can initialise an object with specific values in a single line, rather than setting each field separately after construction.

Java
class Student {
    String name;
    int    age;
    String batch;

    // Parameterized constructor — takes all three fields
    Student(String s1, int a, String b) {
        name  = s1;
        age   = a;
        batch = b;
    }
}

// Now you can create a fully-initialized object in one line:
Student st  = new Student("Mahavendra", 25, "LLD");
Student st1 = new Student("Amit", 24, "LLD");
Student st2 = new Student("Deepak", 30);  // uses 2-arg overload
Tip — this keyword

When a constructor parameter has the same name as a field, use this.name = name to disambiguate. this refers to the current object. This is the conventional Java style and makes the constructor self-documenting.

04

Constructor Overloading

A class can have multiple constructors with different parameter lists. This is called constructor overloading. The compiler picks the right one based on the arguments you pass — the same rule as method overloading.

Java
class Student {
    String name; int age; String batch;

    Student() { /* no-arg */ }

    Student(String n, int a) {        // 2-arg overload
        this.name = n; this.age = a;
    }

    Student(String n, int a, String b) {  // 3-arg overload
        this.name = n; this.age = a; this.batch = b;
    }
}

Student a = new Student();                        // no-arg
Student b = new Student("Deepak", 30);           // 2-arg
Student c = new Student("Amit", 24, "LLD");    // 3-arg
05

Copy Constructor

A copy constructor takes an object of the same class as its parameter and creates a new object with the same field values. This is the correct way to duplicate an object — as opposed to assignment (st2 = st1), which only copies the reference.

Java
class Student {
    String name; int age; String batch;

    // Copy constructor — accepts another Student
    Student(Student s) {
        this.name  = s.name;
        this.age   = s.age;
        this.batch = s.batch;
    }
}

Student st     = new Student("Mahavendra", 25, "LLD");

// ✗ Reference copy — SAME object, modifying stCopy modifies st too
Student bad    = st;

// ✓ Copy constructor — NEW object at a different address
Student stCopy = new Student(st);   // st → @780, stCopy → @900
06

Shallow Copy

The copy constructor above performs a shallow copy. For primitive fields (int, double, boolean), this is a true independent copy. But for non-primitive fields (objects and references like String, arrays, or other objects), a shallow copy copies only the reference — both the original and the copy end up pointing to the same underlying object.

st : @780
Student @ 780
name : @140 ──→ "Mahavendra"
age : 25
batch: @400 ──→ "LLD"
stCopy : @900
Student @ 900 (shallow copy)
name : @140 ──→ same "Mahavendra"!
age : 25 (independent)
batch: @400 ──→ same "LLD"!

Both objects share the same String objects at @140 and @400. Modifying the String through one object affects the other.

Note

In Java, String is immutable — you can't change a String object once created, only reassign the reference. So shallow copying Strings is usually safe. The real risk is with mutable objects like arrays, List, or custom objects (e.g., a List<Lecture> inside Student).

07

Deep Copy

A deep copy recursively copies every object in the graph. Each non-primitive field gets its own independent copy in memory — modifying one object's internals will never affect the other.

Java
class Student {
    String         name;
    int            age;
    String         batch;
    List<Lecture>  lectures;   // mutable non-primitive — needs deep copy

    // Deep copy constructor
    Student(Student s) {
        this.name  = s.name;        // String is immutable — safe
        this.age   = s.age;         // primitive — always independent
        this.batch = s.batch;       // String is immutable — safe

        // Deep copy the list — new list, with new Lecture objects
        this.lectures = new ArrayList<>();
        for (Lecture l : s.lectures) {
            this.lectures.add(new Lecture(l));  // copy each Lecture too
        }
    }
}
AspectShallow CopyDeep Copy
PrimitivesIndependent ✓Independent ✓
Non-primitives (references)Shared — same objectsIndependent — new objects
When to useObject has only primitives / immutable fieldsObject contains mutable nested objects
RiskModifying nested object affects both copiesNo cross-contamination; higher memory cost
08

Pass by Value in Java

Java is always pass by value — but what is "the value" depends on the type. For primitives, the value is the number itself. For objects, the value is the reference (the heap address). This distinction trips up many developers.

Java
void changeAge(int age) {
    age = 999;   // only the LOCAL copy changes — original unaffected
}

void changeName(Student s) {
    s.name = "Hacked";  // s is a copy of the REFERENCE — same heap object!
                          // original Student's name IS changed
}

void reassign(Student s) {
    s = new Student();   // s now points to a new object, but the
                          // original reference outside is unaffected
}
Rule

Java passes a copy of the reference for objects — not a copy of the object, and not the reference itself. You can mutate the object the reference points to (and the caller will see those changes), but you cannot make the caller's reference point to a different object.

Quick Recap

Access Modifiers

public → protected → default → private. Always use private for fields; expose via getters/setters.

Default Constructor

Auto-provided if you write none. Sets all fields to zero / null / false. Lost once you define your own.

Parameterized Constructor

Initialises fields from arguments in a single new call. Use this.field = param to avoid naming conflicts.

Constructor Overloading

Multiple constructors with different parameter lists. Compiler selects the right one at call site.

Copy Constructor

Takes Student s as parameter; creates a new independent object. Correct way to duplicate an object.

Shallow Copy

Copies references for non-primitives — both objects share the same nested objects.

Deep Copy

Recursively copies all nested objects — fully independent. Required when nested objects are mutable.

Pass by Value

Java always passes by value. For objects, the value is a copy of the reference. Mutation of fields is visible to caller; reassignment of the reference is not.
