From Program to Process

A program is a static file sitting on disk (HDD/SSD) — just bytes of compiled code. It does nothing by itself. When you launch it, the OS loads it into RAM as a process — a live, running instance of that program with its own memory space, registers, and execution state.

⚠ CPU cannot directly read from HDD — the speed mismatch is too large. The OS must first load the program into RAM.

Definition

A process is a program in execution. It is an isolated, self-contained unit of computation with its own memory address space, code, data, stack, and CPU registers. Multiple processes cannot directly access each other's memory.

02

Process Anatomy & the PCB

The OS tracks every running process using a data structure called the Process Control Block (PCB). A process's own memory is divided into regions:

Code
Data
Registers
PC (Program Counter)
Stack
⌇ Thread(s)

Process memory layout

Code — the compiled instructions of your program.

Data — static/global variables allocated at startup.

Registers — CPU register values saved when the process is paused.

PC (Program Counter) — the address of the next instruction to execute. Essential for resuming after preemption.

Stack — method call frames, local variables, return addresses.

Thread(s) — execution flows inside the process (at least one: the main thread).

03

What is a Thread?

A thread is the smallest unit of CPU execution. Every process has at least one thread (the main thread). A process can spawn additional threads to perform tasks in parallel — this is multithreading.

Analogy — Word Processor

When you type in Google Docs, multiple tasks happen simultaneously: spell checking (T1), grammar checking (T2), auto-saving (T3), and auto-suggestions (T4). Each runs on its own thread. From a single main(), the app spawns threads for each background task so they don't block each other.

Threads within the same process share the process's code, data, and resources. But each thread has its own private program counter, stack, and registers. This shared memory is what makes threads efficient — and what makes synchronization necessary.

Key Rule

Threads share the same heap memory (objects, static fields). Each thread has its own stack. This is both the strength (cheap communication) and the danger (data races) of multithreading.

Cores & Context Switching

A CPU core is the execution unit that actually runs instructions. A single core can execute only one thread at a time. At 2 GHz, a core performs 2 × 10⁹ operations per second — so a task requiring 10⁶ operations completes in about 0.5 ms.

Context Switching (Single Core)

When a single core runs multiple threads, it rapidly switches between them — pausing one thread (saving its PC and registers) and resuming another. This creates the illusion of parallelism even on one core. Each switch is called a context switch.

06

Concurrency vs Parallelism

Concurrency

Multiple threads are in different stages of execution at the same time, but they are NOT necessarily making progress simultaneously. Achieved via context switching on a single core. The OS interleaves threads so each gets CPU time.

Parallelism

Concurrency plus simultaneous progress. Multiple threads actually execute at the exact same instant — each on its own CPU core. Requires a multi-core CPU. Every parallel program is concurrent, but not every concurrent program is parallel.

Formula

Parallelism = Concurrency + Progress at the same time (multi-core)
Concurrency is about structure (multiple threads exist). Parallelism is about execution (multiple threads run simultaneously).

07

Creating Threads in Java

The canonical pattern for running a task on a separate thread in Java has four steps:

1
Create a Task class

Create a class that implements the Runnable interface and overrides run() with the work to be done.

2
Instantiate the Task

Create an object of your task class.

3
Wrap in a Thread

Pass the task to new Thread(task).

4
Start the Thread

Call thread.start(). This tells the JVM to create a new OS-level thread and call run() on it.

Java
// Step 1: Task class — implements Runnable
class PrintTask implements Runnable {
    private final int number;

    PrintTask(int n) { this.number = n; }

    @Override
    public void run() {
        System.out.println("Thread printing: " + number);
    }
}

public class Main {
    public static void main(String[] args) {
        // Steps 2, 3, 4 — create 100 threads, each printing one number
        for (int i = 1; i <= 100; i++) {
            Runnable task   = new PrintTask(i);   // Step 2
            Thread   thread = new Thread(task);   // Step 3
            thread.start();                          // Step 4
        }
    }
}

// Output order is non-deterministic — threads are scheduled by the OS
Important

Call thread.start(), NOT thread.run(). Calling run() directly executes the task on the current thread — no new thread is created. start() is what actually spins up a new OS-level thread.

08

Thread Lifecycle

A Java thread passes through several states during its lifetime:

StateMeaning
NEWThread object created but start() not yet called.
RUNNABLEReady to run or currently running. Waiting for CPU time from the scheduler.
BLOCKEDWaiting to acquire a monitor lock (e.g., trying to enter a synchronized block held by another thread).
WAITINGIndefinitely waiting — e.g., after calling Object.wait() or Thread.join().
TIMED_WAITINGWaiting for a specified duration — e.g., Thread.sleep(1000).
TERMINATEDThe run() method has completed (normally or via exception).

Quick Recap

Process

Program in execution. Own isolated address space. Tracked by the OS via the PCB (Process Control Block).

Thread

Lightweight process. Unit of CPU execution. Shares process's heap; owns its own PC, stack, and registers.

Core

Execution unit inside a CPU. One core runs one thread at a time. Multi-core CPUs can run multiple threads truly simultaneously.

Context Switching

CPU saves a thread's state and restores another's. Enables concurrency on single-core by rapidly interleaving threads.

Concurrency

Multiple threads in different stages of execution. Does NOT require simultaneous progress. Achievable on single core via context switching.

Parallelism

Concurrency + simultaneous progress. Requires multi-core CPU. Multiple threads literally execute at the same instant.

Runnable

Interface with one method: run(). Implement it to define a task. Pass to new Thread(task).start() to execute concurrently.

start() vs run()

start() creates a new OS thread and calls run() on it. Calling run() directly just executes on the current thread.
