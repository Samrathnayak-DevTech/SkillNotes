01

What is OOP?

Before OOP, the dominant paradigm was procedural programming (think C). A program was a sequence of functions that operated on data. As systems grew, data and functions became increasingly hard to keep in sync — a function somewhere could modify a variable that another function depended on, with no mechanism to prevent misuse.

Object-Oriented Programming organises code around entities called objects. Each object bundles its own data (attributes) and the operations it can perform (behaviours/methods) into a single unit. Instead of asking "what functions do I need?", you ask "what entities exist in my problem domain, and what can they do?"

Definition

OOP is a software engineering paradigm that structures programs as collections of objects — each representing a real-world or conceptual entity with its own state (attributes) and behaviour (methods). In Java, OOP is the foundation of everything.

Analogy — PhonePe Example

In the real world, PhonePe sends money through a bank. In OOP, you model this exactly: a PhonePe object knows about a Bank object and asks it to transfer money. Data and behaviour live together. Later sessions will show how to make this relationship flexible using interfaces.

02

Class & Object

A class is a blueprint — it describes what attributes and behaviours an entity will have, but it doesn't occupy meaningful memory on its own (only metadata). An object is a concrete instance of that blueprint: it occupies real memory in the heap and holds actual values.

Java
// CLASS — the blueprint
class Student {
    // Attributes (state)
    String name;
    int    age;
    String batch;
    double psp;         // problem-solving percentage
    int    attendance;

    // Behaviours (methods)
    void solveProblem() { /* ... */ }
    void bookMockInterview() { /* ... */ }
}

// OBJECT — a concrete instance
Student st = new Student();
st.name = "Amit";
st.age  = 25;
st.solveProblem();   // calling behaviour on this specific object
Key Properties of Objects

Every object is a real instance of its class. It occupies memory on the heap. All objects of the same class are independent of each other — changing one object's name field does not affect any other object.

03

Memory: Stack & Heap

When you write Student st = new Student(), two things happen in two different memory areas. Understanding this is essential for understanding references, copying, and garbage collection.

Stack (references)
st : @1750
st1 : @1850
Local variables & references live here. Each method call gets its own stack frame.
→
Heap (objects)
Student @ 1750
name = "Amit"
age = 25
Student @ 1850
name = null
age = 0
Objects are allocated here by new. The heap is managed by the JVM's garbage collector.

The variable st on the stack holds a reference — a memory address (e.g., @1750). The actual Student object lives at that address on the heap. This distinction becomes critical when you assign one variable to another:

Java
Student st  = new Student();  // st → @1750
Student st1 = new Student();  // st1 → @1850 (different object)

Student st2 = st1;              // st2 → @1850 (SAME object as st1!)
                                  // NO new object is created

st2.name = "Changed";
System.out.println(st1.name);  // prints "Changed" — st1 and st2 point to same object
Common Pitfall

Student st2 = st1 does NOT create a copy of the object. It copies the reference (the address). Both st1 and st2 now point to the exact same object in heap memory. Modifying through one variable affects the other. To create a true copy, you need a copy constructor (covered in Session 03).

05

Pillar 1: Abstraction

Abstraction means hiding internal complexity and exposing only what the user needs. When you call st.solveProblem(), you don't need to know how the method works internally — you just need to know it exists and what it does. The implementation details are hidden (abstracted) away.

Abstraction also means representing things in terms of ideas rather than implementation details. A BankAPI interface abstracts all banks — you don't need to know whether it's ICICI or YesBank underneath.

Analogy

When you drive a car, you press the accelerator — you don't need to understand the combustion engine, the fuel injection system, or the transmission. Those details are abstracted away behind a simple interface: pedals, a wheel, and a gear shift.

06

Pillar 2: Encapsulation

Encapsulation has two complementary meanings. First: bundling — keeping an entity's attributes and behaviours together in one class. Second: protection — controlling who can access those attributes using access modifiers.

Analogy — The Medicine Capsule

A capsule holds the medicine together (bundling) and protects it from the external environment (protection). A class is a capsule for an object's data and behaviour.

Java
class BankAccount {
    private double balance;   // protected — can't be accessed directly

    public void deposit(double amount) {
        if (amount > 0) balance += amount;  // controlled access
    }

    public double getBalance() { return balance; }  // read-only access
}

BankAccount acc = new BankAccount();
// acc.balance = -9999;  ✗ Compile error — private field
acc.deposit(1000);       // ✓ Controlled through method
07

Pillar 3: Inheritance

Inheritance allows a child class to automatically acquire all the attributes and behaviours of a parent class — without rewriting them. This models real-world IS-A relationships and dramatically reduces code duplication.

Java
class User {             // Parent class
    String name, email;
    void login()  { /* ... */ }
    void signUp() { /* ... */ }
}

class Mentor extends User {  // Mentor IS-A User
    String company;
    double avgRating;
    // Automatically has name, email, login(), signUp() from User
}

Mentor m = new Mentor();
m.name = "Deepak";    // ✓ inherited from User
m.login();             // ✓ inherited from User

Inheritance is covered in depth in Session 04, but the key idea here is the IS-A relationship: a Mentor IS-A User, a Dog IS-AN Animal. Whenever this relationship holds, inheritance is appropriate.

08

Pillar 4: Polymorphism

Polymorphism means "many forms." A single method name or reference can behave differently depending on the actual object behind it. This is what lets you write general code that works with many specific types.

Java
User u;
u = new Mentor();      // ✓ a Mentor IS-A User
u = new Instructor(); // ✓ an Instructor IS-A User

// One list holds all user types:
List<User> users = new ArrayList<>();
users.add(new Mentor());
users.add(new Instructor());
users.add(new Student());

// One method handles all user types:
void changePassword(User user, String newPass) { /* ... */ }

Polymorphism is covered in depth in Session 04. For now, the key takeaway is: you can store a child class object in a parent class reference, but not vice versa.

Rule

User xyz = new Instructor() ✓ — a parent reference holds a child object.
Instructor x = new User() ✗ — a child reference cannot hold a parent object. A User is not necessarily an Instructor.

09

Access Modifiers

Access modifiers control the visibility of classes, methods, and fields. Java has four, ordered from most to least permissive:

ModifierSame ClassSame PackageChild Class (other package)Everywhere
public✓✓✓✓
protected✓✓✓✗
default (no keyword)✓✓✗✗
private✓✗✗✗
Best Practice

Default to private for fields and expose them through public getter/setter methods. This is the essence of encapsulation — you control exactly how data is read and written.

Quick Recap

Pillar 1
Abstraction

Hide complexity; expose only what matters. Users work with ideas, not implementations.

Pillar 2
Encapsulation

Bundle data + behaviour together. Protect data using access modifiers.

Pillar 3
Inheritance

Child class inherits attributes & behaviours from parent. Models IS-A relationships.

Pillar 4
Polymorphism

One reference, many forms. Parent reference can hold any child object.

Class

Blueprint of an entity. Defines attributes and behaviours. No memory by itself.

Object

Real instance of a class. Lives on the heap. Each object is independent.

Reference

A variable on the stack that stores an object's heap address. Assignment copies the address, not the object.

Stack vs Heap

Stack: references + method frames. Heap: objects. new allocates heap; constructor initialises.

Access Modifiers

public → protected → default → private. Strictness increases; use private by default for fields.
