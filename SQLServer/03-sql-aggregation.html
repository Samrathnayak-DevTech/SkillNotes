Aggregation: GROUP BY & HAVING 📊

So far you've selected and joined individual rows. Aggregation is the opposite move: it collapses many rows into summary values — a count, a total, an average. GROUP BY lets you summarize per category (per department, per month), and HAVING filters those groups. Master the difference between filtering rows (WHERE) and filtering groups (HAVING) and you've got the heart of every report and dashboard. No prior coding background assumed.

🔢 COUNT / SUM 📐 AVG / MIN / MAX 🗂️ GROUP BY 🚦 HAVING ⚖️ WHERE vs HAVING 🧠 Interview Prep
01

What Aggregation Is

Every query you've written so far returns one output row per input row. Aggregation breaks that rule on purpose — it boils many rows down to a single summary number.

Imagine a stack of 500 sales receipts. A normal query hands you back all 500. But the questions people actually ask are summaries: "How many receipts are there?" "What's the total?" "What's the average sale?" Answering those means combining many rows into one value. That combining is aggregation, and the tools that do it are aggregate functions like COUNT, SUM, and AVG.

🧾 Analogy

Aggregation is the subtotal at the bottom of a receipt. The individual line items are your rows; the "Total: $84.20" and "Items: 7" at the bottom are aggregates. GROUP BY is like printing a subtotal per category — one for groceries, one for household, one for snacks — instead of one grand total.

The aggregation toolkit 🗺️

🔑 The core shift in thinking

Normal query: row in → row out. Aggregate query: many rows in → one row out (or one row per group). Once you group, you can no longer show plain per-row columns unless they're part of the grouping — a rule we'll unpack in section 06.

03

Aggregate Functions

Five workhorses cover almost everything. Each takes a whole set of values and returns one.

🔢

COUNT

How many rows (or non-NULL values). COUNT(*) counts rows; COUNT(col) counts non-NULLs.

➕

SUM

Adds up all the values in a numeric column. Ignores NULLs.

📐

AVG

The mean: SUM ÷ count of non-NULL values. NULLs are skipped, not treated as zero.

⬇️

MIN

The smallest value — works on numbers, dates, and text.

⬆️

MAX

The largest value — also works on numbers, dates, and text.

sql
-- Whole-table summary: one row of numbers describing all employees
SELECT
    COUNT(*)      AS Headcount,
    SUM(Salary)   AS TotalPay,
    AVG(Salary)   AS AvgPay,
    MIN(Salary)   AS LowestPay,
    MAX(Salary)   AS HighestPay
FROM dbo.Employees;

Result

HeadcountTotalPayAvgPayLowestPayHighestPay
3363,000121,00098,000145,000
⚠️ Integer division surprise

If you AVG an integer column, SQL Server does integer math and drops the decimals (e.g., AVG of 3 and 4 gives 3, not 3.5). To get a decimal average, cast first: AVG(CAST(col AS DECIMAL(19,4))). Our Salary is already DECIMAL, so it's fine.

04

COUNT in Detail

COUNT has three forms that give three different answers. Confusing them is one of the most common interview stumbles.

sql
SELECT
    COUNT(*)                AS AllRows,       -- every row, NULLs included
    COUNT(ManagerId)        AS WithManager,   -- non-NULL ManagerId values only
    COUNT(DISTINCT DeptId)  AS DistinctDepts -- unique department values
FROM dbo.Employees;

Result

AllRowsWithManagerDistinctDepts
312

Why these numbers? COUNT(*) = 3 (all employees). COUNT(ManagerId) = 1 because only Alan has a manager — Ada's and Grace's ManagerId are NULL, and COUNT(column) skips NULLs. COUNT(DISTINCT DeptId) = 2 because employees span only Engineering and Sales (two distinct departments).

🔑 The three COUNTs

COUNT(*) — number of rows, regardless of NULLs. COUNT(column) — number of non-NULL values in that column. COUNT(DISTINCT column) — number of unique non-NULL values. Interviewers love asking the difference, because it reveals whether you understand how NULLs flow through aggregates.

05

GROUP BY

Instead of one summary for the whole table, GROUP BY gives you one summary per category.

GROUP BY sorts rows into buckets that share the same value, then runs your aggregate once per bucket. "Headcount per department" means: bucket employees by DeptId, then COUNT(*) inside each bucket.

sql
-- One row per department, with that department's headcount
SELECT DeptId, COUNT(*) AS Headcount, AVG(Salary) AS AvgPay
FROM dbo.Employees
GROUP BY DeptId;

Result

⚠️ Support is missing again

Support (Dept 3) doesn't appear — it has no employee rows, so there's no bucket for it. To show zero-count groups you must start from Departments and LEFT JOIN the employees (see section 12). This is the aggregation cousin of the "INNER JOIN drops unmatched rows" idea.

Grouping by more than one column

You can group by several columns at once — the buckets are then every unique combination. For example, GROUP BY DeptId, HireYear gives one row per department-per-year.

06

The Golden Rule

Once you use GROUP BY, every column in your SELECT must be either grouped or aggregated — no exceptions.

Here's the logic: after grouping, each output row represents a whole bucket of rows. If you ask for a plain column that isn't in the GROUP BY, SQL Server can't know which row's value to show — the bucket may hold many different values. So it refuses.

sql
-- ❌ ERROR: Name is neither grouped nor aggregated.
-- Which of the two Engineering names should this row show? Ambiguous.
SELECT DeptId, Name, COUNT(*) AS Headcount
FROM dbo.Employees
GROUP BY DeptId;

-- ✅ FIX 1: group by Name too (one row per dept + name combination)
SELECT DeptId, Name, COUNT(*) AS Headcount
FROM dbo.Employees
GROUP BY DeptId, Name;

-- ✅ FIX 2: aggregate it, so the bucket collapses to one value
SELECT DeptId, MAX(Name) AS AnyName, COUNT(*) AS Headcount
FROM dbo.Employees
GROUP BY DeptId;
🔑 The rule in one line

Every non-aggregated column in SELECT must appear in GROUP BY. If you want a column's value without grouping by it, wrap it in an aggregate (like MAX()). This is the error beginners hit most: "Column is invalid in the select list because it is not contained in either an aggregate function or the GROUP BY clause."

07

HAVING

A filter that runs after grouping, so it can test aggregate results — something WHERE cannot do.

Once you've produced one summary row per group, you often want to keep only some of those groups: "only departments with more than one employee," "only months where sales exceeded 10,000." That's a filter on the group's aggregate, and it lives in the HAVING clause.

sql
-- Departments that have MORE THAN ONE employee
SELECT DeptId, COUNT(*) AS Headcount
FROM dbo.Employees
GROUP BY DeptId
HAVING COUNT(*) > 1;

Result

DeptIdHeadcount
12

Engineering (Dept 1) has two employees and stays; Sales (Dept 2) has one and is filtered out. Notice we filtered on COUNT(*) — an aggregate — which only HAVING can do.

🔑 HAVING is "WHERE for groups"

WHERE asks "should this row be included?" HAVING asks "should this group be included?" Because groups don't exist until after GROUP BY runs, HAVING is the only place you can reference an aggregate like SUM() or COUNT() in a filter.

08

WHERE vs HAVING

The single most important distinction in this topic — and a guaranteed interview question.

WHERE filters individual rows before they're grouped. HAVING filters whole groups after aggregation. Using both together is common and powerful: trim the rows first, group what's left, then trim the groups.

sql
SELECT DeptId, AVG(Salary) AS AvgPay
FROM dbo.Employees
WHERE  Salary > 50000          -- 1. drop low-paid rows BEFORE grouping
GROUP BY DeptId                    -- 2. bucket the survivors by department
HAVING AVG(Salary) > 120000;   -- 3. keep only high-average groups

Result

DeptIdAvgPay
1121,500

All three salaries survive the WHERE (all > 50,000). Grouped by department, Engineering averages 121,500 and Sales 120,000. The HAVING AVG(Salary) > 120000 keeps Engineering and drops Sales.

WHEREHAVING
FiltersIndividual rowsWhole groups

Can use aggregates?NoYes
Typical use"only orders from 2024""only groups totaling over $1,000"
✅ Performance nudge

Put a condition in WHERE whenever you can — filtering rows early means fewer rows to group, which is faster. Reserve HAVING for conditions that genuinely need an aggregate (something you can only know after grouping).

09

Order of Execution

SQL runs clauses in a fixed logical order that is different from how you write them. This one diagram explains almost every "why can't I…?" question in aggregation.

📁 1. FROM / JOIN Assemble the raw rows from the tables (apply joins).
⚖️ 2. WHERE Filter individual rows. Aggregates don't exist yet, so you can't use them here.
🗂️ 3. GROUP BY Collapse the surviving rows into groups. Aggregates are computed now.
🚦 4. HAVING Filter whole groups. Aggregates now exist, so this is where you test them.
🎯 5. SELECT Pick columns and compute expressions. Column aliases are created here.
🔀 6. ORDER BY Sort the final result. Because it runs last, it can use SELECT aliases.
🔑 This order explains everything

Why can't WHERE use an aggregate? Because WHERE (step 2) runs before GROUP BY (step 3), so the aggregate doesn't exist yet. Why can HAVING use one? Because it runs at step 4, after grouping. Why can ORDER BY use a SELECT alias but WHERE can't? Because aliases are born in step 5 (SELECT), before ORDER BY (step 6) but after WHERE.

⚠️ Logical vs physical

This is the logical order the language guarantees — the mental model to reason with. The engine's optimizer may physically execute steps differently (and faster) as long as the result matches. For interviews and for writing correct queries, always reason with the logical order above.

10

NULLs in Aggregates

Aggregates quietly ignore NULLs — which is usually helpful, but occasionally the source of a "wrong" average.

With one exception (COUNT(*)), aggregate functions skip NULL values entirely. SUM adds only the real numbers; AVG divides by how many non-NULL values there are, not the total row count; MIN/MAX ignore NULLs too.

FunctionHow it treats NULL
COUNT(*)Counts the row anyway (it counts rows, not values)
COUNT(col)Skips NULLs — only non-NULL values counted
SUM(col)Skips NULLs; if all values are NULL, returns NULL
AVG(col)Skips NULLs in both the sum and the divisor
MIN / MAXSkips NULLs
⚠️ The "average" gotcha

Suppose 5 employees have a bonus, but 2 of those bonus values are NULL (not yet decided). AVG(Bonus) divides by 3 (the non-NULL bonuses), not 5. If you meant "treat missing bonuses as 0," you must say so explicitly: AVG(COALESCE(Bonus, 0)). Otherwise your average is silently higher than you expect.

🔑 Turn NULL into a value when you need to

Use COALESCE(col, 0) or ISNULL(col, 0) to replace NULLs with a real value before aggregating, whenever "missing" should count as zero. And use COUNT(*) vs COUNT(col) deliberately depending on whether you're counting rows or actual values.

11

GROUP BY vs DISTINCT

Both can remove duplicates and produce a unique list — but they exist for different jobs.

sql
-- These two return the SAME unique list of department ids: 1, 2
SELECT DISTINCT DeptId FROM dbo.Employees;

SELECT DeptId FROM dbo.Employees GROUP BY DeptId;
🔑 Choose by intent

Use DISTINCT when you only want to remove duplicate rows and show a unique list. Use GROUP BY when you want to compute an aggregate per group (a count, a total). If there's no aggregate involved, DISTINCT reads more clearly. If you're summarizing, GROUP BY is the right tool — and it's the only one that lets you add COUNT, SUM, and HAVING.

✅ Don't stack them needlessly

SELECT DISTINCT ... GROUP BY ... is almost always redundant — grouping already produces one row per group. Seeing both in one query usually signals a misunderstanding worth cleaning up.

12

Aggregates + JOINs

Combining Day 2's joins with aggregation is where real reports live — and where a subtle double-counting trap waits.

To show department names (not just ids) with their totals — and to include departments that have no employees — start from Departments and LEFT JOIN the employees, then group by the department name.

sql
SELECT d.Name AS Department,
       COUNT(e.EmpId) AS Headcount,   -- counts real employees (NULLs skipped)
       SUM(e.Salary)  AS TotalPay
FROM dbo.Departments AS d
LEFT JOIN dbo.Employees AS e ON e.DeptId = d.DeptId
GROUP BY d.Name;

Result

DepartmentHeadcountTotalPay
Engineering2243,000
Sales1120,000
Support0NULL
🔑 COUNT(e.EmpId) vs COUNT(*) here

For Support, the LEFT JOIN produces one row with all employee columns NULL. COUNT(*) would count that phantom row as 1. COUNT(e.EmpId) counts non-NULL employee ids, giving the correct 0. When aggregating after an outer join, count a column from the optional table, not *.

⚠️ The double-counting trap

If you join Employees to Orders (one-to-many) and then SUM(e.Salary), each employee's salary is repeated once per order and summed multiple times — inflating the total. When you need totals from two different one-to-many relationships, aggregate each in its own subquery first, then join the summaries. This is the aggregation version of the duplicate-row trap from the joins topic.

13

Rollups & Best Practices

A quick look at subtotal/grand-total extensions, plus the habits that keep aggregate queries correct and readable.

Subtotals with ROLLUP 🧮

Sometimes you want each group's total and a grand total in the same result. GROUP BY ROLLUP(...) adds those extra summary rows automatically.

sql
SELECT d.Name AS Department, SUM(e.Salary) AS TotalPay
FROM dbo.Departments AS d
JOIN dbo.Employees AS e ON e.DeptId = d.DeptId
GROUP BY ROLLUP(d.Name);
-- adds a final row where Department = NULL, TotalPay = 363,000 (grand total)

Related extensions: CUBE gives totals for every combination of the grouped columns, and GROUPING SETS lets you list exactly which groupings you want in one query. All three are senior-level "I know these exist" knowledge; ROLLUP is the one you'll actually use most.

Conditional aggregation 🎛️

To aggregate only rows meeting a condition — without splitting into separate queries — wrap a CASE inside the aggregate:

sql
-- Count active vs total in one pass (pivot-style)
SELECT
    COUNT(*) AS Total,
    SUM(CASE WHEN Salary >= 120000 THEN 1 ELSE 0 END) AS HighEarners
FROM dbo.Employees;
⚖️

Filter early with WHERE

Trim rows before grouping; only use HAVING for conditions that need an aggregate.

🔢

Mind your COUNTs

Pick COUNT(*), COUNT(col), or COUNT(DISTINCT col) on purpose.

👻

Decide on NULLs

Use COALESCE when "missing" should count as zero in a SUM or AVG.

🧮

Beware double counting

Aggregate each one-to-many relationship in its own subquery before joining totals.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is aggregation, and what is an aggregate function? Junior

Aggregation means combining many rows into a single summary value. An aggregate function is the tool that does it — like COUNT (how many), SUM (total), or AVG (average). Instead of returning every row, it returns one number describing the whole set.

Q02 Name the five main aggregate functions and what each does. Junior

COUNT = how many rows or values; SUM = the total of a numeric column; AVG = the average (mean); MIN = the smallest value; MAX = the largest value. MIN and MAX also work on dates and text (earliest/latest, first/last alphabetically).

Q03 What does GROUP BY do? Junior

It sorts rows into groups that share the same value (say, all employees in the same department) and then runs the aggregate once per group. The result has one summary row per group — for example, one headcount and average salary per department.

Q04 What's the difference between COUNT(*) and COUNT(column)? Junior

COUNT(*) counts every row, no matter what's in it. COUNT(column) counts only the rows where that column has a real (non-NULL) value. So if a column has some blanks, COUNT(column) will be smaller than COUNT(*).

Q05 How does AVG treat NULL values? Junior

It ignores them. AVG adds up only the non-NULL values and divides by how many non-NULL values there were — not by the total number of rows. So NULLs are treated as "not there," not as zero. If you want blanks counted as zero, convert them first with COALESCE(col, 0).

Q06 What is the HAVING clause? Junior

HAVING filters groups after they've been formed by GROUP BY. It's how you keep only certain groups — for example, "only departments with more than one employee" or "only months with total sales above $10,000." It's the one place you can filter using an aggregate like COUNT() or SUM().

Q07 Why can't you use an aggregate function in a WHERE clause? Junior

Because WHERE runs before the rows are grouped, so aggregates like SUM or COUNT don't exist yet at that point. To filter based on an aggregate, you use HAVING, which runs after grouping.

Q08 What does SUM return if there are no matching rows or all values are NULL? Junior

It returns NULL, not 0. This surprises people who expect an empty total to be zero. If you want a zero instead, wrap it: COALESCE(SUM(col), 0). (COUNT, by contrast, returns 0 when there are no rows.)

Q09 What is the difference between WHERE and HAVING? Mid

WHERE filters individual rows before grouping and cannot use aggregates. HAVING filters whole groups after grouping and can use aggregates. A common query uses both: WHERE to trim rows first, then GROUP BY, then HAVING to keep only certain groups. Filtering early with WHERE is also faster.

Q10 What's the rule about columns in SELECT when you use GROUP BY? Mid

Every column in the SELECT list must either appear in the GROUP BY or be wrapped in an aggregate function. If you try to select a plain column that isn't grouped, SQL Server errors, because each output row represents a whole group and it wouldn't know which underlying row's value to show.

Q11 Explain COUNT(*) vs COUNT(col) vs COUNT(DISTINCT col). Mid

COUNT(*) = number of rows. COUNT(col) = number of rows where col is not NULL. COUNT(DISTINCT col) = number of different non-NULL values in col. Example: a table of 3 employees spanning 2 departments where only 1 has a manager gives COUNT(*)=3, COUNT(ManagerId)=1, COUNT(DISTINCT DeptId)=2.

Q12 GROUP BY vs DISTINCT — when do you use each? Mid

Use DISTINCT when you just want a unique list of rows/values, with no aggregation. Use GROUP BY when you want to compute an aggregate per group (a count, sum, average). For a plain unique list they give the same result, but only GROUP BY supports aggregates and HAVING. Stacking both is usually redundant.

Q13 How do aggregate functions handle NULLs in general? Mid

They skip NULLs. SUM, AVG, MIN, MAX, and COUNT(col) all ignore NULL values. The sole exception is COUNT(*), which counts rows regardless of content. This is why an AVG can be higher than expected — it divides by the number of non-NULL values, not the total row count.

Q14 Can you use a SELECT column alias in WHERE, GROUP BY, HAVING, or ORDER BY? Mid

Only in ORDER BY. Aliases are created in the SELECT step, which runs after WHERE, GROUP BY, and HAVING but before ORDER BY. So those earlier clauses must repeat the full expression, while ORDER BY can use the short alias. This falls straight out of the logical order of execution.

Q15 How do you count rows per group and keep only groups above a threshold? Mid

Group by the category, count with COUNT(*), and filter with HAVING: SELECT DeptId, COUNT(*) FROM Employees GROUP BY DeptId HAVING COUNT(*) > 1;. The HAVING tests the per-group count, keeping only groups that clear the bar.

Q16 A department with no employees vanishes from a GROUP BY — why, and how do you include it? Mid

Grouping the Employees table only creates buckets for departments that have employees, so an empty department never forms a group. To include it, start from Departments and LEFT JOIN the employees, group by the department, and use COUNT(EmpId) so the empty department shows a count of 0.

Q17 Explain the logical order of execution and why it matters. Senior

Logically SQL processes: FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. It explains almost every aggregation rule: WHERE can't use aggregates (they aren't computed until GROUP BY); HAVING can (it runs after); and ORDER BY can use SELECT aliases while earlier clauses can't. The optimizer may physically execute differently, but the guaranteed result matches this logical order.

Q18 What is the double-counting trap when aggregating across joins, and how do you avoid it? Senior

Joining to a one-to-many table multiplies the "one" side's rows — so summing a value from the "one" side counts it once per matching child, inflating the total. For example, summing employee salary after joining to their orders counts salary once per order. The fix is to aggregate each one-to-many relationship in its own subquery (or CTE) first, then join the pre-computed summaries together so nothing is multiplied.

Q19 What are ROLLUP, CUBE, and GROUPING SETS? Senior

They extend GROUP BY to produce extra summary rows in one query. ROLLUP(a, b) adds subtotals along a hierarchy plus a grand total. CUBE(a, b) adds totals for every combination of the grouped columns. GROUPING SETS lets you specify exactly which groupings you want. The extra summary rows carry NULL in the columns they total over; the GROUPING() function distinguishes those from real NULLs.

Q20 How does COUNT(e.EmpId) differ from COUNT(*) after a LEFT JOIN? Senior

A LEFT JOIN keeps unmatched left rows with the right table's columns set to NULL. COUNT(*) counts those phantom rows too, so an empty group shows 1 instead of 0. COUNT(e.EmpId) counts only non-NULL ids from the optional table, correctly returning 0 for a group with no matches. Rule of thumb: after an outer join, count a column from the optional side, not *.

Q21 When do you need a window function instead of GROUP BY? Senior

GROUP BY collapses rows — you lose the individual rows and keep only the summary. A window function (like SUM() OVER (...)) computes an aggregate alongside each original row, so you keep both the detail and the summary (e.g., each employee's salary next to their department's total, or a running total). If you must show detail rows and a summary together, reach for a window function rather than GROUP BY.

Q22 What affects the performance of a GROUP BY? Senior

Grouping requires the engine to organize rows by the grouping key, typically via a sort or a hash aggregate. An index on the grouping (and filtering) columns can let it group with a cheap ordered scan instead of a costly sort. Filtering early with WHERE reduces the rows to group. High-cardinality groupings and large row counts increase memory and CPU. Good indexes and current statistics are the main levers.

Q23 Can you use HAVING without GROUP BY? Senior

Yes. Without GROUP BY, the whole table is treated as a single group, so HAVING filters that one group. For example, SELECT SUM(Salary) FROM Employees HAVING SUM(Salary) > 1000000; returns the total only if it exceeds a million, otherwise no rows. It's uncommon but valid — useful for "return this summary only if it meets a condition."

Q24 How do you compute a conditional aggregate, like counting only certain rows? Senior

Put a CASE expression inside the aggregate. To count high earners: SUM(CASE WHEN Salary >= 120000 THEN 1 ELSE 0 END), or equivalently COUNT(CASE WHEN Salary >= 120000 THEN 1 END) (since COUNT ignores the NULLs the ELSE-less CASE produces). This lets you compute several conditional totals in a single pass — the basis of pivot-style reporting.
