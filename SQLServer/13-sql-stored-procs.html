Stored Procedures & Safe Dynamic SQL 🛡️

A stored procedure is a named, reusable block of T-SQL saved in the database — parameters in, work done, results out. Sometimes a query's shape must be built at runtime (dynamic SQL), and that's where danger lives: naively pasting user input into a query string opens the door to SQL injection. This page shows how to build procs well and — the part interviewers care about — how to write dynamic SQL safely with parameters and QUOTENAME. No prior coding background assumed.

01

What a Proc Is

A stored procedure is a named batch of SQL statements saved in the database, callable by name, usually with parameters — like a function for your data layer.

Instead of sending the same query text from your application every time, you define it once as a stored procedure and call it by name: EXEC dbo.GetEmployeesByDept @DeptId = 1. The logic lives in the database, is compiled and cached, and can be reused by any caller. Procedures can take inputs, return outputs, run multiple statements, use control flow, and wrap work in transactions.

📞 Analogy

A stored procedure is like a saved contact with a speed-dial. Rather than typing out the full number (the whole query) every time, you press one button with a name on it and optionally say who it's for (the parameters). The instructions live in one place; everyone dials the same, well-tested routine.

🔑 The through-line of this page

Procedures and parameters are the safe, default way to run SQL. Dynamic SQL is a powerful escape hatch for when a query's structure must vary — but it reintroduces the risk that parameters removed. The whole skill is: prefer parameters, and when you must build SQL as text, build it safely.

03

Create & Execute

Define a procedure with CREATE PROCEDURE; run it with EXEC.

sql
CREATE PROCEDURE dbo.GetEmployeesByDept
    @DeptId INT                     -- an input parameter
AS
BEGIN
    SET NOCOUNT ON;                   -- suppress "N rows affected" messages
    SELECT EmpId, Name, Salary
    FROM dbo.Employees
    WHERE DeptId = @DeptId;
END;
GO

-- Call it
EXEC dbo.GetEmployeesByDept @DeptId = 1;
🔑 SET NOCOUNT ON is a habit worth keeping

It stops SQL Server from sending a "N rows affected" message after every statement. Those extra messages add network chatter and can confuse some client libraries, so putting SET NOCOUNT ON at the top of a procedure is a standard best practice. Use ALTER PROCEDURE to change an existing proc, or CREATE OR ALTER PROCEDURE to create-or-update in one statement.

04

Parameters

Procedures take input parameters (values passed in), can hand back output parameters, and can give parameters default values.

sql
CREATE PROCEDURE dbo.GetDeptHeadcount
    @DeptId INT = NULL,          -- default: NULL means "all departments"
    @Count  INT OUTPUT            -- hands a value back to the caller
AS
BEGIN
    SET NOCOUNT ON;
    SELECT @Count = COUNT(*)
    FROM dbo.Employees
    WHERE (@DeptId IS NULL OR DeptId = @DeptId);
END;
GO

DECLARE @n INT;
EXEC dbo.GetDeptHeadcount @DeptId = 1, @Count = @n OUTPUT;
SELECT @n AS Headcount;
➡️

Input

The default kind — a value the caller passes in, like @DeptId.

⬅️

Output

Marked OUTPUT; the proc writes to it and the caller reads it back.

🔧

Default

Give a value (e.g., = NULL) so callers may omit the argument.

🔑 Parameters are typed and safe by design

A parameter has a declared data type, and its value is passed separately from the SQL text — the engine never treats it as code. That single fact is the foundation of injection safety: whatever a user types into @DeptId is only ever a value, never a command. Keep this in mind for the dynamic-SQL sections below.

05

Why Use Them

Procedures aren't just tidy — they bring concrete performance, security, and maintainability benefits.

♻️

Reuse & encapsulation

Write the logic once; every app and report calls the same tested routine instead of duplicating SQL.

🛡️

Security surface

Grant users EXECUTE on the proc without direct table access, and parameters resist injection.

⚡

Plan reuse

The execution plan is cached and reused across calls, avoiding repeated compilation.

🔧

Central maintenance

Change the logic in one place; callers don't need redeploying to get the fix.

📉

Less network traffic

Send a short EXEC name @args instead of a long query string each time.

🔗

Transaction control

Wrap multi-step operations in a transaction inside the proc for atomicity.

🔑 The security angle interviewers like

With stored procedures you can give an application account permission to EXECUTE a specific procedure while denying it direct SELECT/UPDATE on the underlying tables. Through ownership chaining, the proc can still touch those tables. That narrows what a compromised account can do to exactly the operations the procedures expose — a real least-privilege win.

06

Control Flow

Procedures can branch, loop, validate, and manage transactions — the reason they're more than just saved queries.

sql
CREATE PROCEDURE dbo.GiveRaise
    @EmpId  INT,
    @Amount DECIMAL(19,4)
AS
BEGIN
    SET NOCOUNT ON;
    BEGIN TRY
        BEGIN TRANSACTION;
            IF @Amount <= 0
                THROW 50001, N'Raise must be positive.', 1;

            UPDATE dbo.Employees
            SET Salary = Salary + @Amount
            WHERE EmpId = @EmpId;

            IF @@ROWCOUNT = 0
                THROW 50002, N'No such employee.', 1;
        COMMIT TRANSACTION;
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0 ROLLBACK TRANSACTION;
        THROW;                       -- re-raise to the caller
    END CATCH;
END;
🔑 Validate, act, and protect atomically

This proc checks its input (@Amount > 0), performs the update inside a transaction, verifies a row was actually changed (@@ROWCOUNT), and rolls back on any error. That combination — validation, transaction, and TRY…CATCH with re-THROW — is the reliable template for any procedure that changes data. Other tools available: IF/ELSE, WHILE loops, and local variables via DECLARE.

07

Dynamic SQL: What & Why

Dynamic SQL is a query built as a string at runtime and then executed. You need it only when the query's structure — not just its values — has to vary.

A normal parameterized query can vary its values (WHERE DeptId = @DeptId) but not its structure: you can't parameterize a table name, a column name, or which columns to sort by. When those must change at runtime, you assemble the SQL text yourself and run it. Legitimate cases include:

  • A dynamic sort — the user picks which column to order by.
  • Optional filters — building a WHERE only from the filters actually supplied.
  • A dynamic table/schema name — e.g., the same logic run against a table chosen at runtime.
⚠️ Use it sparingly — it's a last resort, not a default

Dynamic SQL is harder to read, harder for the optimizer to reuse plans for, and — most importantly — it reopens the door to SQL injection if you build the string by pasting in user input. Always ask first whether a plain parameterized query or proc can do the job. Only reach for dynamic SQL when the structure genuinely must vary, and then build it defensively (the next sections).

Do I need dynamic SQL — and how do I keep it safe? 🧭

✅ Can a normal parameterized query or proc do it? Yes → use that. No dynamic SQL, no injection risk. This is the default.
🔌 Does only a value vary (WHERE col = ?)? Pass it as a sp_executesql parameter — never concatenate it.
🏷️ Does an identifier vary (table / column / sort)? It can't be a parameter → validate against a whitelist, then wrap with QUOTENAME.
🧵 Building the string? Run it with sp_executesql + parameters — never EXEC of concatenated input.
08

EXEC vs sp_executesql

There are two ways to run a SQL string. One accepts parameters and one doesn't — and that difference is the whole ballgame for safety.

sql · avoid: concatenated EXEC
-- ❌ EXEC of a concatenated string: user input becomes part of the SQL text
--    This is the pattern that enables injection. Do not build SQL this way.
EXEC (N'SELECT EmpId, Name FROM dbo.Employees WHERE DeptId = ' + @DeptInput);
sql · prefer: sp_executesql with parameters
-- ✅ sp_executesql: the value is passed as a typed PARAMETER, never as text
EXEC sys.sp_executesql
    N'SELECT EmpId, Name FROM dbo.Employees WHERE DeptId = @DeptId',
    N'@DeptId INT',          -- parameter definition
    @DeptId = @DeptInput;         -- parameter value, supplied separately
EXEC (string)sp_executesql
Accepts parameters?No — values must be concatenated inYes — typed parameters
Injection-safe for values?No (if input is concatenated)Yes
Plan reuse?Poor (text differs each time)Good (parameterized text is stable)
Recommended?Avoid for anything with inputThe standard for dynamic SQL
🔑 Always prefer sp_executesql

sp_executesql lets you write dynamic SQL that still uses parameters, so values are passed separately and never interpreted as commands — plus the parameterized text is stable enough for the optimizer to cache and reuse a plan. EXEC(@string) has no parameters, so any variability must be concatenated, which is exactly what makes it unsafe. The rest of the page uses sp_executesql.

09

The Risk

SQL injection happens when user-supplied text is concatenated into a query string, so that input can change the query's meaning rather than just its data. Understanding the mechanism is what makes the defense obvious.

sql · vulnerable pattern (for understanding only)
-- ❌ The user's text is glued directly into the SQL string.
--    Whatever they type is parsed as part of the command.
DECLARE @sql NVARCHAR(MAX) =
    N'SELECT * FROM dbo.Employees WHERE Name = ''' + @UserInput + '''';
EXEC (@sql);

The intended query filters by a name. But because @UserInput is concatenated into the command text, a crafted value can close the string early and append its own SQL. A classic illustration is an input that makes the WHERE condition always true (so every row returns), and more damaging inputs can append additional statements. The database can't tell the difference between the developer's SQL and the attacker's — it's all just one string of text now.

⚠️ Why this is so dangerous

Because the input is treated as code, a successful injection can read data the user shouldn't see, bypass authentication filters, or modify and destroy data — limited only by what the database account is permitted to do. It's consistently ranked among the most serious and common web-application vulnerabilities. The root cause is always the same: mixing untrusted input into command text.

🔑 The one-sentence takeaway

Injection is possible only when user input is concatenated into SQL text. If input is never concatenated — if it's always passed as a parameter — there is no string for an attacker to break out of, and injection simply cannot occur. That's why the fixes below all amount to "stop concatenating, start parameterizing."

10

Defense: Parameters

The primary defense: pass every user-supplied value as a parameter, never as concatenated text. Then it can only ever be data.

sql · safe: value passed as a parameter
-- ✅ The name is a parameter. Whatever the user types is only ever a value.
EXEC sys.sp_executesql
    N'SELECT EmpId, Name, Salary
      FROM dbo.Employees
      WHERE Name = @Name',
    N'@Name NVARCHAR(100)',
    @Name = @UserInput;           -- passed separately -- not part of the SQL text

Here the query text is fixed: WHERE Name = @Name. The user's input is handed to @Name as a separate, typed value. Even if someone types SQL-looking text, it's compared as a literal name — there's no string for it to escape from, so the query's meaning can't be altered.

✅ This is the same idea as app-side parameterized queries

In application code (ADO.NET, Entity Framework, etc.), the equivalent is using command parameters or an ORM rather than building SQL with string concatenation. Whether in T-SQL via sp_executesql or in C# via parameters, the principle is identical: separate the code from the data. Stored procedures called with parameters get this safety automatically.

⚠️ "Escaping quotes" is not a real defense

Trying to sanitize input by doubling apostrophes or stripping keywords is fragile and repeatedly bypassed — different encodings, comment tricks, and edge cases defeat it. Don't rely on blacklists or manual escaping for values. Parameterization is the robust, complete defense for values; use it everywhere.

11

Defense: QUOTENAME

Some things — table names, column names, sort columns — can't be parameters. For those identifiers, defend with a whitelist plus QUOTENAME.

You cannot write ORDER BY @columnName as a parameter — a parameter is a value, and a column name is part of the query's structure. So a dynamic sort column must be built into the text. The safe way is to (1) validate the name against a known list, and (2) wrap it with QUOTENAME, which safely brackets an identifier and escapes any special characters.

sql · safe dynamic sort: whitelist + QUOTENAME
CREATE PROCEDURE dbo.SearchEmployees
    @SortColumn SYSNAME = N'Name'     -- SYSNAME is the type for identifiers
AS
BEGIN
    SET NOCOUNT ON;

    -- 1) WHITELIST: reject anything not in the known-good set
    IF @SortColumn NOT IN (N'Name', N'Salary', N'HireDate')
        THROW 50010, N'Invalid sort column.', 1;

    DECLARE @sql NVARCHAR(MAX) =
        N'SELECT EmpId, Name, Salary, HireDate
          FROM dbo.Employees
          ORDER BY ' + QUOTENAME(@SortColumn);   -- 2) bracket-escape the identifier

    EXEC sys.sp_executesql @sql;
END;
🔑 Two layers for identifiers

The whitelist is the strongest protection: if the requested column isn't one you explicitly allow, reject it outright. QUOTENAME is defense-in-depth — it wraps the name in [brackets] and doubles any embedded closing bracket, so even an unexpected identifier can't break out of its position. Use both. Any values in the same query still go through parameters as before.

✅ The rule of thumb

Values (things compared with =, >, IN) → always parameters. Identifiers (table, column, sort names that must vary) → whitelist + QUOTENAME. If you follow those two rules, your dynamic SQL is safe from injection.

Where do stored procedures fit next to parameterized application queries and ORMs? All can be safe; they differ in where the logic lives.

🔑 Safety is about parameterization, not the tool

Stored procedures, ad-hoc parameterized queries, and ORMs are all safe because they parameterize. The only universally unsafe option is hand-concatenating input into SQL. The proc-vs-ORM choice is really about architecture — where you want logic to live, how you manage permissions and deployments — not about injection safety, which every parameterized approach provides.

🧯

If you remember one thing: never build SQL by concatenating untrusted input. Values become parameters; identifiers get whitelisted and QUOTENAMEd. 15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is a stored procedure? Junior

A named, reusable block of T-SQL saved in the database and called by name, usually with parameters. Instead of sending the same query text repeatedly, you define the logic once and run it with EXEC procName @args. It can take inputs, return outputs, run multiple statements, and use control flow and transactions.

Q02 How do you create and run one? Junior

Define it with CREATE PROCEDURE name @param TYPE AS BEGIN … END, and run it with EXEC name @param = value. Use ALTER PROCEDURE to change it, or CREATE OR ALTER PROCEDURE to create-or-update in one step.

Q03 What is an input parameter? Junior

A typed value the caller passes into the procedure, like @DeptId INT. It's the default kind of parameter. Inside the proc you use it like a variable, and crucially its value is passed separately from the SQL text, so it can never be interpreted as code.

Q04 What is an output parameter? Junior

A parameter marked OUTPUT that the procedure writes to so the caller can read the result back. You declare a variable, pass it with OUTPUT in the EXEC, and after the call it holds the value the proc set — useful for returning a single computed value like a count.

Q05 Why use stored procedures? Junior

They centralize and reuse logic, reduce network traffic (send a short call instead of a long query), let you grant execute permission without exposing tables, cache and reuse execution plans, and keep maintenance in one place. They also make parameterized, injection-resistant access the natural default.

Q06 What does SET NOCOUNT ON do? Junior

It stops SQL Server from sending the "N rows affected" informational message after each statement. That reduces network chatter and avoids confusing some client libraries, so it's a common best practice at the top of a stored procedure.

Q07 What is dynamic SQL? Junior

SQL that is built as a text string at runtime and then executed, rather than written as a fixed statement. You use it when the query's structure must vary — for example, sorting by a column the user chooses, or adding filters only for the criteria supplied. It should be used sparingly and built safely.

Q08 What is SQL injection? Junior

A vulnerability where user-supplied input is concatenated into a SQL command string, so the input can change the query's meaning instead of just its data. Because the database parses the whole string as code, a crafted input can read or alter data it shouldn't. The fix is to never concatenate input — pass it as a parameter.

Q09 What are the main benefits of stored procedures? Mid

Reuse and encapsulation (logic in one tested place), a smaller security surface (grant EXECUTE without table access, and parameters resist injection), plan caching and reuse, reduced network traffic, and centralized maintenance so a fix doesn't require redeploying every caller. They also make transaction control and validation easy to keep server-side.

Q10 EXEC versus sp_executesql — what's the difference? Mid

EXEC(@string) just runs a text string and has no parameters, so any variability must be concatenated in — which enables injection and hurts plan reuse. sp_executesql runs a string but accepts typed parameters, so values are passed separately (safe) and the parameterized text is stable enough for the optimizer to reuse a cached plan. Prefer sp_executesql for all dynamic SQL.

Q11 When do you genuinely need dynamic SQL? Mid

When the query's structure must change at runtime — things you can't parameterize: a table or column name, a sort column the user picks, or a WHERE clause built from whichever optional filters were supplied. If only values vary, a normal parameterized query or proc is better and safer.

Q12 How do parameters prevent injection? Mid

A parameter's value is sent to the engine separately from the SQL text and is never parsed as part of the command — it's treated purely as data. So there's no string boundary for an attacker to break out of; whatever they type stays a value. Since injection requires input to become part of the command text, parameterization removes the possibility entirely.

Q13 Why is string concatenation of input dangerous? Mid

Because the concatenated input becomes part of the command that the database parses. Crafted input can close out the intended string and add its own SQL, changing the query's meaning — for instance turning a filter into an always-true condition, or appending extra statements. The database can't distinguish the developer's SQL from the injected SQL; it's all one string. Parameterizing avoids this completely.

Q14 How do you handle errors in a procedure? Mid

Wrap the work in BEGIN TRY … END TRY BEGIN CATCH … END CATCH. In the CATCH, roll back if a transaction is open (IF @@TRANCOUNT > 0 ROLLBACK) and then THROW to re-raise the error to the caller. Combine that with input validation (raising a custom error via THROW) and a transaction around any data changes.

Q15 Can you use transactions inside a procedure? Mid

Yes — that's a core use. You BEGIN TRANSACTION, do the multi-step work, and COMMIT, with a TRY…CATCH that rolls back on error, so the whole operation is atomic. Just keep the transaction short, and be careful about nesting: an inner rollback affects the whole transaction, so use the @@TRANCOUNT/savepoint patterns when a proc might be called inside another transaction.

Q16 What is QUOTENAME and when do you use it? Mid

QUOTENAME wraps a string as a safely-delimited SQL identifier — by default in [brackets], doubling any closing bracket inside it. You use it when you must place a dynamic identifier (a table, column, or sort-column name) into dynamic SQL, since identifiers can't be parameters. Pair it with a whitelist of allowed names for real safety.

Q17 Explain how injection works and the layered defenses. Senior

Injection occurs when untrusted input is concatenated into command text, letting the input alter the parsed statement rather than just supply data. The layered defenses: (1) parameterize all values via sp_executesql or app parameters, so values are never text — this is the primary, complete defense for values; (2) for identifiers that can't be parameters, use a whitelist of allowed names plus QUOTENAME; (3) apply least privilege so even a slip limits the damage; and (4) validate input types and lengths. Manual escaping/blacklists are not reliable and shouldn't be the defense.

Q18 Why can't you parameterize a table or column name, and what do you do instead? Senior

Parameters supply values, but a table/column name is part of the query's structure, which the engine must know at parse/compile time — so ORDER BY @col or FROM @table isn't allowed. Instead, you build the identifier into the dynamic SQL text safely: validate it against a fixed whitelist of permitted names (reject anything else) and wrap it with QUOTENAME so it's bracket-escaped. Values in the same statement still go through parameters.

Q19 How does sp_executesql help with plan reuse and parameter sniffing? Senior

Because the SQL text is parameterized and stays constant across calls (only the parameter values change), the optimizer can cache one plan and reuse it — unlike concatenated EXEC strings that differ every time and bloat the plan cache. The flip side is parameter sniffing: the cached plan is optimized for the first values seen, which can be suboptimal for very different later values. Remedies include OPTION (RECOMPILE), OPTIMIZE FOR hints, or local-variable patterns when needed.

Q20 Stored procs versus ORM/parameterized ad-hoc queries — trade-offs? Senior

All three are injection-safe as long as they parameterize; the choice is architectural. Stored procs keep logic and permissions in the database (great for central control, fine-grained EXECUTE grants, and shielding tables), but can scatter business logic server-side and add deployment coordination. ORMs and parameterized app queries keep logic in code (more productive and testable, easier refactoring) but move data logic into the app and rely on developers not dropping to unsafe raw SQL. Many teams blend them: procs for critical/complex operations, ORM for routine CRUD.

Q21 How do EXECUTE AS, ownership chaining, and least privilege apply to procs? Senior

Ownership chaining means that when a proc and the tables it touches share an owner, a user with only EXECUTE on the proc can run it without direct table permissions — enabling least privilege. EXECUTE AS lets a proc run under a specified security context (e.g., a limited principal) rather than the caller's, useful when a proc needs a permission the caller lacks. Together they let you expose exactly the operations you intend and nothing more — but note that dynamic SQL inside a proc breaks the ownership chain, so the caller does need underlying rights (or EXECUTE AS) for dynamic statements.

Q22 What are the pitfalls of dynamic SQL beyond injection? Senior

Plan-cache bloat and poor reuse if the text varies each call (mitigated by parameterizing via sp_executesql); broken ownership chaining, so permissions on underlying objects are re-checked against the caller; harder debugging and readability; the need to use NVARCHAR for the sp_executesql statement; scope isolation (variables and temp tables created inside the dynamic batch aren't visible outside it, though #temp tables created before it are); and subtler parameter-sniffing behavior. These are why dynamic SQL should be a considered last resort, not a default.

Q23 How do you safely validate a dynamic identifier? Senior

Strongest is a whitelist: compare the requested name against an explicit set of allowed values (or look it up in sys.columns/sys.tables to confirm it's a real, permitted object) and reject anything else. Then apply QUOTENAME as defense-in-depth to bracket-escape it. Avoid trying to sanitize by removing characters — that's fragile. The combination of "must be one of these known-good names" plus QUOTENAME makes a dynamic identifier safe to embed.

Q24 Output parameters vs return values vs result sets — when do you use each? Senior

A result set (a SELECT) is for returning rows/tabular data — the usual output. Output parameters return one or a few scalar values back to the caller (like a computed count or a new id). The return value of a proc (RETURN n) is an integer conventionally used only for status/success codes, not data. So: rows → result set; scalars → output params; status code → return value. Mixing them up (e.g., using RETURN to pass real data) is a common anti-pattern.
