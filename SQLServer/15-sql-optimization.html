Optimization Capstone & Self-Mock 🏁

The final piece ties the whole series together. Optimization isn't a bag of tricks — it's a repeatable loop: measure, read the plan, fix the access path, fix the query shape, and re-measure. Everything you've learned — indexes, plans, SARGability, isolation, functions, temp storage — plugs into that loop. Then comes the self-mock: a protocol, an interactive rapid-fire round, and scenario drills to rehearse explaining it all out loud. No prior coding background assumed.

01

The Mindset

Optimization is empirical: you measure the problem, form a hypothesis, change one thing, and measure again. Guessing is the enemy.

The single most important habit is to never guess. Slow queries have causes you can see — in the execution plan and the I/O statistics — so the job is to look, diagnose, and verify rather than sprinkle indexes and hope. Every technique from the past two weeks is a possible fix; this page is about the method that decides which fix to reach for.

🩺 Analogy

A good doctor doesn't prescribe before examining. They take measurements (vitals, tests), read the results, form a diagnosis, treat one thing, and re-check. Query tuning is identical: STATISTICS IO and the plan are your vitals and X-ray; adding an index or rewriting a predicate is the treatment; re-measuring is the follow-up. Treating without measuring is malpractice in both.

🔑 The interview meta-skill

When asked "how would you speed this up?", the strong answer isn't a single trick — it's the process: "First I'd capture the actual plan and I/O stats, find the most expensive operator, decide whether it's an access problem or a query-shape problem, apply one targeted fix, and re-measure." Demonstrating the loop shows seniority far more than naming a random index.

03

The Loop

The whole method in five repeatable steps. Every tuning session — and every good interview answer — follows this shape.

📏 1 · Measure Capture the actual plan + SET STATISTICS IO, TIME ON. Get real numbers, not hunches.
👓 2 · Find the bottleneck Highest cost %, fattest arrows, warning icons, biggest estimated-vs-actual gap.
🧭 3 · Diagnose the type Access problem (scan / lookup / non-SARGable) or shape problem (UDF / bad estimate / row-by-row)?
🔧 4 · Apply ONE fix Index or SARGable rewrite for access; set-based rewrite / temp table / fresh stats for shape.
✅ 5 · Re-measure & compare Fewer logical reads? Keep it. Then loop back to step 2 for the next bottleneck.
↺ repeat until fast enough
⚠️ One change at a time

The discipline that separates tuning from thrashing is changing a single thing per iteration and re-measuring. Add three indexes and a rewrite at once and you won't know what helped, what did nothing, and what quietly slowed your writes. Isolate each change so the before/after comparison is honest.

04

Step 1: Measure

Get objective numbers first. Logical reads and the actual plan tell you the truth; wall-clock time lies (it swings with caching and load).

sql
SET STATISTICS IO, TIME ON;   -- logical reads + CPU/elapsed per statement
-- (In SSMS also enable the actual plan: Ctrl+M, then run.)

-- ...run the query you are tuning...

SET STATISTICS IO, TIME OFF;
📈

Logical reads

Pages read from the buffer — the stable yardstick. Fewer reads ≈ better plan, regardless of load.

🗺️

Actual plan

The chosen operators plus real row counts — the map to the bottleneck.

🗄️

Query Store

Tracks query performance and plans over time; great for finding regressions in production.

🔑 Baseline before you touch anything

Record the starting logical reads and plan before making changes. That baseline is what every later iteration is compared against — without it, "it feels faster" is all you've got. In production, Query Store gives you historical baselines and can even force a known-good plan.

05

Step 2: Read

Find the one operator doing the most damage. You rarely need to read the whole plan — three signals point straight at it.

💰

Highest cost %

The operator with the largest share of estimated cost — a starting suspect (verify against actuals).

➰

Fattest arrows

The thickest data flows carry the most rows — often where the work piles up.

⚠️

Warning icons

Yellow triangles: implicit conversions, spills to tempdb, missing-index hints.

🔑 The estimated-vs-actual tell

The most valuable single clue is a big gap between an operator's estimated and actual row counts. It means the optimizer planned on bad information — usually stale or missing statistics — and every downstream choice (join type, memory grant) is suspect. Fixing the estimate often fixes the whole plan at once.

✅ Cross-check cost against reality

Cost percentages are the optimizer's guesses, so a "2%" operator can be the real villain when its actual row count dwarfs the estimate. Always sanity-check the plan's cost story against actual rows and the STATISTICS IO numbers before deciding what to fix.

06

Step 3: Fix Access

If the bottleneck is how rows are fetched — a big-table scan, a fat key lookup, a predicate that won't seek — the fixes are indexing and SARGability.

Symptom in the planDiagnosisFix
Scan of a large table for few rowsNo usable index, or non-SARGable predicateAdd an index; rewrite predicate to keep the column bare
Fat key lookupIndex doesn't cover the queryINCLUDE the returned columns
Non-SARGable filter (YEAR(col)=…)Function/expr on the column blocks the seekRewrite as a range on the bare column
Unexpected hash joinMissing index on the join columnIndex the join key
Costly sortNo index in the required orderProvide an index ordered to match ORDER BY
🔑 Access fixes in one breath

Make the predicate SARGable (column bare, matched types) so an index can seek; add or adjust an index so the seek exists and is selective; and cover the query with INCLUDE so there are no key lookups. Those three moves resolve the large majority of access-path problems — and they're drawn straight from the indexes and execution-plan topics.

07

Step 4: Fix Shape

If the query is fetching rows fine but still slow, the problem is often its shape — row-by-row logic, opaque constructs, or bad estimates.

🐢

Scalar UDF per row

Replace with an inline expression or inline TVF (or rely on 2019 UDF inlining where it applies).

🔁

Cursor / loop

Rewrite row-by-row logic as a single set-based statement wherever possible.

🪶

Big table variable

Swap a large @table for a #temp table so the optimizer gets real statistics.

📊

Multi-statement TVF

Convert to an inline TVF so it's optimized as part of the calling query.

📉

Stale statistics

Update statistics so estimates match reality and the optimizer picks the right plan.

🎯

Parameter sniffing

Consider OPTION (RECOMPILE) or OPTIMIZE FOR when one plan doesn't fit all values.

🔑 The set-based principle

SQL is happiest doing work in sets, not one row at a time. Most shape problems are some form of hidden row-by-row execution — a scalar UDF, a cursor, a multi-statement TVF — or a bad estimate that misleads the optimizer. The cure is to make the operation set-based and give the optimizer accurate information to plan with.

Anti-Pattern Checklist

The greatest hits from the whole series — the recurring mistakes that make queries slow. Scan this list first when something's wrong.

Anti-patternWhy it hurtsBetter
WHERE YEAR(col) = 2024Non-SARGable → scanRange on the bare column

Scalar UDF in a big queryRow-by-row, hidden cost, serialInline expression / inline TVF
Large table variablePoor cardinality estimate#temp table with statistics
Implicit type conversionSilently blocks the seekMatch column & parameter types
Leading-wildcard LIKE '%x'Can't seek an indexTrailing wildcard / full-text

✅ Two rules cover most of these

Keep the indexed column bare and its type matched (so seeks work), and prefer set-based operations over row-by-row ones (so the optimizer can do its job). A large share of real-world slow queries is just one of these two rules being broken.

09

A Worked Example

One query, three problems, fixed with the loop. Watch a slow query become fast by applying access and shape fixes together.

sql · before — slow
-- Employees hired in 2020, with a bonus. Three problems hiding here:
--   (1) YEAR(HireDate) is non-SARGable -> table scan
--   (2) dbo.fnAnnualBonus is a scalar UDF -> runs row-by-row
--   (3) no supporting index on HireDate
SELECT e.Name, dbo.fnAnnualBonus(e.Salary) AS Bonus
FROM dbo.Employees e
WHERE YEAR(e.HireDate) = 2020;

Measure & read: the plan shows a clustered index scan (from the non-SARGable YEAR()), the scalar UDF running once per row, and a forced serial plan — high logical reads for a query that should touch few rows.

sql · after — fast
-- Fix access: a covering index on the date, so the filter can SEEK
CREATE NONCLUSTERED INDEX IX_Emp_HireDate
ON dbo.Employees (HireDate) INCLUDE (Name, Salary);

-- Fix the predicate (SARGable range) and the shape (inline the math)
SELECT e.Name, e.Salary * 0.10 AS Bonus
FROM dbo.Employees e
WHERE e.HireDate >= '2020-01-01'
  AND e.HireDate <  '2021-01-01';

Re-measure: the plan now shows an index seek on IX_Emp_HireDate, no key lookup (the index covers Name and Salary), no scalar-UDF operator, and a plan free to go parallel — logical reads drop dramatically for the same result.

🔑 Notice how the topics combined

This one fix used SARGability (Day 10) to turn the scan into a seek, an index with INCLUDE (Day 9) to cover the query and remove lookups, and the scalar-UDF lesson (Day 14) to inline the bonus math and restore parallelism. Real optimization is exactly this: several small, measured changes from different topics, applied through the one loop.

The Self-Mock Protocol

Knowing the material and performing it under pressure are different skills. A structured self-mock builds the second one.

  1. Timebox it. 30 minutes, no notes. Set a timer — interview pressure is part of what you're training for.
  2. Answer out loud. Speak every answer as if to a person. Silent recognition ("yeah, I know that") hides the gaps that talking exposes.
  3. Mix the levels. Pull a few Junior, Mid, and Senior questions from across the 14 topic pages (that's 336 questions in total) so you jump between areas like a real interview.
  4. Self-grade each answer with the rubric below, and log the ones you fumble.
  5. Re-drill the misses tomorrow. Spaced repetition on your weak spots beats re-reading what you already know.

Deep-Dive Scenarios

Open-ended prompts like a senior round. There's no single right answer — practice thinking out loud, asking clarifying questions, and walking through the loop.

Scenario 1 · The slow report

A nightly report over Orders and Employees that used to run in seconds now takes 20 minutes. Nothing in the query changed. Walk me through how you'd find the cause and what you'd check first.

Scenario 2 · The intermittent timeout

A stored procedure is usually fast but times out a few times a day, always around the same busy period. The plan looks fine when you run it manually. What could be going on, and how would you confirm it?

Scenario 3 · The deadlock reports

Users occasionally see "transaction was deadlocked" errors on a screen that updates two related tables. How do you diagnose it, and what design changes would you propose?

Scenario 4 · The write-heavy table

A table with twelve indexes has fast reads but painfully slow inserts. How do you decide which indexes to keep, and how do you make that call safely in production?

Scenario 5 · The design review

A teammate wants to add a scalar function that formats an address, to be called in a query returning millions of rows. What's your feedback, and what alternative would you suggest?

✅ How to structure a scenario answer

Say your assumptions, ask a clarifying question or two, then narrate the loop: "I'd first measure with the actual plan and Query Store, look for the biggest change or the costliest operator, decide if it's access or shape or concurrency, and test one fix at a time." Naming which earlier topic each step draws on (SARGability, parameter sniffing, isolation) is what signals depth.

15

Capstone Questions

Cross-cutting questions that span the whole series — the synthesis kind asked to gauge overall depth. Click to reveal. Tagged by level.

Q01 What's the first thing you do when a query is slow? Junior

Measure it — capture the actual execution plan and turn on SET STATISTICS IO, TIME ON to see the real logical reads and where time goes. You look before you change anything, so you're fixing the actual bottleneck rather than guessing.

Q02 Why measure logical reads instead of elapsed time? Junior

Elapsed (wall-clock) time swings with server load and caching, so it's an unreliable comparison. Logical reads — pages read from the buffer — are stable and repeatable, so comparing them before and after a change tells you honestly whether the query is doing less work.

Q03 What's the difference between a seek and a scan? Junior

A seek navigates an index straight to the matching rows — fast, touching few pages. A scan reads all the rows and checks each one. Seeks are ideal for selective queries; a scan of a large table when you only want a few rows is usually a sign of a missing index or a predicate that can't use one.

Q04 What makes a query SARGable? Junior

Writing the WHERE clause so an index can seek — mainly, leaving the indexed column by itself on one side of the comparison. Wrapping it in a function (YEAR(col)) or arithmetic, or comparing mismatched types, makes it non-SARGable and forces a scan.

Q05 Why does an index speed up reads but slow writes? Junior

An index is a pre-sorted copy of some columns, so reads can find rows quickly. But every insert, update, or delete must also keep that copy up to date, so each extra index adds work to every write. That's the core trade-off — index the queries you actually run, not everything.

Q06 What's the default isolation level, and why does it matter? Junior

READ COMMITTED — you only read committed data, so no dirty reads, with good concurrency. It matters for tuning because isolation level controls how many locks are taken and how long they're held, which affects blocking and deadlocks; enabling read-committed snapshot (RCSI) can remove a lot of reader/writer blocking.

Q07 What is a covering index? Junior

An index that contains every column a query needs — the key columns to seek/filter on, plus any other returned columns added via INCLUDE. Because the index has everything, the query is answered from it alone with no extra trip (key lookup) back to the table, which is a big, cheap speedup.

Q08 When would you use a stored procedure? Junior

When you want reusable, centralized database logic that's called by name — for encapsulation, plan reuse, reduced network traffic, and security (grant EXECUTE without exposing tables). Procedures called with parameters are also naturally resistant to SQL injection.

Q09 Walk me through your query-tuning process. Mid

Measure first (actual plan + STATISTICS IO, and a baseline). Find the bottleneck via cost %, fat arrows, warnings, and estimated-vs-actual gaps. Diagnose whether it's an access-path problem (scan, lookup, non-SARGable) or a shape problem (scalar UDF, bad estimate, row-by-row). Apply one targeted fix — an index/rewrite for access, a set-based rewrite or temp table or fresh statistics for shape. Re-measure logical reads, and loop to the next bottleneck.

Q10 How do you decide whether to add an index? Mid

Identify the real query it would help and confirm the filter is selective enough that a seek beats a scan. Weigh the read benefit against the write cost (every index slows inserts/updates/deletes) and check for existing overlapping indexes you could extend instead. Treat missing-index hints as input, not gospel, and verify with before/after logical reads.

Q11 A query does a scan on a big table — what do you check? Mid

Whether the predicate is SARGable (no function/arithmetic on the column, matched types, no leading-wildcard LIKE); whether a useful index exists on the filter/join columns; whether the query genuinely needs most of the table (in which case a scan is correct); and whether statistics are current so the optimizer estimates correctly. Usually it's a non-SARGable predicate or a missing index.

Q12 How do you remove a key lookup? Mid

Make the non-clustered index cover the query by adding the columns it currently has to fetch from the table via INCLUDE (for returned-only columns) or as key columns (if you also filter/sort on them). Once the index contains everything the query needs, the seek returns the data directly and the per-row lookup disappears.

Q13 When does a table variable hurt performance? Mid

When it holds many rows and feeds a join or aggregation: the optimizer historically assumed one row (no column statistics), producing bad plans — wrong join type, tiny memory grants, spills. The fix is a #temp table, which has real statistics and can be indexed; on SQL Server 2019+ deferred compilation improves table-variable estimates, but temp tables remain the safe choice for large sets.

Q14 How do you prevent deadlocks? Mid

The biggest lever is accessing objects in a consistent order across all code paths, so a circular wait can't form. Also keep transactions short, add indexes so statements lock few rows, consider RCSI/snapshot to drop read locks, and make the app catch the deadlock error (1205) and retry the transaction for the unavoidable remainder.

Q15 Inline vs multi-statement TVF — which for performance? Mid

Inline. An inline TVF is a single SELECT the optimizer expands into the calling query (like a parameterized view), so it gets accurate estimates and good plans. A multi-statement TVF builds rows into a returned table variable the optimizer can't see into, historically misestimating its size and planning poorly. Use inline whenever the logic can be expressed as one query.

Q16 How do you tell if statistics are the problem? Mid

Look at the actual plan for a large gap between an operator's estimated and actual row counts. If the optimizer expected a few rows and got millions (or vice versa), it planned on stale or missing statistics, which cascades into wrong join types and memory grants. Updating statistics (or adding supporting/filtered statistics) and re-checking the estimates confirms it.

Q17 Design an indexing strategy for a read-heavy reporting workload. Senior

Start from the actual report queries: give each table a sensible clustered key (narrow, increasing), then add covering non-clustered indexes whose key columns match the common filters/joins/sorts and whose INCLUDE columns cover the selected columns. For heavy aggregations over stable data, consider indexed views or columnstore indexes. Since it's read-heavy, the write cost of extra indexes is more tolerable, but still consolidate overlapping indexes and validate each against real query plans and logical reads rather than adding blindly.

Q18 A proc is fast sometimes and slow other times — what's happening? Senior

Classic parameter sniffing: SQL Server compiled and cached a plan optimized for the parameter values seen at first execution, and that plan is great for some values but poor for others (e.g., one that suits a rare value being reused for a common one). Confirm by comparing the cached plan's compiled parameters against the slow-case values. Remedies include OPTION (RECOMPILE), OPTIMIZE FOR a representative value, local-variable patterns, or forcing a good plan via Query Store — trading recompile cost against plan quality.

Q19 How do you tune when you can't add indexes? Senior

Focus on the query and the data. Rewrite predicates to be SARGable so existing indexes can seek; eliminate scalar UDFs and row-by-row logic in favor of set-based operations; reduce the columns and rows touched (avoid SELECT *, filter earlier); stage large intermediates in temp tables with statistics; update statistics; and address parameter sniffing. You can also reshape joins, pre-aggregate, or use query hints as a last resort — all without touching the index design.

Q20 Walk through diagnosing a production slowdown end to end. Senior

Scope it first: one query or system-wide? Use Query Store to spot regressed queries and plan changes, and DMVs (sys.dm_exec_requests, wait stats, sys.dm_tran_locks) to see blocking, waits, and resource pressure. For a specific query, capture the actual plan and I/O, find the costliest operator and any estimate gaps, and decide access vs shape vs concurrency. Check for recent changes — data growth, stale stats, a plan flip from parameter sniffing, new blocking patterns. Then apply one fix, verify against the baseline, and monitor. Communicate findings and roll changes out carefully.

Q21 How do isolation level and indexing interact to affect concurrency? Senior

Indexing controls how many rows a statement locks: a selective seek locks a few rows briefly, while a scan locks many (and may escalate to a table lock), massively increasing contention. Isolation level controls how long and how strictly those locks are held: higher levels hold read locks longer and add range locks, raising blocking and deadlock risk, while row-versioning (RCSI/snapshot) removes read locks entirely. So good indexing plus an appropriate isolation level together minimize the lock footprint — narrow locks, held briefly, or no read locks at all.

Q22 When is denormalization or an indexed view the right call? Senior

When read performance on expensive joins/aggregations matters more than the extra write cost and storage, and the source data is relatively stable. An indexed view precomputes and maintains an aggregation automatically — ideal for frequently-read rollups over data that changes infrequently. Manual denormalization (duplicating columns to avoid joins) can help hot paths but risks update anomalies, so it needs disciplined maintenance. Both are deliberate trade-offs: measure that the read savings outweigh the write/consistency cost before committing.

Q23 How do you balance read performance against write cost across a schema? Senior

Treat indexes as a budget. Profile the workload's read/write mix per table: read-heavy tables can carry more (and covering) indexes and even indexed views; write-heavy tables should be lean, keeping only indexes that earn their keep. Use DMVs to find unused and duplicate indexes and drop them, consolidate overlapping ones, and re-evaluate as the workload evolves. The goal is the smallest set of indexes that satisfies the important queries — every index beyond that is a tax on every write.

Q24 How would you approach optimizing a query you've never seen? Senior

Understand its intent and the tables/volumes first, then run the loop: capture the actual plan and I/O for a baseline; find the costliest operator and any estimate gaps; classify the problem as access (scan/lookup/non-SARGable), shape (UDF/row-by-row/bad estimate), or concurrency (blocking/locking); apply one targeted fix and re-measure. Throughout, prefer the least invasive change, verify with logical reads, and watch for side effects on writes and other queries. The method matters more than any single trick — that's the whole point of this series.
