Locking & Deadlocks 🔐

Isolation levels are enforced by locks — short-lived claims on rows, pages, or tables that stop transactions from corrupting each other. Usually locks just cause brief blocking (one transaction waits its turn). But when two transactions each hold what the other needs, they wait forever — a deadlock — and SQL Server must kill one to break the cycle. This page covers lock modes, blocking vs deadlock, and how to prevent the jams. No prior coding background assumed.

01

What Locking Is

A lock is a temporary claim a transaction places on data so that others can't interfere with it while it's being read or changed.

The isolation levels from the previous topic are promises ("no dirty reads," "repeatable reads"). Locks are how SQL Server keeps those promises. When a transaction reads or writes a row, it takes a lock on it; other transactions that want an incompatible lock on the same data must wait until the first one releases (at commit or rollback). Locking is automatic — the engine manages it — but understanding it explains both slow queries (blocking) and mysterious errors (deadlocks).

🚻 Analogy

A lock is like occupying a single-stall restroom. While you're inside (holding the lock), the door is latched and others wait outside (blocking). When you leave (commit), the next person goes in. A deadlock is the absurd case where two people each hold a key the other needs and refuse to move — someone has to be made to leave for anyone to proceed.

🔑 Locks are the mechanism, not the goal

You rarely take locks manually — SQL Server acquires them automatically to enforce your isolation level. The reason to understand them is diagnostic: nearly every "the app randomly hangs" or "we get deadlock errors" problem traces back to how locks are being taken and held. Knowing the modes and lifetimes lets you fix the cause.

Two sessions (T1 and T2) both want to update the same two employees — the setup for both blocking and the classic deadlock.

Whenever a transaction updates Ada, it takes an exclusive lock on Row A; updating Alan locks Row B. If two transactions touch these two rows in different orders, they can end up each holding one row and waiting for the other — the deadlock we'll build in section 08.

✅ Keep "Row A = Ada, Row B = Alan" in mind

Every example below uses just these two rows and two sessions. That's all it takes to demonstrate a lock, a block, and a full deadlock — real systems are bigger, but the mechanics are identical.

03

Lock Modes

Different operations take different modes of lock. The mode determines what others can and can't do to the same data at the same time.

👀

Shared (S)

Taken to read data. Multiple transactions can hold S locks on the same row at once — readers don't block each other.

✍️

Exclusive (X)

Taken to modify (INSERT/UPDATE/DELETE). Incompatible with everything — no one else may read or write that data.

🔄

Update (U)

Taken while searching for rows to update. Only one U at a time; it converts to X when the change happens. Prevents a common deadlock.

📣

Intent (IS/IX/IU)

Placed on higher levels (page, table) to signal a lock exists below, so the engine checks compatibility cheaply.

🔑 Why the Update (U) lock exists

Consider two transactions that both read a row (S lock) intending to update it, then both try to upgrade to X — each waits for the other to drop its S lock: an instant deadlock. The U lock solves this: only one transaction can hold a U lock while searching, so they naturally serialize instead of deadlocking. It's compatible with shared reads but not with another U or an X.

✅ Intent locks in one line

Before locking a row, SQL Server puts an intent lock (like IX) on the page and table above it. That way, a transaction wanting a full table lock can see "someone holds a row lock in here" instantly, without scanning every row's lock — a performance optimization for the lock manager.

04

Lock Compatibility

Whether one transaction must wait for another comes down to whether their lock modes are compatible. This little matrix drives all blocking.

Held ↓ / Requested →Shared (S)Update (U)Exclusive (X)
Shared (S)✔ compatible✔ compatible✗ blocks
Update (U)✔ compatible✗ blocks✗ blocks
Exclusive (X)✗ blocks✗ blocks✗ blocks
🔑 Read the matrix in plain English

Many readers, one writer. Shared locks coexist (lots of readers are fine). An exclusive lock is compatible with nothing — while someone is writing a row, no one else may read or write it. An update lock tolerates shared readers but not another update or an exclusive, which is exactly what makes it a safe "I'm about to write this" marker.

✅ This is why writes block and reads (mostly) don't

Two SELECTs on the same row run side by side (S + S). A SELECT and an UPDATE on the same row conflict (S vs X), so one waits. Under row-versioning (RCSI/snapshot), reads don't take S locks at all, which is why enabling it dramatically cuts reader/writer blocking — a point we'll return to.

05

Granularity & Escalation

Locks can be taken at different sizes — a single row, a page, or a whole table. Bigger locks cost less to manage but block far more.

lock granularity — smaller = more concurrency, more overhead
Row
one row
→
Page
~8 KB of rows
→
Table
every row

A row lock blocks only that row — maximum concurrency, but each lock uses memory and there's management overhead. A table lock is one cheap lock but blocks everyone. SQL Server usually starts with fine-grained row locks and picks the level automatically.

⚠️ Lock escalation

When a single statement acquires too many row/page locks (by default around 5,000), SQL Server escalates them to a single table lock to save memory. That's efficient for the lock manager but a concurrency disaster: one statement now blocks the whole table. Escalation is often a sign a query is touching far more rows than it should — usually a missing index causing a scan.

✅ How to avoid unwanted escalation

The best fix is to touch fewer rows: add an index so the statement seeks a small set instead of scanning, or process large modifications in batches (e.g., delete 5,000 rows at a time in a loop). You can also disable escalation per table as a last resort, but reducing the row count is the real cure.

06

Blocking

Blocking is simply one transaction waiting for another to release a lock. It's normal, expected, and usually brief — it only becomes a problem when the wait is long.

T1
T2
UPDATE Ada → holds X lock on Row A
UPDATE Ada → waits (X vs X)…
COMMIT → releases Row A
…lock granted, proceeds, COMMIT

T2 isn't broken — it's just waiting its turn. The moment T1 commits and releases the lock, T2 continues. This one-directional wait always resolves on its own once the blocker finishes.

🔑 Blocking is only a problem when it's long

Short blocking is the normal cost of consistency and you'll always have some. It turns pathological when a transaction holds locks for a long time — a slow query, a transaction left open waiting for user input, or a big unindexed update. The fix is almost always to make the blocking transaction shorter or faster, not to remove locking.

07

Deadlocks: The Idea

A deadlock is a circular wait: two (or more) transactions each hold a lock the other needs, so neither can ever proceed. Unlike blocking, it never resolves on its own.

the circular wait that defines a deadlock
T1
holds Row A
wants Row B
⇄
T2
holds Row B
wants Row A

T1 is waiting for T2 to release Row B; T2 is waiting for T1 to release Row A. Each is waiting for the other, and neither will give up its lock until it finishes — but neither can finish. That mutual, circular dependency is the deadlock.

🔑 Deadlock ≠ heavy blocking

A deadlock is not "very slow blocking." Blocking is a line at one door that clears when the front person leaves. A deadlock is a genuine cycle with no natural end — waiting longer would never help. That's precisely why SQL Server has to step in and forcibly break it.

08

One in Action

The classic two-row deadlock, step by step — caused entirely by the two sessions touching the rows in opposite orders.

T1 (Ada → Alan)
T2 (Alan → Ada)
UPDATE Ada → X lock on Row A ✅
UPDATE Alan → X lock on Row B ✅
UPDATE Alan → wants Row B… waits on T2
UPDATE Ada → wants Row A… waits on T1
💀 chosen as victim → rolled back (error 1205)
lock granted → proceeds → COMMIT ✅

Each transaction grabbed one row, then reached for the row the other already held. The cycle formed the instant T2 asked for Row A. SQL Server's deadlock monitor detected it and killed T1 to let T2 finish.

✅ Spot the root cause

The bug isn't the locks — it's the ordering. T1 went Ada→Alan while T2 went Alan→Ada. If both had touched the rows in the same order, one would simply have blocked briefly and then proceeded — no cycle. That single insight is the key to prevention (section 10).

09

Detection & Victim

SQL Server actively watches for deadlock cycles and, when it finds one, kills a transaction to break it. That transaction is the "deadlock victim."

A background process (the deadlock monitor, waking roughly every few seconds) builds a wait-for graph and looks for cycles. When it finds one, it picks a victim, rolls that transaction back entirely, and raises error 1205 to that session. The other transaction(s) then get their locks and continue.

🔑 How the victim is chosen

By default, SQL Server picks the transaction that is cheapest to roll back — the one that has done the least work (smallest amount of transaction log to undo). You can influence this with SET DEADLOCK_PRIORITY (LOW / NORMAL / HIGH, or a number): a lower-priority session is chosen as the victim first, regardless of cost. Ties fall back to the rollback-cost rule.

⚠️ The victim's transaction is fully rolled back

Being the victim means all of that transaction's work is undone, not just the last statement. The application receives error 1205 and must decide what to do — which is why the standard response is to catch 1205 and retry the whole transaction (covered next). A deadlock isn't a corruption; it's a "please try again."

10

Prevention

Deadlocks are almost always preventable. A short ladder of habits eliminates the vast majority.

🔢 Access objects in a consistent order everywhere If every transaction touches Ada before Alan, the cycle can't form. The single biggest fix.
⏱️ Keep transactions short and fast Hold locks for less time → smaller window for a cycle. Never wait on user input mid-transaction.
📑 Add indexes so locks are narrow A seek locks a few rows; a scan locks many and invites escalation and conflicts.
📸 Use RCSI / snapshot to drop read locks Row-versioning means readers don't block writers, removing a whole class of deadlocks.
🔁 Catch error 1205 and retry Some deadlocks are unavoidable; make the app retry the victim transaction a few times.
sql
-- Retrying a deadlock victim (error 1205) a few times
DECLARE @retries INT = 3;
WHILE @retries > 0
BEGIN
    BEGIN TRY
        BEGIN TRANSACTION;
            -- ...always touch Ada, then Alan (consistent order)...
            UPDATE dbo.Employees SET Salary = Salary - 5000 WHERE Name = N'Ada';
            UPDATE dbo.Employees SET Salary = Salary + 5000 WHERE Name = N'Alan';
        COMMIT TRANSACTION;
        BREAK;                       -- success -> leave the loop
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0 ROLLBACK TRANSACTION;
        IF ERROR_NUMBER() = 1205 SET @retries -= 1;  -- deadlock: try again
        ELSE THROW;                    -- other errors: bubble up
    END CATCH;
END
🔑 Consistent ordering is the headline fix

Almost every classic deadlock comes from two code paths acquiring the same locks in different orders. If all transactions always access tables/rows in the same order, a cycle is impossible — the second transaction simply blocks and then proceeds. Combine that with short transactions and good indexes, and deadlocks become rare; the retry loop mops up the unavoidable remainder.

11

Blocking vs Deadlock

The distinction interviewers love, because people conflate them. They're fundamentally different situations.

BlockingDeadlock
ShapeOne-directional waitCircular (mutual) wait
Resolves itself?Yes — when the blocker releasesNo — never on its own
SQL Server intervenes?No (just waits)Yes — kills a victim
OutcomeWaiter proceeds after a delayVictim rolled back, error 1205
Normal?Yes, expected; bad only if longA bug/anti-pattern to prevent
🔑 The one-sentence answer

Blocking is a transaction waiting for a lock that will eventually free up; deadlock is two transactions each waiting for the other so neither ever frees up, and SQL Server must kill one to resolve it. Blocking is a queue; a deadlock is a standoff.

12

Diagnosing

The tools that show you what's locked, who's waiting, and how a deadlock happened.

🕸️

Deadlock graph

An XML/visual diagram of the exact deadlock — the transactions, resources, and who held/wanted what. Captured via Extended Events (the system_health session records them automatically).

🔒

sys.dm_tran_locks

A DMV listing every current lock: mode, resource, and which session holds or is waiting for it. The go-to for "what's locked right now."

⏳

sys.dm_exec_requests

Shows running requests with their blocking_session_id and wait type — how you find who is blocking whom.

📋

Blocked process report

An event that fires when blocking exceeds a threshold, capturing the blocked and blocking statements for later analysis.

✅ Where deadlocks are already recorded

You usually don't need to reproduce a deadlock to investigate it: SQL Server's built-in system_health Extended Events session automatically captures recent deadlock graphs. Open it, read the graph to see which two statements and resources formed the cycle, and you'll immediately see the inconsistent lock ordering to fix.

13

Mistakes & Best Practices

The habits that keep locking healthy and deadlocks rare.

🔢

Consistent access order

Always touch tables/rows in the same order across all code paths — the number-one deadlock preventer.

⏱️

Short transactions

Do the minimum between BEGIN and COMMIT; never hold a transaction open awaiting user input.

📑

Index to narrow locks

Seeks lock few rows; scans lock many and trigger escalation. Good indexes reduce contention.

📸

Consider RCSI

Row-versioning removes read locks, eliminating a large class of reader/writer deadlocks.

🔁

Retry on 1205

Treat deadlocks as retryable: catch error 1205 and re-run the transaction a few times.

🚫

Don't fight it with NOLOCK

Sprinkling NOLOCK to "avoid blocking" trades correctness for speed; fix the cause instead.

⚠️ Escalation and scans go hand in hand

A statement that scans a large table takes many locks and may escalate to a table lock, blocking everyone and raising deadlock risk. If you see escalation or table-level blocking, look for a missing index turning a seek into a scan — fixing the query usually fixes the locking as a side effect.

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is a lock? Junior

A temporary claim a transaction places on data (a row, page, or table) so that other transactions can't interfere while it reads or changes that data. Locks are how SQL Server enforces isolation — they're acquired and released automatically as transactions run.

Q02 Why does SQL Server use locks? Junior

To keep concurrent transactions from corrupting each other — preventing the anomalies like dirty reads and lost updates that the isolation levels promise to avoid. Locks make sure that while one transaction is modifying data, others don't read or overwrite it in conflicting ways.

Q03 What's the difference between a shared and an exclusive lock? Junior

A shared (S) lock is taken to read data, and several transactions can hold shared locks on the same row at once — readers don't block each other. An exclusive (X) lock is taken to modify data and is incompatible with every other lock, so while one transaction writes a row, no one else may read or write it.

Q04 What is blocking? Junior

Blocking is when one transaction has to wait because another holds an incompatible lock on the data it wants. It's normal and temporary — as soon as the first transaction releases its lock (by committing or rolling back), the waiting one proceeds. It's only a problem when the wait is long.

Q05 What is a deadlock? Junior

A deadlock is when two (or more) transactions each hold a lock the other needs, so they wait on each other in a circle and neither can ever finish. Because it never resolves on its own, SQL Server steps in and kills one of them to break the cycle.

Q06 What happens when a deadlock occurs? Junior

SQL Server detects the circular wait, picks one transaction as the "victim," rolls it back completely, and returns error 1205 to that session. The other transaction then gets its locks and continues. The victim's application typically retries the transaction.

Q07 What's the difference between blocking and a deadlock? Junior

Blocking is a one-way wait that resolves itself when the blocker releases its lock. A deadlock is a mutual, circular wait that never resolves on its own, so SQL Server must kill a victim. Blocking is a queue; a deadlock is a standoff.

Q08 Do reads take locks? Junior

Under the default locking behaviour, yes — a read takes a shared (S) lock, though it's usually released quickly and multiple reads can share it. Under row-versioning (RCSI/snapshot), reads don't take shared locks at all; they read a consistent version instead, which is why versioning greatly reduces reader/writer blocking.

Q09 Name the main lock modes. Mid

Shared (S) for reads, Exclusive (X) for modifications, and Update (U) taken while searching for rows to update. There are also Intent locks (IS/IX/IU) placed on pages and tables to signal that finer-grained locks exist below, plus specialized ones like schema and key-range locks.

Q10 What is an update (U) lock and why does it exist? Mid

A U lock is taken while a statement searches for the rows it's about to update. Only one transaction can hold a U lock on a resource, and it's compatible with shared reads but not with another U or an X. It exists to prevent a common deadlock: without it, two transactions could both hold shared locks intending to upgrade to exclusive, each waiting for the other to release — the U lock serializes them instead.

Q11 What are intent locks? Mid

Intent locks (IS, IX, IU) are placed on higher-level objects — the page and table — to indicate that a transaction holds or intends to hold locks on rows beneath them. They let the lock manager check compatibility efficiently: a transaction wanting a full table lock can see "row locks exist here" from the intent lock, without scanning every row.

Q12 What is lock granularity? Mid

The size of the thing being locked: a single row, a page (about 8 KB of rows), or an entire table. Finer granularity (row locks) allows more concurrency but uses more memory and management overhead; coarser granularity (table locks) is cheap to manage but blocks far more. SQL Server chooses automatically, usually starting fine.

Q13 What is lock escalation? Mid

When a single statement acquires too many fine-grained locks (by default around 5,000), SQL Server converts them into one coarser lock — typically a full table lock — to save memory. It's efficient for the lock manager but hurts concurrency because the statement now blocks the whole table. Frequent escalation usually means a query is touching too many rows, often due to a missing index.

Q14 How does SQL Server detect and resolve a deadlock? Mid

A background deadlock monitor periodically builds a wait-for graph of which sessions are waiting on which locks and checks for cycles. When it finds one, it chooses a victim, rolls that transaction back entirely, and raises error 1205 to it, freeing the locks so the other transaction can proceed.

Q15 What is the deadlock victim and how is it chosen? Mid

The victim is the transaction SQL Server rolls back to break the cycle. By default it picks the one that's cheapest to roll back (least transaction-log work done so far). You can override this with SET DEADLOCK_PRIORITY: the session with the lower priority is chosen as the victim regardless of cost.

Q16 How do you prevent deadlocks? Mid

The biggest lever is accessing tables and rows in a consistent order across all code paths, so a cycle can't form. Also keep transactions short, add indexes so statements lock few rows (seek not scan), consider RCSI/snapshot to remove read locks, and make the application catch error 1205 and retry the transaction for the deadlocks that remain.

Q17 Explain lock compatibility between S, U, and X. Senior

Shared is compatible with Shared and Update (many readers, plus one searcher, can coexist). Update is compatible with Shared but not with another Update or Exclusive (only one intended writer). Exclusive is compatible with nothing — while a row is being modified, no other lock is allowed on it. In short: reads share; a single U can sit among readers; a write excludes everyone.

Q18 Walk through a classic deadlock scenario. Senior

T1 updates Row A (takes X on A); concurrently T2 updates Row B (takes X on B). Then T1 tries to update Row B and waits on T2's lock; T2 tries to update Row A and waits on T1's lock. Now each holds what the other needs — a cycle. The deadlock monitor detects it and kills the cheaper-to-roll-back transaction, which gets error 1205; the survivor proceeds. The root cause is the two transactions locking A and B in opposite orders.

Q19 How does consistent access order prevent deadlocks? Senior

A deadlock cycle requires two transactions to acquire the same locks in different orders. If every transaction always acquires locks in the same global order (say, always Row A before Row B), then when a second transaction wants a lock the first holds, it simply blocks and waits — there's no way for the first to also be waiting on the second, so no cycle can form. It converts a potential deadlock into ordinary, self-resolving blocking.

Q20 How do indexes and isolation level affect locking and deadlocks? Senior

Indexes let a statement seek a small, precise set of rows, so it locks few rows and holds them briefly; without one, a scan locks many rows (and may escalate to a table lock), massively increasing contention and deadlock odds. Isolation level sets how many locks are taken and for how long: higher levels (repeatable read, serializable) hold read locks longer and add range locks, raising deadlock risk, while row-versioning (RCSI/snapshot) removes read locks entirely, cutting reader/writer deadlocks.

Q21 How do DEADLOCK_PRIORITY and retrying error 1205 work together? Senior

SET DEADLOCK_PRIORITY lets you nominate which session should be sacrificed first — e.g., set a low-value background job to LOW so a user-facing transaction is never the victim. But since being a victim is expected, robust code also wraps the transaction in a retry loop: catch error 1205 in CATCH, roll back, and re-run the whole transaction a few times (ideally with a tiny backoff). Priority decides who loses; the retry decides how the loser recovers.

Q22 How do you diagnose deadlocks and blocking in production? Senior

For deadlocks, read the deadlock graph — SQL Server's built-in system_health Extended Events session captures them automatically, showing the two statements, the resources, and who held/wanted what (revealing the inconsistent lock order). For live blocking, use sys.dm_exec_requests (its blocking_session_id and wait type), sys.dm_tran_locks (every current lock), and the blocked-process report for waits over a threshold.

Q23 What problems does lock escalation cause, and how do you mitigate it? Senior

Escalation turns thousands of row/page locks into a single table lock, so one statement suddenly blocks the whole table and concurrency collapses. Mitigations, in order of preference: add an index so the statement touches far fewer rows (avoiding the threshold); process large modifications in batches so no single statement crosses it; and, as a last resort, disable escalation on the table (ALTER TABLE … SET (LOCK_ESCALATION = DISABLE)), accepting higher lock memory. The row-count reduction is the real fix.

Q24 How does row-versioning reduce blocking and deadlocks? Senior

With RCSI or snapshot isolation, readers get a consistent committed version of each row from the version store instead of taking shared locks. So readers never block writers and writers never block readers, which eliminates the very common reader-vs-writer blocking and the deadlocks that arise from lock upgrades and read/write ordering. The costs are extra tempdb usage for the version store and (for full snapshot) possible update-conflict errors to retry — but for read-heavy systems it dramatically smooths concurrency.
