Functions, Views & Temp Storage 🧰

Three toolkits for organizing SQL — and each hides a performance trade-off. Functions package reusable logic (but scalar ones can be secretly slow). Views save a query behind a name (and, when indexed, can materialize it). And for scratch data, you choose among temp tables, table variables, and CTEs. Knowing when each is right — and when it quietly hurts — is the whole skill. No prior coding background assumed.

01

Overview

Functions return values or tables, views name a query, and temp storage holds intermediate rows. They overlap — and choosing well is mostly about performance.

All three exist to reuse and organize logic, but they behave very differently under the hood. A function is called inside a query and returns something (a value or a table). A view is a saved SELECT you query like a table. Temp storage holds a working set of rows partway through a process. The catch is that the "obvious" choice is often the slow one — scalar functions, multi-statement table functions, and table variables all have well-known performance pitfalls.

🔑 The theme: convenience vs the optimizer

Each tool trades developer convenience against how well the query optimizer can reason about it. The recurring lesson is that the set-based, inline options (inline TVFs, plain views, temp tables with statistics) usually let the optimizer produce good plans, while the procedural, opaque options (scalar UDFs, multi-statement TVFs, table variables) can hide cost and mislead its estimates.

03

Scalar Functions

A scalar user-defined function (UDF) takes inputs and returns a single value — usable anywhere an expression is allowed.

sql
CREATE FUNCTION dbo.fnAnnualBonus (@Salary DECIMAL(19,4))
RETURNS DECIMAL(19,4)
AS
BEGIN
    RETURN @Salary * 0.10;   -- 10% bonus
END;
GO

SELECT Name, dbo.fnAnnualBonus(Salary) AS Bonus
FROM dbo.Employees;

It reads cleanly and centralizes the bonus rule. For a single value or a handful of rows, it's fine. The problem appears when you call it across many rows — the subject of the next section.

🔑 Great for reuse, risky at scale

A scalar UDF is the natural way to name a reusable calculation. But because it's invoked once per row and (historically) runs as an opaque, procedural black box, using one in the SELECT or WHERE of a large query can be dramatically slower than the equivalent inline expression.

04

The Scalar Trap

The classic performance gotcha: a scalar UDF over many rows is called once per row, its cost is hidden from the plan, and it can force the whole query to run single-threaded.

🐢

Row-by-row

Called once for every row the query touches — millions of calls for a big table.

🕶️

Hidden cost

Historically the plan showed the UDF as nearly free, so the real expense was invisible.

🚫

Serial plans

A scalar UDF could force the entire query to run single-threaded (no parallelism).

🙈

Opaque to the optimizer

The engine couldn't see inside to estimate or optimize it well.

⚠️ The fix, and what changed in SQL Server 2019

Traditionally the remedy was to avoid scalar UDFs in hot queries — inline the expression, or convert the logic into an inline table-valued function and CROSS APPLY it. SQL Server 2019 added Scalar UDF Inlining, which automatically rewrites many scalar UDFs into the surrounding query so they behave like inline expressions — often a big, free speedup. Not every UDF qualifies, so on any version, prefer set-based/inline logic for performance-critical paths.

05

Inline TVFs

An inline table-valued function returns a table from a single SELECT. It's essentially a parameterized view — and the optimizer expands it inline, so it's fast.

sql
CREATE FUNCTION dbo.fnEmployeesByDept (@DeptId INT)
RETURNS TABLE            -- note: RETURNS TABLE, no BEGIN/END
AS
RETURN
(
    SELECT EmpId, Name, Salary
    FROM dbo.Employees
    WHERE DeptId = @DeptId
);
GO

SELECT * FROM dbo.fnEmployeesByDept(1);   -- query it like a table
🔑 An inline TVF = a view that takes parameters

Because it's a single RETURN (SELECT …) with no procedural body, SQL Server can inline its definition into the calling query and optimize the whole thing together — just as it would with a view. That means good estimates and good plans. Whenever you're tempted to write a scalar UDF or a multi-statement TVF for performance-sensitive code, see if an inline TVF does the job instead.

06

Multi-Statement TVFs

A multi-statement TVF declares a table, fills it with several statements, then returns it. Flexible — but historically a performance hazard because the optimizer can't see inside.

sql
CREATE FUNCTION dbo.fnEmployeesByDeptMulti (@DeptId INT)
RETURNS @result TABLE (        -- declares the shape of the returned table
    EmpId  INT,
    Name   NVARCHAR(100),
    Salary DECIMAL(19,4)
)
AS
BEGIN
    INSERT INTO @result
    SELECT EmpId, Name, Salary FROM dbo.Employees WHERE DeptId = @DeptId;
    -- ...could run more statements here...
    RETURN;
END;
⚠️ Why multi-statement TVFs can be slow

The returned table variable is a black box to the optimizer, which historically guessed a fixed row count (long assumed to be 1, later 100) regardless of how many rows it really holds. Bad estimates lead to bad plans — wrong join types, too-small memory grants — especially when the function feeds a larger query. SQL Server 2017+ added interleaved execution, which pauses to get a real row count for multi-statement TVFs, easing this considerably.

✅ Prefer inline whenever you can

If the logic can be written as a single SELECT, use an inline TVF — it's almost always faster because it's optimized as part of the calling query. Reserve multi-statement TVFs for genuinely procedural cases that can't be expressed inline, and be mindful of the estimate risk on older versions.

07

Views

A view is a named, saved SELECT. You query it like a table, but it stores no data — it's just a stored definition the engine expands when used.

sql
CREATE VIEW dbo.vwHighEarners
AS
SELECT EmpId, Name, Salary, DeptId
FROM dbo.Employees
WHERE Salary > 100000;
GO

SELECT Name, Salary FROM dbo.vwHighEarners WHERE DeptId = 1;
🧩

Abstraction

Hide a complex join/filter behind a simple name so callers don't repeat it.

🔒

Security

Expose only certain columns/rows; grant access to the view, not the base tables.

🔧

Stable interface

Change the underlying query while callers keep using the same view name.

🔑 A view stores a query, not data

By default a view holds no rows — each time you query it, SQL Server substitutes its definition into your statement and runs it against the base tables. So a plain view doesn't make things faster; it makes them simpler and safer to reuse. (The exception is an indexed view, which does store results — next.)

08

Indexed Views

Add a unique clustered index to a view and it becomes materialized — its results are physically stored and kept up to date, which can hugely speed up expensive aggregations.

sql
CREATE VIEW dbo.vwDeptTotals
WITH SCHEMABINDING          -- required for an indexed view
AS
SELECT DeptId,
       COUNT_BIG(*) AS EmpCount,   -- COUNT_BIG, not COUNT, is required
       SUM(Salary) AS TotalSalary
FROM dbo.Employees          -- two-part name (schema.table) required
GROUP BY DeptId;
GO

-- Materialize it: the unique clustered index stores the results on disk
CREATE UNIQUE CLUSTERED INDEX IX_vwDeptTotals
ON dbo.vwDeptTotals (DeptId);
🔑 Materialized = precomputed and stored

An indexed view actually stores its result rows and maintains them automatically as the base data changes. Queries can then read the precomputed aggregates instead of recomputing them every time — a big win for costly, frequently-run rollups. The requirements are strict: WITH SCHEMABINDING, a unique clustered index, two-part object names, COUNT_BIG(*) for counts, and only deterministic, allowed constructs.

⚠️ Not free — writes pay the price

Because the stored results must stay correct, every insert/update/delete to the base tables also maintains the indexed view — extra write cost, just like any index. Use indexed views for read-heavy aggregation where the query savings clearly outweigh the write overhead, not for volatile, write-heavy tables.

09

Updates & SCHEMABINDING

Simple views can be written through, and SCHEMABINDING locks a view to its base tables' shape.

You can INSERT/UPDATE/DELETE through a view when it maps cleanly to a single base table — no aggregates, no DISTINCT, no GROUP BY, and the modification touches only one underlying table. Complex or multi-table views generally aren't directly updatable (though an INSTEAD OF trigger can make them so).

🔑 What SCHEMABINDING does

WITH SCHEMABINDING binds the view (or function) to the exact schema of the objects it references, so those columns/tables can't be dropped or altered in a way that would break it — you'd have to drop the view first. It's required for indexed views and deterministic schema-bound functions, and it's a good safety habit anywhere you want to prevent surprise breakage from an underlying change.

✅ WITH CHECK OPTION

Adding WITH CHECK OPTION to an updatable view stops modifications that would produce rows no longer visible through the view. For example, updating a row through vwHighEarners so its salary drops below the threshold would be rejected — keeping writes consistent with the view's filter.

10

Temp Tables

A temp table (#name) is a real table living in tempdb for the duration of your session. It behaves like any table — you can index it and it gets statistics.

sql
-- Stage a working set, then reuse it several times
CREATE TABLE #HighEarners (
    EmpId  INT,
    Name   NVARCHAR(100),
    Salary DECIMAL(19,4)
);

INSERT INTO #HighEarners
SELECT EmpId, Name, Salary FROM dbo.Employees WHERE Salary > 100000;

CREATE INDEX IX_tmp ON #HighEarners (Salary);   -- yes, you can index it

SELECT * FROM #HighEarners WHERE Salary > 140000;
DROP TABLE #HighEarners;   -- (also auto-dropped when the session ends)
🔑 The big advantage: statistics and indexes

Because a temp table is a real table, SQL Server keeps statistics on it and you can add indexes — so the optimizer estimates its row counts accurately and joins to it efficiently. That makes temp tables the right choice for large intermediate results that you'll reference several times. The cost is more tempdb usage and potential recompiles as statistics change.

11

Table Variables

A table variable (@name) also holds rows, but it's a lighter, batch-scoped construct with limited statistics — great for small sets, risky for large ones.

sql
DECLARE @HighEarners TABLE (
    EmpId  INT,
    Name   NVARCHAR(100),
    Salary DECIMAL(19,4)
);

INSERT INTO @HighEarners
SELECT EmpId, Name, Salary FROM dbo.Employees WHERE Salary > 100000;

SELECT * FROM @HighEarners;
⚠️ The estimate pitfall

Historically the optimizer assumed a table variable held one row regardless of reality, because it doesn't maintain column statistics on them. Feed a large table variable into a join and you can get a badly wrong plan. SQL Server 2019+ added table-variable deferred compilation, which gets a real row count on first use and improves estimates — but on any version, table variables remain best for small result sets.

✅ Where table variables shine

They cause fewer recompiles, are convenient in functions (a multi-statement TVF returns one), and are simple for small, well-bounded sets. And unlike temp tables, changes to a table variable are not rolled back by a transaction rollback — occasionally useful (e.g., to keep audit rows) but also a surprise to watch for.

12

The Trade-offs

CTE, table variable, or temp table? The decision comes down to size, reuse, and whether the optimizer needs accurate statistics.

Which scratch space should I use? 🧭

📖 Just naming a step for readability within one query? Use a CTE — no storage, purely a named subquery (see the CTE topic).
🪶 Small result, used across a few statements, estimates don't matter much? Use a table variable (@t) — light, fewer recompiles.
🗃️ Large result, reused several times, needs indexes / good estimates? Use a temp table (#t) — real statistics and indexes.
💾 Expensive aggregation read constantly, base data changes rarely? Consider an indexed view — precomputed and maintained.

🔑 The one-line heuristic

CTE for readability in a single query; table variable for small sets; temp table for large or heavily-reused sets where the optimizer needs real statistics. When a query that uses a table variable is mysteriously slow on big data, swapping to a temp table is the classic fix.

Mistakes & Best Practices

The habits that keep these tools fast and predictable.

🐢

Avoid scalar UDFs in hot paths

Inline the expression or use an inline TVF; rely on 2019 UDF inlining only where it applies.

📊

Prefer inline TVFs

Over multi-statement TVFs — they optimize as part of the calling query.

🗃️

Temp table for big sets

When a table variable's poor estimates cause bad plans, switch to a #temp table.

💾

Indexed views cost writes

Use them for read-heavy aggregations, not volatile write-heavy tables.

🔒

SCHEMABINDING for safety

Prevents underlying changes from silently breaking views/functions; required for indexed views.

📖

CTE for clarity, not caching

A CTE is a named subquery, not stored — a heavily reused one may recompute; use a temp table instead.

⚠️ The convenient default is often the slow one

Scalar UDFs, multi-statement TVFs, and table variables are all easy to reach for and all carry a performance footgun at scale. When something is unexpectedly slow, check whether one of these is hiding a row-by-row execution or a bad cardinality estimate — and reach for the inline/set-based/temp-table alternative.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is a user-defined function? Junior

A reusable routine you create that takes inputs and returns either a single value (a scalar function) or a table (a table-valued function). You call it inside queries to package logic — like a bonus calculation or a parameterized result set — so it lives in one place instead of being repeated.

Q02 What is a scalar function? Junior

A user-defined function that returns one single value, usable anywhere an expression is allowed — in a SELECT list, a WHERE clause, and so on. For example, a function that takes a salary and returns 10% of it. Handy for reuse, but it can be slow when called across many rows.

Q03 What is a view? Junior

A named, saved SELECT statement that you query as if it were a table. It doesn't store data on its own — each time you use it, SQL Server runs its underlying query. Views are used to simplify complex queries, provide a stable interface, and control which columns/rows users can see.

Q04 What is a temp table? Junior

A real but temporary table (named with a # prefix) stored in the tempdb database for the life of your session. You can insert into it, index it, and query it like any table, and it's automatically dropped when the session ends. It's used to hold intermediate results during a multi-step process.

Q05 What is a table variable? Junior

A variable that holds a set of rows, declared with DECLARE @name TABLE (…). It's similar to a temp table but lighter and scoped to the current batch. It's best for small result sets; for large ones it can cause poor query plans because the optimizer doesn't keep detailed statistics on it.

Q06 What is a table-valued function? Junior

A user-defined function that returns a table instead of a single value, so you can query its result like a table (often with a parameter). There are two kinds: inline (a single SELECT, like a parameterized view) and multi-statement (builds the table with several statements). Inline ones perform much better.

Q07 How is a view different from a table? Junior

A table stores actual rows on disk; a view is just a saved query with no data of its own. When you query a view, the engine runs its underlying SELECT against the real tables. (The one exception is an indexed view, which does store its results.)

Q08 Does a view store data? Junior

Normally no — a view stores a query definition, not rows, and is evaluated each time you use it. The exception is an indexed view, which has a unique clustered index and physically stores (materializes) its results, kept in sync as the base data changes.

Q09 What's the difference between a scalar and a table-valued function? Mid

A scalar function returns a single value (used in expressions); a table-valued function returns a set of rows (used in the FROM clause like a table). Table-valued functions come in inline and multi-statement forms. Performance-wise, scalar UDFs and multi-statement TVFs have historic pitfalls, while inline TVFs optimize well.

Q10 Inline vs multi-statement TVF — what's the difference? Mid

An inline TVF is a single RETURN (SELECT …) with no procedural body — essentially a parameterized view that the optimizer expands and optimizes inline, so it's fast. A multi-statement TVF declares a table variable, fills it with several statements, and returns it; the optimizer can't see inside and historically misestimated its row count, often producing slower plans. Prefer inline whenever the logic allows.

Q11 What is the scalar function performance problem? Mid

A scalar UDF used across many rows is invoked once per row, its cost was hidden from the execution plan (so it looked cheap), it could force the whole query to run single-threaded, and the optimizer couldn't see inside it to optimize. The result is queries that are far slower than the equivalent inline expression. SQL Server 2019's scalar UDF inlining automatically fixes many cases by rewriting the UDF into the query.

Q12 What is an indexed (materialized) view? Mid

A view with a unique clustered index, which causes SQL Server to physically store its result rows and keep them updated as the base tables change. Queries can then read the precomputed results — great for expensive, frequently-run aggregations. It requires WITH SCHEMABINDING, COUNT_BIG(*), two-part names, and deterministic constructs, and it adds maintenance cost to every base-table write.

Q13 Temp table vs table variable — key differences? Mid

Both hold rows in tempdb, but a temp table (#t) is a full table with statistics and indexes and session scope, so the optimizer estimates it accurately — ideal for large or reused sets. A table variable (@t) is lighter, batch-scoped, has no column statistics (historically estimated as one row), causes fewer recompiles, and isn't affected by transaction rollback — best for small sets.

Q14 When would you use a CTE versus a temp table? Mid

Use a CTE to make a single query readable (or for recursion) — it's a named subquery with no stored data and one-statement scope. Use a temp table when a large intermediate result is expensive to compute and you'll reference it multiple times, because it stores the rows once and gives the optimizer statistics. A heavily reused CTE may be recomputed each reference, which a temp table avoids.

Q15 What is SCHEMABINDING? Mid

WITH SCHEMABINDING binds a view or function to the schema of the objects it references, so those tables/columns can't be dropped or altered in a way that would break it — you'd have to drop the view/function first. It's required for indexed views and for schema-bound deterministic functions, and it's a useful safeguard against accidental breaking changes.

Q16 Can you update data through a view? Mid

Yes, if the view is simple enough — it maps to a single base table with no aggregates, DISTINCT, or GROUP BY, and the modification affects only that one table. Complex or multi-table views generally aren't directly updatable unless you add an INSTEAD OF trigger. WITH CHECK OPTION can prevent updates that would push a row outside the view's filter.

Q17 Why are scalar UDFs historically slow, and what changed in 2019? Senior

Before 2019, a scalar UDF executed as a separate, procedural black box invoked once per row: its cost wasn't reflected in the plan, it prevented parallelism (forcing serial execution), and the optimizer couldn't inline or estimate it, so it often produced far more work than an equivalent expression. SQL Server 2019 introduced Scalar UDF Inlining, which transforms qualifying scalar UDFs into relational expressions embedded in the calling query — restoring costing, parallelism, and set-based optimization. Not all UDFs qualify, so avoiding them in hot paths remains good practice.

Q18 Why prefer inline TVFs over multi-statement TVFs? Senior

An inline TVF is expanded into the calling query and optimized as a whole, so it gets accurate estimates and can use indexes on the base tables directly — effectively a parameterized view. A multi-statement TVF materializes rows into a returned table variable that the optimizer treats opaquely, historically guessing a fixed cardinality (1, later 100), which produces poor plans when the function feeds a larger query. Interleaved execution (2017+) helps multi-statement TVFs by obtaining a real row count, but inline remains the faster default.

Q19 What are the requirements and trade-offs of indexed views? Senior

Requirements: the view must be created WITH SCHEMABINDING, reference base tables by two-part names, use COUNT_BIG(*) (not COUNT) if aggregating, avoid non-deterministic and disallowed constructs (many outer joins, subqueries, DISTINCT, etc.), and have a unique clustered index as its first index. Trade-off: reads of the precomputed results are fast (and Enterprise editions can auto-match queries to the view), but every base-table modification must maintain the view, adding write overhead. Best for read-heavy aggregation over relatively stable data.

Q20 Temp table vs table variable — statistics, recompiles, tempdb, transactions? Senior

Temp tables have full statistics (accurate estimates), support indexes, live in tempdb, participate in transactions (rolled back on rollback), but their statistics changes can trigger recompiles. Table variables lack column statistics (leading to poor estimates on large sets, mitigated by 2019 deferred compilation), support only limited indexing via constraints, cause fewer recompiles, also use tempdb, and are not affected by transaction rollback. So: temp table for large/reused sets needing good plans; table variable for small sets or when you want to avoid recompiles or preserve rows across a rollback.

Q21 When does a table variable's cardinality estimate hurt, and how do you mitigate it? Senior

It hurts when the table variable holds many rows and feeds a join or aggregation: the optimizer's low estimate leads to nested-loops where a hash join was needed, tiny memory grants and spills, and generally bad plans. Mitigations: switch to a temp table (real statistics); on 2019+ rely on table-variable deferred compilation; add an OPTION (RECOMPILE) on the statement so it's optimized with the actual row count; or, for multi-statement TVFs, lean on interleaved execution. The most reliable fix for large sets is simply the temp table.

Q22 What are the benefits and pitfalls of using views for security and abstraction? Senior

Benefits: a view can expose a curated subset of columns/rows and be granted independently of the base tables, giving a clean abstraction and a stable interface that shields callers from schema changes. Pitfalls: views don't inherently improve performance (a plain view is just its query), and nesting views deeply or layering them can produce surprisingly complex plans and make it hard to see the real cost. Column-level security and row-level security features are often a better fit than relying solely on views for access control, and over-abstraction can obscure performance problems.

Q23 How do determinism and SCHEMABINDING relate to functions and indexed views? Senior

A function is deterministic if it always returns the same output for the same input (no dependence on GETDATE(), randomness, etc.). Determinism plus SCHEMABINDING lets a function/view be used in places that require stability — most notably, an indexed view (and computed columns you want to index) must be schema-bound and built only from deterministic, precise constructs, so its stored results can be reliably maintained. SCHEMABINDING also prevents underlying schema changes from silently invalidating the object.

Q24 How do you choose among CTE, temp table, table variable, and indexed view for a workload? Senior

Match the tool to the need. A CTE when you only need to structure/readability a single query (or recurse), with no reuse concern. A table variable for small intermediate sets where estimate accuracy doesn't matter and you want fewer recompiles. A temp table when the intermediate set is large or referenced multiple times and the optimizer needs statistics/indexes to plan well. An indexed view when a costly aggregation is read very frequently over data that changes rarely, so precomputation pays off despite the write cost. In short: readability → CTE; small → @t; large/reused → #t; hot aggregation on stable data → indexed view.
