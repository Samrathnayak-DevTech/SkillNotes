Window Functions II: Offsets & Running Totals 📈

Part I ranked rows. Part II uses the same OVER clause to compare a row to its neighbors (LAG, LEAD) and to accumulate values in order (running totals, moving averages). The secret ingredient is the frame clause — the sliding window of rows each calculation sees — and its most infamous trap, ROWS vs RANGE. These power every "growth vs last month" and "cumulative sales" report. No prior coding background assumed.

01

The Window, Extended

Same OVER clause as ranking — but now the function either peeks at nearby rows or accumulates across an ordering.

In Part I, OVER (PARTITION BY … ORDER BY …) assigned each row a rank. Part II keeps that exact structure but swaps in two new families of functions: offset functions that fetch a value from another row (the previous month, the next row), and aggregate window functions (SUM, AVG, …) that, once you add ORDER BY, accumulate as they go — producing running totals and moving averages.

🏦 Analogy

Think of a bank statement. LAG is "what was my balance last line?" LEAD is "what's the next transaction?" A running total is the Balance column — each row's balance is the sum of everything up to and including that line. A moving average is "my average spend over the last three days." All of them look at a window of nearby rows.

🔑 The single new idea

Adding ORDER BY to an aggregate window function changes its meaning: without it, SUM(x) OVER () is the grand total repeated on every row; with it, SUM(x) OVER (ORDER BY d) becomes a running total up to each row. That one addition is the difference between "total" and "total so far."

The Sample Data

Running totals and month-over-month changes need an ordered time series, so this topic summarizes the Orders table into monthly totals.

sql
-- Derived from Orders by grouping on month (see the Aggregation notes).
-- For this page, treat MonthlySales as our table.
WITH MonthlySales AS (
    SELECT * FROM (VALUES
        ('2024-01', 500),
        ('2024-02', 800),
        ('2024-03', 300),
        ('2024-04', 700),
        ('2024-05', 600)
    ) AS m(SalesMonth, Sales)
)
SELECT * FROM MonthlySales;

✅ Keep these totals handy

Grand total = 2,900. Running totals build to it: 500 → 1,300 → 1,600 → 2,300 → 2,900. These recur in every running-total example below, so it's worth glancing at the numbers now.

03

LAG — Look Back

LAG fetches a value from an earlier row in the window — by default, the row immediately before the current one.

sql
SELECT SalesMonth, Sales,
       LAG(Sales) OVER (ORDER BY SalesMonth) AS PrevMonth
FROM MonthlySales;

Result

SalesMonthSalesPrevMonth
2024-01500NULL
2024-02800500
2024-03300800
2024-04700300
2024-05600700
🔑 Optional offset and default

LAG(Sales, 2) looks two rows back; LAG(Sales, 1, 0) looks one row back but returns 0 instead of NULL when there's no previous row. So the first row shows 0 rather than a blank. The general form is LAG(column, offset, default).

⚠️ The first row has no "previous"

January is the first row, so LAG returns NULL — there's nothing before it. Any arithmetic with that NULL (like a change calculation) is also NULL unless you supply a default or use COALESCE.

04

LEAD — Look Ahead

LEAD is the mirror of LAG: it fetches a value from a later row — by default, the next one.

sql
SELECT SalesMonth, Sales,
       LEAD(Sales) OVER (ORDER BY SalesMonth) AS NextMonth
FROM MonthlySales;

Result

SalesMonthSalesNextMonth
2024-01500800
2024-02800300
2024-03300700
2024-04700600
2024-05600NULL
🔑 LAG vs LEAD in one line

LAG = "the value behind me"; LEAD = "the value ahead of me." The last row's LEAD is NULL (nothing after it), just as the first row's LAG is NULL. Both take the same optional (offset, default) arguments.

05

Month-over-Month

The headline use of LAG: compare each row to the previous one to get a change or a growth percentage.

sql
SELECT SalesMonth, Sales,
       Sales - LAG(Sales) OVER (ORDER BY SalesMonth) AS MoMChange,
       CAST(
         (Sales - LAG(Sales) OVER (ORDER BY SalesMonth)) * 100.0
         / LAG(Sales) OVER (ORDER BY SalesMonth)
       AS DECIMAL(5,1)) AS PctChange
FROM MonthlySales;

Result

SalesMonthSalesMoMChangePctChange
2024-01500NULLNULL
2024-02800+300+60.0
2024-03300−500−62.5
2024-04700+400+133.3
2024-05600−100−14.3
⚠️ Two things bite here

First, integer division: multiply by 100.0 (a decimal), not 100, or the percentage truncates to a whole number. Second, the first month's change is NULL because there's no prior month — wrap in COALESCE(..., 0) if you'd rather show 0. Watch too for a prior value of 0, which would cause a divide-by-zero.

06

Aggregate Windows

The familiar aggregates — SUM, AVG, COUNT, MIN, MAX — become window functions the moment you add OVER. Without ORDER BY, they summarize the whole partition beside every row.

sql
-- No ORDER BY inside OVER: the grand total is repeated on every row
SELECT SalesMonth, Sales,
       SUM(Sales) OVER () AS GrandTotal
FROM MonthlySales;

Result

SalesMonthSalesGrandTotal
2024-015002,900
2024-028002,900
2024-033002,900
......2,900
🔑 This is the "% of total" building block

SUM(Sales) OVER () gives you the total right next to each row's own value — perfect for computing each month's share of the whole (Sales / SUM(Sales) OVER ()), all without a GROUP BY or a self-join. Add ORDER BY and the very same function turns into a running total, which is the next section.

07

Running Totals

Add ORDER BY to an aggregate window and it accumulates — each row shows the total of everything up to and including itself.

sql
SELECT SalesMonth, Sales,
       SUM(Sales) OVER (ORDER BY SalesMonth) AS RunningTotal
FROM MonthlySales;

Result

SalesMonthSalesRunningTotal
2024-01500500
2024-028001,300
2024-033001,600
2024-047002,300
2024-056002,900

Each RunningTotal is the sum of all sales up to that month: 500, then 500+800, then +300, and so on, ending at the grand total 2,900.

Which windowed calculation do I need? 🧭

⏮️ Need the previous or next row's value? Use LAG / LEAD (e.g., month-over-month change).
➕ Need a cumulative total up to each row? Use SUM() OVER (ORDER BY …) — the running total.
📊 Need a smoothed average over recent rows? Use AVG() OVER (… ROWS n PRECEDING) — a moving average.
🥧 Need each row's share of the whole group? Divide by SUM() OVER (PARTITION BY grp) — no ORDER BY.
✅ Reset per group with PARTITION BY

Add PARTITION BY and each group gets its own running total that restarts from zero — e.g., SUM(Sales) OVER (PARTITION BY Region ORDER BY SalesMonth) gives a separate cumulative line per region.

08

The Frame Clause

The frame is the exact set of rows — relative to the current row — that an aggregate window looks at. It's what makes "running total" and "3-row moving average" different.

sql
-- A running total, with the frame written out explicitly
SELECT SalesMonth, Sales,
       SUM(Sales) OVER (
           ORDER BY SalesMonth
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS RunningTotal
FROM MonthlySales;

🔑 Two common frames

Running total: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (everything so far). Moving average of the last 3: ROWS BETWEEN 2 PRECEDING AND CURRENT ROW (this row plus the two before it). The frame slides down with the current row, which is why these are called "windowed" calculations.

09

ROWS vs RANGE

The subtlest, most interview-worthy detail in window functions — and the reason a running total can quietly give "wrong" numbers.

ROWS counts physical rows: "the 2 rows before me." RANGE works on values: it includes every row that ties with the current row on the ORDER BY value — its "peers." When your ordering column is unique (like our distinct months), they give the same result. When there are ties, they diverge.

A tie makes them differ

Imagine two sales rows sharing the same date, then a third:

DaySalesSUM … ROWS (row by row)SUM … RANGE (peers lumped)
Day 1100100200
Day 1100200200
Day 2100300300

With RANGE, both Day-1 rows are peers, so both show the same running total (200) — RANGE includes all tied rows through the current one. With ROWS, they're counted one at a time (100, then 200). Same data, different running totals.

⚠️ The default frame is RANGE — and that surprises people

When you write SUM(x) OVER (ORDER BY d) with no explicit frame, SQL Server uses RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW by default. On tied ordering values this gives the "peer-lumped" behavior above, which is usually not what people expect from a running total — and RANGE can also be slower (it may spool to a worktable).

✅ Habit: write ROWS explicitly

For running totals and moving averages, spell out ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. It gives the row-by-row behavior you almost always want, is unaffected by ties, and typically performs better. Only reach for RANGE when you specifically want peer rows grouped together.

10

Moving Averages

Change the frame from "everything so far" to "the last N rows" and a running sum becomes a moving average — the classic way to smooth a noisy trend.

sql
-- 3-month moving average (this month + the two before it)
-- CAST to DECIMAL so we don't get integer-division truncation
SELECT SalesMonth, Sales,
       AVG(CAST(Sales AS DECIMAL(10,2))) OVER (
           ORDER BY SalesMonth
           ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
       ) AS MovingAvg3
FROM MonthlySales;

Result

SalesMonthSalesMovingAvg3Averaged over
2024-01500500.00500
2024-02800650.00500, 800
2024-03300533.33500, 800, 300
2024-04700600.00800, 300, 700
2024-05600533.33300, 700, 600
🔑 Early rows have a shorter window

January averages just itself (only one row exists before the frame fills), February averages two rows, and from March on the full three-row window applies. That's expected behavior — the frame simply can't reach rows that don't exist yet. Once the series is long enough, every row uses the full N.

11

FIRST_VALUE & LAST_VALUE

Grab the first or last value in the window — useful for "compare each row to the opening/ closing figure." LAST_VALUE hides a famous trap.

sql
SELECT SalesMonth, Sales,
       FIRST_VALUE(Sales) OVER (ORDER BY SalesMonth) AS FirstMonth,
       LAST_VALUE(Sales) OVER (
           ORDER BY SalesMonth
           ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
       ) AS LastMonth
FROM MonthlySales;

Result: FirstMonth = 500 (January) on every row; LastMonth = 600 (May) on every row — because the explicit frame extends all the way to UNBOUNDED FOLLOWING.

⚠️ The LAST_VALUE trap

Write LAST_VALUE(Sales) OVER (ORDER BY SalesMonth) without a frame and you'll get the current row's value, not the partition's last — a baffling result at first. Why? The default frame ends at CURRENT ROW, so "the last value seen so far" is always the current row. The fix is the explicit ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING shown above, which lets the window see all the way to the end.

🔑 Often MIN/MAX OVER is simpler

If you want the highest or lowest value in a partition (rather than the value at a specific ordered position), MAX(Sales) OVER (PARTITION BY …) or MIN(…) avoids the frame headache entirely. Reach for FIRST_VALUE/LAST_VALUE when the order matters (opening vs closing), not just the extreme.

12

Percent of Total

Combine an ordered window (running total) with an unordered one (grand total) to express each row as a share — and a cumulative share.

sql
SELECT SalesMonth, Sales,
       CAST(Sales * 100.0 / SUM(Sales) OVER ()
            AS DECIMAL(5,1)) AS PctOfTotal,
       CAST(SUM(Sales) OVER (ORDER BY SalesMonth) * 100.0
            / SUM(Sales) OVER ()
            AS DECIMAL(5,1)) AS RunningPct
FROM MonthlySales;

Result

SalesMonthSalesPctOfTotalRunningPct
2024-0150017.217.2
2024-0280027.644.8
2024-0330010.355.2
2024-0470024.179.3
2024-0560020.7100.0
🔑 Two windows, one query

PctOfTotal divides each row by SUM() OVER () (the grand total). RunningPct divides the running total by the grand total, so it climbs to 100% at the last row. You can mix an ordered window and an unordered window in the same SELECT — a compact, self-join-free way to build share-of-total reports.

13

Mistakes & Best Practices

The habits that keep running-total and offset queries correct and fast.

🪟

Write ROWS explicitly

For running totals and moving averages, spell out ROWS BETWEEN … to avoid the surprising default RANGE frame.

🎯

Fix LAST_VALUE's frame

LAST_VALUE needs ROWS … UNBOUNDED FOLLOWING, or it just returns the current row.

🔢

Avoid integer division

Multiply by 100.0 and cast to DECIMAL in percentages and averages, or they truncate.

👻

Handle NULL edges

The first LAG / last LEAD is NULL; use a default arg or COALESCE, and guard divide-by-zero.

🗂️

PARTITION BY to reset

Add it to restart running totals and moving averages per group (per region, per customer).

⚡

Index the order columns

An index matching PARTITION BY + ORDER BY lets the engine skip a sort — the main perf lever.

⚠️ Don't reach for a self-join or cursor

Before window functions, running totals were done with slow self-joins or row-by-row cursors. Those are far slower and harder to read. If you catch yourself joining a table to itself "to add up everything up to this row," that's a running total — use SUM() OVER (ORDER BY …) instead.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What does LAG do? Junior

LAG returns a value from an earlier row in the ordered window — by default the row immediately before the current one. It's how you compare each row to the previous one, like "this month's sales vs last month's." The very first row has no previous row, so it returns NULL.

Q02 What does LEAD do? Junior

LEAD is the mirror of LAG: it returns a value from a later row — by default the next one. Useful for "compare this row to the one that follows." The last row has nothing after it, so LEAD returns NULL there.

Q03 What is a running total and how do you compute one? Junior

A running (cumulative) total is the sum of all values up to and including the current row — like a bank balance. You compute it with an aggregate window and an order: SUM(x) OVER (ORDER BY d). Each row shows the total so far, building up to the grand total on the last row.

Q04 What's the difference between SUM() OVER () and SUM() OVER (ORDER BY …)? Junior

Without ORDER BY, SUM(x) OVER () gives the grand total of the whole partition, repeated on every row. With ORDER BY, it becomes a running total that accumulates up to each row. Adding the order is what turns "total" into "total so far."

Q05 What does LAG return on the first row (or LEAD on the last)? Junior

NULL, because there's no row before the first (or after the last). You can supply a default as the third argument — LAG(x, 1, 0) returns 0 instead of NULL — or wrap the result in COALESCE. Remember that arithmetic involving that NULL is also NULL.

Q06 Can you make LAG look more than one row back? Junior

Yes — the second argument is the offset. LAG(x, 2) looks two rows back, LAG(x, 3) three, and so on. The full form is LAG(column, offset, default), where the optional default is what to return when there's no row at that distance.

Q07 What is a moving average? Junior

An average computed over a sliding window of the most recent N rows, used to smooth out noise in a trend. For a 3-period moving average you average the current row and the two before it: AVG(x) OVER (ORDER BY d ROWS BETWEEN 2 PRECEDING AND CURRENT ROW). As you move down the rows, the window slides along.

Q08 How is a running total different from GROUP BY? Junior

GROUP BY collapses rows into one total per group. A running total keeps every row and shows the cumulative sum at each row. So GROUP BY answers "what's the total?" once, while the window function answers "what's the total so far?" on every line — detail and accumulation together.

Q09 How do you compute month-over-month change and percent change? Mid

Subtract the previous row from the current using LAG: Sales - LAG(Sales) OVER (ORDER BY month) for the change. For percent change, divide that by the previous value and multiply by 100.0 (a decimal, to avoid integer truncation): (Sales - LAG(Sales) OVER (…)) * 100.0 / LAG(Sales) OVER (…). Guard against a previous value of 0 (divide-by-zero) and the NULL first row.

Q10 What is the frame clause? Mid

It defines exactly which rows — relative to the current row — an aggregate window includes. You write it as ROWS BETWEEN <start> AND <end>, using boundaries like UNBOUNDED PRECEDING, n PRECEDING, CURRENT ROW, n FOLLOWING, and UNBOUNDED FOLLOWING. It's what distinguishes a running total (everything so far) from a moving average (the last N rows).

Q11 How do you compute a 3-month moving average? Mid

AVG(Sales) OVER (ORDER BY month ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) — the current month plus the two before it. Cast Sales to DECIMAL first if it's an integer, or the average truncates. Early rows (before three exist) average over whatever rows are available.

Q12 What's the difference between ROWS and RANGE? Mid

ROWS counts physical rows relative to the current one ("the 2 rows before me"). RANGE works on the ORDER BY values and includes all rows that tie with the current row (its "peers"). On a unique ordering they're identical; with ties, RANGE lumps the tied rows together, which can change a running total.

Q13 What's the default frame when you use ORDER BY but no explicit frame? Mid

RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. That means "everything from the start of the partition through the current row's peer group." On tied ordering values it produces peer-lumped results that surprise people, and RANGE can be slower. Writing ROWS explicitly gives the row-by-row behavior you usually want.

Q14 What is the LAST_VALUE trap and how do you fix it? Mid

With just LAST_VALUE(x) OVER (ORDER BY d), the default frame ends at the current row, so "the last value so far" is always the current row — not the partition's final value. Fix it by specifying the full frame: ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING, so the window can see all the way to the end.

Q15 How do you compute each row's percent of the total? Mid

Divide the row's value by the grand total from an unordered window: Sales * 100.0 / SUM(Sales) OVER (). The OVER () with no order gives the total of the whole set on every row. Use 100.0 and a cast to DECIMAL so you get real decimals, not truncated integers.

Q16 How do you reset a running total per group? Mid

Add PARTITION BY to the window: SUM(Sales) OVER (PARTITION BY Region ORDER BY month ROWS …). Each region then gets its own cumulative total that restarts from zero at the start of that region's rows, independent of the others.

Q17 Explain ROWS vs RANGE with an example and their performance implications. Senior

Given two rows tied on the order column, ROWS processes them individually (running totals of, say, 100 then 200), while RANGE treats them as peers and gives both the same total (200). ROWS is generally faster because it computes on a fixed count of rows; RANGE may build a worktable/spool to resolve peer groups. Since RANGE is the default with a bare ORDER BY, explicitly specifying ROWS is both a correctness and a performance habit.

Q18 Why does LAST_VALUE return the current row by default? Senior

Because when you supply ORDER BY, the default frame is … UNBOUNDED PRECEDING AND CURRENT ROW. LAST_VALUE returns the last row within the frame, and since the frame ends at the current row, that's the current row itself. Extending the frame to UNBOUNDED FOLLOWING makes it look at the whole partition and return the true final value.

Q19 What are the implications of the default RANGE frame for running totals with ties? Senior

With ties on the order column, the default RANGE frame gives every tied row the same cumulative value (the total through the end of that peer group), rather than a distinct step per row. If a stakeholder expects each transaction to add its own increment, the numbers look wrong. Using ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW restores the expected row-by-row accumulation regardless of ties.

Q20 What drives window-aggregate performance, and how does the frame matter? Senior

The dominant cost is ordering the rows for the PARTITION BY + ORDER BY; an index whose key matches those columns lets the engine avoid a Sort operator. The frame matters too: ROWS frames can often use an efficient streaming "window spool," while RANGE and wide frames may spill to a worktable in tempdb. Check the plan for Sort and Window Spool operators, and provide a supporting index plus an explicit ROWS frame.

Q21 LAG/LEAD vs a self-join — what are the trade-offs? Senior

Before window functions, "previous row" comparisons used a self-join on row numbers or dates, which is verbose, error-prone (especially with gaps/ties), and often slower because it scans the table twice. LAG/LEAD express the same intent in one clause, compute in a single ordered pass, and handle offsets cleanly. Self-joins linger only where you need arbitrary, non-positional matching that offsets can't express.

Q22 How would you build a running total that resets on a condition? Senior

You create a grouping key that increments whenever the reset condition occurs, then run the window total partitioned by that key. A common technique: use LAG (or a flag) to detect the reset boundary, take a running SUM of that flag to form island ids, then SUM(x) OVER (PARTITION BY island_id ORDER BY d ROWS …). This "gaps-and-islands" pattern is the general way to reset accumulations mid-stream.

Q23 How do integer division and NULLs affect windowed averages and percentages? Senior

If the input is an integer type, AVG and percentage math do integer division and drop the fraction — cast to DECIMAL (or multiply by 100.0) to fix it. NULLs are ignored by aggregate windows (so an AVG divides by the count of non-NULLs, like regular aggregates), and LAG/LEAD at the edges return NULL — which propagates through arithmetic. Use defaults or COALESCE, and guard divide-by-zero when a denominator can be 0.

Q24 When would you use FIRST_VALUE/LAST_VALUE versus MIN/MAX OVER? Senior

Use FIRST_VALUE/LAST_VALUE when position in an ordering matters — the opening or closing value of a series (first login, latest status). Use MIN/MAX OVER when you want the smallest/largest value regardless of position. The value-based ones also sidestep the LAST_VALUE frame trap, so if "the maximum in the group" is really what you need, MAX() OVER (PARTITION BY …) is simpler and safer.
