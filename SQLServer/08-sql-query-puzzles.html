Classic Query Puzzles 🧩

Three problems appear in SQL interviews again and again: find the Nth-highest value, find duplicate rows, and delete duplicates keeping one. This page solves all three on our schema — Nth-highest three different ways (subquery, DENSE_RANK, OFFSET…FETCH) — and dwells on the two traps that sink candidates: TOP/DISTINCT tricks that break on ties, and deleting duplicates with no deterministic rule for which row to keep. No prior coding background assumed.

01

The Puzzle Landscape

Three canonical puzzles, and the tools from earlier days that solve them — ranking, subqueries, and CTEs, now working together.

None of these puzzles needs a new feature. They're a workout that combines what you already know: DENSE_RANK from window ranking, the CTE-and-filter pattern, correlated subqueries, and ROW_NUMBER. What interviewers actually test is whether you handle ties and determinism correctly — the places where quick tricks silently give wrong answers.

🔑 The theme of the whole page

Every puzzle has an "obvious" answer that works on clean data and fails on messy data. The two failure modes are ties (equal values that a row-counting trick can't distinguish) and non-determinism (no rule for which of several equal candidates to pick). Master those two and the puzzles become routine.

03

Puzzle A · The Problem

"Find the 2nd-highest salary." Simple to say — but the first thing to do is pin down what it means, because ties make it ambiguous.

Before writing SQL, clarify the definition. "2nd highest" can mean three different things:

  • 2nd distinct value — ignore duplicates; the second-largest different salary. (Here: 120,000.) This is what people usually mean.
  • 2nd row — the second row when sorted, which could equal the first if the top value is tied.
  • Medal-style 2nd place — ties at the top consume the position.
🔑 Say the definition out loud in the interview

Stating "I'll assume you mean the 2nd-highest distinct salary" is itself a point in your favour — it shows you've spotted the tie ambiguity. All three approaches below target the distinct-value meaning, and each can be adjusted for the others. Our expected answer throughout is 120,000.

04

Way 1 · Correlated Subquery

The oldest, most portable approach: a salary is the Nth-highest if exactly N−1 distinct salaries are strictly greater than it.

sql
-- 2nd highest: the salary with exactly 1 distinct salary above it (N - 1 = 1)
SELECT DISTINCT e1.Salary
FROM dbo.Employees AS e1
WHERE (
    SELECT COUNT(DISTINCT e2.Salary)
    FROM dbo.Employees AS e2
    WHERE e2.Salary > e1.Salary
) = 1;   -- change 1 to N-1 for the Nth highest

For salary 120,000, the only distinct salary strictly greater is 145,000 — a count of 1, which matches N−1 = 1. So it's kept. Result: 120,000. The COUNT(DISTINCT …) is what makes it count values, not rows, so the tie doesn't fool it.

🔑 Why this is portable

It uses nothing but SELECT, COUNT(DISTINCT), and a comparison — supported by essentially every SQL database and every version. That's its main appeal. Its downside is performance: it's a correlated subquery that conceptually re-counts for each row, so it scales poorly on large tables.

05

Way 2 · DENSE_RANK

The clearest modern approach, and the one to reach for by default: rank distinct salaries and pick rank N.

sql
WITH Ranked AS (
    SELECT Salary,
           DENSE_RANK() OVER (ORDER BY Salary DESC) AS dr
    FROM dbo.Employees
)
SELECT DISTINCT Salary
FROM Ranked
WHERE dr = 2;   -- dr = N for the Nth highest

DENSE_RANK gives tied salaries the same rank with no gaps, so distinct values map exactly to 1, 2, 3, 4. Rank 2 is 120,000 (shared by Alan and Edsger); DISTINCT collapses those two rows to a single value. Result: 120,000.

✅ Why DENSE_RANK is the default choice

It reads exactly like the requirement ("the 2nd distinct highest"), it handles ties correctly by design, and changing N is a one-character edit. Use RANK instead if you want medal-style semantics, or ROW_NUMBER if you truly want the 2nd row regardless of ties — the choice of ranking function is the choice of meaning.

06

Way 3 · OFFSET…FETCH

The pagination-style approach: sort the distinct salaries, skip N−1, then take one.

sql
-- Distinct salaries first, so we skip distinct VALUES (not tied rows)
SELECT Salary
FROM (SELECT DISTINCT Salary FROM dbo.Employees) AS s
ORDER BY Salary DESC
OFFSET 1 ROWS          -- skip N-1 = 1 value
FETCH NEXT 1 ROW ONLY;  -- take just the next one

After the DISTINCT subquery, the ordered values are 145,000 · 120,000 · 110,000 · 95,000; skipping 1 lands on 120,000, and we fetch that one row.

⚠️ Two things to remember

First, OFFSET…FETCH requires an ORDER BY — it's meaningless without a defined order. Second, the DISTINCT is essential: skipping over raw rows would skip a tied duplicate instead of a distinct value, giving the wrong "Nth highest." OFFSET…FETCH is available in SQL Server 2012 and later.

Comparing the Three

All three return 120,000. The differences are readability, performance, portability, and how they extend.

Which one should I pick? 🧭

🎖️ Want the clearest, tie-safe answer for "Nth distinct highest"? Use DENSE_RANK() = N in a CTE — the default recommendation.
🌍 Need it to run on any database / very old SQL? Use the correlated subquery with COUNT(DISTINCT …).
📄 Thinking in pagination terms (skip N, take M)? Use OFFSET N-1 ROWS FETCH NEXT 1 over a DISTINCT set.
🚫 Tempted by a quick TOP/DISTINCT trick? Don't — it breaks on ties. See the next section.
08

The TOP/DISTINCT Trap

The "clever" one-liners people reach for under pressure — and exactly why they produce wrong answers when values tie.

sql
-- The classic "TOP N then take the smallest" trick
SELECT MIN(Salary) AS SecondHighest
FROM (
    SELECT TOP 2 Salary
    FROM dbo.Employees
    ORDER BY Salary DESC
) AS t;

On our data this happens to return 120,000 — but it's fragile. The trap: TOP 2 returns two rows, not two distinct values. Suppose the top salary were tied — say two people both earned 145,000:

Salaries (desc)TOP 2 returnsMIN of thoseCorrect 2nd distinct
145,000 · 145,000 · 120,000 · …145,000 · 145,000145,000 ❌120,000
⚠️ Row-counting tricks can't see distinct values

Because the top value repeats, TOP 2 grabs both copies of 145,000, and MIN reports 145,000 as the "2nd highest" — wrong. The same flaw hits OFFSET…FETCH or ROW_NUMBER = 2 applied to raw rows: they skip a duplicate instead of a distinct value. The fixes are to work on a DISTINCT set or, better, use DENSE_RANK, which is built to count distinct values.

🔑 The rule to remember

"Nth highest value" is a question about distinct values, but TOP, ROW_NUMBER, and a bare OFFSET all count rows. Whenever the two can differ (i.e., whenever ties are possible), use DENSE_RANK or de-duplicate first. Don't rely on a trick that only works because your sample data happened to have no ties at the top.

09

Puzzle B · Finding Duplicates

Group by the columns that define a duplicate, then keep only the groups that have more than one row.

sql
-- Which (Name, DeptId, Salary) combinations appear more than once?
SELECT Name, DeptId, Salary, COUNT(*) AS Copies
FROM dbo.EmployeeStaging
GROUP BY Name, DeptId, Salary
HAVING COUNT(*) > 1;

Result

NameDeptIdSalaryCopies
Alan1120,0002
Grace2110,0002
🔑 GROUP BY defines what "duplicate" means

The columns you list in GROUP BY are your definition of a duplicate. HAVING COUNT(*) > 1 then keeps only the groups appearing more than once. (Recall from the aggregation notes: WHERE filters rows before grouping, HAVING filters groups after.) Choose the grouping columns to match the business rule for "the same record."

10

Which Rows Repeat

Grouping tells you which values repeat; to act on the actual rows (to delete extras) you need to label each copy. ROW_NUMBER does that.

sql
SELECT StagingId, Name, DeptId, Salary,
       ROW_NUMBER() OVER (
           PARTITION BY Name, DeptId, Salary   -- one counter per duplicate group
           ORDER BY StagingId                  -- lowest id = the "keeper"
       ) AS rn
FROM dbo.EmployeeStaging;

Result

StagingIdNameSalaryrnFate
1Ada145,0001keep
2Alan120,0001keep
3Alan120,0002delete
4Grace110,0001keep
5Grace110,0002delete
6Katherine95,0001keep
🔑 rn = 1 is the survivor

Within each duplicate group, ROW_NUMBER stamps one row as rn = 1 and any extras as rn = 2, 3, …. That instantly separates the "keeper" (rn = 1) from the rows to remove (rn > 1). The ORDER BY decides which row becomes rn = 1 — which is the crux of the next two sections.

11

Puzzle C · Deleting Duplicates

Wrap that numbering in a CTE and delete every row with rn > 1. Deleting through the CTE removes the rows from the real table.

sql
WITH Dupes AS (
    SELECT StagingId,
           ROW_NUMBER() OVER (
               PARTITION BY Name, DeptId, Salary
               ORDER BY StagingId          -- keep the lowest id
           ) AS rn
    FROM dbo.EmployeeStaging
)
DELETE FROM Dupes
WHERE rn > 1;   -- removes StagingId 3 and 5

After the delete

StagingIdNameDeptIdSalary
1Ada1145,000
2Alan1120,000
4Grace2110,000
6Katherine295,000
🔑 You can DELETE from a CTE

Because this CTE maps to a single underlying table, DELETE FROM Dupes deletes the matching rows from EmployeeStaging itself. Rows 3 and 5 (the rn > 1 copies) are removed; one Alan and one Grace remain. This ROW_NUMBER-in-a-CTE-then-DELETE pattern is the standard, safe way to de-duplicate a table in SQL Server.

✅ Preview before you delete

Run the query from section 10 (the SELECT with rn) first to eyeball exactly which rows will go. Swapping the final DELETE back to a SELECT * is a cheap safety check — especially in production, ideally inside a transaction you can roll back.

12

Which One to Keep

The most important detail in the whole dedup puzzle: the ORDER BY inside ROW_NUMBER must be deterministic, or your delete becomes a coin flip.

If two rows tie on the ORDER BY columns, which one gets rn = 1 is undefined — it can differ run to run. When the duplicate rows are byte-for-byte identical, that doesn't matter (either copy is fine). But when they differ in a column you care about, picking the wrong survivor is a real bug.

sql
-- Keep the MOST RECENT record per business key (e.g., duplicate customers)
WITH Dupes AS (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY Email                 -- the business key
               ORDER BY UpdatedAt DESC, CustomerId DESC  -- newest wins; id breaks ties
           ) AS rn
    FROM dbo.Customers
)
DELETE FROM Dupes WHERE rn > 1;
⚠️ Add a unique tie-breaker

Ordering only by UpdatedAt DESC is not deterministic if two rows share the same timestamp — the survivor is arbitrary. Append a unique column (like the primary key: , CustomerId DESC) so exactly one row is guaranteed rn = 1 every time. A deterministic keep-rule is the difference between a repeatable cleanup and a random one.

🔑 Decide the rule before you write the query

"Keep the earliest," "keep the most recent," "keep the one with the most complete data" — these are business decisions, and they map directly to the ORDER BY. Always be able to state, in one sentence, which duplicate survives and why. That sentence is what the interviewer is listening for.

13

Mistakes & Best Practices

The habits that keep these puzzles correct under interview pressure and in production.

🎖️

DENSE_RANK for Nth-highest

It counts distinct values, so it's tie-safe and reads like the requirement. Default to it.

🚫

Distrust TOP/DISTINCT tricks

They count rows, not values, and silently break when the top value is tied.

❓

Clarify "Nth highest"

Distinct value vs Nth row vs medal-style — state your assumption before coding.

🎯

Deterministic keep-rule

Include a unique tie-breaker in ROW_NUMBER's ORDER BY so exactly one row survives.

👀

Preview before DELETE

Run the SELECT with rn first; wrap the delete in a rollback-able transaction.

🔑

Group by the right key

Match GROUP BY / PARTITION BY columns to the true definition of "the same record."

⚠️ OFFSET/FETCH and ORDER BY nondeterminism

Both OFFSET…FETCH and ROW_NUMBER depend entirely on the ORDER BY. If it isn't unique, "the Nth row" and "which duplicate to keep" are not stable. The single most common interview slip is an ORDER BY that ties — always ensure it fully determines the order.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What does "Nth highest salary" mean, and why is it ambiguous? Junior

It usually means the Nth-largest distinct salary — ignoring duplicates. It's ambiguous because ties muddy it: "2nd highest" could mean the 2nd distinct value, the 2nd row when sorted (which might equal the 1st if the top is tied), or a medal-style 2nd place. Clarifying which you mean is the first move, and here the 2nd distinct value is 120,000.

Q02 How do you find the single highest salary? Junior

The simplest way is SELECT MAX(Salary) FROM dbo.Employees. MAX returns the top value directly, and it's unaffected by ties (if two people share the top salary, MAX still returns that one value). The Nth-highest is the harder generalization of this.

Q03 What does GROUP BY … HAVING COUNT(*) > 1 find? Junior

It finds duplicate groups. You group by the columns that define a duplicate, and HAVING COUNT(*) > 1 keeps only the groups that occur more than once. On the staging table, grouping by (Name, DeptId, Salary) reveals Alan and Grace each appear twice.

Q04 What is a duplicate row? Junior

A row that repeats an existing row on the columns you consider meaningful. It might be an exact copy of every column, or a repeat of just a "business key" (like the same email or the same Name+Dept). You decide which columns count — that decision is exactly what you put in GROUP BY or PARTITION BY.

Q05 How do you get the 2nd highest salary with DENSE_RANK? Junior

Rank salaries high-to-low with DENSE_RANK() OVER (ORDER BY Salary DESC) in a CTE, then select the distinct salary where the rank equals 2. Because DENSE_RANK gives tied salaries the same rank with no gaps, rank 2 is the 2nd distinct salary — 120,000 here.

Q06 What does OFFSET … FETCH do? Junior

It skips a number of rows and then returns a limited number — the standard way to paginate. OFFSET 1 ROWS FETCH NEXT 1 ROW ONLY skips the first row and returns the next one. It always requires an ORDER BY, since "skip" and "take" only make sense in a defined order.

Q07 Why do you often need DISTINCT when finding the Nth-highest? Junior

Because "Nth highest" is about distinct values, but many techniques operate on rows. If a value is tied (appears in multiple rows), skipping or counting rows would treat those copies as separate steps and land on the wrong value. Using DISTINCT (or DENSE_RANK, which is value-based) fixes that.

Q08 What role does ROW_NUMBER play in removing duplicates? Junior

It labels each row within a duplicate group with 1, 2, 3, … So the first copy is rn = 1 (the one to keep) and any extras are rn > 1 (the ones to delete). That turns "remove duplicates" into the simple filter "delete where rn > 1."

Q09 Name three ways to find the Nth-highest salary. Mid

(1) A correlated subquery: the salary with exactly N−1 distinct salaries above it, via COUNT(DISTINCT). (2) DENSE_RANK() OVER (ORDER BY Salary DESC) in a CTE, filtered to rank N. (3) OFFSET N-1 ROWS FETCH NEXT 1 ROW ONLY over a DISTINCT, ordered set. All three give 120,000 for the 2nd highest here.

Q10 DENSE_RANK vs ROW_NUMBER vs RANK for the Nth-highest — which and why? Mid

Use DENSE_RANK for the Nth distinct value (ties share a rank, no gaps) — the usual intent. Use ROW_NUMBER if you want the exact Nth row regardless of ties. Use RANK for medal-style semantics where ties consume positions (creating gaps). The function choice literally defines what "Nth" means.

Q11 How do you delete duplicates while keeping one row? Mid

Number the rows within each duplicate group using ROW_NUMBER() OVER (PARTITION BY key ORDER BY keeprule) inside a CTE, then DELETE FROM cte WHERE rn > 1. That keeps the rn = 1 row in each group and removes the rest, deleting straight from the underlying table.

Q12 Why delete via a CTE with ROW_NUMBER rather than some other way? Mid

Because it's precise and safe: ROW_NUMBER unambiguously marks exactly one keeper per group, and you control which one via the ORDER BY. Alternatives (like matching on MIN(id) with a self-join) work but are wordier and easier to get wrong, and older SET ROWCOUNT hacks are obsolete. The CTE pattern is the clear, modern standard.

Q13 What is the "which one to keep" rule and why does it matter? Mid

It's the rule that decides which duplicate survives, expressed as the ORDER BY inside ROW_NUMBER — for example "keep the most recent" (ORDER BY UpdatedAt DESC) or "keep the lowest id." It matters because duplicates often differ in some column; deleting the wrong one loses data. State the rule explicitly.

Q14 Why does OFFSET/FETCH require ORDER BY? Mid

Because "skip N and take M" is only meaningful relative to a defined order — without one, which rows are "first" is undefined, and results could vary between runs. SQL Server enforces this by requiring an ORDER BY with OFFSET…FETCH. For stable results the order should also be unique.

Q15 How does the correlated-subquery approach to Nth-highest work? Mid

For each candidate salary, it counts how many distinct salaries are strictly greater. A salary is the Nth-highest when exactly N−1 distinct salaries sit above it. For the 2nd highest, you keep the salary with exactly 1 distinct salary above it. COUNT(DISTINCT) is what makes it count values rather than rows, so ties are handled.

Q16 How do you find duplicates on a business key rather than the whole row? Mid

Group by just the key columns that define "the same record" — say GROUP BY Email — with HAVING COUNT(*) > 1, ignoring columns that may legitimately differ. For deletion, partition ROW_NUMBER by that same key. The whole-row case is just the special case where the key is every column.

Q17 Why exactly do TOP/DISTINCT tricks break on ties? Give an example. Senior

Because TOP (and a bare OFFSET or ROW_NUMBER) counts rows, not distinct values. If the top salary is tied — say two people earn 145,000 — then SELECT TOP 2 … ORDER BY Salary DESC returns two 145,000 rows, and taking MIN reports 145,000 as the "2nd highest," which is wrong; the real 2nd distinct value is 120,000. The trick only works when the top values happen not to tie. DENSE_RANK avoids it by counting distinct values.

Q18 Compare the trade-offs of the three Nth-highest approaches. Senior

The correlated subquery is the most portable (plain SQL, any version) but scales poorly — it's effectively O(rows²) in the naive model. DENSE_RANK is the clearest and reads like the requirement, handles ties by design, and performs well with a supporting index on the ordering column. OFFSET…FETCH is concise and index-friendly but requires an explicit ORDER BY, a DISTINCT step for value semantics, and SQL Server 2012+. For most work, DENSE_RANK is the best default.

Q19 How do you deduplicate a table with no primary key? Senior

You can still delete through a ROW_NUMBER CTE, because the delete targets the underlying rows even without a key. If duplicate rows are byte-for-byte identical, any survivor is fine, so an arbitrary ORDER BY (SELECT NULL) works. If they differ, add a deterministic ORDER BY; if there's genuinely no column to distinguish, you may need to add a temporary identity column (or use %%physloc%%-style physical locators cautiously) to break ties. The DELETE FROM cte WHERE rn > 1 pattern still applies.

Q20 What makes an ORDER BY deterministic, and what happens if it isn't? Senior

An ORDER BY is deterministic when its columns uniquely order every row — typically by including a unique key as the final tie-breaker. If it isn't, rows that tie on the ordering columns can be arranged differently between executions, so ROW_NUMBER may crown a different "keeper" and OFFSET…FETCH may return a different row each run. In a dedup delete that means an unpredictable survivor — a subtle, data-losing bug.

Q21 How do you keep the most recent duplicate instead of an arbitrary one? Senior

Order the ROW_NUMBER so the newest row becomes rn = 1: PARTITION BY key ORDER BY UpdatedAt DESC, id DESC. The UpdatedAt DESC puts the latest first; the id DESC is a unique tie-breaker for rows with identical timestamps, guaranteeing a single deterministic survivor. Then delete rn > 1 as usual.

Q22 How would you extend Nth-highest to "per group" (e.g., per department)? Senior

Add PARTITION BY to the ranking: DENSE_RANK() OVER (PARTITION BY DeptId ORDER BY Salary DESC) in a CTE, then filter to rank N. That restarts the ranking within each department, giving the Nth-highest salary per department in one pass — a direct extension of the single-group solution and a very common follow-up.

Q23 What indexing helps these queries perform? Senior

For Nth-highest via DENSE_RANK or OFFSET…FETCH, an index ordered on the ranking/sort column (e.g., Salary DESC) lets the engine avoid a sort. For the dedup query, an index on the PARTITION BY key (plus the ORDER BY tie-breaker) supports the windowing efficiently. Check the plan for Sort operators; a matching index typically removes them.

Q24 What edge cases should you consider? Senior

N larger than the number of distinct values (should return no rows, not an error); NULL salaries (NULLs sort and compare specially — decide whether they're excluded); an empty table; all values identical (only one distinct value exists); and, for dedup, groups of size one (untouched) versus large groups. Confirming these behave sensibly — and saying so — signals senior-level care.
