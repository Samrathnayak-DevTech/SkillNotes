Schema & DDL Foundation 🗂️

Before you can store a single fact, you have to decide what shape your data will take: which tables exist, what columns they hold, which values are allowed, and how the tables connect. That blueprint is your schema, and the language you use to build it is DDL — Data Definition Language. This page teaches all of it from zero, then builds one complete, realistic company database you can reuse. No prior coding background assumed.

01

The Big Picture

What actually happens between "I have some information to store" and "there's a real, rule-enforced table living inside SQL Server."

Imagine a giant, well-organized filing cabinet. A database is the whole cabinet. Inside it are tables — think of each table as one labeled drawer holding cards of the same kind (a drawer of employee cards, a drawer of order cards). Every card has the same printed fields (Name, Salary, Hire Date). In database language, each card is a row and each printed field is a column.

But a good filing system is more than empty drawers. Someone decided the drawers, the fields on each card, the rules ("Salary can't be blank," "no two departments share a name"), and how drawers relate ("this order card points to that employee card"). Designing all of that up front is schema design, and you express it with a set of commands called DDL.

🏗️ Analogy

DDL is the architect's blueprint and building crew; the data you add later is the furniture. You don't rearrange furniture with a blueprint — you use a different set of commands (called DML: INSERT, UPDATE, DELETE) for that. DDL builds and reshapes the rooms; DML moves the stuff in and out of them.

🔑 The one idea to hold onto

A schema is the shape and rules of your data. DDL is the language that defines that shape. Get the shape right early, because everything you store and every query you write later depends on it.

02

What DDL Is

The small family of commands that create and change the structure of a database — not the data inside it.

SQL (Structured Query Language) is the language for talking to relational databases. It is split into a few families by job. The one that defines structure is DDL — Data Definition Language. Its verbs are few, and this whole page is really about using them well.

🧱

CREATE

Make a brand-new object: a database, table, view, index, or sequence. This is where most schema work happens.

🔧

ALTER

Change an object that already exists — add a column, widen a data type, add or drop a constraint.

🗑️

DROP

Delete an object and everything in it, permanently. The table and all its rows are gone.

🧹

TRUNCATE

Empty a table of all rows fast, but keep the empty table (its structure) in place.

DDL vs the other SQL families 📚

FamilyFull nameJobExample verbs
DDLData Definition LanguageDefine / change structureCREATE, ALTER, DROP, TRUNCATE
DMLData Manipulation LanguageRead / change the data itselfSELECT, INSERT, UPDATE, DELETE
DCLData Control LanguagePermissions & securityGRANT, REVOKE, DENY
TCLTransaction Control LanguageGroup work into safe unitsBEGIN TRAN, COMMIT, ROLLBACK
⚠️ Watch out

In SQL Server, most DDL commands are auto-committed the moment they run — but they can be wrapped in a transaction and rolled back. Regardless, DROP and TRUNCATE are destructive and easy to run by accident. Always double-check which database and table you're pointed at before running them.

🔑 "Schema" means two things — don't get tripped up

1) Schema (the design) — the overall blueprint of all your tables, columns, and rules. This is the sense in the page title.
2) Schema (the object) — in SQL Server a schema is also a literal namespace, a named folder that groups tables and controls permissions. The default one is called dbo (database owner). That's why tables are written dbo.Employees: it means "the Employees table inside the dbo schema."

sql
-- Create a database, then a named schema (namespace) inside it
CREATE DATABASE CompanyDB;
GO
USE CompanyDB;
GO

-- Group HR-related tables under their own 'hr' schema for tidiness + security
CREATE SCHEMA hr;
GO

-- A table written with its schema prefix. If you omit it, SQL Server assumes dbo.
CREATE TABLE hr.Employees (
    EmpId INT PRIMARY KEY
);
✅ Good habit

Always write the schema prefix (dbo.Employees, not just Employees). It makes your intent explicit, avoids surprises when multiple schemas exist, and lets SQL Server reuse query plans more efficiently.

04

Data Types

Every column must declare what kind of value it holds. Choosing the right type is the single most impactful early decision.

A data type is a promise: "this column will only ever hold whole numbers," or "text up to 100 characters," or "a date." SQL Server uses that promise to reserve the right amount of storage, reject bad values automatically, and run faster. Picking the wrong type is a bug you feel for years.

📅 Date & Time
  • DATE just the day

⚠️ Money mistake #1

Never store money in FLOAT or REAL — those are approximate and will quietly produce rounding errors (a total that ends in ...0000001). Use DECIMAL (exact). Avoid the old MONEY type too; it has fixed 4-decimal precision and rounding quirks. DECIMAL(19,4) is the safe default for currency.

🔑 VARCHAR vs NVARCHAR

VARCHAR stores 1 byte per character and only handles basic Latin text. NVARCHAR stores 2 bytes per character (Unicode) and can hold any language, emoji, and accented names. For real-world names and user input, default to NVARCHAR. The N literally stands for "National" (international) characters.

✅ Right-sizing rule

Pick the smallest type that will always fit the real data. Smaller types mean less storage, more rows per page, and faster reads. But don't be too tight — widening a type later (INT → BIGINT) on a giant table is painful.

05

CREATE TABLE

The core DDL command. Let's read one line by line, so every piece makes sense.

sql
-- Anatomy of a table definition
CREATE TABLE dbo.Employees (
    EmpId     INT            IDENTITY(1,1) PRIMARY KEY,  -- auto-numbered unique ID
    Name      NVARCHAR(100) NOT NULL,               -- text, required
    Salary    DECIMAL(19,4) NOT NULL,               -- exact money, required
    IsActive  BIT            NOT NULL DEFAULT (1),     -- flag, defaults to "yes"
    HireDate  DATE           NOT NULL DEFAULT (GETDATE())  -- fills today if omitted
);

Reading it as plain English: "Create a table called Employees in the dbo schema. It has five columns. EmpId is a whole number that SQL Server fills in automatically and that uniquely identifies each row. Name is required text. Salary is an exact decimal and required. IsActive is a yes/no flag that defaults to yes. HireDate is a date that defaults to today if you don't provide one."

🔑 NULL — the "we don't know" value

NULL is not zero and not an empty string — it means "no value / unknown." Marking a column NOT NULL forces every row to supply a real value. Marking it NULL (the default) allows blanks. Decide deliberately for each column: is missing data ever acceptable here?

06

Keys

Keys give each row an identity and let tables point at one another. They are the skeleton of a relational database.

🔑

Primary Key (PK)

One column (or set of columns) that uniquely identifies each row. It can never be NULL and never repeat. Every table should have exactly one.

🔗

Foreign Key (FK)

A column that points to the primary key of another table, creating a link — e.g., an order pointing to the employee who handled it.

🧩

Composite Key

A primary key made of two or more columns together, used when no single column is unique on its own (e.g., StudentId + CourseId).

🌱

Natural vs Surrogate

A natural key is real data (a national ID). A surrogate key is an invented number (an auto ID). Surrogates are usually the safer default.

sql
-- The "one" side: each department has a unique id
CREATE TABLE dbo.Departments (
    DeptId INT           IDENTITY(1,1) PRIMARY KEY,
    Name   NVARCHAR(100) NOT NULL
);

-- The "many" side: many employees point to one department via a FOREIGN KEY
CREATE TABLE dbo.Employees (
    EmpId  INT           IDENTITY(1,1) PRIMARY KEY,
    Name   NVARCHAR(100) NOT NULL,
    DeptId INT           NOT NULL,
    CONSTRAINT FK_Employees_Dept
        FOREIGN KEY (DeptId) REFERENCES dbo.Departments (DeptId)
);
🎟️ Analogy

A primary key is like the unique ticket number on a coat-check tag — no two are the same. A foreign key is the matching stub you keep: it doesn't hold the coat, it just points to the exact tag that does. Follow the stub and you always find the right coat.

🔑 PK quietly builds an index

When you declare a PRIMARY KEY, SQL Server by default also creates a clustered index on it — meaning the rows are physically stored in primary-key order. That's why lookups by primary key are extremely fast. (You'll go deep on indexes on a later day; just note the connection.)

07

Constraints

Rules baked into the table itself so bad data can never get in — no matter which app or person tries.

A constraint is a guarantee the database enforces automatically. Because the rule lives in the table (not in one application), it protects your data from every direction: your app, a colleague's script, a manual edit. This is the database earning its keep.

ConstraintWhat it enforcesEveryday example
NOT NULLA value must be presentEvery employee must have a name
UNIQUENo duplicates in this columnNo two departments share a name
CHECKValue must satisfy a ruleSalary must be greater than 0
DEFAULTAuto-fill when no value is givenHireDate defaults to today
PRIMARY KEYUnique + not null (row identity)Each employee's EmpId
FOREIGN KEYValue must exist in another tableDeptId must be a real department
sql
CREATE TABLE dbo.Employees (
    EmpId   INT           IDENTITY(1,1) PRIMARY KEY,
    Name    NVARCHAR(100) NOT NULL,                 -- required
    Email   NVARCHAR(255) NULL,
    Salary  DECIMAL(19,4) NOT NULL,
    Status  NVARCHAR(20)  NOT NULL DEFAULT ('Active'),

    -- Named constraints are easier to find and drop later
    CONSTRAINT UQ_Employees_Email UNIQUE (Email),
    CONSTRAINT CK_Employees_Salary CHECK (Salary > 0),
    CONSTRAINT CK_Employees_Status
        CHECK (Status IN ('Active', 'OnLeave', 'Terminated'))
);
✅ Always name your constraints

If you don't name a constraint, SQL Server auto-generates a cryptic name like CK__Employee__Salar__3B75D760. Naming them yourself (CK_Employees_Salary) makes error messages readable and lets you drop or change them later without guesswork.

⚠️ UNIQUE allows one NULL

A UNIQUE constraint permits a single NULL value (since NULL means "unknown," SQL Server treats it as not-a-duplicate of another NULL — with standard settings, only one NULL is allowed). A PRIMARY KEY forbids NULL entirely. So a PK is really "UNIQUE + NOT NULL."

08

IDENTITY & SEQUENCE

Two ways to let the database generate unique numbers for you, so you never have to invent IDs by hand.

You rarely want to type ID numbers yourself — it's error-prone and hard to keep unique. SQL Server can generate them automatically in two ways.

🔢

IDENTITY — column-bound

A property on a single column. IDENTITY(1,1) means "start at 1, add 1 each new row." Simple and by far the most common way to make a surrogate primary key.

🎰

SEQUENCE — standalone

A separate, reusable number generator (added in SQL Server 2012). One sequence can hand out numbers to several tables, and you can grab a number before inserting a row.

sql
-- IDENTITY: the number is generated when you insert the row
CREATE TABLE dbo.Invoice (
    InvoiceId INT IDENTITY(1,1) PRIMARY KEY,
    Total     DECIMAL(19,4) NOT NULL
);

-- SEQUENCE: a shareable generator you can call whenever you like
CREATE SEQUENCE dbo.OrderNumberSeq
    AS INT
    START WITH 1000
    INCREMENT BY 1;

-- Ask the sequence for the next number, then use it
SELECT NEXT VALUE FOR dbo.OrderNumberSeq;   -- returns 1000, then 1001, ...
🔑 When to use which

Reach for IDENTITY for a normal auto-numbered primary key — it's the default choice. Reach for a SEQUENCE when you need the number before the insert, want one shared counter across multiple tables, or need a custom cycle/range. Both give unique numbers; they differ in flexibility.

⚠️ Gaps are normal

IDENTITY and SEQUENCE values can have gaps (from rolled-back inserts or server restarts). That's fine — their only job is uniqueness, not being a perfect unbroken count. Never rely on "no gaps," and never assume the newest ID is exactly "previous + 1."

09

Normalization

A step-by-step way to organize columns so each fact is stored exactly once — killing duplication and the bugs that come with it.

Normalization means splitting data into well-focused tables so you don't repeat the same fact in many places. Why care? If a department's name is copied onto 500 employee rows and it gets renamed, you'd have to fix 500 rows — miss one and your data now disagrees with itself. Store the name once, and everyone points to it.

The first three normal forms 🧩

1️⃣ First Normal Form (1NF) One value per cell — no lists. Split "Phone: 555-1, 555-2" into separate rows/columns.
2️⃣ Second Normal Form (2NF) Every non-key column depends on the whole key, not just part of a composite key.
3️⃣ Third Normal Form (3NF) No column depends on another non-key column. Move "DepartmentName" out of Employees and into its own Departments table.
📇 Analogy

Normalization is tidying a messy address book where you scribbled each friend's company address next to their name — over and over. Instead, keep one Companies page and just note which company each friend belongs to. Update the company's address once, and every friend's entry is instantly correct.

Before vs after 3NF

🚫

Not normalized

One big Employees table repeats DeptName and DeptLocation on every row. Rename a department → hundreds of edits → easy to get out of sync.

✅

Normalized (3NF)

A Departments table holds each department once; Employees stores just a DeptId that points to it. One edit updates everyone.

✅ How far to go

3NF is the practical sweet spot for most business databases. Higher forms exist, but aim for 3NF first: every fact in one place, tables focused on one thing, links via keys. You can deliberately break the rules later for speed (see the next callout) — but only on purpose.

🔑 Denormalization is a choice, not a mistake

Sometimes you copy data on purpose (e.g., storing a total instead of recomputing it) to make reads faster. That's denormalization. It's valid — but it's a conscious trade of "faster reads" for "more places to keep in sync." Normalize first; denormalize only where measurements prove you need to.

10

Referential Integrity

The guarantee that every link between tables actually points to something real — no "orphan" rows allowed.

A foreign key doesn't just document a relationship — it enforces it. If Orders.EmpId is a foreign key to Employees, SQL Server will refuse to insert an order for an employee that doesn't exist, and refuse to delete an employee who still has orders (unless you tell it what to do). A leftover row pointing to a deleted parent is called an orphan, and referential integrity prevents them.

What happens when a parent is deleted? 🌩️

You control this with cascading referential actions on the foreign key:

ActionMeaning when the parent row is deleted/updated
NO ACTIONDefault. Block the delete if children exist — you must handle them first.
CASCADEAutomatically delete (or update) the matching child rows too.
SET NULLSet the child's foreign-key column to NULL (the link column must allow NULL).
SET DEFAULTSet the child's foreign-key column to its DEFAULT value.
sql
-- If a department is removed, set its employees' DeptId to NULL
-- rather than deleting the employees. (DeptId must allow NULL for this.)
ALTER TABLE dbo.Employees
    ADD CONSTRAINT FK_Employees_Dept
        FOREIGN KEY (DeptId) REFERENCES dbo.Departments (DeptId)
        ON DELETE SET NULL
        ON UPDATE NO ACTION;
⚠️ CASCADE is powerful and dangerous

ON DELETE CASCADE feels convenient, but deleting one parent can silently wipe out thousands of related rows across many tables. Use it only where that chain reaction is truly intended (e.g., deleting an order also deletes its order-lines). For important records, prefer NO ACTION and delete deliberately.

11

ALTER, DROP, TRUNCATE

Changing a table after it exists, and three very different ways of "removing" things — one of the most common interview traps.

ALTER — reshape an existing table 🔧

sql
-- Add a new column (existing rows get NULL or the default)
ALTER TABLE dbo.Employees ADD Email NVARCHAR(255) NULL;

-- Add a rule after the fact
ALTER TABLE dbo.Employees
    ADD CONSTRAINT UQ_Employees_Email UNIQUE (Email);

-- Widen a column's type
ALTER TABLE dbo.Employees ALTER COLUMN Name NVARCHAR(200) NOT NULL;

-- Remove a column
ALTER TABLE dbo.Employees DROP COLUMN Email;

DELETE vs TRUNCATE vs DROP — pick the right eraser 🧽

CommandFamilyRemovesKeeps the table?Can filter rows?
DELETEDMLSome or all rows✅ Yes✅ Yes (WHERE)
TRUNCATEDDLAll rows, fast✅ Yes (empty)❌ No (all or nothing)
DROPDDLThe entire table❌ No — gone❌ No
🔑 The distinction interviewers love

DELETE removes rows one by one and can target specific rows with a WHERE clause. TRUNCATE empties the whole table in one fast operation, resets the IDENTITY counter, and is minimally logged — but it's all-or-nothing and won't run if a foreign key points at the table. DROP deletes the table itself, structure and all.

⚠️ There is no undo button

Outside an open transaction, TRUNCATE and DROP are effectively permanent. Before running either, confirm the database (SELECT DB_NAME();) and consider wrapping the work in BEGIN TRAN so you can ROLLBACK if something looks wrong.

The Sample Schema

Everything so far, assembled into one realistic company database — the exact foundation you'll reuse for joins, indexing, and every later SQL topic.

We'll model three related things: Departments, Employees (each in a department, and possibly reporting to a manager who is also an employee), and Orders (each handled by an employee). Notice how keys, constraints, and defaults from earlier sections all show up together.

How the tables relate 🔗

one department → many employees → many orders
Departments
DeptId (PK)
1 → ∞
Employees
EmpId (PK) · DeptId (FK)
1 → ∞
Orders
OrderId (PK) · EmpId (FK)
sql
-- ============ 0. Fresh database ============
CREATE DATABASE CompanyDB;
GO
USE CompanyDB;
GO

-- ============ 1. Departments (the "one" side) ============
CREATE TABLE dbo.Departments (
    DeptId INT           IDENTITY(1,1) PRIMARY KEY,
    Name   NVARCHAR(100) NOT NULL,
    CONSTRAINT UQ_Departments_Name UNIQUE (Name)   -- no duplicate dept names
);

-- ============ 2. Employees (belongs to a dept, may have a manager) ==========
CREATE TABLE dbo.Employees (
    EmpId     INT           IDENTITY(1,1) PRIMARY KEY,
    Name      NVARCHAR(100) NOT NULL,
    DeptId    INT           NOT NULL,
    ManagerId INT           NULL,                       -- NULL = top of the tree
    Salary    DECIMAL(19,4) NOT NULL,
    HireDate  DATE          NOT NULL DEFAULT (GETDATE()),

    CONSTRAINT CK_Employees_Salary CHECK (Salary > 0),
    CONSTRAINT FK_Employees_Dept
        FOREIGN KEY (DeptId)    REFERENCES dbo.Departments (DeptId),
    CONSTRAINT FK_Employees_Manager        -- self-reference: manager is an employee
        FOREIGN KEY (ManagerId) REFERENCES dbo.Employees (EmpId)
);

-- ============ 3. Orders (each handled by one employee) ============
CREATE TABLE dbo.Orders (
    OrderId   INT          IDENTITY(1,1) PRIMARY KEY,
    EmpId     INT          NOT NULL,
    Amount    DECIMAL(19,4) NOT NULL,
    OrderDate DATETIME2(0) NOT NULL DEFAULT (SYSUTCDATETIME()),

    CONSTRAINT CK_Orders_Amount CHECK (Amount >= 0),
    CONSTRAINT FK_Orders_Employee
        FOREIGN KEY (EmpId) REFERENCES dbo.Employees (EmpId)
);

Seeding a little data 🌱

sql
-- Note the N prefix: it marks the text as Unicode for the NVARCHAR columns
INSERT INTO dbo.Departments (Name) VALUES
    (N'Engineering'), (N'Sales'), (N'Support');

-- Ada has no manager (NULL); Alan reports to Ada (EmpId 1)
INSERT INTO dbo.Employees (Name, DeptId, ManagerId, Salary, HireDate) VALUES
    (N'Ada Lovelace', 1, NULL, 145000, '2019-03-01'),
    (N'Alan Turing',  1, 1,     98000, '2021-06-15'),
    (N'Grace Hopper', 2, NULL, 120000, '2018-01-20');

INSERT INTO dbo.Orders (EmpId, Amount) VALUES
    (2, 499.99), (3, 1250.00), (2, 75.50);
✅ Day-1 success check

Your schema works if these all hold true: inserting a negative salary is rejected (CK_Employees_Salary), a duplicate department name is rejected (UQ_Departments_Name), and an employee with a made-up DeptId is rejected (FK_Employees_Dept). If the database pushes back on bad data, your constraints are doing their job.

13

Best Practices & Common Pitfalls

The habits that separate a schema you'll thank yourself for from one you'll be fighting in six months.

🔑

Every table gets a primary key

A table without a PK has no reliable row identity, and many tools and indexes assume one. Default to an IDENTITY surrogate key.

🛡️

Enforce rules in the database

Put NOT NULL, CHECK, and foreign keys on the table, not only in app code. The database protects data from every source.

🏷️

Name things consistently

Pick a convention (FK_Child_Parent, CK_Table_Rule) and stick to it, so errors and scripts read clearly.

📦

Right-size data types

Smallest correct type wins. DECIMAL for money, NVARCHAR for real text, DATETIME2 over the legacy DATETIME.

🧩

Normalize to 3NF first

Store each fact once. Denormalize later only where real performance measurements demand it.

💥

Handle CASCADE with care

Chain-deletes are convenient but can erase far more than intended. Prefer NO ACTION for important records.

⚠️ Classic beginner traps

Storing money as FLOAT; making everything NULL "just in case"; using VARCHAR for names that need accents or non-Latin scripts; forgetting foreign keys so orphan rows creep in; and leaving out the schema prefix (dbo.). Each one is cheap to fix now and expensive to fix once real data has piled up.

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level so you can pace yourself.

Q01 What is a database schema? Junior

It's the blueprint of a database — which tables exist, what columns each has, the data types of those columns, and the rules connecting them. It describes the shape and rules of the data, not the data itself. In SQL Server, "schema" also has a second meaning: a named container (like dbo) that groups tables and manages permissions.

Q02 What does DDL stand for, and what is it for? Junior

Data Definition Language. It's the set of SQL commands that create and change the structure of a database — mainly CREATE, ALTER, DROP, and TRUNCATE. Think of it as construction commands, versus DML (SELECT/INSERT/UPDATE/DELETE), which works with the data inside the structures.

Q03 What are a table, a row, and a column? Junior

A table stores one kind of thing (all employees, all orders). A column is one attribute every item shares (Name, Salary). A row is one actual item — one employee, with a value in each column. Picture a spreadsheet: the sheet is the table, the headers are columns, and each filled line is a row.

Q04 Why does the data type of a column matter? Junior

The data type tells SQL Server what a column is allowed to hold (numbers, text, dates). That lets the database reserve the right amount of storage, automatically reject wrong values (you can't put "hello" in a number column), sort and compare correctly, and run faster. A wrong type causes bugs and wasted space for years.

Q05 What is a primary key? Junior

A column (or combination of columns) whose value uniquely identifies each row in a table. It can never be empty (NULL) and can never repeat. It's how the database — and other tables — pinpoint exactly one row, like a unique membership number.

Q06 What is a foreign key? Junior

A column that points to the primary key of another table, creating a relationship. For example, Orders.EmpId points to Employees.EmpId, linking each order to the employee who handled it. The database enforces that the value it points to actually exists.

Q07 What does NOT NULL mean, and what is NULL? Junior

NULL means "no value / unknown" — it is not zero and not an empty string. A NOT NULL column requires every row to have a real value, so it can't be left blank. You mark required fields (like a person's name) as NOT NULL.

Q08 What is a DEFAULT constraint? Junior

A value the database fills in automatically when you insert a row without specifying that column. For example, HireDate DEFAULT (GETDATE()) stamps today's date if you don't provide one, and Status DEFAULT ('Active') starts everyone as active. It saves effort and keeps data consistent.

Q09 What's the difference between CHAR, VARCHAR, and NVARCHAR? Junior

CHAR(n) is fixed length — it always uses n characters of space, padding with blanks. VARCHAR(n) is variable length (uses only what it needs) but stores basic Latin text only. NVARCHAR(n) is variable length and Unicode, so it handles any language, accents, and emoji. For real-world names and user input, default to NVARCHAR.

Q10 Primary key vs unique constraint — what's the difference? Mid

Both prevent duplicate values. The differences: a table has only one primary key but can have many unique constraints; a primary key cannot contain NULL, while a unique constraint allows a single NULL; and a primary key is the row's official identity (and creates a clustered index by default). In short, a primary key is essentially "UNIQUE + NOT NULL + row identity."

Q11 What is a composite key, and when do you use one? Mid

A primary key made of two or more columns combined, used when no single column is unique by itself. Classic example: a class-enrollment table where neither StudentId nor CourseId is unique alone, but the pair (a specific student in a specific course) is. The combination becomes the key.

Q12 IDENTITY vs SEQUENCE — how do they differ? Mid

Both auto-generate unique numbers. IDENTITY is a property of one column and produces its number only when a row is inserted. A SEQUENCE is a standalone object (SQL Server 2012+) that can feed numbers to several tables and can be asked for a value before inserting. Use IDENTITY for a normal auto-key; use a SEQUENCE when you need the number early or shared across tables.

Q13 What is a CHECK constraint? Give an example. Mid

A rule that a column's value must satisfy, enforced on every insert and update. For example CHECK (Salary > 0) refuses any non-positive salary, and CHECK (Status IN ('Active','OnLeave','Terminated')) restricts a column to an approved list. It moves validation into the database so no application can bypass it.

Q14 What is the difference between DELETE, TRUNCATE, and DROP? Mid

DELETE (DML) removes rows and can target specific ones with a WHERE clause; the table remains. TRUNCATE (DDL) removes all rows very fast, keeps the empty table, and resets the IDENTITY counter — but it's all-or-nothing and won't run if a foreign key references the table. DROP (DDL) deletes the entire table, structure and data together.

Q15 What is referential integrity, and what is an orphan row? Mid

Referential integrity is the guarantee that every foreign-key value points to a row that actually exists in the referenced table. An orphan is a child row whose parent has been deleted — e.g., an order pointing to an employee who no longer exists. Foreign keys prevent orphans by blocking such deletes or by cascading the change.

Q16 In SQL Server, what does "schema" mean as an object (like dbo)? Mid

Beyond the "overall design" meaning, a schema is a named namespace inside a database that groups objects and controls permissions. Tables are referenced as schema.table (e.g., dbo.Employees). The default schema is dbo (database owner). Grouping tables into schemas like hr or sales keeps large databases organized and lets you grant access per schema.

Q17 Why prefer DECIMAL over FLOAT (or MONEY) for currency? Mid

FLOAT/REAL are approximate — they can't represent many decimal fractions exactly, so sums drift by tiny amounts and produce wrong totals. DECIMAL is exact, storing the precise value you asked for. The legacy MONEY type is exact but has a fixed 4-decimal scale and awkward rounding in division. DECIMAL(19,4) is the reliable default for money.

Q18 What happens to a table's clustered index when you define a primary key? Mid

By default, declaring a PRIMARY KEY also creates a clustered index on those columns, which means the rows are physically stored on disk in primary-key order. A table can have only one clustered index, which is why PK lookups are so fast. You can override this (make the PK non-clustered and cluster on a different column), but the default is a clustered PK.

Q19 Explain normalization (1NF, 2NF, 3NF) in plain terms. Senior

Normalization organizes columns so each fact is stored once. 1NF: one value per cell — no lists crammed into a single field. 2NF: every non-key column depends on the whole primary key, not just part of a composite key. 3NF: no non-key column depends on another non-key column — so a department's name lives in a Departments table, not copied onto every employee. The payoff is no duplication and no data that can disagree with itself.

Q20 When would you deliberately denormalize? Senior

When reads are far more frequent than writes and the joins/aggregations needed to reassemble normalized data are measurably too slow. Examples: storing a precomputed order total, or keeping a copy of a rarely-changing label to avoid a join on a hot query path. The trade-off is that the duplicated value must be kept in sync, so denormalize only where profiling proves the need, and document it.

Q21 What are cascading referential actions, and what are their risks? Senior

They tell a foreign key what to do to child rows when the parent is deleted or its key updated: NO ACTION (block it — the default), CASCADE (delete/update the children too), SET NULL, or SET DEFAULT. The main risk is ON DELETE CASCADE: one delete can silently remove huge numbers of related rows across many tables, and multiple cascade paths can even conflict. Use cascade only where the chain reaction is genuinely intended.

Q22 How do you choose a primary key — natural vs surrogate? Senior

A natural key uses real business data (an ISBN, a national ID); a surrogate key is a meaningless generated number (an IDENTITY id). Surrogates are usually preferred because business values can change, may not be truly unique, and can be wide (slowing indexes and foreign keys). A surrogate key is stable, compact, and never needs to change. You can still enforce the natural value's uniqueness with a separate UNIQUE constraint.

Q23 What are the trade-offs of adding many constraints and indexes at design time? Senior

Constraints and foreign keys guarantee correctness and let the query optimizer make better decisions — big wins. The cost is that every constraint and index must be checked or maintained on writes, so heavy insert/update workloads slow down, and too many indexes waste storage. The balance: always enforce correctness constraints (keys, NOT NULL, CHECK, FK), and add indexes based on real query patterns rather than guessing up front.

Q24 How would you model an employee-to-manager relationship, and what pitfalls exist? Senior

Use a self-referencing foreign key: add a ManagerId column to Employees that points back to Employees.EmpId. A top-level manager has ManagerId = NULL. Pitfalls: ManagerId must allow NULL (or you can never insert the first person); avoid accidental cycles (A manages B who manages A); ON DELETE CASCADE is unsafe on a self-reference (SQL Server disallows it), so handle manager deletions deliberately; and querying the full hierarchy needs a recursive query (a recursive CTE) rather than a simple join.
