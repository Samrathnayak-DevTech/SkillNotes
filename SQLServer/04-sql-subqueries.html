Subqueries & EXISTS vs IN 🔍

A subquery is a query nested inside another query — a small question you answer first, then use in the bigger one. They power "employees earning above their department's average," "customers who have placed an order," and countless reports. This page covers scalar subqueries, the tricky correlated kind that reruns per row, and the classic EXISTS vs IN comparison — including the NOT IN + NULL trap that silently returns nothing. No prior coding background assumed.

🔢 Scalar 📋 IN 🔗 Correlated ✅ EXISTS ⚠️ NOT IN + NULL 🧠 Interview Prep
01

What a Subquery Is

A query tucked inside another query. You answer the small question first, and its result feeds the bigger one.

Say you want "employees who earn more than the company average." That's really two questions: first, what is the average salary? — then, who earns more than that? A subquery lets you write the first question inside parentheses, right where the second question needs its answer: WHERE Salary > (SELECT AVG(Salary) FROM Employees). The inner query is the subquery (or inner query); the outer one is the outer query.

🧠 Analogy

A subquery is a parenthetical thought in a sentence. "Invite everyone who scored above (the class average)." You compute the parenthetical first, then use it to finish the main idea. SQL does exactly this: it resolves the inner query, then hands the result to the outer query.

The landscape of subqueries 🗺️

📦 In FROM

🔑 Two ways to slice subqueries

By where they sit (in SELECT, FROM, or WHERE), and by what shape they return (one value, one column of values, or a whole table of rows). And crucially, by whether they're correlated — depending on the outer query's current row — or self-contained. Those distinctions organize the entire topic.

03

Where They Live

The same subquery idea shows up in three positions, each doing a slightly different job.

🎯

In SELECT

Returns one value to display as a column — e.g., show each employee next to the company average. Must be scalar (exactly one value).

📦

In FROM

Returns a set of rows that the outer query treats as a temporary table (a "derived table"). Needs an alias.

📍

In WHERE / HAVING

Used to filter: compare against a scalar, test membership with IN, or test existence with EXISTS.

🔑 Shape must match the position

A subquery in SELECT or compared with = must return one value. A subquery after IN returns one column of values. A subquery in FROM returns a full table. Use the wrong shape and SQL Server errors — for example, a scalar comparison that gets back many rows throws "Subquery returned more than 1 value."

04

Scalar Subqueries

A subquery that returns exactly one value — usable anywhere a single value is expected.

sql
-- Show each employee's salary next to the company average (one value)
SELECT Name, Salary,
       (SELECT AVG(Salary) FROM dbo.Employees) AS CompanyAvg
FROM dbo.Employees;

-- Or use it to filter: who earns above the company average?
SELECT Name, Salary
FROM dbo.Employees
WHERE Salary > (SELECT AVG(Salary) FROM dbo.Employees);

Result of the filter

NameSalary
Ada145,000

The inner query computes the company average (121,000) once. Only Ada's 145,000 clears it; Alan (98,000) and Grace (120,000) don't.

⚠️ Exactly one value

A scalar subquery must return a single value. If it returns several rows (say you forgot a filter), SQL Server errors with "Subquery returned more than 1 value." If it returns no rows, the result is NULL, which can quietly make comparisons false.

05

Subqueries with IN

When the subquery returns a list of values, IN checks whether a column's value is somewhere in that list.

sql
-- Employees who have handled at least one order
-- The subquery returns the list of EmpIds that appear in Orders: (2, 3)
SELECT Name
FROM dbo.Employees
WHERE EmpId IN (SELECT EmpId FROM dbo.Orders);

Result

Name
Alan
Grace

Orders were placed by employees 2 (Alan, twice) and 3 (Grace). The subquery yields the list (2, 3); each employee whose EmpId is in that list is kept. Ada (1) isn't, so she's excluded.

🔑 IN is "is it one of these?"

IN (list) is shorthand for a chain of "equals" checks joined by OR: EmpId = 2 OR EmpId = 3. The list can be written out by hand — IN (2, 3) — or produced by a subquery, as here. Either way, it tests membership.

06

Correlated Subqueries

A subquery that refers back to the outer query's current row. It can't run on its own — it re-runs for each outer row, with that row's values plugged in.

The classic example: "employees earning above their own department's average." The inner query must know which department — and that changes with each outer employee. So the inner query references the outer row (e.DeptId). That reference is the correlation.

sql
SELECT e.Name, e.Salary
FROM dbo.Employees AS e
WHERE e.Salary > (
    SELECT AVG(inner_e.Salary)
    FROM dbo.Employees AS inner_e
    WHERE inner_e.DeptId = e.DeptId   -- correlation: e is the OUTER row
);

Result

Ada145,000

🔑 How to spot a correlated subquery

If the inner query mentions a table (or alias) from the outer query, it's correlated — it cannot be run by itself. A non-correlated subquery is self-contained: you could copy it out, run it alone, and get an answer.

07

Correlated vs Not

The difference isn't just academic — it changes how (and how often) the subquery runs.

Non-correlatedCorrelated
References outer row?NoYes
Can run standalone?YesNo
Runs how often (logically)?OnceOnce per outer row
Typical use"vs the company average""vs their own group's average"

⚠️ "Runs per row" is the logical model

Conceptually a correlated subquery executes once for every outer row, which sounds slow. In practice the query optimizer often rewrites it into something far more efficient (like a join or a single grouped pass). Reason about correctness with the "per row" model, but don't assume it's automatically slow — measure.

08

EXISTS & NOT EXISTS

EXISTS asks a yes/no question: "does the subquery find at least one matching row?" It's the cleanest way to test for presence or absence.

sql
-- Employees who have handled at least one order (presence)
SELECT e.Name
FROM dbo.Employees AS e
WHERE EXISTS (
    SELECT 1 FROM dbo.Orders AS o WHERE o.EmpId = e.EmpId
);

-- Employees who have NEVER handled an order (absence / anti-join)
SELECT e.Name
FROM dbo.Employees AS e
WHERE NOT EXISTS (
    SELECT 1 FROM dbo.Orders AS o WHERE o.EmpId = e.EmpId
);

Results

EXISTS → has ordersNOT EXISTS → no orders
Alan, GraceAda
🔑 SELECT 1 vs SELECT *

Inside EXISTS, what you select doesn't matter — the engine only checks whether any row comes back, not the values. SELECT 1, SELECT *, and SELECT NULL behave identically. SELECT 1 is a common convention that signals "I only care about existence." EXISTS also short-circuits: it stops at the first matching row.

09

EXISTS vs IN

Both test membership/existence, and often return the same rows — but they handle NULLs very differently, and that difference is the headline of this whole topic.

For the positive case (IN vs EXISTS), results usually match. The danger is in the negative case: NOT IN and NOT EXISTS can disagree the moment a NULL appears in the list.

sql
-- Suppose Orders.EmpId is nullable and one order has EmpId = NULL.
-- The subquery then returns the list (2, 3, NULL).

-- ❌ NOT IN breaks: "x NOT IN (2, 3, NULL)" is UNKNOWN for EVERY x,
-- so the whole query returns ZERO rows -- silently wrong.
SELECT Name FROM dbo.Employees
WHERE EmpId NOT IN (SELECT EmpId FROM dbo.Orders);

-- ✅ NOT EXISTS is NULL-safe and returns the correct answer (Ada)
SELECT e.Name FROM dbo.Employees AS e
WHERE NOT EXISTS (
    SELECT 1 FROM dbo.Orders AS o WHERE o.EmpId = e.EmpId
);
⚠️ The NOT IN + NULL trap

Why does NOT IN break? "Is x NOT equal to 2 AND NOT equal to 3 AND NOT equal to NULL?" The comparison to NULL is UNKNOWN, and "true AND UNKNOWN" is UNKNOWN — never true — so no row passes. A single stray NULL turns a correct-looking query into one that silently returns nothing. This is one of the most infamous SQL bugs, and a favorite senior interview question.

IN / NOT INEXISTS / NOT EXISTS

Correlated?Usually notUsually yes

✅ Rule of thumb

For "does a related row exist?" prefer EXISTS; for "does not exist?" strongly prefer NOT EXISTS over NOT IN (it's NULL-safe). Reserve IN for short, literal, guaranteed-non-NULL lists like Status IN ('Active','OnLeave').

10

ANY, ALL, SOME

Comparison operators that pair a value against every value a subquery returns. Less common, but they show up in interviews.

sql
-- Employees paid more than EVERY Sales-department salary
-- Sales salaries = (120,000); "> ALL" means "> the maximum" = > 120,000
SELECT Name, Salary
FROM dbo.Employees
WHERE Salary > ALL (SELECT Salary FROM dbo.Employees WHERE DeptId = 2);

Result: only Ada (145,000 > 120,000). Alan (98,000) and Grace (120,000) don't clear the bar.

🔑 The mental shortcuts

> ALL (list) means "bigger than the maximum." > ANY (list) (same as > SOME) means "bigger than the minimum" — i.e., bigger than at least one. In fact = ANY is exactly IN, and <> ALL is exactly NOT IN. Most people use IN/EXISTS and aggregates instead, but knowing the equivalences is handy.

11

Derived Tables & CTEs

A subquery in the FROM clause becomes a temporary table you can query further. A CTE is the same idea, written more readably.

sql
-- Derived table: summarize first, then filter the summary
SELECT DeptId, MaxSalary
FROM (
    SELECT DeptId, MAX(Salary) AS MaxSalary
    FROM dbo.Employees
    GROUP BY DeptId
) AS dept_max            -- a derived table MUST have an alias
WHERE MaxSalary > 100000;

-- Same logic as a CTE (Common Table Expression) -- often easier to read
WITH dept_max AS (
    SELECT DeptId, MAX(Salary) AS MaxSalary
    FROM dbo.Employees
    GROUP BY DeptId
)
SELECT DeptId, MaxSalary FROM dept_max WHERE MaxSalary > 100000;
🔑 Derived table vs CTE

They do the same job: give a subquery a name so you can select from it. A derived table lives inline in FROM and needs an alias. A CTE (written with WITH at the top) is named up front, reads top-to-bottom, can be referenced multiple times, and supports recursion. For anything beyond a trivial nesting, CTEs are usually clearer — they're a full topic later in the course.

12

Subquery vs JOIN

Often you can solve the same problem either way. Here's how to choose — and a quick decision guide.

🧩 Need columns from both tables in the output? Use a JOIN — subqueries can't easily return the other table's columns.
✅ Only checking whether a related row exists? Use EXISTS — no duplicate rows, short-circuits, reads clearly.
🚫 Checking that a related row does NOT exist? Use NOT EXISTS (NULL-safe) — avoid NOT IN with a subquery.
📋 Filtering against a short, fixed, non-NULL list? Use IN (…) — simple and readable.
📊 Need a per-row summary alongside the detail? Consider a window function or a derived table instead of a correlated subquery.
⚠️ JOIN can duplicate rows; EXISTS can't

If you use an INNER JOIN just to check existence and the other table has multiple matches, each outer row is duplicated once per match — so you'd need DISTINCT to fix it. EXISTS sidesteps this entirely: it returns each outer row at most once, because it only asks "is there any match?"

13

Mistakes & Best Practices

The handful of habits that keep subqueries correct and readable.

🚫

Avoid NOT IN with subqueries

Use NOT EXISTS instead — it's immune to the NULL trap that makes NOT IN return nothing.

🎯

Keep scalar subqueries scalar

Make sure a subquery used with = or in SELECT returns exactly one value, or it errors.

🏷️

Alias derived tables

A subquery in FROM must be given an alias, and qualify columns to avoid ambiguity.

📖

Prefer CTEs when nesting deepens

Two or three levels of nested subqueries become unreadable; a named CTE reads top-to-bottom.

✅

Use EXISTS for existence

It short-circuits, never duplicates rows, and ignores the columns you select inside it.

📊

Don't fear correlation — but measure

Correlated subqueries are fine; if one is slow on big data, try a join or window-function rewrite and compare.

✅ Readability beats cleverness

A subquery that's correct but hard to read is a maintenance risk. If you find yourself nesting three levels deep, stop and rewrite with CTEs or joins. The clearest version that produces the right answer is almost always the best one.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is a subquery? Junior

A subquery is a query written inside another query, usually in parentheses. You answer the small inner question first, and its result is used by the outer query — for example, computing the average salary in an inner query and then filtering employees who earn above it in the outer query.

Q02 Where can a subquery appear? Junior

Most commonly in three places: in the SELECT list (to produce a value as a column), in the FROM clause (as a temporary "derived table"), and in the WHERE or HAVING clause (to filter, via IN, EXISTS, or a comparison).

Q03 What is a scalar subquery? Junior

A subquery that returns exactly one value — one row, one column. Because it boils down to a single value, you can use it anywhere a value is expected, like in a SELECT column or a comparison such as WHERE Salary > (SELECT AVG(Salary) FROM Employees).

Q04 What does IN do with a subquery? Junior

IN checks whether a value matches any value in a list. When the list comes from a subquery, it keeps rows whose column value appears in the subquery's results — e.g., "employees whose EmpId is in the list of EmpIds found in the Orders table" gives employees who have placed at least one order.

Q05 What does EXISTS check? Junior

EXISTS answers a yes/no question: does the subquery return at least one row? It doesn't care about the values, only about presence. It's the natural way to ask "does this customer have any orders?" — and NOT EXISTS asks the opposite, "does this customer have none?"

Q06 What is a derived table? Junior

A subquery placed in the FROM clause and treated like a temporary table by the outer query. You give it an alias and then select from it as if it were a real table. It's handy for "summarize first, then filter or join the summary."

Q07 What happens if a scalar subquery returns more than one row? Junior

SQL Server raises an error: "Subquery returned more than 1 value." A scalar context (like = or a SELECT column) expects a single value, so returning several is invalid. You either add a filter so it returns one row, or switch to IN/EXISTS if a set is what you actually mean.

Q08 What's the difference between IN and =? Junior

= compares to exactly one value; IN compares to a list of values and is true if it matches any of them. x = 5 checks one thing; x IN (5, 7, 9) checks membership in a set. = ANY(list) is actually the same as IN(list).

Q09 What is a correlated subquery? Mid

A subquery that references a column from the outer query, so it can't run on its own — it depends on the current outer row. Conceptually it runs once per outer row, plugging that row's values in. The classic example is "employees earning above their own department's average," where the inner query filters by the outer employee's department.

Q10 How do correlated and non-correlated subqueries differ? Mid

A non-correlated subquery is self-contained — you could copy it out and run it alone; it's evaluated once. A correlated subquery references the outer query's row, can't run standalone, and logically re-evaluates for each outer row. The tell is whether the inner query mentions an outer table/alias.

Q11 What are the differences between EXISTS and IN? Mid

IN compares a value against a list of values; EXISTS checks whether a (usually correlated) subquery returns any row. For positive checks they often match. The big difference is NULLs: NOT EXISTS is NULL-safe, while NOT IN returns zero rows if the list contains a NULL. EXISTS also ignores the columns you select inside it and never duplicates outer rows.

Q12 How do you find rows with no matching related row? Mid

Use NOT EXISTS with a correlated subquery: WHERE NOT EXISTS (SELECT 1 FROM Orders o WHERE o.EmpId = e.EmpId) finds employees with no orders. It's the NULL-safe anti-join. The LEFT-JOIN-then-IS NULL pattern works too, but NOT EXISTS is usually clearest and avoids duplicate-row issues.

Q13 Subquery vs JOIN — when do you use each? Mid

Use a JOIN when you need to return columns from both tables. Use a subquery (especially EXISTS) when you only need to test something — existence, membership, or a computed value. A join used purely to check existence can duplicate rows if there are multiple matches (needing DISTINCT), which EXISTS avoids.

Q14 What do ANY and ALL do? Mid

They compare a value against every value a subquery returns. > ALL is true if the value beats every one (i.e., it's greater than the maximum). > ANY (same as > SOME) is true if it beats at least one (greater than the minimum). Useful equivalences: = ANY equals IN, and <> ALL equals NOT IN.

Q15 Why might NOT IN return no rows unexpectedly? Mid

If the subquery's list contains a NULL. NOT IN (2, 3, NULL) is evaluated as "not equal to 2 AND not equal to 3 AND not equal to NULL," and the comparison to NULL is UNKNOWN, which makes the whole condition never true — so every row is filtered out. Switching to NOT EXISTS fixes it because it handles NULLs correctly.

Q16 Does it matter what you SELECT inside EXISTS? Mid

No. EXISTS only checks whether any row comes back, not the values, so SELECT 1, SELECT *, and SELECT NULL are equivalent. Writing SELECT 1 is a convention that signals "I only care about existence." The engine also short-circuits at the first matching row.

Q17 Explain the NOT IN + NULL trap in detail. Senior

x NOT IN (a, b, c) is logically x <> a AND x <> b AND x <> c. If any list value is NULL, that comparison yields UNKNOWN (three-valued logic), and "true AND UNKNOWN" is UNKNOWN — which is not true — so the row is excluded. Since this happens for every row, the query returns nothing. It's silent and easy to miss. The fixes: use NOT EXISTS, or filter NULLs out of the subquery, or add WHERE col IS NOT NULL inside it.

Q18 How does a correlated subquery execute, and what are the performance implications? Senior

Logically it runs once per outer row, substituting that row's values into the inner query. Naively that's O(rows × inner-cost), which can be slow on large tables. In practice SQL Server's optimizer frequently rewrites correlated subqueries into joins, semi-joins, or a single grouped pass, so the physical plan is often much cheaper. The guidance: write it clearly, then check the execution plan; if it's slow, consider a join or window-function rewrite and compare.

Q19 Is EXISTS always faster than IN (or vice versa)? Senior

No — there's no universal winner. On modern SQL Server the optimizer often produces the same or similar plans for equivalent EXISTS and IN queries (both become semi-joins). Differences show up with NULLs (where NOT EXISTS is safer), data distribution, and indexing. The real reasons to prefer EXISTS are correctness (NULL handling) and clarity, not a blanket speed claim. Always measure on your data rather than assuming.

Q20 When would you rewrite a correlated subquery as a JOIN or window function? Senior

When you need the related data as columns, a JOIN is clearer than a correlated scalar subquery repeated per column. When you need a per-group aggregate alongside each detail row (like "each salary next to its department average"), a window function (AVG(Salary) OVER (PARTITION BY DeptId)) computes it in one pass and is usually both cleaner and faster than a correlated subquery that recomputes the average for every row.

Q21 Express semi-join and anti-join using EXISTS. Senior

A semi-join returns outer rows that have at least one match, without duplicating them — that's WHERE EXISTS (…). An anti-join returns outer rows with no match — that's WHERE NOT EXISTS (…). Both are "existence" operations rather than "combine-columns" operations, which is exactly why EXISTS fits them better than a plain join (which would multiply rows on multiple matches).

Q22 Can subqueries be nested arbitrarily, and what are the readability trade-offs? Senior

Yes, subqueries can nest many levels deep, but readability collapses quickly — deeply nested parentheses are hard to follow and maintain. The practical fix is CTEs (WITH), which name each step so the query reads top-to-bottom like a pipeline, can be reused, and support recursion. Beyond one or two levels of nesting, refactoring to CTEs (or joins) is almost always the better choice.

Q23 What's the performance concern with a scalar subquery in the SELECT list? Senior

If it's correlated, it can be evaluated for every row of the outer query, which is expensive when that per-row query is itself costly. For example, a scalar subquery computing each row's department average recomputes that average repeatedly. A window function or a join to a pre-aggregated derived table typically computes it once and is far cheaper. Again, verify with the execution plan — the optimizer sometimes handles it well on its own.

Q24 How do derived tables and CTEs differ, and when do you use each? Senior

Both name a result set you can query. A derived table is inline in FROM with an alias — fine for a single, local use. A CTE is declared up front with WITH, reads sequentially, can be referenced multiple times in the same query, and uniquely supports recursion (for hierarchies). Neither is inherently faster — they're mostly a readability and reuse choice; pick the CTE when the query has several steps or needs recursion.
