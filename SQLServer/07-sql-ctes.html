CTEs & Recursive CTEs 🪜

A Common Table Expression (CTE) is a named, temporary result set defined with WITH at the top of a query. It turns a tangle of nested subqueries into a readable, top-to-bottom pipeline. And in its recursive form, a CTE can reference itself — the standard way to walk a hierarchy like an org chart or a folder tree, one level at a time. No prior coding background assumed.

01

What a CTE Is

A CTE gives a name to a query result so you can use it, like a temporary table that exists only for the single statement that follows.

You've already met CTEs in passing — the "rank, then filter" pattern used WITH. A Common Table Expression is a query you define up front and give a name; the main query then refers to that name as if it were a table. It lives only for the duration of the one statement, then disappears. Nothing is stored on disk.

📋 Analogy

A CTE is like jotting a labeled note before doing a calculation: "Let DeptAvg = the average salary per department." Once you've named it, you can refer to "DeptAvg" in the rest of your work instead of rewriting the whole calculation each time. When you finish the task, you throw the note away.

🔑 Scope: one statement only

A CTE exists only for the single query immediately after it. You can't define a CTE and then reference it in a separate statement later — unlike a temp table or a view, which persist. That short lifetime is exactly why CTEs are great for organizing one complex query without leaving anything behind.

02

The Sample Schema

We use the same Employees table — and this time lean on its self-referencing ManagerId, which forms a small org tree ideal for recursion.

sql
-- Employees with a self-referencing ManagerId (see the Schema & DDL notes)
-- ManagerId points to another employee's EmpId; NULL = top of the tree.
INSERT INTO dbo.Employees (EmpId, Name, ManagerId) VALUES
    (1, N'Ada',       NULL),   -- CEO (no manager)
    (2, N'Alan',      1),      -- reports to Ada
    (3, N'Grace',     1),      -- reports to Ada
    (4, N'Edsger',    2),      -- reports to Alan
    (5, N'Katherine', 2);      -- reports to Alan
Ada (Level 1 — CEO)
├─ Alan (Level 2)
│  ├─ Edsger (Level 3)
│  └─ Katherine (Level 3)
└─ Grace (Level 2)
✅ Why this matters for recursion

The hierarchy is three levels deep. A single self-join only reaches one level (employee → their manager). To walk the whole chain from Ada down to Edsger, you need a query that repeats itself — a recursive CTE — which is the second half of this page.

03

Basic Syntax

A CTE is written with WITH name AS ( query ), followed immediately by a statement that uses that name.

sql
-- Define DeptAvg once, then use it to find above-average earners
WITH DeptAvg AS (
    SELECT DeptId, AVG(Salary) AS AvgSalary
    FROM dbo.Employees
    GROUP BY DeptId
)
SELECT e.Name, e.Salary, d.AvgSalary
FROM dbo.Employees AS e
JOIN DeptAvg AS d ON d.DeptId = e.DeptId
WHERE e.Salary > d.AvgSalary;

Read it top to bottom: first compute DeptAvg (the average salary per department), then join each employee to their department's average and keep those who earn more. The CTE reads like a labeled first step.

⚠️ The CTE must be followed immediately by its statement

The SELECT (or INSERT/UPDATE/DELETE) that uses the CTE has to come right after the closing parenthesis — no other statements in between. Also, if the statement before a WITH didn't end in a semicolon, SQL Server may complain; a common habit is to prefix the CTE with ;WITH to be safe.

04

Why Use a CTE

CTEs don't usually make queries faster — they make them readable. That's the whole point, and it's a big one.

sql
-- The same logic as a NESTED SUBQUERY -- harder to read, inside-out
SELECT e.Name, e.Salary
FROM dbo.Employees AS e
JOIN (
    SELECT DeptId, AVG(Salary) AS AvgSalary
    FROM dbo.Employees GROUP BY DeptId
) AS d ON d.DeptId = e.DeptId
WHERE e.Salary > d.AvgSalary;
📖

Reads top-to-bottom

Steps are named and ordered like a recipe, instead of nested inside-out.

♻️

Reuse a step

Reference the same CTE several times in one query without repeating its SQL.

🧩

Break up complexity

Split a monster query into a few small, understandable pieces.

🪜

Enables recursion

The recursive form (self-reference) is only possible with a CTE.

🔑 Readability is a real benefit

Both queries above produce the same result and usually the same execution plan. The CTE version wins on maintainability: a colleague (or future you) can read the named steps and follow the logic. On a team, clarity prevents bugs — which is why CTEs are the default style for anything beyond a trivial query.

05

Multiple CTEs

You can define several CTEs in one WITH, separated by commas — and a later CTE can build on an earlier one.

sql
WITH
DeptAvg AS (                        -- step 1: per-department averages
    SELECT DeptId, AVG(Salary) AS AvgSalary
    FROM dbo.Employees GROUP BY DeptId
),
AbovePeers AS (                     -- step 2: uses step 1
    SELECT e.EmpId, e.Name, e.Salary
    FROM dbo.Employees AS e
    JOIN DeptAvg AS d ON d.DeptId = e.DeptId
    WHERE e.Salary > d.AvgSalary
)
SELECT * FROM AbovePeers ORDER BY Salary DESC;
🔑 One WITH, commas between CTEs

You write WITH once, then list each CTE separated by a comma. Each CTE can reference any CTE defined before it, forming a readable pipeline: step 1 → step 2 → final query. This is how you decompose a genuinely complex report into digestible stages.

⚠️ Common comma slips

Two frequent syntax errors: putting a comma before the first CTE (right after WITH), or forgetting the comma between two CTEs. The pattern is WITH a AS (...), b AS (...) — commas only between definitions.

06

CTE vs Alternatives

CTEs, derived tables, views, and temp tables all hold intermediate results, but they differ in lifetime, reuse, and cost.

OptionLives forReusable?Stored?Best when
CTEOne statementWithin that statementNoReadability; recursion; single complex query
Derived tableOne statementNo (inline, once)NoA quick, one-off inline subquery
ViewPermanent (a saved definition)Across many queriesDefinition onlyReusing the same logic in many places
Temp table (#t)The sessionAcross statementsYes (in tempdb)Big intermediate sets reused several times
Table variable (@t)The batchAcross statementsYes (small)Small sets; simple procedural work
🔑 Quick decision

Reach for a CTE to make one query readable or to do recursion. Use a view when the same logic is reused across many queries. Use a temp table when a large intermediate result is expensive to compute and you'll reference it several times (a CTE would recompute it each time it's referenced). Use a derived table for a simple one-off inline subquery.

⚠️ A CTE is not a stored table

Unlike a temp table, a CTE holds no data of its own — it's a named query. If you reference the same CTE three times, SQL Server may execute its definition three times. When that recomputation is costly, a temp table (computed once, then reused) can be faster.

07

Recursive CTEs: The Idea

A recursive CTE is one that refers to itself. It's the standard tool for data that forms a chain or a tree of unknown depth.

Some data is naturally hierarchical: employees reporting to managers, folders inside folders, categories with sub-categories, parts made of parts. You can't know in advance how many levels deep it goes, so a fixed number of joins won't do. A recursive CTE handles it by starting at the top and repeatedly applying the same step — "find the children of everything I've found so far" — until there are no more children.

🪜 Analogy

Imagine walking down a family tree. You start with one ancestor (the anchor). Then you ask, "who are their children?" — that's the recursive step. Then you ask the same question of those children, and their children, level by level, until you reach people with no children and the tree runs out. A recursive CTE automates exactly that repeated question.

🔑 When to reach for it

Use a recursive CTE whenever you must traverse a variable-depth hierarchy: full management chains, folder trees, bill-of-materials, threaded comments, or any "walk from a node to all its descendants (or ancestors)" problem. For a single known level, a plain self-join is enough.

08

Anchor + Recursive

Every recursive CTE has exactly two parts, glued together by UNION ALL: a starting point, and a step that references the CTE itself.

sql
WITH cte AS (
    -- 1) ANCHOR MEMBER: the starting rows. Runs ONCE.
    SELECT ... FROM SomeTable WHERE <base condition>

    UNION ALL                         -- must be UNION ALL

    -- 2) RECURSIVE MEMBER: references cte. Runs REPEATEDLY.
    SELECT ... FROM SomeTable AS t
    JOIN cte ON t.parent = cte.id
)
SELECT * FROM cte;

How it actually runs 🔁

🌱 1. Run the anchor once Produces the starting rows (e.g., the top boss). These become the current working set.
🔁 2. Run the recursive member on the last batch It joins the table back to the rows just produced, finding the next level down.
➕ 3. Add those new rows & repeat Feed the new rows back into step 2. Keep going, level by level.
🛑 4. Stop when a pass returns no new rows No more children found → recursion ends. The union of every batch is the result.
⚠️ It must be UNION ALL, not UNION

Recursive CTEs require UNION ALL between the anchor and recursive members — UNION (which removes duplicates) isn't allowed here. Also, the recursive member may reference the CTE only once, and can't use aggregates, TOP, ORDER BY, DISTINCT, or outer joins against the CTE.

09

The Org Chart

The canonical example: start at the CEO and walk down the entire management tree, tagging each person with their level.

sql
WITH OrgChart AS (
    -- ANCHOR: the top boss has no manager
    SELECT EmpId, Name, ManagerId, 1 AS Level
    FROM dbo.Employees
    WHERE ManagerId IS NULL

    UNION ALL

    -- RECURSIVE: people who report to someone already in OrgChart
    SELECT e.EmpId, e.Name, e.ManagerId, oc.Level + 1
    FROM dbo.Employees AS e
    JOIN OrgChart AS oc ON e.ManagerId = oc.EmpId
)
SELECT Name, Level FROM OrgChart ORDER BY Level, Name;

Result

NameLevelFound in pass
Ada1anchor
Alan2pass 1
Grace2pass 1
Edsger3pass 2
Katherine3pass 2

The anchor finds Ada (Level 1). Pass 1 finds everyone reporting to Ada — Alan and Grace (Level 2). Pass 2 finds everyone reporting to them — Edsger and Katherine (Level 3). Pass 3 finds nobody (no one reports to Edsger or Katherine), so recursion stops.

✅ Flip it to walk upward

To find a person's chain of managers instead of reports, swap the join direction: anchor on the specific employee, and in the recursive member join e.EmpId = oc.ManagerId. Same structure, opposite direction — from an employee up to the CEO.

10

Level & Path

Because each pass can carry columns forward, you can accumulate a running path string that shows the full chain from the root to each person.

sql
WITH OrgChart AS (
    SELECT EmpId, Name, ManagerId, 1 AS Level,
           CAST(Name AS VARCHAR(500)) AS Path
    FROM dbo.Employees
    WHERE ManagerId IS NULL

    UNION ALL

    SELECT e.EmpId, e.Name, e.ManagerId, oc.Level + 1,
           CAST(oc.Path + ' > ' + e.Name AS VARCHAR(500))
    FROM dbo.Employees AS e
    JOIN OrgChart AS oc ON e.ManagerId = oc.EmpId
)
SELECT Name, Level, Path FROM OrgChart ORDER BY Path;

Result

NameLevelPath
Ada1Ada
Alan2Ada > Alan
Edsger3Ada > Alan > Edsger
Katherine3Ada > Alan > Katherine
Grace2Ada > Grace
⚠️ CAST the path in the anchor

The recursive member appends text to Path, so Path must be wide enough from the start. If you don't CAST the anchor's Path to a large enough VARCHAR, SQL Server sizes the column to the anchor's short value and later passes fail with a truncation error. Cast both members to the same type and length.

11

Termination & Limits

A recursive CTE stops on its own when a pass finds nothing — but bad data can make it loop forever, so SQL Server ships with a safety brake.

Recursion ends naturally when the recursive member returns zero new rows (in the org chart, when we reach people with no reports). But if the data contains a cycle — say A manages B and B manages A — the recursion never runs out of "children" and would loop endlessly.

sql
-- The default recursion limit is 100 levels. Change it with OPTION.
WITH OrgChart AS ( ... anchor + recursive ... )
SELECT * FROM OrgChart
OPTION (MAXRECURSION 50);   -- cap at 50; use 0 for unlimited (careful!)
🔑 MAXRECURSION

By default, SQL Server stops a recursive CTE after 100 levels and raises an error, to prevent runaway queries. OPTION (MAXRECURSION n) sets a different cap (0 through 32767); MAXRECURSION 0 means no limit — only use it when you're certain the data can't cycle, or you risk an infinite loop.

✅ Guard against cycles

To survive cyclic data, carry the accumulated Path and, in the recursive member, only follow a child whose id isn't already in the path (e.g., WHERE oc.Path NOT LIKE '%' + child + '%', or track visited ids). That way a repeated node is never expanded twice and the recursion terminates safely.

12

Generating Series

Recursion isn't only for trees — it can manufacture rows, like a sequence of numbers or a run of dates.

sql
-- Generate the numbers 1 through 10
WITH Numbers AS (
    SELECT 1 AS n              -- anchor: start at 1
    UNION ALL
    SELECT n + 1 FROM Numbers    -- recursive: add 1 each pass
    WHERE n < 10               -- stop condition (no rows once n = 10)
)
SELECT n FROM Numbers;

-- A run of dates works the same way
WITH Days AS (
    SELECT CAST('2024-01-01' AS DATE) AS d
    UNION ALL
    SELECT DATEADD(DAY, 1, d) FROM Days
    WHERE d < '2024-01-07'
)
SELECT d FROM Days;
🔑 Note the built-in alternative

Recursive series generation is a classic trick, but for plain integer ranges SQL Server 2022 and later offer the simpler built-in GENERATE_SERIES(start, stop, step). For very large ranges, a permanent "tally" (numbers) table is often the fastest option. Recursive CTEs remain ideal for modest series and, above all, for genuine hierarchies.

13

Mistakes & Best Practices

The habits that keep CTEs correct — especially the recursive ones.

➕

UNION ALL, not UNION

Recursive CTEs require UNION ALL; UNION is not allowed between the members.

🌱

Give the anchor a real base

The anchor needs a condition (like ManagerId IS NULL) so recursion has a defined starting point.

📏

Size accumulated columns

CAST a growing path/level column to a wide type in the anchor, or later passes truncate.

🛑

Mind MAXRECURSION

Default cap is 100 levels; raise it deliberately and never blindly use 0 on data that might cycle.

🔁

Detect cycles

Track visited nodes (via the path) so cyclic data can't loop forever.

🗃️

Temp table for reuse

If a heavy CTE is referenced many times, materialize it in a temp table so it's computed once.

⚠️ A CTE doesn't cache its result

Referencing the same non-recursive CTE multiple times can cause SQL Server to execute its definition each time — a CTE is a named query, not a saved table. If the query is expensive and reused, that's wasted work; switch to a temp table computed once. Measure with the execution plan if performance matters.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is a CTE? Junior

A Common Table Expression — a named, temporary result set you define with WITH at the top of a query and then use in the statement that follows. It behaves like a temporary table that exists only for that one statement, and it's mainly used to make complex queries readable.

Q02 What keyword introduces a CTE? Junior

WITH. You write WITH name AS ( query ) and then immediately follow it with the SELECT (or INSERT/UPDATE/DELETE) that references name. Because of a parsing quirk, people often write ;WITH to make sure the previous statement is properly terminated.

Q03 Why use a CTE instead of a subquery? Junior

Mainly readability. A CTE gives a name to a step and lets the query read top-to-bottom like a recipe, instead of being nested inside-out. It also lets you reference the same step more than once, break a big query into pieces, and — uniquely — enables recursion. It usually doesn't change performance.

Q04 How long does a CTE exist? Junior

Only for the single statement that immediately follows its definition. Once that statement finishes, the CTE is gone. You can't define a CTE in one statement and use it in a later, separate statement — for that you'd need a temp table or a view.

Q05 Can you have more than one CTE in a query? Junior

Yes. You write WITH once and separate the CTEs with commas: WITH a AS (...), b AS (...). Each CTE can reference any CTE defined before it, so you can build a readable pipeline of steps that culminates in the final query.

Q06 What is a recursive CTE, in simple terms? Junior

A CTE that refers to itself in order to walk through hierarchical data — like an org chart or a folder tree — one level at a time. It starts at a known point and keeps applying the same step ("find the next level") until there's nothing left to find.

Q07 What are the two parts of a recursive CTE? Junior

The anchor member, which produces the starting rows and runs once (for example, the employee with no manager), and the recursive member, which references the CTE itself and runs repeatedly to find each next level. Together they build up the full result.

Q08 What connects the two parts of a recursive CTE? Junior

UNION ALL. It sits between the anchor member and the recursive member and combines their rows. It must be UNION ALL specifically — plain UNION (which removes duplicates) is not permitted in a recursive CTE.

Q09 How does a CTE differ from a derived table? Mid

Both are temporary, single-statement result sets. A derived table is a subquery written inline in the FROM clause; a CTE is defined up front with WITH and given a name. CTEs read more clearly, can be referenced multiple times in the same query, can chain, and can be recursive — things a derived table can't do as cleanly.

Q10 How does a CTE differ from a temp table, and when do you use each? Mid

A CTE is a named query with no stored data and a one-statement lifetime; a temp table (#t) physically stores rows in tempdb for the whole session and can be reused across statements. Use a CTE for readability and recursion within one query. Use a temp table when a large intermediate result is expensive to compute and you'll reference it several times — the temp table computes it once, whereas a CTE may recompute it.

Q11 How does a CTE differ from a view? Mid

A view is a permanently saved query definition you can reuse across many different queries and sessions; a CTE exists only for the single statement it precedes. If the same logic is needed in many places, make it a view. If it's a one-off building block for a single complex query, use a CTE.

Q12 How does a recursive CTE know when to stop? Mid

It stops automatically when a recursive pass returns no new rows — for example, when you reach employees who have no reports. As a safety net, SQL Server also stops after a maximum number of recursion levels (default 100) and raises an error, which protects against runaway or cyclic data.

Q13 What is MAXRECURSION and its default? Mid

MAXRECURSION is a query hint that caps how many recursion levels a CTE may run. The default is 100; exceeding it raises an error. You override it with OPTION (MAXRECURSION n) where n is 0–32767, and 0 means unlimited — use 0 only when you're sure the data can't cycle, or you risk an infinite loop.

Q14 Can a CTE reference itself or another CTE? Mid

A recursive CTE references itself (that's what makes it recursive). Any CTE can also reference other CTEs defined earlier in the same WITH clause. It cannot reference a CTE defined after it, and a non-recursive CTE cannot reference itself.

Q15 Can you INSERT, UPDATE, or DELETE using a CTE? Mid

Yes. A CTE can precede an INSERT, UPDATE, or DELETE, and you can modify data through the CTE when it maps to a single underlying table. A common use is deduplication: number duplicates with ROW_NUMBER in a CTE, then DELETE FROM cte WHERE rn > 1 to remove all but one of each.

Q16 Why must recursive CTEs use UNION ALL rather than UNION? Mid

Recursion is defined to accumulate rows pass by pass with UNION ALL, which simply appends without deduplicating — that's required by the recursive CTE grammar. UNION would try to remove duplicates across the whole growing set, which isn't allowed here. If you need to avoid revisiting nodes (cycles), you handle that with a visited/path check, not with UNION.

Q17 Explain step by step how a recursive CTE executes. Senior

The anchor member runs once, producing the initial set. Then the recursive member runs, joining the source table to the rows produced in the previous pass to find the next level, and those new rows are added to the result. The recursive member runs again on just that latest batch, and so on. When a pass returns zero rows, recursion stops, and the final result is the union of every batch. It's breadth-first, level by level.

Q18 How do you prevent infinite loops from cyclic data? Senior

Two layers. First, the built-in MAXRECURSION cap aborts with an error after a set number of levels (default 100). Second, and better, detect cycles yourself: carry the accumulated path (or a set of visited ids) and, in the recursive member, only follow a child that isn't already on the path — e.g., exclude ids already visited. That makes the recursion terminate cleanly instead of erroring out.

Q19 Are CTEs materialized, and what are the performance implications? Senior

In SQL Server, a non-recursive CTE is generally inlined (expanded into the main query), not materialized into a stored temporary result. So referencing the same CTE multiple times can cause its definition to be evaluated multiple times. When that recomputation is expensive, materializing into a temp table (computed once) is faster. The optimizer treats the CTE like a subquery, so a CTE per se doesn't add or remove performance versus the equivalent subquery.

Q20 Can you reference a CTE multiple times, and is it re-evaluated? Senior

Yes, a non-recursive CTE can be referenced multiple times in the statement, but because it's typically inlined rather than cached, each reference may trigger a separate evaluation of its definition. If the CTE is costly and used several times, that's duplicated work — the standard remedy is to compute it once into a temp table (or an indexed intermediate) and reference that.

Q21 Recursive CTE vs a cursor or loop for hierarchies — trade-offs? Senior

A recursive CTE is set-based and declarative: it expresses "walk the whole tree" in one statement and lets the engine process level by level, which is usually far faster and cleaner than a row-by-row cursor or a procedural WHILE loop. Cursors/loops are slow and verbose but occasionally justified for very deep or complex traversals with per-node side effects. For pure hierarchy retrieval, prefer the recursive CTE.

Q22 How do you track depth and detect cycles in a recursive CTE? Senior

Add a level column in the anchor (1) and increment it in the recursive member (parent.Level + 1) to track depth. Build a path string the same way (append each node), casting it to a wide type in the anchor. To detect cycles, check in the recursive join that the child isn't already in the path (or in a visited set), so a node is never expanded twice. That combination gives you depth, full path, and loop safety.

Q23 How would you generate a number or date series, and what are the alternatives? Senior

A recursive CTE works: anchor at the first value, recursive member adds a step, with a WHERE stop condition. But for integer ranges, SQL Server 2022+ has the built-in GENERATE_SERIES(start, stop, step), which is simpler and efficient. For large or frequently used ranges, a permanent "tally"/numbers table is typically the fastest. Recursion is best for modest series and, primarily, for real hierarchies rather than plain sequences.

Q24 What are the limitations of the recursive member? Senior

The recursive member may reference the CTE only once, must be combined with the anchor via UNION ALL, and cannot use TOP, DISTINCT, GROUP BY or aggregate functions, HAVING, scalar aggregation, subqueries that reference the CTE, or OUTER/right joins against the CTE. Data types of a column must match between the anchor and recursive members (hence casting accumulated columns). These rules exist so the engine can process the recursion one bounded level at a time.
