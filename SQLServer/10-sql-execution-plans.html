Execution Plans & SARGability 🗺️

You write what you want; the query optimizer decides how to get it. The execution plan is that decision made visible — the tree of operators (seeks, scans, joins, sorts) the engine will run. Learning to read it turns "the query is slow" into "this operator is the problem." And the single biggest lever you control is SARGability: writing predicates that let an index seek instead of scan. No prior coding background assumed.

01

What a Plan Is

The step-by-step strategy the optimizer chose to run your query — shown as a tree of operators, each doing one small job.

SQL is declarative: you state the result you want, not the procedure. The query optimizer then compares many possible strategies and picks the one it estimates is cheapest. The execution plan is that chosen strategy, drawn as connected boxes (operators) — read it and you can see exactly how the engine will fetch, join, sort, and return your rows.

🗺️ Analogy

Think of asking a GPS for directions. You say "get me downtown" (the query); the GPS weighs routes by estimated time and picks one (the optimizer). The turn-by-turn route it shows you is the execution plan. If the trip is slow, you look at the route to find the traffic jam — the same way you read a plan to find the expensive operator.

03

Getting a Plan

There are two flavours — estimated (before running) and actual (after) — plus text statistics that reveal the real I/O cost.

sql
-- In SSMS: Ctrl+L = estimated plan, Ctrl+M then run = actual plan.

-- See real logical reads and time (the numbers that matter for tuning)
SET STATISTICS IO, TIME ON;

SELECT Name, Salary
FROM dbo.Employees
WHERE DeptId = 1;

SET STATISTICS IO, TIME OFF;
🔮

Estimated plan

Built from statistics without running the query. Fast to get, but row counts are guesses.

✅

Actual plan

The plan plus real runtime numbers (actual rows, executions). Best for tuning.

📈

STATISTICS IO

Reports logical reads per table — the clearest measure of how much data was touched.

🔑 Logical reads are the tuning yardstick

Wall-clock time varies with server load and caching, but logical reads (pages read from the buffer) are stable and comparable. When you rewrite a query, compare logical reads before and after: fewer reads almost always means a better plan. A scan of a big table shows huge read counts; a good seek shows tiny ones.

Each operator box (in a real tool) shows its type, the object it touches, an estimated cost % of the whole query, and row counts. The connecting arrows are drawn thicker when more rows flow through them — a fat arrow is a visual alarm that a lot of data is moving.

🔑 What to look at first

Scan the plan for (1) the operator with the highest cost %, (2) the fattest arrows (most rows), and (3) any warning icons (yellow triangles — implicit conversions, spills, missing indexes). Those three signals point straight at the bottleneck; you rarely need to read every operator.

05

Data Access

How a plan gets rows from a table — the operators you met in the indexes topic, now seen in the plan itself.

OperatorWhat it meansUsually good or bad?
Index SeekJumped to matching rows via the B-treeGood for selective filters
Clustered Index SeekSeek on the clustered key (returns full row)Good
Index ScanRead all leaf pages of a non-clustered indexOK if covering / low selectivity
Clustered Index Scan / Table ScanRead the whole tableBad for a few-row query on a big table
Key Lookup / RID LookupExtra hop to fetch missing columnsFine for few rows; costly in bulk
⚠️ "Scan" isn't automatically bad

A scan is the right choice when the query genuinely needs most of the table, or the table is tiny, or the filter isn't selective. The red flag is a scan of a large table for a query that returns few rows — that's where a missing index or a non-SARGable predicate is usually to blame. Judge scans by context, not reflex.

06

Join Operators

The optimizer has three physical ways to join tables. Which one appears tells you a lot about the data sizes and available indexes.

🔁

Nested Loops

For each row of the small outer input, seek matches in the (indexed) inner input. Great for small row counts.

🔀

Merge Join

Both inputs already sorted on the join key, zipped together in one pass. Efficient for large, sorted sets.

#️⃣

Hash Match

Builds a hash table from one input, probes with the other. For large, unsorted inputs; uses memory.

JoinBest whenWatch for
Nested LoopsSmall outer input + index on inner join columnBlows up if the "small" side is actually large
Merge JoinBoth inputs sorted on the join key (or an index provides order)May add a costly Sort if inputs aren't ordered
Hash MatchLarge, unsorted inputs; no useful indexMemory-hungry; can spill to tempdb
🔑 The operator is a diagnosis, not a verdict

A Hash Match on a query you expected to be a quick lookup often signals a missing index on the join column (the optimizer couldn't do efficient nested loops). A Nested Loops processing millions of rows suggests a bad row estimate. You usually don't force a join type — you fix the underlying cause (an index, a rewrite, fresh statistics) and let the optimizer choose better.

07

Sorts & Spools

A few other operators are worth spotting because they're common, costly, and often avoidable.

🔃

Sort

Orders rows for ORDER BY, merge joins, or distinct. Expensive; an index in the right order can remove it.

🌀

Spool

Stashes intermediate rows in a worktable (tempdb) to reuse them. Can indicate the engine is working around a gap.

💧

Spill (warning)

A Sort or Hash ran out of memory and spilled to tempdb — a yellow-triangle warning and a real slowdown.

⚠️ Spills are a red flag

When a Sort or Hash Match is granted too little memory (often because of a bad row estimate), it spills to disk, and disk is far slower than memory. A spill warning on the plan is a strong hint that statistics are stale or the estimate is off — fix the estimate and the spill usually disappears.

✅ Sorts you can design away

If a query ends with ORDER BY OrderDate and there's an index already sorted by OrderDate, the plan can read rows in order and skip the Sort operator entirely. Providing the right index order is one of the cheapest ways to remove an expensive sort.

08

Estimated vs Actual

The optimizer guesses row counts from statistics. When those guesses are far from reality, it picks the wrong plan. The gap is one of the most useful things a plan tells you.

Every operator shows an estimated number of rows (from statistics) and, in an actual plan, the actual number. If an operator expected 5 rows but processed 5 million, the optimizer may have chosen nested loops or a tiny memory grant that's now catastrophically wrong.

🔑 Big estimate gaps = statistics problem

A large divergence between estimated and actual rows almost always points to stale or missing statistics, or a predicate the optimizer can't estimate well (like a local variable or a complex expression). The usual fixes: update statistics, add a supporting index, or rewrite the predicate so the engine can estimate it. Fix the estimate and the plan often corrects itself.

09

SARGability: The Idea

A predicate is SARGable — "Search ARGument able" — when it lets SQL Server use an index seek. The single rule: leave the indexed column bare on one side of the comparison.

For a seek to work, the engine must be able to navigate the index by the column's stored values. If you wrap the column in a function or arithmetic, its stored order no longer matches what you're comparing, so the engine has to compute the expression for every row — that's a scan. Keep the column untouched and the seek is possible.

sql
-- ❌ Non-SARGable: the column is wrapped in a function -> SCAN
WHERE YEAR(OrderDate) = 2024

-- ✅ SARGable: the column stands alone, compared to constants -> SEEK
WHERE OrderDate >= '2024-01-01'
  AND OrderDate <  '2025-01-01'
📖 Analogy

Back to the phone book, sorted by last name. "Find everyone whose name is Lovelace" is easy — flip straight to L. But "find everyone whose name has 8 letters" forces you to read every entry and count letters, because the book isn't sorted by length. Wrapping the column in a function is exactly like asking a length-based question of a name-sorted book: the sorting no longer helps.

🔑 SARGability is the tuning lever you control most

You can't always add indexes (they cost writes), but you can almost always write predicates that use the indexes you have. A huge share of "slow query" problems are simply non-SARGable predicates turning a seek into a scan — and the fix is a rewrite, not new hardware.

10

What Breaks It

The recurring patterns that quietly disable index seeks. Learn to spot these on sight.

sql
-- 1) Function on the column
WHERE YEAR(OrderDate) = 2024
WHERE UPPER(Name) = 'ADA'

-- 2) Arithmetic on the column
WHERE Amount * 1.1 > 100
WHERE Salary + 1000 > 50000

-- 3) Leading wildcard in LIKE
WHERE Name LIKE '%ace'      -- can't seek; 'Ada%' is fine

-- 4) Implicit type conversion (column type != compared type)
WHERE VarcharCode = 12345      -- number vs string -> converts the column
PatternWhy it breaks the seek
Function on the columnThe index stores raw values, not YEAR(...) of them; every row must be computed
Arithmetic on the columnSame reason — the stored order doesn't match the computed value
Leading % wildcardAn index is sorted by prefix; a missing prefix gives nothing to seek on
Implicit conversionTo compare, SQL Server converts the column, which wraps it in a function invisibly
OR across different columnsOften can't seek one index for the whole condition; may scan or need UNION
⚠️ Implicit conversion is the sneaky one

When a column's type doesn't match what it's compared to (say a VARCHAR column compared to a number, or VARCHAR vs NVARCHAR), SQL Server silently converts the column to the other type — which is a hidden function wrap that kills the seek. The plan flags it with a yellow-triangle warning. Match the types (and parameter types) to avoid it.

11

Making It Seekable

Most non-SARGable predicates have a clean rewrite: move the work off the column and onto the constant side.

sql
-- Date: replace YEAR(col) with a range on the bare column
-- ❌  WHERE YEAR(OrderDate) = 2024
-- ✅  WHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'

-- Arithmetic: move the math to the constant
-- ❌  WHERE Amount * 1.1 > 100
-- ✅  WHERE Amount > 100 / 1.1

-- LIKE: keep the wildcard trailing, not leading
-- ❌  WHERE Name LIKE '%ace'
-- ✅  WHERE Name LIKE 'Ada%'

-- Types: compare like-with-like (and type your parameters correctly)
-- ✅  WHERE VarcharCode = '12345'

Is this predicate seek-friendly? 🧭

🧮 Is the indexed column wrapped in a function or arithmetic? Yes → rewrite so the column stands alone (range instead of YEAR(); math on the constant).
🔤 Does a LIKE start with a wildcard ('%…')? Yes → no seek possible; use a trailing wildcard, or full-text search for contains-queries.
🔀 Are the column and the value different data types? Yes → fix the implicit conversion by matching types / parameter types.
🎯 Column bare, same type, compared to a constant/parameter? Seekable — the optimizer can use an index seek (if a suitable index exists).
12

Tuning From a Plan

A repeatable workflow: read the plan, find the worst operator, apply the matching fix, then re-check.

  1. Capture the actual plan and turn on STATISTICS IO, TIME to get real numbers.
  2. Find the bottleneck: the highest cost %, the fattest arrows, and any warning icons.
  3. Match the symptom to a fix (see the table below).
  4. Change one thing, re-run, and compare logical reads. Repeat.
What you see in the planLikely causeFix to try
Scan of a big table for few rowsNo usable index or non-SARGable predicateAdd/adjust index; rewrite predicate to be SARGable
Fat Key LookupIndex doesn't cover the queryAdd needed columns via INCLUDE
Unexpected Hash MatchMissing index on the join columnIndex the join key
Costly SortNo index in the required orderProvide an index ordered to match ORDER BY
Estimated ≠ actual rowsStale/missing statisticsUpdate statistics; simplify the predicate
Spill warningUnder-estimated memory grantFix the estimate (stats/predicate)
Implicit conversion warningType mismatch on a predicateMatch column and value/parameter types
✅ Change one thing at a time

Tuning is empirical. Make a single change, re-read the plan, and compare logical reads — don't add three indexes and a rewrite at once, or you won't know what helped (or what hurt your writes). The plan plus STATISTICS IO gives you an honest before/after every time.

13

Mistakes & Best Practices

The habits that make plan-reading and SARGability second nature.

🎯

Keep the column bare

No functions or math on the indexed column; move work to the constant side to preserve the seek.

📅

Use date ranges

Replace YEAR(col)= and CAST(col AS DATE)= with half-open range predicates.

🔀

Match data types

Compare like-with-like and type parameters correctly to avoid implicit-conversion scans.

📈

Trust logical reads

Compare STATISTICS IO before/after, not wall-clock time, which caching distorts.

🔎

Chase the estimate gap

A big estimated-vs-actual difference means stats or predicate problems — fix those first.

🧭

Fix causes, not operators

Don't force a join type; give the optimizer a better index/estimate and let it choose.

⚠️ Don't chase cost % blindly

The plan's cost percentages are the optimizer's estimates, and they can be wrong — exactly when row estimates are off. A "2%" operator can be the real villain if its actual row count dwarfs the estimate. Cross-check cost % against actual rows and STATISTICS IO rather than trusting the percentage alone.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is an execution plan? Junior

It's the strategy the query optimizer chose to run your query, shown as a tree of operators (seeks, scans, joins, sorts). SQL is declarative — you say what you want, and the optimizer decides how to get it. The plan makes that decision visible so you can see exactly how the engine fetches and combines rows.

Q02 Why would you look at an execution plan? Junior

To understand and fix performance. The plan shows which operator is expensive, whether your indexes are being used (seek vs scan), and where lots of rows are flowing. It turns a vague "the query is slow" into a specific "this scan/lookup/sort is the problem," which tells you what to fix.

Q03 What's the difference between an estimated and an actual plan? Junior

An estimated plan is produced from statistics without running the query, so its row counts are predictions. An actual plan is captured after running and includes real numbers (actual rows, executions). The actual plan is more useful for tuning because you can compare what the optimizer expected against what really happened.

Q04 Which direction do you read a plan? Junior

Data flows right to left (and top to bottom). The rightmost operators fetch rows from tables or indexes, and each operator passes its output leftward to the next, ending at the final SELECT on the far left. So you start reading at the right to see where the data comes from.

Q05 What do the arrow thicknesses mean? Junior

The width of the arrows between operators is proportional to the number of rows flowing along them. A thick arrow means many rows are being passed, which is a quick visual cue for where a lot of work is happening — often near the bottleneck.

Q06 What's an index seek versus a scan in a plan? Junior

An index seek means the engine navigated the index directly to the matching rows — fast and efficient for selective filters. A scan means it read all the rows (of the table or index) and checked each. A scan of a big table when you only wanted a few rows usually indicates a problem.

Q07 What does SARGable mean? Junior

SARGable ("Search ARGument able") means a WHERE condition is written so an index can seek on it. The rule of thumb is to leave the indexed column by itself on one side of the comparison. If you do, the engine can jump to matching rows; if you wrap the column in a function, it can't.

Q08 Give an example of a non-SARGable predicate. Junior

WHERE YEAR(OrderDate) = 2024 is non-SARGable, because the column is wrapped in the YEAR function, so the index can't seek and the engine scans. The SARGable version is a date range: WHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'.

Q09 How do you get an actual execution plan? Mid

In SQL Server Management Studio, enable "Include Actual Execution Plan" (Ctrl+M) and run the query; the plan appears as a tab. You can also capture plans via SET STATISTICS XML ON, Extended Events, or Query Store. Pair it with SET STATISTICS IO, TIME ON to see logical reads and timing alongside the plan.

Q10 What does STATISTICS IO tell you? Mid

It reports how many pages each table's operation read — most importantly logical reads (pages read from the buffer pool). It's the clearest, most stable measure of how much data a query touched. When tuning, compare logical reads before and after a change: fewer reads generally means a better plan, independent of server load or caching.

Q11 Name the three physical join operators. Mid

Nested Loops, Merge Join, and Hash Match. Nested loops seek the inner input once per outer row (great when the outer set is small and the inner is indexed); merge join zips two inputs already sorted on the join key; hash match builds a hash table from one input and probes with the other (for large, unsorted inputs).

Q12 When is a hash join used instead of nested loops? Mid

When the inputs are large and unsorted and there's no useful index on the join column, so seeking per row (nested loops) would be too expensive. The optimizer builds a hash table instead. Seeing a hash match where you expected a quick lookup often signals a missing index on the join key or a bad row estimate.

Q13 Why does YEAR(OrderDate) = 2024 prevent a seek, and how do you fix it? Mid

The index stores raw OrderDate values in order, not YEAR(OrderDate), so the engine must compute the function for every row — a scan. Rewrite it as a range on the bare column: OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'. Now the column is untouched and the index can seek the range.

Q14 What is a key lookup in a plan, and how do you remove it? Mid

After a non-clustered index seek, a key lookup is the extra hop back to the clustered index to fetch columns the index didn't contain — one lookup per matching row. It's fine for a few rows but costly in bulk. Remove it by making the index cover the query: add the returned columns via INCLUDE so no lookup is needed.

Q15 What does a big gap between estimated and actual rows indicate? Mid

That the optimizer's cardinality estimate was wrong — usually because statistics are stale or missing, or the predicate is hard to estimate (a local variable, a complex expression, a multi-column correlation). Bad estimates lead to bad choices: the wrong join type, too little memory (spills), or a scan vs seek mistake. Updating statistics or rewriting the predicate often fixes it.

Q16 Does a leading-wildcard LIKE use an index? Mid

No. LIKE '%ace' can't seek, because an index is sorted by prefix and a leading wildcard means there's no known prefix to jump to — so it scans. LIKE 'Ada%' (trailing wildcard) is SARGable and can seek. For genuine "contains" searches, consider full-text search instead of LIKE '%…%'.

Q17 Walk through how you'd read a plan to find the bottleneck. Senior

Capture the actual plan with STATISTICS IO, TIME on. Read right to left, and scan for three signals: the operator with the highest cost %, the thickest arrows (most rows), and any warning icons (implicit conversions, spills, missing-index hints). Cross-check cost % against actual row counts, since costs are estimates. That triangulation points at the expensive operator — then map its symptom to a fix (index, rewrite, stats) and re-measure logical reads.

Q18 Explain each join operator's cost profile and when the optimizer picks it. Senior

Nested loops cost roughly outer-rows × inner-seek-cost — cheap when the outer input is small and the inner is indexed, terrible when the outer is large. Merge join is a single linear pass but requires both inputs sorted on the join key, so it's chosen when order already exists (or a cheap sort/index provides it). Hash match pays to build a hash table but handles large unsorted inputs in roughly linear time, at the cost of memory (and tempdb spills if under-granted). The optimizer picks based on estimated input sizes, sort/index availability, and memory.

Q19 What causes bad cardinality estimates, and how do you address them? Senior

Common causes: stale or missing statistics, correlated predicates the estimator treats as independent, local variables/parameters the optimizer can't sniff, non-SARGable or complex expressions, and table variables (historically poor estimates). Fixes include updating statistics, adding filtered or multi-column statistics, rewriting predicates to be estimable, using temp tables instead of table variables when estimates matter, and in tough cases query hints or the Query Store to force a known-good plan.

Q20 How does implicit conversion silently cause scans? Senior

When a predicate compares a column to a value of a different type, SQL Server applies data-type precedence and often converts the column side to match — effectively wrapping the column in a CONVERT, which is non-SARGable and forces a scan. Classic cases are a VARCHAR column compared to an NVARCHAR parameter, or a numeric-looking string column compared to an integer. The plan shows a yellow-triangle "type conversion may affect cardinality/seek" warning. Fix by matching the column and parameter types.

Q21 What are spills, and why do they matter? Senior

A spill happens when a memory-consuming operator (Sort or Hash Match) is granted less memory than it actually needs and has to write intermediate data to tempdb on disk. Disk is far slower than memory, so spills badly hurt performance and show as a warning on the operator. They usually trace back to an under-estimated row count, so the real fix is correcting the estimate (fresh statistics, a better predicate, or an appropriate index/rewrite).

Q22 How do parameter sniffing and plan reuse affect plans? Senior

SQL Server caches and reuses a compiled plan for a parameterized query, and it optimizes that plan for the parameter values present at first compilation ("sniffing"). If those first values are atypical, the cached plan can be great for them but poor for other values (e.g., a plan tuned for a rare value used later for a common one). Remedies include OPTION (RECOMPILE), OPTIMIZE FOR hints, local-variable patterns, or Query Store plan forcing — balancing recompilation cost against plan quality.

Q23 When is a scan actually the right choice? Senior

When the query needs most of the table, when the table is small enough that a scan is trivial, or when the predicate is low-selectivity so a seek would return a large fraction of rows anyway (each with a key lookup) — past that "tipping point," a scan is genuinely cheaper. A covering index can also make an index scan the efficient choice. The mistake is assuming "seek good, scan bad" universally; judge by rows returned relative to table size.

Q24 How do OR predicates hurt seekability, and what rewrites help? Senior

An OR across different columns often can't be satisfied by a single index seek, so the optimizer may scan or split the work — and if each side is separately indexed, it may do an index-union but can also miss that. Rewrites that help: split the query into a UNION/UNION ALL of two seekable halves (one per condition), ensure each side has a supporting index, or restructure to IN when the OR is on a single column (col = a OR col = b → col IN (a, b)), which is seekable.
