Transactions & Isolation Levels 🔒

A transaction groups several statements into a single all-or-nothing unit — it all commits, or it all rolls back. The ACID properties guarantee that even under crashes and concurrency. But when many transactions run at once, they can step on each other, producing dirty reads, phantoms, and more. Isolation levels are the dial that trades consistency against concurrency to control exactly which of those anomalies can happen. No prior coding background assumed.

01

What a Transaction Is

A group of statements treated as one indivisible unit of work: either every change happens, or none of them do.

Consider moving money between two bank accounts: subtract from one, add to the other. If the first update succeeds but the second fails (a crash, an error), you've destroyed money — the accounts don't balance. A transaction wraps both steps so they succeed together or fail together. Nothing partial ever becomes permanent.

💸 Analogy

A transaction is like a bank transfer receipt that only prints if both the withdrawal and the deposit go through. If the machine jams halfway, the whole transfer is voided and your money reappears in the original account — never stuck in limbo. "All or nothing" is the entire idea.

🔑 Two problems, one topic

Transactions solve two different things. ACID is about a single transaction being reliable (all-or-nothing, survives crashes). Isolation levels are about many transactions running at the same time without corrupting each other's view of the data. Keep the two ideas separate as you read.

02

The Sample Scenario

We'll use the Employees table and a running example: transferring a 5,000 bonus from Ada's salary budget to Alan's.

NameSalary (before)
Ada145,000
Alan120,000

The "transfer" is two updates: subtract 5,000 from Ada, add 5,000 to Alan. Correct end state: Ada 140,000, Alan 125,000 — and the total across the two is unchanged. If only one update lands, the total is wrong. That's exactly what a transaction protects.

✅ Why a transfer is the perfect example

It needs atomicity (both updates or neither) and it's vulnerable to concurrency (two people editing the same salaries at once). So a single scenario illustrates both halves of the topic. Keep Ada = 145,000 and Alan = 120,000 in mind for the examples below.

03

ACID

The four guarantees a transaction provides. Every reliable database transaction upholds all four.

⚛️

Atomicity

All statements succeed or all are undone. No partial transfer — both updates or neither.

✔️

Consistency

The database moves from one valid state to another; all constraints hold before and after.

🚧

Isolation

Concurrent transactions don't corrupt each other; each behaves as if it ran alone (to a degree set by the level).

💾

Durability

Once committed, changes survive crashes and power loss — persisted via the write-ahead log.

🔑 The one that's a dial

Atomicity, Consistency, and Durability are essentially all-or-nothing guarantees. Isolation is the exception: it's adjustable. The isolation level decides how strictly concurrent transactions are kept apart — stricter means safer but slower, looser means faster but riskier. The rest of this page is about that dial.

✅ How durability actually works

SQL Server uses write-ahead logging: changes are written to the transaction log on disk before the commit is acknowledged. So even if the server crashes right after a commit, recovery replays the log to restore every committed change — and rolls back anything that hadn't committed. That's what makes "committed" mean permanent.

04

Transaction Control

You mark a transaction's boundaries explicitly, and pair it with error handling so a failure rolls everything back.

sql
-- The transfer, wrapped so both updates commit together or not at all
BEGIN TRY
    BEGIN TRANSACTION;
        UPDATE dbo.Employees SET Salary = Salary - 5000 WHERE Name = N'Ada';
        UPDATE dbo.Employees SET Salary = Salary + 5000 WHERE Name = N'Alan';
    COMMIT TRANSACTION;        -- make both changes permanent
END TRY
BEGIN CATCH
    IF @@TRANCOUNT > 0
        ROLLBACK TRANSACTION;    -- undo everything on any error
    THROW;                       -- re-raise so the caller knows it failed
END CATCH;
▶️

BEGIN TRANSACTION

Marks the start; changes are now provisional until you commit.

✅

COMMIT

Makes every change in the transaction permanent and durable.

↩️

ROLLBACK

Undoes every change since BEGIN, as if the transaction never ran.

🔢

@@TRANCOUNT

How many transactions are open on the connection; useful to check before rolling back.

⚠️ Always handle errors

Without TRY…CATCH, an error midway can leave a transaction open and holding locks, blocking everyone else. The pattern above — roll back in CATCH if a transaction is open, then THROW to surface the error — is the safe default. Also keep transactions short: don't wait for user input while one is open.

05

Why Isolation Matters

The moment two transactions touch the same data at the same time, one can see the other's half-finished or changing work — unless isolation prevents it.

A database serves many connections at once. If transaction A is midway through the salary transfer while transaction B reads or edits the same rows, B might see money that's about to vanish, or two edits might overwrite each other. Isolation is the set of rules that decides how much of another transaction's in-progress work you're allowed to see.

🔑 The fundamental trade-off

Perfect isolation (each transaction acting as if it's completely alone) is the safest but slowest — it means heavy locking and transactions waiting on each other. Looser isolation lets more run concurrently but exposes anomalies. Higher isolation = more consistency, less concurrency; lower isolation = the reverse. The isolation level is where you pick your point on that spectrum.

06

The Four Phenomena

The concurrency anomalies isolation levels are designed to prevent. Each is a specific way one transaction can be misled by another.

Dirty read — seeing uncommitted data

T1 (writer)
T2 (reader)
UPDATE Ada salary → 999,999 (not committed)
reads Ada = 999,999 😱
ROLLBACK (salary back to 145,000)
…but T2 already acted on a value that never existed

Non-repeatable read — a value changes under you

T1 (reader)
T2 (writer)
reads Ada = 145,000
UPDATE Ada → 150,000, COMMIT
reads Ada again = 150,000 (different!)

Phantom read — new rows appear

T1 (reader)
T2 (writer)
COUNT Dept 1 employees → 2
INSERT new Dept 1 employee, COMMIT
COUNT again → 3 (a phantom appeared)
🔑 Non-repeatable read vs phantom — the distinction

A non-repeatable read is when an existing row you already read changes (or is deleted) on re-read. A phantom is when a new row appears (or vanishes) in a range you re-query, changing the set of rows that match — not a row you'd already seen. One is about a value changing; the other is about the membership of a result set changing.

⚠️ Plus: the lost update

A fourth classic problem: two transactions both read Ada = 145,000, both add 1,000, and both write 146,000. One update silently overwrites the other — the result should be 147,000. This lost update is prevented by higher isolation levels (or optimistic conflict detection under snapshot), and is a very common real-world bug in read-modify-write code.

07

Levels Overview

The four ANSI isolation levels form a ladder — each higher rung prevents one more phenomenon, at the cost of concurrency. Snapshot sits alongside as an optimistic alternative.

READ UNCOMMITTEDPossiblePossiblePossible
READ COMMITTED (default)PreventedPossiblePossible
REPEATABLE READPreventedPreventedPossible
SERIALIZABLEPreventedPreventedPrevented
SNAPSHOTPreventedPreventedPrevented
🔑 This table is the whole topic in one grid

Memorize the staircase: each level down the list prevents one more anomaly. READ UNCOMMITTED prevents nothing; READ COMMITTED stops dirty reads; REPEATABLE READ also stops non-repeatable reads; SERIALIZABLE stops everything including phantoms. SNAPSHOT achieves the same "prevent all three" as serializable but through row-versioning instead of blocking.

Which level should I use? 🧭

✅ Just want committed data, good default behaviour? Use READ COMMITTED — SQL Server's default; blocks dirty reads.
🔁 Must re-read the same rows and get the same values? Use REPEATABLE READ — holds read locks for the transaction.
👻 Need the result set frozen, no new rows either? Use SERIALIZABLE — strictest; range locks; most blocking.
📸 Want consistency without readers blocking writers? Use SNAPSHOT / RCSI — optimistic row-versioning.
08

READ UNCOMMITTED

The loosest level: reads don't wait for anyone and can see other transactions' uncommitted changes. Fast, but allows dirty reads.

sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- The NOLOCK hint does the same thing for one table -- same risks
SELECT Salary FROM dbo.Employees WITH (NOLOCK) WHERE Name = N'Ada';
⚠️ NOLOCK is not "go faster" — it's "maybe wrong"

READ UNCOMMITTED (and its per-table twin WITH (NOLOCK)) permits dirty reads: you can read a value that later gets rolled back, and you can even read rows twice or skip rows during page movement. It avoids read locks, so it doesn't block — but the data may be flat-out wrong. Reserve it for rough, approximate reads where accuracy truly doesn't matter (and even then, prefer snapshot).

09

READ COMMITTED

SQL Server's default. You only ever read data that has been committed — so no dirty reads — but values can still change between two reads in the same transaction.

sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;   -- the default; usually implicit

Under the classic (locking) implementation, a read waits until any in-progress write on that row commits or rolls back, then reads the committed value — and releases the read lock immediately after. That's why a non-repeatable read is still possible: nothing stops another transaction from changing the row before you read it a second time.

🔑 It prevents dirty reads, nothing more

READ COMMITTED guarantees every value you read was committed at the moment you read it. It does not guarantee that value stays the same if you read again, nor that the same query returns the same set of rows later. For most applications that balance of safety and concurrency is exactly right, which is why it's the default.

10

REPEATABLE READ

Guarantees that any row you read once will hold the same value if you read it again in the same transaction — but new rows can still slip into a range.

sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

It works (in the locking model) by holding the read locks on every row it reads until the transaction ends, so no one else can modify those rows meanwhile. That kills non-repeatable reads and lost updates on rows you've read. But because it only locks the rows that exist, another transaction can still INSERT a new row that matches your filter — a phantom.

⚠️ More locks held longer = more blocking

Holding read locks for the whole transaction increases contention: other writers must wait, and the chance of deadlocks rises. REPEATABLE READ buys stronger consistency at a real concurrency cost, so use it only when you genuinely need re-read stability.

11

SERIALIZABLE

The strictest level. Transactions behave as if they ran one after another, in some serial order — no dirty reads, no non-repeatable reads, and no phantoms.

sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

To also stop phantoms, it takes range locks: it locks not just the rows that match your filter but the gaps too, so no one can insert a new matching row while your transaction is open. The result is the strongest guarantee — the outcome is equivalent to running the transactions one at a time.

⚠️ Strongest safety, weakest concurrency

SERIALIZABLE causes the most blocking and the highest deadlock risk, because it locks ranges and holds everything until commit. Use it only when correctness demands full serial equivalence (e.g., certain financial or inventory checks). For many "I need a consistent view" cases, SNAPSHOT gives similar guarantees without the blocking.

12

Snapshot & RCSI

An optimistic approach: instead of locking, SQL Server keeps older versions of rows so readers see a consistent snapshot without blocking writers.

sql
-- Enable row-versioning features at the database level first
ALTER DATABASE MyDb SET ALLOW_SNAPSHOT_ISOLATION ON;      -- enables SNAPSHOT
ALTER DATABASE MyDb SET READ_COMMITTED_SNAPSHOT ON;      -- makes READ COMMITTED use versions (RCSI)

-- Then a transaction can request the SNAPSHOT level
SET TRANSACTION ISOLATION LEVEL SNAPSHOT;

When row-versioning is on, each write keeps the previous version of the row in a version store (in tempdb). Readers get the version that was committed when their statement (RCSI) or transaction (SNAPSHOT) started — so readers never block writers and writers never block readers. It's the same "prevent all three phenomena" strength as serializable, achieved without read locks.

SNAPSHOTRead Committed Snapshot (RCSI)
Consistent view scopeThe whole transactionEach statement
How enabledOpt-in per transaction (SET…SNAPSHOT)Database setting; changes READ COMMITTED behaviour automatically
Update conflictsCan raise a conflict error (must retry)No snapshot-conflict errors
PreventsDirty, non-repeatable, phantomDirty reads (non-blocking); statement-level consistency
🔑 Optimistic vs pessimistic

The locking levels are pessimistic — assume conflict, block to prevent it. Snapshot is optimistic — assume no conflict, let everyone proceed on their own version, and (for full SNAPSHOT) detect a genuine write-write conflict at commit and raise an error to retry. The cost is maintaining the version store in tempdb, but the payoff — readers and writers not blocking each other — is huge for busy systems.

13

Mistakes & Best Practices

The habits that keep transactions correct and concurrency healthy.

⏱️

Keep transactions short

Do the minimum work between BEGIN and COMMIT; never wait on user input while one is open.

🧯

Wrap in TRY…CATCH

Roll back on error and re-THROW, so failures don't leave open transactions holding locks.

🚫

Avoid NOLOCK by reflex

It permits dirty/inconsistent reads; prefer RCSI/snapshot for non-blocking reads that are still correct.

🎚️

Use the lowest safe level

Higher isolation blocks more; pick the least strict level that prevents the anomalies you actually care about.

🔁

Handle update conflicts

Under SNAPSHOT, be ready to catch a conflict error and retry the transaction.

🔒

Guard read-modify-write

Prevent lost updates with a higher level, a single atomic UPDATE, or version/rowversion checks.

⚠️ Long transactions are the root of most concurrency pain

A transaction that stays open too long holds its locks (or grows the version store) the entire time, blocking others and inviting deadlocks. The most impactful habit isn't choosing an exotic isolation level — it's keeping each transaction as small and fast as possible, and committing promptly.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is a transaction? Junior

A group of one or more statements treated as a single, indivisible unit of work: either all of them take effect (commit) or none of them do (roll back). The classic example is a money transfer — subtract from one account and add to another — which must happen entirely or not at all.

Q02 What does ACID stand for? Junior

Atomicity (all-or-nothing), Consistency (the database stays valid, all constraints hold), Isolation (concurrent transactions don't corrupt each other), and Durability (once committed, changes survive crashes). Together they're the guarantees that make transactions reliable.

Q03 What do COMMIT and ROLLBACK do? Junior

COMMIT makes all the changes in the transaction permanent and durable. ROLLBACK undoes every change since BEGIN TRANSACTION, leaving the data as if the transaction never happened. You commit on success and roll back on failure.

Q04 What is atomicity? Junior

The guarantee that a transaction is all-or-nothing: every statement in it succeeds together, or if any fails, all of them are undone. There's no partial result — you can't end up with money withdrawn but never deposited.

Q05 What is durability? Junior

Once a transaction commits, its changes are permanent and survive crashes, restarts, or power loss. SQL Server achieves this with write-ahead logging: it records changes to the transaction log on disk before acknowledging the commit, so recovery can replay them.

Q06 Why do we need isolation levels? Junior

Because many transactions run at the same time and can interfere with each other — one reading another's half-finished work, or two overwriting each other. Isolation levels let you choose how strictly transactions are kept apart, trading consistency against concurrency depending on what your application needs.

Q07 What is a dirty read? Junior

Reading data that another transaction has changed but not yet committed. If that other transaction later rolls back, you acted on a value that never officially existed. Only the loosest level, READ UNCOMMITTED (or NOLOCK), allows dirty reads.

Q08 What's the default isolation level in SQL Server? Junior

READ COMMITTED. It ensures you only read committed data (no dirty reads) while allowing good concurrency. It does still permit non-repeatable reads and phantoms, but for most applications that balance is the right default.

Q09 Explain the concurrency phenomena. Mid

Dirty read: reading uncommitted data that may be rolled back. Non-repeatable read: a row you read changes value when you read it again in the same transaction. Phantom: new rows appear (or vanish) in a range you re-query. Lost update: two read-modify-writes overwrite each other so one change is lost. Isolation levels are defined by which of these they prevent.

Q10 Non-repeatable read vs phantom read — what's the difference? Mid

A non-repeatable read is about an existing row you already read changing (or being deleted) on re-read — a value changes. A phantom is about the set of rows changing: a new row appears (or disappears) in a range you re-query. Repeatable read stops the first; only serializable (or snapshot) stops the second.

Q11 Name the four ANSI isolation levels. Mid

From loosest to strictest: READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, and SERIALIZABLE. Each higher level prevents one more phenomenon. SQL Server also offers SNAPSHOT (and read-committed snapshot) using row-versioning, outside the strict ANSI ladder.

Q12 What does READ COMMITTED prevent and allow? Mid

It prevents dirty reads — you only ever read committed data. It still allows non-repeatable reads and phantoms, because it doesn't hold read locks (or, under RCSI, doesn't freeze a whole-transaction snapshot) after a statement finishes. It's SQL Server's default and the usual sweet spot of safety and concurrency.

Q13 What does REPEATABLE READ do? Mid

It guarantees that any row you read once will have the same value if you read it again within the transaction, by holding read locks on those rows until the transaction ends. That prevents dirty and non-repeatable reads (and lost updates on read rows), but not phantoms — another transaction can still insert new rows that match your filter.

Q14 What does SERIALIZABLE do? Mid

It makes transactions behave as if they ran one after another in some serial order, preventing all three phenomena including phantoms. It does this with range locks that lock the gaps as well as the rows, so no one can insert a matching row while your transaction is open. It's the strictest and least concurrent level.

Q15 What is snapshot isolation? Mid

An optimistic level that uses row-versioning instead of locking: each transaction reads a consistent snapshot of the data as it existed when the transaction started, from versions kept in tempdb. Readers don't block writers and writers don't block readers. It prevents dirty, non-repeatable, and phantom reads, at the cost of maintaining the version store (and possible update-conflict errors).

Q16 What is a lost update and how do you prevent it? Mid

When two transactions both read the same value, both modify it, and both write back — the second write overwrites the first, so one update is lost (e.g., two +1,000 raises produce +1,000 instead of +2,000). Prevent it by doing the change in one atomic statement (SET Salary = Salary + 1000), using a higher isolation level, or checking a rowversion/version column before updating (optimistic concurrency).

Q17 Map the phenomena to the isolation levels. Senior

READ UNCOMMITTED: allows dirty, non-repeatable, and phantom. READ COMMITTED: prevents dirty; allows non-repeatable and phantom. REPEATABLE READ: prevents dirty and non-repeatable; allows phantom. SERIALIZABLE: prevents all three. SNAPSHOT: prevents all three via row-versioning rather than locking. Each ANSI step up the ladder removes exactly one more anomaly.

Q18 Pessimistic vs optimistic concurrency — what's the difference? Senior

Pessimistic concurrency (the locking levels) assumes conflicts will happen and blocks to prevent them — readers and writers wait on each other's locks. Optimistic concurrency (snapshot/row-versioning, or app-level version checks) assumes conflicts are rare, lets everyone proceed against their own consistent version, and detects a genuine conflict only at write/commit time (raising an error to retry). Optimistic favours throughput and avoids blocking; pessimistic favours simplicity and immediate serialization.

Q19 SNAPSHOT vs READ COMMITTED SNAPSHOT (RCSI) — how do they differ? Senior

Both use row-versioning, but the snapshot scope differs. SNAPSHOT gives a consistent view for the whole transaction (opt-in per transaction) and can raise update-conflict errors you must retry. RCSI is a database setting that changes READ COMMITTED to take a fresh consistent snapshot per statement (no whole-transaction freeze), doesn't produce snapshot-conflict errors, and is often turned on to make the default level non-blocking with minimal app changes.

Q20 What are the trade-offs of higher isolation levels? Senior

Higher isolation prevents more anomalies but reduces concurrency: it holds more locks for longer (or for snapshot, grows the version store), which increases blocking, lowers throughput, and raises deadlock risk. Serializable's range locks are the extreme. The engineering goal is to use the lowest level that still prevents the anomalies your workload actually cares about — often READ COMMITTED with RCSI.

Q21 How does row-versioning work under the hood? Senior

When row-versioning is enabled, each time a row is modified SQL Server writes the prior committed version to the version store in tempdb, tagged with a timestamp. A snapshot reader follows a pointer chain to find the version that was committed as of its snapshot point, so it reads consistent data without taking locks. The cost is extra tempdb usage and a 14-byte versioning tag per row; long-running transactions can bloat the version store.

Q22 What is a deadlock, and how does isolation relate to it? Senior

A deadlock is when two transactions each hold a lock the other needs, so neither can proceed; SQL Server detects the cycle and kills one as the "victim." Higher isolation levels hold more locks for longer, which increases deadlock frequency, and lock-order differences between transactions are the usual trigger. Row-versioning (snapshot/RCSI) reduces reader/writer deadlocks by removing read locks. (Locking and deadlocks are the focus of the next topic.)

Q23 How do nested transactions and @@TRANCOUNT behave? Senior

SQL Server doesn't have truly independent nested transactions: BEGIN TRANSACTION increments @@TRANCOUNT, and only the outermost COMMIT (when the count returns to 0) actually commits; inner "commits" just decrement the counter. Critically, a ROLLBACK (without a savepoint name) rolls back everything and resets @@TRANCOUNT to 0. For partial rollback you use SAVE TRANSACTION (a savepoint) and ROLLBACK TRANSACTION savepoint_name.

Q24 What are implicit, explicit, and autocommit transactions? Senior

Autocommit is the default: each individual statement is its own transaction, committed automatically if it succeeds. Explicit transactions are ones you delimit yourself with BEGIN TRANSACTION … COMMIT/ROLLBACK. Implicit transaction mode (SET IMPLICIT_TRANSACTIONS ON) auto-starts a transaction before certain statements but requires you to explicitly commit or roll back — a mode that surprises people by leaving transactions open. Most code uses autocommit plus explicit transactions where atomicity is needed.
