Window Functions I: Ranking 🏅

GROUP BY squashes many rows into one summary. A window function does the opposite: it computes across a set of related rows but keeps every row, adding the result as a new column. The ranking family — ROW_NUMBER, RANK, and DENSE_RANK — numbers rows in order, which powers the two most-asked SQL interview problems: the Nth-highest value and top-N per group. No prior coding background assumed.

🪟 OVER 🔢 ROW_NUMBER 🥇 RANK 🎖️ DENSE_RANK 🗂️ PARTITION BY 🧠 Interview Prep
01

The Big Idea

A window function calculates over a group of rows — a "window" — yet returns one value per row, leaving all your detail rows intact.

On Day 3, GROUP BY collapsed employees into one row per department. That's great for a summary, but you lose the individual employees. A window function lets you keep every employee row and show, say, their rank within the department right beside them. It "looks through a window" at related rows to compute a value, without merging anything.

🏁 Analogy

Picture a race. GROUP BY hands you a single card that says "5 runners, average time 12s." A window function hands back every runner's card, each now stamped with "3rd place." You keep all the individuals and gain the ranking. That per-row stamp is what window functions do.

🔑 The defining trait

Every window function is written with an OVER (...) clause. If you see OVER, you're looking at a window function. It computes using a set of rows but does not reduce the row count — that's the single idea that separates it from GROUP BY.

03

The OVER Clause

Every window function is followed by OVER (...), which describes the "window" of rows to work over. It has two knobs.

🗂️

PARTITION BY

Splits rows into independent groups — the function restarts for each. PARTITION BY DeptId ranks within each department separately. Optional; omit it to treat the whole table as one window.

🔀

ORDER BY

Sets the order used to assign ranks inside the window. ORDER BY Salary DESC means "highest salary gets rank 1." Required for ranking functions.

sql
-- The anatomy: FUNCTION() OVER (PARTITION BY ... ORDER BY ...)
ROW_NUMBER() OVER (PARTITION BY DeptId ORDER BY Salary DESC)
--            ^ split into per-dept windows   ^ rank high salary first
⚠️ ORDER BY inside OVER ≠ ORDER BY of the query

The ORDER BY inside OVER only controls how ranks are assigned — it does not sort your final result. If you want the output rows sorted too, add a separate ORDER BY at the end of the query. They're two different things that happen to share a keyword.

04

ROW_NUMBER

Assigns a unique sequential number to each row in the window — 1, 2, 3, … — with no ties, ever.

sql
SELECT Name, Salary,
       ROW_NUMBER() OVER (ORDER BY Salary DESC) AS RowNum
FROM dbo.Employees;

Result

NameSalaryRowNum
Ada145,0001
Alan120,0002
Edsger120,0003
Grace110,0004
Katherine95,0005
⚠️ Ties get an arbitrary order

Alan and Edsger both earn 120,000, yet ROW_NUMBER still gives them 2 and 3 — it must pick an order, but which one gets 2 is not guaranteed unless you add a tie-breaker to the ORDER BY (e.g., ORDER BY Salary DESC, EmpId). Without one, the same query can return them in different orders on different runs.

🔑 Use ROW_NUMBER when you need exactly one per position

It's the right choice for "give me exactly the single top row per group," pagination (rows 11–20), and deduplication (keep one row per key). Because it never ties, it always produces a clean 1-per-position sequence.

05

RANK

Gives tied rows the same rank, then skips numbers so the next rank reflects how many rows came before — like Olympic medal standings.

sql
SELECT Name, Salary,
       RANK() OVER (ORDER BY Salary DESC) AS Rnk
FROM dbo.Employees;

Result

NameSalaryRnk
Ada145,0001
Alan120,0002
Edsger120,0002
Grace110,0004
Katherine95,0005
🔑 Ties share, then a gap appears

Alan and Edsger both get rank 2. Because two people occupy rank 2, the next rank jumps to 4 (rank 3 is skipped). This mirrors real medal tables: two silver medalists means no bronze — the next competitor is 4th.

06

DENSE_RANK

Like RANK, tied rows share a number — but it does not skip. The next distinct value gets the very next integer.

sql
SELECT Name, Salary,
       DENSE_RANK() OVER (ORDER BY Salary DESC) AS DenseRnk
FROM dbo.Employees;

Result

NameSalaryDenseRnk
Ada145,0001
Alan120,0002
Edsger120,0002
Grace110,0003
Katherine95,0004
🔑 "Dense" = no gaps

Alan and Edsger share rank 2, and Grace is 3 (not 4) — DENSE_RANK counts distinct values, so there are no holes. This is exactly what you want for "the Nth distinct highest salary": the 2nd distinct salary is 120,000, the 3rd is 110,000, and so on.

07

The Tie Test

All three side by side on the same data. This single table is the thing to memorize.

sql
SELECT Name, Salary,
       ROW_NUMBER() OVER (ORDER BY Salary DESC) AS RowNum,
       RANK()       OVER (ORDER BY Salary DESC) AS Rnk,
       DENSE_RANK() OVER (ORDER BY Salary DESC) AS DenseRnk
FROM dbo.Employees;
NameSalaryROW_NUMBERRANKDENSE_RANK
Ada145,000111
Alan120,000222
Edsger120,000322
Grace110,000443
Katherine95,000554

Read the tied pair (Alan, Edsger) and the row after it (Grace): ROW_NUMBER = 2, 3, 4 (unique, no sharing); RANK = 2, 2, 4 (share, then skip a number); DENSE_RANK = 2, 2, 3 (share, no skip).

Which one should I use? 🧭

🔢 Need exactly one row per position (no ties allowed)? Use ROW_NUMBER — top row per group, pagination, dedupe. Add a tie-breaker.
🎖️ Ties should share a rank, and you count distinct values (no gaps)? Use DENSE_RANK — the go-to for "Nth distinct highest."
🥇 Ties should share a rank, with gaps (medal-style standings)? Use RANK — reflects how many rows outrank you.
🪣 Just splitting rows into N equal buckets? Use NTILE(N) — quartiles, percentiles, batches.
08

NTILE

The fourth ranking function: it splits the ordered rows into a given number of roughly equal buckets and labels each row with its bucket number.

sql
-- Split employees into 2 pay bands (top half / bottom half)
SELECT Name, Salary,
       NTILE(2) OVER (ORDER BY Salary DESC) AS PayBand
FROM dbo.Employees;

Result

NameSalaryPayBand
Ada145,0001
Alan120,0001
Edsger120,0001
Grace110,0002
Katherine95,0002
🔑 Uneven splits fill from the top

5 rows into 2 buckets can't divide evenly, so NTILE puts the extra row in the earlier bucket: bucket 1 gets 3 rows, bucket 2 gets 2. Use NTILE(4) for quartiles, NTILE(100) for percentiles, or NTILE(10) for deciles. It groups by position, not by value, so tied values can land in different buckets.

09

PARTITION BY

Add PARTITION BY and the ranking restarts for each group — the key to "per department," "per customer," "per month" rankings.

sql
-- Rank employees by salary WITHIN their own department
SELECT Name, DeptId, Salary,
       ROW_NUMBER() OVER (PARTITION BY DeptId ORDER BY Salary DESC) AS RankInDept
FROM dbo.Employees;

Result

NameDeptIdSalaryRankInDept
Ada1145,0001
Alan1120,0002
Edsger1120,0003
Grace2110,0001
Katherine295,0002

Notice the count resets at the department boundary: Engineering runs 1-2-3, then Sales starts fresh at 1 again. Without PARTITION BY, the numbering would run 1–5 straight through.

🔑 PARTITION BY is the "per group" of window functions

It plays the same role that GROUP BY plays for aggregation — defining the groups — but it doesn't collapse the rows. Each partition is ranked independently, and every original row still comes back.

10

Nth & Top-N per Group

The two signature interview problems fall right out of ranking + a CTE.

Top earner in each department

sql
WITH Ranked AS (
    SELECT Name, DeptId, Salary,
           ROW_NUMBER() OVER (PARTITION BY DeptId ORDER BY Salary DESC) AS rn
    FROM dbo.Employees
)
SELECT Name, DeptId, Salary
FROM Ranked
WHERE rn = 1;          -- rn = 2 for top-2, etc.

Result: Ada (Engineering) and Grace (Sales). Change rn = 1 to rn <= 2 for the top two per department.

The 2nd-highest salary overall

sql
WITH Ranked AS (
    SELECT Salary,
           DENSE_RANK() OVER (ORDER BY Salary DESC) AS dr
    FROM dbo.Employees
)
SELECT DISTINCT Salary
FROM Ranked
WHERE dr = 2;          -- the 2nd DISTINCT highest salary

Result: 120,000 (the second distinct salary). Using DENSE_RANK is deliberate: it treats the tied 120,000 pair as a single "2nd place," which is what "2nd highest salary" almost always means.

⚠️ Nth-highest: pick the function to match the definition

"2nd highest distinct value" → DENSE_RANK() = 2. "The exact 2nd row regardless of ties" → ROW_NUMBER() = 2. "2nd place, medal-style (ties block the position)" → RANK() = 2. Clarify what "Nth highest" means before choosing — interviewers often probe exactly this.

11

Filtering Results

You cannot filter directly on a window function in WHERE. Understanding why makes the fix obvious.

Recall the logical order of execution from the aggregation notes: WHERE runs before SELECT, but window functions are computed in the SELECT step. So at the time WHERE runs, the rank doesn't exist yet — you can't reference it.

sql
-- ❌ ERROR: "Windowed functions ... cannot be used in the WHERE clause"
SELECT Name, ROW_NUMBER() OVER (ORDER BY Salary DESC) AS rn
FROM dbo.Employees
WHERE rn = 1;

-- ✅ FIX: compute the rank in a CTE (or subquery), then filter it outside
WITH Ranked AS (
    SELECT Name, ROW_NUMBER() OVER (ORDER BY Salary DESC) AS rn
    FROM dbo.Employees
)
SELECT Name FROM Ranked WHERE rn = 1;
🔑 The universal pattern

Compute the window function in a CTE or derived table, then filter its result in an outer query. This "rank, then filter" two-step is the standard shape for every top-N, Nth-highest, and deduplication problem. It also explains why those solutions almost always start with WITH.

12

Window vs GROUP BY

They look similar — both work "per group" — but one collapses rows and the other keeps them.

sql
-- Window: keep EVERY employee, and show their department's max salary beside them
SELECT Name, DeptId, Salary,
       MAX(Salary) OVER (PARTITION BY DeptId) AS DeptMax
FROM dbo.Employees;
-- returns all 5 rows, each tagged with its department's top salary

-- GROUP BY: collapse to ONE row per department
SELECT DeptId, MAX(Salary) AS DeptMax
FROM dbo.Employees
GROUP BY DeptId;
-- returns 2 rows (one per department)

🔑 The choosing rule

If you need to show each row's own values and a group-level number on the same line (rank, department average, running total, "% of group total"), use a window function. If you only want the summary rows, use GROUP BY. Window aggregates like SUM() OVER (...) and running totals are the subject of the next topic (Window Functions II).

13

Mistakes & Best Practices

The habits that keep ranking queries correct and predictable.

🎯

Add a tie-breaker

With ROW_NUMBER, ties get an arbitrary order unless your ORDER BY includes a unique tie-breaker column.

🚦

Rank, then filter

You can't use a window function in WHERE; wrap it in a CTE and filter the result in the outer query.

🤝

Match the function to ties

Distinct-Nth → DENSE_RANK; exact row → ROW_NUMBER; medal-style → RANK.

🔀

OVER's ORDER BY ≠ final sort

Add a separate query-level ORDER BY if you also want the output rows sorted.

🗂️

PARTITION BY for per-group

Restarts the ranking per group without collapsing rows — the "per department" tool.

📖

Prefer window over correlated subquery

For "each row vs its group," a window function is usually clearer and faster than a per-row correlated subquery.

⚠️ Nondeterministic ROW_NUMBER bites in production

A pagination or dedupe query that ranks with ROW_NUMBER() OVER (ORDER BY SomeNonUniqueColumn) can return rows in a different order run-to-run when values tie — causing pages to shuffle or the "wrong" duplicate to survive. Always include a unique tie-breaker (like the primary key) in the ORDER BY.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is a window function? Junior

A function that computes a value across a set of related rows (a "window") but returns a result for every row, without collapsing them. It's written with an OVER (...) clause. Unlike GROUP BY, which merges rows into summaries, a window function keeps all your detail rows and just adds a computed column, like a rank.

Q02 What does the OVER clause do? Junior

It defines the "window" — the set of rows the function works over. It has two optional parts: PARTITION BY splits rows into groups (the ranking restarts per group), and ORDER BY sets the order used to assign values within each group. Seeing OVER is how you recognize a window function.

Q03 What does ROW_NUMBER do? Junior

It assigns a unique sequential number — 1, 2, 3, … — to each row in the window, in the order you specify. It never produces ties: even rows with equal values get different numbers. It's ideal for "give me exactly one top row," pagination, and removing duplicates.

Q04 What does PARTITION BY do? Junior

It divides rows into independent groups, and the window function restarts for each group. For example, PARTITION BY DeptId ranks employees within each department separately, so each department's ranking begins again at 1. It's the "per group" control for window functions.

Q05 How is a window function different from GROUP BY? Junior

GROUP BY collapses many rows into one summary row per group, losing the detail. A window function computes a per-group value too, but keeps every original row and just adds the value as a new column. Use a window function when you need the individual rows and a group-level number together.

Q06 Does ORDER BY inside OVER sort the final result? Junior

No. The ORDER BY inside OVER only decides how ranks are assigned within the window. It does not sort the rows your query returns. If you want the output sorted too, add a separate ORDER BY at the end of the whole query.

Q07 What does RANK do? Junior

RANK numbers rows in order, but rows that tie get the same rank, and then the next rank skips ahead to account for them. Two rows tied at rank 2 means the next row is rank 4 (rank 3 is skipped) — just like Olympic standings with two silver medalists and no bronze.

Q08 What is DENSE_RANK? Junior

Like RANK, tied rows share a number — but DENSE_RANK does not skip afterward. It counts distinct values, so after two rows tied at rank 2, the next row is rank 3 (no gap). It's the natural choice for "the Nth distinct highest value."

Q09 Explain ROW_NUMBER vs RANK vs DENSE_RANK. Mid

All number rows in an order; they differ on ties. Take two rows tied at second place: ROW_NUMBER gives them 2 and 3 (always unique). RANK gives both 2, then jumps to 4 (shares, then a gap). DENSE_RANK gives both 2, then 3 (shares, no gap). ROW_NUMBER = positions, RANK = medal standings, DENSE_RANK = distinct-value ranking.

Q10 How do you get the top row per group? Mid

Number the rows within each group and keep number 1. Put ROW_NUMBER() OVER (PARTITION BY GroupCol ORDER BY SortCol DESC) in a CTE, then in the outer query filter WHERE rn = 1. For the top N per group, filter rn <= N.

Q11 How do you find the 2nd highest salary? Mid

Rank salaries descending with DENSE_RANK() OVER (ORDER BY Salary DESC) in a CTE, then select the distinct salary where the rank equals 2. DENSE_RANK is used so tied top salaries count as one "1st place" and the genuine 2nd distinct salary is returned. (Confirm whether "2nd highest" means distinct value or exact row, since that changes the function.)

Q12 Why can't you filter on a window function in WHERE, and how do you fix it? Mid

Because of the logical order of execution: WHERE runs before SELECT, but window functions are computed in the SELECT step — so the rank doesn't exist yet when WHERE runs. The fix is to compute the window function in a CTE or subquery and filter its result in the outer query ("rank, then filter").

Q13 What does NTILE do? Mid

NTILE(n) splits the ordered rows into n roughly equal buckets and labels each row with its bucket number. NTILE(4) gives quartiles, NTILE(100) percentiles. When rows don't divide evenly, the earlier buckets get the extra rows. It groups by position, so tied values can fall in different buckets.

Q14 How is PARTITION BY different from GROUP BY? Mid

Both define groups, but GROUP BY collapses each group into a single summary row, while PARTITION BY keeps every row and computes the window value within each group. Think of PARTITION BY as "GROUP BY that doesn't throw away the detail."

Q15 Which ranking function produces no gaps when there are ties? Mid

DENSE_RANK. Tied rows share a rank and the next distinct value gets the very next integer, so the sequence has no holes (1, 2, 2, 3, …). RANK also lets ties share but then skips numbers (1, 2, 2, 4, …).

Q16 Can you use a ranking function without ORDER BY inside OVER? Mid

For ranking functions (ROW_NUMBER, RANK, DENSE_RANK, NTILE), ORDER BY inside OVER is required — there's no meaningful ranking without an order. (Aggregate window functions like SUM() OVER (PARTITION BY ...) can omit ORDER BY, in which case they compute over the whole partition.)

Q17 For "Nth highest salary," which ranking function and why? Senior

It depends on the definition. For the Nth distinct value (the usual intent), use DENSE_RANK() = N so tied values count once. For the exact Nth row regardless of ties, use ROW_NUMBER() = N. For medal-style semantics where ties consume positions, use RANK() = N. A strong answer states the ambiguity and picks accordingly — usually DENSE_RANK.

Q18 How would you implement top-N per group, and what's an alternative to window functions? Senior

The standard way: ROW_NUMBER() OVER (PARTITION BY grp ORDER BY sort DESC) in a CTE, then filter rn <= N. An alternative is CROSS APPLY with a correlated TOP (N) subquery per group — sometimes faster when there's a supporting index and few groups. Which wins depends on data size, the number of groups, and available indexes; test both on real data.

Q19 Where do window functions sit in the logical order of execution? Senior

They're evaluated in the SELECT phase, after FROM, WHERE, GROUP BY, and HAVING, but before ORDER BY and any TOP/OFFSET-FETCH. That's precisely why you can't reference a window function in WHERE/GROUP BY/HAVING (they run earlier) and must wrap it in a CTE to filter on it.

Q20 What drives window-function performance? Senior

The main cost is sorting: the engine must order rows by the PARTITION BY + ORDER BY columns. An index whose key matches those columns lets it read rows already sorted and skip an expensive sort operator. Large partitions, spills to tempdb, and multiple different window orderings in one query add cost. Check the execution plan for a Sort operator; a supporting index often removes it.

Q21 How does an aggregate window function differ from a ranking function? Senior

Ranking functions (ROW_NUMBER, RANK, …) assign a position number. Aggregate window functions (SUM(), AVG(), COUNT(), MAX() with OVER) compute an aggregate across the window but, unlike GROUP BY, attach it to every row — e.g., each employee's salary next to their department total. Adding ORDER BY to an aggregate window turns it into a running/cumulative calculation (the focus of Window Functions II).

Q22 Why can ROW_NUMBER be nondeterministic, and how do you fix it? Senior

If the OVER (ORDER BY ...) columns aren't unique, tied rows can be numbered in any order, and that order isn't guaranteed to be stable between executions. This breaks pagination (pages shuffle) and dedupe (an unpredictable duplicate survives). The fix is to add a unique tie-breaker — typically the primary key — to the ORDER BY, making the numbering deterministic.

Q23 Where in a query can window functions appear? Senior

Only in the SELECT list and the ORDER BY clause. They cannot appear in WHERE, GROUP BY, or HAVING, because those run before the window functions are computed. To filter or group by a window result, materialize it first in a CTE or derived table, then reference it in the outer query.

Q24 How would you deduplicate rows, keeping one per key, using ROW_NUMBER? Senior

Number rows within each duplicate group and keep number 1: ROW_NUMBER() OVER (PARTITION BY DuplicateKey ORDER BY KeepPriority) in a CTE, then keep rn = 1 (or DELETE the rows where rn > 1). The ORDER BY decides which duplicate to keep, so choose it deliberately (e.g., most recent, or lowest id) and include a unique tie-breaker so the choice is deterministic.
