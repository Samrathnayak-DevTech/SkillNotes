Indexes 📑

An index is a sorted lookup structure that lets SQL Server find rows without reading the whole table — exactly like the index at the back of a book. The two big kinds are the clustered index (which physically orders the table itself) and non-clustered indexes (separate structures that point back to rows). Used well, indexes turn slow scans into instant seeks; overused, they quietly tax every write. No prior coding background assumed.

01

What an Index Is

A pre-sorted structure that points to rows, so the engine can jump straight to what it needs instead of reading every row.

Without an index, answering "find the employee named Grace" means checking every single row — a table scan. With an index on Name, the rows are pre-sorted, so SQL Server can home in on Grace almost instantly — an index seek. The trade is that the index takes extra storage and must be kept up to date on every insert, update, and delete.

📖 Analogy

A textbook's back-of-book index is the perfect mental model. To find every mention of "photosynthesis" you don't read all 900 pages — you flip to the alphabetical index, find the word, and jump to the listed page numbers. A database index works the same way: it's a sorted list of values, each pointing to where the full row lives. The book's pages are the table; the index is the shortcut.

🔑 The core trade-off, stated once

Indexes make reads faster and writes slower. Every index is another copy of some columns that must be updated when data changes. So the art of indexing is adding just enough to speed up the queries you actually run, without drowning your inserts and updates in maintenance. Keep this tension in mind for the rest of the page.

03

The B-Tree

SQL Server indexes are balanced trees. A seek walks from the top down a few levels to reach the row — a handful of steps even in a huge table.

seeking EmpId = 4 in a clustered B-tree
root1–100
interm. 1–3 4–6 7–100
leaf 4: Edsger… 5: Katherine… 6: …

The root and intermediate levels hold ranges of key values that route the search. The leaf level holds the data itself (for a clustered index) or the key plus a pointer (for a non-clustered index). To find EmpId = 4, the engine reads the root, follows the "4–6" branch, and lands on the leaf page — just a few page reads.

🔑 Why seeks are so fast

Because the tree is balanced and each node points to a wide range, the number of levels grows very slowly as the table grows — a table with millions of rows is typically only 3–4 levels deep. A seek touches roughly one page per level, so it stays fast no matter how big the table gets. A scan, by contrast, reads every leaf page.

04

Heaps

A table with no clustered index is a "heap" — rows sit in no particular order, like an unsorted pile of papers.

If you create a table without a clustered index (and without a primary key that makes one), its rows are stored as a heap: an unordered collection. Finding a specific row means scanning the whole heap unless a non-clustered index helps. Rows in a heap are located by a physical address called a RID (row identifier: file:page:slot).

⚠️ Heaps are usually not what you want

For most tables, a heap performs worse than a clustered table for typical queries: there's no order to exploit, range scans are inefficient, and non-clustered indexes must store the bulkier RID pointer. The common best practice is to give nearly every table a clustered index — very often on the primary key. Heaps have niche uses (e.g., fast bulk-load staging), but they're the exception.

05

Clustered Index

The clustered index is the table, physically sorted by its key. There can be only one, because a table can only be stored in one order.

sql
-- A PRIMARY KEY makes a clustered index by default...
CREATE TABLE dbo.Employees (
    EmpId  INT IDENTITY(1,1) PRIMARY KEY,   -- clustered on EmpId
    Name   NVARCHAR(100),
    DeptId INT
);

-- ...or create one explicitly
CREATE CLUSTERED INDEX IX_Employees_EmpId
ON dbo.Employees (EmpId);
🔑 The leaf level of a clustered index holds the whole row

Because the table is physically ordered by the clustered key, the index's leaf pages are the data pages — there's no separate "table" to jump to. A seek on the clustered key therefore returns the full row directly. This is why range queries on the clustered key (e.g., EmpId BETWEEN 10 AND 20) are very efficient: the matching rows are physically adjacent.

✅ What makes a good clustered key

Ideally narrow, unique, unchanging, and ever-increasing (an IDENTITY integer is the classic choice). Narrow because every non-clustered index carries a copy of it; unchanging because updating the key physically moves the row; ever-increasing to avoid page splits from inserting into the middle.

06

Non-Clustered Index

A separate, sorted structure on other columns. Its leaf level holds the key plus a pointer back to the actual row. You can have many.

sql
-- Speed up filtering/joining Employees by department
CREATE NONCLUSTERED INDEX IX_Employees_DeptId
ON dbo.Employees (DeptId);

Think of it as a second back-of-book index sorted by a different topic. It stores DeptId values in order, and next to each one a row locator — the clustered key (if the table has one) or the RID (if it's a heap) — telling SQL Server where to find the rest of the row.

🔑 The row locator is the clustered key

In a clustered table, every non-clustered index uses the clustered key as its pointer back to the row. That's the concrete reason a wide clustered key is costly: its bytes are duplicated inside every non-clustered index. It's also why a seek on a non-clustered index that needs extra columns has to do a second hop — a key lookup — back into the clustered index.

07

Seek vs Scan

The two ways the engine reads data. A seek jumps to the rows it needs; a scan reads them all. Seeks are what indexes are for.

🎯

Index Seek

Navigates the B-tree to the matching rows. Fast; touches a few pages. Wants a selective, usable predicate.

📃

Index / Table Scan

Reads every leaf page. Fine for small tables or when most rows match; slow on big tables for a few rows.

🔁

Key Lookup

After a non-clustered seek, hops to the clustered index to fetch columns the index didn't have.

How the engine decides 🧭

🔍 Is there an index whose key starts with the filter column? No → it must scan (read every row).
🧩 Is the predicate usable for a seek (SARGable)? A function wrapping the column (e.g., YEAR(col)=…) disables the seek → scan.
🛡️ Does the index contain every column the query needs? Yes → a clean index seek. Fastest path — nothing else to fetch.
🔁 No — needs more columns? Seek + a key lookup per row. If too many rows match, the optimizer may just scan instead.
⚠️ A seek isn't always chosen — the "tipping point"

If a non-clustered seek would match a large fraction of the table, each row also needs a key lookup, and thousands of lookups can cost more than simply scanning the whole table. Past a certain threshold the optimizer rationally prefers a scan. This is why a covering index (next section), which removes the lookups, matters so much.

08

Covering & INCLUDE

An index that contains every column a query needs "covers" it — the query is answered from the index alone, with no lookup back to the table.

sql
-- Target query:  SELECT Name, Salary FROM dbo.Employees WHERE DeptId = 1;

-- Plain index on DeptId: seeks, then does a key lookup for Name & Salary
CREATE NONCLUSTERED INDEX IX_Emp_Dept ON dbo.Employees (DeptId);

-- Covering index: keeps Name & Salary IN the index -- no lookup needed
CREATE NONCLUSTERED INDEX IX_Emp_Dept_Covering
ON dbo.Employees (DeptId)
INCLUDE (Name, Salary);
🔑 Key columns vs INCLUDE columns

Key columns (in the parentheses) are what the index is sorted by — use them for filtering, joining, and ordering. INCLUDE columns ride along only in the leaf level; they're not sorted, but they're available to be returned. Put columns you search/sort by in the key, and columns you merely select in INCLUDE. That covers the query while keeping the key narrow.

✅ Covering kills key lookups

Once the index has DeptId (to seek) plus Name and Salary (to return), the query is satisfied entirely from the index — no trips back to the clustered index. Eliminating those per-row lookups is one of the biggest, easiest performance wins, and it also raises the tipping point before the optimizer abandons the seek.

09

Composite & Key Order

A composite index has multiple key columns, and their order is everything. An index is only seekable from its leftmost column onward.

sql
CREATE NONCLUSTERED INDEX IX_Orders_Emp_Date
ON dbo.Orders (EmpId, OrderDate);   -- sorted by EmpId, then OrderDate
Query predicateCan it seek this index?Why
WHERE EmpId = 2 AND OrderDate >= '2024-01-01'Yes — idealUses both key columns, left to right
WHERE EmpId = 2YesUses the leftmost column
WHERE OrderDate >= '2024-01-01'No seekOrderDate isn't the leftmost key → scan
🔑 The leftmost-prefix rule

Think of a phone book sorted by (LastName, FirstName). You can find everyone named "Lovelace," or "Lovelace, Ada" — but you can't efficiently find everyone whose first name is "Ada," because the book isn't sorted by first name. Same with a composite index: it supports seeks on the first key column, the first two, the first three, and so on — never a middle or trailing column on its own.

✅ Ordering the key columns

Put the column used with equality filters first, then columns used for ranges or ordering. Also favour the more selective column early (the one that narrows results most). Column order is the single most common composite-index mistake — design it around the queries you actually run.

10

When They Help

Indexes speed up any operation that benefits from finding or ordering rows quickly.

🔎

WHERE filters

Selective predicates seek straight to matching rows instead of scanning.

🔗

JOINs

An index on the join column lets each match be found by seek — crucial for large joins.

🔀

ORDER BY / GROUP BY

If the index is already in the needed order, SQL Server can skip an expensive sort.

🥇

MIN / MAX / TOP

The extreme value sits at one end of the index — found instantly.

🚦

Uniqueness checks

A unique index enforces and rapidly verifies "no duplicates."

🛡️

Covering queries

Answer entirely from the index — no table access at all.

🔑 Selectivity is the deciding factor

An index helps most when the filter is selective — it narrows to a small fraction of rows. Indexing a column with few distinct values (like a yes/no flag) rarely helps, because a seek would still return half the table, and the optimizer will just scan. High-selectivity columns (ids, emails, dates in a narrow range) are where non-clustered indexes shine.

11

When They Hurt

Every index is a liability as well as an asset. Too many, or the wrong ones, degrade writes and waste space.

✍️

Slower writes

Every INSERT/UPDATE/DELETE must maintain every affected index.

💾

Storage

Indexes are extra copies of data; wide or numerous indexes bloat the database and backups.

🗑️

Unused indexes

An index no query uses is pure cost — maintained forever, benefiting no one.

👯

Duplicates & overlap

Redundant indexes (same leading columns) double the write cost for no read gain.

⚠️ The write tax is real

On a write-heavy table, each extra index makes every modifying statement do more work — updating the index tree, splitting pages, and updating statistics. A table with a dozen indexes can spend more effort maintaining them than doing the actual insert. Index the queries that matter; drop indexes that no query uses (SQL Server's DMVs can show usage stats).

🔑 The balancing question

For each candidate index, ask: which real queries will it speed up, and how much will it slow down my writes and grow my storage? If you can't name a query it helps, don't create it. Indexing is always a bet that the read savings outweigh the write cost — make that bet deliberately.

12

Other Index Types

Beyond clustered and non-clustered, a few specialized index types solve specific problems.

sql
-- UNIQUE index: enforces no duplicates AND speeds lookups
CREATE UNIQUE NONCLUSTERED INDEX UX_Dept_Name ON dbo.Departments (Name);

-- FILTERED index: indexes only a subset of rows (smaller, cheaper)
CREATE NONCLUSTERED INDEX IX_Emp_TopBrass
ON dbo.Employees (Name)
WHERE ManagerId IS NULL;   -- only the handful of top-level people

UniqueEnforces uniqueness & speeds lookupsA column/combination must have no duplicates

🔑 Rowstore vs columnstore in one line

Traditional (rowstore) B-tree indexes are ideal for OLTP — finding and modifying individual rows. Columnstore indexes store each column together, highly compressed, and are built for analytics — scanning and aggregating millions of rows. Use rowstore for transactional apps, columnstore for large reporting/warehouse workloads.

13

Mistakes & Best Practices

The habits that keep indexes an asset rather than a tax.

📍

Clustered on nearly every table

Usually the PK — narrow, unique, increasing. Avoid leaving big tables as heaps.

🧩

Keep predicates SARGable

Don't wrap the indexed column in a function; it forces a scan (deep dive in the next topic).

🛡️

Cover hot queries

Use INCLUDE to eliminate key lookups on frequently run queries.

🔢

Order composite keys well

Equality columns first, then range/sort columns; more selective columns earlier.

🗑️

Drop unused/duplicate indexes

They only cost writes and space. Check usage DMVs before keeping them.

📈

Keep statistics fresh

The optimizer relies on statistics to choose seek vs scan; stale stats cause bad plans.

⚠️ Don't index blindly from "missing index" hints

SQL Server suggests missing indexes, but those hints are per-query and ignore write cost and existing indexes. Blindly creating every suggestion leads to overlapping, redundant indexes that hammer writes. Treat the hints as input, consolidate them, and weigh them against the table's full workload before creating anything.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is an index? Junior

A sorted lookup structure that lets the database find rows quickly without reading the whole table — like the index at the back of a book. Instead of scanning every page, you look up the value in the sorted index and jump straight to the matching rows.

Q02 Why use an index? Junior

To make reads faster. Filtering, joining, sorting, and finding min/max values can all use an index to avoid scanning the entire table. The cost is extra storage and slower writes, since the index must be kept up to date — so you add indexes for the queries that matter.

Q03 What is a clustered index? Junior

An index that determines the physical order of the table — the rows are actually stored sorted by the clustered key. Because a table can only be stored in one order, there's just one clustered index, and its leaf level holds the full rows themselves.

Q04 How many clustered indexes can a table have? Junior

Exactly one, because the clustered index defines how the rows are physically ordered, and data can only be stored in a single order. A table can have many non-clustered indexes, but only one clustered index.

Q05 What is a non-clustered index? Junior

A separate sorted structure on one or more columns, kept apart from the table. Its leaf level stores the indexed key values plus a pointer (the row locator) back to the actual row. A table can have several non-clustered indexes for different query patterns.

Q06 What's the difference between an index seek and an index scan? Junior

A seek navigates the index tree directly to the matching rows — fast, touching only a few pages. A scan reads every row (or every leaf page) and checks each one. Seeks are efficient for selective queries; scans are fine for small tables or when most rows match anyway.

Q07 Does the primary key create an index? Junior

Yes. In SQL Server, declaring a PRIMARY KEY automatically creates a unique index on those columns — clustered by default (unless a clustered index already exists or you specify NONCLUSTERED). So the primary key is usually also the table's clustered index.

Q08 What's the downside of indexes? Junior

They slow down writes and use extra storage. Every insert, update, or delete must also update each affected index, and each index is another copy of some data taking up space. Too many indexes can make a write-heavy table noticeably slower, so they should be added deliberately.

Q09 Clustered vs non-clustered — what are the key differences? Mid

A clustered index is the table, physically ordered by its key, with full rows at the leaf; there's one per table. A non-clustered index is a separate structure whose leaf holds the key plus a pointer to the row; you can have many. A clustered-key seek returns the row directly, while a non-clustered seek may need a key lookup to fetch extra columns.

Q10 What is a heap? Mid

A table with no clustered index, so its rows are stored in no particular order. Rows are located by a physical RID (file:page:slot), and non-clustered indexes on a heap store that RID as their pointer. Heaps are usually slower for typical queries and are generally avoided except for niche cases like bulk-load staging.

Q11 What is a covering index, and what does INCLUDE do? Mid

A covering index contains every column a query needs, so the query is answered from the index alone — no lookup back to the table. INCLUDE adds non-key columns to the index's leaf level: they're not part of the sort key (so they don't bloat it) but are available to be returned. You seek/sort on key columns and INCLUDE the extra columns you only select.

Q12 What is a key lookup and why should you care? Mid

After seeking a non-clustered index, if the query needs columns the index doesn't have, the engine hops back to the clustered index once per matching row to fetch them — that hop is a key lookup (a RID lookup on a heap). It's fine for a few rows, but many lookups get expensive and can push the optimizer to scan instead. A covering index removes them.

Q13 In a composite index, does column order matter? Mid

Very much. A composite index is sorted by its columns left to right, so it's seekable only on a leftmost prefix: the first column, the first two, and so on. An index on (EmpId, OrderDate) helps queries filtering on EmpId (or both), but not a query filtering on OrderDate alone. Order equality columns first, then range/sort columns, more selective ones earlier.

Q14 Which query operations benefit from indexes? Mid

WHERE filters (seek to matching rows), JOINs (find matches by seek), ORDER BY/GROUP BY (reuse the index's order to skip a sort), MIN/MAX/TOP (grab the value at one end), and uniqueness checks. Covering indexes can answer a query entirely from the index. The benefit depends on the filter being selective.

Q15 What's the difference between a unique index and a unique constraint? Mid

Functionally they're almost the same — a unique constraint is implemented by a unique index under the hood, and both prevent duplicate values and speed up lookups. The difference is intent and metadata: a constraint expresses a business rule and can be referenced by foreign keys, while a unique index is described as a performance/uniqueness structure. Creating either one gives you the same enforcing index.

Q16 What is a filtered index and when would you use one? Mid

A non-clustered index with a WHERE clause, so it indexes only the rows matching that condition. It's smaller and cheaper to maintain than a full index. Use it when queries repeatedly target a well-defined subset — e.g., only active records, only rows where a status is 'Open', or only the few rows where a nullable column is set.

Q17 Explain the B-tree structure and how a seek traverses it. Senior

An index is a balanced tree with a root page, intermediate levels, and a leaf level. The root and intermediate pages hold key ranges that route the search; the leaf holds the data (clustered) or key-plus-locator (non-clustered). A seek starts at the root, follows the branch whose range covers the target value, descends one level at a time, and lands on the leaf — roughly one page read per level. Because the tree stays shallow (a few levels even for millions of rows), seeks remain fast as the table grows.

Q18 What makes a predicate SARGable, and why does it matter? Senior

SARGable ("Search ARGument able") means the predicate can use an index seek because the indexed column is left untouched — e.g., OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'. Wrapping the column in a function or expression (YEAR(OrderDate) = 2024, col + 0, leading-wildcard LIKE '%x') makes it non-SARGable: the engine must compute the expression for every row and can't seek, forcing a scan. Rewriting to leave the column bare restores the seek. (The next topic covers this in depth.)

Q19 How do you decide which columns go in the key versus INCLUDE? Senior

Put columns you filter, join, or order by in the key, ordered by how they're used (equality first, then ranges) and by selectivity. Put columns you only return in INCLUDE, so the index covers the query without enlarging the sort key (which would bloat every level and slow maintenance). The goal is a narrow, seekable key plus just enough included columns to avoid key lookups on your hot queries.

Q20 What is index selectivity and how does it affect index use? Senior

Selectivity is how well a column's values distinguish rows — high selectivity means many distinct values / few rows per value (like an id or email); low selectivity means few distinct values (like a boolean flag). The optimizer uses statistics on selectivity to estimate how many rows a predicate returns. A highly selective predicate favours a seek; a low-selectivity one returns so many rows that a scan (or a different plan) is cheaper, so an index on a low-selectivity column often goes unused.

Q21 What ongoing maintenance do indexes need? Senior

Over time, inserts/updates/deletes cause fragmentation (logical order diverging from physical order, plus partially empty pages), which you address by reorganizing or rebuilding indexes. Fill factor leaves free space on pages to reduce page splits from inserts. And statistics must stay current so the optimizer estimates row counts correctly — stale stats are a common cause of bad plans. Modern SQL Server updates stats automatically, but heavy workloads may need scheduled maintenance.

Q22 When would the optimizer ignore an existing index? Senior

When a scan is estimated to be cheaper. Common cases: the predicate is non-SARGable (function on the column); the query returns a large fraction of rows, so a non-clustered seek plus many key lookups costs more than a scan (the "tipping point"); the column is low-selectivity; statistics are stale and misestimate row counts; or an implicit type conversion prevents the seek. A covering index or a rewrite often gets the seek back.

Q23 Columnstore vs rowstore indexes — when do you use each? Senior

Rowstore (traditional B-tree) indexes store whole rows together and excel at OLTP: quickly finding and modifying individual rows and small ranges. Columnstore indexes store each column separately and highly compressed, and excel at analytics: scanning and aggregating millions of rows for reporting/data-warehouse queries. Use rowstore for transactional systems; use columnstore for large analytic workloads — some systems combine both on the same table.

Q24 How does the clustered key affect non-clustered indexes? Senior

Every non-clustered index uses the clustered key as its row locator — it's stored in the non-clustered leaf (and, for uniqueness, in the upper levels too). So a wide clustered key inflates every non-clustered index, increasing their size and maintenance cost. It also means a non-clustered seek that needs more columns does a key lookup via that clustered key. This is the core reason to keep the clustered key narrow, unique, static, and ever-increasing.
