Day 11 · ASP.NET Core Fundamentals

🌐 HttpClient: Talking to the Outside World

Your app rarely lives alone — it calls payment gateways, weather services, other microservices. Today you'll learn to make those calls the right way: IHttpClientFactory instead of the deceptively simple new HttpClient(), clean typed clients, and turning JSON responses into C# objects with System.Text.Json.

📞 HTTP Requests 🏭 IHttpClientFactory 📦 Typed Clients 🔤 System.Text.Json 🛡️ Resilience

01📞 Calling Other APIs

A REST API call is a simple round trip: your app sends an HTTP request (a verb like GET, a URL, headers, maybe a body) to another server, and gets back an HTTP response (a status code, headers, and usually a body — most often JSON). In .NET, the class that performs this round trip is HttpClient.

The using is easy. The using it correctly is where nearly every developer stumbles — because the obvious way (new HttpClient() for each call) quietly destabilizes production. This whole lesson exists to teach the correct way from the start.

☎️ Analogy

An HttpClient is like a phone line to another office. Setting up a line takes real work — dialing, handshaking, establishing the connection. The naive approach is to install and rip out a brand-new phone line for every single call, then leave the dead line's wires warm in the wall for minutes afterward. Do that a few thousand times a second and the building runs out of wall sockets. The professional approach keeps a small switchboard of reusable lines — that's what today's factory gives you.

🔑 Key Idea

Calling an external API is I/O — exactly the kind of waiting Day 10 was about. Every method here is async, takes a CancellationToken, and is awaited all the way up. HttpClient and async are inseparable partners.

03⚠️ The HttpClient Trap

Here's the code that looks perfect and is quietly dangerous:

C# · The tempting mistake
// ❌ NEVER do this in a request handler or hot path:
public async Task<Post?> GetPostAsync(int id)
{
    using var client = new HttpClient();   // 💣 new client + new socket every call
    return await client.GetFromJsonAsync<Post>(
        $"https://jsonplaceholder.typicode.com/posts/{id}");
}

It compiles, it works in a demo, and it will destabilize your app in production. Two distinct failures:

🔌Socket exhaustion

Each HttpClient holds its own connection pool. Disposing it doesn't free the underlying TCP socket immediately — the OS holds it in TIME_WAIT for minutes. Create thousands per second and you run out of ports; new connections fail with obscure errors.

📡Stale DNS

The opposite fix — one static HttpClient forever — solves sockets but caches DNS for the app's whole life. If the target service's IP changes (failover, scaling), your app keeps calling the dead address.

⚠️ HttpClient Is Meant to Be Reused

HttpClient implements IDisposable, which screams "wrap me in using!" — but that instinct is exactly wrong here. It's designed to be long-lived and shared, not created and disposed per call. The disposable interface is a genuine footgun.

🔑 The Dilemma

Per-request client → socket exhaustion. One eternal static client → stale DNS. You need something that reuses connections yet periodically refreshes them to pick up DNS changes. That's precisely the problem IHttpClientFactory was built to solve.

04🏭 IHttpClientFactory

IHttpClientFactory threads the needle. It manages the expensive part — the HttpMessageHandler that owns the connection pool — separately from the cheap, disposable HttpClient wrapper you actually use. Handlers are pooled and reused (no socket exhaustion) but rotated on a timer (default ~2 minutes, so DNS changes get picked up).

🚕 Analogy

Think of a taxi rank. The cars (message handlers, with their engines warm and routes known) are expensive to prepare, so the depot keeps a managed fleet and reuses them — and retires each car every couple of hours so it re-checks the maps (DNS). You just grab whichever taxi ticket (HttpClient) is next in line, take your ride, and toss the ticket — the car keeps running for the next passenger. You never build a car per trip.

What you get 🎁

  • 🔌 No socket exhaustion — handlers and their connections are pooled and reused.
  • 📡 No stale DNS — handlers are recycled periodically so new IPs are picked up.
  • ⚙️ Central configuration — base address, default headers, and timeout live in one place per client.
  • 🧅 A handler pipeline — plug in cross-cutting middleware (logging, auth, retries) as delegating handlers.
  • 🛡️ Resilience hooks — one line adds retries, timeouts, and circuit breakers (section 09).
✅ Registration Is One Line

You register HTTP clients in Program.cs with builder.Services.AddHttpClient(...). That single call wires the factory into DI; from then on you ask DI for a client (or a typed client) and the lifetime management is handled for you.

⚠️ A Factory Client Is Still Short-Lived

The clients the factory hands you are meant to be requested fresh and used briefly — don't cache one in a field for hours. And never inject a typed client into a singleton service: the singleton would pin one handler forever, defeating the DNS-refresh rotation. (More on typed-client lifetime in section 05.)

05📦 Named vs Typed Clients

There are three ways to consume the factory, escalating in structure. For real applications, typed clients are the sweet spot.

1 — Basic (unnamed) 🌱

C# · Just get a client
builder.Services.AddHttpClient();   // register the factory

// Inject IHttpClientFactory, create a default client per use:
var client = _factory.CreateClient();
var post = await client.GetFromJsonAsync<Post>("https://.../posts/1", ct);

2 — Named 🏷️

C# · Configuration behind a name
builder.Services.AddHttpClient("jsonplaceholder", client =>
{
    client.BaseAddress = new Uri("https://jsonplaceholder.typicode.com/");
    client.Timeout = TimeSpan.FromSeconds(30);
    client.DefaultRequestHeaders.Add("Accept", "application/json");
});

// Ask for it by name — you must remember the string everywhere:
var client = _factory.CreateClient("jsonplaceholder");
var post = await client.GetFromJsonAsync<Post>("posts/1", ct);

3 — Typed (recommended) 🌟

A typed client is a class of your own that receives a pre-configured HttpClient in its constructor and exposes clean, intention-revealing methods. It hides the URLs and JSON plumbing behind a tidy API and is fully DI-friendly:

C# · Services/JsonPlaceholderClient.cs
public class JsonPlaceholderClient(HttpClient http)
{
    // The HttpClient is already configured (base address, headers, timeout)
    // by the registration below — this class just uses it.
    public async Task<Post?> GetPostAsync(int id, CancellationToken ct = default)
        => await http.GetFromJsonAsync<Post>($"posts/{id}", ct);

    public async Task<List<Post>> GetPostsByUserAsync(int userId, CancellationToken ct = default)
        => await http.GetFromJsonAsync<List<Post>>($"posts?userId={userId}", ct) ?? [];
}
C# · Program.cs — register & inject
builder.Services.AddHttpClient<JsonPlaceholderClient>(client =>
{
    client.BaseAddress = new Uri("https://jsonplaceholder.typicode.com/");
    client.Timeout = TimeSpan.FromSeconds(30);
});

// Anywhere via DI — no factory, no magic strings, just a clean dependency:
app.MapGet("/posts/{id}", async (int id, JsonPlaceholderClient api, CancellationToken ct) =>
    await api.GetPostAsync(id, ct) is { } post
        ? Results.Ok(post)
        : Results.NotFound());
StyleHow you get itBest for
🌱 Basic_factory.CreateClient()One-off calls, quick scripts.
🏷️ Named_factory.CreateClient("name")Several configs, or when you can't add a class.
🌟 TypedInject MyApiClient directlyReal apps — strongly typed, testable, self-documenting.
🔑 Why Typed Wins

A typed client turns "call this URL and parse JSON" scattered across your codebase into a single, named capability — weatherApi.GetForecastAsync(city). Callers depend on your interface, not on HttpClient or magic strings, which makes the code readable and trivial to mock in tests. It's the same "depend on abstractions" lesson from Days 3, 6, and 7.

⚠️ Typed Client Lifetime

A typed client's lifetime is tied to the injected HttpClient, so it's transient/short-lived by design. That's why you must not capture it in a singleton — doing so pins one message handler forever and breaks the DNS-refresh rotation, quietly reintroducing the stale-DNS problem the factory exists to prevent.

06📤 Making Requests

Two levels of API. The JSON helpers from System.Net.Http.Json do request + deserialize in one line — reach for these first. The raw methods give you the full HttpResponseMessage when you need the status code or headers.

C# · The JSON convenience helpers
using System.Net.Http.Json;

// GET + deserialize in one call (throws on non-success status):
Post? post = await http.GetFromJsonAsync<Post>("posts/1", ct);

// POST an object as JSON, read the created result back:
var response = await http.PostAsJsonAsync("posts", new NewPost("Title", "Body", 1), ct);
response.EnsureSuccessStatusCode();
Post? created = await response.Content.ReadFromJsonAsync<Post>(ct);

// PUT and DELETE follow the same shape:
await http.PutAsJsonAsync($"posts/{id}", updated, ct);
await http.DeleteAsync($"posts/{id}", ct);
C# · Raw methods when you need the response details
// Get the full response so you can inspect status & headers:
HttpResponseMessage res = await http.GetAsync("posts/1", ct);

if (res.StatusCode == HttpStatusCode.NotFound)
    return null;                                    // handle 404 without an exception

res.EnsureSuccessStatusCode();                      // throw for other failures
Post? post = await res.Content.ReadFromJsonAsync<Post>(ct);
⚠️ Never Mutate the Injected Client's Defaults

Don't set http.BaseAddress, http.Timeout, or http.DefaultRequestHeaders inside a request handler on a factory-provided client. Those clients come from a shared pool; mutating defaults causes race conditions and header bleed between requests (one request's auth header leaking into another's). Configure defaults once, at registration. For per-request headers, set them on an individual HttpRequestMessage instead.

✅ Per-Request Headers Done Right

Need a per-call header (say a request-specific auth token)? Build an HttpRequestMessage, add the header to it, and send with http.SendAsync(request, ct). The shared client's defaults stay untouched, so nothing bleeds between requests.

07🔤 System.Text.Json: Mapping the Response

JSON is text; your app wants objects. Deserialization is the translation, and System.Text.Json is .NET's built-in, high-performance engine (the GetFromJsonAsync<T> you've been using calls it under the hood). The core idea: define a C# type shaped like the JSON, and the serializer fills it in by matching names.

JSON · what the API returns
{
  "userId": 1,
  "id": 1,
  "title": "sunt aut facere",
  "body": "quia et suscipit..."
}
C# · the model to map onto
// A record is perfect for immutable API responses.
// System.Text.Json matches JSON names to properties CASE-INSENSITIVELY
// by default in the web helpers, so "userId" → UserId works out of the box.
public record Post(int UserId, int Id, string Title, string Body);

When names don't line up: attributes 🏷️

Real APIs use snake_case, odd names, or fields you want renamed. [JsonPropertyName] maps a JSON key to a differently named property; [JsonIgnore] drops one:

C# · Explicit name mapping
using System.Text.Json.Serialization;

public class WeatherReading
{
    [JsonPropertyName("temp_c")]          // JSON "temp_c" → C# TemperatureC
    public double TemperatureC { get; set; }

    [JsonPropertyName("condition")]
    public string Condition { get; set; } = "";

    [JsonIgnore]                          // never read/written to JSON
    public bool IsStale { get; set; }
}

Global options 🎛️

Rather than attribute every property, configure the serializer once — the cleanest fix for a whole snake_case API:

C# · JsonSerializerOptions
var options = new JsonSerializerOptions
{
    PropertyNamingPolicy = JsonNamingPolicy.SnakeCaseLower,  // temp_c ↔ TempC
    PropertyNameCaseInsensitive = true,
    NumberHandling = JsonNumberHandling.AllowReadingFromString,  // "42" → 42
    DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull
};

// Pass options to the helper:
var reading = await http.GetFromJsonAsync<WeatherReading>(url, options, ct);

Handy option flags

Option / attributeWhat it does
PropertyNameCaseInsensitiveMatch names ignoring case (on by default in the *Json* web helpers).
PropertyNamingPolicyBulk-map casing: CamelCase, SnakeCaseLower, etc.
[JsonPropertyName("x")]Map one property to a specific JSON key.
[JsonIgnore]Skip a property entirely.
NumberHandlingRead numbers that arrive as strings ("42").
DefaultIgnoreConditionOmit null (or default) values when serializing.
ConvertersCustom logic for tricky types (enums-as-string, odd date formats).
🔑 System.Text.Json vs Newtonsoft

You'll see Newtonsoft.Json (Json.NET) in older code — it's mature and feature-rich. System.Text.Json is the modern built-in default: faster, lower-allocation, no extra package. It's stricter by default (case-sensitive when used directly, though the web helpers relax that), which occasionally surprises people migrating. For new code, start with System.Text.Json and only reach for Newtonsoft if you hit a feature it lacks.

⚠️ Map Defensively

You don't control the external API's payload — fields can be missing, null, or renamed without warning. Model only the fields you actually use, make reference types nullable where the API might omit them, and never assume a field is present. A missing field deserializes to null/default, not an exception — so guard before you use it.

08🧯 Error Handling

External calls fail in more ways than local code: the network drops, the server returns 500, the request times out, the JSON is malformed. Each surfaces differently, so handle them deliberately:

HttpRequestException
network / DNS / non-success*
TaskCanceledException
timeout / cancellation
JsonException
malformed body
null result
404 or empty
The failure modes of an HTTP call · *EnsureSuccessStatusCode & the Json helpers throw HttpRequestException on 4xx/5xx
C# · Handling the failure modes
public async Task<Post?> TryGetPostAsync(int id, CancellationToken ct)
{
    try
    {
        var res = await _http.GetAsync($"posts/{id}", ct);

        if (res.StatusCode == HttpStatusCode.NotFound)
            return null;                          // expected "missing" — not an error

        res.EnsureSuccessStatusCode();            // 4xx/5xx → HttpRequestException
        return await res.Content.ReadFromJsonAsync<Post>(ct);
    }
    catch (HttpRequestException ex)
    {
        _logger.LogError(ex, "Network/HTTP error fetching post {PostId}", id);  // Day 6
        throw;
    }
    catch (TaskCanceledException) when (!ct.IsCancellationRequested)
    {
        // Cancellation the caller DIDN'T ask for = the HttpClient timeout fired.
        _logger.LogWarning("Timeout fetching post {PostId}", id);
        throw new TimeoutException($"Timed out fetching post {id}");
    }
    catch (JsonException ex)
    {
        _logger.LogError(ex, "Malformed JSON for post {PostId}", id);
        throw;
    }
}
⚠️ The Timeout Disguise

When HttpClient's own timeout elapses it throws TaskCanceledException — the same type as caller cancellation. Distinguish them with the token: when (!ct.IsCancellationRequested) means "nobody asked to cancel, so this was a timeout." (Newer .NET wraps the inner cause as a TimeoutException, but the catch pattern above is the portable idiom.)

✅ 4xx/5xx Aren't Network Failures

A 404 or 400 is a successful HTTP round trip that returned an error status — very different from the connection failing. Decide per status: often a 404 should become a null/NotFound in your domain rather than a thrown exception, while 5xx is worth retrying (next section). Check response.IsSuccessStatusCode or the specific StatusCode before calling EnsureSuccessStatusCode.

09🛡️ Resilience: Surviving Flaky Networks

Networks are unreliable by nature — packets drop, services hiccup, a dependency briefly overloads. A robust app expects transient failure and absorbs it. In .NET 8+ the built-in Microsoft.Extensions.Http.Resilience package (built on Polly v8) adds this with essentially one line:

C# · One-line standard resilience
// dotnet add package Microsoft.Extensions.Http.Resilience
builder.Services.AddHttpClient<JsonPlaceholderClient>(client =>
{
    client.BaseAddress = new Uri("https://jsonplaceholder.typicode.com/");
})
.AddStandardResilienceHandler();   // retry + circuit breaker + timeout + rate limiter

That standard handler bundles the four patterns every external call should have:

🔁Retry

Automatically re-attempt transient failures (5xx, timeouts) with exponential backoff — waits grow between tries so you don't hammer a struggling service.

⚡Circuit breaker

After too many failures, "trip" and fail fast for a cool-down period instead of piling requests onto a service that's clearly down — then test the waters again.

⏱️Timeout

A per-attempt (and total) time cap so one slow call can't hang a request forever.

🚦Rate limiter

Bound how many requests you fire concurrently, protecting both you and the downstream service.

☎️ Analogy

Resilience is polite phone etiquette for a line that keeps dropping. Retry = call back a couple of times if it cuts out — but wait a little longer each time (backoff), not redial furiously. Circuit breaker = if the number's clearly dead, stop dialing for a while instead of jamming the exchange. Timeout = hang up if nobody answers in 10 seconds. Together they keep one flaky dependency from dragging down your whole system.

C# · Custom pipeline when you need control
.AddResilienceHandler("custom", builder =>
{
    builder
        .AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 3,
            BackoffType = DelayBackoffType.Exponential,
            Delay = TimeSpan.FromMilliseconds(500)
        })
        .AddTimeout(TimeSpan.FromSeconds(10));
});
⚠️ Only Retry Idempotent Operations Blindly

Retrying a GET is safe — it just reads. Retrying a POST that creates something risks duplicates (two orders!) if the first attempt actually succeeded but the response got lost. Retry non-idempotent writes only with care — an idempotency key, or by scoping retries to reads. The standard handler is tuned conservatively, but you own this judgment.

🔑 It Composes with the Handler Pipeline

Resilience plugs in as delegating handlers in the same pipeline as logging and auth (section 04). The factory wires them in order, so retries, timeouts, and your own middleware all wrap each outgoing request cleanly — configured once at registration, invisible at the call site.

10🛠️ Hands-On: Call a Public API, Map the Response

Today's Do: call a real public API and map the JSON to a model. A free, no-key option like https://jsonplaceholder.typicode.com is perfect. The path:

  1. 🏭 In Program.cs, register a typed client with AddHttpClient<PostsApiClient>(...), setting BaseAddress and a Timeout.
  2. 🧱 Define a model matching the JSON — start with a record Post(int UserId, int Id, string Title, string Body). Note that case-insensitive matching means you don't need attributes here.
  3. 📦 Write the typed client class with an async method: GetPostAsync(int id, CancellationToken ct) using GetFromJsonAsync<Post>.
  4. 🔌 Expose an endpoint that injects the typed client and returns the mapped model — return NotFound when the API 404s.
  5. 🏷️ Try an API with awkward names (or pretend one does): rename a field with [JsonPropertyName], or set a PropertyNamingPolicy, and confirm mapping still works.
  6. 🧯 Add error handling: wrap the call, distinguish a 404 (→ null), a timeout (TaskCanceledException with no caller cancellation), and a network failure (HttpRequestException).
  7. 🛡️ Add .AddStandardResilienceHandler() (install Microsoft.Extensions.Http.Resilience) and note that transient failures now retry automatically.
  8. 🔬 Prove correctness: hit your endpoint, confirm the JSON became a real object, and — bonus — log the mapped title with structured logging (Day 6). Try requesting a nonexistent id to see your NotFound path.
C# · A complete, runnable slice
// ── Model ──
public record Post(int UserId, int Id, string Title, string Body);

// ── Typed client ──
public class PostsApiClient(HttpClient http, ILogger<PostsApiClient> logger)
{
    public async Task<Post?> GetPostAsync(int id, CancellationToken ct = default)
    {
        var res = await http.GetAsync($"posts/{id}", ct);
        if (res.StatusCode == HttpStatusCode.NotFound) return null;
        res.EnsureSuccessStatusCode();

        var post = await res.Content.ReadFromJsonAsync<Post>(ct);
        logger.LogInformation("Fetched post {PostId}: {Title}", id, post?.Title);
        return post;
    }
}

// ── Program.cs ──
builder.Services.AddHttpClient<PostsApiClient>(c =>
    c.BaseAddress = new Uri("https://jsonplaceholder.typicode.com/"))
    .AddStandardResilienceHandler();

app.MapGet("/posts/{id:int}", async (int id, PostsApiClient api, CancellationToken ct) =>
    await api.GetPostAsync(id, ct) is { } post
        ? Results.Ok(post)
        : Results.NotFound());

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is HttpClient used for?🌱 JUNIOR

It's the .NET class for sending HTTP requests to other servers and reading their responses — calling REST APIs, downloading data, posting to a webhook. You give it a verb (GET/POST/…), a URL, optional headers and body, and await the response, which carries a status code and usually a JSON body you then turn into objects.

Q2Why is "new HttpClient()" for every request a problem?🌱 JUNIOR

Each HttpClient owns a connection pool tied to network sockets. Creating and disposing one per request means sockets aren't released immediately — the OS holds them in a waiting state for a couple of minutes. Under load you create thousands, run out of available ports, and new connections start failing with confusing errors. This is "socket exhaustion." HttpClient is meant to be reused, not created per call.

Q3What is IHttpClientFactory and why use it?🌱 JUNIOR

It's the built-in, recommended way to create HttpClient instances. It manages the expensive underlying message handler (and its connection pool) for you — pooling handlers so connections are reused (no socket exhaustion) and periodically recycling them so DNS changes are picked up (no stale DNS). It also gives you one place to configure base address, headers, and timeouts, and hooks for resilience. You register it in Program.cs and get clients through DI.

Q4What is a typed client?🌱 JUNIOR

A typed client is your own class that receives a pre-configured HttpClient in its constructor and exposes clean methods like GetPostAsync(id). You register it with AddHttpClient<MyClient>(...) and inject it wherever you need it. It hides URLs and JSON parsing behind a tidy, strongly-typed API, so callers depend on your interface instead of raw HttpClient or magic strings — which is cleaner and easy to mock in tests.

Q5What is System.Text.Json and what does deserialization mean?🌱 JUNIOR

System.Text.Json is .NET's built-in JSON library. Deserialization is turning JSON text into C# objects — the serializer reads the JSON and fills in a matching class or record by matching property names. (Serialization is the reverse: object to JSON.) The helper GetFromJsonAsync<Post> combines an HTTP GET with deserialization into a Post in one call.

Q6How do you map a JSON field whose name doesn't match your C# property?🌱 JUNIOR

Put a [JsonPropertyName("json_key")] attribute on the property, giving the exact JSON name. For example [JsonPropertyName("temp_c")] public double TemperatureC maps the JSON temp_c onto TemperatureC. If the whole API uses one convention like snake_case, it's cleaner to set a PropertyNamingPolicy in JsonSerializerOptions once instead of attributing every property.

Q7Which methods read JSON from an HTTP response, and what's the simplest one?🌱 JUNIOR

The simplest is GetFromJsonAsync<T>(url) from System.Net.Http.Json — it does the GET and deserializes to T in one line. For posting, PostAsJsonAsync serializes an object and sends it. When you need the raw response (to check status or headers), use GetAsync then response.Content.ReadFromJsonAsync<T>().

Q8Why are all these HttpClient calls async?🌱 JUNIOR

Calling another server is I/O — your app spends the time waiting for the network, exactly the situation Day 10's async lesson addressed. Making the calls async and awaiting them releases the thread during that wait so it can serve other requests, letting the server scale. That's why every HttpClient method is async, takes a CancellationToken, and should be awaited all the way up the call chain.

🟠 Mid

Q9Explain the socket exhaustion AND stale DNS problems, and how the factory solves both.🟠 MID

They're opposite failures of the two naive approaches. Per-request clients → socket exhaustion: each disposed client leaves its socket in TIME_WAIT for minutes, so high request rates run out of ports. One eternal static client fixes sockets but → stale DNS: the underlying handler caches DNS for the app's lifetime, so if the target's IP changes (failover, autoscaling) you keep calling a dead address. IHttpClientFactory resolves both by separating the cheap, short-lived HttpClient wrapper from the expensive HttpMessageHandler: handlers are pooled and reused (connections shared → no exhaustion) but recycled on a timer (~2 min default) so DNS refreshes. You get reuse and freshness at once.

Q10Why must you not modify a factory-provided client's DefaultRequestHeaders in a request handler?🟠 MID

Because the factory hands out clients backed by a shared, pooled message handler; mutating a client's default headers (or base address/timeout) affects state that can be observed across requests. In concurrent scenarios you get race conditions and "header bleed" — one request's auth token leaking into another's, headers accumulating over time. Configure defaults once at registration. For per-request values, build an HttpRequestMessage, set the header on it, and send with SendAsync so nothing shared is touched.

Q11Why should you never inject a typed client into a singleton service?🟠 MID

A typed client's lifetime is tied to the HttpClient injected into it, which is meant to be short-lived (transient). A singleton captures its dependencies for the app's entire life — so it would hold onto one HttpClient, and therefore one message handler, forever. That defeats the factory's handler rotation, so DNS changes are never picked up: you've silently reintroduced the stale-DNS problem the factory exists to prevent. If a long-lived service needs to make calls, inject IHttpClientFactory (or the typed client via a scope/factory) and get a fresh client per operation.

Q12How do you tell a timeout apart from a caller-requested cancellation?🟠 MID

Both surface as TaskCanceledException (a subclass of OperationCanceledException), so the type alone doesn't tell you. The trick is to inspect the token you passed: catch (TaskCanceledException) when (!ct.IsCancellationRequested) means "the caller did not ask to cancel, so this was the HttpClient timeout firing." If ct.IsCancellationRequested is true, it was genuine cancellation (client disconnected) and you usually let it propagate. Newer .NET also exposes the timeout as an inner TimeoutException, but the token-check pattern is the portable idiom.

Q13A 404 comes back. Should that be an exception? How do you handle HTTP error statuses generally?🟠 MID

A 404 is a successful HTTP round trip that returned "not found" — it's a valid response, not a network failure, so throwing is often the wrong model. Usually you check response.StatusCode and translate 404 into a domain null/NotFound. General approach: inspect the status deliberately — 2xx read the body; 404 → null; 4xx client errors are usually your bug (bad request/auth) and shouldn't be blindly retried; 5xx are transient server problems worth retrying. The convenience helpers and EnsureSuccessStatusCode() throw HttpRequestException on any non-success, which is fine when you want any error to bubble, but for nuanced handling check the status first.

Q14System.Text.Json vs Newtonsoft.Json — how do you choose?🟠 MID

System.Text.Json is the modern built-in default: significantly faster, lower memory, no extra dependency, and integrated with the ASP.NET Core pipeline and the *Json* HttpClient helpers. Newtonsoft.Json (Json.NET) is older, extremely feature-rich, and more lenient by default. Choose System.Text.Json for new code; reach for Newtonsoft only when you need a specific feature it still handles more easily (certain complex custom serialization, some polymorphism scenarios, or legacy compatibility). Be aware System.Text.Json is stricter — case-sensitive when used directly (the web helpers relax that), and less forgiving of malformed input — which occasionally surprises teams migrating.

Q15What is exponential backoff and why is it better than immediate retries?🟠 MID

Exponential backoff means increasing the wait between retry attempts — e.g., 0.5s, then 1s, then 2s — rather than retrying immediately or at a fixed interval. It's better because a service that's failing is often overloaded; hammering it with instant retries makes things worse and can cause a "retry storm" that keeps it down. Growing delays give it room to recover. In practice you also add jitter (small random variation) so many clients don't retry in synchronized waves. The standard resilience handler does exponential backoff with jitter for you.

Q16What does AddStandardResilienceHandler give you, and which package provides it?🟠 MID

It's from Microsoft.Extensions.Http.Resilience (the .NET 8+ package built on Polly v8, which replaced the older Microsoft.Extensions.Http.Polly). Chained onto AddHttpClient, one call adds a sensible default pipeline of four patterns: retry with exponential backoff for transient failures, a circuit breaker that fails fast when a dependency is clearly down, a timeout per attempt and total, and a rate limiter bounding concurrency. It's conservatively tuned and configurable, giving most apps production-grade resilience for external calls with essentially one line.

Q17Explain the circuit breaker pattern and its states. When is it essential?🔴 SENIOR

A circuit breaker prevents a struggling dependency from dragging your whole system down. It has three states: Closed (normal — requests flow, failures are counted); when failures cross a threshold it trips to Open (requests fail fast immediately without even trying, for a cool-down period — protecting both you and the downstream service); after the cool-down it moves to Half-Open, allowing a trial request or two — success closes the circuit, failure re-opens it. It's essential in distributed systems where a slow dependency would otherwise cause cascading failure: threads pile up waiting on it, your thread pool starves (Day 10!), and the outage spreads upstream. Failing fast contains the blast radius and lets the dependency recover without being hammered.

Q18Retries and non-idempotent operations: what's the danger and how do you handle it?🔴 SENIOR

The danger: a retry can't tell "the request failed" from "the request succeeded but the response was lost." For a GET that's harmless — read it again. For a POST that creates a resource (place order, charge card), a blind retry can create a duplicate if the first attempt actually landed. Handling: prefer idempotency keys — the client sends a unique key with the write, the server deduplicates so repeats are no-ops; this makes the operation safely retriable. Otherwise, scope automatic retries to idempotent verbs (GET/PUT/DELETE are naturally idempotent; POST usually isn't), and for critical writes require explicit, careful handling rather than default retry policies. Design the API to be idempotent where you can — it's the clean solution.

Q19Walk through how you'd add authentication and logging to all outgoing calls of a client.🔴 SENIOR

Use DelegatingHandlers — middleware in the outgoing pipeline, the mirror image of ASP.NET Core's request middleware. Write a handler that overrides SendAsync: attach the auth token (fetched/refreshed from a token service, ideally cached), call base.SendAsync, and on the way back inspect the response. Register it with AddHttpClient<MyClient>().AddHttpMessageHandler<AuthHandler>(). Handlers compose in order, so you can stack auth, logging, and resilience cleanly, each concern isolated. Caveats specific to the factory: message handlers live in their own DI scope separate from the request scope, so you can't rely on scoped services like the current HttpContext inside a handler — inject IHttpContextAccessor or pass what you need explicitly. For auth token refresh, guard against concurrent refreshes. Logging often you get for free (the factory logs requests), but a handler lets you add correlation IDs and timing.

Q20The factory rotates handlers every ~2 minutes. What happens to in-flight requests, and how would you tune this?🔴 SENIOR

Rotation is graceful: when a handler's lifetime expires it's marked for retirement but not disposed while any HttpClient is still using it or requests are in flight — the factory disposes it only once it's truly idle. New clients get a fresh handler; existing work finishes on the old one. So there's no torn request. Tuning: SetHandlerLifetime(TimeSpan) on the registration adjusts the interval. Lower it if you expect frequent DNS changes (aggressive failover, blue-green deploys) and want faster pickup; raise it to reduce the small cost of re-establishing connection pools if your endpoints are stable. Setting it to Timeout.InfiniteTimeSpan disables rotation entirely — occasionally right for a load-balancer VIP with a stable IP, but you lose DNS refresh, so it's a deliberate trade-off, not a default.

Q21How do you unit-test code that calls an external API via a typed client?🔴 SENIOR

Don't hit the real network in unit tests. Two clean levels. (1) Test consumers of the typed client by mocking the typed client's interface — extract an IPostsApiClient and mock it; the code under test never sees HttpClient. (2) Test the typed client itself by giving it an HttpClient wired to a fake HttpMessageHandler — either a small custom handler that returns canned HttpResponseMessages (asserting the request URL/headers/body), or a library like WireMock.NET / a mocked handler, so you verify serialization, URL construction, status handling, and error paths deterministically. For resilience, WireMock or a fault-injecting handler lets you simulate 500s and timeouts to prove retries and circuit breaking behave. Integration tests against a sandbox API are a separate, slower tier.

Q22An external API is your bottleneck: slow, rate-limited, called on every request. What's your strategy?🔴 SENIOR

Layered approach, cheapest first. Cache — if the data tolerates staleness, cache responses (in-memory or distributed) keyed by request, with a TTL matched to how fresh it must be; this removes most calls entirely. Coalesce duplicate concurrent requests for the same key so a cache miss triggers one upstream call, not fifty. Respect the rate limit proactively with a client-side limiter (the resilience rate limiter) rather than getting 429'd. Batch if the API supports fetching many items per call. Degrade gracefully — a fallback (last-known-good cached value, or a sensible default) when the dependency is unavailable, so your app stays up. Decouple — if the call isn't needed synchronously, move it to a background job/queue and return immediately. And measure: add metrics on latency, cache hit rate, and 429s to know which lever actually helps. The meta-point is reducing dependence on the synchronous hot-path call, not just calling it faster.

Q23A response is a large JSON array. How do you deserialize it efficiently?🔴 SENIOR

Avoid buffering the whole payload into a string first — that allocates a huge string and then a huge object graph, spiking memory. Instead stream: get the response with HttpCompletionOption.ResponseHeadersRead so you start processing before the full body arrives, take the content stream, and deserialize from it. For a big array, JsonSerializer.DeserializeAsyncEnumerable<T> yields items one at a time as they're parsed, so you can process (or project into DTOs) without ever holding the entire collection in memory — ideal for pipelines that transform or forward records. Also project to a lean model (only the fields you need) so you don't materialize unused data. If the API supports it, prefer pagination over one giant response in the first place. The principle mirrors EF's: don't materialize more than you must, and stream when the data is large.

Q24Your app calls twelve different external services. How do you structure and standardize that cleanly?🔴 SENIOR

One typed client per external service, each behind its own interface, each registered with its own base address, timeout, headers, and resilience policy tuned to that dependency's characteristics (a flaky partner API gets aggressive retries; a fast internal one gets tight timeouts). Standardize the cross-cutting concerns via shared DelegatingHandlers (correlation ID propagation, structured logging, auth) applied consistently, and consider a small registration helper/extension method so every client is wired the same way — reducing copy-paste drift. Keep response models per service (don't share DTOs across unrelated APIs — they evolve independently). Put configuration (URLs, timeouts) in appsettings/options bound per client, not hard-coded, so environments differ cleanly. Add per-client metrics and health checks so you can see which dependency is degrading. The goals: each integration is isolated and independently tunable, the boilerplate is centralized, and a new service is a small, uniform addition rather than a fresh reinvention.
