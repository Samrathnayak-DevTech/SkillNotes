📋 Logging: Your App's Black Box Recorder

When your app misbehaves at 3 a.m. on a server you can't touch, logs are the only witness. Today you'll learn how .NET's built-in logging works, how to write logs that machines can search, and how to plug in Serilog — the most popular logging library in the .NET world.

💉 ILogger<T> 🚦 Log Levels 🗂️ Categories 🧱 Structured Logging 🔭 Scopes 🌲 Serilog

01📢 Why Logging Matters

Imagine an airplane's black box recorder. It quietly records everything that happens during a flight — altitude, speed, cockpit conversations. Nobody looks at it when everything goes well. But when something goes wrong, it becomes the single most important piece of evidence.

Logging is exactly that for software: your application continuously writes short notes about what it's doing — "user 42 logged in", "order 1001 saved", "payment gateway timed out" — so that later, a human (or a monitoring tool) can reconstruct what happened.

🍳 Analogy

Think of a restaurant kitchen keeping a daily journal: "6:02 pm — table 4 ordered pasta. 6:14 pm — oven #2 overheated, switched to oven #3. 6:30 pm — pasta served." If a customer later complains their food was late, the manager doesn't have to guess — the journal tells the story. Logs are your app's kitchen journal.

Why not just use Console.WriteLine?

Beginners often print debug messages with Console.WriteLine("here!"). That works on your laptop but falls apart in real applications, because a proper logging system gives you things plain printing never can:

🚦Severity levels

Mark each message as Trace, Debug, Information, Warning, Error, or Critical — then show only what matters.

🎚️Filtering without code changes

Turn detail up or down per component via configuration — no recompiling, no redeploying.

📦Multiple destinations

Send the same log event to the console, a file, and a cloud dashboard at once — each destination is a "provider" or "sink."

🔍Searchable data

Structured logs store values as named properties, so you can query "show all errors for OrderId 1001" instead of grepping text.

🏷️Automatic context

Each entry carries a timestamp, category (which class wrote it), and can carry a request ID, user ID, and more.

⚡Performance control

Messages below the minimum level are skipped cheaply — the framework doesn't even build the message string.

🗺️ The big picture

Everything in today's lesson hangs off one central idea: .NET ships a logging abstraction (Microsoft.Extensions.Logging) that your code talks to, while pluggable providers decide where the messages actually go.

🔑 Key Idea

Your code never talks to a file or the console directly. It talks to the ILogger abstraction, and configuration decides where messages end up. Swap destinations without touching a single service class.

02🔀 The Logging Pipeline

Before writing any code, it helps to see the journey a single log message takes — from the line you write to the place it's stored. ASP.NET Core sets most of this up automatically when you call WebApplication.CreateBuilder(args): it registers the logging services and adds default providers (Console, Debug, and EventSource).

✍️
1. Your code writes a log_logger.LogInformation("Order {OrderId} placed", id) — a message template plus values.
🏷️
2. A log event is createdThe framework attaches the level, the category (your class's full name), a timestamp, and an optional event ID and exception.
🎚️
3. Filters decide: keep or drop?Configuration rules compare the event's level and category against minimum levels. Too low? The event is discarded cheaply — this is the critical gate.
📮
4. Providers receive the eventEvery registered provider (Console, Debug, Serilog…) gets the surviving event.
🗄️
5. Output is writtenEach provider formats and delivers the event to its destination: terminal, file, database, or a cloud dashboard like Seq or Application Insights.

The three actors

PieceWhat it isKitchen analogy 🍳
ILogger / ILogger<T>The interface your code calls to write messages.The chef shouting an update.
ILoggerProviderA plugin that knows how to deliver messages somewhere (console, file, cloud).The person writing shouts into a specific journal.
ILoggerFactoryCreates loggers and wires them to all registered providers.The manager who hands each chef a megaphone connected to every journal.
✅ Tip

You almost never touch ILoggerFactory or providers directly in day-to-day code. You just ask for ILogger<YourClass> in a constructor and start logging — dependency injection (Day 3!) handles the rest.

03💉 ILogger<T> & Dependency Injection

ILogger<T> is the generic logging interface where T is the class doing the logging. Why the generic parameter? Because the full name of T becomes the log category — a label stamped on every message that tells you exactly where it came from (more in section 05).

Getting a logger is as simple as adding a constructor parameter. The DI container already knows how to supply it — no registration needed on your side:

C# · OrderService.cs
public class OrderService
{
    private readonly ILogger<OrderService> _logger;

    // The DI container injects a logger whose category is
    // "MyShop.Services.OrderService" (the full type name).
    public OrderService(ILogger<OrderService> logger)
    {
        _logger = logger;
    }

    public void PlaceOrder(Order order)
    {
        _logger.LogInformation(
            "Placing order {OrderId} for customer {CustomerId}",
            order.Id, order.CustomerId);

        // ... business logic ...

        _logger.LogInformation("Order {OrderId} placed successfully", order.Id);
    }
}

Modern C# lets you shorten this with a primary constructor (C# 12+):

C# · Primary constructor style
public class OrderService(ILogger<OrderService> logger)
{
    public void PlaceOrder(Order order)
    {
        logger.LogInformation("Placing order {OrderId}", order.Id);
    }
}

Logging methods you'll use

The interface itself has one core method, Log(...), but you'll almost always use the friendlier extension methods — one per level:

  • LogTrace(...) 🔬 — ultra-fine detail
  • LogDebug(...) 🐛 — developer diagnostics
  • LogInformation(...) ℹ️ — normal app flow
  • LogWarning(...) ⚠️ — something odd, app continues
  • LogError(...) ❌ — an operation failed
  • LogCritical(...) 🔥 — the app itself is in danger

Every one of these accepts an optional Exception as the first argument — always pass it there rather than gluing it into the message, so providers can capture the full stack trace as data:

C# · Logging an exception
try
{
    await _paymentGateway.ChargeAsync(order);
}
catch (PaymentException ex)
{
    // ✅ Exception goes FIRST — full stack trace is preserved as data
    _logger.LogError(ex,
        "Payment failed for order {OrderId} with amount {Amount}",
        order.Id, order.Total);
    throw;
}
⚠️ Common Mistake

Don't write LogError("Failed: " + ex.Message). You lose the exception type, stack trace, and inner exceptions. Pass the exception object itself as the first parameter — the logging system knows what to do with it.

🔑 Key Idea

ILogger<T> vs plain ILogger: the generic version is just a convenience that auto-fills the category from T. Non-generic ILogger instances are created from ILoggerFactory.CreateLogger("SomeCategory") when you want a custom category name.

04🚦 Log Levels

Every log message carries a severity level — a number from 0 to 5 (plus a special "None"). Levels answer two questions at once: how important is this message? and should we bother recording it right now?

Setting a minimum level means "record this level and everything more severe." If the minimum is Warning, then Warning, Error, and Critical get through; Trace, Debug, and Information are dropped.

🚨 Analogy

Think of a hospital triage desk. Trace/Debug = "patient sneezed" (only interesting to a researcher). Information = routine check-in notes. Warning = elevated temperature — watch closely. Error = broken arm — needs treatment now. Critical = cardiac arrest — all hands on deck. The triage nurse (your minimum level) decides which cases get written up.

C# · One method, all levels in action
public async Task ProcessOrderAsync(Order order)
{
    _logger.LogDebug("Validating order {OrderId}", order.Id);

    if (order.Items.Count == 0)
    {
        _logger.LogWarning("Order {OrderId} arrived with no items — rejecting", order.Id);
        return;
    }

    _logger.LogInformation("Processing order {OrderId} with {ItemCount} items",
        order.Id, order.Items.Count);

    try
    {
        await _repository.SaveAsync(order);
    }
    catch (DbException ex)
    {
        _logger.LogError(ex, "Could not persist order {OrderId}", order.Id);
        throw;
    }
}
✅ Rule of Thumb

If a support engineer at 3 a.m. would need the message to understand a problem → Warning or above. If it just narrates normal life → Information. If only a developer stepping through code cares → Debug/Trace.

⚠️ Watch Out

Trace may contain sensitive data (raw payloads, tokens) and should never be enabled in production by default. Also never log passwords, credit card numbers, or personal data at any level — logs are often widely readable within a company.

05🗂️ Log Categories & Filtering

Every logger has a category — a string label attached to all its messages. When you inject ILogger<OrderService>, the category is the class's fully qualified name, e.g. MyShop.Services.OrderService. Framework code logs with its own categories, like Microsoft.AspNetCore.Hosting or Microsoft.EntityFrameworkCore.Database.Command.

📻 Analogy

Categories are like radio stations. Every part of your app broadcasts on its own station named after its namespace. Filtering is your mixing desk: turn up the volume on MyShop.Services (Debug) while turning down the chatty Microsoft.* stations (Warning only).

Filtering by prefix in appsettings.json

Filters match categories by longest prefix wins. The rule "Microsoft.AspNetCore": "Warning" applies to every category starting with that text. "Default" covers anything with no better match.

JSON · appsettings.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning",
      "Microsoft.EntityFrameworkCore.Database.Command": "Information",
      "MyShop.Services.OrderService": "Debug"
    }
  }
}

Per-environment settings

ASP.NET Core layers configuration files: appsettings.Development.json overrides appsettings.json when you run locally. Typical practice — verbose locally, calm in production:

JSON · appsettings.Development.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Debug",
      "Microsoft.AspNetCore": "Information"
    }
  }
}
🔑 Key Idea

Categories + levels = a two-dimensional dial. Level answers "how important?", category answers "from where?". Together they let you silence framework noise while cranking up detail on the one service you're debugging — without changing any code.

✅ Tip

Config changes to appsettings.json are picked up while the app runs (the default configuration sources reload on change). You can raise a production service from Information to Debug for ten minutes to investigate, then dial it back — no restart.

06🧱 Structured Logging

This is the most important concept of the day. Structured logging means a log entry isn't just a sentence — it's a small packet of data: a message template plus named property values. Instead of storing the string "Order 1001 placed by customer 42", the system stores:

  • Template: "Order {OrderId} placed by customer {CustomerId}"
  • Property OrderId = 1001
  • Property CustomerId = 42

Why does this matter? Because once values are stored as named properties, a log tool can answer questions like "show every event where OrderId = 1001" or "average Elapsed time for slow queries" — instantly, without fragile text searching.

🗃️ Analogy

Plain-text logging is a shoebox of handwritten receipts — to find anything you must read every slip. Structured logging is a spreadsheet — every receipt has Date, Amount, and Customer columns you can sort and filter in one click.

The golden rule: templates, not interpolation

C# · Do vs Don't
// ❌ DON'T: string interpolation — the values melt into plain text
_logger.LogInformation($"Order {order.Id} placed by {order.CustomerId}");

// ❌ DON'T: string concatenation — same problem, plus allocation cost
_logger.LogInformation("Order " + order.Id + " placed");

// ✅ DO: message template with named placeholders + arguments
_logger.LogInformation("Order {OrderId} placed by customer {CustomerId}",
    order.Id, order.CustomerId);

Three reasons the template form wins:

  1. Searchability 🔍 — OrderId becomes a queryable field in tools like Seq, Elasticsearch, or Application Insights.
  2. Performance ⚡ — with interpolation, C# builds the final string before calling the logger, even if the message will be filtered out. With templates, formatting only happens if the message survives filtering.
  3. Grouping 🧮 — all "Order {OrderId} placed…" events share one template, so dashboards can count "how often does this event happen?" regardless of the values inside.
⚠️ Positional, Not Named

Placeholders are filled by position, not by name. In "Order {OrderId} by {CustomerId}" the first argument fills {OrderId} and the second fills {CustomerId} — swap the arguments and your logs silently lie. The names exist purely to label the stored properties, so keep names and argument order aligned.

Naming and formatting placeholders

  • Use PascalCase names: {OrderId}, {ElapsedMs}, {UserEmail}.
  • Format specifiers work like string.Format: {Total:C} (currency), {Elapsed:0.00}, {Date:yyyy-MM-dd}.
  • Keep names consistent across the codebase — if one service logs {OrderId} and another logs {Order_ID}, your queries have to chase both.
  • With Serilog, prefix a placeholder with @ — {@Order} — to capture a whole object's properties (destructuring) rather than just its ToString().

What the output looks like

The same event, rendered two ways. A console shows friendly text; a JSON sink stores the searchable version:

Text vs JSON output
# Human-friendly console rendering:
[14:23:07 INF] Order 1001 placed by customer 42

# Structured JSON stored by a sink (simplified):
{
  "Timestamp": "2026-07-06T14:23:07.412Z",
  "Level": "Information",
  "MessageTemplate": "Order {OrderId} placed by customer {CustomerId}",
  "OrderId": 1001,
  "CustomerId": 42,
  "SourceContext": "MyShop.Services.OrderService"
}
🔑 Key Idea

The message template is the identity of an event; the properties are its data. Once you internalize "logs are data, not prose," everything else — levels, categories, Serilog sinks — clicks into place.

07🔭 Scopes & the LoggerMessage Pattern

Two power tools round out the built-in system: scopes attach shared context to a whole block of logs, and the LoggerMessage source generator makes hot-path logging both faster and tidier.

Scopes: shared context for a block of work 🔭

Suppose ten log lines all belong to the same order. Instead of repeating {OrderId} in every call, open a scope: every log written inside it automatically carries those properties.

C# · BeginScope
using (_logger.BeginScope(new Dictionary<string, object>
{
    ["OrderId"] = order.Id,
    ["CustomerId"] = order.CustomerId
}))
{
    _logger.LogInformation("Validating items");      // carries OrderId + CustomerId
    _logger.LogInformation("Charging payment");       // carries OrderId + CustomerId
    _logger.LogInformation("Sending confirmation");   // carries OrderId + CustomerId
}

ASP.NET Core already uses scopes for you: with "IncludeScopes": true on the console provider (or any structured sink), each request's logs carry the request path and a RequestId, letting you group every line belonging to one HTTP request.

✅ Tip

Scopes shine for "correlation": trace one order, one user session, or one background job across dozens of log lines by filtering on a single property that the scope stamped everywhere.

LoggerMessage: compile-time logging ⚡

For messages written thousands of times per second, .NET offers a source generator: you declare a partial method with an attribute, and the compiler generates highly optimized logging code (cached delegates, zero boxing, a built-in level check).

C# · [LoggerMessage] source generator
public static partial class OrderLogs
{
    [LoggerMessage(EventId = 1001, Level = LogLevel.Information,
        Message = "Order {OrderId} placed by customer {CustomerId}")]
    public static partial void OrderPlaced(
        this ILogger logger, int orderId, int customerId);

    [LoggerMessage(EventId = 1002, Level = LogLevel.Error,
        Message = "Payment failed for order {OrderId}")]
    public static partial void PaymentFailed(
        this ILogger logger, Exception ex, int orderId);
}

// Usage — reads like a domain event, runs faster than LogInformation:
_logger.OrderPlaced(order.Id, order.CustomerId);

Boxing / allocationsavoided Level checkbuilt-in Event IDscentralized

🔑 When To Bother

Plain LogInformation(...) is perfectly fine for most code. Reach for [LoggerMessage] in hot paths (per-request middleware, tight loops) or when your team wants a single catalog of named, numbered events.

08🌲 Adding Serilog

The built-in providers cover the console nicely, but real projects usually want more: rolling log files, JSON output, a searchable dashboard, extra context on every event. Serilog is the most widely used logging library in .NET — built around structured logging from day one — and it plugs straight into the ILogger<T> world you already know. Your services don't change at all.

Serilog vocabulary

🚰Sink

A destination for log events: Console, File, Seq, Elasticsearch, Application Insights… hundreds exist as NuGet packages.

🧂Enricher

Adds extra properties to every event automatically — machine name, thread ID, or anything pushed via LogContext.

🎚️Minimum level & overrides

Serilog's own filtering: a global minimum plus per-namespace overrides, mirroring the categories idea.

🧾Output template / formatter

Controls how each event is rendered — a friendly text line for the console, compact JSON for machines.

Step 1 — install packages 📦

Terminal
# The one package most apps need — it pulls in the console and
# file sinks plus configuration support:
dotnet add package Serilog.AspNetCore

Step 2 — wire it up in Program.cs 🔌

The recommended pattern is two-stage initialization: a tiny "bootstrap" logger catches crashes that happen before the host and configuration are ready (these otherwise vanish silently), and the real logger — configured from appsettings.json — replaces it once the host is built.

C# · Program.cs
using Serilog;

// Stage 1: bootstrap logger — catches startup crashes
Log.Logger = new LoggerConfiguration()
    .WriteTo.Console()
    .CreateBootstrapLogger();

try
{
    Log.Information("Starting web application");

    var builder = WebApplication.CreateBuilder(args);

    // Stage 2: the real logger, built from appsettings.json + DI services.
    // AddSerilog replaces the default providers with Serilog.
    builder.Services.AddSerilog((services, lc) => lc
        .ReadFrom.Configuration(builder.Configuration)
        .ReadFrom.Services(services)
        .Enrich.FromLogContext());

    var app = builder.Build();

    // One tidy "HTTP GET /orders responded 200 in 12.3 ms" line per request
    app.UseSerilogRequestLogging();

    app.MapControllers();
    app.Run();
}
catch (Exception ex)
{
    Log.Fatal(ex, "Application terminated unexpectedly");
}
finally
{
    // Flush buffered events so nothing is lost on shutdown
    Log.CloseAndFlush();
}
⚠️ Version Note

builder.Services.AddSerilog(...) is the recommended registration in current Serilog.AspNetCore versions; the older builder.Host.UseSerilog(...) still works. Match the package's major version to your target framework (e.g., .NET 8 → Serilog.AspNetCore 8.x). Also note the final logger fully replaces the bootstrap logger — if you configure sinks in JSON, the bootstrap stage still needs its own WriteTo.Console() in code.

Step 3 — configure sinks in appsettings.json 🗄️

Because we used ReadFrom.Configuration(...), sinks and levels live in config — changeable per environment without recompiling. Note Serilog uses a "Serilog" section (replacing the "Logging" section) and its own level names: Verbose instead of Trace, Fatal instead of Critical.

JSON · appsettings.json
{
  "Serilog": {
    "MinimumLevel": {
      "Default": "Information",
      "Override": {
        "Microsoft.AspNetCore": "Warning",
        "MyShop.Services": "Debug"
      }
    },
    "WriteTo": [
      { "Name": "Console" },
      {
        "Name": "File",
        "Args": {
          "path": "logs/myshop-.txt",
          "rollingInterval": "Day",
          "retainedFileCountLimit": 14
        }
      }
    ],
    "Enrich": [ "FromLogContext", "WithMachineName" ]
  }
}

That's it. Every existing _logger.LogInformation(...) call in your services now flows through Serilog to the console and a daily rolling file like logs/myshop-20260706.txt that keeps the last 14 days.

✅ Bonus: Destructuring

Serilog's @ operator captures whole objects: _logger.LogInformation("Created {@Order}", order) stores the order's properties as structured data instead of calling ToString(). Use it deliberately — destructure small DTOs, not giant entity graphs.

🔑 Why Serilog "Just Works"

Serilog registers itself as a logging provider behind the ILogger abstraction. Your services keep depending on ILogger<T>; only Program.cs and configuration know Serilog exists. That is the payoff of coding against abstractions (Day 3's DI lesson).

09🛠️ Hands-On: Add Structured Logs to Your Services

Today's Do: take the services you built earlier this week and add structured logs with parameters. Work through this checklist:

  1. 📦 Install Serilog.AspNetCore and set up two-stage initialization in Program.cs (copy the section 08 snippet as your starting point).
  2. 💉 Inject ILogger<YourService> into at least two services via constructor injection.
  3. 🧱 Add Information logs at the start and successful end of each important operation — always with named placeholders, never interpolation.
  4. ⚠️ Add a Warning for a recoverable oddity (empty input, retry, fallback used).
  5. ❌ Wrap a risky call in try/catch and log an Error with the exception object first, then rethrow.
  6. 🎚️ In appsettings.Development.json, set your own namespace to Debug and Microsoft.AspNetCore to Warning — watch the console noise drop.
  7. 🗄️ Add the rolling File sink, run the app, hit a few endpoints, and open the log file. Find one specific order/entity by its ID.
C# · Target quality for your logs
public class BookingService(ILogger<BookingService> logger, IBookingRepository repo)
{
    public async Task<Booking> CreateAsync(BookingRequest request)
    {
        logger.LogInformation(
            "Creating booking for {GuestEmail} from {CheckIn:yyyy-MM-dd} to {CheckOut:yyyy-MM-dd}",
            request.GuestEmail, request.CheckIn, request.CheckOut);

        if (request.CheckIn >= request.CheckOut)
        {
            logger.LogWarning(
                "Rejected booking: check-in {CheckIn} is not before check-out {CheckOut}",
                request.CheckIn, request.CheckOut);
            throw new ValidationException("Invalid date range");
        }

        try
        {
            var booking = await repo.AddAsync(request);
            logger.LogInformation("Booking {BookingId} created for {GuestEmail}",
                booking.Id, request.GuestEmail);
            return booking;
        }
        catch (Exception ex)
        {
            logger.LogError(ex, "Failed to create booking for {GuestEmail}",
                request.GuestEmail);
            throw;
        }
    }
}
✅ Self-Check

Open your log file and ask: Could a teammate who has never seen my code follow one booking from request to database using only these lines? If yes, your logging is doing its job.

11🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is logging and why do applications need it?🌱 JUNIOR

Logging is the practice of having an application write short records of what it's doing — like a diary or a flight recorder. It matters because once an app runs on a server, you can't watch it or pause it like on your laptop. When something goes wrong, logs are often the only evidence of what happened, in what order, and with what data.

Q2What is ILogger<T> and how do you get one in ASP.NET Core?🌱 JUNIOR

ILogger<T> is the standard interface for writing logs, where T is the class doing the logging. You get one through dependency injection: just declare it as a constructor parameter (e.g., ILogger<OrderService>) and the framework hands you a ready-to-use logger. The type T is used to label every message with your class's full name, so you always know where a message came from.

Q3Name the log levels in order of severity.🌱 JUNIOR

From least to most severe: Trace, Debug, Information, Warning, Error, Critical (values 0 through 5), plus None which turns logging off. A handy memory trick: severity rises as the situation gets worse — from "just narrating" (Trace/Debug) to "normal news" (Information) to "something's off" (Warning) to "something broke" (Error) to "everything is on fire" (Critical).

Q4What does setting a "minimum level" of Warning actually do?🌱 JUNIOR

It means "record Warning and anything more severe." So Warning, Error, and Critical messages get through, while Trace, Debug, and Information are discarded. It's a threshold, not an exact match — like saying "only tell me about problems of level 3 or worse."

Q5Why is Console.WriteLine a bad substitute for a logging framework?🌱 JUNIOR

Because it can only print plain text to one place, always. It has no severity levels, can't be filtered or turned off via configuration, can't write to files or monitoring dashboards, carries no timestamp or source information, and stores no searchable data. A logging framework gives you all of that with roughly the same one-line effort.

Q6What is a log category?🌱 JUNIOR

A category is a text label attached to every message a logger writes, saying which component produced it. When you inject ILogger<OrderService>, the category is the full type name, like MyShop.Services.OrderService. Categories let you filter by source — for example, showing only warnings from Microsoft's framework code while showing debug detail from your own services.

Q7How do you log an exception correctly?🌱 JUNIOR

Pass the exception object itself as the first argument: _logger.LogError(ex, "Failed to save order {OrderId}", id). Don't stuff ex.Message into the text — you'd lose the exception type, the stack trace, and any inner exceptions. Passed as an object, the full details are preserved and shown by every provider.

🟠 Mid

Q8What is structured logging and how is it different from plain-text logging?🟠 MID

Plain-text logging stores a finished sentence: "Order 1001 placed by customer 42." Structured logging stores the event as data: a message template ("Order {OrderId} placed by customer {CustomerId}") plus named property values (OrderId=1001, CustomerId=42). The difference matters when searching: with structured logs, a tool can instantly answer "show all events where OrderId equals 1001" or compute averages over a numeric property — things that are painful or impossible with free text.

Q9Why should you avoid string interpolation ($"...") in log calls?🟠 MID

Two reasons. First, interpolation bakes the values into plain text before the logger ever sees them, so you lose the named, searchable properties that make structured logging valuable. Second, performance: the string is fully built even when the message will be filtered out (say, a Debug message in production). With a template plus arguments, the framework checks the level first and skips formatting entirely for filtered messages.

Q10How does category-based filtering resolve when multiple rules could apply?🟠 MID

By longest matching prefix. Given rules for "Default", "Microsoft", and "Microsoft.AspNetCore", a message from category Microsoft.AspNetCore.Routing uses the "Microsoft.AspNetCore" rule because it's the most specific prefix that matches. "Default" applies only when no named rule matches. This lets you set a broad baseline and then carve out exceptions for specific namespaces.

Q11What is a logging scope (BeginScope) and when would you use one?🟠 MID

A scope attaches a set of properties to every log message written inside a block of code. For example, opening a scope with OrderId=1001 means ten subsequent log lines all carry that property without repeating it in each call. It's ideal for correlation: tracing one HTTP request, one user session, or one background job across many log lines. ASP.NET Core itself uses scopes to attach a RequestId to all logs of a request.

Q12What is Serilog and why would you add it when .NET already has logging?🟠 MID

Serilog is a third-party logging library built around structured logging, with a huge ecosystem of "sinks" — destinations like rolling files, Seq, Elasticsearch, and cloud monitoring services. The built-in system defines the abstraction and basic providers (mainly console); Serilog plugs in as a provider behind that abstraction and adds rich outputs, JSON formatting, enrichers, and flexible configuration. Your services keep using ILogger<T> unchanged — only Program.cs and configuration know Serilog exists.

Q13What is a Serilog sink? Name a few.🟠 MID

A sink is a destination that receives and stores log events. Each is a NuGet package, and you can write to several at once. Common ones: Console (terminal output), File (rolling text or JSON files), Seq (a searchable log dashboard), Elasticsearch/OpenSearch, and Application Insights (Azure monitoring).

Q14In a message template, do the placeholder names have to match the argument names?🟠 MID

No — arguments fill placeholders by position, left to right. The names inside the braces exist to label the stored properties, not to bind arguments. That's a real bug source: if the template says "{OrderId} … {CustomerId}" but you pass the customer ID first, the log will silently record swapped values. Keep template order and argument order in sync, and enable the analyzer warnings that catch mismatched counts.

Q15What kinds of data should never appear in logs?🟠 MID

Secrets and personal data: passwords, API keys, tokens, session cookies, full credit card numbers, national ID numbers, and health details. Logs get copied to files, dashboards, and third-party services, and are readable by many people — treat them as semi-public within your organization. Mask or omit sensitive fields (log the last four digits, or just an internal ID), and be extra careful with Trace-level logging of raw request bodies.

Q16How can you change logging detail on a running production app without redeploying?🟠 MID

Log levels live in configuration (appsettings.json or environment-specific files), and the default configuration providers reload when the file changes. So you can edit the file — for example raising your service's namespace from Information to Debug — and the running app picks it up. Investigate, then dial it back. This is exactly why filtering belongs in config rather than in code.

🔴 Senior

Q17Explain Serilog's two-stage initialization. What problem does the bootstrap logger solve?🔴 SENIOR

The full logger reads its setup from appsettings.json and DI services — but those only exist after the host is built. If the app crashes while building the host (bad config, missing connection string), a config-driven logger doesn't exist yet, so the crash vanishes silently. The fix: create a simple, code-configured bootstrap logger on the very first line of Program.cs (CreateBootstrapLogger()), wrap startup in try/catch so fatal startup errors are logged, and let the fully configured logger replace the bootstrap one once the host is ready. A finally block calls Log.CloseAndFlush() so buffered events aren't lost on shutdown.

Q18What is the LoggerMessage source generator and when is it worth using?🔴 SENIOR

It's a compile-time feature: you declare a partial method decorated with [LoggerMessage(EventId, Level, Message)], and the compiler generates optimized logging code — cached delegates, no boxing of value-type arguments, and a built-in "is this level enabled?" check. It's worth using on hot paths (middleware, tight loops, high-throughput services) where ordinary extension methods allocate on every call, and in teams that want a central, numbered catalog of log events. For ordinary business code, plain LogInformation is fine.

Q19How would you design logging for correlation across services in a distributed system?🔴 SENIOR

Ensure every log line carries a shared correlation identifier so one user action can be followed across services. Practically: rely on W3C trace context (the traceparent header), which ASP.NET Core and HttpClient propagate automatically; surface the TraceId/RequestId in logs via scopes or an enricher; and ship all services' logs to one central, structured store (Seq, Elasticsearch, a cloud APM) where you can filter by that ID. For deeper visibility, pair logging with distributed tracing (OpenTelemetry) — logs tell you what happened, traces show where time went.

Q20What are the performance considerations of heavy logging, and how do you mitigate them?🔴 SENIOR

Costs come from three places: building messages (string formatting, boxing), I/O to sinks (disk, network), and log volume (storage, ingestion fees). Mitigations: use templates instead of interpolation so filtered messages cost almost nothing; guard expensive argument computation with IsEnabled(LogLevel.Debug); use the LoggerMessage generator on hot paths; write to sinks asynchronously/buffered where supported; keep production minimums at Information or above with per-namespace overrides; and prefer one rich event over five thin ones (e.g., Serilog's request logging emits a single summarized line per HTTP request).

Q21Console noise: production logs are drowning in framework messages. Walk me through your fix.🔴 SENIOR

First, identify the noisy categories (they're printed with each message — usually Microsoft.AspNetCore.* and Microsoft.EntityFrameworkCore.*). Then raise their minimums via category overrides — e.g., "Microsoft.AspNetCore": "Warning" — while keeping application namespaces at Information. With Serilog, the same idea lives under MinimumLevel:Override, and replacing the framework's many per-request messages with UseSerilogRequestLogging() gives one clean summary line per request. Finally, verify the volume drop and confirm errors still surface — the goal is less noise, not less signal.

Q22When would you destructure an object with {@Obj}, and what are the risks?🔴 SENIOR

Destructuring ({@Order} in Serilog) stores an object's properties as structured data rather than its ToString(). It's great for small, stable DTOs where you genuinely want the fields queryable. Risks: accidentally serializing huge object graphs (an EF entity dragging its navigation properties), leaking sensitive fields buried in the object, and creating high-cardinality noise that bloats storage. Mitigate by destructuring purpose-built log DTOs, configuring destructuring policies/attributes to redact fields, and capping depth.

Q23Logging vs. metrics vs. tracing — how do they differ and when do you reach for each?🔴 SENIOR

They're the three pillars of observability. Logs are discrete, detailed event records — best for investigating what exactly happened in one case. Metrics are cheap numeric aggregates over time (requests/sec, error rate, memory) — best for dashboards and alerting on trends. Traces follow one request across services with timing per step — best for finding where latency or failure occurs in a distributed flow. Mature systems use all three, increasingly through OpenTelemetry, with logs enriched by trace IDs so you can pivot between pillars.

Q24A teammate proposes logging every method's entry and exit at Information level. Your response?🔴 SENIOR

Push back constructively: that pattern floods logs with low-value noise, raises costs, and buries real signals. Information should mark business-meaningful events (order placed, payment failed), not code mechanics. If method-level flow is genuinely needed, put it at Trace/Debug so it's off by default and can be enabled per namespace during an investigation. Better still, get flow information from tracing (spans time the operations automatically) and keep logs for events and decisions. Set the norm in a short team logging guideline: what each level means, standard property names, and what never goes in a log.
