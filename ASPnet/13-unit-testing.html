Day 13 · ASP.NET Core Fundamentals

🧪 Unit Testing: Proving Your Code Works

Everything you've built this fortnight — services, EF queries, HTTP clients, workers — was designed around dependency injection. Today that pays off: because your classes depend on interfaces, you can swap real dependencies for fakes and test your logic in complete isolation. You'll use xUnit to run tests and Moq to build the fakes.

✅ xUnit 🎭 Moq 📐 Arrange-Act-Assert 🔌 Testing DI Services 🔍 Verify Interactions

01🎯 Why Unit Test?

A unit test is a small, fast, automated check that one piece of code — usually one method — does what it should. "Unit" means in isolation: you test your service's logic without touching a real database, real network, or real anything. If it fails, you know the bug is right there, not somewhere in a tangle of systems.

🏭 Analogy

Unit tests are the quality-control station on an assembly line. Each part is checked before it goes into the product, on its own, against a spec — is this bolt the right size? does this switch click? You don't wait until the whole car is built and someone crashes it (that's the slow, expensive end-to-end test). Catching a bad part at the station is cheap; catching it after assembly is not.

What tests buy you 💰

🐛Catch bugs early

Failures surface in seconds on your machine, not days later in production where they're costly to trace.

🛡️Refactor without fear

A green test suite is a safety net — change the internals freely, and tests tell you instantly if behavior broke.

📖Living documentation

A well-named test states exactly what the code is supposed to do — and unlike comments, it can't drift out of date.

🎨Better design pressure

Code that's hard to test is usually badly coupled. Writing tests nudges you toward clean, injectable design.

The test pyramid 🔺

Not all tests are equal. The classic guidance: lots of fast unit tests, fewer integration tests, very few end-to-end tests — because the higher you go, the slower, more brittle, and more expensive they get.

🧪 Unit
many · ms · isolated
🔗 Integration
some · seconds · real DB
🌐 End-to-End
few · slow · whole system
The test pyramid · a broad base of unit tests, a thin cap of E2E · today is the base
🔑 The Golden Properties

A good unit test is F.I.R.S.T.: Fast (milliseconds), Isolated (no shared state, no real dependencies), Repeatable (same result every run), Self-validating (passes or fails, no manual inspection), and Timely (written alongside the code). If a test hits a real database or the network, it's an integration test, not a unit test — a different, slower tier.

03✅ xUnit Basics

xUnit is the most popular test framework for modern .NET. It runs your tests, reports pass/fail, and gives you the attributes and assertions to write them. You put tests in a separate test project that references the project it tests.

Terminal · set up a test project
# Create an xUnit test project next to your app:
dotnet new xunit -n MyShop.Tests

# Reference the project under test so you can use its classes:
dotnet add MyShop.Tests reference MyShop

# Add Moq for mocking dependencies:
dotnet add MyShop.Tests package Moq

# Run everything:
dotnet test

Facts and Theories 🏷️

Two attributes mark a method as a test:

C# · [Fact] and [Theory]
using Xunit;

public class CalculatorTests
{
    // [Fact]: a single test with no parameters — one fixed scenario.
    [Fact]
    public void Add_TwoAndThree_ReturnsFive()
    {
        var calc = new Calculator();
        int result = calc.Add(2, 3);
        Assert.Equal(5, result);   // expected first, actual second
    }

    // [Theory]: one test run once per data row — same logic, many inputs.
    [Theory]
    [InlineData(2, 3, 5)]
    [InlineData(-1, 1, 0)]
    [InlineData(0, 0, 0)]
    public void Add_VariousInputs_ReturnsSum(int a, int b, int expected)
    {
        var calc = new Calculator();
        Assert.Equal(expected, calc.Add(a, b));
    }
}

Common assertions

AssertionChecks that…
Assert.Equal(expected, actual)Two values are equal (note the order: expected first).
Assert.True(x) / Assert.False(x)A boolean is true / false.
Assert.Null(x) / Assert.NotNull(x)A reference is null / not null.
Assert.Contains(item, collection)A collection contains an item.
Assert.Throws<T>(() => ...)The code throws exception type T.
await Assert.ThrowsAsync<T>(...)Async version for awaited calls.
Assert.IsType<T>(x)An object is exactly type T (great for testing controller results).
🔑 A Fresh Instance Per Test

xUnit creates a new instance of the test class for every test method. That's deliberate — it guarantees tests don't share state and can't contaminate each other. Any setup you put in the test class's constructor runs fresh before each test (xUnit's replacement for old "SetUp" methods); cleanup goes in Dispose.

✅ Naming Convention

The widely-used pattern is MethodName_Scenario_ExpectedBehavior, e.g. PlaceOrder_EmptyCart_ThrowsValidationException. When a test fails, its name alone tells you what broke and under what condition — no need to open the code. Self-documenting names are half the value of a test suite.

04📐 Arrange-Act-Assert

Every good test has the same three-part shape, known as AAA. Keeping the parts visually separate makes tests instantly readable — anyone can see the setup, the action, and the check at a glance.

🧰
ArrangeSet up everything the test needs: create the object under test, build its fake dependencies, prepare input data and expected values.
▶️
ActCall the one method you're testing. Just one action — that's what this test is about.
✅
AssertCheck the outcome: the return value is what you expected, or the right thing happened, or the right exception was thrown.
C# · AAA labeled
[Fact]
public void ApplyDiscount_Over1000_Gives10PercentOff()
{
    // Arrange
    var pricing = new PricingService();
    decimal orderTotal = 2000m;

    // Act
    decimal result = pricing.ApplyDiscount(orderTotal);

    // Assert
    Assert.Equal(1800m, result);
}
🔑 One Act, One Reason to Fail

Keep a single Act per test. If a test calls three methods, and it fails, you can't tell which one broke. One action, focused assertions, a self-explaining name — that's a test that diagnoses, not just detects. Multiple scenarios? Write multiple tests (or a [Theory]).

⚠️ Don't Test Trivia, Don't Test Privates

Test behavior, not implementation details. Don't write tests for getters/setters with no logic, and don't try to reach into private methods — they're implementation details tested indirectly through the public method that calls them. If a private method feels like it needs its own test, that's often a hint it should be its own class.

05🎭 Test Doubles: Fakes, Stubs & Mocks

Your OrderService depends on an IOrderRepository (which hits the database) and an IEmailSender (which hits the network). To unit-test OrderService in isolation, you replace those real dependencies with test doubles — stand-ins you control. This is the whole reason we've been coding against interfaces all fortnight.

🎬 Analogy

Test doubles are stunt doubles and stage props on a film set. You don't crash a real car or hire the real A-list star for a dangerous shot — you use a stunt double that behaves how the scene needs. A test double stands in for a real dependency so you can test your scene (your method) safely, cheaply, and exactly the way you want, without the real database or network showing up.

The three flavors (they overlap in casual use) 🎭

📋Stub

Provides canned data so the code can run. "When asked for order 1, return this fixed order." It answers questions; it doesn't judge the test.

🔍Mock

Also verifies interactions. "Was SendEmail called exactly once with the right address?" A mock can make the test fail based on how it was used.

🛠️Fake

A lightweight working implementation — e.g. an in-memory list standing in for a repository. Real logic, just not production-grade.

🔑 The Distinction That Matters

A stub feeds your code inputs (state-based testing: "given this data, did I get the right answer?"). A mock checks outputs to collaborators (interaction-based testing: "did I call the right dependency the right way?"). Moq creates objects that can do both — you Setup them like stubs and Verify them like mocks. In everyday speech people call them all "mocks," but knowing which behavior you're relying on keeps tests focused.

⚠️ Don't Over-Mock

Mock the boundaries — things that are slow, external, or nondeterministic (databases, HTTP, clocks, file systems). Don't mock simple value objects, DTOs, or the very class you're testing. Over-mocking produces tests that verify your code calls methods in a certain order rather than that it produces correct results — brittle tests that break on every refactor while proving little.

06🎬 Moq: Setting Up Fakes

Moq generates a test double from an interface at runtime — no hand-written fake class needed. You create one, teach it how to respond, and hand its .Object to your service.

C# · Creating and setting up a mock
using Moq;

// 1. Create a mock of the interface:
var repoMock = new Mock<IOrderRepository>();

// 2. Setup: "when GetByIdAsync(1) is called, return this order":
repoMock
    .Setup(r => r.GetByIdAsync(1))
    .ReturnsAsync(new Order { Id = 1, Total = 500m });

// 3. Hand the fake to your real service via its constructor (DI!):
var service = new OrderService(repoMock.Object);

Matching arguments with It 🎯

You rarely want to hard-code every argument. The It helpers match flexibly:

C# · Argument matchers
// Match ANY int argument:
repoMock.Setup(r => r.GetByIdAsync(It.IsAny<int>()))
    .ReturnsAsync(new Order { Id = 99 });

// Match arguments meeting a condition:
repoMock.Setup(r => r.GetByIdAsync(It.Is<int>(id => id > 0)))
    .ReturnsAsync(new Order());

// Return a value that depends on the input:
repoMock.Setup(r => r.GetByIdAsync(It.IsAny<int>()))
    .ReturnsAsync((int id) => new Order { Id = id });

// Make it throw, to test error paths:
repoMock.Setup(r => r.GetByIdAsync(0))
    .ThrowsAsync(new ArgumentException("Invalid ID"));
MethodUse for
.Returns(value)A synchronous method's return value.
.ReturnsAsync(value)An async method returning Task<T>.
.ThrowsAsync(ex) / .Throws(ex)Simulating a failure.
It.IsAny<T>()Match any argument of type T.
It.Is<T>(predicate)Match arguments meeting a condition.
.Callback(...)Capture the argument passed, or run side-effect code.
⚠️ Unconfigured Methods Return Defaults — Silently

By default a Moq mock is "loose": call a method you didn't set up and it returns a harmless default (null, 0, an empty collection) rather than complaining. Convenient, but it can hide the fact that your test isn't exercising what you think. For critical dependencies, consider new Mock<T>(MockBehavior.Strict), which throws on any unconfigured call — forcing you to be explicit about every interaction the test expects.

07🔍 Moq: Verifying Interactions

Some behavior has no return value to assert on — did the service actually send the email? save the order? That's an interaction, and you check it with Verify: assert that a method on a mock was (or wasn't) called, how many times, and with what arguments.

C# · Verify calls
// Assert SaveAsync was called exactly once, with any Order:
repoMock.Verify(r => r.SaveAsync(It.IsAny<Order>()), Times.Once);

// Assert an email went to a specific address:
emailMock.Verify(e => e.SendAsync(
    It.Is<string>(to => to == "customer@example.com"),
    It.IsAny<string>()), Times.Once);

// Assert something did NOT happen (payment bounced → no shipment):
shipMock.Verify(s => s.ShipAsync(It.IsAny<Order>()), Times.Never);
Times.Asserts the call happened…
OnceExactly one time.
NeverZero times — great for "this should NOT happen" safety checks.
Exactly(n)Exactly n times.
AtLeastOnce / AtMostOnceAt least / at most one time.
AtLeast(n) / AtMost(n)At least / at most n times.
🔑 Verify Behavior Worth Verifying

The Times.Never check is quietly powerful: it proves your code didn't do something dangerous — didn't ship an unpaid order, didn't charge a card twice, didn't email when it shouldn't. Those "it must not happen" guarantees are often more valuable than the happy-path ones, and only interaction verification can express them.

⚠️ Assert on Results Where You Can

Prefer asserting on return values and state over verifying calls when both are possible. A test full of Verify calls tends to mirror the implementation line-by-line, so any refactor breaks it even when behavior is unchanged. Reach for Verify when the interaction is the observable behavior (an email was sent, a row was saved) and there's no return value to check.

08🔌 Testing a Service That Uses DI

Now the payoff. Here's a realistic service with two dependencies — exactly the shape you've built all fortnight. We'll test it thoroughly without a database or an email server in sight.

C# · The service under test
public class OrderService(IOrderRepository repo, IEmailSender email)
{
    public async Task<Order> PlaceOrderAsync(Order order, CancellationToken ct = default)
    {
        if (order.Items.Count == 0)
            throw new ValidationException("Order must have at least one item.");

        order.PlacedAt = DateTime.UtcNow;
        await repo.SaveAsync(order, ct);
        await email.SendAsync(order.CustomerEmail, "Order confirmed", ct);
        return order;
    }
}

The test class arranges the mocks once in its constructor (fresh per test, remember), so each test reads cleanly:

C# · OrderServiceTests.cs
public class OrderServiceTests
{
    private readonly Mock<IOrderRepository> _repoMock = new();
    private readonly Mock<IEmailSender> _emailMock = new();
    private readonly OrderService _sut;   // "System Under Test"

    public OrderServiceTests()
    {
        // Runs fresh before EVERY test — clean mocks each time.
        _sut = new OrderService(_repoMock.Object, _emailMock.Object);
    }

    [Fact]
    public async Task PlaceOrder_ValidOrder_SavesAndSendsEmail()
    {
        // Arrange
        var order = new Order
        {
            CustomerEmail = "priya@example.com",
            Items = { new OrderItem { ProductId = 1, Quantity = 2 } }
        };

        // Act
        var result = await _sut.PlaceOrderAsync(order);

        // Assert — state:
        Assert.NotEqual(default, result.PlacedAt);
        // Assert — interactions: it saved once and emailed the right person:
        _repoMock.Verify(r => r.SaveAsync(order, It.IsAny<CancellationToken>()), Times.Once);
        _emailMock.Verify(e => e.SendAsync("priya@example.com",
            It.IsAny<string>(), It.IsAny<CancellationToken>()), Times.Once);
    }

    [Fact]
    public async Task PlaceOrder_EmptyCart_ThrowsAndSavesNothing()
    {
        // Arrange — an order with no items
        var order = new Order { CustomerEmail = "x@example.com" };

        // Act + Assert — the exception:
        await Assert.ThrowsAsync<ValidationException>(
            () => _sut.PlaceOrderAsync(order));

        // Assert — the safety guarantee: nothing was saved or emailed:
        _repoMock.Verify(r => r.SaveAsync(It.IsAny<Order>(),
            It.IsAny<CancellationToken>()), Times.Never);
        _emailMock.Verify(e => e.SendAsync(It.IsAny<string>(),
            It.IsAny<string>(), It.IsAny<CancellationToken>()), Times.Never);
    }
}
🔑 This Is Why We Used Interfaces

Notice what made this effortless: OrderService takes IOrderRepository and IEmailSender, not concrete classes. So in the test we simply pass fakes into the same constructor DI uses in production. Testability isn't a testing technique — it's a design property, and dependency injection (Day 3) is what granted it. Code that news up its own dependencies internally is nearly impossible to unit test.

✅ What About the DbContext?

Notice we tested against IOrderRepository, not a raw DbContext. Mocking EF Core directly is painful and often misleading (it doesn't replicate real query translation — Day 8). Best practice: put data access behind a repository interface or test the query logic with integration tests against a real (containerized) database. Unit tests are for your business logic; the database deserves the real thing.

09🎲 Theories & Edge Cases

Real confidence comes from testing the sad paths, not just the happy one. And when the same logic needs checking against many inputs, a [Theory] saves you from copy-pasting near-identical tests.

C# · Data-driven tests
// InlineData: quick literal rows, checked in one test method:
[Theory]
[InlineData(100, 100)]      // under threshold → no discount
[InlineData(1000, 900)]     // at threshold → 10% off
[InlineData(2000, 1800)]    // well over → 10% off
public void ApplyDiscount_VariousTotals_DiscountsCorrectly(decimal total, decimal expected)
{
    var pricing = new PricingService();
    Assert.Equal(expected, pricing.ApplyDiscount(total));
}

// MemberData: for complex objects that don't fit in an attribute:
[Theory]
[MemberData(nameof(InvalidOrders))]
public async Task PlaceOrder_InvalidInput_Throws(Order order)
{
    await Assert.ThrowsAsync<ValidationException>(() => _sut.PlaceOrderAsync(order));
}

public static IEnumerable<object[]> InvalidOrders => new[]
{
    new object[] { new Order { Items = { } } },              // empty
    new object[] { new Order { CustomerEmail = null! } },   // no email
};

The edge cases worth remembering 🧩

0️⃣Empty & zero

Empty collections, empty strings, zero amounts, the number 0. Boundaries are where bugs live.

🚫Null

Null inputs, null returns from dependencies. What does your code do when a mock returns null?

💥Failure

Dependency throws — set up ThrowsAsync and assert your code handles it (retries, logs, rethrows).

📏Boundaries

Exactly at the threshold (1000), one below (999), one above (1001). Off-by-one errors hide here.

✅ Test the Failure Paths First

The happy path is the one you'll notice is broken anyway. The sad paths — empty input, a downstream failure, the boundary value — are where silent bugs hide and where good tests earn their keep. A suite that only tests success is a suite that lies about its coverage.

🔑 InlineData vs MemberData

[InlineData] is for simple compile-time constants (numbers, strings, bools) written right on the method. [MemberData] (and [ClassData]) pull rows from a property or method, so they can supply objects and computed data that won't fit in an attribute. Reach for MemberData the moment your cases stop being simple literals.

10🛠️ Hands-On: Test a Service, Mock Its Dependencies

Today's Do: pick one service you built earlier — ideally one with a dependency or two (a repository, an email sender, an HTTP client) — and write a focused set of unit tests, mocking those dependencies. The path:

  1. 📦 Create the test project: dotnet new xunit, reference your app project, and dotnet add package Moq.
  2. 🏗️ Set up the test class: fields for each Mock<IDependency>, and a constructor that builds the SUT with the mocks' .Object (fresh per test).
  3. ✅ Happy path: arrange the mocks to return sensible data (Setup + ReturnsAsync), act on the method, and assert on the return value.
  4. 🔍 Interaction: add a Verify(..., Times.Once) to confirm the service called a key dependency correctly (saved the order, sent the email).
  5. 🚫 Safety check: write a test where the input is invalid, assert it throws with ThrowsAsync, and verify the dangerous dependency was called Times.Never.
  6. 💥 Failure path: set up a dependency to ThrowsAsync, and assert your service handles or propagates it as designed.
  7. 🎲 Data-driven: convert a set of similar cases (boundaries: 999 / 1000 / 1001) into one [Theory] with [InlineData].
  8. ▶️ Run dotnet test, watch them pass, then break the service on purpose (change a >= to >) and confirm a test catches it. A test that never fails proves nothing.
C# · A complete failure-path test
[Fact]
public async Task PlaceOrder_WhenRepositoryFails_PropagatesAndSkipsEmail()
{
    // Arrange — the repository will blow up on save:
    var order = new Order
    {
        CustomerEmail = "a@b.com",
        Items = { new OrderItem { ProductId = 1, Quantity = 1 } }
    };
    _repoMock
        .Setup(r => r.SaveAsync(It.IsAny<Order>(), It.IsAny<CancellationToken>()))
        .ThrowsAsync(new DbUpdateException("DB down"));

    // Act + Assert — the failure surfaces:
    await Assert.ThrowsAsync<DbUpdateException>(() => _sut.PlaceOrderAsync(order));

    // Assert — no confirmation email went out after a failed save:
    _emailMock.Verify(e => e.SendAsync(It.IsAny<string>(),
        It.IsAny<string>(), It.IsAny<CancellationToken>()), Times.Never);
}
✅ Self-Check

From your own tests: (1) Which dependencies did you mock, and why those and not others? (2) Did you assert on a return value and verify at least one interaction? (3) Do you have at least one Times.Never test proving something bad didn't happen? (4) When you deliberately broke the service, did a test go red — and did its name tell you what broke?

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is a unit test and how is it different from other tests?🌱 JUNIOR

A unit test is a small, fast, automated check that one piece of code (usually one method) behaves correctly in isolation — no real database, network, or file system. If it fails, you know the problem is in that specific code. It differs from integration tests (which test several components working together, often with a real database) and end-to-end tests (which drive the whole running system). Unit tests are the fastest and most numerous; the others are slower and fewer — that's the test pyramid.

Q2What is xUnit, and what do [Fact] and [Theory] mean?🌱 JUNIOR

xUnit is a popular testing framework for .NET — it discovers and runs your tests and reports results. [Fact] marks a method as a single test with no parameters — one fixed scenario. [Theory] marks a parameterized test that runs once per data row (supplied by [InlineData] or [MemberData]), letting you check the same logic against many inputs without duplicating the test.

Q3Explain the Arrange-Act-Assert pattern.🌱 JUNIOR

It's the three-part structure of a clean test. Arrange: set up everything needed — create the object under test, its fake dependencies, and input data. Act: call the one method you're testing. Assert: check the outcome — the return value, a thrown exception, or that a dependency was called. Keeping the sections visually separate (often with comments) makes the test easy to read: setup, action, verification.

Q4What is a mock, and why would you use one?🌱 JUNIOR

A mock is a fake object that stands in for a real dependency during a test. You use it to isolate the code you're testing: instead of your service calling a real database or email server, it calls a mock you control — you tell the mock what to return, and you can check how it was called. This makes tests fast, repeatable, and focused on your code's logic rather than the behavior of external systems.

Q5In Moq, what do Setup and Verify do?🌱 JUNIOR

Setup configures how the mock responds — "when GetByIdAsync(1) is called, return this order" (via .ReturnsAsync(...)), or make it throw. Verify checks, after the act, that a method was actually called — how many times and with what arguments — e.g. Verify(r => r.SaveAsync(order), Times.Once). Setup controls inputs to your code; Verify checks the calls your code made to its dependencies.

Q6How do you test that a method throws an exception?🌱 JUNIOR

Use Assert.Throws<TException>(() => ...) for synchronous code, or await Assert.ThrowsAsync<TException>(() => ...) for async. You pass the code that should throw as a lambda; the assertion fails if it doesn't throw, or throws a different type. You can also capture the returned exception to assert on its message. It's good practice to also verify that no unwanted side effects (like a save or email) happened when the exception was thrown.

Q7Why does dependency injection make code easier to test?🌱 JUNIOR

Because a class that receives its dependencies through its constructor (as interfaces) lets you pass in fakes during a test — the exact same way the DI container passes real ones in production. You just call the constructor with mock objects. In contrast, a class that creates its own dependencies internally (new SqlRepository()) gives you no seam to substitute a fake, making it nearly impossible to unit test in isolation. Testability is a benefit you get "for free" from good DI design.

Q8What makes a good unit test? Name a few properties.🌱 JUNIOR

The FIRST properties: Fast (runs in milliseconds), Isolated (no shared state, no real external systems), Repeatable (same result every time — no dependence on dates, randomness, or order), Self-validating (clearly passes or fails, no manual checking), and Timely (written close to the code). Add a clear name (Method_Scenario_Expected) and a single Act, and you have a test that both catches bugs and documents intent.

🟠 Mid

Q9What's the difference between a stub and a mock?🟠 MID

A stub provides canned data so your code can run — it answers queries but doesn't affect whether the test passes ("when asked for order 1, return this"). It supports state-based testing: given this input, did I compute the right result? A mock additionally verifies interactions — it can make the test fail based on whether (and how) it was called. It supports interaction-based testing: did my code call the dependency the right way? In strict literature they're distinct; in practice Moq objects do both — you Setup them like stubs and Verify them like mocks — so people loosely call everything a "mock." Knowing which behavior you're relying on keeps a test focused on one thing.

Q10When should you prefer asserting on return values versus verifying interactions?🟠 MID

Prefer asserting on return values and observable state whenever the behavior produces one — it's more robust because it tests what the code achieves, not how, so refactoring the internals doesn't break the test. Reach for interaction verification (Verify) when the side effect is the observable behavior and there's nothing to return — an email was sent, a row was saved, a message was published. Over-relying on Verify produces brittle tests that mirror the implementation line-by-line and break on every refactor while proving little about correctness.

Q11What is Times.Never useful for? Give an example.🟠 MID

It proves your code didn't do something it shouldn't — a negative guarantee that's often more valuable than a positive one. Example: in a checkout service, if payment fails, you must not ship the order. shipMock.Verify(s => s.ShipAsync(It.IsAny<Order>()), Times.Never) asserts exactly that. Other cases: not charging a card twice, not sending an email when validation failed, not saving an invalid record. These "must not happen" rules can only be expressed through interaction verification, since there's no positive result to assert on.

Q12Moq mocks return defaults for unconfigured methods. What problem can that cause, and what's the alternative?🟠 MID

By default Moq uses "loose" behavior: any method you didn't Setup returns a harmless default (null, 0, empty collection) without complaint. The risk is silent false confidence — your test passes because a method returned null, not because the code is correct, and you may not realize a dependency wasn't exercised as intended. The alternative is new Mock<T>(MockBehavior.Strict), which throws on any unconfigured call, forcing you to explicitly set up every interaction the code is expected to make. Strict mocks catch unexpected calls and make the test's assumptions explicit, at the cost of more setup and more brittleness — use them where precise interaction control matters.

Q13How do you avoid shared state between tests in xUnit?🟠 MID

Largely, xUnit does it for you: it creates a new instance of the test class for every test method, so instance fields (like your mocks and SUT) are re-created fresh each time — no leakage between tests. Put per-test setup in the constructor (xUnit's equivalent of a SetUp method) and cleanup in Dispose. Avoid static fields and shared mutable state, which do persist across tests and cause order-dependent, flaky failures. For expensive shared setup that's genuinely read-only (like a database fixture), use IClassFixture/ICollectionFixture deliberately rather than static state.

Q14Why is mocking a DbContext usually discouraged, and what do you do instead?🟠 MID

Because a mocked DbContext/DbSet doesn't behave like the real thing — LINQ that a real database translates to SQL runs as in-memory LINQ against your fakes, so queries can pass in tests but fail (or behave differently) in production; you end up testing your mock setup, not your data access. Instead: put data access behind a repository interface and mock that for unit tests of business logic, and/or write integration tests that run against a real database (SQL Server in a container via Testcontainers, or LocalDB) so query translation is genuinely exercised. The EF Core in-memory provider is tempting but also diverges from relational behavior. Unit-test the logic; integration-test the data layer.

Q15What are InlineData, MemberData, and ClassData, and when do you use each?🟠 MID

All three feed rows to a [Theory]. [InlineData(...)] puts literal values right on the method — best for simple constants (numbers, strings, bools). [MemberData(nameof(X))] pulls rows from a static property or method returning IEnumerable<object[]> — needed when cases are complex objects or computed values that can't be constant expressions. [ClassData(typeof(X))] uses a dedicated class implementing IEnumerable<object[]> — good for reusing a data set across multiple test classes or when the generation logic is substantial. Rule of thumb: literals → InlineData; objects/computed → MemberData; reusable/complex → ClassData.

Q16How would you unit-test code that depends on the current date/time?🟠 MID

Don't call DateTime.Now directly in the code — that makes tests non-repeatable (they depend on when they run). Inject an abstraction for the clock. On .NET 8+, the framework provides TimeProvider: your code takes a TimeProvider and calls GetUtcNow(), and in tests you pass a FakeTimeProvider (from Microsoft.Extensions.TimeProvider.Testing) whose time you set and even advance. Before .NET 8 the pattern was a custom IClock/IDateTimeProvider interface, mocked in tests. Either way the principle is the same as any dependency: make the nondeterministic thing injectable so the test controls it.

🔴 Senior

Q17Your team's tests are brittle — they break on every refactor. Diagnose and fix.🔴 SENIOR

Brittleness almost always means tests are coupled to implementation rather than behavior. Common causes: over-verification (asserting every internal call and its order with Verify, so any restructure fails even when the result is identical); over-mocking (mocking collaborators that aren't real boundaries, turning the test into a mirror of the code); testing private methods or internal structure; and mocks configured with exact arguments where It.Is conditions would be more meaningful. Fixes: assert on outputs and observable state wherever possible; reserve Verify for genuine side effects (email sent, row saved); mock only true boundaries (I/O, external services, clock); test through the public API; and treat the test's job as "does the behavior meet the contract," not "does the code do these exact steps." Well-designed behavioral tests survive refactors — that's the whole point of the safety net.

Q18What is code coverage, and why is a high coverage number not the goal?🔴 SENIOR

Coverage measures how much of your code executes during tests (lines/branches). It's a useful diagnostic — low coverage reliably flags untested areas — but a poor target, because it measures execution, not verification. You can hit 100% coverage with tests that call every line and assert nothing meaningful, or that only exercise happy paths. Chasing a number invites gaming: tests written to touch lines rather than to catch bugs, and tests for trivial getters that add noise. Better to use coverage to find blind spots (especially untested branches and error paths), aim for meaningful assertions on important logic, and accept that some code (thin wiring, generated code) doesn't warrant unit tests. Quality of assertions > quantity of covered lines.

Q19Where's the line between unit and integration tests, and how do you decide what belongs where?🔴 SENIOR

A unit test isolates one unit of behavior with all real dependencies replaced by doubles — fast, deterministic, no I/O. An integration test exercises real collaboration across a boundary — a real database, the real DI container, the real HTTP stack — accepting slower, less isolated runs for higher-fidelity confidence. Decide by what you're actually validating: business logic and branching (discount rules, validation, orchestration) → unit tests, mocking the boundaries; things that only fail against the real component (EF query translation, migrations, SQL constraints, serialization on the wire, middleware pipeline behavior) → integration tests, because a mock would give false confidence. Practically: broad base of unit tests for logic, a targeted set of integration tests for the seams that mocks can't faithfully represent, and a few end-to-end tests for critical user journeys. The anti-pattern is unit-testing things that need the real component (mocked EF) or integration-testing pure logic (slow and redundant).

Q20How do you test the async and cancellation behavior of a service?🔴 SENIOR

For async methods: make the tests async Task and await the SUT; set up mocks with ReturnsAsync/ThrowsAsync; assert exceptions with Assert.ThrowsAsync. Don't block with .Result in tests (same deadlock/starvation risks as production, Day 10). For cancellation: pass a token you control. To test that cancellation is honored, create a CancellationTokenSource, cancel it (immediately or via CancelAfter), pass its token, and assert the method throws OperationCanceledException — and, importantly, verify it propagates the token to its dependencies (Verify the mock was called with the expected token, or set the mock to throw on the cancelled token). You can also verify that no side effects ran after cancellation. The subtle part is proving the token actually flows through, not just that the top-level call cancels — a common real bug is a token dropped halfway down.

Q21Explain TDD and its cycle. What are its real benefits and costs?🔴 SENIOR

Test-Driven Development follows Red-Green-Refactor: write a failing test for the next small piece of behavior (Red), write the minimum code to make it pass (Green), then improve the design while keeping tests green (Refactor). Benefits: it forces you to define behavior before implementing, yields a comprehensive test suite as a byproduct, drives clean, decoupled, testable design (you feel coupling immediately when a test is hard to write), and gives fast feedback. Costs and caveats: it's a skill with a learning curve; done dogmatically it can produce over-specified tests or waste effort on trivial code; it works best for logic with clear input/output and less well for exploratory or heavily I/O-bound work. Many strong teams practice it selectively — TDD for gnarly business rules, test-after for straightforward wiring. The value isn't ritual purity; it's that thinking about verifiability up front tends to produce better-designed code.

Q22A test passes locally but fails intermittently in CI. How do you approach a flaky test?🔴 SENIOR

Flakiness is a bug in the test (or the code) — never "just re-run it." Common causes and fixes: shared/leaking state between tests exposed by parallel execution or different ordering (xUnit runs tests in parallel by default) → remove static mutable state, ensure per-test isolation; time dependence (DateTime.Now, time zones) → inject a clock; ordering assumptions (relying on another test's side effect) → make each test self-contained; real async races or arbitrary Task.Delay-based waits → use deterministic synchronization or fake timers instead of sleeping; external resources (network, real DB with unclean state) → those belong in integration tests with proper isolation (transaction rollback, fresh DB per test), not unit tests. Approach: reproduce by running repeatedly and in parallel, isolate the offending test, identify the nondeterministic input, and remove it. A flaky suite erodes trust until people ignore failures — so flakiness is treated as a priority defect, not a nuisance.

Q23How do you keep a large test suite maintainable as the codebase grows?🔴 SENIOR

Treat test code as production code — it needs the same care. Practices: reduce duplication with builders/object mothers for test data (a OrderBuilder that produces valid orders with overridable fields) rather than repeating construction everywhere; keep setup close to the test (helper methods over sprawling shared fixtures, so a test is readable top-to-bottom); test behavior not implementation so refactors don't cascade into test churn (the biggest maintenance sink); name tests clearly so failures are self-diagnosing; keep tests fast so the suite stays run-on-every-change; delete redundant or obsolete tests deliberately rather than accumulating them; and organize tests to mirror the code structure. Also invest in the boundary contracts you mock — if an interface changes, you want one place to update. The meta-principle: a test suite is an asset only while it's trusted and fast; the moment it's slow, flaky, or brittle it becomes a liability people route around, so maintainability is the point.

Q24When is mocking the wrong choice, and what are the alternatives?🔴 SENIOR

Mocking is wrong when it makes the test less truthful or more brittle than the alternative. Cases: mocking types you don't own (a third-party SDK, EF Core) — the mock encodes your assumption of their behavior, which can be wrong and won't update when they change; prefer a thin adapter you own and mock that, or an integration test against the real thing. Mocking things that are cheap and deterministic (value objects, pure functions, simple in-memory collaborators) — just use the real object; a mock adds noise and couples the test to call patterns. Complex interaction-heavy mock setups that become harder to read than the code — a hand-written fake (e.g. an in-memory repository implementing the interface) is often clearer and reusable across tests. And for anything where fidelity matters more than speed — query translation, serialization, concurrency — an integration test beats a mock. The judgment: mock true, owned, expensive boundaries; use real objects for cheap deterministic ones; use fakes for stateful collaborators; and reach for integration tests when only the real component tells the truth.
