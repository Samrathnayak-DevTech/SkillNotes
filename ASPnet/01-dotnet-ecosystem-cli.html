Ecosystem & the CLI ⚙️

Before writing C#, it helps to know what .NET actually is: a runtime that executes your code, an SDK that builds it, and a command-line tool — dotnet — that ties it all together. In this first day you'll learn the SDK-vs-runtime distinction, the core CLI verbs (new / build / run / test), how projects and solutions are laid out, what lives in a .csproj, and how NuGet pulls in packages — then build a real solution with a console app and a class library. No prior .NET background assumed.

01

What .NET Is

.NET is a free, cross-platform, open-source developer platform — a runtime that runs your code, a set of libraries, languages (C#, F#, VB), and tooling to build all kinds of apps.

Modern .NET (the successor to the older, Windows-only .NET Framework) runs on Windows, Linux, and macOS. You write code in a language like C#, the compiler turns it into an intermediate form, and the runtime executes it. Around that sit thousands of built-in and third-party libraries and a unified command-line tool. With one platform you can build console tools, web APIs and sites (ASP.NET Core), desktop and mobile apps, cloud services, and more.

🏭 Analogy

Think of .NET as a kitchen. The runtime is the stove that actually cooks your dish (executes your program). The SDK is the full set of chef's tools — knives, mixers, recipes — used to prepare the dish (compile and build). NuGet is the pantry you pull pre-made ingredients from. And the dotnet CLI is the head chef issuing the commands that make everything happen.

🔑 One name, a bit of history

".NET Core" was the name of the cross-platform rewrite (versions 1–3.1). From version 5 onward Microsoft dropped "Core" and calls it simply ".NET" (.NET 5, 6, 7, 8, 9, 10…). So ".NET Core" and modern ".NET" refer to the same lineage — distinct from the legacy Windows-only ".NET Framework." When people say ".NET Core" today they almost always mean modern .NET.

02

SDK vs Runtime

The single most important Day-1 distinction: the runtime runs .NET apps; the SDK builds them (and includes a runtime).

RuntimeSDK
PurposeExecute a finished appCreate, compile, test, publish apps
Who needs itEnd users / servers running your appDevelopers writing code
IncludesCLR + base librariesRuntime + compiler + CLI + templates + MSBuild
Has the dotnet build command?NoYes
🔑 The SDK is a superset

Installing the SDK gives you everything: the compiler, the CLI, project templates, MSBuild, and a runtime to execute what you build. Installing only the runtime lets a machine run .NET apps but not build them — which is exactly what you want on a production server. As a developer, you install the SDK.

✅ Check what you have installed

dotnet --version shows the active SDK version; dotnet --list-sdks and dotnet --list-runtimes list all installed SDKs and runtimes; and dotnet --info prints a full report of both plus your OS and architecture. These are the first commands to run when something's off.

03

Versions & Support

.NET ships a new major version every November. Each is either LTS (long-term support) or STS (standard-term support).

Release typeCadenceSupported forChoose when
LTS (Long-Term Support)Odd-numbered years (e.g. .NET 8, .NET 10)3 yearsProduction apps wanting stability
STS (Standard-Term Support)Even-numbered years (e.g. .NET 9)2 yearsWanting the newest features sooner
🔑 Current version context (mid-2026)

.NET 10 (released November 2025) is the current LTS, supported into November 2028 — the safe default for new projects. .NET 8 (LTS) and .NET 9 (STS) both reach end of support on November 10, 2026, so projects on those should plan to move to .NET 10. All releases are equal in quality; LTS vs STS differs only in how long it's patched. This series targets net10.0, but the CLI and concepts are identical across .NET 6–10.

✅ Pinning an SDK with global.json

A global.json file at the root of a repo can pin which SDK version the CLI uses, so everyone on the team builds with the same one: { "sdk": { "version": "10.0.100" } }. Without it, the CLI uses the latest installed SDK that's compatible. This keeps builds reproducible across machines and CI.

04

The dotnet CLI

The dotnet command is the entry point to everything: scaffolding, building, running, testing, packaging, and managing references and packages.

CommandWhat it does
dotnet newScaffold a new project, file, or solution from a template
dotnet restoreDownload the NuGet packages a project needs
dotnet buildCompile the code into assemblies (DLLs/EXEs)
dotnet runBuild (if needed) and run the app
dotnet testBuild and run the project's unit tests
dotnet publishProduce a deployable, self-contained output folder
dotnet addAdd a package or project reference to a project
dotnet slnManage projects inside a solution file
bash
# The shape of most commands: dotnet <verb> [target] [options]
dotnet --info                 # SDKs, runtimes, OS — your environment report
dotnet new list               # show all available project/file templates
dotnet build                   # compile the project/solution in the current folder
dotnet run --project App       # run a specific project
🔑 The CLI is the real interface

Visual Studio and other IDEs are friendly front-ends, but underneath they call the same CLI and MSBuild engine. Learning the CLI means you can build, test, and run .NET anywhere — your laptop, a Linux server, a CI pipeline — with no IDE at all. It's also exactly what interviewers expect you to reach for.

05

dotnet new

dotnet new scaffolds a starting point from a template — a console app, a class library, a web API, a solution file, and many more.

bash
dotnet new console   -n PeopleApp.Cli    # a runnable console application
dotnet new classlib  -n PeopleApp.Core   # a reusable class library (a DLL)
dotnet new sln       -n PeopleApp        # an empty solution to group projects
dotnet new webapi    -n PeopleApp.Api    # an ASP.NET Core web API (later in the series)
dotnet new xunit     -n PeopleApp.Tests  # an xUnit test project
TemplateProducesRunnable?
consoleA command-line app with an entry pointYes
classlibA library of reusable code (a DLL)No — referenced by others
slnA solution file that groups projects—
webapi / mvcAn ASP.NET Core web appYes
xunit / nunitA unit-test projectVia dotnet test
✅ The -n flag names it

-n PeopleApp.Cli creates a folder and project of that name. Without -o/-n, the template uses the current directory name. Run dotnet new list anytime to see every installed template — there are dozens, and you can install more.

06

Project Structure

A project is one .csproj plus its source files. Here's what a fresh console app looks like on disk.

PeopleApp.Cli/
├── PeopleApp.Cli.csproj   # the project file: settings, references
├── Program.cs             # your source code (the entry point)
├── bin/                    # build OUTPUT (DLLs/EXEs) — do not commit
│   └── Debug/net10.0/
└── obj/                    # build INTERMEDIATE files — do not commit
📄

.csproj

The project definition — target framework, references, and settings. The heart of the project.

💻

Program.cs

Your code. In a console app it holds the entry point where execution begins.

📦

bin/

The compiled output — the DLLs and EXE the build produces. Regenerated every build.

🔧

obj/

Intermediate build artifacts and restore data. Temporary scaffolding for the build.

⚠️ Never commit bin/ and obj/

Both folders are generated by the build and are recreated from your source at any time, so they don't belong in source control — a .gitignore for .NET excludes them by default. Committing them bloats the repo and causes needless merge conflicts. Your source (the .cs and .csproj files) is what matters.

07

The .csproj File

The project file is XML that tells the build how to compile the project. Modern SDK-style projects are refreshingly short.

xml · PeopleApp.Cli.csproj
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>            <!-- Exe = runnable; Library = a DLL -->
    <TargetFramework>net10.0</TargetFramework> <!-- which .NET version -->
    <ImplicitUsings>enable</ImplicitUsings>   <!-- auto-import common namespaces -->
    <Nullable>enable</Nullable>             <!-- opt in to null-safety warnings -->
  </PropertyGroup>

  <ItemGroup>
    <ProjectReference Include="..\PeopleApp.Core\PeopleApp.Core.csproj" />
  </ItemGroup>

</Project>
ElementMeaning
Sdk="Microsoft.NET.Sdk"Uses the standard SDK, which supplies sensible defaults (so the file stays tiny)
<TargetFramework>The framework you compile against — the "TFM," e.g. net10.0
<OutputType>Exe for an app, Library for a class library
<ImplicitUsings>Auto-imports common namespaces so you write fewer using lines
<Nullable>Turns on nullable-reference-type warnings to catch null bugs early
<PackageReference>A NuGet dependency (in an <ItemGroup>)
<ProjectReference>A dependency on another project in your solution
🔑 SDK-style vs the old format

Old .NET Framework .csproj files listed every source file explicitly and ran to hundreds of lines. The modern SDK-style project (introduced with .NET Core) automatically includes all .cs files in the folder and relies on SDK defaults, so a whole project fits in a dozen readable lines. If you ever see a huge, verbose project file, it's the legacy format.

08

Solutions

A solution (.sln) is a container that groups related projects so you can build and manage them together.

bash
dotnet new sln -n PeopleApp                              # create PeopleApp.sln
dotnet sln add PeopleApp.Cli/PeopleApp.Cli.csproj       # add a project
dotnet sln add PeopleApp.Core/PeopleApp.Core.csproj      # add another
dotnet sln list                                         # show projects in the solution
dotnet build                                            # builds every project in the .sln
🔑 Project vs solution

A project produces one output (an app or a library) and is defined by one .csproj. A solution is just an organizational grouping of projects — running dotnet build against the .sln builds them all in the right dependency order. A small program might be a single project; a real application is usually several projects in one solution (an API, a core library, tests, etc.).

✅ You don't strictly need a solution

You can build and run a single project without any .sln at all. Solutions become valuable once you have multiple related projects, want to open everything at once in an IDE, or need one command to build the whole set. For today's exercise we'll use one because it mirrors real-world structure.

09

NuGet

NuGet is .NET's package manager — the pantry of reusable libraries you pull into a project instead of writing everything yourself.

bash
dotnet add package Newtonsoft.Json            # add a dependency (latest version)
dotnet add package Serilog --version 4.0.0    # pin a specific version
dotnet restore                               # download all referenced packages
dotnet list package                          # show this project's packages
dotnet remove package Newtonsoft.Json         # drop a dependency

Adding a package writes a <PackageReference> line into your .csproj. On the next build (or an explicit dotnet restore), the CLI downloads that package — and everything it depends on — from the public nuget.org feed (or a private one you configure) and caches it locally.

🔑 PackageReference = a version, not a file

You don't commit the downloaded library files — you commit the <PackageReference> with its version, and restore fetches the exact packages on any machine. This keeps repos small and builds reproducible. Restore happens automatically as part of build, run, and test, so you rarely call it explicitly.

✅ Transitive dependencies come along

If package A depends on package B, adding A automatically brings B too — you don't list B yourself. NuGet resolves this whole dependency graph for you and picks compatible versions, which is why a single add package can pull in several files behind the scenes.

10

Build · Run · Test

The everyday inner loop. Each verb does one job — and each does a restore first if needed.

🔨

dotnet build

Compiles source into assemblies in bin/. Reports compiler errors and warnings. Doesn't run anything.

▶️

dotnet run

Builds if needed, then launches the app. For quick iteration on a runnable project.

🧪

dotnet test

Builds and executes the test project's tests, reporting pass/fail counts.

bash
dotnet build                          # compile; Debug config by default
dotnet build -c Release               # optimized Release build
dotnet run --project PeopleApp.Cli    # run the console app
dotnet test                           # run all tests in the solution
🔑 Debug vs Release

By default the CLI builds the Debug configuration — unoptimized, with full debugging info, for development. Pass -c Release for the Release configuration — optimized and stripped of debug overhead — which is what you ship. Same code, different build settings; always publish Release.

11

Hands-On Build

The Day-1 exercise: create a solution containing a console app and a class library, wire them together, and run it. This is our running example for the whole series.

The sequence 🧭

📁 1 · Create the solution dotnet new sln -n PeopleApp
💻 2 · Create the console app & the library dotnet new console -n PeopleApp.Cli · dotnet new classlib -n PeopleApp.Core
➕ 3 · Add both projects to the solution dotnet sln add …Cli.csproj · dotnet sln add …Core.csproj
🔗 4 · Make the app reference the library dotnet add …Cli.csproj reference …Core.csproj
▶️ 5 · Build & run dotnet build · dotnet run --project PeopleApp.Cli → Hello, Ada!
bash · the full script
# 1) A solution to hold everything
dotnet new sln -n PeopleApp

# 2) A runnable console app and a reusable library
dotnet new console  -n PeopleApp.Cli
dotnet new classlib -n PeopleApp.Core

# 3) Register both projects with the solution
dotnet sln add PeopleApp.Cli/PeopleApp.Cli.csproj
dotnet sln add PeopleApp.Core/PeopleApp.Core.csproj

# 4) The app depends on the library
dotnet add PeopleApp.Cli/PeopleApp.Cli.csproj reference PeopleApp.Core/PeopleApp.Core.csproj

# 5) Build the whole solution, then run the app
dotnet build
dotnet run --project PeopleApp.Cli
csharp · PeopleApp.Core/Greeter.cs
namespace PeopleApp.Core;

public class Greeter
{
    public string Greet(string name) => $"Hello, {name}!";
}
csharp · PeopleApp.Cli/Program.cs
using PeopleApp.Core;              // pull in the library's namespace

var greeter = new Greeter();
Console.WriteLine(greeter.Greet("Ada"));   // → Hello, Ada!
✅ What just happened

You built a two-project solution: PeopleApp.Core compiles to a DLL that holds reusable logic, and PeopleApp.Cli is the runnable app that references that DLL and calls into it. The using PeopleApp.Core; line makes the library's Greeter class visible. This library-plus-app split is the foundation of almost every real .NET codebase — keep this PeopleApp around; later days build on it.

12

Publishing

dotnet publish packages your app for deployment. The key choice is whether the target machine already has the .NET runtime.

bash
# Framework-dependent: small output; the machine must have the .NET runtime
dotnet publish -c Release

# Self-contained: bundles the runtime; runs anywhere, but larger output
dotnet publish -c Release -r linux-x64 --self-contained true
Framework-dependentSelf-contained
Includes the runtime?NoYes
Output sizeSmallLarger
Target needs .NET installed?YesNo
Best forServers/images with .NET presentMachines without .NET; isolated deploys
🔑 build vs publish

build produces assemblies for development (with dependencies scattered in bin/). publish produces a deployable folder — everything the app needs, arranged to copy to a server or container. You develop with build/run and ship with publish.

13

Mistakes & Practices

Day-1 habits that save pain later.

🎯

Target an LTS

New projects → net10.0 (current LTS) for the longest support window.

🙈

Ignore bin/ & obj/

Never commit build output; add the standard .NET .gitignore from the start.

📌

Pin the SDK

Use global.json so the team and CI build with the same SDK version.

🗂️

Split app from library

Keep reusable logic in a class library, thin runnable apps on top — easier to test and reuse.

📥

Commit references, not packages

Commit <PackageReference> lines; let restore fetch the files.

🚢

Publish in Release

Ship -c Release builds; Debug is for development only.

⚠️ "It builds on my machine" — pin your versions

Most reproducibility problems come from different SDK versions or unpinned package versions across machines. A global.json to pin the SDK and explicit package versions (or central package management, which later days cover) make builds identical everywhere — locally and in CI. Set this up early rather than debugging mystery differences later.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is .NET (Core)? Junior

A free, cross-platform, open-source developer platform for building many kinds of apps — console tools, web APIs, desktop, mobile, cloud. It includes a runtime that executes your code, a large set of libraries, languages like C#, and command-line tooling. "Core" was the name of versions 1–3.1; from version 5 it's just ".NET," but the same modern, cross-platform lineage.

Q02 What's the difference between the SDK and the runtime? Junior

The runtime runs finished .NET apps — it's what a production server needs. The SDK builds apps: it includes the compiler, the CLI, templates, MSBuild, and a runtime. Developers install the SDK (a superset); servers that only run apps need just the runtime.

Q03 What does dotnet build do? Junior

It compiles your project's source code into assemblies (DLLs and, for apps, an executable) placed in the bin/ folder, reporting any compiler errors and warnings. It restores NuGet packages first if needed, but it doesn't run the program.

Q04 What does dotnet run do? Junior

It builds the project if necessary and then launches the resulting app. It's the quick way to run a project during development. You can target a specific project with --project. For shipping, you'd use publish instead.

Q05 What is a .csproj file? Junior

The project file — XML that defines how a project is built: which .NET version it targets (TargetFramework), whether it's an app or a library (OutputType), and its package and project references. Modern SDK-style project files are short because they rely on SDK defaults and auto-include source files.

Q06 What is a solution (.sln)? Junior

A container that groups related projects so you can manage and build them together. Building the solution compiles all its projects in dependency order. A single project doesn't require a solution, but real applications usually organize several projects (app, library, tests) into one.

Q07 What is NuGet? Junior

.NET's package manager. It lets you pull in reusable libraries — published on nuget.org or a private feed — instead of writing everything yourself. You add a package with dotnet add package, which records a <PackageReference> in the .csproj; restore then downloads it and its dependencies.

Q08 What's the difference between a console app and a class library? Junior

A console app is runnable — it has an entry point and produces an executable. A class library is a reusable bundle of code (a DLL) with no entry point; it isn't run directly but is referenced by other projects. You typically put shared logic in a library and have thin apps reference it.

Q09 How do you add a project reference via the CLI? Mid

dotnet add <consumer.csproj> reference <dependency.csproj>. That writes a <ProjectReference> into the consuming project so it can use the other project's public types. For example, a console app referencing a class library so it can call the library's classes.

Q10 What is a target framework (TFM)? Mid

The Target Framework Moniker specifies which framework version your project compiles against — for example net10.0. It determines which APIs are available and which runtime the output expects. A project can even target multiple frameworks. It's set via <TargetFramework> (or <TargetFrameworks>, plural) in the .csproj.

Q11 What do ImplicitUsings and Nullable do in the .csproj? Mid

ImplicitUsings (enabled) auto-imports a set of common namespaces (like System) so you write fewer using statements. Nullable (enabled) turns on nullable-reference-type analysis, giving compiler warnings when you might dereference null — a big help for catching null-related bugs early. Both are on by default in new templates.

Q12 What's the difference between LTS and STS releases? Mid

Both are full-quality releases; they differ only in support length. LTS (Long-Term Support) releases come in odd-numbered years and get three years of patches — good for production stability. STS (Standard-Term Support) releases come in even-numbered years and get two years — good for adopting new features sooner. .NET 10 is the current LTS; .NET 9 is STS.

Q13 What does dotnet restore do, and when does it run automatically? Mid

It reads the project's package references and downloads those packages (and their transitive dependencies) into the local cache, writing restore data into obj/. It runs automatically as part of build, run, test, and publish, so you rarely call it directly — mainly in CI or to force a refresh.

Q14 What's in the bin and obj folders? Mid

bin/ holds the build output — the compiled DLLs and executables, organized by configuration and target framework. obj/ holds intermediate build artifacts and restore metadata. Both are generated from your source, so neither should be committed to source control; the standard .NET .gitignore excludes them.

Q15 What is global.json used for? Mid

It pins which .NET SDK version the CLI uses for a folder and its subfolders, so everyone on a team and the CI server build with the same SDK. Without it, the CLI picks the latest compatible installed SDK. It can also set roll-forward behavior. It's about build reproducibility, not the runtime the app targets (that's the TFM).

Q16 Framework-dependent vs self-contained deployment? Mid

A framework-dependent deployment ships just your app and relies on a .NET runtime already installed on the target — small output, but the machine needs the right runtime. A self-contained deployment bundles the runtime with your app — larger output, but it runs on a machine with no .NET installed. You choose with --self-contained and a runtime identifier (-r) at publish time.

Q17 Explain SDK-style projects versus the old .csproj format. Senior

Legacy .NET Framework project files explicitly listed every source file and were long and merge-conflict prone. SDK-style projects (introduced with .NET Core, declared via Sdk="Microsoft.NET.Sdk") implicitly include all source files in the folder, provide sensible defaults through the SDK, and express dependencies as concise <PackageReference> and <ProjectReference> items — so a project file is often just a dozen lines. They're also the basis for cross-platform builds and modern features like implicit usings.

Q18 How does the CLI decide which SDK version to use? Senior

By default it uses the highest installed SDK compatible with the command. If a global.json is present in the current directory or an ancestor, its sdk.version (and roll-forward policy) determines the SDK selected — pinning the exact version and failing if it's missing (depending on roll-forward settings). This lets teams guarantee everyone, including CI, builds with an identical SDK, which matters because SDK versions can change build behavior and defaults.

Q19 How does NuGet dependency resolution and transitive restore work? Senior

Each <PackageReference> pulls in that package plus its own dependencies, forming a graph. NuGet walks the graph and resolves a single version per package — by default choosing the lowest version that satisfies all constraints (nearest-wins/lowest-applicable), with direct references taking precedence over transitive ones. The result is written to a lock/assets file in obj/, and the packages are pulled from the global cache or downloaded. Version conflicts surface as warnings you resolve by adding an explicit reference or unifying versions.

Q20 What's the difference between .NET Framework, .NET Core, .NET Standard, and .NET 5+? Senior

.NET Framework is the original Windows-only platform (latest 4.8, still supported but not evolving). .NET Core was the cross-platform, open-source rewrite (1.0–3.1). .NET Standard was a set of API specifications that libraries could target to run on multiple runtimes (Framework, Core, Xamarin) — a compatibility bridge. .NET 5+ unified everything into one cross-platform product (dropping "Core" from the name), and is where all new development happens. New libraries usually target modern .NET; .NET Standard is now mainly for libraries that must also support old Framework apps.

Q21 How do PackageReference and ProjectReference differ, and how do transitive references flow? Senior

A <PackageReference> depends on a published NuGet package (by name and version); a <ProjectReference> depends on another project in your own solution, built from source. Both flow transitively by default: if app A references library B, and B references package/project C, then A also compiles against C without listing it. You can control this flow with attributes like PrivateAssets (to stop a dependency leaking to consumers) — useful for build-only or analyzer packages you don't want to propagate.

Q22 What happens during dotnet build versus dotnet publish? Senior

build compiles each project to assemblies in bin/<config>/<tfm>/ for development — dependencies are resolved but the layout isn't a finished deployment. publish runs a build and then gathers everything the app needs into a single publish folder arranged for deployment: your assemblies, dependencies, config, and — for self-contained or single-file/ trimmed/ReadyToRun options — the runtime and optimizations. So build is for the inner dev loop; publish produces the artifact you copy to a server or container image.

Q23 How do you manage package versions centrally across a solution? Senior

With Central Package Management: a Directory.Packages.props file at the repo root declares each package's version once via <PackageVersion>, and individual projects reference packages by name only (no version). This guarantees every project uses the same version, prevents drift and conflicts, and makes upgrades a one-line change. Related mechanisms include Directory.Build.props for shared MSBuild properties across projects. It's the standard way to keep a large multi-project solution consistent.

Q24 How would you structure a multi-project solution for a real app? Senior

Typically separate concerns into projects: a thin entry-point/host (e.g., the web API or console app), one or more class libraries for domain/business logic and data access, and dedicated test projects mirroring them — all grouped in one solution. Dependencies flow inward (the app references the libraries, not vice versa), keeping logic reusable and testable independent of the host. Shared build settings go in Directory.Build.props and versions in Directory.Packages.props. The exact split depends on the architecture, but the principle is: reusable logic in libraries, thin runnable hosts on top, tests alongside.
