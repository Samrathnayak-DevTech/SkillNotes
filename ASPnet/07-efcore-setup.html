🗄️ EF Core: Talking to SQL Server in C#

Databases speak SQL; your application speaks C# objects. Entity Framework Core is the translator that sits between them. Today you'll set up a DbContext, connect it to SQL Server, and learn the two ways of telling EF exactly how your classes map to tables — data annotations and the Fluent API.

🧠 ORM Concept 🗄️ DbContext 🔌 SQL Server Connection 🏷️ Data Annotations 🪄 Fluent API 🧬 Migrations

01🧠 What Is EF Core?

Entity Framework Core (EF Core) is Microsoft's ORM — an Object-Relational Mapper. That's a mouthful for a simple idea: it lets you work with database rows as ordinary C# objects, and it writes the SQL for you.

Without an ORM, saving a customer means hand-writing an INSERT statement, opening a connection, binding parameters, and copying results back into objects field by field. With EF Core, the same job is:

C# · Life with EF Core
// EF Core translates this into parameterized SQL, runs it,
// and fills in the generated Id — no SQL written by you.
var customer = new Customer { Name = "Priya Sharma", Email = "priya@example.com" };
db.Customers.Add(customer);
await db.SaveChangesAsync();

// Querying uses LINQ — EF turns it into a SELECT with a WHERE clause:
var recent = await db.Orders
    .Where(o => o.CustomerId == customer.Id)
    .OrderByDescending(o => o.PlacedAt)
    .ToListAsync();
🗣️ Analogy

EF Core is a professional interpreter between two people who speak different languages. You (C#) say "save this customer"; the interpreter turns it into fluent SQL for the database, listens to the reply, and translates it back into objects you understand. You could learn the other language and speak directly (raw SQL — sometimes worth it!), but for everyday conversation the interpreter is faster and makes fewer grammar mistakes.

The vocabulary you'll use all week

🧱Entity

A plain C# class that represents a table — e.g., a Customer class mapping to a Customers table. One object = one row.

🗄️DbContext

Your session with the database: it holds the entity sets, tracks changes, and turns SaveChanges into SQL. The heart of EF Core.

📇DbSet<T>

A property on the context representing one table — the thing you query (db.Customers) and add to.

🔌Provider

A NuGet package teaching EF one database's dialect: SQL Server, SQLite, PostgreSQL, Cosmos… Today: SQL Server.

🗺️The model

EF's internal map of how classes, properties, and relationships correspond to tables, columns, and foreign keys.

🧬Migration

A generated script of schema changes, so your database structure evolves in step with your C# classes.

🔑 Key Idea

EF Core doesn't replace SQL — it generates SQL. Everything it does ends up as ordinary SELECT/INSERT/UPDATE/DELETE statements against your database. Good EF developers can always answer: "what SQL will this produce?"

03🔌 Packages & the Connection String

Step 1 — install the provider 📦

EF Core ships as NuGet packages. The SQL Server provider pulls in the EF Core runtime with it:

Terminal
# The SQL Server provider (brings EF Core core packages along):
dotnet add package Microsoft.EntityFrameworkCore.SqlServer

# Needed later for migrations/scaffolding commands:
dotnet add package Microsoft.EntityFrameworkCore.Design

# The command-line tool itself (one-time, global):
dotnet tool install --global dotnet-ef
⚠️ Version Pairing

EF Core providers generally do not work across major versions — a provider built for EF Core 8 won't work with EF Core 9. Keep the provider's major version matched to your EF Core (and usually your .NET) version, and keep all EF packages on the same version number.

Step 2 — the connection string 🧾

A connection string is the database's address plus entry instructions: which server, which database, and how to prove who you are. It lives in configuration, never hard-coded:

JSON · appsettings.json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=localhost;Database=MyShop;Trusted_Connection=True;TrustServerCertificate=True"
  }
}

Dissecting the parts:

SegmentMeaningCommon values
Server=Which SQL Server instance to talk to.localhost · .\SQLEXPRESS · (localdb)\mssqllocaldb · a hostname
Database=Which database on that server.Your app's DB name, e.g. MyShop
Trusted_Connection=TrueLog in as the current Windows user (integrated security).Alternative: User Id=...;Password=... for SQL logins
TrustServerCertificate=TrueAccept the server's TLS certificate without validation — fine for local dev only.In production, use a real certificate instead
⚠️ The Classic First-Run Error

Recent versions of the underlying driver (Microsoft.Data.SqlClient) encrypt connections by default. Against a local SQL Server with no proper certificate you'll hit "The certificate chain was issued by an authority that is not trusted." The dev-machine fix is adding TrustServerCertificate=True — but never ship that to production, where certificate validation is your protection against man-in-the-middle attacks.

✅ Secrets Hygiene

A connection string with a password is a secret. Locally, keep it out of source control with user secrets (dotnet user-secrets set "ConnectionStrings:DefaultConnection" "..."); in production, use environment variables or a vault (e.g., Azure Key Vault). Configuration layering means the same GetConnectionString call works everywhere.

📮 Analogy

Think of the connection string as a courier's delivery instruction card: building address (Server), apartment number (Database), and the door code (credentials). You wouldn't print the door code on the outside of every parcel — that's why it lives in secrets, not in code.

04🗄️ DbContext & DbSet

The DbContext is your session with the database. You create a class inheriting from it, expose one DbSet<T> per table you care about, and give it a constructor that accepts options (so DI can configure it):

C# · Data/ShopContext.cs
using Microsoft.EntityFrameworkCore;

public class ShopContext : DbContext
{
    // The options carry the provider + connection string, supplied by DI.
    public ShopContext(DbContextOptions<ShopContext> options) : base(options) { }

    // One DbSet per table you want to work with:
    public DbSet<Customer> Customers => Set<Customer>();
    public DbSet<Product>  Products  => Set<Product>();
    public DbSet<Order>    Orders    => Set<Order>();
}

Registering it in Program.cs 🔧

C# · Program.cs
var connectionString = builder.Configuration.GetConnectionString("DefaultConnection")
    ?? throw new InvalidOperationException("Connection string 'DefaultConnection' not found.");

builder.Services.AddDbContext<ShopContext>(options =>
    options.UseSqlServer(connectionString,
        sql => sql.EnableRetryOnFailure()));   // retries transient network errors

AddDbContext registers your context with a scoped lifetime: in a web app, each HTTP request gets its own fresh ShopContext, which is disposed when the request ends. That maps perfectly onto EF's design — a context is meant to be a short-lived unit of work, not a long-lived shared object.

⚠️ Not Thread-Safe

A DbContext instance must never be used by two operations at the same time. That means: always await EF's async methods immediately (don't fire off two queries in parallel on one context), and never store a context in a singleton service. Scoped lifetime + one request at a time keeps you safe by default.

Using it in a service 💉

Because the context is in the DI container, any service can simply ask for it — the same pattern as ILogger<T> yesterday:

C# · Services/ProductService.cs
public class ProductService(ShopContext db, ILogger<ProductService> logger)
{
    public async Task<List<Product>> GetActiveAsync()
    {
        logger.LogInformation("Loading active products");
        return await db.Products
            .Where(p => p.IsActive)
            .OrderBy(p => p.Name)
            .ToListAsync();
    }
}

The change tracker: how SaveChanges knows what to do 🕵️

The context quietly watches every entity it hands out or receives. Each tracked entity is in one of five states, and SaveChanges generates SQL based on that state:

Added
→ INSERT
Unchanged
→ nothing
Modified
→ UPDATE
Deleted
→ DELETE
Detached
not tracked
EntityState · SaveChanges walks all tracked entities and emits SQL per state

LifetimeScoped (per request) Thread-safe?No Unit of workSaveChanges = 1 transaction

🔑 Key Idea

You don't tell EF how to update the database — you change objects and call SaveChangesAsync(). The change tracker computes the difference and wraps all resulting statements in a single transaction: either everything saves or nothing does.

05📐 Conventions First: What EF Assumes for Free

Before you write a single configuration line, EF Core builds a complete mapping from conventions — sensible rules applied to your class shapes. Meet today's three entities in their plainest form:

C# · Models (convention-only)
public class Customer
{
    public int Id { get; set; }              // convention: primary key + identity
    public string Name { get; set; } = "";   // non-nullable → NOT NULL column
    public string? Phone { get; set; }        // nullable → NULL column
    public List<Order> Orders { get; set; } = [];  // navigation → 1-to-many
}

public class Order
{
    public int Id { get; set; }
    public DateTime PlacedAt { get; set; }
    public int CustomerId { get; set; }        // convention: foreign key to Customer
    public Customer Customer { get; set; } = null!;
}

The convention cheat sheet

You write…EF assumes…
Property named Id or CustomerId (TypeName + Id)Primary key; for int/long, values generated by the database (identity).
DbSet<Customer> CustomersTable named Customers (the DbSet property name).
Property Name of type stringColumn Name, type nvarchar(max).
Non-nullable type (int, string with NRT on)NOT NULL column; nullable types (int?, string?) allow NULL.
Customer navigation + CustomerId propertyA one-to-many relationship with CustomerId as the foreign key, plus an index on it.
decimal propertySQL decimal(18,2) on SQL Server — often you'll want to set precision yourself.
🏨 Analogy

Conventions are like a hotel's default room setup — bed made, towels folded, thermostat at 22°. Most guests never call reception. Data annotations are quick requests on the door card ("extra pillow, please"). The Fluent API is calling the manager to rearrange the furniture. Same room, three levels of customization.

⚠️ Defaults Aren't Always Right

Two conventions bite people in production: string → nvarchar(max) (unbounded columns hurt indexing and invite junk data) and decimal → decimal(18,2) (may silently truncate values needing more scale — EF warns about this). Fixing those two is often your first configuration job.

06🏷️ Data Annotations

Data annotations are attributes placed directly on your entity classes and properties — small square-bracketed labels that override conventions where they stand:

C# · Models/Product.cs (annotated)
using System.ComponentModel.DataAnnotations;
using System.ComponentModel.DataAnnotations.Schema;

[Table("Products", Schema = "catalog")]       // map to catalog.Products
public class Product
{
    [Key]                                       // explicit primary key
    public int Id { get; set; }

    [Required]                                  // NOT NULL (matters pre-NRT)
    [MaxLength(120)]                            // nvarchar(120), not (max)
    public string Name { get; set; } = "";

    [Column("UnitPrice", TypeName = "decimal(10,2)")]  // rename + exact SQL type
    public decimal Price { get; set; }

    [MaxLength(40)]
    public string? Sku { get; set; }

    [NotMapped]                                 // exists in C# only, no column
    public string DisplayLabel => $"{Name} (₹{Price})";
}

The annotations you'll actually use

AttributeEffect on the databaseNotes
[Key]Marks the primary key.Only needed when the name isn't Id/TypeId.
[Required]Column becomes NOT NULL.With nullable reference types on, non-nullable string already implies this.
[MaxLength(n)] / [StringLength(n)]nvarchar(n) instead of nvarchar(max).Also used by input validation — double duty.
[Table("Name", Schema="s")]Maps the class to a specific table/schema.Essential when matching an existing database.
[Column("Name", TypeName="...")]Renames a column / sets its exact SQL type.Great for legacy column names.
[Precision(10, 2)]Sets decimal precision/scale.Cleaner than TypeName for decimals.
[NotMapped]Property gets no column at all.For computed, C#-only helpers.
[ForeignKey("Nav")]Declares which property is the FK for a navigation.Only when names don't follow convention.
✅ Strength

Annotations are self-documenting: anyone reading the entity sees its rules right there. Several (like [Required] and [MaxLength]) also feed ASP.NET Core's model validation, so one attribute guards both the API input and the database column.

⚠️ Limitation

Annotations can't express everything: composite keys, most index definitions, relationship delete behavior, value conversions, and many mapping tricks are Fluent-API-only. They also couple your entity classes to persistence concerns — which some teams prefer to keep out of domain models.

07🪄 The Fluent API

The Fluent API configures the model in code, inside your context's OnModelCreating method, by chaining method calls (hence "fluent"). It can do everything annotations can — and everything they can't:

C# · ShopContext.OnModelCreating
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Customer>(e =>
    {
        e.ToTable("Customers");
        e.HasKey(c => c.Id);

        e.Property(c => c.Name)
            .IsRequired()
            .HasMaxLength(120);

        e.HasIndex(c => c.Email)          // unique index — annotations can't do this well
            .IsUnique();
    });

    modelBuilder.Entity<Order>(e =>
    {
        e.ToTable("Orders");

        e.Property(o => o.Total)
            .HasPrecision(10, 2);

        e.HasOne(o => o.Customer)          // relationship, spelled out:
            .WithMany(c => c.Orders)
            .HasForeignKey(o => o.CustomerId)
            .OnDelete(DeleteBehavior.Restrict);  // don't cascade-delete orders
    });
}
🪄 Analogy

If annotations are sticky notes on each piece of furniture, the Fluent API is the architect's master blueprint kept in one office. The furniture (entities) stays clean and label-free; every layout decision lives in a single, authoritative document that can express far more detail.

Keeping it tidy: IEntityTypeConfiguration 🗂️

With many entities, OnModelCreating becomes a monster. The standard fix: one small configuration class per entity, auto-discovered by assembly scanning:

C# · Data/Configurations/ProductConfiguration.cs
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

public class ProductConfiguration : IEntityTypeConfiguration<Product>
{
    public void Configure(EntityTypeBuilder<Product> builder)
    {
        builder.ToTable("Products", "catalog");

        builder.Property(p => p.Name)
            .IsRequired()
            .HasMaxLength(120);

        builder.Property(p => p.Price)
            .HasColumnName("UnitPrice")
            .HasPrecision(10, 2);

        builder.HasIndex(p => p.Sku).IsUnique();
    }
}

// In ShopContext.OnModelCreating — one line finds every configuration class:
modelBuilder.ApplyConfigurationsFromAssembly(typeof(ShopContext).Assembly);

Coverageeverything Runsonce per app (model is cached) Best practice1 config class per entity

🔑 Fluent-Only Superpowers

Reach for the Fluent API when you need: composite keys (HasKey(x => new { x.A, x.B })), unique or filtered indexes, delete behavior (Cascade/Restrict/SetNull), value conversions (store an enum as a string), seed data (HasData), and global query filters (auto-apply "not deleted" to every query).

08⚖️ Annotations vs Fluent: Choosing & Precedence

Both techniques feed the same model, so the real questions are which wins a conflict and which should your team prefer. The precedence rule is simple — more explicit beats less explicit:

✅ A Sane Team Policy

Pick one primary home for configuration and stay consistent. A common, pragmatic split: use annotations for simple, validation-adjacent facts ([MaxLength], [Required]) and the Fluent API for all real mapping decisions (tables, keys, relationships, indexes, precision). Many teams go further and put everything in Fluent configuration classes — one obvious place to look.

09🧬 Getting a Database: Migrations & Scaffolding

You've described the model — now the database schema must match it. There are two directions, depending on which side already exists:

🧑‍🎨Code First → new tables

Your C# classes are the source of truth. Migrations generate and apply SQL to create/evolve the schema. Best for new projects.

🏛️Database First → existing tables

An existing database is the source of truth. Scaffolding (reverse engineering) generates entity classes and a context from it.

The migrations loop (Code First) 🔁

✏️
1. Change the modelAdd a property, a new entity, or a Fluent rule — e.g., give Product a Sku.
🧬
2. Add a migrationdotnet ef migrations add AddProductSku — EF compares the new model with a snapshot of the old one and writes C# Up/Down steps.
🔍
3. Review the generated codeOpen the migration file and read it. Renames can be misread as drop-and-recreate — catching that here saves your data.
🚀
4. Update the databasedotnet ef database update — EF runs pending migrations as SQL and records each in the __EFMigrationsHistory table.
✅
5. Commit migration + model togetherMigrations are code: version-controlled, repeatable on teammates' machines and in deployment pipelines.
Terminal · Code First commands
# Create the first migration (generates Migrations/xxxx_InitialCreate.cs):
dotnet ef migrations add InitialCreate

# Apply all pending migrations to the connected database:
dotnet ef database update

# Peek at the SQL a migration will run (great for review):
dotnet ef migrations script

# Made a mistake before applying? Remove the last migration:
dotnet ef migrations remove

Scaffolding an existing database (Database First) 🏛️

Terminal · Reverse engineering
# Generate entities + context from an existing SQL Server database:
dotnet ef dbcontext scaffold \
  "Server=localhost;Database=LegacyShop;Trusted_Connection=True;TrustServerCertificate=True" \
  Microsoft.EntityFrameworkCore.SqlServer \
  --output-dir Models --context-dir Data --context LegacyShopContext \
  --tables Customers --tables Orders --tables Products

Scaffolding writes classes whose annotations/Fluent config mirror the real tables — column names, lengths, keys, and relationships included. From there you can keep re-scaffolding when the DB changes, or adopt the generated code and switch to migrations.

⚠️ Mapping to Existing Tables by Hand

When you write entities against an existing database yourself (today's exercise!), the names must line up exactly: use [Table]/ToTable and [Column]/HasColumnName wherever your C# names differ from the real table and column names. A mismatch shows up at runtime as "Invalid object name" or "Invalid column name" errors from SQL Server.

✅ Sanity-Check the Mapping

Fastest verification loop: run the app with EF's SQL visible (Day 6 skill! set Microsoft.EntityFrameworkCore.Database.Command to Information) and read the generated SELECT — the table and column names EF uses are printed right there.

10🛠️ Hands-On: Map 2–3 Entities to Tables

Today's Do: map two or three entities to tables — new ones via a migration, or existing ones if you already have a database. Suggested path:

  1. 📦 Install Microsoft.EntityFrameworkCore.SqlServer + Microsoft.EntityFrameworkCore.Design, and the dotnet-ef tool.
  2. 🧱 Create 2–3 entities with a relationship between two of them — e.g., Customer, Order (many orders per customer), and Product.
  3. 🗄️ Create a ShopContext with one DbSet<T> per entity; register it with AddDbContext + UseSqlServer, reading the connection string from configuration.
  4. 🏷️ Configure one entity with data annotations ([MaxLength], [Required], [Table]) — feel how it reads.
  5. 🪄 Configure another with a Fluent IEntityTypeConfiguration<T> class — set a max length, a HasPrecision on a decimal, a unique index, and the relationship with OnDelete(DeleteBehavior.Restrict).
  6. 🧬 New tables: dotnet ef migrations add InitialCreate, read the generated migration, then dotnet ef database update. Existing tables: match names with ToTable/HasColumnName instead, no migration.
  7. 🔎 Prove it works: insert one row of each entity and query them back; open the database (SSMS / Azure Data Studio) and look at the created columns, types, and the FK index.
  8. 📋 Bonus (Day 6 crossover): log "Created customer {CustomerId}" after SaveChangesAsync — the generated ID is already populated on the object.
C# · Minimal end-to-end smoke test
// e.g. in a temporary minimal-API endpoint:
app.MapPost("/seed", async (ShopContext db) =>
{
    var customer = new Customer { Name = "Priya Sharma", Email = "priya@example.com" };
    customer.Orders.Add(new Order { PlacedAt = DateTime.UtcNow, Total = 1499.50m });

    db.Customers.Add(customer);          // the related Order is tracked too
    await db.SaveChangesAsync();          // 2 INSERTs, 1 transaction

    return Results.Ok(new { customer.Id });
});

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is an ORM, and what problem does EF Core solve?🌱 JUNIOR

An ORM (Object-Relational Mapper) is a translator between two worlds: object-oriented code (classes, properties, references) and relational databases (tables, columns, foreign keys). EF Core lets you read and write database rows as ordinary C# objects, writing the SQL for you. It removes the repetitive plumbing of hand-written SQL, connections, and copying values between rows and objects — while still producing normal SQL underneath.

Q2What is a DbContext and what are its main jobs?🌱 JUNIOR

A DbContext is your session with the database. Its jobs: expose DbSet<T> properties you query and add entities to; translate LINQ queries into SQL through the provider; track changes to the entities it knows about; and, when you call SaveChanges, write all pending inserts, updates, and deletes to the database in one transaction. Think of it as a short-lived "unit of work" — created, used, disposed.

Q3What is a DbSet<T>?🌱 JUNIOR

A DbSet<T> is the context's handle to one table. db.Customers is where you start queries (Where, OrderBy, ToListAsync) and where you register new rows (Add) or removals (Remove). By convention, the DbSet property's name also becomes the default table name.

Q4How does EF Core decide which property is the primary key without any configuration?🌱 JUNIOR

By convention: a property named Id or <TypeName>Id (like CustomerId on the Customer class) is treated as the primary key automatically. If it's an int or long, EF also assumes the database generates the value (an identity column). If your key has a different name, you mark it with [Key] or configure it with HasKey in the Fluent API.

Q5What is a connection string and where should it be stored?🌱 JUNIOR

It's the set of instructions for reaching the database: which server, which database, and how to authenticate. It should live in configuration, never hard-coded: appsettings.json for harmless local settings, user secrets for anything with credentials during development, and environment variables or a secret vault in production. Code just calls Configuration.GetConnectionString("Name") and works in every environment.

Q6What are data annotations? Give three examples.🌱 JUNIOR

Data annotations are attributes written directly on entity classes and properties that adjust how EF maps them. Examples: [Required] makes the column NOT NULL; [MaxLength(120)] limits a string column to 120 characters instead of unlimited; [Table("Products")] maps the class to a specific table name. Others include [Key], [Column], and [NotMapped].

Q7What is the Fluent API in one sentence, and where does it live?🌱 JUNIOR

The Fluent API is EF Core's code-based configuration language — chained method calls like Property(p => p.Name).IsRequired().HasMaxLength(120) — and it lives in the context's OnModelCreating(ModelBuilder) method (often split into one IEntityTypeConfiguration<T> class per entity for tidiness).

Q8What is a migration?🌱 JUNIOR

A migration is a generated, versioned recipe for changing the database schema so it matches your current C# model. When you run dotnet ef migrations add SomeName, EF compares your model to a snapshot of the previous one and writes the difference as code (with Up and Down steps). dotnet ef database update then applies pending migrations, and EF records which ones ran in a history table so they're never applied twice.

🟠 Mid

Q9Why is DbContext registered with a scoped lifetime, and why would singleton be wrong?🟠 MID

Scoped means one context per HTTP request — each request gets a fresh, isolated unit of work that's disposed when the request ends. That fits DbContext's design: it accumulates tracked entities and isn't thread-safe. A singleton context would be shared by all concurrent requests, causing two problems: concurrent access (crashes or corrupted state, since one instance mustn't run parallel operations) and unbounded growth of the change tracker, effectively a memory leak with increasingly stale data.

Q10What happens, step by step, when you call SaveChangesAsync?🟠 MID

The change tracker walks every entity the context knows and checks its state: Added, Modified, Deleted, or Unchanged. For each non-Unchanged entity it generates the matching SQL (INSERT, UPDATE with only the changed columns, DELETE), orders the statements to respect foreign-key dependencies, wraps them in a single transaction, and executes them. Database-generated values (like new identity IDs) are read back and written onto your objects, and the saved entities are reset to Unchanged. If anything fails, the transaction rolls back and nothing is persisted.

Q11Conventions, annotations, and Fluent API all configure the model. Who wins a conflict?🟠 MID

Precedence goes from least to most explicit: conventions are the baseline defaults, data annotations override conventions, and the Fluent API overrides both. So if an attribute says [MaxLength(50)] but Fluent configuration says HasMaxLength(120), the column ends up as 120. Practical takeaway: when debugging a surprising mapping, check Fluent configuration first — it has the final word.

Q12Name three things you can only do with the Fluent API, not with annotations.🟠 MID

Common Fluent-only capabilities: composite primary keys (HasKey(x => new { x.OrderId, x.ProductId })); configuring relationship delete behavior (OnDelete(DeleteBehavior.Restrict) vs cascade); unique or filtered indexes via HasIndex(...).IsUnique(). Others include value conversions (enum ↔ string), seed data with HasData, and global query filters.

Q13How do you map an entity to an existing table whose names don't match your C# names?🟠 MID

Override the naming conventions explicitly. Point the class at the real table with [Table("tbl_customer", Schema = "sales")] or ToTable("tbl_customer", "sales"), and map individual properties to real column names with [Column("cust_name")] or HasColumnName("cust_name"). Also match types and nullability (HasColumnType, IsRequired). If names don't line up, you'll get runtime errors like "Invalid object name" or "Invalid column name" from SQL Server.

Q14What is scaffolding (reverse engineering) and when do you use it?🟠 MID

Scaffolding generates C# entity classes and a DbContext from an existing database: dotnet ef dbcontext scaffold "<connection>" Microsoft.EntityFrameworkCore.SqlServer. EF reads the tables, columns, keys, and foreign keys and writes matching code, including the configuration needed to mirror them exactly. You use it when the database already exists and is the source of truth — legacy systems, DBA-managed schemas — instead of the code-first migrations flow.

Q15How does EF Core's convention treat a C# string property, and why can that be a problem?🟠 MID

By default a string becomes nvarchar(max) — an unlimited-length Unicode column — nullable or not depending on nullable reference types. Unlimited columns are a problem because they can't be efficiently indexed, invite unbounded junk input, and hide the real business rule ("a name is at most ~120 characters"). The fix is one line: [MaxLength(120)] or HasMaxLength(120). Similarly, decimal defaults to decimal(18,2), which may silently truncate values needing more scale — set HasPrecision deliberately for money and measurements.

Q16What does EnableRetryOnFailure do and why enable it?🟠 MID

It turns on an execution strategy that automatically retries operations that fail with known transient errors — brief network blips, database failovers, throttling (especially common with cloud databases like Azure SQL). Instead of one unlucky packet loss becoming a user-facing 500 error, EF waits briefly and retries a few times. You enable it in the provider options: UseSqlServer(cs, sql => sql.EnableRetryOnFailure()). One caveat: with a retry strategy, user-initiated transactions need special handling since the whole unit must be retriable.

🔴 Senior

Q17Your team debates annotations vs Fluent API. What policy do you propose, and why?🔴 SENIOR

The goal is predictability: a developer must know where to look for mapping truth. My default proposal: Fluent configuration classes as the primary home (one IEntityTypeConfiguration<T> per entity, applied via ApplyConfigurationsFromAssembly), because they cover 100% of EF's features, keep persistence concerns out of domain classes, and win precedence anyway — so nothing can silently override them. Annotations remain acceptable for validation-adjacent facts that ASP.NET also uses ([Required], [MaxLength]) if the team values that dual use. The anti-pattern to ban is mixing both for the same concern on the same entity — that's where "why is this column 120 chars?" archaeology begins.

Q18Walk me through how you'd manage migrations safely across a team and in deployment.🔴 SENIOR

Team side: migrations are code — committed together with the model change, reviewed in PRs (actually reading the Up method — renames can scaffold as drop-and-recreate, which destroys data), and kept linear by regenerating when two branches both added migrations. Deployment side: don't auto-run Database.Migrate() from every app instance at startup in a multi-instance production environment (race conditions, surprise locks at boot). Prefer generating an idempotent SQL script (dotnet ef migrations script --idempotent) or a migrations bundle, reviewed and applied as an explicit deployment step by the pipeline, with a backup taken first and destructive changes split into safe multi-release phases (add new column → migrate data → drop old column).

Q19How do you decide the delete behavior (cascade vs restrict) for a relationship?🔴 SENIOR

Ask: if the parent disappears, is the child meaningless or valuable evidence? Cascade fits true composition — order lines without their order are meaningless, so deleting the order deleting its lines is correct. Restrict (or SetNull) fits association with independent value — deleting a customer should not silently erase their order history, which may be needed for accounting or legal reasons; better to block the delete or use soft-delete flags. EF's convention cascades required relationships by default, so I set OnDelete explicitly for every relationship rather than inheriting a default someone will discover in production. SQL Server adds a wrinkle: it rejects schemas with multiple cascade paths, which forces Restrict in diamond-shaped models anyway.

Q20What is the difference between AddDbContext and AddDbContextPool, and when does pooling matter?🔴 SENIOR

Both give you a scoped context per request. AddDbContextPool additionally recycles instances: instead of constructing and disposing a context object per request, finished contexts are reset and returned to a pool (default max 1024) for reuse, cutting allocation and setup cost. It matters in high-throughput APIs where thousands of contexts per second are created. Constraints: the context must be stateless apart from EF's own machinery — no per-request fields set in the constructor, since instances are reused across requests. For most apps it's a safe, cheap win; for low-traffic apps the difference is negligible.

Q21Where do value conversions fit, and what trade-offs do they carry?🔴 SENIOR

A value conversion (Fluent: Property(...).HasConversion(...)) translates between the C# type and the stored type — classic uses: enum stored as a readable string, a strongly-typed ID wrapper stored as an int, encrypted-at-rest strings. Trade-offs: the database only sees the converted value, so queries can only use operations valid on that stored form (an enum-as-string orders alphabetically, not by enum value); some LINQ constructs can't translate and will fail or fall back; and indexes work on the stored representation. Rule of thumb: conversions are for representation, not for hiding heavy logic — keep them cheap, deterministic, and both-ways.

Q22The app throws "Invalid column name 'X'" in production but works locally. Diagnose.🔴 SENIOR

That error means EF's model and the actual database disagree — EF generated SQL referencing a column the target database doesn't have. Systematic diagnosis: (1) confirm which database production actually connects to (connection string per environment — very often the real culprit is pointing at an old or wrong DB); (2) check pending migrations: does __EFMigrationsHistory on that database contain the latest migration? If not, the deployment step that applies migrations didn't run; (3) check for drift — did someone hand-edit the production schema or does a scaffolded model no longer match after a DBA change; (4) reproduce by pointing a local run (read-only!) at a copy of the production schema and comparing dotnet ef migrations script output against reality. Fix the process, not just the column: schema changes must flow through migrations applied by the pipeline.

Q23When would you NOT use EF Core, or bypass it within an EF-based app?🔴 SENIOR

EF earns its keep for typical transactional CRUD with a rich object model. I'd bypass or skip it when: queries are heavily set-based/analytical (complex reporting, window functions, bulk aggregation) where hand-written SQL or a lightweight mapper like Dapper is clearer and faster; for bulk operations on many thousands of rows (though EF's ExecuteUpdate/ExecuteDelete now cover many set-based cases without loading entities); when the schema is exotic or DBA-controlled with heavy stored-procedure conventions; or in ultra-hot paths where materialization overhead measurably matters. These aren't either/or — a common architecture is EF for commands and workflows, raw SQL (via FromSql or Dapper) for the handful of read paths that need it.

Q24How do you keep EF Core from becoming a performance problem as an app grows?🔴 SENIOR

Principles I enforce early: know your SQL (log Database.Command at Information in dev and read what EF generates); use AsNoTracking for read-only queries so the change tracker doesn't pay for entities you'll never modify; project with Select into DTOs instead of loading whole entities with all columns; watch for the N+1 pattern (a query in a loop) and fix it with Include or better, a single projected query; use pagination rather than unbounded ToList; consider split queries for wide Include trees; add the indexes your real query patterns need via HasIndex; and enable context pooling on hot services. Most "EF is slow" incidents are actually "we accidentally asked for 40,000 rows and tracked them all."
