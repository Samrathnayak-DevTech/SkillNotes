Day 10 · ASP.NET Core Fundamentals

⏳ Async/Await: Waiting Without Wasting

You've written ToListAsync and SaveChangesAsync all week. Today you'll understand why that Async suffix matters — how Task, async, and await let a server serve thousands of users on a handful of threads, how to cancel work that's no longer needed, and how one careless .Result can freeze your whole application.

🧵 Threads & Waiting 📦 Task<T> ⏳ async / await 🚦 CancellationToken 💥 .Result Deadlocks

01🍳 Why Async Exists

Your web server has a limited pool of threads — worker "hands" that execute code. Every incoming request borrows one. The trouble is that most of what a request does is wait: wait for the database (Days 7–9!), wait for an HTTP call, wait for a file. During that wait, the thread does nothing useful — but it's still held.

🍳 Analogy

Picture a chef (a thread) who puts a cake in the oven and then stands there staring at the oven for 40 minutes, ignoring every other order. That's synchronous code. An asynchronous chef puts the cake in, sets a timer, and immediately starts chopping vegetables for other dishes — returning to the cake only when the timer dings. Same oven time; one chef feeds the whole restaurant, the other feeds one table.

Async programming is about that second chef. When code hits an await on a database call, the thread isn't parked staring at the oven — it's returned to the pool to serve other requests. When the database replies, the code resumes on an available thread. The single query took the same wall-clock time, but the server did far more work in parallel.

🔑 Key Idea

Async is not about speed — one operation isn't faster. It's about scalability: freeing threads during I/O waits so a small pool can handle a large load. This is why every I/O method in modern .NET has an async version, and why ASP.NET Core is async from top to bottom.

⚠️ I/O-bound vs CPU-bound

Async shines for I/O-bound work (waiting on something external — DB, network, disk). For CPU-bound work (heavy calculation that keeps a core busy), await alone helps nothing — there's no "waiting" to release, only computing. That's a job for Task.Run / parallelism, a different tool. Today is about I/O.

The through-line: Task is the data type for a future result, async/await is the syntax for working with it readably, and the pitfalls are all variations on one sin — blocking a thread instead of awaiting.

03📦 What a Task Actually Is

A Task is a promise: an object that represents work that will finish later and, in the case of Task<T>, will eventually hold a value of type T. It's not the work itself and not the result — it's the receipt you hold while you wait.

🎫 Analogy

A Task is a coat-check ticket. You hand over your coat (start the work) and get a numbered stub. The stub isn't the coat — but it's your claim on it. Later you present the stub (await) and receive the coat (the result). A Task<Order> is a ticket that will get you an Order; a plain Task is a ticket for work with no return value — proof it happened, nothing to collect.

A Task has a status 🚥

Every task moves through a small lifecycle. Understanding it demystifies exceptions and cancellation later:

Task lifecycle · a completed task carries its outcome — a value, an exception, or cancellation
C# · Task types you'll return
// Task<T> — async work that produces a value:
public Task<Order> GetOrderAsync(int id) { ... }

// Task — async work with no return value (the async "void" done right):
public Task SaveOrderAsync(Order o) { ... }

// ValueTask<T> — an optimization for hot paths that often complete
// synchronously (e.g., a cache hit). Rare in everyday code; don't reach
// for it by default, and never await a ValueTask more than once.
public ValueTask<Order> GetCachedOrderAsync(int id) { ... }
🔑 Key Idea

A method that returns Task has usually already started the work by the time it hands you the task — the task is "hot." Your job is to await it to get the result (or the exception) out. Holding a Task without awaiting it means the work may still be running unobserved.

04⏳ async & await

The async keyword marks a method as one the compiler should rewrite into a state machine. The await keyword marks the pause points. Together they let you write asynchronous code that reads like ordinary top-to-bottom code:

C# · Anatomy of an async method
public async Task<OrderDto> GetOrderSummaryAsync(int id)
{
    // Runs synchronously on the calling thread up to the first await:
    logger.LogInformation("Loading order {OrderId}", id);

    // await: kick off the DB call, then RELEASE the thread while waiting.
    Order? order = await db.Orders.FindAsync(id);

    // ...resumes here (possibly on a DIFFERENT thread) once data arrives:
    if (order is null) throw new NotFoundException(id);

    return new OrderDto(order.Id, order.Total);
}

What await really does 🎬

▶️
1. Runs synchronously until the first awaitAn async method starts on the caller's thread and executes normally right up to the first await.
🎫
2. Reaches await on an incomplete taskIf the awaited task isn't finished, the method hands a Task back to its caller and stops here.
🔓
3. Releases the threadThe thread returns to the pool to do other work. Nothing is blocked. This is the entire point.
🔔
4. The task completesWhen the I/O finishes, the runtime schedules the rest of the method (the "continuation") to run.
↩️
5. Resumes after the awaitExecution picks up on the next line — on a thread-pool thread in ASP.NET Core — with the result (or a thrown exception) in hand.
⚠️ async Doesn't Mean "New Thread"

A common myth: writing async spins up a background thread. It doesn't. For I/O, there's often no thread at all during the wait — the operating system notifies .NET when the data is ready. Async is about not occupying a thread while waiting, not about adding threads.

✅ Naming & Signatures

Convention: async methods end in Async and return Task or Task<T>. A method with async but no await inside is a code smell (it warns at compile time). And await can only appear inside a method marked async — the two travel together.

05🔝 Async All the Way

The single most important rule: once you go async, go async everywhere up the call stack. An async method should be awaited by an async caller, awaited by its async caller, all the way to the framework entry point (your controller action or minimal-API handler, which ASP.NET Core awaits for you).

🔗 Analogy

Async-all-the-way is a relay race where the baton is passed cleanly. The moment one runner stops to tie their shoe in the middle of the track (a blocking .Result), the whole team stalls behind them — it doesn't matter how fast everyone else runs. One blocking call in the chain wastes a thread and can starve the pool under load.

C# · The chain, done right
// Endpoint (ASP.NET Core awaits this for you):
app.MapGet("/orders/{id}", async (int id, OrderService svc, CancellationToken ct) =>
    await svc.GetSummaryAsync(id, ct));

// Service:
public async Task<OrderDto?> GetSummaryAsync(int id, CancellationToken ct)
    => await _repo.FindSummaryAsync(id, ct);

// Repository → EF Core → database. Async, unbroken, top to bottom.
public async Task<OrderDto?> FindSummaryAsync(int id, CancellationToken ct)
    => await _db.Orders
        .Where(o => o.Id == id)
        .Select(o => new OrderDto(o.Id, o.Total))
        .FirstOrDefaultAsync(ct);
🔑 Return the Task When You Can

If a method's only job is to call one async method and return its result, you can skip async/await and just return the Task directly (as in the service above written with =>). It's a small efficiency — no state machine generated. But keep the await if you need a using block, a try/catch, or work after the call.

06🔀 Running Work Concurrently: WhenAll

Awaiting one task at a time is sequential — fine when step B needs step A's result. But when operations are independent, awaiting them one by one wastes time. Task.WhenAll starts them together and awaits the whole batch:

C# · Sequential vs concurrent
// ⏱️ Sequential: total time ≈ A + B + C (each waits for the previous)
var weather = await _weatherApi.GetAsync(city, ct);
var news    = await _newsApi.GetAsync(city, ct);
var events  = await _eventsApi.GetAsync(city, ct);

// ⚡ Concurrent: total time ≈ the SLOWEST of the three.
// Start all three (don't await yet), then await them together:
Task<Weather> wTask = _weatherApi.GetAsync(city, ct);
Task<News>    nTask = _newsApi.GetAsync(city, ct);
Task<Events>  eTask = _eventsApi.GetAsync(city, ct);

await Task.WhenAll(wTask, nTask, eTask);

var dashboard = new Dashboard(wTask.Result, nTask.Result, eTask.Result);
// ↑ .Result is SAFE here: WhenAll already guaranteed they finished.
⚠️ Not on One DbContext!

This concurrency pattern is for independent operations — separate HTTP clients, separate contexts. It does not apply to a single EF Core DbContext, which forbids parallel operations (Days 7–8). Two un-awaited EF queries in a WhenAll on the same context throws. For parallel database work, give each task its own context via IDbContextFactory.

✅ Cousins of WhenAll

Task.WhenAny completes when the first task finishes (useful for timeouts or "fastest wins"). Note the difference from the blocking, thread-wasting Task.WaitAll/WaitAny — prefer the When* versions, which you await. And when using WhenAll, if multiple tasks fault, the awaited exception is the first one; inspect the tasks individually if you need them all.

07🚦 Cancellation Tokens

Work sometimes becomes pointless before it finishes: the user closes the browser, a timeout elapses, a newer request supersedes this one. A CancellationToken is how you ask running async work to stop, cleanly and cooperatively.

🛎️ Analogy

A cancellation token is a "never mind" signal to the kitchen. The customer leaves; the waiter tells the chef "table 4 walked out." A cooperative chef checks that note between steps and stops cooking, freeing the stove. The token doesn't yank the pan off the heat by force — the chef has to look. Code that never checks its token keeps cooking a meal no one will eat.

How it flows through your app 🌊

ASP.NET Core creates a token for each request and cancels it if the client disconnects. You accept it in your handler and thread it down through every async call — EF Core, HttpClient, and your own methods all take one:

C# · Threading a token top to bottom
// The framework hands you the request's token automatically:
app.MapGet("/reports/sales", async (SalesService svc, CancellationToken ct) =>
    await svc.BuildReportAsync(ct));

public async Task<Report> BuildReportAsync(CancellationToken ct)
{
    // Pass ct to EVERY async call — EF forwards it to SQL Server, which
    // stops executing the query if the request is abandoned:
    var rows = await _db.Sales
        .Where(s => s.Year == 2026)
        .ToListAsync(ct);

    foreach (var batch in Chunk(rows))
    {
        // In your own loops, check the token between chunks of work:
        ct.ThrowIfCancellationRequested();
        await ProcessAsync(batch, ct);
    }

    return Summarize(rows);
}

Creating your own tokens ⏲️

C# · Timeouts and linked tokens
// A timeout: cancel automatically after 5 seconds.
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));
var data = await _api.GetAsync(url, cts.Token);

// Combine the request's token with your timeout — cancels if EITHER fires:
using var linked = CancellationTokenSource
    .CreateLinkedTokenSource(requestToken, cts.Token);
var result = await _api.GetAsync(url, linked.Token);

When cancellation fires, the awaited call throws an OperationCanceledException (or its subclass TaskCanceledException). That's expected control flow, not a bug — the operation stopping is the whole point.

✅ Habits

(1) Accept a CancellationToken parameter on every async method that does I/O, conventionally last, often defaulted to default. (2) Pass it onward to every async call you make. (3) In long CPU loops, call ThrowIfCancellationRequested() periodically. (4) Don't catch OperationCanceledException just to swallow it — usually let it bubble; the framework treats a canceled request gracefully.

⚠️ A Token You Ignore Does Nothing

Cancellation is cooperative. Accepting a token and then never passing or checking it means the work runs to completion regardless — the "never mind" note sits unread. The most common cancellation bug isn't misuse; it's a token quietly dropped halfway down the call chain.

08🧯 Error Handling in Async Code

Good news: try/catch works exactly as you'd hope. When an awaited task faults, await re-throws the exception at the await point, so you catch it like synchronous code:

C# · try/catch across await
try
{
    var order = await _repo.GetAsync(id, ct);   // exception surfaces HERE
    await _payments.ChargeAsync(order, ct);
}
catch (OperationCanceledException)
{
    // Expected when the request is abandoned — usually just let it propagate,
    // or log at a low level. Don't treat it as a failure.
    throw;
}
catch (PaymentException ex)
{
    logger.LogError(ex, "Payment failed for order {OrderId}", id);  // Day 6!
    throw;
}
⚠️ The Silent Killer: async void

An async void method has no Task to carry its exception — so a throw inside one can't be caught by the caller and typically crashes the whole process. The only legitimate use of async void is a top-level event handler (rare in web apps), and even then you wrap the body in try/catch. Rule: async methods return Task, never void — this is the single most important async signature rule.

⚠️ Fire-and-Forget Loses Exceptions Too

Calling an async method without awaiting it — DoWorkAsync(); on its own line — starts the work but discards the task, so any exception vanishes unobserved and the work may outlive the request. If you truly need background work, use a proper mechanism (a hosted service / Channel / background queue), not a dropped task. The compiler warns you (CS4014); heed it.

✅ Exceptions Are Stored, Then Re-Thrown

Remember the task lifecycle (section 03): a faulted task holds its exception until observed. await is what observes it and re-throws. That's why an un-awaited task can hide a failure — nobody ever looked at the ticket.

09💥 Pitfalls & the Classic Deadlock

Nearly every async disaster is one mistake wearing different hats: blocking a thread to wait for async work instead of awaiting it. This is "sync-over-async," and its tools are .Result, .Wait(), and .GetAwaiter().GetResult().

The deadlock, mechanically 🔒

In frameworks with a SynchronizationContext — classic ASP.NET (not Core), WPF, WinForms — an await tries to resume on the original thread. Now watch the trap:

🧵
1. UI/request thread calls .Resultvar x = GetDataAsync().Result; — the thread now blocks, waiting for the task to finish.
⏳
2. Inside, the await completes its I/OThe async method's await finishes and wants to resume its continuation.
🔒
3. The continuation needs the original thread…but that thread is blocked at step 1, waiting for this very task. Each waits for the other. Frozen — forever.
🔑 ASP.NET Core Changes the Stakes

ASP.NET Core has no SynchronizationContext, so this exact deadlock usually won't trigger there. But that is not permission to block. Sync-over-async still causes thread-pool starvation: each blocked thread is one fewer serving requests, and under load the pool empties, latency spikes, and the app grinds to a halt. The advice is identical everywhere: don't block — await.

C# · The sins and their fixes
// ❌ Blocking — deadlock risk (with a context) or thread starvation (always):
var order = GetOrderAsync(id).Result;
GetOrderAsync(id).Wait();
var o2 = GetOrderAsync(id).GetAwaiter().GetResult();

// ✅ Await it — the thread is freed while waiting:
var order = await GetOrderAsync(id);

// ❌ async void — exceptions escape and crash the process:
public async void Handle() { await DoAsync(); }

// ✅ async Task — awaitable, catchable:
public async Task HandleAsync() { await DoAsync(); }

// ❌ Blocking wait for many tasks (wastes a thread):
Task.WaitAll(t1, t2);

// ✅ Await them:
await Task.WhenAll(t1, t2);

What about ConfigureAwait(false)? 🎛️

You'll see await something.ConfigureAwait(false) in older code and libraries. It says "don't bother resuming on the captured context" — which historically prevented the deadlock above and shaves a little overhead. The current picture:

Where your code runsNeed ConfigureAwait(false)?Why
ASP.NET Core app code (controllers, services)❌ No effectThere's no SynchronizationContext to capture — it does nothing.
Reusable library code (unknown host)✅ Yes, on every awaitYour library might run in WPF/WinForms/legacy ASP.NET where the context does matter — protects those callers.
WPF / WinForms / legacy ASP.NET✅ Where you don't need the UI/request threadAvoids deadlocks and needless marshaling back to the special thread.
✅ The Real Fix Isn't ConfigureAwait

In application code, reaching for ConfigureAwait(false) to "avoid deadlocks" is treating the symptom. The cure is not blocking in the first place — go async all the way. Save ConfigureAwait(false) for library code that may run anywhere, and turn on async analyzers (the Microsoft.VisualStudio.Threading.Analyzers package) to flag .Result, .Wait(), and async void at build time.

The pitfall roll-call 📋

🚫.Result / .Wait()

Blocks the thread. Deadlock (with context) or starvation (always). Await instead.

🕳️async void

Uncatchable exceptions, process crashes. Return Task.

👻Fire-and-forget

Dropped task = lost exception + orphaned work. Use a background service.

🔁await in a tight loop

Sequential when it could be concurrent. Consider WhenAll for independent items.

🔐lock across await

A lock held over an await can deadlock. Use SemaphoreSlim.WaitAsync.

🧊Thread.Sleep in async

Blocks the thread. Use await Task.Delay(...) instead.

10🛠️ Hands-On: Convert Sync to Async End-to-End

Today's Do: take a synchronous method and make it async all the way down. If you have blocking data-access code from earlier days, that's the perfect victim. Work the chain:

  1. 🔎 Find a blocking chain — an endpoint or service calling db.Orders.ToList(), SaveChanges(), or (worse) something with .Result / .Wait().
  2. 🔩 Start at the bottom: swap EF calls for their async twins — ToListAsync, FirstOrDefaultAsync, SaveChangesAsync, FindAsync.
  3. ⏳ Make that method async Task<T> and await the calls. The compiler will now complain that its caller blocks — good, follow the errors upward.
  4. 🔝 Convert each caller in turn — service, then endpoint — until the whole stack is async and ASP.NET Core awaits your handler. Never bridge with .Result to "stop the spread."
  5. 🚦 Thread a CancellationToken from the endpoint parameter down through every async call. Confirm EF receives it.
  6. 🧯 Verify error handling: keep a try/catch across an await and confirm the exception still lands there. Make sure no method is async void.
  7. 🔀 Find two independent calls (e.g., two different HTTP fetches, or two queries on separate contexts) and parallelize them with Task.WhenAll. Note what you must not parallelize (one DbContext).
  8. 🔬 Prove the payoff: enable EF command logging (Day 6/8), hit the endpoint, and confirm it still returns correctly — now without holding a thread during the DB wait.
C# · Before → after
// ❌ BEFORE — synchronous, blocks a thread for the whole DB round trip:
public List<OrderDto> GetRecentOrders(int customerId)
{
    return _db.Orders
        .Where(o => o.CustomerId == customerId)
        .OrderByDescending(o => o.PlacedAt)
        .Select(o => new OrderDto(o.Id, o.Total))
        .ToList();
}

// ✅ AFTER — async end-to-end, thread freed during the wait, cancellable:
public async Task<List<OrderDto>> GetRecentOrdersAsync(
    int customerId, CancellationToken ct)
{
    return await _db.Orders
        .Where(o => o.CustomerId == customerId)
        .OrderByDescending(o => o.PlacedAt)
        .Select(o => new OrderDto(o.Id, o.Total))
        .ToListAsync(ct);
}
✅ Self-Check

After converting: (1) Search your solution for .Result, .Wait(), .GetAwaiter().GetResult(), and async void — are there zero? (2) Does a CancellationToken reach every EF call? (3) If you close the browser mid-request against a slow query, does the work stop? (4) Did you resist the urge to bridge one stubborn sync caller with .Result?

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1In plain terms, what problem does async/await solve?🌱 JUNIOR

It stops threads from sitting idle while waiting on slow external things — a database, an API, a file. A web server has a limited pool of threads; if each request holds its thread the whole time it waits for the database, the server can only handle a few requests at once. Async releases the thread during the wait so it can serve other requests, and resumes the original work when the data arrives. The single operation isn't faster — the server just handles far more of them at the same time.

Q2What is a Task in C#?🌱 JUNIOR

A Task represents work that will finish later — a promise, or a coat-check ticket for a result. Task<T> promises a value of type T (like Task<Order>); a plain Task represents work with no return value, just completion. You get the result by awaiting the task. A completed task ends in one of three states: succeeded (has a value), faulted (holds an exception), or canceled.

Q3What does the await keyword actually do?🌱 JUNIOR

It pauses the method at that point until the awaited task completes — but crucially, while paused it releases the thread back to the pool instead of blocking it. When the task finishes, the rest of the method (the "continuation") is scheduled to run, picking up right after the await with the result in hand. So the code reads top-to-bottom like normal, but doesn't hold a thread hostage during the wait.

Q4Does using async create a new thread?🌱 JUNIOR

No — that's a common misconception. For I/O-bound work (database, network), there's typically no thread at all during the wait; the operating system notifies .NET when the data is ready and the continuation is then picked up by a thread-pool thread. Async is about not occupying a thread while waiting, not about spinning up new ones. Adding threads for CPU-bound work is a different tool (Task.Run).

Q5What return type should an async method have?🌱 JUNIOR

Task if it returns nothing, Task<T> if it returns a value. Never void — with the single exception of a top-level event handler. Returning a Task is what lets the caller await the method, observe its result, and catch its exceptions. By convention the method name also ends in Async.

Q6What is a CancellationToken for?🌱 JUNIOR

It's a signal that asks running async work to stop early — because the user navigated away, a timeout elapsed, or the result is no longer needed. ASP.NET Core gives each request a token and cancels it if the client disconnects. You accept it as a parameter and pass it into every async call (EF Core, HttpClient); those operations then stop and throw an OperationCanceledException. It's cooperative: the work has to actually check or pass the token for anything to happen.

Q7Why shouldn't you use Thread.Sleep in an async method?🌱 JUNIOR

Thread.Sleep blocks the thread — it just sits there occupied for the whole duration, which is exactly what async is trying to avoid. The async equivalent is await Task.Delay(...), which pauses the method and releases the thread during the delay, so it can do other work and resume afterward. Same visible pause, no wasted thread.

Q8How does exception handling work with async/await?🌱 JUNIOR

Just like synchronous code: wrap the await in a try/catch. When the awaited task faults, the exception is stored on the task and then re-thrown at the await point, so your catch block sees it normally. The big exception to this is async void methods — their exceptions have no task to travel on and can crash the app instead of being catchable, which is a key reason to always return Task.

🟠 Mid

Q9What does "async all the way" mean and why does it matter?🟠 MID

It means every method in the call chain from the framework entry point down to the I/O call is async and awaited — no synchronous blocking anywhere in between. It matters because the benefit of async is releasing threads during waits; a single blocking call (a .Result to "bridge" to sync code) re-occupies a thread and defeats the purpose for that whole request. Worse, it's the exact pattern that causes deadlocks in context-based frameworks and thread-pool starvation everywhere. Partially async is often worse than fully synchronous, so the guidance is to convert the whole chain at once.

Q10Explain the classic .Result deadlock step by step.🟠 MID

In a framework with a SynchronizationContext (classic ASP.NET, WPF, WinForms), an await by default captures that context and tries to resume its continuation on the original thread. Now: the original thread calls SomethingAsync().Result, which blocks that thread until the task completes. The task's internal await finishes its I/O and needs to resume — but the only thread it's allowed to resume on is the one that's blocked waiting for it. The continuation waits for the thread; the thread waits for the continuation. Neither can proceed — a permanent freeze with no exception.

Q11ASP.NET Core has no SynchronizationContext. So is blocking with .Result safe there?🟠 MID

No. It removes the classic deadlock, because there's no special thread the continuation must return to — it resumes on any thread-pool thread. But blocking still causes thread-pool starvation: every thread parked on a .Result is one that can't serve a request. Under load, requests arrive faster than blocked threads free up, the pool grows slowly or exhausts, latency spikes, and the app can appear hung. The rule is unchanged across all environments: don't block, await.

Q12When is Task.WhenAll appropriate, and when is it a bug?🟠 MID

Appropriate for independent async operations you want to run concurrently — three different API calls, work across separate resources — where total time drops to roughly the slowest one instead of the sum. It's a bug when the operations share a resource that forbids concurrency: most notably a single EF Core DbContext, which throws on parallel operations. For parallel database work you need a separate context per task (via IDbContextFactory). Also watch exception handling — if several tasks fault, awaiting WhenAll surfaces only the first, so inspect the individual tasks when you need them all.

Q13Why is async void dangerous, and when is it acceptable?🟠 MID

async void returns nothing awaitable, so the caller can't await it, can't know when it finished, and — critically — can't catch its exceptions; an unhandled throw in an async void typically tears down the whole process. It also can't be composed with WhenAll or cancellation. The only acceptable use is a top-level event handler that the framework calls directly (e.g., a UI button click), and even then you wrap the entire body in try/catch. In web apps you essentially never write it — always async Task.

Q14What's wrong with holding a lock across an await, and what's the alternative?🟠 MID

A lock (Monitor) is thread-affine — it must be released by the same thread that took it. But after an await, the continuation may resume on a different thread, so releasing the lock can happen on the wrong thread (the compiler actually forbids await directly inside a lock block). Beyond that, holding any lock across an await keeps it held during the entire I/O wait, blocking other threads that need it — a recipe for contention and deadlock. The async-friendly alternative is SemaphoreSlim with await WaitAsync() / Release(), which doesn't have thread affinity and cooperates with async flow.

Q15What does ConfigureAwait(false) do, and do you need it in an ASP.NET Core service?🟠 MID

ConfigureAwait(false) tells the await not to capture and resume on the current SynchronizationContext — the continuation runs on any thread-pool thread. In ASP.NET Core application code it has no effect, because ASP.NET Core doesn't install a SynchronizationContext in the first place; adding it there is just noise. Where it still matters is reusable library code, which might be consumed by a WPF, WinForms, or legacy ASP.NET app that does have a context — there, ConfigureAwait(false) avoids needless marshaling and prevents deadlocks for those callers. So: library code, yes; ASP.NET Core app code, no.

Q16What's a fire-and-forget task, and what's the risk?🟠 MID

It's calling an async method without awaiting it — DoWorkAsync(); on its own line — so execution continues immediately while the task runs unobserved. Risks: any exception it throws is stored on the discarded task and never seen (silent failure), the work may outlive the request that started it (and touch disposed scoped services like a DbContext), and there's no way to know if or when it completed. The compiler warns (CS4014). If you genuinely need background work, use a proper mechanism — a hosted/background service, a Channel-based queue — that owns the task, handles its exceptions, and manages its own service scope.

Q17A production ASP.NET Core API intermittently hangs under load with no exceptions. How do you investigate?🔴 SENIOR

The pattern screams thread-pool starvation from sync-over-async. Investigation: capture a dump during an incident and look at thread-pool stats — many threads blocked in Task.Result/.Wait()/.GetAwaiter().GetResult() or in a Monitor.Wait, while the queue of pending work grows, is the tell. Tools: dotnet-counters to watch ThreadPool queue length and thread count climb, dotnet-trace/PerfView for stacks, and a dump analyzed for blocked threads. Grep the codebase for the blocking trio and async void. The fix is structural — make the offending paths async all the way — not raising the thread-pool minimum, which only masks it. Add the async analyzers to CI so it can't regress, and consider load tests that reproduce the concurrency level.

Q18You must call an async method from a constructor or a genuinely synchronous interface you can't change. What are your options?🔴 SENIOR

First, resist — constructors shouldn't do I/O. Options, best to worst: (1) redesign to an async factory method (public static async Task<Thing> CreateAsync()) so construction stays synchronous and initialization is awaited separately — the cleanest fix. (2) The async initialization pattern: a lightweight constructor plus an awaitable InitializeAsync that callers await, or lazy async initialization (AsyncLazy). (3) If you truly must bridge — a fixed third-party sync interface — isolate the blocking to a single, documented spot and offload with Task.Run(() => asyncMethod()).GetAwaiter().GetResult() so it runs off any potential context, accepting the thread cost knowingly. But (3) is a last resort; the presence of the need usually signals a design that should be async at that boundary.

Q19Compare Task and ValueTask. When would you actually use ValueTask?🔴 SENIOR

Task is a reference type — allocating one has a small cost, incurred even when the operation completes synchronously. ValueTask<T> is a struct that can wrap either an already-available value (no allocation) or an underlying Task, so it avoids the allocation on the common synchronous-completion path. Use it in hot, high-throughput APIs that frequently complete synchronously — a cache lookup that usually hits, a buffered stream read — where the allocation savings measurably matter. Costs: a ValueTask must be awaited exactly once and not stored, cached, or awaited concurrently (its backing may be reused), so it's more error-prone. Default to Task; reach for ValueTask only after profiling shows the allocations matter and you can honor its consumption rules.

Q20How do you implement a reliable timeout for an async operation, including cancelling the underlying work?🔴 SENIOR

The robust approach threads a token driven by a timeout so the underlying work is actually cancelled, not just abandoned: create a CancellationTokenSource(timeout) (or link it with the request's token via CreateLinkedTokenSource), pass that token into the async operation, and let it throw OperationCanceledException on expiry. On .NET 8+, Task.WaitAsync(timeout) gives a clean timeout wrapper. Avoid the naive Task.WhenAny(work, Task.Delay(timeout)) — it lets you observe a timeout but leaves the original work running unobserved (and swallows its exceptions) unless you also cancel it via a token. The principle: a timeout should cancel the work, not just stop waiting for it.

Q21Walk through what the C# compiler generates for an async method.🔴 SENIOR

The compiler rewrites the method into a state machine — a generated struct implementing IAsyncStateMachine with a MoveNext method and an integer state field. Local variables that must survive an await become fields on the struct. The method body is chopped into segments at each await; MoveNext runs the current segment, and at an await it asks the awaiter whether the task is already complete — if so it continues synchronously, if not it hooks the continuation (which calls MoveNext again, advancing state) via the awaiter and returns, releasing the thread. A builder (AsyncTaskMethodBuilder) produces the Task handed to the caller and, on completion, sets its result or exception. Understanding this explains why locals persist across awaits, why exceptions land on the returned task, and why there's real (if small) overhead — which is what ValueTask and returning-the-task-directly optimize away.

Q22Design correct background/fire-and-forget work in an ASP.NET Core app.🔴 SENIOR

Never bare fire-and-forget from a request — the task outlives the request, its scoped services (DbContext!) get disposed underneath it, and its exceptions vanish. Proper design: enqueue the work item to an in-memory queue backed by System.Threading.Channels, and drain it in a BackgroundService (an IHostedService) running for the app's lifetime. Inside the background worker, create a fresh DI scope per item (IServiceScopeFactory.CreateScope()) so it gets its own DbContext and scoped services; wrap each item in try/catch with structured logging so failures are observed; and honor the host's stopping token for graceful shutdown. For durable or cross-process needs, escalate to a real queue (e.g., a message broker) with retries and dead-lettering. The key principles: owned lifetime, own scope, observed exceptions, graceful shutdown.

Q23You have 500 items to process, each an async HTTP call. How do you run them efficiently and safely?🔴 SENIOR

Not 500 concurrent calls — that hammers the remote service, exhausts sockets/connections, and can trip rate limits. And not one-at-a-time in a loop — too slow. The answer is bounded concurrency: cap the number in flight. Clean options: Parallel.ForEachAsync with MaxDegreeOfParallelism set (built for exactly this on modern .NET), or a SemaphoreSlim as a gate around each task with a WhenAll over the whole set, or a TPL Dataflow / Channel pipeline for streaming. Also: pass a shared CancellationToken so one failure/timeout can stop the batch, decide your error policy (fail-fast vs collect-all-results, e.g. via results objects rather than exceptions), and reuse a single HttpClient/IHttpClientFactory. The judgment is picking the concurrency limit from the downstream service's tolerance, not the item count.

Q24Your team keeps introducing async bugs. What guardrails do you put in place?🔴 SENIOR

Make the mistakes impossible or loud. Tooling: add Microsoft.VisualStudio.Threading.Analyzers (and enable the built-in async warnings) and treat them as errors in CI, so .Result/.Wait(), async void, and un-awaited tasks (CS4014) fail the build. Conventions in a short team doc: async methods return Task and end in Async; every I/O method takes and forwards a CancellationToken; no blocking bridges — convert the whole chain; background work goes through the hosted-service pattern, never bare fire-and-forget. Review habits: a reviewer scans any new data/HTTP path for an unbroken await chain and a threaded token. Testing: load tests at realistic concurrency to surface starvation before production, and cancellation tests that abandon a request and assert the work stops. Culture: pair the rules with the why (thread-pool starvation, lost exceptions) so people internalize them rather than cargo-culting.
