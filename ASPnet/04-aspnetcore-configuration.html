// .NET FUNDAMENTALS · DAY 4

Configuration in ASP.NET Core ⚙️

How a modern .NET app reads its settings — connection strings, feature toggles, API keys — from many places at once, keeps secrets out of source control, and behaves differently in development versus production, all through one tidy interface.

📄 appsettings.json 🌍 environment files 🔧 env variables 🔐 user secrets 🧩 IConfiguration 📦 Options pattern

01Why Configuration Exists

Every real application has values that shouldn't be baked into the code. The address of the database, the port for the email server, whether a new feature is switched on, the secret key that talks to a payment provider — these change between your laptop, the testing server, and the live production site. If you hard-code them, you'd have to edit and recompile the program every time one changes. Configuration is the mechanism that pulls those values out of the code and lets the running app read them from the outside.

🍳 Analogy

Think of your app as a recipe and configuration as the pantry labels. The recipe says "add the oil from the bottle on the shelf." It doesn't care whether the bottle holds olive oil at home or sunflower oil at a friend's kitchen — it just reads the label at cooking time. Swap the pantry, and the same recipe produces the right dish for that kitchen. Configuration lets the same compiled code run correctly in every "kitchen" (environment).

In ASP.NET Core, configuration data is simply a big collection of key–value pairs. A key like ConnectionStrings:DefaultConnection maps to a value like a database address. Those pairs can come from a JSON file, from the operating system's environment variables, from the command line, or from several other places — and the framework merges them all into one unified view your code can read.

🔑 Key idea

Configuration in .NET is an ordered chain of providers stacked on top of each other. Each provider contributes key–value pairs, and if two providers set the same key, the one added later wins. That single rule explains almost everything about how settings behave.

02The Configuration Model

When your app starts, the line var builder = WebApplication.CreateBuilder(args); quietly wires up a set of default configuration providers. Each provider knows how to read one kind of source. Together they fill a single object you'll interact with everywhere: IConfiguration.

🧩 IConfiguration
one merged key–value store
📄 JSON files
  • appsettings.json — base settings
  • appsettings.{Env}.json — per-environment overrides
  • Loaded by the JSON provider
🔐 User secrets
  • A JSON file outside the project
  • Development only, by default
  • Keeps secrets out of Git
🔧 Environment variables
  • Set by the OS, container, or host
  • Ideal for servers & production
  • Use __ for nested keys
⌨️ Command-line args
  • Passed to dotnet run
  • Highest priority by default
  • Great for quick overrides
➕ Others
  • Azure Key Vault, Azure App Config
  • In-memory, INI, XML, key-per-file
  • Even custom providers you write

Your code never asks "which file did this come from?" It just asks IConfiguration for a key and gets a value. That indirection is the whole point: the same reading code works no matter where a value physically lives.

✅ Tip

Configuration keys are case-insensitive, and every value is fundamentally a string. When you ask for an int or a bool, the framework converts the string for you. A null value can't be stored in configuration.

03appsettings.json

This is the primary, human-readable settings file that ships in the root of almost every ASP.NET Core project. It's plain JSON, it's checked into source control, and it holds your non-secret defaults: logging levels, allowed hosts, and your own custom sections.

json · appsettings.json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=localhost;Database=ShopDb;Trusted_Connection=True;TrustServerCertificate=True;"
  },
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning"
    }
  },
  "AllowedHosts": "*",
  "EmailSettings": {
    "SenderName": "Shop Notifications",
    "FromAddress": "no-reply@myshop.com",
    "SmtpPort": 587
  }
}

Notice the JSON is nested (an object inside an object). Internally the configuration system flattens this hierarchy into flat keys, joining the levels with a colon (:). So the nested value above is addressed as EmailSettings:SmtpPort.

✅ Good to know

With the default host, appsettings.json is loaded with reload-on-change turned on. Edit and save the file while the app is running and the new values take effect immediately — no restart needed (whether your code sees the change depends on how you read it; more in §09).

⚠️ Warning

appsettings.json is committed to source control and readable by anyone with the repository. Never put passwords, API keys, or other secrets here. A local, password-free connection string is fine; a real production password is not.

04Environment-Specific Files

Alongside appsettings.json you can add files named appsettings.{Environment}.json. The {Environment} part matches the app's current environment name, which comes from the ASPNETCORE_ENVIRONMENT environment variable. The three conventional names are Development, Staging, and Production, but the value is free-form, so you can invent your own.

💻
appsettings.Development.json
Loaded when ASPNETCORE_ENVIRONMENT=Development — usually on your laptop. Verbose logging, local database.
🚀
appsettings.Production.json
Loaded in production. Quieter logging, real service endpoints. Overrides the base file's values.
🧪
appsettings.Staging.json
A pre-production rehearsal environment. Optional — create it only if you use a staging tier.

The key behaviour is layering: the base appsettings.json loads first, then the environment file loads on top and overrides any keys they share. You only put the differences in the environment file, not a full copy.

json · appsettings.Development.json
{
  // Only the keys that differ in Development.
  "Logging": {
    "LogLevel": {
      "Default": "Debug"       // noisier logs while coding
    }
  },
  "EmailSettings": {
    "SmtpPort": 1025          // a local fake mail server
  }
}

At runtime, Logging:LogLevel:Default becomes Debug and EmailSettings:SmtpPort becomes 1025, while every other key (like SenderName and AllowedHosts) is still inherited from the base file.

🔑 Where does the environment name come from?

From the ASPNETCORE_ENVIRONMENT variable. On your machine it's typically set to Development inside Properties/launchSettings.json. If it's not set anywhere, the framework defaults to Production — a safe default, since Production applies the strictest behaviour.

05Environment Variables

Environment variables are name–value pairs maintained by the operating system, a container, or a cloud host rather than by a file in your project. They're the standard way to feed settings into an app running on a server, in Docker, or in Kubernetes — because you can change them per-machine without touching the code or any committed file.

By default, ASP.NET Core reads environment variables that are not prefixed with ASPNETCORE_ or DOTNET_ (those two prefixes are reserved for configuring the host itself).

The double-underscore rule

Configuration keys are nested with a colon (ConnectionStrings:DefaultConnection), but many operating systems and shells (Bash, for example) don't allow a colon in a variable name. The solution is a double underscore __, which .NET automatically converts into a colon when it reads the variable.

Environment variable name → configuration key
ConnectionStrings__DefaultConnection
env variable
→ converts to →
__ becomes :

bash · setting variables on Linux / macOS
# Nested key: __ turns into : inside the app
export ConnectionStrings__DefaultConnection="Server=prod-db;Database=ShopDb;User Id=app;Password=…;"
export EmailSettings__SmtpPort=2525

# Choose the environment file that loads
export ASPNETCORE_ENVIRONMENT=Production

dotnet run
powershell · setting variables on Windows
$env:ConnectionStrings__DefaultConnection = "Server=prod-db;Database=ShopDb;User Id=app;Password=…;"
$env:ASPNETCORE_ENVIRONMENT = "Production"
dotnet run
✅ Best fit

Environment variables are appropriate for any environment, production included, and are the usual home for real production secrets when you're not using a dedicated secret store like Azure Key Vault. Because they're set on the host, they never end up in your repository.

⚠️ Not encrypted

Environment variables are stored as plain text. If a machine or process is compromised, they can be read. For highly sensitive production secrets, a managed secret store (Azure Key Vault, AWS Secrets Manager) is stronger.

06User Secrets

During development you often need a real secret — a password, an API key — but you must never commit it. User Secrets (managed by the "Secret Manager" tool) solve this by storing those values in a JSON file that lives outside your project folder, in your personal user profile. Because it's outside the project tree, it can't be accidentally checked into Git.

bash · the Secret Manager workflow
# 1) Run once, from the project folder. Adds a UserSecretsId (a GUID) to the .csproj
dotnet user-secrets init

# 2) Store secrets. The key uses the same colon syntax as appsettings.json
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Server=localhost;Database=ShopDb;User Id=dev;Password=Sup3rSecret!;"
dotnet user-secrets set "EmailSettings:ApiKey" "sk_test_51H…"

# 3) Inspect or clean up
dotnet user-secrets list
dotnet user-secrets remove "EmailSettings:ApiKey"
dotnet user-secrets clear

After init, your project file gains a marker that links the project to its secret store:

xml · MyShop.csproj
<PropertyGroup>
  <TargetFramework>net10.0</TargetFramework>
  <UserSecretsId>a1b2c3d4-5e6f-47a8-9012-3456789abcde</UserSecretsId>
</PropertyGroup>

Where the file actually lives

OSsecrets.json path
Windows%APPDATA%\Microsoft\UserSecrets\<UserSecretsId>\secrets.json
Linux / macOS~/.microsoft/usersecrets/<UserSecretsId>/secrets.json
⚠️ Development only — and not encrypted

User Secrets are loaded only when the app runs in the Development environment, and the file is stored as plain, unencrypted text. Their value is convenience and keeping secrets out of source control — not cryptographic protection. Use environment variables or Key Vault for staging and production.

🔐 Analogy

User Secrets are like a sticky note you keep in your own desk drawer instead of taped to the shared office whiteboard. It's still readable by anyone who opens your drawer — but it won't be photographed and shared with the whole company (committed to the repo) by accident.

07Precedence & Load Order

When the same key is defined in more than one place, who wins? The provider added later overrides the one added earlier. With the default host builder, the sources are registered in a fixed order, so the effective priority (highest to lowest) is:

⌨️ 1 · Command-line arguments
Highest priority. Passed to dotnet run. Overrides everything else — perfect for one-off tweaks.
🔧 2 · Environment variables
Non-prefixed OS variables. The usual channel for server and production values.
🔐 3 · User Secrets (Development only)
Loaded after the JSON files, so a secret here overrides the same key in appsettings.
🌍 4 · appsettings.{Environment}.json
Per-environment overrides layered on top of the base file.
📄 5 · appsettings.json
The base defaults. Lowest priority of the app sources.
🏠 6 · Host configuration (fallback)
The bottom layer the host itself established.

A concrete example: suppose EmailSettings:SmtpPort is set in three places. The final value is decided top-down.

Resolving EmailSettings:SmtpPort — last writer wins
587
appsettings.json
1025
Development.json
2525
env variable ✅ wins
🔑 The mental model

Picture the providers as sheets of tracing paper laid down one after another. Later sheets sit on top. When you look straight down at a key, you see whatever is written on the topmost sheet that has it. That's precedence in one image.

✅ Override anything from the command line

Because command-line args sit at the very top, you can override any setting instantly:
dotnet run --EmailSettings:SmtpPort=9000

08Reading With IConfiguration

You get to the merged settings through the IConfiguration service. In Program.cs it's available as builder.Configuration; anywhere else you receive it through dependency injection. Here are the reading methods you'll use most.

🔤
Indexer [ ]
The raw way. Returns a string?. config["EmailSettings:SmtpPort"] gives "587" as text.
🔢
GetValue<T>
Reads one value and converts its type, with an optional default. GetValue<int>("…:SmtpPort", 25).
🔗
GetConnectionString
A shortcut that reads the ConnectionStrings section by name. Cleaner than the full key.
📂
GetSection
Returns a sub-tree of related keys. Never returns null — an empty section instead.
csharp · Program.cs
var builder = WebApplication.CreateBuilder(args);
IConfiguration config = builder.Configuration;

// Raw string (may be null)
string? portText = config["EmailSettings:SmtpPort"];

// Typed read with a fallback default of 25
int smtpPort = config.GetValue<int>("EmailSettings:SmtpPort", 25);

// Connection-string shortcut — reads ConnectionStrings:DefaultConnection
string? conn = config.GetConnectionString("DefaultConnection");

// Grab a whole section, then read children by their short keys
IConfigurationSection email = config.GetSection("EmailSettings");
string? sender = email["SenderName"];

To read configuration inside a controller or service, just ask for IConfiguration in the constructor and the framework hands it to you:

csharp · injecting into a service
public class ReportService(IConfiguration config)
{
    public void Run()
    {
        var host = config["EmailSettings:FromAddress"];
        Console.WriteLine($"Sending report from {host}");
    }
}
⚠️ The magic-string problem

Reading with string keys like "EmailSettings:SmptPort" is fragile: a typo compiles fine but silently returns null at runtime, and any class holding IConfiguration can read every setting, including secrets it shouldn't touch. Section 09 fixes both problems.

09The Options Pattern

Instead of scattering string keys through your code, the Options pattern binds a whole configuration section to a plain C# class once, then injects that strongly-typed class wherever it's needed. You get autocomplete, compile-time safety, and each class only sees its own settings.

Step 1 — Define a class that mirrors the section

csharp · EmailSettings.cs
public class EmailSettings
{
    public const string SectionName = "EmailSettings";

    public string SenderName  { get; set; } = string.Empty;
    public string FromAddress { get; set; } = string.Empty;
    public int    SmtpPort    { get; set; }
}

Step 2 — Bind the section during startup

csharp · Program.cs
builder.Services.Configure<EmailSettings>(
    builder.Configuration.GetSection(EmailSettings.SectionName));

Step 3 — Inject and use it

csharp · EmailSender.cs
using Microsoft.Extensions.Options;

public class EmailSender(IOptions<EmailSettings> options)
{
    private readonly EmailSettings _settings = options.Value;

    public void Send()
    {
        Console.WriteLine(
            $"From {_settings.SenderName} on port {_settings.SmtpPort}");
    }
}

Three ways to receive options

There are three accessor interfaces, and choosing the right one is a favourite interview topic. They differ in lifetime and whether they notice config changes after startup.

InterfaceLifetimeReloads on change?Use when…
IOptions<T>Singleton❌ No — read once at startupValues are effectively constant (connection strings, fixed URLs). Fastest, simplest — the default choice.
IOptionsSnapshot<T>Scoped✅ Yes — fresh per web requestYou want each request to pick up edited config. Can't be injected into a singleton.
IOptionsMonitor<T>Singleton✅ Yes — current value any timeLong-lived / background services that need live values, plus change notifications via OnChange.
✅ Rule of thumb

Default to IOptions<T>. Reach for IOptionsSnapshot<T> only when a per-request refresh matters, and IOptionsMonitor<T> when a singleton or background worker must always see the latest value.

You can also validate options — catch a bad or missing value at startup instead of at 3 a.m. in production:

csharp · validate at startup
builder.Services.AddOptions<EmailSettings>()
    .Bind(builder.Configuration.GetSection(EmailSettings.SectionName))
    .ValidateDataAnnotations()
    .ValidateOnStart();

10Hands-On Walkthrough

Here's the day's task assembled end to end: load a connection string and bind a settings section in a minimal API. Follow the flow, then read the full Program.cs.

1 · Put defaults in appsettings.json 📄
Add a ConnectionStrings block and an EmailSettings section (the non-secret parts).
2 · Move the real password into User Secrets 🔐
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "…with password…" — overrides the base value in Development.
3 · Read the connection string 🔗
builder.Configuration.GetConnectionString("DefaultConnection") — the framework has already merged every source, so you just ask once.
4 · Bind the settings section 📦
Configure<EmailSettings>(...GetSection("EmailSettings")) registers the strongly-typed class.
5 · Use both via injection ✅
Endpoints and services receive IOptions<EmailSettings> and the DB context reads the connection string.
csharp · complete Program.cs
using Microsoft.Extensions.Options;

var builder = WebApplication.CreateBuilder(args);

// (1) Read the connection string. In Development this resolves to the
//     User Secrets value; in Production, to an environment variable.
string? connectionString =
    builder.Configuration.GetConnectionString("DefaultConnection");

if (string.IsNullOrWhiteSpace(connectionString))
    throw new InvalidOperationException(
        "Missing ConnectionStrings:DefaultConnection.");

// Hand it to your data layer, e.g. Entity Framework Core:
// builder.Services.AddDbContext<ShopContext>(o => o.UseSqlServer(connectionString));

// (2) Bind the EmailSettings section to a strongly-typed class
builder.Services.Configure<EmailSettings>(
    builder.Configuration.GetSection(EmailSettings.SectionName));

var app = builder.Build();

// (3) Read the connection string from an endpoint (masking the value)
app.MapGet("/db", (IConfiguration cfg) =>
{
    var cs = cfg.GetConnectionString("DefaultConnection");
    return Results.Ok(new { Configured = !string.IsNullOrEmpty(cs) });
});

// (4) Consume the bound settings section via the Options pattern
app.MapGet("/email-config", (IOptions<EmailSettings> opts) =>
{
    EmailSettings s = opts.Value;
    return Results.Ok(new { s.SenderName, s.FromAddress, s.SmtpPort });
});

app.Run();

// ---- supporting class ----
public class EmailSettings
{
    public const string SectionName = "EmailSettings";
    public string SenderName  { get; set; } = string.Empty;
    public string FromAddress { get; set; } = string.Empty;
    public int    SmtpPort    { get; set; }
}
🔑 What just happened

The exact same code runs everywhere. On your laptop the connection string comes from User Secrets and SmtpPort from appsettings.Development.json; on the server both may come from environment variables. Your Program.cs neither knows nor cares — that's the payoff of the provider chain.

11Best Practices & Pitfalls

🚫
No secrets in appsettings
Keep passwords and keys out of committed JSON. Use User Secrets locally; env vars or Key Vault in production.
📦
Prefer the Options pattern
Bind sections to classes rather than sprinkling config["…"] everywhere. Fewer typos, real type safety.
🧩
Only the diffs in env files
Environment files should hold overrides, not a full copy of the base file. Less duplication, fewer surprises.
🔁
Pick the right Options accessor
IOptions for constants; IOptionsMonitor when a singleton must see live changes.
✅
Validate on start
Use ValidateOnStart() so a missing or malformed setting fails loudly at launch, not mid-request.
🔤
Mind the separators
Colon : in code and JSON; double underscore __ in environment variables.
⚠️ Common gotchas

Wrong environment name means the wrong file loads — check ASPNETCORE_ENVIRONMENT first when settings look "stale." User Secrets ignored? They only load in Development. Value not overriding? An env variable or command-line arg sits above your JSON — remember precedence. And a misspelled key returns null silently rather than throwing.

13Interview Questions

Click any question to reveal a plain-English answer. Grouped by level — start with the 🌱 Junior set and work down.

🌱 Junior (0–2 yrs)

Q1What is configuration in ASP.NET Core, in one sentence?Junior

It's the system that lets an app read its settings — things like a database address, a port number, or a feature toggle — from outside the code, so those values can change without recompiling the program.

Q2What is the appsettings.json file used for?Junior

It's the main settings file in the project root. It holds non-secret default values in JSON — logging levels, allowed hosts, and your own custom settings. It's committed to source control, so it must never contain passwords or secret keys.

Q3Why have an appsettings.Development.json as well?Junior

To keep settings that differ between environments. When the app runs in Development, that file loads on top of the base file and overrides matching keys — so you can use a local database or verbose logging while developing, without changing what production uses.

Q4How is the app's environment (Development, Production) decided?Junior

By an environment variable named ASPNETCORE_ENVIRONMENT. On a developer machine it's usually set to Development in launchSettings.json. If it isn't set anywhere, the framework defaults to Production.

Q5How do you read a value from configuration in code?Junior

You use the IConfiguration object. In Program.cs it's builder.Configuration; elsewhere you receive it through the constructor. Then you read a value by its key, for example config["EmailSettings:SmtpPort"].

Q6What does the colon mean in a key like EmailSettings:SmtpPort?Junior

It's a separator for nested settings. The JSON has SmtpPort inside an EmailSettings object, and the colon walks down that hierarchy — outer name, colon, inner name.

Q7Where should a real database password live during development?Junior

In User Secrets, not in appsettings.json. User Secrets store the value in a file outside the project folder so it can't be committed to source control by accident. In production you'd use an environment variable or a managed secret store instead.

Q8What is a connection string, and how do you fetch one?Junior

A connection string is the piece of text that tells the app how to reach a database — server, database name, and credentials. You fetch it with the shortcut config.GetConnectionString("DefaultConnection"), which reads the ConnectionStrings section by name.

🟠 Mid (2–5 yrs)

Q9If the same key is set in two places, which value wins?Mid

The provider added later wins. With the default host the priority, highest to lowest, is: command-line arguments, then environment variables, then User Secrets (Development only), then the environment-specific JSON file, then the base appsettings.json. Picture stacked tracing paper — the top sheet with the key is what you see.

Q10Why do environment variables use __ instead of a colon?Mid

Because many shells and operating systems (Bash, for instance) don't allow a colon inside a variable name. The framework accepts a double underscore __ on every platform and automatically converts it to a colon when it reads the value, so ConnectionStrings__DefaultConnection becomes ConnectionStrings:DefaultConnection.

Q11What is the Options pattern and why prefer it over reading IConfiguration directly?Mid

The Options pattern binds a configuration section to a strongly-typed C# class, which you then inject where needed. It's better because you get compile-time safety and autocomplete (no fragile string keys), and each class only sees its own settings rather than the entire configuration, which is cleaner and more secure.

Q12Walk through binding a settings section with the Options pattern.Mid

Three steps: define a class whose properties mirror the JSON keys; register it in startup with builder.Services.Configure<EmailSettings>(builder.Configuration.GetSection("EmailSettings")); then inject IOptions<EmailSettings> and read .Value wherever you need the settings.

Q13Are User Secrets encrypted or safe for production?Mid

No on both counts. User Secrets are stored as plain, unencrypted text and only load in the Development environment by default. Their purpose is to keep secrets out of source control and off shared config files during development — not to provide cryptographic protection. Production should use environment variables or a secret store such as Azure Key Vault.

Q14What's the difference between the indexer, GetValue<T>, and GetSection?Mid

The indexer config["key"] returns the raw value as a string (or null). GetValue<T>("key", default) reads one value, converts it to the type you ask for, and lets you supply a fallback. GetSection("name") returns a whole sub-tree of related keys that you can read or bind; it never returns null — you get an empty section if the name isn't found.

Q15What does reloadOnChange do, and does it always update running code?Mid

It makes the JSON provider watch the file and re-read it when you save changes, so the merged configuration updates without a restart. But whether your code sees the new value depends on how you read it: a value captured once at startup (like a plain IOptions<T>) won't change; you need IOptionsSnapshot or IOptionsMonitor to observe reloads.

Q16A setting you added to appsettings.json isn't taking effect. How do you debug it?Mid

Check precedence first — an environment variable or command-line argument with the same key sits above the JSON and overrides it. Then confirm the key spelling and nesting exactly match (a typo returns null silently). Verify the environment name is what you expect, since the wrong appsettings.{Env}.json may be loading. Enumerating app.Configuration.AsEnumerable() at startup can print the effective keys and values.

🔴 Senior (5–7 yrs)

Q17Explain IOptions vs IOptionsSnapshot vs IOptionsMonitor.Senior

IOptions<T> is a singleton computed once at startup — it never reflects later config changes; it's the fastest and the default choice for stable values. IOptionsSnapshot<T> is scoped, so it's recomputed once per web request and picks up changes on the next request; because it's scoped it can't be injected into a singleton. IOptionsMonitor<T> is a singleton that always exposes the current value through CurrentValue and fires an OnChange callback — the right choice for long-lived or background services that must react to live updates.

Q18Why can't IOptionsSnapshot<T> be injected into a singleton, and what happens if you try?Senior

IOptionsSnapshot is registered as a scoped service, and injecting a scoped service into a singleton is a lifetime violation — the singleton would capture one request's scope forever (a "captive dependency"). The DI container detects this and throws at startup when scope validation is on. For live values in a singleton, use IOptionsMonitor, which is itself a singleton.

Q19How would you add a custom configuration source, and when is that justified?Senior

You implement IConfigurationSource and a matching ConfigurationProvider that populates the key–value dictionary in its Load method, then register it via an extension on the configuration builder. It's justified when settings live somewhere the built-in providers don't cover — a database table, a feature-flag service, or a bespoke vault. You'd weigh cost against alternatives like Azure App Configuration, and decide whether reload-on-change is worth implementing.

Q20How do you validate configuration and fail fast on a bad value?Senior

Use the options builder: AddOptions<T>().Bind(section).ValidateDataAnnotations().ValidateOnStart(). Data-annotation attributes (or a custom Validate delegate) express the rules, and ValidateOnStart() forces evaluation at launch so a missing or malformed setting crashes the app immediately rather than surfacing as a confusing runtime failure later. This is far preferable to discovering a bad connection string on the first request in production.

Q21What are the security trade-offs across appsettings, env vars, User Secrets, and Key Vault?Senior

appsettings.json is committed and world-readable to anyone with the repo — never for secrets. Environment variables aren't committed but are plain text and readable by anyone who can inspect the process, so they're acceptable but not strong. User Secrets solve the "don't commit it" problem in development only and are unencrypted. A managed store like Azure Key Vault adds encryption at rest, access policies, auditing, and rotation — the right answer for sensitive production secrets. The general principle: raise the protection level as you move from dev to prod and as the sensitivity of the value increases.

Q22Explain the difference between host configuration and app configuration.Senior

Host configuration is read very early to bootstrap the host itself — the environment name, content root, and application name — and draws mainly from DOTNET_- and ASPNETCORE_-prefixed environment variables and the command line. App configuration is the broader set your application code reads through IConfiguration, and it takes priority over host configuration. Some settings, such as URLS, are read from app configuration even though host configuration can act as a fallback for them.

Q23A config array merged from two sources gives unexpected results. Why?Senior

Arrays are flattened into indexed keys (Modules:0, Modules:1, …). Because the last source wins per key, a shorter array from a higher-priority source overwrites only the overlapping indices and leaves trailing elements from the earlier source in place — producing a spliced, surprising result. Fixes: keep array indices aligned across sources, or store the list as a single delimited string and split it yourself so it's replaced atomically.

Q24Why are configuration values always strings, and what are the implications?Senior

Because the providers read from text-based sources (files, environment variables, command lines) that have no native type system, everything arrives as a string, and the binder converts on read. The implications: type conversion happens when you call GetValue<T> or bind, so an unconvertible value ("banana" for an int) becomes a silent default or a bind error rather than a compile error; null can't be represented, so a "cleared" value is an empty string, not null; and culture/format issues can bite numeric and date parsing. Strong typing and validation via the Options pattern are how you tame this.

✅ Study order

Learn it in the order this page is written: first why config exists and the merged IConfiguration view, then each source (appsettings → environment files → env vars → User Secrets), then the precedence rule that ties them together, and finally reading with IConfiguration and the Options pattern. Nail Q1–Q8 cold before touching the mid questions — juniors are expected to know the sources and precedence, while the Options accessor trio (Q17) is the single highest-value senior topic.
