// .NET FUNDAMENTALS · DAY 5

The Options Pattern 📦

Turn loose configuration keys into clean, strongly-typed C# classes you can inject anywhere — with three accessor interfaces for different refresh needs, and validation that fails loudly at startup instead of quietly at 3 a.m. in production.

🧱 IOptions<T> 📸 IOptionsSnapshot<T> 📡 IOptionsMonitor<T> 🔗 binding ✅ validation 🚦 ValidateOnStart

01Why the Options Pattern

On Day 4 you read configuration with string keys — config["EmailSettings:SmtpPort"]. That works, but it has three quiet problems. A misspelled key like "SmptPort" compiles fine and returns null at runtime. There's no type safety — you convert strings by hand. And any class holding IConfiguration can read every setting in the app, including secrets it has no business touching.

The Options pattern fixes all three. You define a plain C# class whose properties mirror a configuration section, bind the section to it once at startup, and then inject that strongly-typed class wherever you need it. You get autocomplete, compile-time checking, a clear contract, and each class sees only its own slice of configuration.

🍳 Analogy

Reading raw configuration is like grabbing ingredients from an unlabeled pantry by shouting their shelf coordinates ("row 3, slot 7") and hoping you got the right jar. The Options pattern is like having a labeled spice rack handed to you: a small, named tray holding exactly the items this recipe needs — clearly marked, nothing extra, and you notice immediately if one is missing.

🔑 One-line definition

The Options pattern binds a configuration section to a strongly-typed class and delivers it through dependency injection, so your code depends on typed settings rather than magic strings.

02The Big Picture

There are only a few moving parts. A settings class, a registration step that binds a config section to it, an optional validation step, and one of three accessor interfaces you inject to read the values. Here's the whole territory at a glance.

The rest of this page walks these parts in order: bind first (§03–04), pick an accessor (§05–08), validate (§09), then assemble the whole thing in the hands-on task (§10).

03Binding a Settings Class

"Binding" means copying the values from a configuration section into the matching properties of a C# object. The framework matches by name: a JSON key called SmtpPort fills a property called SmtpPort. Start with the configuration section you want to represent:

json · appsettings.json
{
  "EmailSettings": {
    "SenderName": "Shop Notifications",
    "FromAddress": "no-reply@myshop.com",
    "SmtpPort": 587,
    "RetryCount": 3
  }
}

Now write a class whose property names line up with those keys. A const holding the section name is a common convention — it keeps the magic string in exactly one place.

csharp · EmailSettings.cs
public class EmailSettings
{
    public const string SectionName = "EmailSettings";

    public string SenderName  { get; set; } = string.Empty;
    public string FromAddress { get; set; } = string.Empty;
    public int    SmtpPort    { get; set; }
    public int    RetryCount  { get; set; }
}

✅ Binding rules worth knowing

Properties need a public setter (or an init accessor) to be bound. Matching is case-insensitive. Properties with no matching key keep their default; keys with no matching property are simply ignored. String values are converted to your property's type, so "587" becomes an int.

⚠️ Nested objects & collections

Binding recurses into nested classes and lists, but be aware that data-annotation validation does not recurse by default — it only checks the top-level properties unless you opt in (more in §09).

04AddOptions & BindConfiguration

There are two ways to register the binding in Program.cs. Both put a bound EmailSettings into the DI container; they differ in how much you can chain onto them.

The classic way — Configure<T>

csharp · Program.cs
builder.Services.Configure<EmailSettings>(
    builder.Configuration.GetSection(EmailSettings.SectionName));

Short and familiar. Its limitation: Configure<T> returns the service collection, so you can't fluently chain validation onto it.

The modern way — AddOptions<T>

Calling AddOptions<T>() returns an OptionsBuilder<T>, which is chainable. You can bind, validate, and configure start-up validation all in one fluent expression. In current .NET, BindConfiguration is the tidiest binder — it takes the section path as a string and also wires up reload-on-change automatically.

csharp · the chainable, recommended form
builder.Services
    .AddOptions<EmailSettings>()
    .BindConfiguration(EmailSettings.SectionName)   // bind the "EmailSettings" section
    .ValidateDataAnnotations()                    // enable [Required], [Range], etc.
    .ValidateOnStart();                            // check at launch, not first use
🔑 Configure vs. AddOptions

Use Configure<T> for a plain bind with nothing else. Reach for AddOptions<T>().BindConfiguration(…) whenever you want to attach validation or start-up checks — it's the pattern current Microsoft docs lead with for new code.

Whichever you choose, the registration is done once. From here on, consuming code just asks for one of the three accessor interfaces.

05IOptions<T>

The simplest accessor. You inject IOptions<EmailSettings> and read its .Value property to get the bound object.

csharp · EmailSender.cs
using Microsoft.Extensions.Options;

public class EmailSender(IOptions<EmailSettings> options)
{
    private readonly EmailSettings _settings = options.Value;

    public void Send()
        => Console.WriteLine($"From {_settings.SenderName} on port {_settings.SmtpPort}");
}
♾️
Lifetime: Singleton
One instance for the whole app. Can be injected into services of any lifetime.
🧊
Value: frozen
Bound once and cached. Editing appsettings.json afterward has no effect — the value never changes.
⚡
Fastest
No per-request recomputation, no file watching. The cheapest of the three.
✅ When to use it

Use IOptions<T> for values that are effectively constant for the app's lifetime — connection strings, fixed service URLs, feature flags set at deploy time. It's the default choice, and the one you'll use most.

⚠️ It does not reload

This is the classic gotcha: change a value in appsettings.json while the app runs and IOptions<T> keeps serving the old value. If you need live updates, you need one of the next two interfaces.

06IOptionsSnapshot<T>

IOptionsSnapshot<T> is scoped: it's computed once per request (a fresh instance per DI scope) and re-bound from the current configuration each time. So within a single web request the value is stable, but the next request picks up any changes you saved to the config file in the meantime.

csharp · reading a snapshot in an endpoint
app.MapGet("/email-config", (IOptionsSnapshot<EmailSettings> snapshot) =>
{
    EmailSettings s = snapshot.Value;   // fresh per request
    return Results.Ok(new { s.SenderName, s.SmtpPort });
});
🔄
Lifetime: Scoped
One instance per request. Recomputed each request from current config.
📸
Reloads: yes
Reflects config edits on the next request — ideal for request-handling code.
🚫
Not for singletons
Being scoped, it can't be injected into a singleton service (see §08).
✅ When to use it

Reach for IOptionsSnapshot<T> in controllers, Razor pages, and other per-request code where you'd like edited configuration to take effect without restarting the app.

⚠️ A small performance cost

Because it re-binds once per request, IOptionsSnapshot<T> is measurably slower than IOptions<T>. For hot paths with truly static config, prefer IOptions<T>.

07IOptionsMonitor<T>

IOptionsMonitor<T> is a singleton that always exposes the latest value through its .CurrentValue property (note: CurrentValue, not .Value). Because it's a singleton it can be injected anywhere — including into other singletons and long-lived background services — and it can notify you the moment configuration changes via OnChange.

csharp · a singleton background worker
using Microsoft.Extensions.Options;

public class MailPollingService : BackgroundService
{
    private readonly IOptionsMonitor<EmailSettings> _monitor;

    public MailPollingService(IOptionsMonitor<EmailSettings> monitor)
    {
        _monitor = monitor;

        // React the instant config changes on disk
        _monitor.OnChange(updated =>
            Console.WriteLine($"Config changed — port is now {updated.SmtpPort}"));
    }

    protected override async Task ExecuteAsync(CancellationToken stop)
    {
        while (!stop.IsCancellationRequested)
        {
            var current = _monitor.CurrentValue;   // always up to date
            Console.WriteLine($"Polling as {current.SenderName}");
            await Task.Delay(TimeSpan.FromSeconds(30), stop);
        }
    }
}
♾️
Lifetime: Singleton
Injectable into any lifetime, including other singletons and hosted services.
📡
CurrentValue + OnChange
Always current; fires a callback when the underlying config reloads.
🎛️
Get(name)
Also reads named options by name — useful for multiple instances of one type.
⚠️ OnChange can fire more than once

File systems sometimes raise multiple change events for a single save, so an OnChange handler may be invoked twice for one edit. Keep the callback idempotent, and remember CurrentValue reflects the newest values at the moment you read it.

08Choosing Between Them

All three read the same bound settings — you don't change the registration, only which interface you inject. They differ in lifetime and whether they see changes after startup. This decision is the single most common Options interview question.

Use this flowchart to pick quickly:

If they're set at deploy time and never touched again, you're on the easy path.
No → IOptions<T> 🧱
Simplest and fastest. Bound once, injectable everywhere. Done.
Yes · Where do you read them? 📍
The consuming service's lifetime decides the rest.

Inside a singleton / background worker → IOptionsMonitor<T> 📡
Use CurrentValue, and OnChange if you must react to updates.
🔑 Why can't a singleton take IOptionsSnapshot?

A snapshot is scoped (per request). Injecting a scoped service into a singleton would trap one request's scope inside the singleton forever — a "captive dependency." The DI container's scope validation detects this and throws at startup. For live values in a singleton, use the singleton-friendly IOptionsMonitor<T>.

09Validation

A missing or nonsensical setting shouldn't surface as a confusing crash on some rare code path. Options validation lets you declare the rules once and, with one extra call, check them the instant the app starts. There are three escalating styles.

1 · Data annotations

Decorate the settings class with attributes, then call ValidateDataAnnotations(). This needs the Microsoft.Extensions.Options.DataAnnotations package, which is already included for web SDK projects.

csharp · EmailSettings.cs with rules
using System.ComponentModel.DataAnnotations;

public class EmailSettings
{
    public const string SectionName = "EmailSettings";

    [Required]
    public string SenderName  { get; set; } = string.Empty;

    [Required, EmailAddress]
    public string FromAddress { get; set; } = string.Empty;

    [Range(1, 65535)]
    public int SmtpPort { get; set; }
}

2 · A custom .Validate(…) lambda

For a rule that spans multiple properties, chain a Validate delegate that returns true when valid, plus a message shown on failure.

csharp · cross-property rule
builder.Services
    .AddOptions<EmailSettings>()
    .BindConfiguration(EmailSettings.SectionName)
    .ValidateDataAnnotations()
    .Validate(s => s.SmtpPort != 25, "Port 25 is blocked; choose another.")
    .ValidateOnStart();

3 · An IValidateOptions<T> class

For complex or reusable logic, move validation into its own testable class. It returns a ValidateOptionsResult — Success or Fail(message).

csharp · EmailSettingsValidator.cs
using Microsoft.Extensions.Options;

public class EmailSettingsValidator : IValidateOptions<EmailSettings>
{
    public ValidateOptionsResult Validate(string? name, EmailSettings options)
    {
        if (options.RetryCount < 0)
            return ValidateOptionsResult.Fail("RetryCount cannot be negative.");

        return ValidateOptionsResult.Success;
    }
}

// register it:
builder.Services.AddSingleton<IValidateOptions<EmailSettings>, EmailSettingsValidator>();

Fail fast with ValidateOnStart

By default, validation runs lazily — the first time the options are actually read. That means a bad value might not blow up until a specific request hits it. ValidateOnStart() forces the checks to run during application startup instead, so a misconfiguration crashes the launch with a clear OptionsValidationException rather than failing later in production.

✅ Rule of thumb

Add .ValidateDataAnnotations().ValidateOnStart() to every options registration. The cost is negligible and it converts "3 a.m. production incident" into "the deployment refused to start." There's also a convenience registration, AddOptionsWithValidateOnStart<T>(), that turns on start-up validation for you.

⚠️ Annotations don't recurse by default

Data-annotation validation only inspects the top-level properties of the options class. To validate nested objects or list items, opt in with the [ValidateObjectMembers] and [ValidateEnumeratedItems] attributes.

10Hands-On Walkthrough

Here's the day's task assembled end to end: bind a strongly-typed settings class and validate it on startup. Follow the flow, then read the full Program.cs.

1 · Add the config section 📄
Put an EmailSettings block in appsettings.json.
2 · Write the class + rules 🧱
Properties mirror the keys; data annotations declare what "valid" means.
3 · Bind it 🔗
AddOptions<EmailSettings>().BindConfiguration("EmailSettings").
4 · Validate on startup 🚦
.ValidateDataAnnotations().ValidateOnStart() — the app refuses to launch if config is wrong.
5 · Inject & use ✅
Consume IOptions<EmailSettings> wherever the settings are needed.
csharp · complete Program.cs
using System.ComponentModel.DataAnnotations;
using Microsoft.Extensions.Options;

var builder = WebApplication.CreateBuilder(args);

// (2–4) Bind EmailSettings and validate it at startup
builder.Services
    .AddOptions<EmailSettings>()
    .BindConfiguration(EmailSettings.SectionName)
    .ValidateDataAnnotations()
    .Validate(s => s.SmtpPort != 25, "Port 25 is blocked; choose another.")
    .ValidateOnStart();

var app = builder.Build();

// (5) Read the strongly-typed settings via IOptions
app.MapGet("/email-config", (IOptions<EmailSettings> opts) =>
{
    EmailSettings s = opts.Value;
    return Results.Ok(new { s.SenderName, s.FromAddress, s.SmtpPort, s.RetryCount });
});

app.Run();

// ---- the settings class ----
public class EmailSettings
{
    public const string SectionName = "EmailSettings";

    [Required]
    public string SenderName  { get; set; } = string.Empty;

    [Required, EmailAddress]
    public string FromAddress { get; set; } = string.Empty;

    [Range(1, 65535)]
    public int SmtpPort { get; set; }

    [Range(0, 10)]
    public int RetryCount { get; set; }
}
🔑 What you get

If FromAddress is missing or SmtpPort is 70000, the app throws an OptionsValidationException during startup with a message naming the bad property — before it ever accepts a request. Fix the config, redeploy, done. That's the whole point of "validate on startup."

11Named Options & Extras

Two features round out the pattern for real-world apps.

Named options — several instances of one class

Sometimes you have multiple sections that share the same shape — say two payment tiers, or two email profiles. Instead of writing two classes, register the same class under different names and fetch each by name with Get(name).

csharp · registering & reading named options
builder.Services
    .Configure<EmailSettings>("Transactional",
        builder.Configuration.GetSection("Email:Transactional"))
    .Configure<EmailSettings>("Marketing",
        builder.Configuration.GetSection("Email:Marketing"));

app.MapGet("/senders", (IOptionsMonitor<EmailSettings> monitor) =>
{
    var tx = monitor.Get("Transactional");
    var mkt = monitor.Get("Marketing");
    return Results.Ok(new { tx.SenderName, mkt.SenderName });
});
⚠️ Named options and IOptions<T>

The plain IOptions<T> interface does not support named options — it only ever gives you the default (unnamed) instance. To read named options, use IOptionsSnapshot<T>.Get(name) or IOptionsMonitor<T>.Get(name).

PostConfigure — adjust after binding

PostConfigure<T> runs after all normal configuration, letting you tweak or force a value once everything else has been applied — handy for derived defaults.

csharp · PostConfigure
builder.Services.PostConfigure<EmailSettings>(s =>
{
    if (s.RetryCount == 0) s.RetryCount = 3;   // enforce a sane default
});

14Interview Questions

Click any question to reveal a plain-English answer. Grouped by level — start with 🌱 Junior and work down.

🌱 Junior (0–2 yrs)

Q1What is the Options pattern, in one sentence?Junior

It's a way to bind a section of configuration to a strongly-typed C# class and inject that class wherever you need it, so your code uses typed settings instead of loose string keys.

Q2Why is it better than reading IConfiguration with string keys?Junior

Three reasons: you get compile-time safety and autocomplete instead of typo-prone strings; values are already the right type, no manual conversion; and each class only sees its own settings rather than the entire configuration, which is cleaner and safer.

Q3What does "binding" mean here?Junior

Binding copies the values from a configuration section into the matching properties of a C# object. The framework matches by property name — a key called SmtpPort fills a property called SmtpPort — and converts the string value to the property's type.

Q4What does a settings class need to be bindable?Junior

It's a plain class with public properties that have setters (or init), whose names match the configuration keys. Matching is case-insensitive; properties without a matching key just keep their default value.

Q5How do you register a settings class in Program.cs?Junior

The simplest way is builder.Services.Configure<EmailSettings>(builder.Configuration.GetSection("EmailSettings")). The modern, chainable alternative is builder.Services.AddOptions<EmailSettings>().BindConfiguration("EmailSettings").

Q6How do you read the settings after binding them?Junior

Inject one of the accessor interfaces and read its value. Most commonly you inject IOptions<EmailSettings> and read the .Value property to get the bound object.

Q7Name the three Options interfaces.Junior

IOptions<T>, IOptionsSnapshot<T>, and IOptionsMonitor<T>. They all read the same bound settings but differ in lifetime and whether they notice configuration changes after startup.

Q8What is the point of validating options?Junior

To catch a missing or wrong setting early and clearly. Instead of the app silently using a wrong default or crashing on some rare code path, validation lets you declare the rules and check them — ideally at startup — so problems surface immediately.

🟠 Mid (2–5 yrs)

Q9Explain the difference between IOptions, IOptionsSnapshot, and IOptionsMonitor.Mid

IOptions<T> is a singleton bound once at startup — it never reflects later changes and is the fastest. IOptionsSnapshot<T> is scoped, recomputed once per request, so it picks up changes on the next request but can't be injected into a singleton. IOptionsMonitor<T> is a singleton that exposes the current value at any time through CurrentValue and can raise OnChange notifications — the right choice for singletons and background services.

Q10Why doesn't IOptions<T> pick up changes to appsettings.json?Mid

Because it binds the value once and caches it as a singleton for the app's lifetime. There's no re-binding and no file watching involved, so an edit after startup is never observed. That's a deliberate trade-off for speed and simplicity; if you need reloads you use Snapshot or Monitor.

Q11How does IOptionsMonitor access its value, and how is that different from the others?Mid

Through CurrentValue, not .Value. Each read of CurrentValue returns the latest bound value, so a long-lived singleton always sees current config. It also offers OnChange to run a callback when the configuration reloads, which the other two don't provide.

Q12What's the difference between Configure<T> and AddOptions<T>().BindConfiguration()?Mid

Both bind a section to a class and register it. Configure<T> returns the service collection, so you can't fluently chain validation onto it. AddOptions<T>() returns an OptionsBuilder that is chainable, letting you add ValidateDataAnnotations, Validate, and ValidateOnStart in one expression — which is why it's preferred for new code.

Q13Walk through the three ways to validate options.Mid

First, data annotations: decorate properties with attributes like [Required] and [Range] and call ValidateDataAnnotations(). Second, a custom lambda: chain .Validate(x => …, "message") for a rule spanning several properties. Third, an IValidateOptions<T> class: move the logic into a testable class whose Validate method returns ValidateOptionsResult.Success or Fail(message).

Q14What does ValidateOnStart() change, and why use it?Mid

By default, options validation is lazy — it runs the first time the options are read, which could be deep into a request. ValidateOnStart() forces validation to run during application startup, so an invalid configuration throws an OptionsValidationException and prevents the app from starting. You use it so config mistakes fail the deployment loudly instead of surfacing unpredictably at runtime.

Q15You edited appsettings.json but a controller still shows old values. What's likely wrong?Mid

The controller is probably injecting IOptions<T>, which is bound once and never reloads. Switching to IOptionsSnapshot<T> makes each request re-read the current configuration, so the edit shows up on the next request. (Also confirm the file's reload-on-change is active and you saved the file.)

Q16What are named options and when would you use them?Mid

Named options let you register the same settings class multiple times under different names — useful when several config sections share the same shape (for example, two email profiles). You register each with a name and read a specific one with Get("Name") on IOptionsSnapshot or IOptionsMonitor. Plain IOptions<T> doesn't support them.

🔴 Senior (5–7 yrs)

Q17Why can't IOptionsSnapshot<T> be injected into a singleton, and what happens if you try?Senior

IOptionsSnapshot is registered as scoped, and injecting a scoped service into a singleton is a lifetime violation: the singleton would capture one request's scope and hold it forever — a "captive dependency" that leaks state and defeats the per-request semantics. With scope validation enabled (the default in development), the DI container detects this and throws at startup. The correct choice for live values inside a singleton is IOptionsMonitor<T>, which is itself a singleton.

Q18How would you validate options in a background/hosted service, and what subtlety applies?Senior

Register validation the usual way with ValidateOnStart() so bad config fails the launch. Inside the hosted service, inject IOptionsMonitor<T> and read CurrentValue so you always act on the latest config, optionally subscribing with OnChange. The subtlety: OnChange can fire more than once for a single file save because the file-system watcher may raise duplicate events, so keep the handler idempotent and avoid heavy work directly inside it.

Q19When would you choose IValidateOptions<T> over data annotations or a lambda?Senior

When validation is complex, cross-cutting, or needs to be unit-tested and reused. A class implementing IValidateOptions<T> can express multi-field rules, return rich failure messages via ValidateOptionsResult, and be tested in isolation without spinning up the host. Data annotations are best for simple per-property rules; inline lambdas suit one-off cross-property checks; a validator class is the maintainable home for real business rules.

Q20Data-annotation validation on a nested object isn't running. Why, and how do you fix it?Senior

By default, ValidateDataAnnotations only validates the top-level properties of the options class; it doesn't recurse into nested objects or into items of a collection. You enable recursive validation explicitly with the [ValidateObjectMembers] attribute on the nested property and [ValidateEnumeratedItems] on collections — or move the logic into an IValidateOptions<T> that walks the graph itself.

Q21What's the role of IOptionsMonitorCache and IOptionsFactory?Senior

IOptionsFactory<T> is responsible for creating options instances — it runs all registered IConfigureOptions then IPostConfigureOptions, and distinguishes named from default instances. IOptionsMonitor<T> uses an IOptionsMonitorCache<T> to cache those instances; its TryRemove invalidates a cached instance so it's recomputed on next access, TryAdd injects one manually, and Clear forces all named instances to be recreated on demand. This machinery is what lets a singleton monitor return fresh values and support selective invalidation.

Q22What does PostConfigure do and when is it the right tool?Senior

PostConfigure<T> registers a delegate that runs after all normal Configure steps have applied, so it always has the last word. It's the right tool for derived or fallback values that must be computed once every other source has been merged — for example, filling a default when a key was left empty, or normalizing a value — without those adjustments being overwritten by a later configuration step. There's also PostConfigureAll to apply the adjustment across every named instance.

Q23Some argue against injecting IOptions<T> directly into domain code. What's the reasoning and alternative?Senior

The concern is that IOptions<T> is an infrastructure abstraction from Microsoft.Extensions.Options; threading it through business classes couples your domain to that framework type and forces consumers to know about .Value. A common alternative is to bind the options at composition root and register the plain settings object itself (for example, resolve IOptions<T>.Value once and add it as a singleton), so domain code depends only on EmailSettings. The trade-off is losing reload and named-options support, so it's a judgment call: keep IOptions* at the edges, inject the POCO into the core when reloads aren't needed.

Q24How does options validation interact with the different accessors' timing?Senior

Validation runs the first time an options instance is created, and it re-runs whenever the options are recomputed. So without ValidateOnStart, an IOptions<T> value is validated at first access and then never again (it's cached); an IOptionsSnapshot<T> is validated on the first access within each request, meaning a bad reload can surface per request; and an IOptionsMonitor<T> re-validates when the value is recomputed after a change. Calling ValidateOnStart forces the first creation to happen at startup, giving you a single, predictable failure point regardless of which accessor consumers later use.
