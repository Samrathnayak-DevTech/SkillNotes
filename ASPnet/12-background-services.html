Day 12 · ASP.NET Core Fundamentals

⚙️ Background Services: Work That Runs Itself

Not everything happens because a user made a request. Emails need sending, caches need refreshing, old data needs cleaning, queues need draining — continuously, in the background, for as long as the app is alive. Today you'll build these with BackgroundService, run work on a timer, and solve the one gotcha that trips everyone: using scoped services like DbContext inside them.

🏠 IHostedService ⚙️ BackgroundService ⏰ PeriodicTimer 🔒 Scoped Services 🛑 Graceful Shutdown

01🏭 Work Without a Request

Every ASP.NET Core feature so far has been request-driven: a user hits an endpoint, code runs, a response goes back. But real applications also have work that isn't tied to any request — it just needs to happen, over and over, while the app runs:

📧Draining a queue

Process outbound emails, notifications, or jobs that other parts of the app enqueued.

🧹Cleanup

Delete expired sessions, archive old orders, purge stale cache entries every hour.

🔄Syncing

Poll an external API for exchange rates or stock levels every few minutes (Day 11 tie-in).

💓Health & metrics

Emit heartbeats, recompute dashboards, or check dependencies on a schedule.

In .NET, this is the job of a hosted service: a class the host starts when the app boots and stops when it shuts down, running independently of the request pipeline. The base class you'll almost always use is BackgroundService.

🏢 Analogy

Your web endpoints are the front-desk staff — they only act when a customer walks in. A background service is the building's night-shift crew and facilities systems: the cleaner who comes through every hour, the boiler that cycles on a timer, the mail room that processes the outgoing pile. Nobody "requests" them each time — they run on their own schedule for as long as the building is open, and they shut down cleanly when it closes.

🔑 Key Idea

A hosted service's lifetime is the application's lifetime. The host starts it at boot and signals it to stop at shutdown. Your job is to do useful work in between — and to stop promptly and cleanly when asked. Everything today builds on the async and cancellation skills from Day 10.

03🏠 IHostedService: The Raw Contract

At the bottom of it all is a tiny interface with just two methods. The host calls one when the app starts and the other when it stops:

C# · The interface
public interface IHostedService
{
    Task StartAsync(CancellationToken cancellationToken);  // runs once at boot
    Task StopAsync(CancellationToken cancellationToken);   // runs once at shutdown
}

You register an implementation with AddHostedService<T>(), and the host takes care of calling it. Use the raw interface directly when you have one-off startup/shutdown work — warm a cache, register with a service discovery system, open and later close a connection:

C# · A simple startup task
public class StartupWarmupService(ILogger<StartupWarmupService> logger) : IHostedService
{
    public Task StartAsync(CancellationToken ct)
    {
        logger.LogInformation("Warming up on startup");
        // ... one-time work ...
        return Task.CompletedTask;
    }

    public Task StopAsync(CancellationToken ct) => Task.CompletedTask;
}

// Program.cs:
builder.Services.AddHostedService<StartupWarmupService>();
⚠️ StartAsync Blocks Startup — Don't Loop In It

StartAsync runs as part of the host starting up, and the host waits for it to return before the app is fully up. So a while (true) loop in StartAsync would block your app from ever finishing startup. Startup work here must be quick and return promptly. For continuous or long-running work, you want BackgroundService (next section) — not the raw interface.

🔑 Registered as a Singleton

Every hosted service, however registered, is a singleton — one instance for the whole app lifetime. That single fact is the root of the scoped-service problem in section 06, so hold onto it.

04⚙️ BackgroundService: The Everyday Choice

Implementing StartAsync/StopAsync correctly for long-running work (kicking off a task without blocking startup, tracking it, stopping it cleanly) is fiddly. BackgroundService is an abstract base class that does all that plumbing and leaves you one method to override: ExecuteAsync. It's the right choice for virtually all continuous background work.

C# · The minimal worker
public class Worker(ILogger<Worker> logger) : BackgroundService
{
    // stoppingToken is signaled when the host is shutting down.
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            logger.LogInformation("Worker running at {Time}", DateTimeOffset.Now);
            await Task.Delay(TimeSpan.FromSeconds(5), stoppingToken);
        }
    }
}

// Program.cs:
builder.Services.AddHostedService<Worker>();

How the host drives it, start to finish:

🚀
1. App boots → host calls StartAsyncBackgroundService's built-in StartAsync kicks off ExecuteAsync and returns immediately — startup isn't blocked.
🔁
2. ExecuteAsync runs your loopYour work runs concurrently with the rest of the app for its whole lifetime — usually a loop that does work, then awaits a delay.
🛑
3. Shutdown → stoppingToken is cancelledOn Ctrl+C or app stop, the host cancels the token. Your loop condition and awaited delays observe it and exit the loop.
⏳
4. Host waits (briefly) for ExecuteAsync to finishStopAsync blocks until your task completes or a shutdown timeout elapses — so finishing promptly matters (section 07).
⚠️ Don't Block Before the First await

The host considers startup "still going" until ExecuteAsync hits its first await. If you do heavy synchronous work at the top of ExecuteAsync before any await, you delay the entire app's startup. Get to an await quickly (the loop's await Task.Delay counts), and do heavy work inside the loop.

Use…When
IHostedService directlyQuick one-off work at startup/shutdown that returns fast.
BackgroundServiceContinuous or periodic long-running work — the overwhelming majority of cases.

05⏰ Running Work Every N Seconds

The naive loop above uses Task.Delay, which works but has a subtle flaw: the interval is delay + work time, so it drifts. If your work takes 2 seconds and you delay 5, ticks land ~7 seconds apart. The modern, purpose-built tool is PeriodicTimer — you await WaitForNextTickAsync and it accounts for elapsed time, and it doesn't waste a thread while waiting.

C# · PeriodicTimer — the recommended pattern
public class PeriodicWorker(ILogger<PeriodicWorker> logger) : BackgroundService
{
    private readonly TimeSpan _period = TimeSpan.FromSeconds(10);

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        using PeriodicTimer timer = new(_period);

        // WaitForNextTickAsync returns true each tick, false when cancelled:
        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            logger.LogInformation("Tick at {Time}", DateTimeOffset.Now);
            // ... do the periodic work ...
        }
    }
}
🚇 Analogy

Task.Delay is like a bus driver who, at each stop, waits exactly 5 minutes after unloading before leaving — so the more passengers (work) at a stop, the later everything downstream runs; the schedule drifts. PeriodicTimer is a metro on a fixed timetable: trains are scheduled every 10 minutes regardless of how long each stop took. Same rhythm every time, no accumulating lateness.

✅ Run Immediately, Then on Schedule

WaitForNextTickAsync waits before the first tick, so nothing runs for the first period. If you want the work to fire once at startup and then every N seconds, call your DoWork() once before the loop, then enter the timer loop — a common and tidy pattern.

⚠️ Long Work vs. the Interval

PeriodicTimer ticks are not reentrant — a new tick won't overlap while you're still awaiting the previous iteration's work. If a tick's work takes longer than the period, the next tick simply fires as soon as you're free (you don't get pile-up, but you do get catch-up). If overlap-free spacing matters, that behavior is exactly what you want; if you need strict fixed timing regardless of work duration, you'd run the work on a separate task — but for most jobs, the built-in non-overlap is the safe default.

06🔒 The Scoped Service Problem

Here's the wall almost everyone hits. You want your worker to use the DbContext from Days 7–9, so you inject it into the constructor — and the app crashes at startup with:

Error · at startup
Cannot consume scoped service 'ShopContext' from singleton
'Microsoft.Extensions.Hosting.IHostedService'.

Why? Recall two facts we've established: hosted services are singletons (section 03), and DbContext is registered scoped — one per request, disposed when the request ends (Day 7). A singleton lives for the whole app; a scoped service is meant to live briefly. Letting a singleton hold a scoped service forever would keep one DbContext alive for the app's entire lifetime — exactly the bug the DI container refuses to let you create.

lives forever
✗ can't inject →
Scoped DbContext
meant to be brief
Lifetime mismatch · a long-lived singleton must not permanently hold a short-lived scoped service

The fix: create a scope per unit of work 🎯

Inject IServiceScopeFactory (a singleton, so allowed), and each tick open a fresh scope, resolve your scoped services from it, do the work, and dispose the scope. That mirrors what a web request does automatically — one scope per unit of work, one DbContext per scope:

C# · The correct pattern
public class OrderArchiverWorker(
    IServiceScopeFactory scopeFactory,
    ILogger<OrderArchiverWorker> logger) : BackgroundService
{
    private readonly TimeSpan _period = TimeSpan.FromSeconds(30);

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        using PeriodicTimer timer = new(_period);
        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            // 🎯 A fresh scope per tick — like a mini-request:
            await using var scope = scopeFactory.CreateAsyncScope();
            var db = scope.ServiceProvider.GetRequiredService<ShopContext>();

            var cutoff = DateTime.UtcNow.AddYears(-1);
            var archived = await db.Orders
                .Where(o => o.Status == OrderStatus.Cancelled && o.PlacedAt < cutoff)
                .ExecuteUpdateAsync(s => s.SetProperty(o => o.IsArchived, true), stoppingToken);

            logger.LogInformation("Archived {Count} old orders", archived);
            // scope disposed here → DbContext disposed, connection released
        }
    }
}
🔑 The Mental Model

A web request creates a DI scope for you; a background service doesn't get one automatically. So you create one yourself, per iteration. Think of each timer tick as a tiny self-made "request": open scope → resolve scoped deps → work → dispose. Never hold the scope (or its DbContext) across ticks — a fresh one each time keeps you clean and avoids stale, ever-growing state.

✅ Delegate to a Scoped Service

Cleaner still: put the actual work in its own scoped service (e.g. IOrderArchiver), and have the worker's only job be "each tick, create a scope, resolve the archiver, call it." The worker handles timing; the scoped service handles logic — testable in isolation, and the scope boundary is obvious.

⚠️ CreateScope vs CreateAsyncScope

Use CreateAsyncScope() (with await using) when the scope contains services that implement IAsyncDisposable — which DbContext does. Plain CreateScope() with using works too but disposes synchronously; for EF Core, the async scope is the tidier match.

07🛑 Graceful Shutdown

When the app stops — a deploy, a container scaling down, Ctrl+C — the host wants your worker to finish what it's doing and exit promptly. Two mechanisms make this work: the stopping token and an optional StopAsync override.

Honor the stopping token everywhere 🚦

The stoppingToken passed to ExecuteAsync is cancelled at shutdown. Your loop condition checks it, and — critically — you pass it into every awaited call (Day 10!) so in-flight work can abort. That's what lets the app stop in milliseconds instead of hanging.

Shutdown signal
Ctrl+C / SIGTERM
stoppingToken cancelled
host fires it
loop & awaits observe it
exit cleanly
StopAsync completes
app exits
Graceful shutdown chain · a token nobody checks means a hang, then a forced kill
⚠️ The Shutdown Timeout

The host gives your service a limited window to stop (default ~30 seconds, configurable via HostOptions.ShutdownTimeout). If ExecuteAsync doesn't return within it — because you ignored the token and are mid-work — the host forcibly kills the process. That can mean half-finished work, an unflushed log, or a dangling transaction. Honoring the token isn't optional politeness; it's how you avoid data corruption on shutdown.

Override StopAsync for cleanup 🧹

For "finish the current item, then stop accepting new ones" logic — common with queues — override StopAsync to run extra cleanup after the token fires:

C# · Custom stop logic
public override async Task StopAsync(CancellationToken cancellationToken)
{
    logger.LogInformation("Worker stopping — finishing in-flight work");
    // e.g. stop pulling new queue items, let the current one complete
    await base.StopAsync(cancellationToken);   // let the base do its part too
}
✅ Two Different Tokens

Don't confuse them: the stoppingToken in ExecuteAsync signals "begin stopping." The cancellationToken parameter of StopAsync represents the shutdown-timeout deadline for cleanup itself. In modern hosting they behave consistently, but the mental split — "stop working" vs "you have this long to clean up" — is worth keeping.

08🧯 Error Handling: Don't Let One Failure Kill the Worker

The nastiest background-service bug is silent death. If an unhandled exception escapes ExecuteAsync, the task faults and your worker simply stops running — permanently, until the app restarts. No more ticks, often no obvious error, just a job that mysteriously quit hours ago.

⚠️ A Thrown Exception Ends the Loop Forever

An exception from inside your loop that isn't caught propagates out of ExecuteAsync, the task enters the faulted state (Day 10!), and the BackgroundService is done. The rest of your app keeps running, which is why this is so easy to miss. The fix is to wrap each iteration's work in try/catch so one bad tick is logged and the loop lives on.

C# · Resilient loop
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    using PeriodicTimer timer = new(_period);
    while (await timer.WaitForNextTickAsync(stoppingToken))
    {
        try
        {
            await using var scope = scopeFactory.CreateAsyncScope();
            var svc = scope.ServiceProvider.GetRequiredService<IEmailDispatcher>();
            await svc.SendPendingAsync(stoppingToken);
        }
        catch (OperationCanceledException) when (stoppingToken.IsCancellationRequested)
        {
            break;   // expected on shutdown — leave the loop quietly
        }
        catch (Exception ex)
        {
            // Log and KEEP GOING — one failed tick shouldn't kill the worker
            logger.LogError(ex, "Email dispatch tick failed; will retry next period");
        }
    }
}
🔑 Catch Inside the Loop, Not Around It

Put the try/catch inside the while so a failure aborts only the current iteration; the next tick tries again. A try/catch around the whole loop would still end the worker on the first error. And distinguish shutdown cancellation (expected — break) from real errors (log and continue).

✅ Since .NET 6: BackgroundServiceExceptionBehavior

By default in modern .NET, an unhandled exception in a hosted service will stop the entire host (fail-fast — arguably better than a silently-dead worker, because you notice). You can tune this with HostOptions.BackgroundServiceExceptionBehavior. But don't rely on it — handle exceptions per-tick as above so a transient blip (a brief DB hiccup) doesn't take anything down. Reserve fail-fast for truly unrecoverable states.

09📮 Queues & Real Scheduling

Two patterns extend the periodic worker into the shapes real systems need.

Queued background work 📮

Remember Day 10's warning against fire-and-forget? This is the correct alternative. An endpoint that needs to hand off slow work (send an email, generate a report) enqueues an item; a background service dequeues and processes it. The modern tool is System.Threading.Channels — a thread-safe async producer/consumer queue:

C# · Channel-based queue + consumer
// A singleton queue both sides share:
public class EmailQueue
{
    private readonly Channel<EmailJob> _channel = Channel.CreateUnbounded<EmailJob>();
    public ValueTask EnqueueAsync(EmailJob job) => _channel.Writer.WriteAsync(job);
    public IAsyncEnumerable<EmailJob> ReadAllAsync(CancellationToken ct) => _channel.Reader.ReadAllAsync(ct);
}

// The consumer runs continuously, one scope per job:
public class EmailWorker(EmailQueue queue, IServiceScopeFactory scopeFactory) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        await foreach (var job in queue.ReadAllAsync(stoppingToken))
        {
            try
            {
                await using var scope = scopeFactory.CreateAsyncScope();
                var sender = scope.ServiceProvider.GetRequiredService<IEmailSender>();
                await sender.SendAsync(job, stoppingToken);
            }
            catch (Exception ex) { /* log & continue */ }
        }
    }
}

Scheduling: "every day at 2 AM" ⏰

PeriodicTimer handles "every N seconds/minutes/hours." For calendar schedules — 2 AM daily, Mondays only — compute the next occurrence and Task.Delay until it, typically with a cron library:

C# · Cron-style scheduling (sketch)
// Using a cron expression (e.g. via the Cronos library):
while (!stoppingToken.IsCancellationRequested)
{
    DateTimeOffset now = DateTimeOffset.Now;
    DateTimeOffset? next = _cron.GetNextOccurrence(now, _timeZone);
    if (next is null) break;

    var delay = next.Value - now;
    if (delay > TimeSpan.Zero)
        await Task.Delay(delay, stoppingToken);   // wait until the scheduled time

    await DoWorkAsync(stoppingToken);
}
⚠️ Multiple Instances = Multiple Runs

Background services run in every instance of your app. Scale to three pods and your "nightly cleanup" runs three times, at once — possibly corrupting each other. For scheduled jobs in a scaled-out deployment you need coordination: a distributed lock (only one instance wins the job), a leader-election mechanism, or a dedicated scheduler like Hangfire/Quartz.NET with a shared store. In-process timers are perfect for single-instance apps and per-instance work (like draining that instance's own queue); think twice before using them for cluster-wide scheduled tasks.

✅ When to Reach for a Real Job Framework

Built-in background services are ideal for simple, in-process, best-effort work. When you need persistence (jobs survive restarts), retries with history, a dashboard, or guaranteed single execution across a cluster, graduate to Hangfire or Quartz.NET, or offload to a message queue (RabbitMQ, Azure Service Bus) with dedicated consumers. Know the ceiling of the built-in tools.

10🛠️ Hands-On: A Worker That Runs Every N Seconds

Today's Do: build a worker that runs a task on an interval. Reuse the shop DbContext from earlier days so you also practice the scoped-service pattern. The path:

  1. ⚙️ Create a class deriving from BackgroundService and register it with builder.Services.AddHostedService<YourWorker>().
  2. ⏰ In ExecuteAsync, use a PeriodicTimer with a period like 10 seconds and a while (await timer.WaitForNextTickAsync(stoppingToken)) loop — log a message each tick to confirm it fires.
  3. 🔒 Do real work that needs a scoped service: inject IServiceScopeFactory, and each tick await using var scope = scopeFactory.CreateAsyncScope(); then resolve ShopContext. Count the orders, or archive old ones.
  4. 🧯 Wrap the per-tick work in try/catch so one failure logs and the loop continues; distinguish shutdown cancellation (break) from real errors (log & continue).
  5. 🛑 Verify graceful shutdown: run the app, watch the ticks, press Ctrl+C — confirm it stops promptly and logs a stopping message rather than hanging.
  6. 💥 Prove the failure mode: temporarily throw inside the loop without a try/catch, watch the worker die silently while the app keeps running — then add the try/catch back and watch it survive.
  7. 🎛️ Bonus: move the actual work into a scoped service (e.g. IOrderArchiver.RunAsync) and have the worker only handle timing + scope creation. Notice how testable the logic becomes.
  8. 🔬 Bonus: try the "run immediately, then every N seconds" variant by calling the work once before the timer loop.
C# · Complete, runnable worker
public class OrderStatsWorker(
    IServiceScopeFactory scopeFactory,
    ILogger<OrderStatsWorker> logger) : BackgroundService
{
    private readonly TimeSpan _period = TimeSpan.FromSeconds(10);

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        logger.LogInformation("OrderStatsWorker started");
        using PeriodicTimer timer = new(_period);

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            try
            {
                await using var scope = scopeFactory.CreateAsyncScope();
                var db = scope.ServiceProvider.GetRequiredService<ShopContext>();

                int total = await db.Orders.CountAsync(stoppingToken);
                decimal today = await db.Orders
                    .Where(o => o.PlacedAt.Date == DateTime.UtcNow.Date)
                    .SumAsync(o => o.Total, stoppingToken);

                logger.LogInformation(
                    "Order stats — total: {Total}, today's revenue: {Today:C}", total, today);
            }
            catch (OperationCanceledException) when (stoppingToken.IsCancellationRequested)
            {
                break;
            }
            catch (Exception ex)
            {
                logger.LogError(ex, "Order stats tick failed; continuing");
            }
        }

        logger.LogInformation("OrderStatsWorker stopping");
    }
}

// Program.cs:
builder.Services.AddHostedService<OrderStatsWorker>();

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is a hosted service and when would you use one?🌱 JUNIOR

A hosted service is a class that runs background work managed by the .NET host — it starts when the app boots and stops when the app shuts down, independently of any HTTP request. You use one for work that isn't triggered by a user: processing a queue of jobs, cleaning up old data on a schedule, polling an external API for updates, emitting metrics. Anything that "just needs to keep happening" while the app runs.

Q2What's the difference between IHostedService and BackgroundService?🌱 JUNIOR

IHostedService is the raw interface with two methods — StartAsync (called at boot) and StopAsync (called at shutdown). BackgroundService is an abstract base class that already implements IHostedService and handles the tricky plumbing for long-running work, leaving you one method to override: ExecuteAsync. Use the raw interface for quick one-off startup/shutdown tasks; use BackgroundService for continuous or periodic work, which is almost always what you want.

Q3How do you register a background service?🌱 JUNIOR

In Program.cs, call builder.Services.AddHostedService<MyWorker>(). That tells the host to create and manage the service's lifetime — starting it when the app runs and stopping it on shutdown. You can register multiple hosted services; they all run concurrently for the app's lifetime.

Q4What is the stoppingToken in ExecuteAsync for?🌱 JUNIOR

It's a CancellationToken the host cancels when the app is shutting down. Your work loop checks it (while (!stoppingToken.IsCancellationRequested)) and passes it into every awaited call, so the service can stop promptly and cleanly when asked instead of running forever. It's the same cooperative-cancellation idea from the async lesson, applied to the service's whole lifetime.

Q5How do you run a task every N seconds in a background service?🌱 JUNIOR

The recommended way is PeriodicTimer: create one with your interval, then loop with while (await timer.WaitForNextTickAsync(stoppingToken)) { /* work */ }. Each tick fires roughly every N seconds. It's better than doing work and then Task.Delay(N), because that approach adds the work's duration to the interval and drifts over time, whereas PeriodicTimer keeps a steady rhythm.

Q6Why can't you inject a DbContext directly into a BackgroundService?🌱 JUNIOR

Because hosted services are registered as singletons (one instance for the whole app), but DbContext is registered as scoped (a short-lived instance meant to last one operation). The DI container refuses to let a long-lived singleton permanently hold a short-lived scoped service — it throws at startup. Instead you inject IServiceScopeFactory, create a scope when you need to work, and resolve the DbContext from that scope.

Q7Roughly how do you use a scoped service inside a background service?🌱 JUNIOR

Inject IServiceScopeFactory into the worker. Each time you need to do work (e.g. each timer tick), create a scope with await using var scope = scopeFactory.CreateAsyncScope();, then get your scoped service from scope.ServiceProvider.GetRequiredService<T>(), use it, and let the scope dispose at the end of the block. Each scope is like a mini-request with its own fresh DbContext.

Q8Give two real-world examples of background service work.🌱 JUNIOR

Examples: processing a queue of outbound emails or notifications that endpoints enqueued; periodically cleaning up expired sessions or archiving old orders; polling an external API every few minutes to refresh cached exchange rates; emitting heartbeat/health metrics on a schedule; running a nightly report generation. The common thread: work that recurs or runs continuously and isn't tied to a specific user request.

🟠 Mid

Q9Why is PeriodicTimer preferred over a work-then-Task.Delay loop?🟠 MID

Two reasons. Accuracy: with "do work, then Task.Delay(period)," the real interval becomes work time + delay, so ticks drift later as work takes longer — a job "every 10s" that does 3s of work actually runs every 13s. PeriodicTimer schedules ticks on the interval itself, accounting for elapsed work, so timing stays steady. Design: it's purpose-built for this, reads cleanly (await WaitForNextTickAsync), integrates cancellation, doesn't tie up a thread while waiting, and its ticks don't overlap — a new tick won't start until the previous iteration's awaited work finishes.

Q10Explain the singleton/scoped lifetime mismatch and exactly how you resolve it.🟠 MID

A hosted service is a singleton — constructed once, lives for the app's lifetime. Scoped services (like DbContext) are designed to live for one unit of work and then be disposed. If a singleton captured a scoped service via constructor injection, that scoped instance would live forever inside the singleton — defeating scoping and, for DbContext, accumulating tracked entities and holding a connection indefinitely. The DI container detects this and throws at startup. Resolution: inject IServiceScopeFactory (itself a singleton, so injectable), and for each unit of work call CreateScope()/CreateAsyncScope(), resolve the scoped service from that scope, do the work, and dispose the scope — recreating per iteration the scope boundary that a web request would provide automatically.

Q11What happens if an unhandled exception escapes ExecuteAsync?🟠 MID

The Task returned by ExecuteAsync faults, and the background service stops running — it won't tick again until the app restarts. In modern .NET (since 6), the default BackgroundServiceExceptionBehavior is actually to stop the whole host (fail-fast), which at least makes the failure loud rather than a silently dead worker; you can change it to Ignore. Either way, the fix isn't to rely on that setting — it's to wrap each iteration's work in try/catch inside the loop, log the error, and continue, so a transient failure (a brief DB blip) costs one tick, not the whole worker.

Q12Why put the try/catch inside the loop rather than around it?🟠 MID

Scope of recovery. A try/catch inside the while catches a failure from a single iteration, logs it, and lets the loop proceed to the next tick — the worker survives a bad tick. A try/catch around the whole loop catches the exception only after the loop has already been torn out from under it, so the worker still stops on the first error. You want per-iteration isolation. You also typically special-case OperationCanceledException when the stopping token is set — that's expected shutdown, so you break out quietly rather than logging it as an error.

Q13How does graceful shutdown work, and what's the shutdown timeout?🟠 MID

On shutdown (Ctrl+C, SIGTERM, deploy), the host cancels the stoppingToken and then waits for ExecuteAsync to return, calling StopAsync. Your loop and awaited calls observe the token and exit promptly. But the host only waits so long — the shutdown timeout (default ~30 seconds, set via HostOptions.ShutdownTimeout). If your service ignores the token and is still mid-work when the timeout elapses, the host forcibly terminates the process, risking half-finished work, unflushed logs, or dangling transactions. So honoring the token (checking it, passing it into every await) is what makes shutdown both graceful and safe.

Q14You need to process work items enqueued by API endpoints. How do you structure that?🟠 MID

Producer/consumer with a shared queue and a background consumer — the correct alternative to fire-and-forget. Use System.Threading.Channels: register a singleton wrapping a Channel<WorkItem>; endpoints call Writer.WriteAsync(item) and return immediately; a BackgroundService loops with await foreach (var item in channel.Reader.ReadAllAsync(stoppingToken)), creating a fresh DI scope per item, resolving scoped dependencies, processing, and catching exceptions per item. This gives the work an owned lifetime (it doesn't die with the request), its own scope (fresh DbContext), and observed errors — everything bare fire-and-forget lacks.

Q15Should you inject a scoped service into StartAsync of a raw IHostedService? What's the catch?🟠 MID

You can't inject scoped services into the constructor (the service is a singleton) — same rule as BackgroundService. Inside StartAsync you can create a scope and resolve scoped services from it for one-time startup work. The bigger catch is that StartAsync runs during host startup and the host waits for it to return before the app is ready — so it must be quick. Don't do a long loop or heavy blocking work there, or you delay (or prevent) the app from starting. For anything long-running, use BackgroundService.ExecuteAsync, which is designed not to block startup.

Q16CreateScope vs CreateAsyncScope — when does the difference matter?🟠 MID

CreateScope() returns a scope disposed synchronously (using); CreateAsyncScope() returns one disposed asynchronously (await using). The difference matters when the scope holds services implementing IAsyncDisposable — notably EF Core's DbContext, which has async disposal to flush/close things properly. With a synchronous scope disposing an async-disposable service, you can miss the async cleanup path (and some setups throw). So when your scope resolves a DbContext or other async-disposable resource, prefer await using var scope = factory.CreateAsyncScope();.

🔴 Senior

Q17Your app scales to multiple instances. What breaks in a naive periodic worker, and how do you fix it?🔴 SENIOR

Background services run in every instance, so a "nightly cleanup" or "send digest emails" job that assumes a single execution now fires once per instance — three pods, three simultaneous runs — causing duplicate emails, double-processing, or writes racing each other. Fixes depend on the work: for per-instance work (draining this instance's own in-memory queue) it's correct as-is. For cluster-wide singleton work you need coordination: a distributed lock (e.g. via Redis or a database row/advisory lock) so only the instance that acquires it runs the job; leader election so one instance owns scheduled tasks; or, best for anything nontrivial, a dedicated scheduler like Hangfire or Quartz.NET backed by a shared store that guarantees single execution and gives retries, history, and a dashboard. The design question to ask first is "is this work per-instance or global?" — that determines the whole approach.

Q18Walk through implementing graceful shutdown for a worker draining a queue with in-flight items.🔴 SENIOR

Goal: on shutdown, stop accepting new items but finish (or safely abandon) what's in progress, within the shutdown budget. Approach: the consumer loops on ReadAllAsync(stoppingToken); when the token cancels, the enumeration ends and no new items are pulled. For in-flight items, decide the policy — either let the current item complete before exiting (track active count, await it), or make item processing itself cancellation-aware and idempotent so a re-run after restart is safe. Override StopAsync to log and coordinate the drain, and mind the shutdown timeout: if draining could exceed it, either raise HostOptions.ShutdownTimeout deliberately or design items to be re-processable so an abrupt stop loses nothing. The robust real-world answer usually combines "finish the current item + idempotent processing + a durable queue," so that whatever wasn't finished is simply picked up again after restart — you stop trying to guarantee completion in-process and instead guarantee it's never lost.

Q19A background worker calls an external API and a database every 30 seconds. How do you make it production-grade?🔴 SENIOR

Layer the concerns from earlier days. Scope per tick for the DbContext (scoped, disposed each iteration). Resilience on the HTTP call — a typed client with AddStandardResilienceHandler (Day 11) so a flaky API retries/circuit-breaks rather than throwing every tick. Cancellation threaded from stoppingToken into both the HTTP and EF calls so shutdown is instant. Per-tick try/catch so one failed cycle logs and the worker survives; classify errors (transient → fine, keep going; persistent → alert). Idempotency so re-running after a restart or a scaled-out double-run doesn't corrupt data. Observability — structured logs with a correlation/run id, metrics on success/failure counts and duration, and a health check reflecting the worker's last successful run so ops can see it silently dying. Config — interval and toggles from configuration, not hard-coded. And decide the single-vs-multi-instance story (Q17). That combination turns a demo loop into something you can trust unattended.

Q20How would you schedule "run at 2 AM every day" reliably, and why isn't PeriodicTimer enough?🔴 SENIOR

PeriodicTimer only does fixed intervals, not calendar times — "every 24 hours from when the app started" drifts relative to wall-clock 2 AM and breaks across restarts and daylight-saving changes. For calendar schedules you compute the next occurrence from a cron expression (Cronos/NCrontab), honoring a specific time zone, and Task.Delay until then, looping. But the harder problems are operational: across multiple instances it'd fire multiple times (needs a distributed lock or leader election); if the app is down at 2 AM the run is simply missed (needs catch-up/misfire handling); and there's no history or retry. Once you need time-zone correctness, missed-run handling, single execution across a cluster, persistence, and a dashboard, the honest answer is to stop hand-rolling it and use Quartz.NET or Hangfire with a shared store — reliable scheduling is a solved problem, and rebuilding it is where subtle bugs live.

Q21How do you unit/integration test a BackgroundService?🔴 SENIOR

Keep the logic out of the timing loop so it's testable. Push the real work into a scoped service (e.g. IOrderArchiver.RunAsync) and unit-test that directly with mocked dependencies — no host, no timer. For the worker itself, test its orchestration: because BackgroundService exposes StartAsync/StopAsync, you can start it, let it run a tick or two (using a short interval, or a fake timer/abstraction you inject so you control ticks deterministically), assert the collaborator was invoked, then stop it and assert it shuts down promptly and honors cancellation. Inject an IServiceScopeFactory from a real test service provider so scope creation is exercised. For end-to-end confidence, an integration test hosts the service against a real (containerized) database and verifies the effect. Avoid tests that sleep for real seconds — abstract the timer so the test drives ticks instantly.

Q22When do the built-in background services stop being the right tool?🔴 SENIOR

They're ideal for simple, in-process, best-effort work whose loss on a restart is acceptable. They stop being enough when you need: durability (jobs must survive process crashes/restarts — an in-memory Channel loses its contents), guaranteed-once execution across a cluster, retries with backoff and a record of attempts, scheduling with missed-run handling, a management dashboard, or heavy throughput that warrants separate scaling. At that point graduate to Hangfire or Quartz.NET (persistent jobs, scheduling, retries, dashboard) for in-app job processing, or offload to a real message broker (RabbitMQ, Azure Service Bus, Kafka) with dedicated consumer services for decoupled, durable, scalable pipelines. The skill is recognizing the ceiling early — before you've reimplemented a fragile half of a job framework inside a while loop.

Q23Two hosted services must run in a specific order at startup, and one does slow initialization. How do you handle it?🔴 SENIOR

First, know the default: hosted services' StartAsync methods run in registration order, and (in modern hosting) the host awaits each before starting the next by default, so registration order gives you basic sequencing — but relying on that for real dependencies is fragile. Better options: for a slow one-time init that must complete before others, do it in a dedicated startup IHostedService whose StartAsync performs the work (accepting that it blocks startup — appropriate if later services genuinely depend on it), or move init out of the hosted-service lifecycle entirely (e.g. run migrations/seeding as an explicit step, Day 9). If services must coordinate at runtime rather than startup, use an explicit signal — a shared TaskCompletionSource/readiness gate that the dependent service awaits — instead of ordering tricks. And avoid heavy blocking work in ExecuteAsync before the first await, which delays overall startup. The principle: make dependencies explicit (a readiness signal) rather than implicit (registration order), and separate one-time init from the long-running loop.

Q24A worker silently stopped in production three days ago and nobody noticed. What went wrong and how do you prevent a repeat?🔴 SENIOR

Almost certainly an unhandled exception escaped ExecuteAsync and faulted the task, so the worker died while the web app kept serving — no crash, no alert, just a job that quietly stopped ticking. Root cause: no per-iteration try/catch, and no visibility. Prevention is two-pronged. Resilience: wrap each tick in try/catch so transient failures log and the loop continues; only fail-fast (stop the host) on truly unrecoverable state, and if you do let it fail-fast, ensure the orchestrator (Kubernetes, etc.) restarts the process. Observability: emit a heartbeat/last-successful-run timestamp and expose it via a health check, then alert when it goes stale ("no successful run in 3× the interval"); add metrics on tick success/failure counts and log every caught exception with structured context. The meta-lesson: a background worker with no health signal is invisible when it dies — the monitoring is as important as the code, because silent failure is the default failure mode here.
