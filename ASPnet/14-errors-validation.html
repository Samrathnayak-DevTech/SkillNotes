Day 14 · ASP.NET Core Fundamentals

🛡️ Errors & Validation: Failing Well

Code that only handles the happy path is a demo, not a product. Real software must decide, for every operation: what counts as invalid input, what to do when something breaks, and how to communicate failure clearly. Today you'll learn exceptions and try/finally, guard clauses that fail fast, input validation, and the increasingly popular Result pattern — plus when to reach for each.

💥 Exceptions 🧹 try / finally 🚧 Guard Clauses ✅ Validation 📦 Result<T>

01💥 What Are Exceptions?

An exception is C#'s built-in mechanism for signaling that something went wrong in a way the current code can't handle. When you throw one, normal execution stops immediately and the runtime unwinds the call stack — walking back up through every calling method — looking for a catch block willing to handle it. If none is found, the program crashes.

🚨 Analogy

An exception is a fire alarm in a building. When someone pulls it, everyone stops what they're doing immediately — no finishing the meeting, no sending the email. The alarm propagates floor by floor (up the call stack) until it reaches someone trained to respond (a catch block). If it reaches the roof with nobody responding, the building is evacuated (the program crashes). Alarms are for emergencies — you don't pull one because the coffee machine is empty.

The exception hierarchy 🌳

All exceptions derive from System.Exception. You'll meet a handful constantly, and knowing them helps you both throw the right one and catch precisely:

ExceptionMeansTypically thrown when
ArgumentNullExceptionA required argument was null.A method got null where it needs a value.
ArgumentExceptionAn argument was invalid.Wrong format, out of range, empty string.
ArgumentOutOfRangeExceptionAn argument was outside its allowed range.Negative quantity, index too large.
InvalidOperationExceptionThe object is in a wrong state for this call.Confirming an already-cancelled order.
KeyNotFoundExceptionA lookup key doesn't exist.Fetching an entity by an id that isn't there.
NotImplementedExceptionThis code isn't written yet.A stubbed method — never ship it live.
🔑 Exceptions Carry Rich Information

An exception object bundles a Message (human-readable), a StackTrace (exactly where it happened, frame by frame), and an InnerException (the original cause, if this one wraps another). That's why, back on Day 6, we passed the exception object to the logger rather than just its message — all of that context would otherwise be lost.

⚠️ Exceptions Are for the Exceptional

Throwing and catching has a real cost — building the stack trace, unwinding frames — and, more importantly, using exceptions for expected outcomes (like "user typed an invalid email") makes normal flow look like an emergency and clutters logs. .NET 9 made exceptions notably faster, but the design principle stands: exceptions are for genuinely unexpected or unrecoverable situations, not routine branching. Sections 05–08 are all about the alternatives for expected failures.

The through-line: expected failures (bad input, "not found", business-rule violations) are best handled with guard clauses, validation, and Result returns; unexpected failures (the database is down, disk full) are what exceptions and try/finally are for. Learning to tell them apart is the real skill.

03🧹 try / catch / finally

The try block wraps code that might fail. A catch block handles a specific exception if it occurs. A finally block runs no matter what — whether the try succeeded, threw, or you returned early — making it the place for cleanup that must always happen.

C# · The full shape
try
{
    // code that might throw
    await ProcessPaymentAsync(order);
}
catch (PaymentDeclinedException ex)   // most specific first
{
    logger.LogWarning(ex, "Payment declined for order {OrderId}", order.Id);
    throw;                            // rethrow, preserving the stack trace
}
catch (Exception ex)                 // broadest last, as a safety net
{
    logger.LogError(ex, "Unexpected error paying for order {OrderId}", order.Id);
    throw;
}
finally
{
    // ALWAYS runs — success, exception, or early return
    tempFile?.Delete();
}

The order matters: catch blocks are checked top to bottom, so a broad catch (Exception) placed first would swallow everything and the specific ones below would never run. Always go most specific → most general.

using: finally you don't have to write 🎁

For disposable resources (files, connections, EF contexts), the using statement is a hidden try/finally — it guarantees Dispose() is called even if an exception is thrown. Prefer it over manual cleanup:

C# · using = automatic finally
// This...
using var stream = File.OpenRead(path);
Process(stream);
// ...is roughly: try { Process(stream); } finally { stream.Dispose(); }
// The file closes whether Process succeeds or throws.
⚠️ Never Swallow Exceptions

An empty catch { } — or one that catches and does nothing — is one of the worst habits in programming. It hides bugs, produces silent failures, and turns a five-minute fix into a multi-day mystery. If you catch, you must do something: handle it meaningfully, log it, or rethrow. "Catch, log, and swallow" is only acceptable at the very top of the app (a global handler) where crashing would be worse.

✅ throw; vs throw ex;

To rethrow after logging, write throw; — bare, no variable. Writing throw ex; resets the stack trace to the current line, erasing where the error actually originated and crippling your debugging. This tiny distinction trips up countless developers; throw; preserves the original trace, which is what you want.

04🎯 Throwing Well

When you do throw, throw the most specific exception type that fits, with a clear message that says what was wrong and ideally how to fix it. A good exception is a message to a future developer (often you) at 3 a.m.

C# · Specific & descriptive
// ❌ Vague — what argument? what's wrong with it?
throw new Exception("error");

// ✅ Specific type, actionable message, names the offending value
throw new ArgumentOutOfRangeException(
    nameof(quantity), quantity, "Quantity must be greater than zero.");

Custom exceptions for your domain 🏷️

For errors specific to your business, define your own exception types. They make catch blocks precise and carry domain data:

C# · A domain exception
public class InsufficientStockException(int productId, int requested, int available)
    : Exception($"Product {productId}: requested {requested} but only {available} in stock.")
{
    public int ProductId { get; } = productId;
    public int Requested { get; } = requested;
    public int Available { get; } = available;
}

// Callers can catch THIS specifically and react (e.g. suggest alternatives):
catch (InsufficientStockException ex)
{
    return Results.Conflict($"Only {ex.Available} left.");
}
🔑 Wrapping with InnerException

When you catch a low-level exception and throw a more meaningful one, pass the original as the inner exception: throw new OrderProcessingException("Could not place order", ex);. This preserves the full causal chain — your high-level message on top, the root cause (say, a SqlException) still attached underneath for debugging. Never discard the original.

⚠️ Don't Catch What You Can't Handle

Only catch an exception if you can do something useful about it — recover, add context, translate it, or make a decision. If you can't, let it propagate to code that can (often a global error handler). Catching broadly "just in case" and rethrowing adds noise; catching and swallowing hides bugs. The default should be to let exceptions flow up to a place designed to handle them.

05🚧 Guard Clauses: Fail Fast

A guard clause is a validation check at the very top of a method that rejects bad input immediately, before any real work begins. It "guards" the method's body from invalid state. The payoff is twofold: invalid data never reaches your logic, and your code stays flat and readable instead of drowning in nested if statements.

🛂 Analogy

Guard clauses are the bouncer at the door. They check IDs before anyone gets inside — no ticket, no entry, turned away immediately at the threshold. The party inside (your business logic) never has to worry about gatecrashers, because they were stopped at the door. Compare that to letting everyone in and then checking IDs table by table, mid-party — chaos.

Guards flatten the "pyramid of doom" 🔺

C# · Nested vs guarded
// ❌ Pyramid of doom — logic buried under nested ifs
public void Register(User user)
{
    if (user != null)
    {
        if (!string.IsNullOrEmpty(user.Email))
        {
            if (user.Age >= 18)
            {
                // the actual work, indented three levels deep...
            }
        }
    }
}

// ✅ Guard clauses — reject early, then the happy path runs unindented
public void Register(User user)
{
    ArgumentNullException.ThrowIfNull(user);
    if (string.IsNullOrWhiteSpace(user.Email))
        throw new ArgumentException("Email is required.", nameof(user));
    if (user.Age < 18)
        throw new ArgumentOutOfRangeException(nameof(user), "Must be 18 or older.");

    // the actual work — flat, clear, guaranteed-valid input
}

Built-in guard helpers (C# 10+) 🧰

Modern .NET ships concise static guards so you write one line instead of an if+throw:

HelperThrows if…
ArgumentNullException.ThrowIfNull(x)x is null.
ArgumentException.ThrowIfNullOrEmpty(s)string is null or empty.
ArgumentException.ThrowIfNullOrWhiteSpace(s)string is null, empty, or whitespace.
ArgumentOutOfRangeException.ThrowIfNegative(n)number is negative.
ArgumentOutOfRangeException.ThrowIfZero(n)number is zero.
ArgumentOutOfRangeException.ThrowIfGreaterThan(a, b)a > b.
✅ For Repetitive Guards, Centralize

If the same checks appear across many methods, a small static Guard helper (or the popular Ardalis.GuardClauses library — Guard.Against.NullOrEmpty(...), Guard.Against.NegativeOrZero(...)) keeps things DRY and reads like plain English. The built-in helpers cover most needs; reach for a library when your domain has many custom rules.

🔑 Guards Encode Preconditions

A guard clause documents a method's contract: "I require a non-null user, a valid email, an adult." Anyone reading the top of the method instantly sees what must be true for it to work — and the method body can then assume those things without re-checking. Preconditions at the door; clean logic inside.

06✅ Input Validation

Guard clauses protect a single method. Input validation is the broader discipline of checking that data entering your system is well-formed and sensible before acting on it. The cardinal rule: never trust input — not from users, not from other services, not from files.

🛃 Analogy

Validation is customs at the border. Every item entering the country is inspected against the rules — right paperwork, nothing prohibited, quantities within limits — before it's allowed in. Once it's cleared customs, everyone downstream can trust it's legitimate. Skip customs and contraband circulates freely through the whole economy (your system).

Two styles of validation

🏷️Declarative (attributes)

Annotate a model with [Required], [EmailAddress], [Range(1, 100)], [StringLength(120)] (Day 7!). ASP.NET Core checks them automatically for incoming requests. Great for simple, per-field rules.

🧠Imperative (code)

Write validation logic explicitly — for cross-field rules ("end date after start date"), database-dependent checks ("email not already taken"), or complex business rules attributes can't express.

C# · Declarative on a request model
public class CreateBookingRequest
{
    [Required, EmailAddress]
    public string GuestEmail { get; set; } = "";

    [Range(1, 10, ErrorMessage = "Guests must be between 1 and 10.")]
    public int Guests { get; set; }

    [Required]
    public DateOnly CheckIn { get; set; }
    [Required]
    public DateOnly CheckOut { get; set; }
}

Collect all the errors, don't stop at the first ✋

A guard clause throws on the first problem — fine for programmer errors. But for user input, that's a frustrating experience ("fix email" → resubmit → "fix date" → resubmit…). Good input validation gathers every problem and reports them together:

C# · Accumulate validation errors
public List<string> Validate(CreateBookingRequest req)
{
    var errors = new List<string>();

    if (string.IsNullOrWhiteSpace(req.GuestEmail))
        errors.Add("Guest email is required.");
    if (req.Guests < 1 || req.Guests > 10)
        errors.Add("Guests must be between 1 and 10.");
    if (req.CheckIn >= req.CheckOut)                     // cross-field rule
        errors.Add("Check-out must be after check-in.");

    return errors;   // empty list = valid; caller reports all at once
}
🔑 Validate at the Boundary

Validate input where it enters your system — the API endpoint, the message consumer — so invalid data is rejected before it touches your business logic or database. Inner layers can then trust their inputs (a guard clause there is a safety net, not the primary check). Validate early, validate at the edge, and never rely on the client to have validated for you.

✅ Beyond the Basics: FluentValidation

For anything complex, the FluentValidation library is the community favorite — it expresses rules as readable, testable code (RuleFor(x => x.Email).NotEmpty().EmailAddress()) separate from your models, handles cross-field and conditional rules elegantly, and integrates with ASP.NET Core. Attributes for simple cases; FluentValidation when rules grow.

07📦 Result-Style Returns

Here's a shift in thinking. For failures that are expected — "email already taken," "order not found," "insufficient stock" — throwing an exception is like pulling the fire alarm because the coffee ran out. These aren't emergencies; they're ordinary outcomes. The Result pattern models them as return values instead: a method returns an object that says "here's the value" or "here's why it failed."

C# · A minimal Result<T>
public record Result<T>(bool IsSuccess, T? Value, string? Error)
{
    public static Result<T> Success(T value) => new(true, value, null);
    public static Result<T> Fail(string error) => new(false, default, error);
}
C# · Using it — failure is a normal return
public async Task<Result<User>> CreateUserAsync(string email, CancellationToken ct)
{
    if (string.IsNullOrWhiteSpace(email))
        return Result<User>.Fail("Email is required.");

    if (await _repo.EmailExistsAsync(email, ct))
        return Result<User>.Fail("That email is already registered.");

    var user = new User(email);
    await _repo.AddAsync(user, ct);
    return Result<User>.Success(user);
}

// The caller MUST look at the result — failure can't be ignored:
var result = await _users.CreateUserAsync(email, ct);
return result.IsSuccess
    ? Results.Ok(result.Value)
    : Results.BadRequest(result.Error);
📬 Analogy

An exception is a fire alarm; a Result is a delivery receipt that says "delivered" or "address not found." The receipt is part of the normal transaction — you expect to read it, and both outcomes are routine. You don't evacuate the building because a parcel couldn't be delivered; you just check the receipt and act accordingly. Result returns make the failure a normal, visible part of the method's signature.

Why teams reach for it 💡

  • 🔎 Explicit — the return type Result<User> tells callers, right in the signature, that this can fail. An exception is invisible until it's thrown.
  • 🧭 Forces handling — you can't use the value without checking IsSuccess, so failures don't get silently ignored.
  • ⚡ No throw/unwind cost — expected failures don't pay the exception performance tax or pollute logs with stack traces.
  • 🧪 Easy to test — assert result.IsSuccess and result.Error, no Assert.Throws gymnastics.
✅ Libraries Do This Well

Rolling your own Result<T> is fine to learn, but mature libraries like ErrorOr, FluentResults, or a Result type from a framework give you richer errors (codes, categories like Validation/Conflict/NotFound), multiple errors, and helpful methods for chaining. ErrorOr<User> returning Error.Conflict("User.Duplicate", "...") maps cleanly to HTTP status codes at the boundary.

08⚖️ Exceptions vs Results: Choosing

This is the judgment call that separates tidy codebases from messy ones. The guiding question: is this failure expected and part of normal flow, or genuinely exceptional?

🔑 The Pragmatic Middle Ground

You don't have to pick one dogmatically. A common, healthy pattern: guard clauses throw for programmer errors (null args, impossible state — these are bugs, fail fast), Result returns for expected business/validation failures (the caller should handle them gracefully), and exceptions propagate for genuine infrastructure failures (handled centrally by a global error handler). Each tool for what it's best at.

⚠️ Don't Use Exceptions for Control Flow

The classic anti-pattern: try { return Parse(x); } catch { return default; } as normal branching, or throwing to break out of loops. It's slow, obscures intent, and makes debugging harder because real errors hide among "expected" ones. If a failure is expected often enough to be part of normal logic, it belongs in a Result or a Try... method (int.TryParse), not an exception.

09🧩 Putting It Together

Here's a service method that layers everything today into a coherent whole — guard clauses for programmer errors, validation for user input, a Result for expected business failures, and exceptions left to propagate for the truly unexpected:

C# · A well-guarded, well-validated service method
public async Task<Result<Order>> PlaceOrderAsync(
    PlaceOrderRequest request, CancellationToken ct = default)
{
    // 1️⃣ GUARD CLAUSES — programmer errors → throw, fail fast & loud
    ArgumentNullException.ThrowIfNull(request);

    // 2️⃣ INPUT VALIDATION — user errors → collect & return as a Result
    var errors = new List<string>();
    if (request.Items.Count == 0)
        errors.Add("Order must contain at least one item.");
    if (string.IsNullOrWhiteSpace(request.CustomerEmail))
        errors.Add("Customer email is required.");
    if (errors.Count > 0)
        return Result<Order>.Fail(string.Join(" ", errors));

    // 3️⃣ BUSINESS RULE — expected failure → Result, not exception
    foreach (var item in request.Items)
    {
        var product = await _repo.GetProductAsync(item.ProductId, ct);
        if (product is null)
            return Result<Order>.Fail($"Product {item.ProductId} not found.");
        if (product.Stock < item.Quantity)
            return Result<Order>.Fail(
                $"Only {product.Stock} of '{product.Name}' available.");
    }

    // 4️⃣ THE WORK — infra failures (DB down) throw & propagate to a
    //     global handler; we don't try/catch what we can't fix here.
    var order = Order.From(request);
    await _repo.SaveOrderAsync(order, ct);

    _logger.LogInformation("Placed order {OrderId}", order.Id);  // Day 6
    return Result<Order>.Success(order);
}

Notice the layered discipline, mapping straight to the endpoint at the boundary:

🚧
GuardNull request → throw immediately. That's a bug in the calling code, not a user's fault.
✅
ValidateCollect all input problems, return them together as a failed Result → maps to 400 Bad Request.
📦
Business rulesNot found / insufficient stock → failed Result → maps to 404 / 409. Expected, handled gracefully.
💥
Let infra throwDB failure bubbles up as an exception to a global handler → 500. We don't mask what we can't fix.
🔑 One Rule to Remember

Match the mechanism to the meaning. Bugs (null args, impossible state) → throw, loudly. User/business failures (invalid, not found, rule violated) → Result, gracefully. Infrastructure failures → let them propagate to one place that logs and returns a clean 500. When the tool matches the meaning, your error handling almost writes itself.

10🛠️ Hands-On: Add Validation & Error Handling to a Service

Today's Do: take one service method you built earlier (yesterday's tested OrderService is ideal — you already have tests to keep green) and give it proper validation and error handling. The path:

  1. 🚧 Add guard clauses at the top for programmer errors: ArgumentNullException.ThrowIfNull(request) and any non-nullable arguments that must never be null.
  2. ✅ Validate user input: check the real rules (required fields, ranges, cross-field rules like check-out after check-in). Collect all errors into a list rather than throwing on the first.
  3. 📦 Introduce a Result<T> (write the small record from section 07, or add ErrorOr) and return failures as values for expected problems — invalid input, not found, business-rule violations.
  4. 🎯 Throw specific exceptions only where appropriate — a custom domain exception, or a built-in like InvalidOperationException for impossible state.
  5. 🧹 Use try/finally or using where you have cleanup that must always run (a resource to dispose, a temp file to delete).
  6. 💥 Let infrastructure failures propagate — don't wrap the whole method in a catch-all that swallows the database being down. If you catch, add context and rethrow with throw;.
  7. 🔌 Map at the boundary: in the endpoint, translate a failed Result to the right HTTP status (400/404/409) and a success to 200/201.
  8. 🧪 Update your tests (Day 13!): add cases asserting a failed Result for each invalid input, and verify no side effects fired on failure (Times.Never). Confirm the happy path still passes.
✅ Self-Check

From your updated method: (1) Which failures did you model as exceptions and which as Results — and can you justify each choice with "expected vs exceptional"? (2) If a user submits three bad fields, do they see all three errors or just the first? (3) Is there any empty catch { } hiding somewhere? (4) If you rethrow, did you write throw; and not throw ex;? (5) Does a failed validation leave the database untouched?

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is an exception, and what happens when one is thrown?🌱 JUNIOR

An exception is an object that signals an error the current code can't handle. When you throw one, normal execution stops immediately and the runtime "unwinds the call stack" — walking back up through each calling method — looking for a catch block that handles that type of exception. If it finds one, execution continues there; if it reaches the top without a handler, the program crashes. The exception object carries a message, a stack trace showing where it happened, and optionally an inner exception (the underlying cause).

Q2What does the finally block do, and when does it run?🌱 JUNIOR

finally contains cleanup code that runs no matter what happens in the try — whether it completed normally, threw an exception, or returned early. It's where you release resources that must always be freed: close a file, dispose a connection, delete a temp file. Because it always runs, it guarantees cleanup even on the error path. In practice, the using statement is a built-in try/finally that disposes a resource automatically, so you often don't write finally by hand.

Q3Why should catch blocks go from specific to general?🌱 JUNIOR

Because C# checks catch blocks top to bottom and uses the first one that matches. If you put the broad catch (Exception) first, it matches everything, so any more specific catch blocks below it can never run — the compiler will even warn you. Ordering most-specific first (e.g. catch (PaymentDeclinedException) before catch (Exception)) lets you handle known errors precisely and keep the general one as a last-resort safety net.

Q4What is a guard clause?🌱 JUNIOR

A guard clause is a check at the very top of a method that rejects invalid input immediately — before any real work runs. For example, ArgumentNullException.ThrowIfNull(user) or "if quantity is negative, throw." It "guards" the method body from bad data. The benefits: invalid input is stopped early ("fail fast"), and the code stays flat and readable instead of wrapping the logic in deeply nested if statements.

Q5What's the difference between throw; and throw ex;?🌱 JUNIOR

throw; (bare, in a catch block) rethrows the current exception while preserving its original stack trace — so you can still see where the error actually started. throw ex; rethrows but resets the stack trace to that line, erasing the original location and making the bug much harder to trace. Almost always you want throw;. It's a small syntax difference with a big debugging impact.

Q6Why is an empty catch block dangerous?🌱 JUNIOR

An empty catch { } "swallows" the exception — the error happens but nothing records it or reacts, so the program silently continues in a possibly broken state. Bugs become invisible: data quietly goes wrong, and there's no log, no crash, no clue. A five-minute fix turns into days of mystery debugging. If you catch an exception, you must do something meaningful with it — handle it, log it, or rethrow. Swallowing is only acceptable in rare, deliberate cases (like a top-level handler), never as a routine habit.

Q7What is input validation and why does it matter?🌱 JUNIOR

Input validation is checking that data entering your system is well-formed and sensible before you act on it — required fields present, values in range, formats correct. It matters because you can never trust input: users make mistakes, and malicious input can break or exploit your system. Validating at the boundary (like the API endpoint) rejects bad data before it reaches your business logic or database, keeping everything downstream safe and simple. The rule of thumb: never trust input, validate early.

Q8Give an example of when to throw an exception versus return an error value.🌱 JUNIOR

Throw an exception for something unexpected — a method received null where it should never be null (a bug), or the database is unreachable. Return an error value (like a Result) for something expected — a user submitted an invalid email, or an order lookup found nothing. The test: "Is this a normal outcome I planned for, or an emergency?" Invalid user input happens all day long, so it's normal → return it; a corrupted state or downed dependency is exceptional → throw.

🟠 Mid

Q9What is the Result pattern and what advantages does it have over exceptions for expected failures?🟠 MID

The Result pattern models failure as a return value instead of a thrown exception: a method returns an object (e.g. Result<User>) that says either "success, here's the value" or "failure, here's why." Advantages for expected failures: it's explicit — the return type tells callers this can fail, whereas an exception is invisible until thrown; it forces handling — you must check IsSuccess before using the value; it avoids the exception cost — no stack unwinding or log noise for routine outcomes; and it's easier to test — assert on the returned object rather than catching. It shines for validation errors, "not found," and business-rule violations that happen as part of normal flow.

Q10How do you decide between the Result pattern and exceptions in a real codebase?🟠 MID

The guiding question is "expected or exceptional?" Expected, recoverable failures that are part of the operation's normal contract — validation failures, not-found, business rules like insufficient stock — are best as Results: the caller should handle them gracefully and they happen routinely. Genuinely exceptional situations — a null argument that indicates a bug, an impossible object state, a failed database connection, out of memory — are best as exceptions: rare, often unrecoverable, and something you want to fail loudly and handle centrally. A common healthy split: guard clauses throw for programmer errors, Results carry expected business failures, and infrastructure exceptions propagate to a global handler. Consistency within the team matters as much as the exact rule.

Q11Why shouldn't you use exceptions for control flow?🟠 MID

Using exceptions for normal branching — like try { return Parse(x); } catch { return 0; } as routine logic, or throwing to exit a loop — has several problems. Performance: throwing and unwinding the stack is far costlier than a simple check (even though .NET 9 improved this). Clarity: it hides intent — a reader expects exceptions to mean "something went wrong," so using them for expected cases is misleading. Debugging: real errors get buried among "expected" ones, and exception-break debugging becomes noisy. Correctness: it's easy to catch too broadly and mask a genuine bug. When a failure is expected often enough to be part of the logic, use a check, a Try... method (int.TryParse), or a Result — reserve exceptions for the genuinely exceptional.

Q12When validating user input, why collect all errors instead of failing on the first?🟠 MID

Because failing on the first error creates a miserable, slow feedback loop for the user: they fix the email, resubmit, learn the date is wrong, resubmit, learn the quantity is invalid, and so on — one round trip per mistake. Collecting all validation errors and returning them together lets the user fix everything at once. This is exactly why input validation differs from guard clauses: a guard clause throws on the first violation (fine for detecting a programmer bug, where one problem is enough), but user-facing input validation should accumulate errors into a collection and report the complete set. Frameworks like ASP.NET Core model binding and FluentValidation do this accumulation for you.

Q13What's the difference between declarative and imperative validation, and when do you use each?🟠 MID

Declarative validation uses attributes on a model — [Required], [Range], [EmailAddress], [StringLength] — and the framework checks them automatically. It's concise and great for simple, per-field rules. Imperative validation is explicit code, needed for anything attributes can't express: cross-field rules ("check-out after check-in"), rules requiring a database or service call ("is this email already taken"), or complex conditional logic. Use declarative for the straightforward field constraints and imperative (often via FluentValidation, which keeps it clean and testable) for the rest. They coexist happily — attributes for the basics, code for the business rules.

Q14How do you preserve the original cause when translating a low-level exception into a domain one?🟠 MID

Pass the caught exception as the inner exception when you throw the new one: catch (SqlException ex) { throw new OrderProcessingException("Failed to place order", ex); }. This keeps the full causal chain — your meaningful high-level message on top, with the original SqlException (and its stack trace and message) attached underneath via InnerException. Logs and debuggers can then show both: what failed at the business level and why at the technical level. Never discard the original by omitting it — you'd lose the actual root cause and be left debugging blind.

Q15Where should validation happen in a layered application, and why?🟠 MID

Validate untrusted input at the boundary where it enters the system — the API endpoint or message consumer — so bad data is rejected before it reaches business logic or the database. That keeps inner layers simpler because they can largely trust their inputs. Guard clauses deeper in the code act as a secondary safety net (defensive checks against programmer errors), not the primary validation. There's nuance: some rules can only be checked deeper (e.g. against database state), and domain models can enforce their own invariants. But the principle is validate-early-at-the-edge for untrusted external input, and don't rely on the client having validated. Defense in depth: boundary validation as the main gate, guards inside as backup.

Q16What are the built-in guard helper methods in modern .NET, and why prefer them?🟠 MID

Modern .NET (C# 10+/.NET 6+) added concise static guards: ArgumentNullException.ThrowIfNull(x), ArgumentException.ThrowIfNullOrEmpty(s) / ThrowIfNullOrWhiteSpace(s), and ArgumentOutOfRangeException.ThrowIfNegative/ThrowIfZero/ThrowIfGreaterThan(...). Prefer them because they replace a two-line if (x is null) throw new ArgumentNullException(nameof(x)) with one readable line, they automatically capture the argument name via caller-argument-expression (so the exception names the right parameter), and they throw the correct, specific exception type with a good message. For repeated domain-specific checks, a small custom Guard class or the Ardalis.GuardClauses library (Guard.Against.NullOrEmpty) keeps things DRY and expressive.

🔴 Senior

Q17Design an error-handling strategy for a layered ASP.NET Core application.🔴 SENIOR

Layered by concern. At the boundary (controllers/endpoints): validate incoming DTOs (model validation + FluentValidation), returning 400 with all errors for invalid input. In the domain/service layer: use guard clauses to fail fast on programmer errors (invariant violations, nulls that shouldn't happen), and return a Result/ErrorOr for expected business failures (not found, conflict, rule violations) that carry an error category. At the edge again: map those result categories to HTTP status codes (Validation→400, NotFound→404, Conflict→409). Globally: a centralized exception handler (middleware or IExceptionHandler) catches anything unhandled, logs it with full context (Day 6 structured logging + correlation id), and returns a clean ProblemDetails 500 without leaking internals. The principles: expected failures are explicit values handled locally; unexpected ones propagate to one place; users get actionable messages; logs get full detail; and internal errors never leak to clients. Consistency (one Result type, one error shape, one handler) is what makes it maintainable.

Q18What are the real costs of exceptions, and how has .NET 9 changed the calculus?🔴 SENIOR

Costs: throwing captures a stack trace and unwinds frames, which is expensive relative to a branch — historically enough that exceptions-as-control-flow in a hot loop could dominate runtime. There are also indirect costs: log noise (every "expected" failure produces a stack trace), obscured intent, and harder debugging. .NET 9 significantly improved exception throughput — throwing is now much faster and lighter — which weakens the pure-performance argument against them. But the design argument is unchanged: even cheap exceptions are still the wrong semantic tool for expected outcomes, because the Result pattern's explicitness, forced handling, and readability aren't about speed. So post-.NET 9 I'd say: don't reach for Results solely for performance anymore, but do reach for them for clarity and maintainability of expected-failure flows; reserve exceptions for the genuinely exceptional regardless of how fast they've become.

Q19How do you handle validation and errors consistently across an API so clients get a uniform experience?🔴 SENIOR

Standardize on a single error contract — ProblemDetails (RFC 9457) is the built-in, well-supported choice: a consistent JSON shape with type, title, status, detail, and a place for validation errors. Wire it up so every failure path produces it: ASP.NET Core already returns ValidationProblemDetails for model-validation failures; map your Result/ErrorOr error categories to the matching status + ProblemDetails at the boundary; and register a global exception handler that converts unhandled exceptions into a ProblemDetails 500 (with a trace/correlation id the client can quote to support, but no internal details or stack traces leaked). The result: whether a request fails validation, hits a business rule, or trips an unexpected error, the client always parses the same structure and knows where to find the message and the correlation id. Document it, and consider a shared error-code enum so clients can branch on stable codes rather than message strings.

Q20A method needs to do several things that must all succeed or all roll back, with cleanup. How do you structure the error handling?🔴 SENIOR

Combine transactional atomicity with guaranteed cleanup. For the all-or-nothing part, use the mechanism appropriate to the resource — a database transaction (EF's BeginTransactionAsync, Day 9) so either everything commits or everything rolls back; wrap the work in try, commit at the end, and roll back in the catch before rethrowing. For cleanup that must always run regardless of outcome (releasing a lock, deleting a temp file, disposing a non-DI resource), use finally or, better, using/await using so disposal is automatic even on the exception path. Order matters: acquire resources, do the transactional work, commit; finally handles disposal; catch handles rollback and adds context. Also make the operation idempotent where possible so a retry after a partial failure is safe. And distinguish expected failures (return a Result, no rollback needed because nothing was committed yet) from exceptions (roll back, then propagate). The shape is: guard → validate → begin transaction → try{ work; commit } catch{ rollback; throw } finally{ dispose }.

Q21What's the argument against the Result pattern, and when would you NOT use it?🔴 SENIOR

The honest downsides: it adds boilerplate — every method signature and call site has to thread the Result type and check it, versus letting an exception bubble; it can propagate verbosely up through many layers (mitigated by railway-oriented helpers like Bind/Map, which add their own learning curve); it doesn't compose as naturally with language features that assume exceptions; and if applied dogmatically to everything it obscures the genuinely exceptional cases among routine ones. When not to use it: for truly exceptional, unrecoverable failures (use exceptions — you don't want to Result-wrap "out of memory"); in code paths where the failure genuinely can't be handled locally and should just crash or hit the global handler; and where the team isn't bought in (inconsistent use of Result and exceptions is worse than consistent use of either). It's a tool for expected failures in your core logic, not a religion — mixing it sensibly with exceptions is the mature position.

Q22How do you avoid leaking sensitive information through error messages?🔴 SENIOR

Separate internal error detail from external error responses. Internally, log everything — full exception, stack trace, parameters (minus secrets), correlation id — so you can diagnose. Externally, return a generic, safe message for unexpected errors ("An error occurred; reference id X") and never echo stack traces, SQL, connection strings, file paths, or internal type names to clients — those hand attackers a map of your system. For expected failures, return specific but non-sensitive messages ("Email already registered" is fine; "SQL constraint UQ_Users_Email violated on server DB01" is not). Watch subtle leaks: don't let "email not found" vs "wrong password" reveal which accounts exist (use a uniform "invalid credentials"); be careful that validation messages don't disclose internal rules that aid enumeration. In ASP.NET Core, ensure developer exception pages are dev-only and production returns ProblemDetails without the exception detail. The rule: detailed inward, minimal and generic outward, with a correlation id bridging the two.

Q23Defensive programming vs fail-fast — where's the balance?🔴 SENIOR

They sound opposed but target different things. Fail-fast says: when you detect an invalid state or a broken precondition (a bug), stop immediately and loudly — don't limp along corrupting data. Guard clauses that throw embody this. Defensive programming says: anticipate that inputs and collaborators may misbehave and handle it gracefully — validate external input, check for null returns, degrade gracefully when a dependency is down. The balance comes from who the failure is from. For your own code's bugs (a precondition that should always hold) → fail fast; masking it hides the bug and makes it worse. For external/untrusted boundaries (user input, network, third-party data) → be defensive; those will misbehave and that's expected, so validate and handle. The anti-patterns are at both extremes: over-defensive code that swallows its own bugs and null-checks everything internally (noise that hides real problems), and under-defensive code that trusts external input (security holes, crashes). Rule of thumb: trust your own invariants and fail fast if they break; distrust the outside world and validate it.

Q24How do you make error handling testable, and what would you assert?🔴 SENIOR

Design for it (Day 13 pays off here). Prefer Results for expected failures precisely because they're trivial to assert — a test checks result.IsSuccess is false and the error/code matches, no Assert.Throws needed and no risk of catching the wrong exception. For paths that legitimately throw, use Assert.ThrowsAsync<TException> and assert on the exception's type and message/properties (a custom domain exception makes this precise). Crucially, also assert the side-effect discipline: on a validation or business-rule failure, verify with mocks (Times.Never) that nothing was saved, no email sent, no downstream call made — proving the guard actually protected the system, not just returned an error. Test each failure branch explicitly (each invalid field, each business rule, the not-found case), not only the happy path, since the error paths are where bugs hide. Inject failures via mocks (ThrowsAsync) to test that infrastructure errors propagate or are wrapped correctly. And test the boundary mapping — that a failed Result becomes the right HTTP status. The meta-point: code that's hard to test for errors usually has tangled error handling; making it testable (explicit Results, injected dependencies, specific exceptions) improves the design itself.
