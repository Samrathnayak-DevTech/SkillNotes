✍️ Writes & Migrations: Changing Data, Changing Schema

Day 8 was all about reading. Today you change things — rows and tables. You'll master the add/update/delete patterns, look inside SaveChangesAsync, wire up relationships properly, evolve your schema with migrations, and seed sample data the modern way.

➕ Add / Update / Delete 💾 SaveChangesAsync 🔗 Relationships 🧬 Migrations 🌱 Seeding

01🧺 The Write Side of EF: Unit of Work

Everything about writing data in EF Core follows one rhythm, built on the change tracker you met on Days 7–8:

  1. Stage your changes — add new entities, modify loaded ones, mark others for removal. Nothing touches the database yet.
  2. Commit them all at once with SaveChangesAsync() — EF computes the SQL and runs it inside a single transaction.
🛒 Analogy

Working with a DbContext is shopping with a cart. You walk the aisles putting items in (Add), swapping one brand for another (update), and putting things back on the shelf (Remove). The store's inventory doesn't change while you browse — only at the checkout (SaveChangesAsync) does everything happen, together, as one receipt. Abandon the cart (let the context dispose without saving) and it's as if nothing happened.

This is the unit of work pattern: a business operation ("place this order") maps to one context scope and one save, so related changes succeed or fail together. Recall the entity states from Day 7 — writes are just you moving entities between these states, and SaveChanges translating states into SQL:

Added
Add() → INSERT
Unchanged
loaded, untouched
Modified
property set → UPDATE
Deleted
Remove() → DELETE
Detached
unknown to tracker
Your write API is really a state machine · SaveChangesAsync turns states into SQL
🔑 Key Idea

You never write INSERT/UPDATE/DELETE. You declare the desired end state of your objects, and EF derives the SQL. The skill of today is knowing which calls put entities into which states — and what SaveChangesAsync does with them.

03➕ Adding Data

Adding is the friendliest write: construct objects, hand them to the context, save.

C# · Add, AddRange, and graphs
// Single entity:
var product = new Product { Name = "Mechanical Keyboard", Price = 4999.00m };
db.Products.Add(product);
await db.SaveChangesAsync();
logger.LogInformation("Created product {ProductId}", product.Id);  // ID is populated!

// Several at once:
db.Products.AddRange(
    new Product { Name = "USB-C Hub",  Price = 1899.00m },
    new Product { Name = "Desk Mat",   Price = 799.00m  });
await db.SaveChangesAsync();

// A whole GRAPH — parent + children in one save.
// EF walks the navigations: everything reachable from an Added
// entity that isn't tracked yet is marked Added too.
var order = new Order
{
    CustomerId = customerId,
    PlacedAt   = DateTime.UtcNow,
    Items =
    {
        new OrderItem { ProductId = 1, Quantity = 2, UnitPrice = 4999.00m },
        new OrderItem { ProductId = 3, Quantity = 1, UnitPrice = 799.00m  }
    }
};
db.Orders.Add(order);
await db.SaveChangesAsync();   // 3 INSERTs, correct order, one transaction —
                                // items get their new OrderId FK automatically

Three details worth noticing:

  • 🎁 Generated keys come back for free. After the save, order.Id and each item's Id hold the database-generated values — no re-query needed.
  • 🕸️ Graph adds are ordered correctly. EF inserts the parent first, takes its new key, and uses it as the FK for the children — you never juggle IDs manually.
  • 🏷️ Add marks reachable entities too. That's usually helpful, but beware attaching an existing entity into a new graph — if EF thinks it's new (no key set), you'll get a duplicate. Set FK values (like ProductId above) instead of attaching loaded entities you don't mean to insert.
✅ Tip

Add vs AddAsync: adding is an in-memory operation, so plain Add is correct and normal. AddAsync exists only for a special value-generation scenario (e.g., HiLo sequences that may hit the database to reserve keys). Don't sprinkle AddAsync out of async habit — the async that matters is SaveChangesAsync.

04✏️ Updating Data

The connected update — the everyday pattern 🔵

When the same context loads and saves, updating is almost invisible: load tracked, set properties, save. The tracker's snapshots make the UPDATE surgical — only changed columns are written:

C# · Load → modify → save
var product = await db.Products.FindAsync(id);   // tracked (checks tracker first)
if (product is null) return Results.NotFound();

product.Price = 4499.00m;                          // state → Modified
product.Name  = "Mechanical Keyboard (2026)";

await db.SaveChangesAsync();
-- UPDATE [Products] SET [Name] = @p0, [Price] = @p1 WHERE [Id] = @p2  (only 2 columns!)

The disconnected update — the web reality 🌐

In an API, the entity that comes back in a PUT request was loaded by a previous request with a context that's long gone. Nothing is tracking it. You have two options:

C# · Disconnected update, two styles
// ✅ Style 1 (recommended): fetch-then-apply.
// One extra SELECT, but: precise UPDATE, validation against current state,
// and the client can't overwrite columns you didn't expose in the DTO.
public async Task<bool> UpdatePriceAsync(int id, UpdateProductDto dto, CancellationToken ct)
{
    var product = await db.Products.FindAsync([id], ct);
    if (product is null) return false;

    product.Name  = dto.Name;
    product.Price = dto.Price;
    await db.SaveChangesAsync(ct);
    return true;
}

// ⚠️ Style 2: Update() attaches the object and marks EVERY column Modified.
// No extra SELECT — but it overwrites all columns with whatever the
// client sent, and throws if the row doesn't exist.
db.Products.Update(productFromClient);
await db.SaveChangesAsync(ct);

The bulk update — set-based writes ⚡

C# · ExecuteUpdateAsync (EF Core 7+)
// One UPDATE statement in the database — no loading, no tracking:
var discounted = await db.Products
    .Where(p => p.Category == "Accessories")
    .ExecuteUpdateAsync(s => s
        .SetProperty(p => p.Price, p => p.Price * 0.9m), ct);

// Runs IMMEDIATELY — independent of SaveChanges and the change tracker.
⚠️ Concurrency: The Last Write Wins By Default

If two users load the same product and both save, the second save silently overwrites the first. When that matters (inventory, balances), add a concurrency token — a rowversion column via [Timestamp] or IsRowVersion(). EF then adds WHERE ... AND RowVersion = @original to updates; if another write got there first, zero rows match and EF throws DbUpdateConcurrencyException, which you handle (reload/merge/inform the user) instead of losing data invisibly.

05🗑️ Deleting Data

C# · Three deletion patterns
// 1) Classic: load, remove, save
var product = await db.Products.FindAsync([id], ct);
if (product is null) return false;
db.Products.Remove(product);          // state → Deleted
await db.SaveChangesAsync(ct);

// 2) Bulk / by predicate: one DELETE statement, nothing loaded (EF Core 7+)
var removed = await db.Orders
    .Where(o => o.Status == OrderStatus.Cancelled
             && o.PlacedAt < DateTime.UtcNow.AddYears(-1))
    .ExecuteDeleteAsync(ct);

// 3) Soft delete: don't delete at all — flag it, filter it out globally
product.IsDeleted = true;
await db.SaveChangesAsync(ct);
// + in OnModelCreating, hide flagged rows from every query:
// modelBuilder.Entity<Product>().HasQueryFilter(p => !p.IsDeleted);

What happens to the children? 👶

Deleting a parent raises the Day 7 question again: the relationship's delete behavior decides. Cascade removes dependents with the parent (right for true composition like order → items); Restrict blocks the delete while dependents exist (right for valuable history like customer → orders); SetNull orphans them gracefully when the FK is nullable. You chose this in your Fluent configuration — deletes are where that choice becomes real.

📁 Analogy

Hard delete is a paper shredder — fast, final, and occasionally regretted. Soft delete is the archive drawer: the file leaves every desk and search, but audits, undo, and "what did this look like last year?" remain possible. Most business data earns the drawer, not the shredder.

⚠️ ExecuteDelete Bypasses EF's Niceties

ExecuteDeleteAsync runs immediately, outside SaveChanges' transaction and the change tracker — already-loaded entities go stale, and EF-level cascade logic is skipped (the database's own FK rules govern). Superb for purging and archiving jobs; wrong for deletes tangled in per-entity business logic.

06💾 SaveChangesAsync Under the Hood

You've called it a dozen times — now watch what actually happens inside that one line:

🔎
1. DetectChangesThe tracker compares every tracked entity against its snapshot, flipping quietly-edited entities from Unchanged to Modified and fixing up relationship changes.
🧮
2. Compute the command listEach Added/Modified/Deleted entity becomes an INSERT/UPDATE/DELETE — updates listing only the changed columns.
🔀
3. Order by dependenciesParents insert before children need their keys; children delete before parents. EF sorts and batches the commands.
🔒
4. Execute inside one transactionAll commands run atomically. Any failure rolls back everything and surfaces as DbUpdateException (or DbUpdateConcurrencyException) — no half-saved orders, ever.
🎁
5. Propagate values & reset statesDatabase-generated IDs and rowversions flow back onto your objects; saved entities become Unchanged, ready for the next unit of work.

Transactionautomatic, per save Batchingmultiple commands per round trip Return valuerows affected

When one save isn't enough: explicit transactions 🔗

One SaveChangesAsync = one transaction, automatically. But if a business operation needs two saves (e.g., you must use a generated ID in a second step) to succeed or fail together, open the transaction yourself:

C# · Explicit transaction across saves
await using var tx = await db.Database.BeginTransactionAsync(ct);
try
{
    db.Orders.Add(order);
    await db.SaveChangesAsync(ct);            // need order.Id below

    db.AuditEntries.Add(new AuditEntry("OrderPlaced", order.Id));
    await db.SaveChangesAsync(ct);

    await tx.CommitAsync(ct);                  // both saves stand or fall together
}
catch
{
    await tx.RollbackAsync(ct);
    throw;
}
✅ Habit

Call SaveChangesAsync once per business operation, at the end — not after every little change. Fewer saves = fewer transactions, natural atomicity, and code that reads like the business action it implements. (Using a retrying execution strategy from Day 7's EnableRetryOnFailure? Explicit transactions then need to be wrapped in strategy.ExecuteAsync(...) so the whole unit can retry.)

07🔗 Relationships in Practice

The three cardinalities, in our shop's terms:

1️⃣→🔢One-to-many

Customer → Orders. FK (CustomerId) lives on the many side. The bread and butter — you built this on Day 7.

🔢↔🔢Many-to-many

Product ↔ Tag. Needs a join table; EF creates a hidden one automatically when both sides have collection navigations.

1️⃣↔1️⃣One-to-one

Customer → CustomerProfile. FK on the dependent side, usually with a unique index. Good for splitting rarely-used columns.

C# · Many-to-many, the easy way
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public List<Tag> Tags { get; set; } = [];      // collection on BOTH sides
}

public class Tag
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public List<Product> Products { get; set; } = [];
}
// EF conjures a "ProductTag" join table in the migration — you never see a class.
// Linking is just collection work:
product.Tags.Add(saleTag);
await db.SaveChangesAsync();     // INSERT into the join table

Two ways to relate — navigation or FK 🎭

Every relationship gives you two equivalent handles. Use whichever fits what you already have in hand:

C# · Same result, two styles
// Style A — set the FK value. No extra query; perfect when you have the ID:
var order = new Order { CustomerId = 42, PlacedAt = DateTime.UtcNow };

// Style B — use the navigation. Natural when you already loaded the entity:
var customer = await db.Customers.FindAsync([42], ct);
customer!.Orders.Add(new Order { PlacedAt = DateTime.UtcNow });

await db.SaveChangesAsync(ct);   // EF keeps FK ↔ navigation in sync ("fixup")
🔑 Fixup

EF continuously reconciles both views: set order.CustomerId = 42 and (if customer 42 is tracked) order.Customer and customer.Orders update too, and vice versa. That's why you never manually maintain both sides — pick one, EF fixes up the rest among tracked entities.

⚠️ The Duplicate-Parent Trap

Building a new order for an existing customer? Set CustomerId — don't do new Order {{ Customer = new Customer {{ Name = "Priya" }} }}. A navigation pointing at an un-tracked, key-less object tells EF "this parent is new too," and you'll insert a duplicate customer. IDs for existing things, navigations for new graphs.

08🧬 Migrations Deep Dive

Day 7 introduced the loop (change model → add migration → review → update). Today, open the hood. Running dotnet ef migrations add AddProductTags generates three files working together:

⬆️ Up()
apply the change
⬇️ Down()
undo the change
📸 ModelSnapshot
current model, for the next diff
Anatomy of a migration · timestamped file (Up/Down) + designer file + updated snapshot
C# · Inside a generated migration
public partial class AddProductTags : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.CreateTable(
            name: "Tags",
            columns: table => new
            {
                Id   = table.Column<int>(nullable: false)
                    .Annotation("SqlServer:Identity", "1, 1"),
                Name = table.Column<string>(maxLength: 50, nullable: false)
            },
            constraints: table => table.PrimaryKey("PK_Tags", x => x.Id));
        // ... join table, FKs, indexes ...
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropTable(name: "Tags");   // the exact reverse
    }
}

How EF knows what's pending 🧾

The database carries a small bookkeeping table, __EFMigrationsHistory, listing every applied migration by ID. dotnet ef database update compares your project's migrations against that table and runs only the missing ones, in order, recording each. That's why the same command is safe to run repeatedly, on any environment.

The commands, one level deeper 🛠️

CommandWhat it doesNotes
dotnet ef migrations add NameDiffs model vs snapshot → new migration files.Doesn't touch the database. Review the Up()!
dotnet ef database updateApplies all pending migrations.Runs UseSeeding/UseAsyncSeeding afterwards too (EF 9+).
dotnet ef database update PreviousNameReverts: runs Down() of newer migrations back to the target.Data created since may be lost — Down drops things.
dotnet ef migrations removeDeletes the last migration + restores the snapshot.Only if not yet applied; otherwise revert first.
dotnet ef migrations script --idempotentGenerates SQL that applies only what's missing.The production-friendly deployment artifact.
dotnet ef migrations has-pending-model-changesChecks if the model drifted from the last migration.Great as a CI guard so nobody forgets a migration.
⚠️ Renames Read as Drop + Add

Rename a property and the diff may scaffold DropColumn + AddColumn — schema correct, data gone. This is exactly why the review step exists: replace the pair with migrationBuilder.RenameColumn(...) by hand. Migrations are generated code, and generated code is editable code.

✅ Applying Migrations at Startup — Dev Convenience Only

await db.Database.MigrateAsync() in Program.cs is lovely for local dev: F5 and the schema is current. In production with multiple instances it's a race-condition machine. There, apply migrations as an explicit deployment step — the idempotent script or a migration bundle, run once by the pipeline (Day 7, Q18!).

09🌱 Seeding Data

A fresh database is uselessly empty — no statuses to pick, no categories, no sample rows to develop against. Seeding fills in that baseline. EF gives you two mechanisms with very different personalities:

1 — HasData: model-managed data 📌

Declared in OnModelCreating, baked into migrations as INSERT/UPDATE/DELETE data operations. The docs now call this "model managed data" rather than general seeding — a deliberate renaming, because it fits only a narrow shape of data:

C# · HasData for fixed lookup rows
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<OrderStatusLookup>().HasData(
        new { Id = 1, Name = "Pending"   },   // ⚠️ PK must be specified explicitly —
        new { Id = 2, Name = "Shipped"   },   // it's how migrations diff the data
        new { Id = 3, Name = "Delivered" });
}
// Next `migrations add` emits InsertData(...) calls into the migration.

Right for: small, static, deterministic lookup data (statuses, country codes) that should exist in every environment and only ever change via a migration. Wrong for: anything needing generated keys, hashing, DateTime.Now, external calls, or large volumes — those bloat snapshots and fight the diffing model.

2 — UseSeeding / UseAsyncSeeding: runtime seeding (EF 9+) 🚿

Plain C# callbacks configured with your context options. They run whenever the database is initialized — after MigrateAsync(), EnsureCreated, or dotnet ef database update — with full access to the context, so they can check what exists, hash, randomize, and call services:

C# · Program.cs — modern runtime seeding
builder.Services.AddDbContext<ShopContext>(options => options
    .UseSqlServer(connectionString)
    .UseAsyncSeeding(async (context, _, ct) =>
    {
        // IDEMPOTENT: check before insert — this runs on every init
        if (await context.Set<Customer>().AnyAsync(ct)) return;

        var priya = new Customer { Name = "Priya Sharma", Email = "priya@example.com" };
        priya.Orders.Add(new Order { PlacedAt = DateTime.UtcNow, Total = 1499.50m });

        context.Set<Customer>().AddRange(priya,
            new Customer { Name = "Arjun Rao", Email = "arjun@example.com" });
        await context.SaveChangesAsync(ct);
    })
    .UseSeeding((context, _) =>
    {
        // Implement the sync twin too — EF tooling (database update)
        // uses THIS one; skip it and CLI-triggered seeding silently no-ops.
        if (context.Set<Customer>().Any()) return;
        context.Set<Customer>().Add(new Customer { Name = "Priya Sharma", Email = "priya@example.com" });
        context.SaveChanges();
    }));

// In dev, trigger initialization at startup:
using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<ShopContext>();
    await db.Database.MigrateAsync();    // applies migrations, then runs seeding
}

Which seeding, when? 🧭

DataMechanismWhy
Fixed lookups: statuses, roles-as-constants, country codes📌 HasDataStatic, deterministic, belongs in every environment — versioned with the schema.
Dev/test sample data: demo customers, fake orders🚿 UseSeeding, gated by environmentDynamic, environment-specific, must not live in migration files.
Data needing services: hashed admin user, API-fetched values🚿 UseAsyncSeeding (or a custom startup seeder with DI)Needs runtime logic HasData can't express.
Large datasets, imports🛠️ Custom job/scriptBoth mechanisms are for baselines, not bulk loads.
🔑 Golden Rule of Seeding

Runtime seeding executes on every initialization, so it must be idempotent: check, then insert. The safe skeleton is always if (already there) return; — forget it and every restart plants another Priya.

10🛠️ Hands-On: Migrate & Seed

Today's Do: create a migration and apply it, then seed sample data. A path that exercises everything above:

  1. 🔗 Extend the model: add a Tag entity with a many-to-many to Product, and an IsDeleted flag on Product with a global query filter (HasQueryFilter).
  2. 🧬 dotnet ef migrations add AddTagsAndSoftDelete — then open the migration file. Find the hidden join table, the FKs, and the new column. Can you read the Down()?
  3. 🚀 Apply it: dotnet ef database update. Check __EFMigrationsHistory in the database — your migration ID is now recorded.
  4. 📌 Add HasData for 3 fixed Tag rows (with explicit IDs) → that requires another migration: add it, read the InsertData calls, apply it.
  5. 🌱 Add UseAsyncSeeding + UseSeeding for dev sample data: 2 customers, each with an order containing 2 items (a graph add!). Make it idempotent, and gate it to Development.
  6. ▶️ Wire MigrateAsync() at startup (dev only), run the app twice — verify the second run seeds nothing extra.
  7. ✏️ Practice the write patterns: update a product's price (fetch-then-apply), soft-delete a product and confirm queries hide it, and bulk-archive old orders with ExecuteUpdateAsync.
  8. ↩️ Finale: roll back — dotnet ef database update AddTagsAndSoftDelete reverts the HasData migration; inspect what Down() did, then re-apply.
✅ Self-Check

Three questions to answer from your own project: (1) Which file decides what the next migration will contain, and why must it be committed? (2) Why did your seed data appear only once despite running twice? (3) In your order-graph insert, how did the items get their OrderId before you ever knew it?

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1Walk through how you insert a new row with EF Core.🌱 JUNIOR

Create the object, hand it to the context, save: db.Products.Add(new Product {...}); await db.SaveChangesAsync();. Add only marks the entity as Added in the change tracker — nothing touches the database until SaveChangesAsync, which generates the INSERT, runs it in a transaction, and copies the database-generated ID back onto your object, so product.Id is usable immediately after the save.

Q2How does EF Core know which columns to UPDATE when you save?🌱 JUNIOR

When a query loads an entity (tracked), the change tracker stores a snapshot of its original values. At save time, DetectChanges compares current values against that snapshot; properties that differ are marked modified, and the generated UPDATE lists only those columns. That's why the everyday update is just "load, set properties, save" — no explicit marking needed.

Q3What does SaveChangesAsync guarantee about multiple pending changes?🌱 JUNIOR

Atomicity: all pending inserts, updates, and deletes in that save run inside one database transaction. Either every change is persisted or — if any statement fails — everything rolls back and an exception (DbUpdateException) is thrown. You never end up with an order saved but its items missing. It also orders statements to respect foreign keys and returns the number of rows affected.

Q4How do you delete an entity, and what happens to its related children?🌱 JUNIOR

Load it (or have it tracked), call db.Products.Remove(entity), then save — EF issues a DELETE. What happens to children depends on the relationship's configured delete behavior: Cascade deletes them too (fits order → items), Restrict blocks the delete while children exist (fits customer → orders), and SetNull clears their foreign key if it's nullable. That behavior was chosen in your model configuration — deleting is where the choice takes effect.

Q5What is a migration's Up and Down method?🌱 JUNIOR

Up() contains the operations that apply the schema change — create table, add column, create index. Down() contains the exact reverse, so the migration can be undone: dotnet ef database update SomeEarlierMigration runs the Down methods of everything newer, walking the schema back. Together they make schema changes replayable forward and backward, like version control for your database structure.

Q6How does the database know which migrations have already been applied?🌱 JUNIOR

EF maintains a bookkeeping table in the database itself, __EFMigrationsHistory, with one row per applied migration. When you run dotnet ef database update (or MigrateAsync()), EF compares your project's migration list against that table and executes only the missing ones, in timestamp order, recording each as it goes. That's why the command is safe to repeat and works identically on a teammate's fresh database.

Q7What is data seeding and why do apps need it?🌱 JUNIOR

Seeding is automatically populating a database with baseline data when it's created or migrated — lookup values (order statuses, roles), a default admin account, or sample data for development. Without it, a fresh database is an empty shell the app can't function against, and every developer would hand-insert rows differently. Seeding makes "clone the repo, run the app" produce a working system with consistent data everywhere.

Q8In a one-to-many relationship, what are the two ways to connect a child to its parent?🌱 JUNIOR

Either set the foreign key value directly — order.CustomerId = 42, ideal when you already have the ID — or use the navigation: add the order to customer.Orders (or set order.Customer = customer) when you've loaded the parent. Both produce the same row; EF's "fixup" keeps the FK and navigations synchronized among tracked entities, so you never maintain both sides manually.

🟠 Mid

Q9Compare fetch-then-apply with DbSet.Update() for a disconnected update. Which do you prefer?🟠 MID

Fetch-then-apply: load the current entity, copy allowed fields from the incoming DTO, save. Costs one SELECT, but gives a precise UPDATE of changed columns, natural 404 handling, validation against current state, and — importantly — the client can only affect fields your code explicitly copies. Update(): attaches the incoming object and marks every property modified — no SELECT, but it overwrites all columns with client-supplied values (an over-posting risk), can't tell "missing" from "cleared," and throws if the row is gone. I default to fetch-then-apply for APIs; Update() suits trusted, full-object sync scenarios.

Q10What happens step by step inside SaveChangesAsync?🟠 MID

(1) DetectChanges compares tracked entities against snapshots, promoting silently-edited ones to Modified and fixing up relationships. (2) Each Added/Modified/Deleted entity becomes an INSERT/UPDATE (changed columns only)/DELETE. (3) Commands are sorted to honor dependencies — parents before children for inserts, the reverse for deletes — and batched. (4) Everything executes in a single transaction; failures roll back and throw. (5) Database-generated values (identities, rowversions) are read back onto the objects and saved entities reset to Unchanged. Return value: total rows affected.

Q11When do you need an explicit transaction if SaveChanges already provides one?🟠 MID

When one business operation requires multiple SaveChangesAsync calls — say you must persist an order first to obtain its generated ID and then write an audit row referencing it — each save would otherwise be its own transaction, and a failure between them leaves partial state. BeginTransactionAsync lets both saves commit or roll back together. Also needed when coordinating EF work with raw ADO/Dapper commands on the same connection. Caveat: with a retrying execution strategy enabled, wrap the transactional unit in strategy.ExecuteAsync so the whole thing can retry.

Q12Explain optimistic concurrency in EF Core and how you'd implement it.🟠 MID

Optimistic concurrency assumes conflicts are rare and detects them instead of locking: each row carries a version (SQL Server rowversion, mapped with [Timestamp] or IsRowVersion()). Updates become UPDATE … WHERE Id = @id AND RowVersion = @originalVersion. If another writer changed the row since you read it, zero rows match and EF throws DbUpdateConcurrencyException. You then choose a policy: tell the user and show fresh data, retry with reload, or merge field-by-field. Without a token, the default is silent last-write-wins — fine for some data, dangerous for money and stock.

Q13HasData vs UseSeeding/UseAsyncSeeding — when is each appropriate?🟠 MID

HasData ("model-managed data") is baked into migrations: it requires explicit primary keys, must be static and deterministic (no DateTime.Now, hashing, or services), and suits small fixed lookups — statuses, codes — that belong in every environment and change only via migration. UseSeeding/UseAsyncSeeding (EF 9+) are runtime callbacks executed after Migrate/EnsureCreated/database update: plain C# with full context access, so they handle dynamic data, environment-specific samples, hashed users, and existence checks. Rules of thumb: static-and-structural → HasData; everything else → runtime seeding, written idempotently, with both the sync and async callbacks implemented (tooling uses the sync one).

Q14Why must runtime seeding be idempotent, and how do you make it so?🟠 MID

Because the seeding callbacks run on every database initialization — every startup that calls MigrateAsync, every database update, even with no pending migrations. Non-idempotent seeding therefore duplicates its data on each run. The pattern: check before insert — if (await context.Set<Customer>().AnyAsync()) return; for wholesale seeding, or per-row existence checks (by a natural key like email) for finer-grained top-ups. Deterministic keys or unique constraints act as a second line of defense.

Q15You renamed a property and the migration drops and re-adds the column. Why, and what do you do?🟠 MID

The migration generator only sees two model snapshots — before and after. A property that disappeared plus a property that appeared is indistinguishable from "delete one column, add another," so it scaffolds DropColumn + AddColumn, which would destroy the data in that column. The fix is the review step: edit the generated migration and replace the pair with a single migrationBuilder.RenameColumn(name, table, newName) (same for RenameTable). This is the canonical example of why migrations must be read before being applied — they're editable generated code, not gospel.

Q16How does EF Core handle a many-to-many relationship without a join entity class?🟠 MID

If both entities expose collection navigations to each other (Product has List<Tag>, Tag has List<Product>), EF conventions create an implicit join table (e.g., ProductTag with both FKs as a composite key) — no C# class needed. Linking is collection work: product.Tags.Add(tag); SaveChangesAsync(); inserts a join row; removing from the collection deletes it. When the relationship itself carries data (quantity, date added), you promote it to an explicit join entity with its own class and two one-to-manys — or use UsingEntity to configure payload on the join table.

Q17Design a zero-downtime strategy for a destructive schema change (e.g., splitting a column).🔴 SENIOR

Never do it in one migration — use the expand/contract pattern across releases. Expand: migration adds the new columns alongside the old; deploy code that writes to both but still reads the old (or reads new with fallback). Migrate data: a backfill — batched SQL in a migration or a background job — populates the new columns for existing rows. Switch: deploy code reading the new columns exclusively; verify. Contract: a final migration drops the old column once nothing references it. Each step is individually reversible, old and new app versions coexist during rollout, and a rollback at any stage loses nothing. Wrap it with backups, an idempotent script per stage applied by the pipeline, and metrics watching the dual-write window.

Q18SaveChanges throws DbUpdateException in production. What's your triage process?🔴 SENIOR

First read the inner exception — that's the actual SqlException with the story: unique constraint violation (duplicate key — race on a natural key? missing upsert logic?), FK violation (inserting a child whose parent was deleted, or delete blocked by Restrict), null constraint (model/DB drift or an unmapped required column), string truncation (data longer than the column). The exception's Entries collection tells you which entities were involved. Then classify: data race → add the missing uniqueness handling or concurrency token; drift → compare model to schema, check pending migrations; bad input → validation gap upstream. Log the failure with the entity keys (Day 6 structured logging), never swallow it, and reproduce with the same data locally before patching.

Q19When do ExecuteUpdate/ExecuteDelete become dangerous, and what guardrails do you apply?🔴 SENIOR

Dangers: they run immediately, outside the SaveChanges unit of work — mixing them with staged changes breaks the "all together" expectation unless you open an explicit transaction spanning both; they bypass the change tracker, so already-loaded entities silently go stale; they skip EF-level cascades and any C# business rules that normally run on save (domain events, auditing interceptors, soft-delete conventions). Guardrails: reserve them for genuinely set-shaped maintenance (archiving, purging, counters); require a Where — never call them on an unfiltered set; wrap in an explicit transaction when combined with tracked work; re-query rather than trust tracked instances afterwards; and route audited entities through the tracked path so interceptors still fire.

Q20How do you keep migrations sane across a team with parallel feature branches?🔴 SENIOR

The failure mode: two branches each add a migration from the same snapshot; after merging, the snapshot only reflects one and the model diverges. Process fixes: keep migrations small and merged quickly; on merge conflicts in the snapshot, the later branch removes its migration (migrations remove) and re-adds it on top of the merged model — regenerating is cheap, hand-merging snapshots is not. Add a CI guard: dotnet ef migrations has-pending-model-changes fails the build when someone changed the model without a migration. Convention: one migration per PR where possible, descriptive names, and the rule that a migration file, once merged to main, is immutable — corrections are new migrations, never edits to applied ones.

Q21Where do you put the seeding of an admin user whose password must be hashed with ASP.NET Identity?🔴 SENIOR

Not in HasData — hashing isn't deterministic and needs services, both disqualifiers for model-managed data. Two good options: UseAsyncSeeding registered where the service provider is available (the AddDbContext overload receiving IServiceProvider lets you resolve UserManager/RoleManager), with an idempotent "does the admin exist" check; or a dedicated startup seeder — a small service invoked from Program.cs (or an IHostedService) inside a scope, which is the most flexible for multi-step initialization and easiest to test. Either way: credentials come from configuration/secrets, never hard-coded; the operation is idempotent; and in clustered production you ensure single execution (the EF 9 seeding callbacks are protected by the migration lock, which is one of their quiet advantages).

Q22Argue for and against applying migrations automatically at application startup.🔴 SENIOR

For: zero-friction dev experience (F5 and the schema is current), guaranteed schema/code alignment per deployment, simplest possible pipeline for single-instance apps and internal tools. Against: in multi-instance production, several nodes race to migrate simultaneously (EF's migration locking mitigates but startup still blocks); the app's runtime identity now needs DDL permissions — a security smell; a broken migration takes the whole app down at boot rather than failing a controlled deploy step; and there's no human review gate before irreversible DDL runs. My line: MigrateAsync at startup for Development (and maybe single-instance staging); for production, migrations are a first-class deployment step — idempotent script or bundle, executed once by the pipeline with elevated credentials, before the new app version rolls out.

Q23A teammate models "order archived" by physically deleting rows into an archive table via cascade tricks. Propose something better.🔴 SENIOR

Recommend soft delete / status modeling for the live system: an ArchivedAt (nullable timestamp beats a boolean — it's a flag and an audit fact) with a global query filter so normal queries never see archived orders, and an explicit IgnoreQueryFilters() path for admin/reporting screens. Archival to cheaper storage, if genuinely needed for volume, becomes a separate scheduled job using set-based operations (ExecuteUpdate/bulk copy) — deliberate, monitored, reversible — rather than a side effect of cascade rules, which are invisible in code review and merciless in accidents. Benefits: undo is trivial, referential integrity stays intact (order items, payments still join), auditors are happy, and "deleted by cascade surprise" stops being an incident category.

Q24How would you test data-layer code that performs writes — unit, integration, or both?🔴 SENIOR

Both, with clear roles. Unit tests cover business logic around the data layer — keep them free of EF where possible by testing domain behavior, not the ORM. For the data layer itself, integration tests against a real database engine are the truth: SQL Server in a container (Testcontainers) or LocalDB, schema created by running your actual migrations (which tests the migrations too!), each test isolated via a transaction rollback or a fresh database per class. The in-memory provider is tempting but lies — no relational behavior, no constraints, no translation validation; SQLite in-memory is closer but still diverges from SQL Server on types and features. Assert on real effects: row counts, constraint violations, concurrency exceptions, cascade outcomes. Fast feedback tier: unit tests on logic; nightly/PR tier: the containerized integration suite.
