Day 08 · ASP.NET Core Fundamentals

🔍 Querying with EF Core: LINQ In, SQL Out

Yesterday you mapped entities to tables. Today you'll pull data back out — and learn what separates a fast, precise query from an accidental table scan. The four levers: turn tracking off for reads, project only the columns you need, always go async, and load related data deliberately with Include. Then we'll open the hood and read the SQL EF actually generates.

🌳 IQueryable 🕵️ Tracking vs No-Tracking 🎯 Projections ⏳ Async Queries 🧩 Include 🔬 Generated SQL

01🌳 How LINQ Becomes SQL

When you write a LINQ query against a DbSet, something subtle happens: nothing runs. You aren't fetching data — you're describing data. EF Core collects your Where, OrderBy, and Select calls into an expression tree: a data structure representing the query as a recipe, not a result.

Two interfaces make this possible, and confusing them is the #1 source of slow EF code:

🌳IQueryable<T>

A description of a query. Every operator you chain extends the recipe. EF translates the whole thing into one SQL statement — the filtering happens in the database.

📦IEnumerable<T>

An in-memory sequence. Operators run in your app's RAM, one item at a time. Once a query becomes IEnumerable, the database can no longer help.

C# · The trap and the fix
// ❌ TRAP: ToList() runs "SELECT * FROM Orders" — the ENTIRE table —
// then filters the rows in your app's memory.
var big = db.Orders.ToList()
    .Where(o => o.Total > 1000)
    .Take(10);

// ✅ FIX: keep it IQueryable until the end — the filter and TOP(10)
// travel into the SQL, and only 10 rows cross the network.
var small = await db.Orders
    .Where(o => o.Total > 1000)
    .OrderByDescending(o => o.PlacedAt)
    .Take(10)
    .ToListAsync();
🛒 Analogy

An IQueryable is a shopping list you hand to a personal shopper (the database): "10 most recent orders over ₹1000, please." She walks the aisles with her expert knowledge (indexes!) and brings back exactly ten items. Calling ToList() too early is like having the entire supermarket delivered to your kitchen and then searching the boxes yourself.

🔑 Key Idea

Every LINQ operator placed before the terminal call (ToListAsync, FirstOrDefaultAsync, CountAsync…) is a candidate for SQL translation. Everything after runs in memory. Push work toward the database — it's built for exactly this.

⚠️ Not Everything Translates

EF can only translate what SQL can express. Call a custom C# method inside a Where and you'll get a runtime InvalidOperationException: "could not be translated." The fix is to rewrite the condition with translatable operations (string methods, math, EF.Functions.Like, date parts) — or, if truly needed, materialize first and accept in-memory filtering knowingly.

03⏱️ Deferred Execution in Your Hands

Because the query is a recipe until a terminal call, you can build queries in stages — adding filters conditionally — and EF still compiles one clean SQL statement at the end:

C# · Composing a search query
public async Task<List<Order>> SearchAsync(int? customerId, decimal? minTotal, int page)
{
    IQueryable<Order> query = db.Orders;          // recipe starts here

    if (customerId is not null)
        query = query.Where(o => o.CustomerId == customerId);

    if (minTotal is not null)
        query = query.Where(o => o.Total >= minTotal);   // Wheres combine with AND

    return await query
        .OrderByDescending(o => o.PlacedAt)
        .Skip((page - 1) * 20)
        .Take(20)                                  // pagination in SQL: OFFSET/FETCH
        .ToListAsync();                            // 🎬 executes exactly here
}

✅ Terminal Operators You'll Use Daily

ToListAsync() (all matches), FirstOrDefaultAsync() (first or null — pair with Where or a predicate), SingleOrDefaultAsync() (asserts at most one match), CountAsync(), AnyAsync() (cheapest existence check — becomes EXISTS), SumAsync()/MaxAsync(), and FindAsync(id) (checks the change tracker first, then the database).

⚠️ Multiple Enumeration

Each terminal call re-executes the query. Iterating the same IQueryable twice = two database round trips, possibly with different results in between. If you need the data more than once, materialize it once into a list and reuse that.

04🕵️ Tracking vs No-Tracking

Remember Day 7's change tracker: by default, every entity a query returns is tracked. EF creates an internal entry and a snapshot of each property value so a later SaveChangesAsync() can detect what you modified. That's essential for updates — and pure overhead for reads.

C# · The two modes
// TRACKED (default) — because you intend to modify and save:
var order = await db.Orders.FirstOrDefaultAsync(o => o.Id == id);
order!.Status = OrderStatus.Shipped;
await db.SaveChangesAsync();      // tracker knows exactly which column changed

// NO-TRACKING — read-only list for an API response:
var recent = await db.Orders
    .AsNoTracking()
    .Where(o => o.PlacedAt >= cutoff)
    .ToListAsync();                // no snapshots, less memory, faster
📚 Analogy

A tracked query is borrowing library books with a lending record — the librarian notes each book's condition so any change is detected at return. AsNoTracking is the reading room: browse as much as you like, no paperwork, but you can't take anything home to modify. If you only came to read, skip the paperwork.

Choosing, at a glance

ScenarioModeWhy
GET endpoint returning a list / report / dashboard🟢 AsNoTracking()You'll never call SaveChanges on these — tracking is wasted work.
Load → modify → SaveChangesAsync()🔵 Tracked (default)The tracker computes the UPDATE for you.
Projection to a DTO via Select⚪ Neither neededDTOs aren't entities — EF doesn't track them regardless.
Read-only query where the same related entity appears many times🟢 AsNoTrackingWithIdentityResolution()No tracking, but duplicate rows still map to one shared instance.
Whole context is read-only (reporting service)🟢 Set once: ChangeTracker.QueryTrackingBehavior = NoTrackingFlip the default instead of repeating the call.

Typical speedup (large reads)~30–50% Memory savedno snapshots Cost of the callone method

🔑 Identity Resolution — the Subtle Difference

Tracked queries guarantee one object per database row: if two orders reference customer 42, both point to the same Customer instance. Plain AsNoTracking gives up that guarantee — you may get duplicate Customer objects. Usually harmless for display; when it matters, AsNoTrackingWithIdentityResolution() restores the one-instance behavior while staying read-only.

⚠️ The Classic Bug

Loading an entity with AsNoTracking(), changing a property, calling SaveChangesAsync() — and nothing happens. No error, no UPDATE. The tracker never knew the entity existed. If you intend to modify, load it tracked (or attach it explicitly).

05🎯 Projections: Ask Only for What You Need

A query that returns whole entities fetches every mapped column — the description, the notes field, the byte-array nobody reads. A projection uses Select to reshape the result into exactly the fields the caller needs, and EF translates it into a SQL statement selecting only those columns:

C# · Entity vs projection
// ❌ Whole entities: SELECT every column of Orders (and you only wanted 3)
var orders = await db.Orders.Where(o => o.CustomerId == id).ToListAsync();

// ✅ Projection to a DTO: SELECT Id, PlacedAt, Total — nothing else
public record OrderSummaryDto(int Id, DateTime PlacedAt, decimal Total);

var summaries = await db.Orders
    .Where(o => o.CustomerId == id)
    .Select(o => new OrderSummaryDto(o.Id, o.PlacedAt, o.Total))
    .ToListAsync();

Projections can reach across relationships 🪄

Inside a Select, you can walk navigation properties and even aggregate children — EF turns it into JOINs and subqueries, often making Include unnecessary for read scenarios:

C# · Projection across navigations
var report = await db.Customers
    .Where(c => c.Orders.Any())                    // only customers with orders
    .Select(c => new
    {
        c.Id,
        c.Name,
        OrderCount = c.Orders.Count(),               // → COUNT subquery
        TotalSpent = c.Orders.Sum(o => o.Total),     // → SUM subquery
        LastOrder  = c.Orders.Max(o => o.PlacedAt)
    })
    .OrderByDescending(x => x.TotalSpent)
    .Take(10)
    .ToListAsync();
🥡 Analogy

Fetching full entities is ordering the entire thali when you only wanted the dal. A projection is ordering à la carte: exactly the dishes you'll eat. Cheaper, faster to arrive, nothing wasted — and the kitchen (database) plates it for you in one trip.

🔑 Three Wins in One

Projections (1) shrink the SQL to the needed columns, (2) skip change tracking entirely — DTOs aren't entities, so there's nothing to track, and (3) decouple your API response shape from your database schema, which pays off enormously when either evolves. For read endpoints, projection is the default best practice.

⚠️ One Constraint

You can't SaveChanges a DTO. Projections are for the read path. For the update path, load the tracked entity (only the one you'll modify) and change it — different tools for different halves of the job.

06⏳ Async Queries

Every terminal operator has an async twin — ToList → ToListAsync, First → FirstAsync, Count → CountAsync — and in a web application you should always use the async ones. Here's why it matters even though the query takes the same time either way:

A database call is mostly waiting — for the network, for the disk, for the query plan. A synchronous call makes the request thread stand still and hold its place while waiting. An async call releases the thread back to the pool to serve other requests, and resumes when the data arrives. One request finishes in the same time; a thousand concurrent requests is where the difference shows — sync blocks a thousand threads, async needs only a handful.

☎️ Analogy

Sync is a receptionist who stays on hold with a supplier, ignoring the ringing front desk until the supplier answers. Async is the receptionist who says "call me back when you have it," hangs up, and serves the queue — resuming the supplier conversation the moment the callback comes. Same supplier delay; wildly different throughput at the desk.

C# · Async patterns done right
// Accept a CancellationToken — ASP.NET provides one per request, and EF
// passes it to SQL Server so abandoned requests stop wasting database time.
public async Task<OrderSummaryDto?> GetSummaryAsync(int id, CancellationToken ct)
{
    return await db.Orders
        .Where(o => o.Id == id)
        .Select(o => new OrderSummaryDto(o.Id, o.PlacedAt, o.Total))
        .FirstOrDefaultAsync(ct);
}

// ❌ NEVER: two awaits "in flight" on the SAME context —
// DbContext forbids parallel operations (Day 7!).
var t1 = db.Orders.CountAsync();
var t2 = db.Customers.CountAsync();
await Task.WhenAll(t1, t2);          // 💥 InvalidOperationException

// ✅ Sequential awaits on one context are fine:
var orderCount    = await db.Orders.CountAsync(ct);
var customerCount = await db.Customers.CountAsync(ct);
✅ Habits

(1) The async method names come from Microsoft.EntityFrameworkCore — add the using. (2) Await immediately; don't stash EF tasks for later. (3) Thread a CancellationToken from your endpoint down through services into every EF call. (4) Never block with .Result or .Wait() — that combines the costs of both worlds and risks deadlocks.

🔑 Key Idea

Async doesn't make one query faster — it makes your server scale. Database time is wait time; async lets threads serve other users while the database works.

07🧩 Include & ThenInclude: Loading Related Data

Navigation properties (order.Customer, customer.Orders) are not loaded automatically. Query an order and its Items list will simply be empty — EF only fetches what you ask for. Include is how you ask:

C# · Eager loading
// Order + its Customer (reference) + its Items (collection), one query:
var order = await db.Orders
    .Include(o => o.Customer)                 // reference nav → LEFT JOIN
    .Include(o => o.Items)                    // collection nav → JOIN + grouping
        .ThenInclude(i => i.Product)          // one level deeper: Item → Product
    .FirstOrDefaultAsync(o => o.Id == id);

// Filtered include (EF Core 5+): load only SOME children —
// e.g., each customer with just their orders from this year:
var customers = await db.Customers
    .Include(c => c.Orders.Where(o => o.PlacedAt.Year == 2026)
                          .OrderByDescending(o => o.PlacedAt))
    .AsNoTracking()
    .ToListAsync();

The N+1 problem — the bug you'll actually meet 🐛

Forget Include and touch navigations in a loop (with lazy loading enabled, or by querying per item), and you get the infamous pattern: 1 query for the parents + N queries for children, one per parent. 200 customers = 201 database round trips.

1 + N queries

1 query
Include (JOIN)
1 query

1 + k queries

Round trips for "customers with their orders" · N = number of customers · fewer, chunkier queries win
C# · N+1 and its fixes
// ❌ N+1: one query per customer inside the loop
var customers = await db.Customers.ToListAsync();
foreach (var c in customers)
{
    var orders = await db.Orders
        .Where(o => o.CustomerId == c.Id).ToListAsync();   // 💥 ×N
}

// ✅ Fix 1 — Include: fetch the graph up front
var withOrders = await db.Customers
    .Include(c => c.Orders)
    .ToListAsync();

// ✅ Fix 2 (often better for reads) — projection does the JOIN too:
var summaries = await db.Customers
    .Select(c => new { c.Name, Orders = c.Orders.Select(o => o.Total).ToList() })
    .ToListAsync();
🔑 Include vs Projection

Use Include when you need real, tracked entities with their graph — typically the update path. Use a projection when you just need related data for display — it fetches fewer columns and skips tracking. Many teams' rule: projections for queries, Include for commands.

⚠️ Lazy Loading — Handle With Care

EF supports lazy loading (navigations that auto-load when first touched, via proxies), but it turns every forgotten Include into a silent N+1 at runtime. Most teams leave it off and load explicitly — a missing Include that shows up as an obvious empty list is easier to catch than 201 hidden queries.

08🍱 Split Queries & the Cartesian Explosion

One more trap hides inside Include. A single SQL statement returns a flat rectangle of rows — so when one query includes multiple collections, the JOINs multiply. A customer with 10 orders and 5 addresses becomes 10 × 5 = 50 rows, with the customer's columns duplicated on every one. That's the cartesian explosion: network and memory wasted on redundant copies.

C# · AsSplitQuery
// Two COLLECTION includes → cartesian risk in a single query:
var customers = await db.Customers
    .Include(c => c.Orders)
    .Include(c => c.Addresses)
    .AsSplitQuery()                // → 3 simple queries instead of 1 giant JOIN
    .AsNoTracking()
    .ToListAsync();

// Roughly what EF sends:
//   SELECT ... FROM Customers ...
//   SELECT ... FROM Orders    WHERE CustomerId IN (…) ...
//   SELECT ... FROM Addresses WHERE CustomerId IN (…) ...

Single vs split — the honest trade-off

AspectSingle query (default)Split query (AsSplitQuery)
Round trips11 per included collection (+1)
Data duplicationMultiplies with each collection includeNone — each table fetched once
ConsistencyOne statement = one consistent snapshotData can change between the queries
Best when0–1 collection includes, small graphs2+ collection includes, large children
✅ Rule of Thumb

Reference includes (o.Customer) never multiply rows — they just add columns; no split needed. One collection include: single query is fine. Two or more collection includes: reach for AsSplitQuery(). You can also make splitting the default for the whole context (UseQuerySplittingBehavior) and opt back with AsSingleQuery() per query.

🍱 Analogy

A single giant JOIN is one enormous thali plate where the rice gets re-served next to every single curry so everything shares one tray — massive and repetitive. Split queries are a bento box: each dish in its own compartment, delivered together, nothing duplicated. Three small trips beat one absurd platter.

09🔬 Inspecting the Generated SQL

The single habit that separates confident EF developers from hopeful ones: read the SQL. Never guess what a query does — look. You have three tools, from quickest to most thorough:

1 — ToQueryString(): instant peek 👀

C# · Print the SQL without running it
var query = db.Orders
    .Where(o => o.Total > 1000)
    .OrderByDescending(o => o.PlacedAt)
    .Take(10);

Console.WriteLine(query.ToQueryString());   // works before executing — great in debugger
SQL · What it prints (SQL Server)
-- Parameters are declared, not inlined — EF always parameterizes:
DECLARE @__p_0 int = 10;

SELECT TOP(@__p_0) [o].[Id], [o].[CustomerId], [o].[PlacedAt], [o].[Status], [o].[Total]
FROM [Orders] AS [o]
WHERE [o].[Total] > 1000.0
ORDER BY [o].[PlacedAt] DESC
⚠️ Small Caveat

For a query using AsSplitQuery(), ToQueryString() shows only the first of the split statements. To see them all, use logging (below), which prints every command actually sent.

2 — Command logging: see everything the app really runs 📋

Day 6 pays off here. EF logs every executed command under the category Microsoft.EntityFrameworkCore.Database.Command — raise it to Information in development and the SQL (with timings!) streams into your console:

JSON · appsettings.Development.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.EntityFrameworkCore.Database.Command": "Information"
    }
  }
}

3 — Dev-only extras: LogTo + sensitive data 🔧

C# · Program.cs (Development only)
builder.Services.AddDbContext<ShopContext>(options =>
{
    options.UseSqlServer(connectionString);

    if (builder.Environment.IsDevelopment())
    {
        options.EnableSensitiveDataLogging();   // show parameter VALUES in logs
        options.EnableDetailedErrors();          // richer materialization errors
    }
});
⚠️ Sensitive Means Sensitive

By default EF logs ? placeholders instead of parameter values — deliberately, because values can be personal data (emails, names). EnableSensitiveDataLogging() reveals them, which is invaluable for debugging and unacceptable in production logs. Gate it behind an environment check, always.

What to look for when you read the SQL 🔎

  • ✅ Is the WHERE there? A missing filter means it ran client-side — your app fetched too much.
  • ✅ Is there a TOP/OFFSET…FETCH when you paginated?
  • ✅ Does the SELECT list only your projected columns, or every column?
  • 🚩 The same parameterized query repeating many times in a burst → N+1.
  • 🚩 A wall of JOINs returning huge row counts → cartesian explosion; consider AsSplitQuery.
  • 🚩 Warnings about client evaluation or untranslatable expressions → rewrite the condition.

10🛠️ Hands-On: Query Your DbContext, Read Your SQL

Today's Do: write LINQ queries against yesterday's ShopContext and inspect the generated SQL for each. Seed ~5 customers, ~20 orders, and a few products first, then work through:

  1. 📋 Turn on SQL visibility: set Microsoft.EntityFrameworkCore.Database.Command to Information and add EnableSensitiveDataLogging() for Development.
  2. 🔎 Filter + sort + page: orders over some total, newest first, page 1 of 10. Confirm in the SQL that WHERE, ORDER BY, and OFFSET…FETCH all appear.
  3. 🕵️ Tracking experiment: load an order tracked, change its status, save — watch the UPDATE list only the changed column. Then repeat with AsNoTracking() and observe that SaveChanges does nothing.
  4. 🎯 Projection: a customer summary (name, order count, total spent) via Select. Compare its SELECT column list against the whole-entity version of the same query.
  5. 🧩 Include: one order with Customer and Items.ThenInclude(Product). Read the JOINs. Then deliberately write the N+1 loop version and count the queries in your log.
  6. 🍱 Split it: a query with two collection includes — run once as a single query, once with AsSplitQuery(), and compare the logged statements and row shapes.
  7. 👀 Use query.ToQueryString() in the debugger on your favorite query — no execution needed.
  8. ⏳ Confirm every terminal call in your services is the …Async version and accepts a CancellationToken.
C# · A worthy target query
// "Top customers this year" — everything from today in one query:
public async Task<List<TopCustomerDto>> GetTopCustomersAsync(CancellationToken ct)
{
    return await db.Customers
        .Where(c => c.Orders.Any(o => o.PlacedAt.Year == 2026))
        .Select(c => new TopCustomerDto(
            c.Id,
            c.Name,
            c.Orders.Count(o => o.PlacedAt.Year == 2026),
            c.Orders.Where(o => o.PlacedAt.Year == 2026).Sum(o => o.Total)))
        .OrderByDescending(x => x.TotalSpent)
        .Take(5)
        .ToListAsync(ct);    // projection → no tracking, minimal columns, 1 query
}

public record TopCustomerDto(int Id, string Name, int OrderCount, decimal TotalSpent);
✅ Self-Check

For each query you wrote today, answer without running it: How many round trips? Which columns come back? Is anything tracked? Then verify against the log. When your predictions match the SQL three times in a row, you've got it.

12🎤 Interview Questions

Click any question to reveal a plain-English answer. Levels: 🌱 Junior (0–2 yrs) · 🟠 Mid (2–5 yrs) · 🔴 Senior (5–7 yrs).

🌱 Junior

Q1What is deferred execution in EF Core?🌱 JUNIOR

Writing a LINQ query doesn't run anything — it builds a description of the query (an expression tree). The database is only contacted when a terminal operator like ToListAsync(), FirstOrDefaultAsync(), or CountAsync() is called, at which point EF translates the whole chain into one SQL statement and executes it. The benefit: you can compose a query in steps (add filters conditionally) and still get a single efficient SQL statement at the end.

Q2What's the difference between IQueryable<T> and IEnumerable<T> in practice?🌱 JUNIOR

IQueryable is a query description that EF can still translate to SQL — operations on it run in the database. IEnumerable is an in-memory sequence — operations run in your app, item by item. The practical consequence: call ToList() too early and every later Where or Take happens in memory after fetching far too many rows. Keep the query as IQueryable until the final terminal call.

Q3What does change tracking do during a query?🌱 JUNIOR

By default, every entity a query returns is registered with the context's change tracker, which stores a snapshot of its property values. Later, when you call SaveChangesAsync(), EF compares current values against the snapshots to figure out exactly which columns changed and generates the matching UPDATE statements. It's what makes "load, modify, save" work without you writing any SQL.

Q4What does AsNoTracking() do and when should you use it?🌱 JUNIOR

It tells EF: "I'm only reading — don't track these entities." EF skips creating tracker entries and snapshots, which makes the query faster and lighter on memory (often dramatically so for large result sets). Use it for any read-only path: GET endpoints, lists, reports, dashboards. Don't use it when you plan to modify and save the loaded entities in the same context — untracked changes are silently ignored by SaveChanges.

Q5What is a projection and why is it useful?🌱 JUNIOR

A projection uses Select to shape the result into exactly the fields you need — typically a small DTO — instead of returning whole entities. EF translates it to a SQL statement selecting only those columns. Benefits: less data over the network, no change-tracking overhead (DTOs aren't entities), and your API's response shape stops being welded to your database schema.

Q6Why should EF Core queries in a web app be async?🌱 JUNIOR

Database calls are mostly waiting. A synchronous call blocks the request thread for that whole wait; an async call (ToListAsync instead of ToList) releases the thread to serve other requests and resumes when the data arrives. One query isn't faster — but the server handles far more concurrent users with far fewer threads. In web applications, async data access is the default, not an optimization.

Q7What does Include do, and what happens if you forget it?🌱 JUNIOR

Include eagerly loads a navigation property along with the main entity — db.Orders.Include(o => o.Items) fetches each order together with its items, via a SQL JOIN. If you forget it (and lazy loading is off, the common setup), the navigation simply stays empty: order.Items is an empty list, with no error. ThenInclude continues one level deeper, e.g. items → their products.

Q8How can you see the SQL that EF Core generates?🌱 JUNIOR

Two easy ways. (1) query.ToQueryString() returns the SQL for a query without executing it — handy in the debugger. (2) Logging: set the category Microsoft.EntityFrameworkCore.Database.Command to Information and every executed command is printed with its parameters and timing. In development you can additionally call EnableSensitiveDataLogging() to see actual parameter values.

🟠 Mid

Q9Explain the N+1 query problem and two ways to fix it.🟠 MID

N+1 happens when you load N parent rows with one query and then trigger one additional query per parent for its children — typically by touching navigation properties in a loop, or querying inside a loop. 200 customers become 201 round trips, and latency multiplies. Fixes: (1) eager-load the children up front with Include, turning it into one (or a few) queries; (2) better for read paths, use a projection whose Select reaches into the navigation — EF generates the JOIN and you also skip tracking and extra columns. You spot N+1 in logs as the same parameterized query repeating in rapid bursts.

Q10What is a cartesian explosion, and how does AsSplitQuery help?🟠 MID

A single SQL statement returns flat rows, so including multiple collections multiplies them: a customer with 10 orders and 5 addresses produces 10×5 = 50 rows, with the customer's data duplicated in each — wasted network, memory, and materialization work. AsSplitQuery() makes EF issue separate simple queries instead (customers; their orders; their addresses) and stitch results in memory: no duplication, at the cost of extra round trips and a slightly looser consistency guarantee (data may change between the statements). Rule of thumb: 2+ collection includes → split.

Q11When is tracked loading the right choice, given no-tracking is faster?🟠 MID

Whenever the point of loading is to modify and save. The tracker is what lets you write entity.Status = X; await SaveChangesAsync(); and get a precise UPDATE of just the changed columns — implementing that by hand (attach, mark properties modified) is more code and more error-prone. Tracked loading also gives identity resolution (one instance per row), which matters when your logic navigates a graph and expects shared references. In short: commands/updates → tracked; queries/reads → no-tracking or projection.

Q12Why must you never run two EF operations in parallel on the same DbContext, and what's the correct alternative?🟠 MID

DbContext is not thread-safe: it has single-instance state (the change tracker, the open connection) that concurrent operations would corrupt, so EF throws an InvalidOperationException if a second operation starts while one is in flight — e.g., two un-awaited queries combined with Task.WhenAll. Correct alternatives: await each call sequentially on the one context (fine for most cases), or, when true parallelism is worth it, give each parallel operation its own context instance via IDbContextFactory<T>.

Q13What is a filtered Include and where does its filter apply?🟠 MID

Since EF Core 5 you can filter and order the children being included: Include(c => c.Orders.Where(o => o.PlacedAt.Year == 2026)) loads each customer with only their 2026 orders. Crucially, the filter applies to which children are loaded, not to which parents are returned — customers with zero 2026 orders still appear, just with empty collections. To filter the parents too, add a normal Where on the root query.

Q14EF throws "the LINQ expression could not be translated." What's going on and how do you fix it?🟠 MID

Your query contains something EF can't express in SQL — commonly a call to your own C# method inside a Where/Select, or an unsupported construct. Since EF Core 3, EF refuses to silently fetch everything and filter in memory; it throws instead. Fixes, in order of preference: rewrite the condition using translatable building blocks (basic operators, supported string/date methods, EF.Functions.Like and friends); precompute values into local variables that become parameters; or — consciously and last — materialize a narrowed result with ToListAsync() and finish the untranslatable part in memory, accepting the cost explicitly.

Q15FirstOrDefaultAsync vs SingleOrDefaultAsync vs FindAsync — when do you use which?🟠 MID

FirstOrDefaultAsync returns the first match or null — use with an ordering or when several matches are acceptable; SQL gets TOP(1). SingleOrDefaultAsync asserts at most one match — it fetches up to 2 rows and throws if it finds a second; use it when duplicates would indicate a data bug you want surfaced. FindAsync(id) looks up by primary key and checks the change tracker first — if the entity is already loaded in this context, it returns it with no database round trip; ideal for key lookups on the update path.

Q16Why does EF Core parameterize queries, and what does that protect against?🟠 MID

Values from your LINQ (variables, method arguments) become SQL parameters (@__p_0) rather than being pasted into the SQL text. Two big wins: (1) security — user input can never change the query's structure, which is the defense against SQL injection; (2) performance — the SQL text stays identical across calls with different values, so the database reuses one cached execution plan instead of compiling a new one each time. You can see the parameters yourself in ToQueryString() output or command logs.

🔴 Senior

Q17Your API endpoint is slow. Walk me through diagnosing an EF query performance issue.🔴 SENIOR

First, get evidence: enable command logging (with timings) or tracing and identify which query dominates — is it one slow statement, or many fast ones (N+1)? For many-fast: find the loop, fix with Include or projection. For one-slow: capture the SQL via ToQueryString()/logs and examine it — is the WHERE translated or is the app filtering in memory? Is it selecting every column when a projection would do? Huge row counts from multiple collection includes → cartesian explosion, consider AsSplitQuery. Then look below EF: run the SQL in SSMS with the actual execution plan — missing index? scan instead of seek? Fix at the right layer (query shape first, then indexes via HasIndex), and verify with the same measurements. Also check the cheap wins: AsNoTracking on reads and pagination on unbounded lists.

Q18Compare eager, explicit, and lazy loading. What's your default policy and why?🔴 SENIOR

Eager (Include): related data fetched with the main query — explicit at the query site, predictable SQL. Explicit (context.Entry(e).Collection(...).LoadAsync()): load navigations on demand with a visible, deliberate call — useful for conditional loading. Lazy (proxy-based auto-loading on first property access): convenient but dangerous — every forgotten Include becomes a silent extra query, and a loop over entities becomes an invisible N+1; it also tempts data access from layers that shouldn't touch the database. My default: lazy loading off; projections for read paths; Include for command paths; explicit loading for the rare conditional case. Predictability beats convenience in data access.

Q19How do split queries interact with consistency, and when would you refuse to use them?🔴 SENIOR

A split query executes several statements; by default each sees the database as of its own execution, so a concurrent write can land between them — parents fetched before the change, children after, yielding a graph that never existed at any single moment. Usually acceptable for display data. I'd avoid splitting (or wrap the operation in an explicit transaction with an appropriate isolation level, e.g. snapshot) when the read feeds a business decision that must see one consistent state — financial calculations, stock allocation, anything audited. Also not worth it when the graph is small or latency per round trip is high; the default single query is one consistent snapshot, and that property is sometimes exactly what you're paying for.

Q20When would you drop to raw SQL from EF Core, and how do you do it safely?🔴 SENIOR

Drop down when the query is clearer or faster expressed as SQL than as LINQ: exotic constructs EF won't translate, hand-tuned reporting queries, hints, or calling stored procedures. Safely means: use FromSql($"...{param}...") / SqlQuery, whose interpolation is converted into parameters (never string-concatenate user input — that's the injection hole); keep the raw SQL composable where possible (FromSql results can still have Where/OrderBy LINQ appended and translated); and keep such queries few, well-named, and covered by integration tests since the compiler no longer validates them against the model. Raw SQL is a scalpel, not the default knife.

Q21What are ExecuteUpdateAsync / ExecuteDeleteAsync, and how do they change the classic load-modify-save pattern?🔴 SENIOR

They're set-based write operations (EF Core 7+): db.Orders.Where(o => o.Status == Stale).ExecuteDeleteAsync() or .ExecuteUpdateAsync(s => s.SetProperty(o => o.Status, Archived)) run a single UPDATE/DELETE in the database — no entities loaded, no tracking, no SaveChanges. For bulk operations this replaces the painful "load 50,000 rows, mutate, save" pattern with one statement. Caveats: they execute immediately and independently of SaveChanges (so they don't participate in the unit of work unless you open an explicit transaction), they bypass the change tracker (already-tracked entities can become stale), and they skip EF-level cascade logic — database-level constraints govern. I use them for archiving, purging, counters, and other set-shaped writes; classic tracked updates remain right for rich per-entity business logic.

Q22A code review shows `.Where(o => o.PlacedAt.ToString("yyyy-MM") == month)`. Critique it.🔴 SENIOR

Multiple problems. It may fail to translate (or translate into a string-formatting expression evaluated per row), and even when it produces SQL, wrapping the column in a function makes the predicate non-sargable — the database can't use an index on PlacedAt and must scan every row, converting each date to a string to compare. Culture/format subtleties add correctness risk. The fix is a range predicate on the raw column: compute var start = new DateTime(y, m, 1); var end = start.AddMonths(1); and write Where(o => o.PlacedAt >= start && o.PlacedAt < end) — index-friendly, translation-safe, and timezone-explicit. General principle: manipulate the parameters, not the column, in filter conditions.

Q23How would you establish query-quality guardrails for a team new to EF Core?🔴 SENIOR

Make the good path the easy path. Conventions: projections for all read endpoints, AsNoTracking where entities are returned read-only, mandatory pagination on list queries, async-only terminal operators with CancellationTokens, lazy loading disabled. Visibility: command logging on in dev, and developers expected to read their SQL before merging (a PR habit: paste the generated SQL for any non-trivial query). Automation where possible: analyzer rules or code review checklists for .Result/.Wait(), ToList() mid-query, and unbounded queries; integration tests that assert query counts for hot endpoints, which catches N+1 regressions mechanically. Finally, a short "query cookbook" in the repo with the team's blessed patterns — people copy what's nearest.

Q24Compiled queries: what problem do they solve and when are they worth it?🔴 SENIOR

Every EF query pays a small cost to process its expression tree and look up (or produce) the cached translation. For most apps that cost is noise. In extremely hot paths — the same query shape executed thousands of times per second — EF.CompileAsyncQuery lets you pre-compile a query into a delegate once and invoke it with parameters, skipping per-call expression handling entirely. Worth it only after measurement shows query compilation/lookup overhead matters (rare), and it carries ergonomics costs: fixed parameters, stored delegates, less flexibility. Order of optimization: fix N+1 and over-fetching first, add indexes second, no-tracking and projections third — compiled queries are the last few percent, not the first fix.
