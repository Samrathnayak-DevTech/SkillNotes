Dependency Injection 🧩

Instead of a class creating its own dependencies with new, dependency injection (DI) has them handed to it — usually through its constructor — by a central container. .NET has one built in: you register your services on the host builder, and the container resolves and constructs them for you, respecting a chosen lifetime (transient, scoped, or singleton). That loose coupling makes code easier to swap, test, and reason about. Today you'll register two services, inject one into the other, and watch the lifetimes behave differently. Builds on the PeopleApp host from Day 2.

01

What DI Is

Dependency injection means a class receives the things it needs from outside, rather than creating them itself. A container does the wiring.

A "dependency" is any object a class needs to do its job — a logger, a data-access service, a clock. With DI, a class declares what it needs (typically as constructor parameters) and something else — the DI container — supplies those dependencies at runtime. The class no longer knows or cares how its dependencies are built; it just uses them. This flips the usual control: instead of your code reaching out to create collaborators, they're handed in. That flip is called Inversion of Control (IoC), and DI is the most common way to achieve it.

🍽️ Analogy

Think of a restaurant kitchen. A chef (your class) doesn't go farm the vegetables, raise the cattle, and mill the flour before cooking — ingredients (dependencies) are delivered to the kitchen by suppliers. The chef just declares what's needed on the order sheet. The container is the supply system that sources and delivers exactly the right ingredients when a dish is prepared, so the chef can focus on cooking.

03

The Problem It Solves

Creating dependencies with new inside a class hard-wires it to specific implementations — making it rigid and hard to test.

csharp · tightly coupled (avoid)
public class GreetingService
{
    // Hard-wired: this class MAKES its own dependencies.
    private readonly SystemClock _clock = new();
    private readonly Greeter _greeter = new();

    public string Greet(string name)
        => $"{_greeter.Greet(name)} It's {_clock.Now:t}.";
}

This looks fine until you need to change something. Want to swap the clock for a fake one in a test? You can't — it's welded in with new. Want to use a different clock implementation in production? Same problem. The class is tightly coupled to concrete types it created itself.

csharp · loosely coupled (with DI)
public class GreetingService : IGreetingService
{
    private readonly IClock _clock;
    private readonly Greeter _greeter;

    // Dependencies are DECLARED and handed in — not created here.
    public GreetingService(IClock clock, Greeter greeter)
    {
        _clock = clock;
        _greeter = greeter;
    }

    public string Greet(string name)
        => $"{_greeter.Greet(name)} It's {_clock.Now:t}.";
}
🔑 Depend on abstractions, not concretions

By depending on the IClock interface rather than a concrete class, the service doesn't care which clock it gets — a real one in production, a fake one in tests. The container decides. This is the heart of DI's value: loose coupling lets you swap implementations and test in isolation without touching the class that uses them.

🔑 Register before Build, resolve after

You add all your services to builder.Services before calling Build(). Build() turns that list into a live IServiceProvider (the container). After that, the registrations are fixed and you resolve services from the provider — usually by injecting them, occasionally by asking the provider directly. 05

Registering Services

You register each service with a lifetime, mapping an interface to the implementation the container should build for it.

csharp · in Program.cs, before Build()
var builder = Host.CreateApplicationBuilder(args);

// interface → implementation, with a lifetime
builder.Services.AddSingleton<IClock, SystemClock>();       // one shared clock
builder.Services.AddTransient<IGreetingService, GreetingService>();

// a concrete type with no interface is fine too
builder.Services.AddSingleton<Greeter>();

var host = builder.Build();

🔑 The interface → implementation mapping

AddSingleton<IClock, SystemClock>() says: "when anyone asks for IClock, give them a SystemClock." Consumers depend only on IClock; the registration is the single place that picks the concrete type. Change one line and the whole app uses a different implementation — that's the swap-ability DI buys you.

06

Constructor Injection

The container reads a class's constructor, resolves each parameter from its registrations, and passes them in. This is the standard, preferred form of injection.

csharp
public class GreetingService : IGreetingService
{
    private readonly IClock _clock;

    // The container sees this needs an IClock and supplies one.
    public GreetingService(IClock clock) => _clock = clock;

    public string Greet(string name)
        => $"Hello {name}, it's {_clock.Now:t}.";
}

When something asks the container for IGreetingService, the container sees that GreetingService needs an IClock, looks up that registration, builds (or reuses) a clock, and passes it to the constructor — then hands you the finished service. If the clock had its own dependencies, they'd be resolved too, all the way down. The container builds the whole dependency graph for you.

🔑 Prefer constructor injection

Constructor injection makes a class's dependencies explicit and required — you can't construct the object without them, so there are no half-initialized instances, and the constructor signature documents exactly what the class needs. Store them in readonly fields. Alternatives (property or method injection) exist but are rarer; constructor injection is the default you should reach for.

✅ You rarely call the provider yourself

In most code you never touch IServiceProvider directly — you just declare constructor parameters and the container fills them in as it builds your objects. Resolving manually with GetRequiredService<T>() is mainly for the composition root (like Program.cs) or special cases; reaching for it deep in your code is usually a smell (the service-locator anti-pattern, section 13).

07

Transient

Transient services are created fresh every time they're requested — a brand-new instance for each injection and each resolve.

csharp
builder.Services.AddTransient<IGreetingService, GreetingService>();

// Every request for IGreetingService returns a NEW GreetingService.
🔑 A fresh instance, always

Ask for a transient service twice and you get two different objects. That makes transients safe for stateless, lightweight services where sharing doesn't matter — there's no shared state to worry about, but you pay to construct one each time. It's the safest default when you're unsure: a new instance can't accidentally leak state between callers.

08

Scoped

Scoped services are created once per scope — the same instance is reused within a scope, but a new one is made for each new scope.

csharp
builder.Services.AddScoped<IdProvider>();

// One IdProvider per scope; everyone in the SAME scope shares it.
using (var scope = host.Services.CreateScope())
{
    var a = scope.ServiceProvider.GetRequiredService<IdProvider>();
    var b = scope.ServiceProvider.GetRequiredService<IdProvider>();
    // a and b are the SAME instance (same scope)
}
🔑 "Scope" usually means "one web request"

A scope is a bounded lifetime you create with CreateScope(). In ASP.NET Core, the framework creates one scope per HTTP request — so a scoped service is effectively "one per request": every component handling that request shares the same instance, and the next request gets a fresh one. This is exactly why DbContext (Entity Framework) is registered scoped — one unit of work per request.

✅ In a console app, you make scopes yourself

A plain console app has no automatic per-request scope, so to use scoped services you explicitly create one with host.Services.CreateScope() (as above). Web apps do this for you on every request. Either way, the rule is the same: one instance per scope, disposed when the scope ends.

09

Singleton

Singleton services are created once and the same instance is shared for the entire lifetime of the application.

csharp
builder.Services.AddSingleton<IClock, SystemClock>();

// ONE SystemClock for the whole app — every request/scope shares it.
🔑 One instance, app-wide, forever

A singleton is built the first time it's needed (or at startup) and then reused everywhere for the life of the process. That's ideal for stateless, expensive-to-create, or genuinely shared services — a clock, a cache, a configuration holder, an HTTP client factory. Because it's shared across all threads and requests, a singleton must be thread-safe.

⚠️ Singletons and mutable state are dangerous

Since one singleton instance serves every request simultaneously, any mutable state inside it is shared across all callers and threads at once. If it isn't carefully made thread-safe, you get race conditions and data corruption. Keep singletons stateless, or protect their state with proper synchronization. Stateless singletons are the easy, safe case.

10

Compared

Three lifetimes, one decision. Pick based on how much state is shared and how long it should live.

LifetimeNew instance…Shared acrossTypical use
TransientEvery request/injectionNothing — always freshLightweight, stateless helpers
ScopedOnce per scopeOne scope / one web requestDbContext, per-request state
SingletonOnce for the appThe whole applicationCaches, clocks, config, factories

Which lifetime should I pick? 🧭

🪶 Stateless & lightweight, no sharing needed? Use Transient — a safe default; a new instance can't leak state.
🔄 Tied to one unit of work / web request (holds per-request state)? Use Scoped — one instance per request, e.g. a database context.
🌐 Shared app-wide, expensive to build, and thread-safe? Use Singleton — one instance forever (keep it stateless or synchronized).
🔑 The golden rule of lifetimes

A service may only depend on others with an equal or longer lifetime. Transient and scoped services can safely use a singleton; a singleton must not hold onto a scoped service (that's the captive dependency in section 12). When unsure, start with the shortest safe lifetime (transient/scoped) and promote to singleton only when you have a reason.

11

Hands-On

The Day-3 exercise: build two services — an IdProvider and a Consumer that receives it — then change the registered lifetime and watch the injected Id behave differently across resolves and scopes.

csharp · PeopleApp.Core/Services.cs
namespace PeopleApp.Core;

// Service 1: holds a unique Id created when the instance is built.
public class IdProvider
{
    public Guid Id { get; } = Guid.NewGuid();
}

// Service 2: depends on Service 1 (injected via the constructor).
public class Consumer
{
    public IdProvider Provider { get; }
    public Consumer(IdProvider provider) => Provider = provider;
}
csharp · PeopleApp.Cli/Program.cs
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using PeopleApp.Core;

var builder = Host.CreateApplicationBuilder(args);

// 🔁 Change this ONE line to Transient / Scoped / Singleton and re-run.
builder.Services.AddScoped<IdProvider>();
builder.Services.AddScoped<Consumer>();

var host = builder.Build();

static void Report(string label, IServiceProvider sp)
{
    var direct   = sp.GetRequiredService<IdProvider>();
    var consumer = sp.GetRequiredService<Consumer>();
    Console.WriteLine($"{label}: direct={direct.Id.ToString()[..8]}  " +
                      $"injected={consumer.Provider.Id.ToString()[..8]}");
}

using (var scope1 = host.Services.CreateScope())
    Report("scope 1", scope1.ServiceProvider);

using (var scope2 = host.Services.CreateScope())
    Report("scope 2", scope2.ServiceProvider);

✅ What just happened

You registered two services and let the container inject IdProvider into Consumer via its constructor. By comparing the Id the Consumer received against one resolved directly — and across two scopes — you can see the lifetime: transient gives a new object every time, scoped shares within a scope but not across them, and singleton is one object for the entire app. Flip the one registration line and re-run to watch the pattern change.

12

Captive Dependency

The most famous DI bug: a long-lived service holds a reference to a shorter-lived one, trapping it and breaking its intended lifetime.

csharp · the trap
builder.Services.AddScoped<IdProvider>();       // meant to be per-scope
builder.Services.AddSingleton<Consumer>();       // ❌ singleton...

// ...but Consumer's constructor needs the SCOPED IdProvider.
// The singleton captures ONE IdProvider forever — it never gets a
// fresh per-scope instance again. Its "scoped" dependency is captive.

Because the singleton Consumer is created once and lives forever, the scoped IdProvider it grabbed at construction is stuck living just as long — it will never be replaced per scope as intended. The scoped service has been captured, silently defeating its lifetime and often leaking state or resources across requests.

🔑 The rule, restated

Dependencies must live at least as long as the thing that holds them. A singleton may depend on singletons only; a scoped service may depend on scoped or singleton; a transient may depend on anything. Going the wrong way — a longer-lived service capturing a shorter-lived one — is the captive-dependency bug.

✅ Scope validation catches it for you

The .NET container has built-in scope validation that throws an error when a singleton tries to consume a scoped service, so the bug fails loudly instead of silently. It's enabled by default in the Development environment (from Day 2!). If a singleton genuinely needs scoped work, the fix is to inject an IServiceScopeFactory and create a scope on demand rather than capturing the scoped service.

13

Mistakes & Practices

DI habits that keep your object graph clean, testable, and bug-free.

🏗️

Use constructor injection

Make dependencies explicit and required; store them in readonly fields.

🔌

Depend on interfaces

Register interface → implementation so you can swap and mock without touching consumers.

📏

Respect the lifetime rule

Never let a longer-lived service capture a shorter-lived one.

🔒

Keep singletons thread-safe

One instance serves all threads; avoid unsynchronized mutable state.

🚫

Avoid the service locator

Don't inject IServiceProvider to fetch things at will — it hides dependencies.

🧪

Lean on it for tests

Inject fakes/mocks of interfaces to test a class in isolation, no container needed.

⚠️ The service-locator anti-pattern

Injecting IServiceProvider everywhere and calling GetService<T>() to grab dependencies on the fly looks convenient, but it hides what a class actually depends on — the constructor no longer tells the truth, and testing gets harder. Prefer declaring real constructor parameters. Resolving from the provider is appropriate at the composition root and in a few framework scenarios, not as an everyday pattern.

15

Interview Questions

Click any question to reveal a plain-English answer. Tagged by experience level.

Q01 What is dependency injection? Junior

A pattern where a class receives the objects it depends on from outside — usually through its constructor — instead of creating them itself. A DI container does the wiring: you register services, and it constructs them and supplies their dependencies. The result is loosely coupled code that's easy to swap and test.

Q02 What problem does DI solve? Junior

Tight coupling. When a class creates its own dependencies with new, it's welded to specific concrete types — you can't swap implementations or substitute fakes for testing. DI has dependencies handed in against interfaces, so you can change or mock them without modifying the class, making code flexible and testable.

Q03 What is constructor injection? Junior

Declaring a class's dependencies as constructor parameters, which the container resolves and passes in when it builds the object. It's the preferred form because the dependencies are explicit and required — you can't create the object without them — and the constructor documents exactly what the class needs.

Q04 What are the three service lifetimes? Junior

Transient (a new instance every time it's requested), Scoped (one instance per scope — typically one per web request), and Singleton (one instance shared for the whole application's lifetime). You choose per service via AddTransient, AddScoped, or AddSingleton.

Q05 What is a transient service? Junior

One that the container creates fresh every time it's requested — two requests give two different objects. It suits lightweight, stateless services where sharing isn't needed and a new instance can't leak state. It's a safe default when you're unsure which lifetime to use.

Q06 What is a singleton service? Junior

One that's created once and shared for the entire life of the application — every request and scope gets the same instance. Good for stateless, expensive-to-create, or genuinely shared services like caches, clocks, and config holders. Because it's shared across all threads, it must be thread-safe.

Q07 How do you register a service? Junior

On the host builder's service collection, before Build(): for example builder.Services.AddSingleton<IClock, SystemClock>(). You pick the lifetime by which method you call (AddTransient/AddScoped/AddSingleton) and map the requested type to its implementation.

Q08 How do you register an interface with its implementation? Junior

Use the two-type generic form: AddScoped<IGreetingService, GreetingService>(). This tells the container "when someone asks for IGreetingService, build a GreetingService." Consumers depend only on the interface, and this single registration decides the concrete type used everywhere.

Q09 Transient vs scoped vs singleton — what are the differences? Mid

Transient: a new instance on every resolve/injection — nothing shared. Scoped: one instance per scope, so everything within a single scope (one web request) shares it, but a new scope gets a new one. Singleton: one instance for the whole app, shared across all scopes and threads. The choice depends on how much state should be shared and how long the object should live.

Q10 What is a "scope"? Mid

A bounded lifetime boundary the container uses to group scoped services. Within a scope, each scoped service is created once and reused; when the scope is disposed, its scoped services are disposed too. You create one explicitly with CreateScope(); in ASP.NET Core the framework creates a scope automatically per HTTP request.

Q11 What does scoped mean in a web app? Mid

Effectively "one instance per HTTP request." ASP.NET Core opens a new scope at the start of each request and disposes it at the end, so every component involved in handling that request shares the same scoped instance, and the next request gets a fresh one. This is why Entity Framework's DbContext is scoped — one unit of work per request.

Q12 How does the container resolve constructor dependencies? Mid

When asked for a type, the container inspects its constructor, resolves each parameter from the registrations (creating or reusing instances per their lifetimes), then invokes the constructor with them — recursively building the whole dependency graph down to leaf services. If a required dependency isn't registered, it throws. This is why you rarely construct objects yourself.

Q13 What's the difference between IServiceCollection and IServiceProvider? Mid

IServiceCollection is the registration list you populate before building — it holds the "recipes" (service type, implementation, lifetime). IServiceProvider is the built container produced by Build(); it actually creates and hands out instances. You register into the collection, then resolve from the provider (usually via injection).

Q14 Why prefer DI over new? Mid

Using new hard-wires a class to concrete types, so you can't swap implementations or substitute test doubles, and lifetime management is manual and scattered. DI centralizes construction, lets you depend on interfaces (swap/mock freely), manages lifetimes and disposal for you, and makes dependencies explicit. The payoff is loose coupling and testability.

Q15 Can you register multiple implementations of an interface? Mid

Yes — register several implementations of the same interface, and inject IEnumerable<IMyService> to receive all of them (useful for plugin-style or strategy scenarios). If you resolve the single interface, you get the last one registered. Newer .NET also supports keyed services (AddKeyedScoped, injected with a [FromKeyedServices] key) to pick a specific one by key.

Q16 What's the default lifetime you should reach for? Mid

Start with the shortest safe lifetime. Transient is the safest default for stateless helpers; use scoped for things tied to a unit of work like a DbContext; and reserve singleton for genuinely shared, thread-safe, or expensive services. Promote to a longer lifetime only when you have a concrete reason, since longer lifetimes bring shared-state and thread-safety concerns.

Q17 What is a captive dependency and how do you avoid it? Senior

It's when a longer-lived service holds a reference to a shorter-lived one — for example a singleton that takes a scoped service in its constructor. The scoped instance gets "captured" and lives as long as the singleton, defeating its per-scope lifetime and often leaking state or resources. Avoid it by obeying the rule that a service may only depend on equal-or-longer lifetimes; if a singleton needs scoped work, inject IServiceScopeFactory and create a scope on demand rather than capturing the scoped service.

Q18 How does scope validation work? Senior

The container can validate scopes: it throws if a singleton attempts to resolve a scoped service, and if scoped services are resolved from the root provider (outside any scope). This turns captive-dependency bugs into loud startup/runtime errors instead of silent leaks. It's enabled by default in the Development environment (and you can force it on always). This is one concrete reason the environment from Day 2 matters — validation is stricter in Development.

Q19 What is IoC and how does DI implement it? Senior

Inversion of Control is the broad principle that a framework or container — not your own code — drives the flow and object creation; your classes are called into and have collaborators supplied to them, rather than reaching out to create and control everything. Dependency injection is a specific way to realize IoC for object construction: instead of a class instantiating its dependencies, an external container constructs and injects them. DI inverts the "who creates my collaborators" decision from the class to the container.

Q20 How are IDisposable services disposed by the container? Senior

The container tracks the disposable instances it creates and disposes them when their owning scope ends: scoped and transient services created within a scope are disposed when that scope is disposed, and singletons are disposed when the root container (the app) shuts down. A caveat: transient IDisposables resolved from a long-lived provider are held until that provider is disposed, which can cause a subtle memory buildup — so be careful with disposable transients captured by long-lived scopes. Instances you create with new yourself, the container won't dispose.

Q21 Constructor vs property/method injection — trade-offs? Senior

Constructor injection makes dependencies required and explicit, guarantees a fully-initialized object, and is what the built-in container supports natively — it's the default choice. Property (setter) injection makes a dependency optional and mutable, which can leave objects half-configured and hides requirements; method injection passes a dependency to a single method call, useful when only one operation needs it. The built-in .NET container primarily does constructor injection; the others are occasional patterns, generally discouraged unless there's a specific reason (e.g., truly optional dependencies).

Q22 How do you register open generics, multiple services, or keyed services? Senior

Open generics: register the generic definition, e.g. AddScoped(typeof(IRepository<>), typeof(Repository<>)), so any IRepository<T> resolves to Repository<T> — used heavily by things like ILogger<T>. Multiple implementations: register several for one interface and inject IEnumerable<T> to get them all. Keyed services (recent .NET): AddKeyedSingleton<I, Impl>("key") and inject with [FromKeyedServices("key")] to select a specific implementation by key. You can also register factory delegates for custom construction logic.

Q23 Why is the service-locator pattern considered an anti-pattern? Senior

Because it hides dependencies. When a class injects IServiceProvider and calls GetService<T>() internally, its constructor no longer reveals what it actually needs — dependencies become implicit and discoverable only by reading the body, which hurts readability, refactoring, and testing (you must set up a whole provider instead of passing fakes). It also makes lifetime and resolution errors surface at runtime rather than construction. Prefer explicit constructor injection; using the provider directly is acceptable only at the composition root or in framework-level infrastructure.

Q24 How does DI aid testability, and how do you mock dependencies? Senior

Because a class receives its dependencies through interfaces via the constructor, in a unit test you simply construct it with test doubles — a hand-written fake or a mock from a library like Moq/NSubstitute implementing the same interface — with no container required. You control the collaborators' behavior and assert on interactions, testing the class in complete isolation. This is DI's biggest practical payoff: depending on abstractions makes substituting behavior trivial, so tests are fast, deterministic, and focused on one unit at a time.
